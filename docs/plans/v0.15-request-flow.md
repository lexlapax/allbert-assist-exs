# Allbert v0.15 Request Flow: Minimal App Registration Contract

This request-flow document is the implementation companion to
`docs/plans/v0.15-plan.md`. It describes how `AllbertAssist.App` modules are
declared, registered at boot, validated, surfaced through registered actions
and the CLI, tagged onto action capabilities, consulted by session active-app
writes, and torn down across restarts. App registration is contract data, not
authority; permission and execution decisions remain at the existing action,
Security Central, confirmation, and resource-access boundaries.

Status: implemented.

## Common Shape

Every v0.15 app-registry interaction follows the same shape:

1. The registry is started under `AllbertAssist.App.Supervisor`, between
   `AllbertAssist.Jido` and `AllbertAssist.Session.Scratchpad` in the
   application supervision tree.
2. `AllbertAssist.App.Bootstrap` reads `Application.get_env(:allbert_assist,
   :apps, [...])` and registers each module through
   `AllbertAssist.App.Registry.register/2`.
3. `AllbertAssist.App.Registry.register/2` calls the module's `validate/1`
   first; on success it stores an ETS entry and starts `child_spec/1` under
   `AllbertAssist.App.DynamicSupervisor`; on failure it records a diagnostic
   and never creates an ETS entry.
4. After bootstrap, runtime callers (Decision validation, session writes,
   action capability lookup, skills resolution, registered actions, CLI)
   consult `AllbertAssist.App.Registry` through its public API. No caller
   touches ETS directly.

App registration never authorizes work, never grants app permissions, never
modifies SQLite tables, and never writes markdown memory. It is contract data,
not authority.

## Bootstrap Flow

On `AllbertAssist.Application.start/2`:

1. Repo, DNSCluster, PubSub, SignalBus, and `AllbertAssist.Jido` start in
   their existing order.
2. `AllbertAssist.App.Supervisor` starts with `:one_for_all` strategy and
   children in order:
   1. `AllbertAssist.App.Registry` creates the named ETS table
      (`:allbert_app_registry`) and an empty diagnostics map.
   2. `AllbertAssist.App.DynamicSupervisor` starts with no children.
   3. `AllbertAssist.App.Bootstrap` reads `Application.get_env(:allbert_assist,
      :apps, [AllbertAssist.App.CoreApp, AllbertAssist.App.StockSageStub])`
      and registers each module:
      - For each `module` in the list, call
        `AllbertAssist.App.Registry.register(module, [])`.
      - The registry GenServer serializes registration calls.
      - On success, log `info`: `"App registered: #{app_id}"`.
      - On failure, log `warning`: `"App registration failed: #{app_id}: #{reason}"`,
        and persist the diagnostic.
3. `AllbertAssist.Session.Scratchpad` starts. From this point on,
   `set_active_app` writes can consult the registry for validation.
4. `AllbertAssist.Jobs.Scheduler` starts when enabled.

Bootstrap is idempotent in the sense that re-running it on the same
configured list produces the same registry state. Re-running it while the
registry already contains entries for some app ids returns
`{:error, {:app_id_taken, app_id}}` for those modules without disturbing the
existing registration; this is the documented duplicate-register behavior.

## App Registration Flow

For `AllbertAssist.App.Registry.register(module, opts)`:

1. The registry validates the module shape:
   - `module` is loaded and exports `app_id/0`, `display_name/0`,
     `version/0`, `validate/1`, `child_spec/1`, `actions/0`, `skill_paths/0`.
   - `module.app_id/0` returns an atom matching
     `~r/^[a-z][a-z0-9_]*$/` after `Atom.to_string/1`.
   - `module.display_name/0` returns a trimmed non-empty string, 1..64
     characters.
   - `module.version/0` returns a non-empty string, 1..32 characters.
2. The registry calls `module.validate(opts)`. Failure returns
   `{:error, {:validation_failed, diagnostics}}`, records the diagnostic in
   the registry diagnostics map under `app_id`, and stops the flow.
3. The registry checks for a duplicate `app_id`. If a registered entry
   already exists, returns `{:error, {:app_id_taken, app_id}}` without
   disturbing the existing entry.
4. The registry validates the optional `surfaces/0` callback:
   - Each entry has `id` (atom), `label` (trimmed string), `path` (string
     starting with `/`, 1..128 chars, no whitespace, no query/fragment),
     `app_id` (atom, must equal `module.app_id/0`), and optional `icon` and
     `description`.
   - Per-app surface `id` values must be unique.
   - Cross-app surface `id` duplicates are recorded in diagnostics but do
     not fail registration.
5. The registry validates `actions/0`:
   - Each module must resolve through
     `AllbertAssist.Actions.Registry.resolve/1`. Unknown action modules
     produce `{:error, {:unknown_action_module, module}}` and fail
     registration.
6. The registry validates `skill_paths/0`:
   - Each path must be a binary string, absolute, and 1..256 chars.
   - Paths are not required to exist on disk at registration time; missing
     paths surface later as skills-loader diagnostics, not as registration
     failures.
7. The registry calls `module.child_spec(opts)`:
   - If `:ignore`, no child is started; the entry stores `child_id:
     :ignore`, `child_pid: :ignore`.
   - Otherwise, `DynamicSupervisor.start_child(AllbertAssist.App.DynamicSupervisor,
     spec)` runs. On success, store the stable child spec `:id` as
     `child_id` and the observed pid as `child_pid`. On failure, return
     `{:error, {:child_spec_failed, detail}}`, record the diagnostic, and do
     not insert an ETS entry.
8. The registry inserts the ETS entry keyed by `app_id` with the documented
   shape. Returns `{:ok, app_id}`.

The registration call is synchronous from the caller's perspective. The
registry GenServer serializes registration requests, so concurrent
`register/2` calls do not race on duplicate detection.

## Unregistration Flow

`AllbertAssist.App.Registry.unregister(app_id)` is exposed for tests and
v0.20 (which removes `StockSageStub` when `./plugins/stocksage` contributes
`StockSage.Plugin` and `StockSage.App`):

1. Look up the ETS entry for `app_id`. Missing entries return `:ok` (idempotent).
2. If `child_pid` is a live pid, call `DynamicSupervisor.terminate_child/2`.
   If the stored child pid is already gone, continue idempotently; the registry
   entry still needs to be removed.
4. Delete the ETS entry.
5. Remove the app's diagnostics entry, if any.
6. Return `:ok`.

Production code does not call `unregister/1` during normal operation.
v0.20's removal of `StockSageStub` follows this pattern: the stub is removed
from the bootstrap defaults, `./plugins/stocksage` contributes
`StockSage.Plugin` and `StockSage.App` through the v0.17 plugin substrate, and
the application restarts. Calling `unregister(:stocksage)` mid-run is a
test-only fallback.

## Validation Failure Flow

For `mix allbert.apps validate AllbertAssist.App.CoreApp`:

1. CLI parses the module name as a binary and resolves it against already
   compiled module atoms from loaded OTP application specs and
   `:code.all_loaded/0`. It does not call `String.to_atom/1`,
   `String.to_existing_atom/1`, or `Module.concat/1` on arbitrary operator
   input.
2. CLI checks the resolved module is loaded and implements
   `AllbertAssist.App`.
   Missing or non-conforming modules return a redacted error and exit
   non-zero without touching the registry.
3. CLI calls `module.validate([])`.
4. CLI prints either `:ok` and the module's `app_id`, `display_name`, and
   `version`, or the structured diagnostics list.
5. CLI does NOT call `AllbertAssist.App.Registry.register/2`, does NOT
   start the module's child spec, and does NOT modify any ETS state.

For a registration-time validation failure (e.g., a real app registered via
Bootstrap whose `validate/1` returns `{:error, [...]}`):

1. Bootstrap logs the failure at `warning` level.
2. The registry diagnostics map gets an entry under that `app_id`.
3. `mix allbert.apps list` shows the failing row with diagnostic kind and a
   redacted message.
4. The registry serves reads for other registered apps without disturbance.
5. The supervision tree continues; no crash, no restart.

## Lookup Flow

### From Session Writes

For:

```sh
mix allbert.sessions set-active-app --user alice --session sess-1 stocksage
```

The flow becomes (replacing the v0.14 allowlist step):

1. CLI normalizes identity and session id exactly as v0.14 defines.
2. CLI calls `AllbertAssist.Actions.Runner.run/3` with `set_active_app`
   and `%{user_id: "alice", session_id: "sess-1", app_id: "stocksage"}`.
3. The action calls `AllbertAssist.Session.AppId.normalize("stocksage")`.
4. `normalize/1` checks nil aliases (`""`, `"none"`, `"general"`) and
   short-circuits to nil for those.
5. `normalize/1` calls
   `AllbertAssist.App.Registry.normalize_app_id("stocksage")`. The registry
   helper safely resolves the binary to an existing atom only when that atom
   is already known to the running code and registered in the app registry.
6. If the registry knows the id, normalization returns `{:ok, :stocksage}`.
7. If the registry does not know the id, normalization returns
   `{:error, :unknown_app}` and `set_active_app` returns an error without
   mutating ETS scratchpad state.
8. On success, the action calls `AllbertAssist.Session.set_active_app/3`
   and the rest of v0.14's flow runs unchanged.

For `mix allbert.sessions set-active-app --user alice --session sess-1 unknown`:

1. `AllbertAssist.App.Registry.normalize_app_id/1` returns
   `{:error, :unknown_app}` for binaries that do not correspond to a
   registered app id. No new atom is created.
2. If `:unknown` happens to exist as an atom, `normalize_app_id/1` still
   returns `{:error, :unknown_app}` because no app registered with that id.
3. No scratchpad entry is created or modified.

### From Decision Validation

For an intent agent that produces a candidate decision including
`active_app: :nonexistent`:

1. `AllbertAssist.Intent.Decision.new/1` runs existing validation.
2. The `active_app` step calls
   `AllbertAssist.App.Registry.normalize_app_id(:nonexistent)`.
3. The registry returns `{:error, :unknown_app}`. The decision is rebuilt
   without the candidate `active_app` (or with the context-supplied
   `active_app` from the v0.14 scratchpad when that context value is known),
   and a structured diagnostic is added.
4. The diagnostic is included in the decision's `diagnostics` list and
   surfaces in trace metadata; it does not throw or crash.
5. The runtime continues with the cleaned-up decision.

If the candidate decision does not set `active_app` and the runtime
context-supplied `active_app` (from v0.14 scratchpad lookup) is a known
app id, the context value is preserved exactly as v0.14 documents.

### From Action Capability Reads

For `AllbertAssist.Actions.Registry.capability(SomeStockSageAction)`:

1. Look up the compile-time capability metadata for the module.
2. Call `AllbertAssist.App.Registry.app_id_for_action(SomeStockSageAction)`.
3. If the registry knows the action, the returned `Capability` struct
   carries `app_id: :stocksage`.
4. If the registry does not know the action (e.g., a core action), the
   returned `Capability` struct carries `app_id: nil`.

`AllbertAssist.Actions.Registry.capabilities_for_app/1`:

1. Call `AllbertAssist.App.Registry.actions_for(app_id)` to get the list
   of registered action modules.
2. Resolve each module through the existing capability map.
3. Return the list of capabilities, each stamped with the input `app_id`.

### From Skills Resolution

`AllbertAssist.Skills.Registry.load/1` is called on every `list/1`,
`get/2`, and `read/2`. The merged source list becomes:

1. Built-in root (`priv/skills`) — trusted, precedence 0.
2. Project native (`.allbert/skills`) — trusted/pending, precedence 1.
3. Project interoperable (`.agents/skills`) — trusted/pending, precedence 2.
4. **App-contributed paths** from
   `AllbertAssist.App.Registry.registered_skill_paths/0` — entries shaped as
   `%{app_id: atom(), path: Path.t()}`, trusted, precedence 3.
5. User native (`~/.allbert/skills`) — trusted, precedence 4.
6. User interoperable (`~/.agents/skills`) — trusted, precedence 5.
7. Configured `skills.scan_paths` from Settings Central — precedence 6+.
8. Imported cache (`~/.allbert/cache/skills`) — precedence 100.

App-contributed paths default to `trust: :trusted` posture because the app
was installed by being added to the `:apps` config list. Paths that do not
exist on disk surface as ordinary skills-loader diagnostics (`{:missing,
path}`) and do not block loading from the other roots.

Unregistering an app removes its paths from the next `load/1` call. The
skills registry does not cache app paths across calls.

The implementation adds `:app` as a `source_scope` in
`AllbertAssist.Skills.Skill` and the skills registry precedence table. App
skills include `source_scope: :app`; diagnostics should include `app_id` when
that context is available.

## Registered Action Flow

### `list_apps`

For an intent agent or CLI calling `list_apps`:

1. `AllbertAssist.Actions.Runner.run(:list_apps, %{}, context)` resolves
   the action module through the action registry and runs it.
2. The action calls `AllbertAssist.App.Registry.registered_apps/0` and
   `AllbertAssist.App.Registry.diagnostics/0`.
3. The action builds a redacted summary:

   ```elixir
   %{
     apps: [
       %{
         app_id: :allbert,
         display_name: "Allbert",
         version: "0.15.0",
         action_count: 0,
         skill_path_count: 0,
         surface_count: 0
       },
       %{
         app_id: :stocksage,
         display_name: "StockSage (placeholder)",
         version: "0.15.0",
         action_count: 0,
         skill_path_count: 0,
         surface_count: 0
       }
     ],
     diagnostics: []
   }
   ```

4. The runner emits `allbert.action.requested` and
   `allbert.action.completed` signals with redacted parameters and result
   metadata.
5. Raw `metadata`, child supervisor PIDs, and full skill paths are not
   included in the action result.

### `show_app`

For `show_app` with `params: %{app_id: "stocksage"}`:

1. Action validates `app_id` with
   `AllbertAssist.App.Registry.normalize_app_id/1`:
   - String input resolves only to existing, registered app atoms. Unknown
     binary returns `{:error, :unknown_app}` without raising or creating an
     atom.
   - Atom input is accepted only when the registry knows it.
2. Action calls `AllbertAssist.App.Registry.lookup(:stocksage)`.
3. On success, returns:

   ```elixir
   %{
     app_id: :stocksage,
     display_name: "StockSage (placeholder)",
     version: "0.15.0",
     module: AllbertAssist.App.StockSageStub,
     action_names: [],
     skill_paths: [],
     surfaces: []
   }
   ```

4. `module` is the implementing module atom for operator inspection.
   `action_names` is a sorted list of strings derived from the registered
   action modules. `skill_paths` is the full list of absolute paths. `surfaces`
   is the full surface entry list including `label`, `path`, `description`,
   and `icon` when present.
5. On lookup miss, returns a completed action-runner response with
   `status: :not_found` and `error: :unknown_app`; the CLI renders this as
   `App not found: APP_ID` and exits non-zero.
6. Lifecycle signals are emitted identically to `list_apps`.

## CLI Flow

### `mix allbert.apps list`

```sh
mix allbert.apps list
```

1. CLI ensures the application is started (loading config, supervisors,
   bootstrap).
2. CLI calls `AllbertAssist.Actions.Runner.run(:list_apps, %{}, %{channel: :cli})`.
3. CLI renders one row per app:

   ```text
   allbert      Allbert                 v0.15.0  actions=0 skills=0 surfaces=0
   stocksage    StockSage (placeholder) v0.15.0  actions=0 skills=0 surfaces=0
   ```

4. CLI renders any diagnostics rows below the table with a clear
   "failed registrations" header.
5. Exit code 0 even when diagnostics are present; only argument errors and
   action-runner exceptions produce non-zero exits.

### `mix allbert.apps show APP_ID`

```sh
mix allbert.apps show stocksage
```

1. CLI calls `AllbertAssist.Actions.Runner.run(:show_app,
   %{app_id: "stocksage"}, %{channel: :cli})`.
2. CLI renders:

   ```text
   App: stocksage
   Display name: StockSage (placeholder)
   Version: 0.15.0
   Actions: (none)
   Skill paths: (none)
   Surfaces: (none)
   ```

3. CLI surfaces non-empty fields with one entry per line for actions,
   skill paths, and surfaces.
4. Missing app returns:

   ```text
   App not found: unknown
   ```

   with the normal non-zero `Mix.Error` exit behavior.

### `mix allbert.apps validate MODULE`

```sh
mix allbert.apps validate AllbertAssist.App.CoreApp
```

1. CLI parses `MODULE` as a binary and resolves it against already compiled
   module atoms from loaded OTP application specs and `:code.all_loaded/0`.
   It must not call `String.to_atom/1`, `String.to_existing_atom/1`, or
   `Module.concat/1` on arbitrary operator input.
2. CLI checks the module declares `@behaviour AllbertAssist.App` (via
   `module.module_info(:attributes)` or a documented helper).
3. CLI calls `module.validate([])` and captures the return value.
4. On `:ok`, CLI prints:

   ```text
   AllbertAssist.App.CoreApp: ok
   app_id: allbert
   display_name: Allbert
   version: 0.15.0
   ```

5. On `{:error, diagnostics}`, CLI prints each diagnostic as one line with
   `kind` and `message`, redacting any `detail` values that look like
   filesystem paths or secrets.
6. CLI does NOT touch the registry. Calling `validate` repeatedly produces
   the same output and does not duplicate the app, even if the app is
   already registered.

## Bootstrap And Restart Flow

### Application Cold Start

1. `AllbertAssist.Application.start/2` walks the children list in order.
2. `AllbertAssist.App.Supervisor` starts with `:one_for_all`:
   1. `Registry` starts, creates ETS table, idle.
   2. `DynamicSupervisor` starts with no children.
   3. `Bootstrap` starts, reads `:apps`, registers each module.
3. `Scratchpad` starts after `App.Supervisor`. From this point on,
   `set_active_app` writes can consult the populated registry.

### Registry Process Crash

1. `Registry` crashes (e.g., from a fatal bug in `register/2`).
2. `:one_for_all` restarts `Registry`, `DynamicSupervisor`, and
   `Bootstrap` as one volatile registry group.
3. `Registry`'s new `init/1` creates a fresh ETS table.
4. `DynamicSupervisor` restarts with no children (the previous children
   are terminated by the supervisor restart cascade).
5. `Bootstrap` re-reads `:apps` and re-registers every configured app.
6. After restart, `registered_apps/0` returns the same set as before,
   though `registered_at_ms` timestamps are fresh.

### Application Restart

1. Cold-start behavior repeats. The registry starts empty before
   Bootstrap re-registers. No app state survives across application
   restarts; persistent state lives in SQLite, markdown memory, settings,
   and the encrypted secret store.

### Scratchpad Restart

1. v0.14's restart behavior is unchanged: the scratchpad supervisor
   restarts with an empty ETS table.
2. `App.Supervisor` is a sibling, not a child of `Scratchpad`. A scratchpad
   crash does NOT restart the app registry.
3. When the scratchpad restarts, the registry still has its registered
   apps. `set_active_app` writes against the empty scratchpad continue to
   validate against the registry exactly as before.

### Bootstrap Failure Tolerance

If a configured app's `validate/1` fails or its `child_spec/1` raises:

1. The single registration call returns an error.
2. Bootstrap logs a `warning` and records the diagnostic.
3. Bootstrap continues with the remaining configured apps.
4. The supervision tree does not restart.
5. `mix allbert.apps list` shows the failing row with diagnostic detail.
6. Sessions can still proceed for any registered app; only the failing
   app's id is unknown to the registry.

## Validation Failure Modes

Failure modes observed during `register/2`:

- Module not loaded or not implementing the behaviour: returns
  `{:error, :invalid_module}`. Bootstrap logs and records the diagnostic.
- Bad `app_id` shape: returns `{:error, {:invalid_app_id, module}}`.
- Empty/oversize `display_name` or `version`: returns
  `{:error, {:invalid_metadata, detail}}`.
- `validate/1` returns `{:error, diagnostics}`: returns
  `{:error, {:validation_failed, diagnostics}}`.
- Unknown action module in `actions/0`: returns
  `{:error, {:unknown_action_module, module}}`.
- Non-absolute or non-string skill path: returns
  `{:error, {:invalid_skill_path, path}}`.
- Malformed surface entry: returns
  `{:error, {:invalid_surface, detail}}`.
- Duplicate `app_id`: returns `{:error, {:app_id_taken, app_id}}`.
- `child_spec/1` raises: returns `{:error, {:child_spec_failed, detail}}`.
- `DynamicSupervisor.start_child/2` returns
  `{:error, {:already_started, pid}}`: registry records
  `:child_spec_failed` unless the app is already registered with the same
  module and `unregister/1` has not been called. v0.15 does not silently
  attach to arbitrary existing children.
- `DynamicSupervisor.start_child/2` returns any other error: registry
  rolls back the in-progress entry and returns
  `{:error, {:child_spec_failed, detail}}`.

In every failure case, no ETS entry is created. Existing registrations are
not disturbed.

## Manual Verification Matrix

### Default Bootstrap

Commands:

```sh
mix allbert.apps list
mix allbert.apps show allbert
mix allbert.apps show stocksage
```

Expected:

- `list` shows both `allbert` and `stocksage` with their display names and
  zero-count action/skill/surface columns.
- `show allbert` returns `CoreApp` metadata with empty action/skill/surface
  lists.
- `show stocksage` returns `StockSageStub` metadata with empty
  action/skill/surface lists and a note that it is a placeholder.

### Missing App

Command:

```sh
mix allbert.apps show unknown
```

Expected:

- Returns a redacted not-found error and exits non-zero.
- No ETS state is modified.
- No `allbert.action.requested`/`completed` lifecycle signals for
  successful payload; the action returns an error promptly.

### Validate

Commands:

```sh
mix allbert.apps validate AllbertAssist.App.CoreApp
mix allbert.apps validate AllbertAssist.App.StockSageStub
mix allbert.apps validate Elixir.NotAnApp
```

Expected:

- The first two print `:ok` plus `app_id`, `display_name`, and `version`.
- The third prints a clear "module does not implement
  `AllbertAssist.App`" error and exits non-zero.
- No registry mutation occurs in any case.

### v0.14 Session Continuity

Commands:

```sh
mix allbert.sessions set-active-app --user alice --session sess-1 stocksage
mix allbert.sessions show --user alice --session sess-1
mix allbert.ask --user alice --session sess-1 "hello from stocksage context"
```

Expected:

- `set-active-app` succeeds because `StockSageStub` is registered. Emits
  `allbert.action.requested`/`completed` for `set_active_app`.
- `show` reports `active_app=stocksage`.
- `ask` propagates `active_app: :stocksage` to the intent agent, signals,
  trace metadata, and the response.
- Without StockSageStub registered (an alternate test config),
  `set-active-app stocksage` fails with `:unknown_app` and no entry is
  created.

### Unknown App Rejection

Command:

```sh
mix allbert.sessions set-active-app --user alice --session sess-1 nonexistent
```

Expected:

- Fails with `:unknown_app` without mutating the scratchpad.
- No ETS scratchpad entry for `{alice, sess-1}` is created.
- The `set_active_app` registered action emits its lifecycle signals with
  the error result.

### Decision Validation Rejection

Process:

1. Submit a runtime prompt that causes the intent agent (a test stub) to
   emit a decision with `active_app: :totally_bogus`.
2. Observe `AllbertAssist.Intent.Decision.new/1` behavior.

Expected:

- The decision is rebuilt without `active_app` (or with the context-supplied
  `active_app` when present).
- A structured diagnostic is recorded.
- The runtime turn completes without crashing.

### Action Tagging

Process:

1. Register a test app whose `actions/0` returns `[Test.SafeReadAction]`,
   a real action module compiled into the test umbrella with
   `AllbertAssist.Actions.Registry` recognition.
2. Call `AllbertAssist.Actions.Registry.capability(Test.SafeReadAction)`.

Expected:

- The returned `Capability` struct carries `app_id: :test_app`.
- `AllbertAssist.Actions.Registry.capabilities_for_app(:test_app)`
  returns a list containing the same capability.
- The core action list is unchanged; existing capabilities continue to
  carry `app_id: nil`.

### Skills Tagging

Process:

1. Register a test app whose `skill_paths/0` returns
   `["/tmp/test_app_skills"]` and pre-populate that directory with a
   minimal valid `SKILL.md`.
2. Call `AllbertAssist.Skills.Registry.list/1`.

Expected:

- The test skill appears in the result with `trust: :trusted`.
- The skill's precedence puts it after project roots and before user
  roots, exactly as the merged source list documents.
- Unregistering the test app removes the skill from the next `list/1`
  call.

### Registry Restart

Process:

1. Cold-start the application; observe `mix allbert.apps list`.
2. Kill `AllbertAssist.App.Registry` with `Process.exit/2` `:kill`.
3. Wait for OTP to restart the supervisor children.
4. Call `mix allbert.apps list` again.

## Failure Modes

- `Application.get_env(:allbert_assist, :apps)` returns a non-list: the
  bootstrap raises a clear `RuntimeError` with the configured value's type;
  this is intentional because mis-configured `:apps` is a deployment bug,
  not runtime input.
- A module in `:apps` is not loaded: bootstrap logs a warning and records a
  diagnostic. Other apps continue.
- A module in `:apps` does not implement `AllbertAssist.App`: bootstrap
  records `:invalid_module`; the supervision tree continues.
- A module's `validate/1` raises: bootstrap catches the exception, records
  `:validation_raised` with redacted detail, and continues.
- A module's `child_spec/1` raises: bootstrap catches the exception,
  records `:child_spec_failed`, and continues. The
  `AllbertAssist.App.DynamicSupervisor` is unaffected.
- A module's `child_spec/1` succeeds but the child fails to start:
  bootstrap records `:child_spec_failed`; the ETS entry is not created.
- `AllbertAssist.App.Registry` GenServer crashes: `:one_for_all` restarts
  Registry, DynamicSupervisor, and Bootstrap; the registry is rebuilt from
  empty ETS state.
- `AllbertAssist.App.DynamicSupervisor` crashes: `:one_for_all` restarts
  Registry, DynamicSupervisor, and Bootstrap together, avoiding stale
  child entries in the registry.
- `AllbertAssist.App.Bootstrap` completes successfully: it transitions
  to a permanent idle state. It is not a permanent Task that exits normally
  after registration, because that would create a supervisor restart loop.
- Concurrent `register/2` calls with the same `app_id`: GenServer
  serialization means one succeeds and the other gets
  `{:error, {:app_id_taken, app_id}}`.
- `unregister/1` for an unknown `app_id`: returns `:ok` idempotently.
- An action capability lookup for an action module never registered with
  any app: returns the compile-time capability with `app_id: nil`.
- A `set_active_app` write for an app id whose atom does not exist:
  `AllbertAssist.App.Registry.normalize_app_id/1` returns
  `{:error, :unknown_app}`. No atom is created. No scratchpad mutation.
- `Application.get_env(:allbert_assist, AllbertAssist.App.Registry)` has
  `enabled?: false`: reads return `:not_found`; writes return `:disabled`;
  `mix allbert.apps list` prints an empty result with a disabled notice.

## Out Of Scope

- Hosted accounts, app marketplaces, third-party app trust elevation, or
  remote app installation.
- AG-UI/A2UI streaming endpoints, MCP Apps protocol, or any external
  remote UI execution.
- App-scoped permission grants, Security Central safety floors, resource
  access operation classes, or confirmation requirements.
- App-scoped intent prioritization. v0.19 introduces hybrid intent
  enrichment that may consume `app_id`; v0.15 only carries the tag.
- App-scoped jobs, app-owned schedules, or per-app job templates. v0.13
  jobs continue to store nil/general `app_id`.
- Dynamic HTTP route mounting, dynamic LiveView module loading from app
  folders, or runtime compilation of app code from arbitrary directories.
- General plugin discovery or plugin-contributed apps. v0.17 owns that
  package/discovery layer; v0.15 owns only the app contract.
- New writable Settings Central keys for `apps.*`.
- `register_app`, `enable_app`, `disable_app`, or `unregister_app`
  registered actions.
- `AllbertAssist.App.SurfaceProvider`, `AllbertAssist.Surface` DSL,
  surface validation tooling, surface encoders, canvas storage, or
  ephemeral UI rendering. Those are v0.18/v0.26 (formerly v0.24).
- `mix allbert.gen.plugin` or `mix allbert.gen.app` generator. That is v0.31
  (formerly v0.29).
- Workspace shell sidebar, `/apps` LiveView, or any new HTTP route.
- Durable registry audit table or `allbert.app.*` lifecycle signal types.
- Generic dynamic atom creation from operator or model input.
