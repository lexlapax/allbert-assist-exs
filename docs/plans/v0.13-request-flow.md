claimed, run, observed, paused, and blocked on confirmation.

## Common Shape

Every v0.13 job run follows one of two paths:

1. Runtime prompt job: `AllbertAssist.Jobs.Runner` calls
   `AllbertAssist.Runtime.submit_user_input/1`.
2. Registered action job: `AllbertAssist.Jobs.Runner` calls
   `AllbertAssist.Actions.Runner.run/3`.

Both paths create a durable `scheduled_job_runs` row, emit job lifecycle
signals, preserve v0.12 `user_id`/`thread_id` context, and stop at the existing
durable confirmation workflow when risky work needs approval.

Jobs never call shell commands, package managers, `Req`, local file readers,
skill importers, memory writers, or generated modules directly.

## Job Creation

For a CLI command such as:

```sh
mix allbert.jobs create runtime-prompt daily-brief \
  --user alice \
  --prompt "give me a safe daily brief" \
  --daily 08:00 \
  --active
```

The flow is:

1. CLI parses args and validates local identity.
2. Omitted `--user` defaults to `"local"`.
3. If `--operator` is added later and differs from `--user`, fail before
   writing.
4. CLI normalizes the schedule into a persisted schedule map.
5. CLI reads `jobs.timezone` unless `--timezone` is supplied.
6. CLI computes `next_due_at` for non-manual active jobs.
7. CLI stores a `scheduled_jobs` row through `AllbertAssist.Jobs`.
8. CLI prints job id, status, schedule, timezone, next due, and user/thread
   context.

Manual jobs use:

```elixir
%{"kind" => "manual"}
```

and store `next_due_at: nil`.

Daily jobs use:

```elixir
%{"kind" => "daily", "at" => "08:00"}
```

Weekly jobs use:

```elixir
%{"kind" => "weekly", "weekday" => "monday", "at" => "08:00"}
```

Cron-like jobs use:

```elixir
%{"kind" => "cron", "expression" => "0 8 * * 1-5"}
```

Invalid schedule maps, unsupported cron syntax, invalid timezones, and unknown
target types fail closed before inserting the job.

## Thread Context At Creation

For runtime prompt jobs:

- `--thread THREAD_ID` stores `thread_id` and `thread_mode:
  "origin_thread"` after verifying the thread belongs to `user_id`.
- `--recent-thread` stores no `thread_id` and `thread_mode:
  "recent_general"`.
- `--new-thread-per-run` stores no `thread_id` and `thread_mode:
  "new_thread_per_run"`.
- Omitted thread flags default to `"recent_general"` for CLI-created jobs.
- Jobs created from an interactive runtime turn should preserve the originating
  `thread_id` and use `"origin_thread"` by default.

Registered action jobs may store `thread_id` as origin metadata, but they must
not write conversation messages directly.

## Manual Run Flow

For:

```sh
mix allbert.jobs run JOB_ID
```

The flow is:

1. CLI loads `JOB_ID` through `AllbertAssist.Jobs`.
2. Archived jobs are rejected.
3. A manual run row is created with `trigger: "manual"` and `status:
   "queued"`.
4. `allbert.job.started` is emitted after the run moves to `"running"`.
5. `AllbertAssist.Jobs.Runner` executes the job target through the runtime or
   action runner.
6. The run row is updated with status, timing, signal ids, trace id,
   confirmation id, and redacted result metadata.
7. `allbert.job.completed`, `allbert.job.needs_confirmation`, or
   `allbert.job.failed` is emitted.
8. CLI prints the run result and any Approval Handoff details.

Manual runs are allowed for paused jobs. Manual runs are rejected for archived
jobs and for jobs already blocked by a pending confirmation unless the command
explicitly supports a later force/retry mode.

## Scheduler Tick Flow

The supervised scheduler repeats a conservative tick:

1. Read `jobs.schedule_policy`.
2. If policy is `"paused"`, stop the tick without creating runs.
3. Select active jobs where `next_due_at <= now` and
   `blocked_confirmation_id` is nil.
4. Skip jobs with a currently running run.
5. Claim each due job in a database transaction.
6. Create a queued run with `trigger: "scheduler"` and the due timestamp.
7. Emit `allbert.job.due`.
8. Execute the run through `AllbertAssist.Jobs.Runner`.
9. Compute the next due time from the job schedule and timezone.
10. Update `last_run_at`, `next_due_at`, and run status.

The scheduler is local-only. It does not need distributed locks, cluster leader
election, an OS cron entry, Oban, or Quantum.

## Runtime Prompt Job Flow

For a runtime prompt job, the runner builds a request map:

```elixir
%{
  text: job.target["text"],
  channel: :job,
  user_id: job.user_id,
  operator_id: job.operator_id,
  thread_id: selected_thread_id,
  new_thread: new_thread?,
  session_id: job.session_id,
  metadata: %{
    job_id: job.id,
    run_id: run.id,
    trigger: run.trigger,
    app_id: job.app_id
  }
}
```

Thread selection:

- `"origin_thread"` passes the stored `thread_id` after user-scoped
  validation.
- `"recent_general"` omits `thread_id` and lets v0.12 select/create the
  user's recent general thread.
- `"new_thread_per_run"` passes `new_thread: true`.

The runtime then follows the v0.12 request flow:

1. Normalize identity/thread context.
2. Emit `allbert.input.received`.
3. Persist the user message.
4. Pass bounded recent context to the intent agent.
5. Run selected registered actions.
6. Emit `allbert.agent.responded`.
7. Persist assistant response and trace metadata.
8. Return a response map.

The job runner copies redacted response metadata into the run record. It does
not write conversation messages itself.

## Registered Action Job Flow

For a registered action job, the runner builds action params and context:

```elixir
params = job.target["params"]

context = %{
  channel: :job,
  job_id: job.id,
  run_id: run.id,
  user_id: job.user_id,
  operator_id: job.operator_id,
  thread_id: job.thread_id,
  session_id: job.session_id,
  app_id: job.app_id,
  trigger: run.trigger
}
```

The flow is:

1. Resolve `job.target["action_name"]` through
   `AllbertAssist.Actions.Registry`.
2. Reject unknown or unsupported actions before running.
3. Call `AllbertAssist.Actions.Runner.run/3`.
4. Store redacted result metadata in the run row.
5. If the action requires confirmation, follow the confirmation flow below.

The runner context is origin metadata, not permission. The target action still
owns schema validation, Security Central checks, confirmation creation,
redaction, and effects.

## Confirmation-Required Flow

For a risky scheduled prompt such as:

```sh
mix allbert.jobs create runtime-prompt url-summary \
  --user alice \
  --prompt "check https://example.com/report and summarize it" \
  --manual
mix allbert.jobs run JOB_ID
```

Expected flow:

1. The runtime or registered action reaches the same v0.11
   `:needs_confirmation` result as CLI/LiveView.
2. The target action creates a durable confirmation and performs no target
   side effect.
3. The job run status becomes `"needs_confirmation"`.
4. The run stores the `confirmation_id`, resource posture, Approval Handoff,
   decision, diagnostics, signal ids, and redacted action summary.
5. The job stores `blocked_confirmation_id` and moves to `"blocked"` for
   automatic scheduling.
6. `allbert.job.needs_confirmation` is emitted.
7. CLI and LiveView render the same Approval Handoff data as interactive
   requests.
8. Approval or denial uses `mix allbert.confirmations approve|deny` or the
   existing LiveView confirmation controls.

Confirmation origin metadata must include:

- `channel: "job"`
- `job_id`
- `run_id`
- `user_id`
- `operator_id`
- `thread_id` when present
- `app_id` when present
- source signal id when available

Jobs must not approve, deny, remember grants, fetch, summarize, import,
install, execute, or resume target work by mutating confirmation records.

## Pause, Resume, And Blocked Jobs

Pause:

```sh
mix allbert.jobs pause JOB_ID
```

- Sets job status to `"paused"`.
- Clears automatic due selection.
- Does not cancel an already running run.
- Does not resolve a pending confirmation.

Resume:

```sh
mix allbert.jobs resume JOB_ID
```

- For a paused job, sets status to `"active"` and recomputes `next_due_at`.
- For a blocked job, first checks the referenced confirmation status.
- If the confirmation is still pending, resume fails and points to
  `mix allbert.confirmations show CONFIRMATION_ID`.
- If the confirmation is resolved, clears `blocked_confirmation_id`, sets
  status to `"active"`, and recomputes `next_due_at`.

Archive is optional in v0.13 but, if implemented, must prevent automatic and
manual runs while preserving historical run records.

## CLI Inspection

List output should include stable, scannable fields:

```text
job_abc daily-brief status=paused schedule=manual user=alice thread=recent next=none last=none
```

Show output should include:

- id and name
- status
- schedule and timezone
- target type and redacted target summary
- user/operator/thread/session/app context
- next due and last run
- blocked confirmation id

Runs output should include:

```text
run_xyz status=completed trigger=manual started=... duration_ms=1234 signal=... trace=none confirmation=none
```

Pending confirmation runs should print enough handoff detail for the operator
to continue with:

```sh
mix allbert.confirmations show CONFIRMATION_ID
mix allbert.confirmations approve CONFIRMATION_ID "reason"
mix allbert.confirmations deny CONFIRMATION_ID "reason"
```

## LiveView Inspection

The web surface should use the same context/actions as CLI:

1. Load jobs/runs through `AllbertAssist.Jobs`.
2. Render status, schedule, next due, last run, user/thread context, and
   blocked confirmation id.
3. Pause/resume/manual-run events call the job context or registered actions.
4. Pending confirmation details reuse existing Approval Handoff rendering and
   confirmation actions.

LiveView must not parse schedules, execute jobs, mutate confirmation files, or
own security policy.

## Acceptance Smoke

Use a disposable `ALLBERT_HOME` and test database.

```sh
export ALLBERT_HOME="$(mktemp -d /tmp/allbert-v013-manual.XXXXXX)"
export DATABASE_PATH="$ALLBERT_HOME/db/allbert.sqlite3"
mkdir -p "$ALLBERT_HOME/db"
mix ecto.create
mix ecto.migrate
```

Manual job:

```sh
mix allbert.jobs create runtime-prompt daily-brief \
  --user alice \
  --prompt "hello from scheduled jobs" \
  --manual
mix allbert.jobs list --user alice
mix allbert.jobs run JOB_ID
mix allbert.jobs runs JOB_ID
mix allbert.threads --user alice
```

Expected:

- Job is created with `user=alice`.
- Manual run emits job lifecycle signals and runtime signals.
- Run status becomes completed.
- Conversation history is written by the runtime, not by job storage.

Automatic due job:

```sh
mix allbert.jobs create runtime-prompt alice-daily \
  --user alice \
  --prompt "safe daily brief" \
  --daily 08:00 \
  --active
mix allbert.jobs show JOB_ID
```

Expected:

- Job stores a normalized daily schedule and timezone.
- `next_due_at` is UTC and derived from the local `08:00` schedule.
- When the scheduler sees the job as due in tests or manual tick helpers, it
  creates exactly one run and advances `next_due_at`.

User/thread isolation:

```sh
mix allbert.ask --user alice --new-thread "job thread seed"
mix allbert.jobs create runtime-prompt alice-thread \
  --user alice \
  --thread THREAD_ID \
  --prompt "continue this thread" \
  --manual
mix allbert.jobs run JOB_ID
mix allbert.jobs create runtime-prompt bad-thread \
  --user bob \
  --thread ALICE_THREAD_ID \
  --prompt "try to use Alice thread" \
  --manual
```

Expected:

- Alice's job can use Alice's thread.
- Bob's create or run fails without revealing Alice as owner.

Confirmation-required job:

```sh
mix allbert.jobs create runtime-prompt url-summary \
  --user alice \
  --prompt "check https://example.com/report and summarize it" \
  --manual
mix allbert.jobs run JOB_ID
mix allbert.jobs show JOB_ID
mix allbert.jobs runs JOB_ID
mix allbert.confirmations show CONFIRMATION_ID
```

Expected:

- Run status is `needs_confirmation`.
- Job is blocked on the confirmation id.
- Approval Handoff and resource posture are visible in job run output.
- The confirmation origin includes job/run/user/thread context.
- No fetch, summary, import, install, shell command, or script execution occurs
  before approval.

Scheduler pause policy:

```sh
mix allbert.settings set jobs.schedule_policy paused
mix allbert.jobs resume JOB_ID
```

Expected:

- Manual runs still work.
- Automatic scheduler ticks do not create due runs while policy is paused.

## Failure Modes

- Invalid schedule: fail before insert/update.
- Invalid timezone: fail before insert/update.
- Unknown job id: return not found without stacktrace.
- Unknown action name: fail before creating a running run.
- Archived job: reject manual and automatic runs.
- Existing running run: skip duplicate due run.
- Pending confirmation: block automatic reruns until resolved.
- Runtime/action exception: mark run failed with redacted error and emit
  `allbert.job.failed`.
- Trace writer failure: keep run status/result and record redacted trace error
  without crashing the scheduler.

## Out Of Scope

- Hosted accounts, auth, roles, or job ownership foreign keys.
- Distributed scheduling or remote workers.
- New execution adapters.
- Channel-specific job creation.
- App-specific jobs or `active_app` routing.
- Job-authored conversation history writes outside runtime.
- Automatic markdown memory promotion.
- Job-specific approval queues or grant stores.
