# Allbert v0.14 Session Scratchpad And Active App Context

Formerly tracked as M-D1b.

## Purpose

v0.14 adds volatile local session state keyed by `{user_id, session_id}`. The
scratchpad lets Allbert carry short-lived workspace context, especially
`active_app`, across turns in the same local session without treating that
state as durable memory, authorization, or app registration.

This milestone is deliberately small. It makes `session_id` useful, gives the
runtime a safe place to read `active_app`, and prepares v0.15 app registration
and v0.20 cross-surface intent enrichment. It does not add the app registry,
app action routing, canvas state, hosted sessions, or a new UI surface.

The acceptance surface is runtime, CLI, signals, traces, and tests. Phoenix
LiveView may pass a `session_id` later, but v0.14 should not add an AgentLive
session sidebar, workspace shell, or canvas controls.

## Expected Inputs

- v0.11 is released and tagged. `AllbertAssist.Intent.Decision` already
  reserves `user_id`, `thread_id`, `session_id`, and `active_app`.
- v0.12 local workspace identity and conversation history are implemented.
  Runtime requests and responses carry canonical string `user_id`,
  compatibility `operator_id`, `thread_id`, and optional `session_id`.
- v0.13 scheduled jobs are implemented. Jobs may carry `session_id`; when they
  enter the runtime after v0.14, the runtime may read scratchpad context for
  that session but jobs must not mutate scratchpad implicitly.
- ADR 0014 local workspace identity is accepted.
- `AllbertAssist.Runtime.submit_user_input/1` remains the runtime entrypoint,
  and channels/jobs continue to call it instead of calling agents directly.

## Roadmap And Vision Fit

The Allbert/Jido vision treats session scratchpad state as local, volatile
context around the signal-driven runtime. It is useful because the workspace is
becoming multi-app, but it is intentionally weaker than durable storage:

- SQLite `Thread` and `Message` rows remain durable conversation history.
- Markdown memory remains the durable, operator-edited source of truth.
- ETS scratchpad entries are fast, local, TTL-expiring session context.

This milestone prepares later work without implementing it:

- v0.15 app registration can use `active_app` as context while registering app
  actions and skill paths through explicit contracts.
- v0.16 channels can map external session/thread identities into local
  `session_id` values without inventing per-channel scratchpads.
- v0.20 intent enrichment can use `active_app` to prioritize app-registered
  actions only after the app registry exists.
- v0.22 StockSage LiveViews can set `active_app: :stocksage` when the user is
  inside StockSage context.
- v0.23 security evals can test cross-session leakage and app-scoped routing.
- v0.26/v0.27 canvas work can add durable or task-scoped surface state later;
  v0.14 reserves no canvas storage shape.

v0.14 must keep Allbert local-first and inspectable. The scratchpad should be a
supervised process with clear APIs, predictable expiry, redacted observability,
and boring failure behavior.

## Handoffs

### v0.12 Handoff

v0.14 keys scratchpad entries from the canonical v0.12 `user_id` and optional
`session_id`. It must not move SQLite conversation history into ETS, change
thread selection semantics, or use scratchpad presence as authorization.

When no `session_id` is present, the runtime behaves as v0.12/v0.13 do today:
no scratchpad lookup, `active_app: nil`, and no scratchpad entry creation.

### v0.13 Handoff

Scheduled jobs may pass a stored `session_id` to the runtime. If a matching
scratchpad entry exists and has not expired, the job's runtime prompt receives
that session context. Jobs must not create, update, or clear scratchpad entries
as a side effect of running. Registered action jobs keep using action context
and do not read scratchpad unless a later plan explicitly adds that behavior.

### v0.15 And v0.20 Handoff

v0.14 carries `active_app`; it does not route on it. The runtime and intent
agent receive `active_app` so later app-aware ranking can consume the field.
Until v0.15 registers apps and v0.20 enriches intent, `active_app` is context
and trace metadata only.

## Implementation Boundaries

Authoritative existing boundaries:

- Runtime entrypoint: `AllbertAssist.Runtime.submit_user_input/1`.
- Runtime agent path: `AllbertAssist.Agents.IntentAgent.respond/1`.
- Conversation history: `AllbertAssist.Conversations`.
- Intent decision contract: `AllbertAssist.Intent.Decision`.
- Tracing: `AllbertAssist.Trace` and the `record_trace` action.
- Settings: `AllbertAssist.Settings`.
- Supervision: `AllbertAssist.Application`.

Preferred new implementation shape:

- `AllbertAssist.Session`: public context/facade used by runtime, registered
  actions, CLI, tests, jobs, and future channels. It owns normalization,
  error shapes, and redacted summaries; external callers should prefer this
  module over calling the GenServer directly.
- `AllbertAssist.Session.Scratchpad`: supervised GenServer that owns an ETS
  table and exposes the public scratchpad API.
- `AllbertAssist.Session.Entry`: optional struct or documented map shape for
  normalized entry data.
- `AllbertAssist.Session.AppId`: small normalization helper for `active_app`
  values, if keeping that logic separate makes the boundary clearer.
- `Mix.Tasks.Allbert.Sessions`: CLI inspection and explicit active-app control
  for manual verification.

The ETS table should be owned by the supervised scratchpad process. Runtime,
CLI, tests, jobs, and future channel adapters should use the scratchpad API
rather than reading or writing ETS directly.

The scratchpad process belongs in the application supervision tree before
anything that can submit runtime work on startup. It may be disabled or named
through application config for tests, but operator-tunable values belong in
Settings Central.

## Supervision And Lifecycle

`AllbertAssist.Session.Scratchpad` runs as a supervised `GenServer` started
from `AllbertAssist.Application`. Boot order:

1. `AllbertAssist.Repo` and migrations.
2. `Phoenix.PubSub` and `Jido.Signal.Bus`.
3. `AllbertAssist.Jido`.
4. `AllbertAssist.Session.Scratchpad` (new).
5. `AllbertAssist.Jobs.Scheduler` (existing, when enabled).

The scratchpad must start before the scheduler so a scheduler-originated run
that carries a `session_id` can read scratchpad state without racing the
supervised process. It must start after Repo/PubSub/Jido so it can use
existing supervision primitives if needed.

Application config under
`Application.get_env(:allbert_assist, AllbertAssist.Session.Scratchpad)`:

- `enabled?`: defaults to `true`. When `false`, the supervised process either
  is not started (test environments that want explicit start_supervised) or
  starts in a disabled mode that returns `not_found` from reads and
  `disabled` errors from writes.
- `table_name`: defaults to `:allbert_session_scratchpad`. Tests may pass an
  alternate name to start an isolated scratchpad instance.
- `sweep_interval_ms`: defaults to `60_000`. Tests may pass a smaller value
  or `0` to disable periodic sweeping.

Restart strategy: `:one_for_one` under the main supervisor with default
backoff. On restart, the scratchpad is empty by design. This restart
contract is the source of v0.14's volatility guarantee; it must not be
weakened by adding ETS-table heir patterns, named table preservation
tricks, or disk persistence.

GenServer responsibilities:

- `init/1`: create the named ETS table, read TTL and sweep interval from
  Settings, schedule the first periodic sweep, and log a startup line.
- `handle_call/3` serializes writes: `set_active_app`, `clear_active_app`,
  `merge_working_memory`, `clear`, `touch`, `sweep_expired`.
- `handle_info/2` handles periodic `:sweep` self-messages.
- `terminate/2` does nothing special; ETS dies with the process.

Reads (`get`, `list`) may go through the GenServer for simplicity or read
the protected table directly through the context module. Either choice is
acceptable as long as every external caller goes through the
`AllbertAssist.Session` context module and never touches ETS directly.

## Settings

Add one writable Settings Central key:

- `sessions.scratchpad_ttl_minutes`: integer, default `30`, minimum `1`,
  maximum `1440`.

Runtime/process config may also expose test-only or deployment-only options:

- `enabled?`: default `true`.
- `table_name`: default `:allbert_session_scratchpad`.
- `sweep_interval_ms`: default `60_000`, test-configurable.

The scratchpad reads this setting at process boot and caches the validated
value. New writes and runtime touches use the cached TTL until the scratchpad
process restarts. Existing entries keep their stored `expires_at_ms` until
touched, updated, or swept.

## Scratchpad State Contract

The ETS table is the only data store v0.14 introduces.

ETS table:

- Name: `:allbert_session_scratchpad` (atom, named table; overridable via
  application config for tests).
- Type: `:set`.
- Access: `:protected`. The supervised `AllbertAssist.Session.Scratchpad`
  process owns the table. Writes are serialized through the GenServer.
- Concurrency: `read_concurrency: true`, `write_concurrency: false`.
- Lifetime: created in the GenServer's `init/1`. When the owning process
  terminates for any reason, the table is destroyed. There is no disk
  persistence and no migration path.

Entries are keyed by `{user_id, session_id}` after normalization.

Stored entry shape:

```elixir
%{
  user_id: "alice",
  session_id: "sess-1",
  active_app: :stocksage,
  working_memory: %{},
  inserted_at_ms: 12_345,
  updated_at_ms: 12_345,
  expires_at_ms: 1_812_345,
  metadata: %{}
}
```

Rules:

- `user_id` is a non-empty string and defaults upstream to `"local"`.
- `session_id` is required for scratchpad operations, non-empty after trim,
  and should be capped at 128 characters.
- `active_app` is `atom | nil`.
- `working_memory` is a bounded map for transient, non-secret context.
- `metadata` is a bounded map for non-secret diagnostics such as channel or
  surface.
- Expiry uses monotonic milliseconds, not wall-clock datetimes.
- CLI/log/trace output may show `active_app`, key names, counts, and expiry
  metadata, but must not dump arbitrary `working_memory` values by default.

`working_memory` is intentionally small:

- Accept maps only.
- Reject the reserved key `:canvas_tiles` with `{:error, :reserved_key}`.
  Canvas tile storage is owned by v0.26; v0.14 must not define, read, write,
  or render it.
- Reject obvious secrets and rely on existing redaction for rendering.
- Reject oversize payloads before storing. A reasonable initial cap is 64 KiB
  measured with `:erlang.external_size/1`.
- Merge semantics are shallow by default: new keys replace existing keys; a
  later milestone may add nested patch semantics if needed.

## Active App Semantics

`active_app` means "the session currently has evidence that the operator is in
an app context." It does not mean "the app exists," "the user is authorized,"
or "app actions should run."

Normalization rules:

- nil, `""`, `"none"`, and `"general"` normalize to nil.
- Atoms are accepted by internal APIs.
- Binary input from CLI flags, registered action params, and channel
  adapters must not create arbitrary atoms. v0.14 maintains a small reserved
  allowlist in `AllbertAssist.Session.AppId`: nil/general (no app),
  `:allbert` (general workspace), and `:stocksage` (the first proving app
  whose atom is compile-time defined here so v0.17+ can register it). Binary
  input is matched against the allowlist first; unknown strings are rejected
  with `:unknown_app`. `String.to_existing_atom/1` is the only conversion
  primitive used.
- v0.15 replaces the static allowlist with `AllbertAssist.App.Registry`
  validation. v0.14 ships the allowlist so the mechanism is exercised end to
  end without depending on v0.15.

Decision validation handoff:

- `AllbertAssist.Intent.Decision` already accepts `:active_app` from the
  request context. v0.14 must ensure context-supplied `active_app` is
  preserved when the decision does not explicitly set one, and that
  model/agent output cannot overwrite a scratchpad-derived `active_app`
  with an unknown atom. Decision validation rejects unknown atoms; v0.14
  must not loosen that rule.

v0.14 does not prioritize StockSage actions, discover app skill paths, or
enforce app-scoped action routing. Those are v0.15/v0.20/v0.23 responsibilities.

## Runtime Request Contract

Extend runtime request normalization with scratchpad lookup:

1. Normalize `user_id`, `operator_id`, `thread_id`, and `session_id` exactly as
   v0.12 defines.
2. If `session_id` is nil, set `active_app: nil` and skip scratchpad lookup.
3. If `session_id` is present, fetch `{user_id, session_id}` from
   `AllbertAssist.Session.Scratchpad`.
4. If no entry exists or the entry expired, use `active_app: nil`.
5. If an entry exists, read `active_app` and pass it into runtime request,
   input signal, intent-agent request, response signal, response map, trace
   context, and assistant message metadata.
6. A runtime read may touch the entry and extend TTL; it must not create a new
   entry when none exists.

The runtime agent request should include:

```elixir
%{
  text: request.text,
  channel: request.channel,
  user_id: request.user_id,
  operator_id: request.operator_id,
  thread_id: request.thread_id,
  session_id: request.session_id,
  active_app: request.active_app,
  thread_context: request.thread_context,
  metadata: request.metadata,
  timeout_ms: request.timeout_ms,
  input_signal_id: signal.id
}
```

Runtime responses should include `active_app` even when nil so channel adapters
have one stable field to render or ignore.

## CLI Contract

Update `mix allbert.ask`:

- Add `--session SESSION_ID`.
- Preserve existing `--user`, `--operator`, `--thread`, and `--new-thread`
  behavior.
- Render `Session: SESSION_ID` when present.
- Render `Active app: stocksage` or `Active app: none`.
- Do not add `--active-app` to `ask` in v0.14; setting session state should be
  explicit through `mix allbert.sessions`.

Add `mix allbert.sessions`:

- `mix allbert.sessions list [--user USER]`
- `mix allbert.sessions show --user USER --session SESSION_ID`
- `mix allbert.sessions set-active-app --user USER --session SESSION_ID APP`
- `mix allbert.sessions clear-active-app --user USER --session SESSION_ID`
- `mix allbert.sessions clear --user USER --session SESSION_ID`
- `mix allbert.sessions sweep`

CLI rules:

- Omitted `--user` defaults to `"local"`.
- `--operator` is accepted as a compatibility alias and must match `--user`
  when both are present.
- `--session SESSION_ID` must be non-empty after trimming and at most 128
  characters; otherwise the command fails before mutating scratchpad state.
- `--session` and `--thread`/`--new-thread` are orthogonal. v0.14 does not
  imply a session id from a thread id, and it does not imply a thread id
  from a session id.
- `set-active-app` creates or updates the scratchpad entry.
- `clear-active-app` keeps the entry but sets `active_app: nil`.
- `clear` deletes the entry.
- `show` prints `user_id`, `session_id`, `active_app`, remaining TTL, metadata
  keys, and working-memory key count. It does not print raw working-memory
  values by default.

## Registered Actions

v0.14 also exposes scratchpad mutations through registered Jido actions so
that v0.15 app activation, v0.16 channels, registered confirmation flows,
and future jobs share a single contract with the CLI. Actions run through
`AllbertAssist.Actions.Runner.run/3` and emit the standard
`allbert.action.requested`/`allbert.action.completed` lifecycle signals.

Registered actions:

- `set_active_app`: params `user_id`, `session_id`, `app_id` (string or nil).
  Permission `:settings_write`, execution mode `:settings_write`,
  confirmation `:not_required`. This reuses the existing safe local
  operator-configuration permission class rather than adding a new session
  permission class. Validates `app_id` against the v0.14 `AppId` allowlist via
  `String.to_existing_atom/1`; unknown values return a structured
  `:unknown_app` error without writing.
- `clear_active_app`: params `user_id`, `session_id`. Permission
  `:settings_write`, execution mode `:settings_write`, confirmation
  `:not_required`. Keeps the entry; sets `active_app: nil`. Missing entries
  return `:not_found` without creating empty entries.
- `show_session_scratchpad`: params `user_id`, `session_id`. Permission
  `:read_only`, execution mode `:settings_read`, confirmation
  `:not_required`. Returns `active_app`, `working_memory_keys` (sorted list
  of keys only), remaining TTL, and metadata keys. Never returns raw
  `working_memory` values. Missing or expired entries return `:not_found`.

v0.14 does not register `merge_working_memory` or `delete_working_memory`
actions. Working memory writes happen only through the internal context
module until a later milestone defines what working-memory data is safe to
move across the action boundary.

The registered action capability metadata must follow the v0.06 contract
(permission, execution mode, confirmation requirement, resumability,
redaction posture). The mutating actions must call `PermissionGate.authorize/2`
with `:settings_write`; the show action must call it with `:read_only`. None
of the actions declare themselves as resource-access consumers; they do not
touch `AllbertAssist.Resources.*`, do not create durable confirmations, and do
not require new Security Central permission classes or safety floors.

CLI subcommands delegate to the same registered actions so the operator
surface, future channels, registered-action jobs, and tests all reach the
same code path. The CLI is the thin surface; the registered actions are
the durable contract.

## Signals, Traces, And Message Metadata

Add `active_app` to:

- `allbert.input.received` signal data.
- `allbert.agent.responded` signal data.
- runtime response maps.
- intent-agent request maps.
- `AllbertAssist.Intent.Decision` context passed through the runtime.
- trace request/context summaries.
- assistant message metadata.

Do not include raw scratchpad `working_memory` values in signals, traces,
logs, response payloads, conversation message metadata, or registered
action results. Sorted lists of `working_memory` keys and counts are
acceptable inspection metadata; values are not. If a future action needs to
expose part of working memory, it must do so through an explicit redacted
summary defined in that milestone.

Trace metadata gains an `active_app` line only when non-nil so existing
no-session traces stay unchanged. Trace metadata does not include
`expires_at_ms` or other monotonic-time internals.

Signals:

- `allbert.input.received` and `allbert.agent.responded` gain optional
  `active_app` fields.
- v0.14 introduces no new signal types. The supervised scratchpad does not
  emit `allbert.session.*` lifecycle signals. Reads, writes, sweeps, and
  expiry are observable only through registered action lifecycle signals,
  the runtime request path, traces, and CLI output.

Scratchpad operations may emit log lines or diagnostics, but v0.14 does not
need a durable scratchpad audit table. Security-sensitive or app-routing
evals arrive in v0.23.

## Expiry, Sweep, And Restart Semantics

- `get` returns nil/not found for expired entries and may delete them.
- `touch` extends `expires_at_ms` from the current monotonic time using the
  current TTL setting.
- The supervised scratchpad process periodically sweeps expired entries.
- `sweep` is exposed for tests and CLI manual verification.
- If the scratchpad process crashes, OTP restarts it with an empty ETS table.
- Application restart starts with an empty scratchpad.
- Expiry or restart must not delete SQLite threads, messages, jobs, memory, or
  settings.

## Acceptance

- `mix allbert.sessions set-active-app --user alice --session sess-1 stocksage`
  creates a volatile entry with `active_app: :stocksage` and emits
  `allbert.action.requested`/`allbert.action.completed` for `set_active_app`.
- `mix allbert.sessions set-active-app --user alice --session sess-1 unknown`
  is rejected with `:unknown_app` and creates no entry.
- `mix allbert.sessions set-active-app --user alice --session "" stocksage`
  fails before mutating ETS.
- `mix allbert.ask --user alice --session sess-1 "hello"` passes
  `active_app: :stocksage` to the intent agent and returns/renders that
  context.
- `mix allbert.ask --user alice "hello"` keeps `active_app: nil`.
- An agent/model that emits a decision with an unknown `active_app` atom is
  rejected by Decision validation; the context-supplied `active_app` from
  the scratchpad is preserved.
- Alice and Bob using the same `session_id` do not see each other's
  scratchpad entries.
- `{alice, sess-1}` and `{alice, sess-2}` are isolated.
- `clear-active-app` sets active app back to nil without deleting the session
  entry.
- `show_session_scratchpad` returns sorted `working_memory_keys` only; raw
  values never appear in CLI, action results, signals, traces, logs,
  conversation `action_log`, or response payloads.
- `merge_working_memory` writes that include `:canvas_tiles` are rejected.
- `merge_working_memory` writes that exceed the 64 KiB soft cap are rejected
  before storage.
- A scheduled job whose stored `session_id` matches a live scratchpad entry
  receives `active_app` in its run's decision/trace metadata. Job storage
  and scratchpad storage do not write to each other.
- Expired entries are ignored by reads and removed by sweep.
- Restarting the scratchpad supervisor with `start_supervised!/2` starts
  with no session entries; application restart starts with no session
  entries.
- Settings validation rejects `sessions.scratchpad_ttl_minutes` outside
  `[1, 1440]`.
- Existing v0.11-v0.13 URL/document/import/package/shell/script/job
  confirmation behavior is unchanged when no session is present, and is
  unchanged when a session is present except for the addition of
  `active_app` in observability output.
- No markdown memory, SQLite conversation rows, job rows, or settings rows
  are created from ordinary scratchpad reads.

## Non-Goals

- No durable session persistence.
- No browser cookie/session-token/auth model.
- No hosted accounts, roles, or permission grants.
- No app registry, app action routing, app skill ranking, or StockSage domain
  behavior.
- No AgentLive session sidebar, workspace shell, canvas state, or ephemeral UI.
- No automatic session creation from arbitrary `session_id` reads.
- No automatic markdown memory or conversation-history promotion from
  scratchpad contents.
- No generic dynamic atom creation from user input.
- No durable scratchpad audit table.

  `AllbertAssist.Actions.Runner.run/3`. Coverage includes happy path,

  `:read_only` permission decisions for show, and that the v0.06 capability

## Definition Of Done

v0.14 is done when:

- `AllbertAssist.Session.Scratchpad` is supervised, TTL-expiring, sweep-
  driven, and restart-empty.
- `AllbertAssist.Session` exposes the public API used by runtime, registered
  actions, CLI, and tests; no caller touches ETS directly.
- `set_active_app`, `clear_active_app`, and `show_session_scratchpad` are
  registered actions with v0.06 capability metadata and pass focused tests
  through the action runner. Mutating session actions reuse the existing
  `:settings_write` permission; show uses `:read_only`. CLI subcommands
  delegate to the same actions.
- Runtime requests with `session_id` read scratchpad once per turn and
  propagate `active_app` to signal, decision, response, trace, and
  conversation `action_log` layers. Sessionless requests behave exactly as
  v0.13.
- `sessions.scratchpad_ttl_minutes` is a validated Settings Central key
  with a default; deployment/test config keys (`enabled?`, `table_name`,
  `sweep_interval_ms`) exist for boot-level overrides only.
- `working_memory` rejects `:canvas_tiles` and oversize payloads;
  observability output never includes `working_memory` values.
- v0.11 Approval Handoff, v0.12 conversation history, and v0.13 scheduled
  jobs behave identically for sessionless requests; `active_app` propagates
  cleanly for session-bearing requests.
- README, CHANGELOG, roadmap, `docs/plans/v0.14-plan.md`,
  `docs/plans/v0.14-request-flow.md`, and version metadata reflect the
  v0.14 release and the v0.15/v0.20/v0.26 handoffs.
- Focused milestone tests and the full warning gate
  (`mix compile --warnings-as-errors`, `mix format --check-formatted`,
  `mix credo --strict`, `mix dialyzer`, `mix precommit`) pass.
