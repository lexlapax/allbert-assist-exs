# Allbert v0.09 Request Flow

v0.09 turns trusted Agent Skill scripts from inert resource inventory into a
confirmed, resource-gated execution capability. The flow keeps Allbert's
architecture stable: channels translate user input, agents select actions,
Security Central evaluates risk, confirmations pause risky work, and only a
registered action may run the script.

## Actors And Components

- User/operator: the local operator for v0.09.
- Origin channel: CLI, LiveView, or future channel that requested script work.
- Resolver channel: CLI, LiveView, or future channel that approved or denied.
- Intent agent: selects a known skill/action path; it does not run scripts.
- Skill registry: discovers trusted/enabled skills and resource inventory.
- `run_skill_script`: registered action and only script execution entrypoint.
- Security Central: evaluates `:skill_script_execute`.
- Confirmation workflow: stores pending requests and resumes eligible targets.
- Script runner: host-process runner using explicit executable plus argv,
  timeout, env filtering, output caps, redaction, trace, and audit.

## Activation Flow Remains Inert

1. User asks to read or activate a skill.
2. Runtime routes to `read_skill` or `activate_skill`.
3. The action loads instructions and resource metadata only.
4. Script resources are listed with path, kind, byte size, and digest.
5. No confirmation is created and no process executes.

This remains true even when the skill declares `run_skill_script`.

## Script Request Flow

1. User asks Allbert to run a script from a selected trusted skill, or an
   operator CLI builds a script request directly.
2. Channel calls runtime/action boundaries; it does not inspect the filesystem
   and does not spawn a process.
3. Runtime or CLI selects registered action `run_skill_script`.
4. The action resolves:
   - skill name
   - trust and enabled status
   - validated capability contract
   - relative script path
   - inventoried resource kind and digest
   - args, cwd, env, timeout, and output cap
5. The resolver rejects missing, untrusted, disabled, non-selected,
   non-inventoried, non-script, hidden, absolute, traversal, invalid-arg, or
   digest-mismatched requests before confirmation.
6. Security Central evaluates `:skill_script_execute` with selected skill,
   script resource, digest, cwd, env, timeout, output cap, origin channel, and
   runner metadata.
7. If policy denies, the action returns a denial and nothing runs.
8. If policy needs confirmation, the action writes a durable pending
   confirmation with redacted script metadata and no side effect.
9. User approves or denies from CLI, `/settings`, or a future approval handoff
   surface.
10. Approval re-reads the record, checks resolver policy, re-checks Security
    Central, re-resolves the skill resource, and re-checks digest.
11. If anything changed or policy now denies, the confirmation resolves without
    executing.
12. If still allowed, approval resumes `run_skill_script` exactly once through
    `AllbertAssist.Actions.Runner.run/3`.
13. The runner executes only the explicit launch vector with bounded cwd, env,
    timeout, and output capture.
14. The confirmation resolves with target result metadata, and trace/audit
    records show what happened.

## Runner Launch Contract

`run_skill_script` owns script launch policy. It does not route through
`run_shell_command` and does not treat arbitrary scripts as allowed shell
commands.

Direct executable launch:

- resolve the inventoried script to an absolute path under the trusted skill
  root
- verify it is a regular file, still matches the stored sha256, and is
  executable by the host user
- call the runner with executable set to the resolved script path and argv set
  to the explicit args
- use operator-supplied cwd only when it is under `execution.local.allowed_roots`
- otherwise create an internal per-run cwd under
  `<ALLBERT_HOME>/execution/skill-scripts/runs/<run-id>/cwd`

Interpreter launch is optional in v0.09 and must be policy-defined. If no
matching explicit interpreter profile exists, non-executable scripts are
denied before confirmation.

Path-like args are constrained by the v0.08 allowed-root policy. Non-path args
are opaque argv values. Shell syntax remains data, not syntax, and must not be
expanded by Allbert.

## Operator Surface Flow

- `mix allbert.skills run SKILL SCRIPT [--cwd PATH] [--timeout MS]
  [--max-output-bytes BYTES] -- [ARGS...]` should create a pending
  `run_skill_script` confirmation when the script spec is valid.
- `mix allbert.confirmations show CONFIRMATION_ID` should display skill name,
  script path, digest, cwd, timeout, output cap, and risk summary.
- `mix allbert.confirmations approve CONFIRMATION_ID --reason "smoke"` should
  resume the script target once and print result metadata.
- `/settings` should render the same pending/resolved script metadata through
  registered confirmation actions.
- Runtime traces should include a bounded Skill Script Metadata section.
- Execution audit should record script request, approval, denial, stale,
  digest mismatch, success, failure, timeout, and truncation events.

Operator-facing messages should be explicit:

- pending: "Skill script is ready for operator approval. Nothing has executed
  yet."
- denied: "Skill script execution was denied: <reason>."
- digest mismatch: "Approved, but not executed: the script changed after the
  request was created."
- completed: "Skill script executed with exit status <status>."
- timed out: "Skill script timed out after <timeout>ms."

## Denial Flow

Denials happen before confirmation when possible.

Examples:

- `execution.skill_scripts.enabled` is false
- `permissions.skill_script_execute` is denied
- skill is not trusted, enabled, discoverable, or selected
- skill contract does not allow `run_skill_script`
- script path is absolute, hidden, traversal, missing, or non-script
- resource digest differs from inventory
- cwd is outside `execution.local.allowed_roots`
- args or env contain blocked/invalid values
- requested timeout or output cap exceeds policy
- launch mode requires an interpreter profile that does not exist
- approval re-check finds changed policy, changed skill status, or digest drift

Denied results should include skill/script metadata, denial reason, no-output
status, and no-side-effect status.

## Trace And Audit Requirements

Every pending, denied, approved, stale, timed-out, failed, or succeeded script
request should include:

- confirmation id, when present
- origin channel and resolver channel
- selected skill name, source scope, trust status, and enablement
- selected action and permission
- script relative path, byte size, and sha256
- launch mode and resolved executable summary
- digest re-check status
- args summary with secret-like values redacted
- cwd summary and allowed-root status
- env allowlist summary, not env values
- timeout and output cap
- Security Central decision before pending creation
- Security Central decision after approval
- runner backend and sandbox level
- exit status or timeout status
- output byte counts and redacted preview
- denial or failure reason

## Implementation Checklist

- Keep channels, CLI tasks, and LiveViews thin.
- Do not execute from `activate_skill`, skill metadata, registry parsing, or
  model output.
- Do not mutate confirmation records directly from channel code.
- Do not trust client-supplied script paths or digests; re-resolve from the
  registry and filesystem.
- Do not add package installs, network adapters, generic scripting engines, or
  container dependencies in v0.09.
- Use temporary Allbert homes, temporary skill roots, and temporary execution
  roots in tests.
