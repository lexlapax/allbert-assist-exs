# Allbert v0.23 Jido State-Machine Convergence

## Status

Status: implemented through M5 closeout on 2026-05-16. Operator manual
verification and release tagging remain pending.

v0.23 is a docs-verified implementation plan. v0.22 is released and tagged as
`v0.22`; its bridge/routing/audit closeouts are now baseline regression
context, not design inputs for v0.23.

v0.23 is a **pure architectural refactor**. It is the substrate-only milestone
the project-direction rethink inserted before v0.24 ships
`Objectives.Engine` as a new Jido.Agent on top of this converged substrate.
No operator-visible default behavior changes; no schema changes; no new
permissions, signals, capability actions, surfaces, channels, objectives, or
app-specific behavior. The only new operator-controllable knob is the
diagnostic setting `allbert.jido.debug_trace`, default `false`.

## Purpose

v0.23 converges the two clearest existing state-machine components onto a
shared Jido substrate before v0.24 ships the objective runtime:

- `AllbertAssist.Confirmations.Store`
- `AllbertAssist.Jobs.Scheduler`

Both conversions go through a single shared behaviour,
`AllbertAssist.JidoBacked`, so the resulting pattern is one substrate
contract rather than two ad-hoc ports. v0.24 `Objectives.Engine` then
adopts the same pattern, keeping the runtime internally consistent with
the Allbert/Jido vision.

This is an architectural refactor, not a feature milestone. It must make the
runtime more internally consistent with the Allbert/Jido vision while keeping
the v0.07 confirmation workflow and v0.13 scheduled-job workflow byte-equivalent
to operators.

The current code matters:

- `AllbertAssist.Confirmations.Store` is a plain file-backed module, not a
  `GenServer` and not SQLite-backed. ADR 0008 makes confirmation records
  durable action state under Allbert Home. v0.23 must preserve that storage
  model. The Jido.Agent layer becomes the state-machine/coordinator around
  those files; it does not migrate confirmations into SQLite.
- `AllbertAssist.Jobs.Scheduler` is a supervised `GenServer` that keeps no
  authoritative job queue in memory. Each tick reads due jobs from SQLite and
  claims runs durably. v0.23 must preserve that contract. The Jido.Agent layer
  owns scheduler lifecycle and tick commands; SQLite remains authoritative for
  jobs and job runs.

## Architecture Readiness Audit

v0.23 is consistent with the current architecture if it preserves these
boundaries:

- ADR 0006: Security Central remains the permission, risk, redaction, and
  audit boundary. No permission classes, safety floors, risk tiers, or
  redaction obligations change.
- ADR 0007: runtime-facing, effectful, security-relevant, or observable work
  still enters through registered Jido actions and `Actions.Runner.run/3`.
  Converting an internal state holder to `Jido.Agent` does not create a second
  action boundary. Internal v0.23 command modules may use `Jido.Action`, but
  they are not registered Allbert capability actions and are not exposed to
  the intent/tool surface.
- ADR 0008: confirmation records stay durable under Allbert Home. The agent's
  in-memory state is a rebuildable projection and transition coordinator. It
  is not authoritative state.
- ADR 0014: local identity semantics are unchanged. `user_id`, `operator_id`,
  `thread_id`, and `session_id` fields in confirmation/job records preserve
  their existing meanings.
- ADR 0017: plugin contract unchanged. The JidoBacked substrate is a core-app
  module; plugins do not gain a new contribution surface in v0.23.
- ADR 0021: v0.24 objectives will consume confirmations/jobs after this
  convergence, but v0.23 does not add `objective_id`, `step_id`, objective
  signals, objective settings, or objective CLI commands.

No new ADR is required for v0.23. ADR 0007 is amended by this readiness pass
to record the pragmatic `Jido.Agent` substrate rule and to clarify that
internal `Jido.Action` command modules are not Allbert capability actions. If
implementation discovers that Jido's current API forces a behavior change
visible outside the internals, stop and write or update an ADR before
proceeding.

## Vision And Roadmap Alignment

The Allbert/Jido vision says:

- Use `Jido.Signal` for events.
- Use `Jido.Action` for capability boundaries.
- Use `Jido.Agent` where state machines, lifecycle hooks, or successor-agent
  stories earn their keep.
- Use plain `GenServer` or plain modules for storage wrappers where an agent
  abstraction adds ceremony without value.

v0.23 implements the clearest convergence case without overcorrecting. It does
not turn Settings, Trace, Memory storage IO, Session.Scratchpad,
Memory.Compiler, or Memory.Promotion into Jido.Agents.

Roadmap dependency order:

- v0.22 supplied the latest confirmation-heavy regression baseline.
- v0.23 converges Confirmations.Store and Jobs.Scheduler through the new
  `JidoBacked` behaviour.
- v0.24 adds Objective Runtime Foundation, with `Objectives.Engine` using the
  same `JidoBacked` behaviour and supervisor pattern.
- v0.25 adds native Jido trading agents consuming objective state from day
  one.

v0.23 must not include StockSage-specific branches, callbacks, settings, or
JidoBacked behavior. Shipped plugins may be used as regression consumers, but
the substrate is core-owned and app-neutral.

## Guiding Rules

1. **No operator-visible behavior change.** CLI output, LiveView behavior,
   trace sections, audit file content, signal payloads, settings, and error
   messages must remain byte-identical or trivially equivalent. Trivially
   equivalent means timestamps, process identifiers, or internal debug
   metadata may differ; operator-meaningful fields may not. v0.23 closeout
   verifies this through focused regression tests, the full test gate, and
   retained fixture snapshots under
   `apps/allbert_assist/test/fixtures/v0.23/` for canonical confirmation
   audit and scheduler summary scenarios. Future milestones should treat
   the public API, CLI, trace, audit, signal tests, and these fixtures as
   the compatibility contract.

2. **No durable schema changes.** No Ecto migration. Confirmation files stay
   under Allbert Home. Job tables stay unchanged.

3. **One internal substrate, two consumers.** Both conversions go through
   `AllbertAssist.JidoBacked` (a shared behaviour). v0.24
   `Objectives.Engine` adopts the same pattern. No per-conversion
   bespoke agent skeleton.

4. **State is rebuildable.** On start/restart, agent state is reconstructed
   from Allbert Home confirmation files or SQLite job rows. In-memory agent
   state is never the source of truth.

5. **Actions remain the authority boundary.** A Jido.Agent command may
   coordinate internal state, but all externally meaningful capability work
   still flows through registered Allbert actions, Security Central,
   confirmations, and the action runner. Internal v0.23 `Jido.Action`
   command modules are private agent implementation details. They are not
   registered in `AllbertAssist.Actions.Registry`, are not intent candidates,
   and are not operator-callable actions.

6. **Use documented Jido APIs.** Prefer Jido.Action command modules,
   `signal_routes`, `Jido.AgentServer` calls, `on_before_cmd/2`,
   `on_after_cmd/3`, and documented directives such as
   `Jido.Agent.Directive.schedule/2`. Do not invent macros or lifecycle hooks.
   If a directive cannot express the needed timing (cron, sub-second jitter,
   etc.), fall back to a tiny `Process.send_after`-based wrapper module and
   document the constraint in the agent's `@moduledoc`.

7. **Keep storage helpers pure.** Path helpers, YAML codec helpers, record
   rendering, schedule parsing, and Ecto changesets remain plain helper
   modules/functions. Do not wrap everything in an agent just because this
   milestone exists.

8. **Atomic rename.** Transitional compatibility modules are temporary
   scaffolding while each conversion lands. Both are deleted at v0.23 M5
   closeout. v0.24 must not depend on them for any reason.

## Substrate Technology

### `AllbertAssist.JidoBacked` behaviour

A single shared behaviour both conversions implement. Goals:

- Bundle the common Jido.Agent boilerplate (use macro, agent server start
  spec, supervisor child spec, lifecycle hook registration, signal-routing
  defaults) into one module so each conversion stays focused on its own
  state-machine semantics.
- Provide a `JidoBacked.child_spec/1` helper that returns a child spec map
  consumable by `AllbertAssist.JidoBacked.Supervisor`.
- Provide a uniform mechanism for declaring private Jido command modules
  used by the agent. These modules use `Jido.Action` for validation and
  state transitions but never enter `AllbertAssist.Actions.Registry`.
- Accept optional `schema:` and optional `signal_routes:` options so v0.24
  can define `Objectives.Engine.Agent` state before every command route is
  attached.
- Implementation note: the macro must forward the caller-provided
  `schema:` and `signal_routes:` compile-time declarations through to
  `use Jido.Agent`. Do not hardcode `schema: []`, require
  `signal_routes:`, or convert rich schema defaults into inert AST data.
  v0.24 objective state needs map defaults like `%{}` to initialize as
  real maps, not quoted syntax trees.
- Provide a uniform debug-trace renderer: when `allbert.jido.debug_trace`
  (new setting; boolean; default `false`) is on, trace rendering appends a
  bounded `## Jido Debug` section for configured JidoBacked agents. Default
  operator traces stay unchanged.
- Dispatch semantics: private commands that need a caller-visible result
  write `last_result: {:ok, value}` or `last_result: {:error, reason}` into
  their state patch. Directive-only commands may return an empty patch plus
  one or more Jido directives; `JidoBacked.dispatch/4` treats that as
  `{:ok, :dispatched}` rather than an error. This keeps v0.24 objective
  stage commands free to emit directives without inventing meaningless
  return payloads.

### `AllbertAssist.JidoBacked.Supervisor`

A new supervisor under `AllbertAssist.Supervisor` that owns the v0.23
JidoBacked agents. v0.24 will register `Objectives.Engine.Agent` under the
same supervisor.

Behavioral notes:

- One-for-one restart strategy.
- Max restarts / max seconds defaults match the existing
  `AllbertAssist.Supervisor` posture; tuning belongs in
  `apps/allbert_assist/lib/allbert_assist/application.ex`, not in plan text.
- Child spec list is built from each JidoBacked agent's `child_spec/1` so the
  agents stay co-located with their domain modules (the supervisor only knows
  it has a list of children).
- `:extra_children` appends later JidoBacked coordinators (for example
  v0.24 `Objectives.Engine.Agent`) without replacing the v0.23 confirmation
  and scheduler agents. `:children` remains an explicit full override for
  focused tests only.
- `AllbertAssist.Application` starts `JidoBacked.Supervisor` instead of
  starting `AllbertAssist.Jobs.Scheduler` as a separate child. The
  `Jobs.Scheduler` module remains a public facade; starting both the facade
  and `Jobs.Scheduler.Agent` would double-start the scheduler and is
  forbidden.
- Existing `config :allbert_assist, AllbertAssist.Jobs.Scheduler, ...`
  options are passed into `JidoBacked.Supervisor` and then to
  `Jobs.Scheduler.Agent`, preserving the v0.13 enable/disable posture.

defmodule AllbertAssist.JidoBacked.Supervisor do
  use Supervisor

  @spec start_link(keyword()) :: Supervisor.on_start()
  def start_link(opts \\ []), do: Supervisor.start_link(__MODULE__, opts, name: __MODULE__)

  @impl true
  def init(opts) do
    children = [
      {AllbertAssist.Confirmations.Store.Agent, Keyword.get(opts, :confirmations, [])},
      {AllbertAssist.Jobs.Scheduler.Agent, Keyword.get(opts, :scheduler, [])}

    Supervisor.init(children, strategy: :one_for_one)
  end
end

| subject | when emitted | payload contract preserved |
|---|---|---|
| `allbert.job.created` | Job context creates a job row. | job_id, kind, user_id, schedule, paused. |
| `allbert.job.updated` | Job context updates a job row (pause/resume/cancel/edit). | job_id, change, paused. |
| `allbert.job.due` | Scheduler finds and durably claims a due run. | run_id, job_id, trigger, user_id, operator_id, thread_id, session_id, app_id, due_at, status, metadata. |
| `allbert.job.started` | Runner begins execution. | run_id, job_id, trigger, user_id, operator_id, thread_id, session_id, app_id, started_at, status, metadata. |
| `allbert.job.completed` | Runner completes successfully. | run_id, job_id, trigger, user_id, operator_id, thread_id, session_id, app_id, finished_at, status, metadata. |
| `allbert.job.failed` | Runner returns `{:error, _}`. | run_id, job_id, trigger, user_id, operator_id, thread_id, session_id, app_id, finished_at, status, metadata.reason (bounded). |
| `allbert.job.skipped` | Run skipped (disabled, locked, blocked, etc.). | run_id, job_id, trigger, user_id, operator_id, thread_id, session_id, app_id, status, metadata.reason. |
| `allbert.job.needs_confirmation` | Target action returned `:needs_confirmation`. | run_id, job_id, trigger, user_id, operator_id, thread_id, session_id, app_id, finished_at, status, metadata.confirmation_id. |

### Trace metadata posture

Default trace output (when `allbert.jido.debug_trace = false`) is
byte-identical to v0.22 for the same input. With `debug_trace = true`,
trace markdown gains a bounded top-level `## Jido Debug` subsection for the
configured JidoBacked agents, capturing:

- agent module name
- last command processed
- last projection rebuild timestamp (Store only)
- last tick timestamp + summary (Scheduler only)

No raw secrets, no full record bodies, no operator-meaningful field
changes.
