## Flow 1: Application Startup — Bridge Initialization

When Allbert starts with the StockSage plugin available:

1. Plugin bootstrap discovers `./plugins/stocksage` and starts
   `StockSage.Supervisor` as a child of the plugin supervisor.
2. `StockSage.Supervisor` starts `StockSage.TraderBridge` as one of its
   children.
3. When `stocksage.bridge_enabled` is `true` (the default), `TraderBridge`
   opens a Port to `bridge.py` using the configured Python interpreter.
4. The bridge process starts and waits for JSON requests on stdin.
5. `StockSage.TraderBridge` sends a ping; the bridge responds with pong.
6. The bridge is now in running state. Allbert core is not aware of the bridge
   directly; the bridge is a plugin-internal child.

The operator observes: Allbert starts without errors. `mix allbert.plugins show
stocksage` shows the StockSage plugin with `RunAnalysis` listed under
contributions. The bridge is available for analysis requests.

When `stocksage.bridge_enabled` is `false`: the bridge is not started.
`RunAnalysis` returns a disabled error before creating any confirmation record.
The operator observes: `mix allbert.plugins show stocksage` shows bridge status
as `disabled`. Analysis requests fail immediately with a clear disabled message.

## Flow 2: Analysis Request Via CLI

The operator requests analysis for a specific ticker and date.

1. The operator runs `mix stocksage.analyze AAPL 2026-05-01 --user local`.
2. The CLI calls `StockSage.Actions.RunAnalysis` through
   `AllbertAssist.Actions.Runner.run/3` with `ticker: "AAPL"`,
   `analysis_date: "2026-05-01"`, `engine: "tradingagents"`, `user_id: "local"`.
3. The action validates ticker and date format. Invalid inputs fail here with a
   clear error; no confirmation record is created.
4. Security Central evaluates `:stocksage_analyze`. The default policy is
   `needs_confirmation`.
5. The action creates a durable confirmation record. The record includes ticker,
   analysis date, engine, user_id, risk tier `high`, and a disclosure that
   TradingAgents will make external market-data API calls as part of this
   analysis. No analysis is run yet.
6. The action returns `:needs_confirmation` with the confirmation id.
7. The CLI displays:
   ```
   Analysis confirmation required. ID: <confirmation-id>
   Run: mix allbert.confirmations approve <confirmation-id> --reason "..."
   ```
8. The operator runs `mix allbert.confirmations list` to inspect the pending
   record and confirm the ticker, date, and disclosure.
9. The operator runs `mix allbert.confirmations approve <id> --reason "smoke test"`.
10. The approval action re-reads the confirmation record, re-checks Security
    Central with confirmation context, and invokes `RunAnalysis` through the
    runner for the approved resume path.
11. `RunAnalysis` calls `StockSage.TraderBridge.analyze/1` with the validated
    params.
12. The bridge process receives a JSON request, dispatches to TradingAgents,
    and returns a JSON response.
13. The action persists the result: one `stocksage_analyses` row and one
    `stocksage_analysis_details` row. The confirmation record moves to
    resolved state.
14. The CLI displays the analysis id, ticker, date, engine, and a bounded
    summary of the result.

The operator observes: the analysis result is persisted and displayed with a
bounded summary. Running `mix allbert.confirmations list --resolved` shows the
resolved confirmation. Running `mix allbert.ask "list my stocksage analyses"`
returns the new analysis in the list.

If the operator denies the confirmation:
`mix allbert.confirmations deny <id> --reason "not now"`
The analysis does not run. The queue entry (if any) remains in its current
state. The confirmation moves to resolved/denied.

## Flow 3: Analysis Request Via Natural Language

The user is in a StockSage session and asks for analysis by text.

1. The user has set `active_app: :stocksage` in their session scratchpad (either
   by explicit session command or by navigating to StockSage context).
2. The user sends: `"analyze TSLA for yesterday"` through any channel.
3. The runtime passes the request to the intent engine.
4. The engine collects candidates. With `active_app: :stocksage`, StockSage
   skill paths and registered StockSage actions are scored higher. The
   `run-analysis` skill with `run_analysis` action binding appears as a high-
   scoring candidate with `source: skill` and `app: stocksage`.
5. The engine selects `run_analysis` as the intent.
6. The runtime calls `RunAnalysis` through the action runner with the extracted
   ticker and date (parsed from the natural language request), and `user_id`
   from the session identity.
7. Security Central evaluates `:stocksage_analyze`: confirmation required.
8. The action creates a pending confirmation record and returns
   `:needs_confirmation`.
9. The runtime response includes the confirmation id and instructs the user to
   approve through their current channel.

The user observes: Allbert responds with a confirmation request explaining that
analysis requires approval and including the confirmation id. The user approves
through CLI (`mix allbert.confirmations approve`) or, in a future channel,
through channel-native approval (inline button for Telegram, typed command for
email). After approval, the analysis runs and the result summary is returned to
the approval channel.

When `active_app` is not set to `stocksage`: `RunAnalysis` is not selected as a
top candidate by the engine for a generic "analyze something" prompt. The engine
routes to general-purpose Allbert actions instead. StockSage analysis is
explicitly scoped to StockSage session context.

## Flow 4: Queue-Driven Analysis

The operator links an analysis run to an existing StockSage queue entry.

1. A `stocksage_analysis_queue` entry exists in status `queued` from a prior
   `QueueAnalysis` action call.
2. The operator runs:
   `mix stocksage.analyze TSLA 2026-05-01 --queue-id <queue_entry_id> --user local`
3. The CLI calls `RunAnalysis` with `ticker: "TSLA"`, `analysis_date: "2026-05-01"`,
   `queue_entry_id: "<id>"`, `user_id: "local"`.
4. The flow proceeds as in Flow 2 (confirmation created, approval required).
5. On approved resume: after the bridge returns, the action marks the queue
   entry as `completed` and writes a `stocksage_queue_runs` row recording the
   run duration and analysis id.
6. If the bridge fails, the queue entry is marked `failed` with a bounded
   reason.

The operator observes: after approval, the queue entry status changes from
`queued` to `completed`. The resulting analysis id is linked to the queue entry.
Running `mix allbert.ask "show stocksage queue"` reflects the updated status.

## Flow 5: Bridge Crash Recovery

The bridge process crashes unexpectedly during a pending analysis.

1. An analysis request has been approved and `RunAnalysis` has called
   `TraderBridge.analyze/1`.
2. The bridge Port exits with a non-zero status.
3. `StockSage.TraderBridge` receives the Port exit signal. All callers waiting
   for a response receive `{:error, :bridge_crashed}`.
4. `RunAnalysis` receives the crash error. It writes a `stocksage_analyses`
   failure row with `status: "failed"` and `summary: "bridge_crashed"`. It
   returns a structured error result.
5. The CLI displays: `Analysis failed: bridge_crashed`. No partial data is
   leaked.
6. The `TraderBridge` GenServer clears the Port reference. On the next
   `analyze/1` call, it attempts to restart the Port. If it succeeds, the
   bridge is back in running state.

The operator observes: the failed analysis is recorded with status `failed`.
A subsequent `mix stocksage.analyze` for a new request creates a new
confirmation and eventually invokes a new bridge call. If the Port continues
to fail, `TraderBridge` logs the error and returns `{:error, :bridge_crashed}`
on each call. Allbert core and other functionality are not affected.

## Flow 6: Bridge Disabled At Runtime

The operator disables the bridge via Settings Central.

1. The operator runs:
   `mix allbert.settings set stocksage.bridge_enabled false`
2. On the next `RunAnalysis` invocation (through CLI or natural language),
   the action reads `bridge_enabled` from Settings Central.
3. The action returns `{:error, :bridge_disabled}` before creating any
   confirmation record.
4. The CLI displays: `Analysis unavailable: bridge is disabled. Use
   mix allbert.settings set stocksage.bridge_enabled true to re-enable.`

The operator observes: no confirmation record is created. No bridge process is
invoked. Re-enabling the setting restores the normal Flow 2 behavior on the
next analysis request.

Note: changing `bridge_enabled` at runtime does not stop the already-running
`TraderBridge` GenServer. The GenServer remains supervised but idle when
`bridge_enabled` is false. A full application restart is not required.

## Failure And Safety Cases

| Situation | Expected behavior |
| --- | --- |
| Invalid ticker format (special chars, >10 chars) | Schema validation error before confirmation; no pending record created. |
| Invalid analysis_date (non-ISO-8601, future century) | Schema validation error before confirmation; no pending record created. |
| `bridge_enabled: false` | Action returns `{:error, :bridge_disabled}` before confirmation; CLI prints clear message. |
| Bridge Port exits before analysis completes | `{:error, :bridge_crashed}` returned to caller; failure row written; no partial data leaked. |
| Bridge times out (`bridge_timeout_ms` exceeded) | `{:error, :timeout}` returned; failure row written; bridge process is still running. |
| Confirmation denied for RunAnalysis | Analysis does not run; no bridge call; queue entry (if any) remains in prior state. |
| Confirmation re-checked at approval and policy is now denied | Confirmation resolves as denied; bridge is not called. |
| `RunAnalysis` called without `active_app: :stocksage` via natural language | Intent engine does not rank RunAnalysis highly; routes to other actions or asks for clarification. |
| `queue_entry_id` references a non-existent or already-consumed entry | Action returns a bounded error; no analysis row is written for that entry. |
| TradingAgents returns malformed JSON | Bridge returns `{status: "error", reason: "invalid_json"}` after Protocol decode fails; action writes failure row. |
| Analysis result exceeds `bridge_max_output_bytes` | Bridge truncates the result body and sets `truncated: true`; truncated row is persisted; no data is lost silently. |
| Bridge restart fails repeatedly | `TraderBridge` logs errors; every `analyze/1` call returns `{:error, :bridge_crashed}`; Allbert core is not affected. |
| Raw analysis body appears in a trace | Must not happen; trace section is bounded and summary-only; redaction tests cover this. |
| Cross-user: accessing another user's analyses | `ListAnalyses` and `ShowAnalysis` filter by user_id; no cross-user data is returned. |

## Non-Goals Visible To Users

- v0.22 does not add StockSage LiveViews or a workspace UI for analysis results.
  Results are queryable through `mix allbert.ask` and CLI actions.
- v0.22 does not confirm individual market-data API calls made by TradingAgents.
  The single `RunAnalysis` confirmation covers the full analysis including
  external calls. v0.28 (formerly v0.26) hardens this.
- v0.22 does not add native Jido trading agents. TradingAgents via the Python
  bridge is the only engine.
- v0.22 does not automatically consume queue entries on a schedule. Queue
  entries are consumed only by explicit operator or confirmed runtime action.
- v0.22 does not canvas-render analysis results. Canvas integration is v0.30
  (formerly v0.28).
- v0.22 does not add memory namespace registration for StockSage. That is
  v0.29 (formerly v0.27).
- Raw analysis output is never shown in traces, CLI list summaries, or signals.
  Only bounded structured metadata is surfaced.
