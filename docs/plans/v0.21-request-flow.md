## Flow 1: Memory Append

The user asks Allbert to remember something through any surface.

1. The user submits a request such as "remember: I prefer concise responses."
2. The intent engine selects the `append_memory` action.
3. Security Central authorizes `:memory_write`.
4. The action writes a markdown entry under
   `<ALLBERT_HOME>/memory/<category>/`. The category is `preferences` when
   heuristic detection identifies an identity or preference statement; otherwise
   `notes`.
5. The entry format follows the existing v0.01 structure: `# Memory: {summary}`,
   metadata fields, and `## Body`. No `## Review` section is added at write time.
6. The response confirms what was saved. The trace includes the entry path,
   category, and source signal id.

The user observes: a confirmation message. The entry file is present under the
correct category directory. No review section is present until an explicit
review action is run.

## Flow 2: Memory List And Read

The operator lists memory entries and reads one in full.

1. The operator runs `mix allbert.memory list` or
   `mix allbert.memory list --category preferences`.
2. The CLI calls the `list_memory_entries` action with optional `category`,
   `review_status`, `limit`, and `since` filters.
3. Security Central authorizes `:read_only`.
4. The action reads markdown files from the memory root for the given user,
   returns bounded `Entry` maps with summary, category, timestamp, and path.
5. The operator copies a path and runs
   `mix allbert.memory show <path>`.
6. The `read_memory_entry` action returns the full entry including body and
   `## Review` section if present.

The operator observes: a bounded list of entries with summary, category,
timestamp, and review_status. The show command renders the full entry
including body. Entries without a `## Review` section show review_status as
`unreviewed`. Cross-user paths return not-found.

## Flow 3: Memory Review And Correction

The operator reviews an entry and optionally corrects its content.

1. The operator runs
   `mix allbert.memory review <path> --status flagged --note "stale"`.
2. The CLI calls `review_memory_entry` with `path`, `status: :flagged`, and
   `note`.
3. Security Central authorizes `:memory_write`.
4. The action reads the entry, validates the path is inside the memory root,
   and writes a `## Review` section with `Status: flagged`,
   `Reviewed: <timestamp>`, `Reviewed by: <user_id>`, and `Correction note:
   stale`.
5. A subsequent `mix allbert.memory show <path>` returns the updated entry
   with the review section visible.

To correct the entry body or summary, the operator runs
`mix allbert.memory update <path> --summary "Concise response preference"`.

1. The CLI calls `update_memory_entry` with `path` and the updated field.
2. Security Central authorizes `:memory_write`.
3. The action writes the updated content while preserving all existing metadata
   headers and the `## Review` section.
4. The entry path and timestamp do not change.

The operator observes: a review or update confirmation message. The subsequent
show renders the updated section. The entry file is still in the same location.

## Flow 4: Memory Deletion With Confirmation

The operator deletes a memory entry through the durable confirmation workflow.

1. The operator runs `mix allbert.memory delete <path>`.
2. The CLI calls `delete_memory_entry` with `path`.
3. Security Central authorizes `:memory_write`.
4. When `memory.delete_requires_confirmation` is true (the default), the
   action creates a durable confirmation record through the v0.07 confirmation
   workflow and returns `:needs_confirmation`.
5. The CLI displays the confirmation id.
6. The operator runs `mix allbert.confirmations list` to see the pending record.
7. The operator runs `mix allbert.confirmations approve <id> --reason "smoke test"`.
8. On approval, the markdown file is moved to
   `<ALLBERT_HOME>/memory/deleted/<YYYY-MM>/`.
9. The file is never hard-deleted; it is archived in the deleted folder.

The operator observes: a pending confirmation is created and the file remains
in place until approval. After approval, the file is absent from the category
directory and visible under `memory/deleted/`. A subsequent
`mix allbert.memory show <path>` returns not-found.

If `memory.delete_requires_confirmation` is set to false, the file is archived
immediately without a confirmation record.

## Flow 5: Memory Prune

The operator prunes stale or nominated entries in bulk.

1. The operator runs `mix allbert.memory prune` (dry-run mode, the default).
2. The CLI calls `prune_memory_entries` with no `write` flag.
3. Security Central authorizes `:memory_write`.
4. The action identifies candidates by three criteria: entries with
   `review_status: :prune_nominated`, trace-category entries older than the
   retention window, and any category where entry count exceeds
   `memory.max_entries_per_category`.
5. Dry-run returns the candidate list with reasons. No file is moved.
6. The operator inspects the list and runs
   `mix allbert.memory prune --write` to proceed.
7. When `memory.prune_requires_confirmation` is true (the default), write mode
   creates a bulk confirmation record listing all candidates and returns
   `:needs_confirmation`.
8. After `mix allbert.confirmations approve <id>`, each candidate is moved to
   `memory/deleted/` archive.
9. If `memory.prune_requires_confirmation` is false, write mode archives the
   candidates immediately. This setting is separate from
   `memory.delete_requires_confirmation`.

The operator observes: dry-run shows candidate count and reasons without any
mutations. Write mode displays a confirmation id. After approval, the
candidates are absent from the active category directories and present in
`memory/deleted/`.

## Flow 6: Compiled Index And Search

The operator builds the memory index and searches entries by text.

1. The operator runs `mix allbert.memory compile-index`.
2. The CLI calls `compile_memory_index`.
3. Security Central authorizes `:read_only`.
4. The action reads all markdown entries through `Memory.Compiler.compile_index/2`,
   builds a JSON index bounded by `memory.max_index_entries`, and writes it to
   `<ALLBERT_HOME>/memory/.index.json`.
5. The index file starts with a derived-artifact header:
   `# DERIVED - DO NOT EDIT`, `# Source: …`, `# Rebuilt: {iso8601}`.
6. The action returns entry count, elapsed time, and categories covered.
7. The operator then runs `mix allbert.memory search "response"`.
8. The `search_memory` action queries through `Memory.Index.query/3`, returns
   ranked `Entry` maps with category, summary, timestamp, and match reasons.
9. When the index is absent or stale, `search_memory` falls back to direct
   markdown reads.

The operator observes: compile-index reports entry count and elapsed time.
Search returns entries ranked by keyword match and recency. Entries with
`review_status: :flagged` or `:prune_nominated` are not returned by search.

## Flow 7: Category Summary

The operator views a compiled summary of entries in a category.

1. The operator runs
   `mix allbert.memory summarize --category preferences`.
2. The CLI calls `summarize_memory_category` with `category: :preferences`.
3. Security Central authorizes `:read_only`.
4. The action writes a derived summary file to
   `<ALLBERT_HOME>/memory/preferences/.summary.md`.
5. The summary file starts with the derived-artifact header, followed by an
   entry table (timestamp, summary, review_status) and the bodies of recently
   reviewed `kept` entries.
6. No model is used; the summary is a compiled view over existing markdown.

The operator observes: a summary file is created and its path is reported.
Running `cat "$ALLBERT_HOME/memory/preferences/.summary.md"` shows the
derived header followed by the entry table.

## Flow 8: Conversation Turn Promotion

The operator promotes one conversation message to a markdown memory entry.

1. The operator runs
   `mix allbert.threads --user local` to find a thread id.
2. The operator identifies a `message_id` from that thread.
3. The operator runs:
   ```sh
   mix allbert.memory promote-turn \
     --thread-id <thread-id> --message-id <message-id> \
     --category notes --user local
   ```
4. The CLI calls `promote_conversation_turn` with `user_id`, `thread_id`,
   `message_id`, and `category`.
5. Security Central authorizes `:memory_write` and `:confirmation_decide`.
6. `Memory.Promotion.from_thread_message/4` validates that the message belongs
   to `user_id` and constructs the entry body (bounded to 2000 chars).
7. When `memory.promotion_requires_confirmation` is true (the default), the
   action creates a confirmation record with a redacted body preview and returns
   `:needs_confirmation`.
8. The CLI displays the confirmation id.
9. The operator approves the confirmation.
10. On approval, `Memory.append/1` writes the markdown entry under the target
    category.

The operator observes: a confirmation record is created with a bounded body
preview. After approval, `mix allbert.memory list --category notes` shows the
new entry. Messages belonging to another user return not-found before any
confirmation is created.

## Flow 9: Memory Candidates In The Intent Trace

After `compile-index` is run with a seeded memory root, the user asks a
question whose text matches existing memory entries.

1. The user runs:
   ```sh
   mix allbert.ask --trace "what do you know about my preferences?"
   ```
2. Runtime passes the request to the intent engine.
3. When `memory.index_enabled` is true and the index exists and is not stale,
   `Engine.collect_candidates/1` calls `Memory.Index.query/3` with the request
   text.
4. Matching entries become `:memory` kind candidates with `source: :memory`,
   a recency + keyword score between 0.1 and 0.5, and a bounded `reason` field
   containing the entry summary.
5. Entries with `review_status: :flagged` or `:prune_nominated` are excluded.
6. Memory candidates appear in the ranked candidate set alongside action, skill,
   and surface candidates.
7. Memory candidate scores can influence ranking but do not grant permissions or
   authorize actions.
8. The trace output includes up to 5 memory candidates in the
   `## Intent Candidates` section, each showing category, summary, and recency.
9. No entry body content appears in the trace.

The user observes: a normal assistant response informed by relevant memory
context. The trace shows `kind: memory` candidates with `source: memory`,
category, and recency. No raw entry bodies, conversation turns, or secrets
appear in the trace.

When `memory.index_enabled` is set to false or the index is absent, the trace
shows no memory candidates and the engine proceeds without error.

## Flow 10: Scheduled Index Rebuild

When `memory.review_cadence` is `"daily"` or `"weekly"`, Settings Central
creates or updates one managed scheduled job that triggers a memory index
rebuild. When it is set back to `"manual"`, that managed job is paused.

1. The operator runs `mix allbert.settings set memory.review_cadence weekly`
   or `daily`.
2. Settings Central validates the value and syncs the managed
   `memory-index-rebuild` job for the local user.
3. The v0.13 job scheduler fires based on the configured cadence.
4. The job invokes `compile_memory_index` through
   `AllbertAssist.Actions.Runner.run/3`.
5. The action rebuilds the index from current markdown files.
6. If the action succeeds, the `allbert.memory.index_compiled` signal is emitted
   with entry count and elapsed time.
7. The job does not auto-delete, auto-promote, or auto-prune entries.
6. Scheduled jobs that require confirmation are blocked through the existing
   job/confirmation path.

The operator observes: the index file's `Rebuilt:` timestamp updates after
each scheduled run. The signal is visible in the trace log. No memory entries
are mutated by the scheduled job.

### Verifying Memory List And Read

```sh
mix allbert.memory list
mix allbert.memory list --category preferences
mix allbert.memory show "$ALLBERT_HOME/memory/preferences/<entry>.md"
```

The operator observes: list shows entries with summary, category, timestamp,
and review_status. Show renders the full entry including body.

### Verifying Review And Correction

```sh
mix allbert.memory review "$ALLBERT_HOME/memory/preferences/<entry>.md" --status flagged
mix allbert.memory show "$ALLBERT_HOME/memory/preferences/<entry>.md"
mix allbert.memory update "$ALLBERT_HOME/memory/preferences/<entry>.md" --summary "Updated summary"
```

The operator observes: the show output includes a `## Review` section with
`Status: flagged`. Update changes only the summary; all other metadata is
preserved.

### Verifying Deletion With Confirmation

```sh
mix allbert.memory delete "$ALLBERT_HOME/memory/preferences/<entry>.md"
mix allbert.confirmations list
mix allbert.confirmations approve <id> --reason "smoke test"
ls "$ALLBERT_HOME/memory/deleted/"
```

The operator observes: a pending confirmation is displayed before any file
move. After approval, the file is absent from preferences/ and present in the
deleted/ archive.

### Verifying Index And Search

```sh
mix allbert.memory compile-index
mix allbert.memory search "response"
mix allbert.memory summarize --category preferences
```

The operator observes: compile-index reports entry count and elapsed time.
Search returns ranked entries. The summary file is written with a derived
header.

### Verifying Memory Candidates In Trace

```sh
mix allbert.memory compile-index
mix allbert.ask --trace "what do you know about my preferences?"
mix allbert.settings set memory.index_enabled false
mix allbert.ask --trace "what do you know about my preferences?"
```

The operator observes: the first ask shows `kind: memory` candidates in the
trace. After setting `memory.index_enabled` to false, the second ask shows
no memory candidates.

### Verifying Settings

```sh
mix allbert.settings get memory.index_enabled
mix allbert.settings get memory.review_cadence
mix allbert.settings get memory.delete_requires_confirmation
mix allbert.settings get memory.prune_requires_confirmation
mix allbert.settings get memory.promotion_requires_confirmation
mix allbert.settings set memory.review_cadence weekly
mix allbert.jobs list
mix allbert.settings set memory.review_cadence manual
```

The operator observes: all `memory.*` settings are readable and writable.
`memory.delete_requires_confirmation`, `memory.prune_requires_confirmation`,
and `memory.promotion_requires_confirmation` default to true. Setting
`memory.review_cadence` to weekly creates or updates one managed
`memory-index-rebuild` job; setting it back to manual pauses that job.

## Failure And Safety Cases

| Situation | Expected behavior |
| --- | --- |
| Entry path is outside the memory root | Action rejects before any read or write. |
| `read_memory_entry` called with a path belonging to another user | Returns not-found; no data leak. |
| `delete_memory_entry` called with `delete_requires_confirmation: true` | Creates pending confirmation; no file move until approved. |
| Confirmation denied for delete or promote | File remains in place; no archive move. |
| `prune_memory_entries` dry-run | Returns candidates with reasons; no files moved. |
| Index file is absent or stale | `search_memory` falls back to direct markdown reads; engine silently skips memory candidates. |
| `memory.index_enabled` is false | No memory candidates in engine; search uses direct reads. |
| `compile_memory_index` run on empty memory root | Returns entry count of 0; writes valid empty index file. |
| Entry body exceeds promotion bound | Body is truncated to 2000 chars before confirmation record is created. |
| `promote_conversation_turn` targets a message belonging to another user | Returns not-found before any confirmation is created. |
| Memory candidate score ties with action candidate | Action candidates win; memory candidates influence ranking only. |
| Memory candidate names a non-existent action | Candidate `source: :memory`; it cannot grant permissions or authorize execution. |
| Scheduled index rebuild fails | Job is marked failed; trace records bounded error; existing index file is not deleted. |
| Flagged or prune-nominated entry in index | Entry is excluded from `search_memory` and engine candidate collection. |
| Deleted entry path requested directly | Returns not-found from the active memory root; archived file is not served. |

## Non-Goals Visible To Users

- v0.21 does not add embeddings, vector databases, or semantic similarity
  search. Index retrieval uses scored keyword and recency weighting.
- v0.21 does not rewrite, summarize, or modify memory entry bodies using a
  language model. `summarize_memory_category` is a compiled view, not an
  LLM-generated document.
- v0.21 does not automatically promote conversation turns to memory without
  explicit operator selection and confirmation.
- v0.21 does not automatically promote StockSage domain `memory_entries`
  records to markdown Allbert memory.
- v0.21 does not hard-delete memory entries. Deleted and pruned entries are
  archived under `memory/deleted/`.
- v0.21 does not expose memory entry bodies in trace output, signals, or CLI
  list summaries.
- v0.21 does not add workspace shell surfaces, canvas components, or LiveView
  memory UIs.
- Memory candidates in the intent engine are proposal data: they can influence
  ranking scores but do not grant permissions or bypass Security Central.
- v0.21 does not add automatic memory mutation (auto-delete, auto-promote,
  auto-prune without confirmation).
