## Purpose

v0.18 matures the v0.15 minimal app registration contract into the full local
app/surface contract, except for memory namespace registration which remains
deferred to v0.29 (formerly v0.27).

This milestone adds `AllbertAssist.App.SurfaceProvider` and
`AllbertAssist.Surface`, then upgrades `AllbertAssist.App.CoreApp` into the
first real `SurfaceProvider`. `CoreApp` declares the existing `/agent`
conversation route as the built-in chat surface. That makes the current
conversation surface the first proven app/surface implementation before
StockSage v0.20 becomes the second.

Starting in v0.18, runtime requests that arrive without an explicit app context
resolve to `active_app: :allbert`. This default is context, not authorization.
It gives every runtime turn a declared home app while preserving the v0.14
scratchpad behavior for explicit app context.

v0.18 also fixes the v0.17 gap where plugin and app settings schema
contributions are stored in their respective registries but never merged into
Settings Central's validation and read paths.

v0.24 later upgrades `CoreApp`'s declared `/agent` surface into the
signal-driven workspace shell. The workspace shell is `CoreApp`'s surface
implementation; it is not a separate UI system floating outside the app
contract.

## Readiness Decision

v0.18 is ready for implementation when the plan, request-flow doc, roadmap,
development guide, and ADRs agree on these points:

- App registration does not grant permission. The v0.18 app contract is a
  richer contract, not a richer authority. Actions still run through
  `Actions.Runner.run/3`, Security Central, confirmations, and audits.
- `SurfaceProvider` is declarative only. It registers metadata; it does not
  mount routes, execute code, produce LiveView components, or authorize access.
- `active_app` is resolved session context, not routing or execution authority.
  Resolving to `:allbert` does not give a turn any fewer or more permissions
  than resolving to `:stocksage`.
- Unknown app ids from channels, jobs, external requests, and model output are
  never atomized. They fall back to `:allbert` with diagnostics.
- Settings schema contributions from app and plugin registries are runtime
  metadata merged at read/validation time. They are not compiled into
  `AllbertAssist.Settings.Schema` at boot.
- Surface DSL validation happens before registration. Invalid nodes, unknown
  components, invalid action bindings, and secret-like props are rejected at
  validation time, not at render time.
- The existing `/agent` Phoenix route stays a reviewed source-tree route.
  v0.18 declares it through the contract; v0.24 upgrades its implementation.
- AG-UI and A2UI must not become runtime dependencies in v0.18. The stub
  documents the future translation boundary and returns a typed
  not-implemented result only.
- Memory namespace registration must not appear in any v0.18 module. It is
  v0.27 work.

## Architecture Readiness Audit

The v0.18 plan is consistent with the current architecture:

- ADR 0001 requires signal-first runtime coordination. v0.18 adds
  `active_app` to input signal metadata so every turn has a declared home.
- ADR 0007 requires effectful, security-relevant behavior to enter through
  registered actions. `mix allbert.validate_app` calls the validation engine
  through the action runner or CLI pattern already established for
  `mix allbert.apps validate`. Surface events that invoke actions still route
  through `Actions.Runner.run/3`.
- ADR 0015 is the canonical app/surface contract decision. v0.18 is the
  M-AppContract-Full implementation that ADR 0015 describes. The ADR must be
  updated after implementation to reflect concrete module names and signatures.
- ADR 0017 is the plugin contract decision and the packaging boundary. v0.18
  does not replace it; plugin-contributed apps continue to register through the
  plugin registry first, then validate through the same App.Validator used for
  core apps.

No new package dependency is required for the local web path. If implementation
later depends on a library API detail not well-known from the existing codebase,
use Context7 before coding.

## Vision And Roadmap Alignment

The vision says Allbert should become a compact, inspectable, workspace-aware
runtime where domain apps plug in through public contracts rather than private
hooks, and where the user can inspect and steer work through declarative,
validated surfaces instead of trusting opaque generated code.

v0.18 makes both things real at the local level:

- The app contract gains its full declared scope: identity, lifecycle, agents,
  actions, signals, skills, settings, and surfaces. Memory namespaces are the
  only remaining deferred layer.
- `AllbertAssist.Surface` establishes the local declarative surface DSL.
  v0.26 (formerly v0.24) canvas and ephemeral UI consume this substrate;
  they do not invent a second one.
- `CoreApp` becomes a real `SurfaceProvider`, which proves the contract before
  StockSage uses it in v0.20.

Roadmap dependency order is intact:

- v0.15 supplied the minimal app contract and registry.
- v0.17 supplied the plugin contract and shipped channel plugins.
- v0.18 supplies the full app/surface contract, settings schema merge, and
  active app defaulting.
- v0.19 can consume registered surface metadata for cross-surface intent
  enrichment.
- v0.20 StockSage implements the full app/surface contract from day one; there
  is no lite-to-full migration path.
- v0.24 upgrades `CoreApp`'s declared `/agent` surface into the workspace
  shell; it consumes the stable app registry and Surface DSL.
- v0.27 adds memory namespace registration as the final deferred layer.

v0.18 must not pull v0.19 through v0.24 forward. Its job is to prove the full
app/surface contract and Surface DSL before StockSage and the workspace shell
consume them.

## Guiding Rules

1. `SurfaceProvider` is declarative only. Surfaces declare navigation paths
   and component metadata. They do not mount routes, start LiveViews, grant
   permissions, execute actions, or own domain state.

2. `active_app` is resolved session context, not execution authority. A turn
   with `active_app: :allbert` has identical permissions to any other turn.
   The intent agent may use `active_app` to prioritize skill paths and actions,
   but it is not routing authority and it does not bypass Security Central.

3. Unknown app ids are never atomized from string input. Channels, jobs,
   external requests, model output, and user input may provide string app ids.
   Use `App.Registry.normalize_app_id/1` for safe lookup; fall back to
   `:allbert` with diagnostics when the id is unknown.

4. Settings schema contributions merge at read/validation time, not at
   compile time. If a registry is unavailable, proceed with the static compiled
   schema and log a warning; do not crash.

5. App registration never grants permission. Plugin or app registration does
   not relax any Security Central safety floor or confirmation requirement.
   Actions registered by apps still run through `Actions.Runner.run/3`.

6. Surface DSL validation is the gate. Unknown components, invalid action
   bindings, secret-like props, raw HTML, remote URL paths, and script strings
   are rejected at validation time. The renderer does not re-validate.

7. Memory namespace registration must not appear in v0.18 code. It is v0.29
   (formerly v0.27) work. Any module, comment, or struct field that implies
   memory namespace registration is out of scope.

8. AG-UI and A2UI must not become runtime package dependencies. The stub
   in `AllbertAssist.Surface.Encoder` documents the future translation boundary
   and returns `{:error, :not_implemented}` only.

9. Cross-app duplicate route paths are registry diagnostics, not boot
   failures. Same-app duplicate surface ids are validation failures that reject
   the app.

10. The full v0.18 app contract must be backward-compatible with the v0.15
    minimal contract. `CoreApp`, `StockSageStub`, and any test fixture that only
    implements `app_id/0`, `display_name/0`, `version/0`, and `validate/1` must
    still validate without change.

## App Behaviour Expansion

`AllbertAssist.App` currently defines: `app_id/0`, `display_name/0`,
`version/0`, `validate/1`, `child_spec/1`, `actions/0`, `skill_paths/0`,
`surfaces/0`.

v0.18 adds optional callbacks with defaults via `use AllbertAssist.App`:

```elixir
@callback agents() :: [module()]
@callback signals() :: %{emits: [String.t()], subscribes: [String.t()]}
@callback settings_schema() :: [schema_entry()]
```

New types added to `AllbertAssist.App`:

```elixir
@type schema_entry :: %{
  required(:key) => String.t(),
  required(:type) => atom(),
  optional(:default) => term(),
  optional(:description) => String.t(),
  optional(:secret?) => boolean()
}

@type signal_declarations :: %{
  emits: [String.t()],
  subscribes: [String.t()]
}
```

Defaults added or extended by `use AllbertAssist.App`:

```elixir
def agents, do: []
def signals, do: %{emits: [], subscribes: []}
def settings_schema, do: []
# existing defaults preserved:
def child_spec(_opts), do: :ignore
def actions, do: []
def skill_paths, do: []
def surfaces, do: []
```

`surfaces/0` is kept for v0.15 legacy navigation summaries. Apps that implement
`AllbertAssist.App.SurfaceProvider` declare interactive surfaces through that
behaviour instead. Both may be present in the same module; `SurfaceProvider`
takes precedence for interactive surface registration.

## SurfaceProvider Behaviour

Add `AllbertAssist.App.SurfaceProvider` as a standalone behaviour module.

Required callbacks:

```elixir
@callback surfaces() :: [AllbertAssist.Surface.t()]
@callback surface_catalog() :: [AllbertAssist.Surface.catalog_entry()]
```

Optional callback with a no-op default:

```elixir
@callback fallback_surface(surface_id :: atom()) ::
            {:ok, String.t()} | {:error, :not_found}
```

Default implementation (added by `use AllbertAssist.App.SurfaceProvider`):

```elixir
def fallback_surface(_surface_id), do: {:error, :not_found}
defoverridable fallback_surface: 1
```

An app may implement both `AllbertAssist.App` and
`AllbertAssist.App.SurfaceProvider` in the same module. `CoreApp` must do this
in v0.18. `StockSage.App` must do the same in v0.20. Surface events that invoke
registered actions still route through `Actions.Runner.run/3`.

`SurfaceProvider` is detected by checking `function_exported?(module, :surfaces,
0)` after confirming the module implements `AllbertAssist.App.SurfaceProvider`
behaviour. The validator must not confuse the legacy `App.surfaces/0` callback
with the `SurfaceProvider.surfaces/0` callback; detection should check
`__behaviours__` or equivalent compile-time metadata.

## Surface DSL

Add `AllbertAssist.Surface` as the local declarative surface substrate.

### Surface Struct

```elixir
defstruct [
  :id,           # atom; compiled from app code only, never from external input
  :app_id,       # atom; must match a registered app
  :label,        # String.t(); 1..64 bytes
  :path,         # String.t(); local route; must start with /; no scheme or host
  :kind,         # atom; :route | :chat | :workspace | :analysis | :canvas | :settings
  :status,       # atom; :available | :placeholder | :disabled
  nodes: [],     # [Node.t()]
  fallback_text: "", # String.t(); 1..512 bytes; shown when rendering cannot use nodes
  metadata: %{}  # map(); bounded; redacted
]
```

### Node Struct

```elixir
defstruct [
  :id,             # String.t(); unique within surface; 1..64 bytes
  :component,      # atom; must be in known_components/0
  props: %{},      # map(); bounded and redacted; restrictions below
  children: [],    # [Node.t()]
  bindings: []     # [ActionBinding.t()]
]
```

### ActionBinding Struct

```elixir
defstruct [
  :action_name,          # String.t(); registered action name; required
  :action_module,        # module() | nil; registered module if known
  :app_id,               # atom() | nil; provenance
  :plugin_id,            # String.t() | nil; provenance
  :permission,           # atom() | nil; from Actions.Registry.capability/1
  :confirmation_required? # boolean()
]
```

### Catalog Entry Type

```elixir
@type catalog_entry :: %{
  component: atom(),          # atom from known_components/0
  allowed_props: [atom()],    # prop keys this app is allowed to pass
  allowed_bindings: [String.t()] # registered action names bindable to this component
}
```

### Public API

- `AllbertAssist.Surface.validate_surface/1` →
  `{:ok, AllbertAssist.Surface.t()} | {:error, [diagnostic()]}`
- `AllbertAssist.Surface.validate_catalog/1` →
  `{:ok, [catalog_entry()]} | {:error, [diagnostic()]}`
- `AllbertAssist.Surface.known_components/0` → `[atom()]`

### Known Components — v0.18 Initial Catalog

`:route`, `:chat`, `:timeline`, `:composer`, `:panel`, `:section`, `:text`,
`:list`, `:empty_state`, `:button`, `:action_button`, `:status_badge`

### Surface Validation Rules

Reject surfaces that:

- Have `path` not starting with `/`, or containing a scheme (`http://`,
  `https://`, `//`), hostname, query string, or fragment.
- Have an unknown `kind` atom.
- Have duplicate node ids within the surface.
- Have nodes whose `component` is not in `known_components/0`.
- Have nodes with secret-like prop keys: keys matching `*_key`, `*_secret`,
  `*_token`, `*_password`, `*_credential`, or `*_api_key` (case-insensitive).
- Have nodes with prop values that are raw HTML strings (values starting with
  `<` and containing `>`).
- Have nodes with prop values that are script strings (values starting with
  `javascript:`, `data:`, or containing `<script`).
- Have nodes with prop values that are remote URLs (values starting with
  `http://`, `https://`, or `//`).
- Exceed reasonable bounded sizes (surface node count > 256, props map size
  > 64 keys, prop string value length > 2048 bytes).

### Action Binding Validation Rules

Reject action bindings that:

- Have no `action_name` or an empty `action_name`.
- Reference an `action_name` not present in `Actions.Registry` (unknown,
  model-invented, or unregistered names).
- Reference an `action_module` that is not a loaded Elixir module.
- Contain any field value that is a remote URL, shell command, script, or
  arbitrary path.
- Reference actions whose capability permission is at a permanently denied
  safety floor (`:command_execute` remains denied; surface bindings cannot
  promote it).

Valid action bindings have their `permission` and `confirmation_required?`
fields populated from `Actions.Registry.capability/1` at validation time.

- `provider_surfaces: [AllbertAssist.Surface.t()]`

```elixir
@spec registered_agents(keyword()) :: [%{app_id: atom(), module: module()}]
@spec registered_signals(keyword()) :: [%{app_id: atom(),
  emits: [String.t()], subscribes: [String.t()]}]
```

## App Validator Expansion

`AllbertAssist.App.Validator.validate/2` must expand to validate:

- `agents/0`: must return a list; each element must be a loaded atom
  (`Code.ensure_loaded?/1`).
- `signals/0`: must return `%{emits: [String.t()], subscribes: [String.t()]}`.
  Each topic string must be a non-empty, dot-namespaced string (e.g.,
  `"allbert.runtime.turn.completed"`). Topic strings must not contain spaces,
  remote URLs, or arbitrary format strings.
- `settings_schema/0`: must return a list of schema entry maps. Each entry
  must have a `key` beginning with `apps.<app_id>.`; a `type` that is a known
  Settings Central atom; and an optional `default` that is JSON-safe.
  Entries with wrong key prefix are rejected.
- `SurfaceProvider` surfaces: if `module` implements
  `AllbertAssist.App.SurfaceProvider`, call `SurfaceProvider.surfaces/0` and
  validate each through `AllbertAssist.Surface.validate_surface/1`. Each
  surface must have `app_id` matching the app's `app_id/0`.
- `SurfaceProvider` catalog: call `SurfaceProvider.surface_catalog/0` and
  validate each entry references only components from
  `AllbertAssist.Surface.known_components/0`.

The v0.18 validator result map expands to include:

```elixir
%{
  app_id: atom(),
  module: module(),
  display_name: String.t(),
  version: String.t(),
  agents: [module()],
  actions: [module()],
  signals: map(),
  skill_paths: [Path.t()],
  settings_schema: [map()],
  surfaces: [map()],             # legacy v0.15 navigation summaries
  surface_provider: module() | nil,
  provider_surfaces: [AllbertAssist.Surface.t()],
  surface_catalog: [map()]
}
```

Backward compatibility: apps that only implement the v0.15 minimal contract
(without agents, signals, settings_schema, or SurfaceProvider) still validate
with empty defaults for all new fields.

## Settings Schema Merge

This is the closure of the v0.17 gap where plugin and app settings schema
contributions are stored in their registries but never consumed by
`AllbertAssist.Settings.Schema`.

Add `AllbertAssist.Settings.Schema.runtime_schema/0`:

```elixir
@spec runtime_schema() :: [entry()]
def runtime_schema do
  static_schema = @schema

  app_entries = safe_registered_settings(:app)
  plugin_entries = safe_registered_settings(:plugin)

  merge_schema(static_schema, app_entries, plugin_entries)
end
```

Merge rules:

1. Static compiled `@schema` always present; its keys always win conflicts.
2. App registry schema entries override plugin registry entries for the same
   key.
3. Plugin registry entries fill remaining gaps.
4. App schema entry keys must start with `apps.<app_id>.`; entries with wrong
   prefixes are excluded with a log warning.
5. Plugin schema entry keys must start with `plugins.<plugin_id>.`; entries
   with wrong prefixes are excluded with a log warning.

`safe_registered_settings/1` catches any exception from a registry call and
returns `[]` with a log warning when the registry is unavailable (crashed, not
yet started, or in a restart cycle). Settings reads during the core boot
sequence before registries are available degrade gracefully to the static
schema.

Update `AllbertAssist.Settings.Schema.validate/2` and
`AllbertAssist.Settings.Schema.get/2` to call `runtime_schema/0` in place of
the static `@schema` reference.

## Active App Default Resolution

Add a single private function in `AllbertAssist.Runtime` responsible for
resolving `active_app` context for all request entry points. Apply it to
`submit_user_input/1`, job turn handlers, and channel adapter turn handlers.

Resolution order:

1. If `request.active_app` is a known atom (validated through
   `App.Registry.known_app_id?/1`), use it as-is.
2. If `request.session_id` is present, read the v0.14 scratchpad for that
   user and session. If the scratchpad has a known `active_app`, use it.
3. If no known app context exists, default to `:allbert`.
4. If `request.active_app` is a non-nil but unknown string or atom, fall back
   to `:allbert` and add an `:unknown_app_id` diagnostic to the response. Do
   not atomize unknown strings.

String-to-atom conversion uses only `App.Registry.normalize_app_id/1`, which
internally uses `String.to_existing_atom` guarded by a registry existence
check. No `String.to_atom/1` call may appear in the active_app resolution path.

The resolved `active_app` must be present in:

- Input signal metadata (`data.active_app`)
- Runtime response metadata
- Trace metadata
- Conversation message metadata (both request and response turns)

The resolved `active_app` is context. It must not change which permissions
apply, which actions are accessible, or whether confirmation is required.

## Signal Declarations

App signal declarations are metadata for routing and future subscriptions.

v0.18 validates the shape and stores it in the registry. It does not add a
new signal bus, automatic subscription runtime, or dynamic topic routing.
Existing Jido signal flow remains the only runtime signal path.

Signal topic format:

- Must be a dot-namespaced string: `<namespace>.<resource>.<event>`.
- Examples: `"stocksage.analysis.started"`,
  `"allbert.runtime.turn.completed"`.
- Must not contain spaces, slashes, remote URL segments, or format strings.
- Length: 1..128 bytes per topic string.

`AllbertAssist.App.CoreApp` becomes the first `SurfaceProvider`.

It must declare one initial surface:

```elixir
%AllbertAssist.Surface{
```

The existing Phoenix `/agent` route in `AllbertAssistWeb.Router` remains the

the same app/surface contract that future apps must use.

`surface_catalog/0` for CoreApp must return at minimum catalog entries for
`:chat`, `:timeline`, and `:composer`.

## mix allbert.validate_app

Add `Mix.Tasks.Allbert.ValidateApp` implementing `mix allbert.validate_app MODULE`.

Behavior:

- Parse MODULE as a module name string. Resolve it to an atom only using
  `Module.concat/1` on a list of string segments; never use `String.to_atom/1`.
  Check `Code.ensure_loaded?/1` before proceeding.
- Call `AllbertAssist.App.Validator.validate/2` on the resolved module.
- If the module also implements `AllbertAssist.App.SurfaceProvider`, validate
  surfaces through `AllbertAssist.Surface.validate_surface/1`.
- Print: app_id, display_name, version, action count, skill path count, agent
  count, settings schema entry count, signal emission count, signal subscription
  count, surface ids and paths (from both legacy surfaces and provider surfaces).
- Print diagnostics without secrets, raw node data, raw provider payloads, or
  raw model output.
- Exit non-zero when the app is invalid.

Existing `mix allbert.apps validate MODULE` may delegate to the same validation
engine. Do not create two divergent validators.

## Non-Goals

- No app or plugin generator. That is v0.31 (formerly v0.29).
- No memory namespace registration in `AllbertAssist.App.Registry`. That is
  v0.29 (formerly v0.27).
- No canvas implementation. That is v0.26 (formerly v0.24).
- No workspace shell replacement for `/agent`. v0.18 declares the current
  route; v0.24 upgrades its surface implementation.
- No permission grants from app registration, surface registration, plugin
  registration, action bindings, or `active_app` context.
- No dynamic route loading from app metadata.
- No runtime compilation or loading of arbitrary app or plugin modules.
- No AG-UI or A2UI package dependency for the local web path.
- No AG-UI protocol emission, streaming endpoint, or wire event.
- No app-scoped memory promotion or semantic retrieval.
- No hosted accounts, teams, roles, or per-user app administration.
- No `mix allbert.gen.app` or `mix allbert.gen.plugin` scaffolding.
