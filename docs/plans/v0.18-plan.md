## Purpose

v0.18 matures the v0.15 minimal app registration contract into the full local
app/surface contract, except for memory namespace registration. Memory
namespace registration remains explicitly deferred to v0.27, where StockSage
polish first consumes it.

This milestone adds `AllbertAssist.App.SurfaceProvider` and
`AllbertAssist.Surface`, then upgrades `AllbertAssist.App.CoreApp` into the
first real `SurfaceProvider`. `CoreApp` declares the existing `/agent`
conversation route as the built-in chat surface. That makes the current
conversation surface the first proven app/surface implementation before
StockSage v0.20 becomes the second app to implement the same contract.

Starting in v0.18, runtime requests that arrive without an explicit app context
resolve to `active_app: :allbert`. This default is context, not authorization.
It gives every runtime turn a declared home app while preserving the v0.14
scratchpad behavior for explicit app context.

v0.24 later upgrades `CoreApp`'s declared `/agent` surface into the
signal-driven workspace shell. The workspace shell is `CoreApp`'s surface
implementation; it is not a separate UI system floating outside the app
contract.

## Current State

- v0.15 provides `AllbertAssist.App`, `AllbertAssist.App.Registry`, app
  validation, app child supervision, app-contributed actions, app skill paths,
  static navigation descriptors, and `mix allbert.apps`.
- v0.17 provides `AllbertAssist.Plugin`, `AllbertAssist.Plugin.Registry`,
  plugin-contributed apps/actions/skills/settings metadata, plugin child
  supervision, and shipped source-tree channel plugins.
- `AllbertAssist.App.CoreApp` exists but only declares v0.15 identity.
- `AllbertAssist.App.StockSageStub` exists as a transitional app id holder.
  v0.20 replaces it with the real `StockSage.App` through the StockSage plugin.
- `AllbertAssist.Session.AppId` validates app ids through the app registry
  without creating atoms from external input.
- Runtime, channel, job, trace, and conversation metadata already carry
  optional `active_app`; v0.18 defines the defaulting rule and makes the
  surface contract visible to users.

## Architecture Decisions

### App Contract Layers

`AllbertAssist.App` expands from the v0.15 minimal contract into these layers:

- Identity and OTP lifecycle: `app_id/0`, `display_name/0`, `version/0`,
  `validate/1`, and `child_spec/1`.
- Agents, actions, and signals: `agents/0`, `actions/0`, `signals/0`, app
  action tagging, and app signal declarations.
- Skills: `skill_paths/0`, still discovered through the existing skill
  registry.
- UI surfaces: `surfaces/0` as the v0.15 compatibility summary and
  `AllbertAssist.App.SurfaceProvider` as the v0.18 canonical interactive
  surface contract.
- Data and settings: `settings_schema/0` for app-owned Settings Central schema
  declarations. Memory namespaces are deferred to v0.27 and must not be added
  in this milestone.

Defaults preserve existing apps:

- `agents/0` defaults to `[]`.
- `signals/0` defaults to `%{emits: [], subscribes: []}`.
- `settings_schema/0` defaults to `[]`.
- `surface_provider/0` is not required. Apps that implement
  `AllbertAssist.App.SurfaceProvider` are detected and validated through that
  behaviour.
- Existing `surfaces/0` remains supported as a legacy navigation summary, but
  new interactive surfaces should come from `SurfaceProvider`.

### Registry Responsibilities

`AllbertAssist.App.Registry` remains the app discovery point. It must not
become a duplicate plugin registry.

The registry stores normalized, bounded app entries:

- app id, display name, version, module, child id/pid, registered timestamp
- action modules and action names
- agent modules
- skill paths
- signal declarations
- settings schema declarations
- legacy navigation summaries
- validated SurfaceProvider surfaces
- diagnostics

Registry additions:

- `registered_agents/0`
- `registered_signals/0`
- `registered_settings_schema/0`
- `registered_surfaces/0` returns validated surface summaries from both
  `SurfaceProvider` and v0.15 legacy summaries, with SurfaceProvider taking
  precedence for the same app/surface id.
- `surface_for/2` or equivalent lookup by `app_id` and surface id/path.
- diagnostics for invalid settings schema, invalid signals, invalid surface
  providers, duplicate surface ids within an app, and cross-app duplicate route
  paths.

Cross-app duplicate surface ids or route paths should be diagnostics, not boot
crashes, unless the duplicate is inside the same app. Same-app duplicate ids
are validation failures because an app cannot address the surface
unambiguously.

### SurfaceProvider Contract

Add `AllbertAssist.App.SurfaceProvider`.

Required callbacks:

- `surfaces/0`: returns the app's declared interactive surfaces.
- `surface_catalog/0`: returns the component catalog entries the app is
  allowed to use.

Optional callback with default:

- `fallback_surface/1`: returns text-only fallback data for a surface id or
  route when the renderer cannot render a node.

An app may implement both `AllbertAssist.App` and
`AllbertAssist.App.SurfaceProvider` in the same module. `CoreApp` should do
this in v0.18. StockSage v0.20 should follow the same pattern.

SurfaceProvider is declarative. It does not mount routes, start LiveViews,
grant permissions, or execute actions. Routes may still be implemented by
reviewed Phoenix modules in the source tree, but the app must declare those
surfaces through the provider contract so the registry, future workspace shell,
and user-facing validation agree on what exists.

### Surface DSL

Add `AllbertAssist.Surface` as the native local DSL and validation layer.

The implementation may use structs or normalized maps, but it must expose a
small, documented public API:

- `AllbertAssist.Surface.surface/1`
- `AllbertAssist.Surface.node/1`
- `AllbertAssist.Surface.action_binding/1`
- `AllbertAssist.Surface.validate_surface/1`
- `AllbertAssist.Surface.validate_catalog/1`
- `AllbertAssist.Surface.known_components/0`

Surface fields:

- `id`: atom or bounded string normalized to atom only when already known from
  compiled app code, not from user/model/channel input.
- `app_id`: atom matching the registered app.
- `label`: bounded string.
- `path`: local route path beginning with `/`; no scheme, host, or remote URL.
- `kind`: atom such as `:route`, `:chat`, `:workspace`, `:analysis`,
  `:canvas`, or `:settings`.
- `status`: `:available`, `:placeholder`, or `:disabled`.
- `nodes`: list of validated nodes.
- `fallback_text`: bounded string shown when rendering cannot use nodes.
- `metadata`: bounded, redacted map.

Node fields:

- `id`: bounded string unique within its surface.
- `component`: atom from the known catalog.
- `props`: bounded redacted map. Props must not contain raw HTML, JavaScript,
  scripts, secrets, credential refs, arbitrary external URLs, or unbounded
  model output.
- `children`: list of validated child nodes.
- `bindings`: list of validated action bindings.

Initial known components:

- `:route`
- `:chat`
- `:timeline`
- `:composer`
- `:panel`
- `:section`
- `:text`
- `:list`
- `:empty_state`
- `:button`
- `:action_button`
- `:status_badge`

Action bindings:

- reference registered action names or modules only
- include app id and plugin id provenance when known
- include permission metadata from `Actions.Registry.capability/1`
- cannot grant permission, change confirmation requirements, or bypass
  Security Central
- are rejected when they reference unknown actions, unregistered modules,
  model-invented action names, remote URLs, scripts, shell commands, or secret
  fields

### CoreApp Surface

`AllbertAssist.App.CoreApp` becomes the first `SurfaceProvider`.

It declares one initial surface:

- `id: :agent`
- `app_id: :allbert`
- `label: "Allbert Chat"`
- `path: "/agent"`
- `kind: :chat`
- `status: :available`
- nodes: at minimum a `:chat` root containing a `:timeline` node and a
  `:composer` node
- fallback text: short text explaining that the Allbert chat surface is
  available at `/agent`

The existing `/agent` route remains a reviewed Phoenix route. v0.18 does not
replace it with a generated UI. The important change is that the route is now
declared through the same app/surface contract future apps must use.

### Active App Default

Runtime request context resolution becomes:

1. If request context includes a known explicit `active_app`, preserve it.
2. If request context includes `session_id`, read v0.14 scratchpad active app.
   If it is known, use it.
3. If no known app context exists, default to `:allbert`.
4. If the request includes an unknown app id, keep the request safe by falling
   back to `:allbert` and record diagnostics. Do not create atoms from strings.

This default applies to runtime turns from CLI, LiveView, scheduled jobs,
Telegram/email channels, and future plugin channels. It must be visible in
runtime response metadata, input/response signals, trace metadata, and
conversation message metadata. It must not authorize an action.

### Settings Schema

App settings schema declarations are metadata for Settings Central.

Required schema fields:

- `key`: bounded string beginning with `apps.<app_id>.`
- `type`: atom from Settings Central's existing type vocabulary, or a new
  bounded app schema type added in this milestone
- `default`: optional JSON/YAML-safe value
- `description`: optional bounded string
- `secret?`: optional boolean; secret values still go through Settings Secrets

v0.18 should merge registered app settings schema declarations into Settings
Central read/validation paths when the app registry is available. If the app
registry is unavailable, settings reads should fail closed or report clear
diagnostics rather than crash.

### Signal Declarations

App signal declarations are metadata for routing and future subscriptions.

Shape:

```elixir
%{
  emits: ["stocksage.analysis.started"],
  subscribes: ["allbert.runtime.turn.completed"]
}
```

v0.18 validates the shape and stores it. It does not add a new signal bus or
automatic subscription runtime. Existing Jido signal flow remains the only
runtime signal path.

### AG-UI/A2UI Stub

Add `AllbertAssist.Surface.Encoder.to_a2ui/1`.

The function must be typed and documented as the future translation boundary
from `AllbertAssist.Surface` validated nodes to AG-UI/A2UI-style state
snapshots. In v0.18 it may return `{:error, :not_implemented}` or equivalent.
It must not emit protocol wire events, add an AG-UI dependency, or participate
in the Phoenix LiveView path.

### CLI And Docs

Add:

- `mix allbert.validate_app MODULE`
- `docs/how-to-create-an-allbert-app.md`
- `docs/plans/v0.18-request-flow.md`

`mix allbert.validate_app` should:

- resolve only loaded/compiled modules; do not create atoms from arbitrary
  operator input
- validate `AllbertAssist.App`
- validate `SurfaceProvider` when implemented
- print app identity, action count, skill path count, settings schema count,
  signal declarations, and surface ids/paths
- print diagnostics without secrets or raw provider payloads
- exit non-zero for invalid apps

Existing `mix allbert.apps validate MODULE` may delegate to the same validation
engine or remain as a compatibility alias. Avoid two divergent validators.

## User Acceptance

A local operator should be able to verify v0.18 without writing custom code:

- `mix allbert.validate_app AllbertAssist.App.CoreApp` passes and prints the
  `/agent` surface.
- `mix allbert.apps show allbert` shows the `/agent` surface and does not show
  raw internals, secrets, provider payloads, or arbitrary node maps.
- `mix allbert.ask "hello"` returns a normal response whose metadata/trace
  includes `active_app: :allbert`.
- `mix allbert.ask --session SESSION "hello"` preserves an explicitly set
  scratchpad app when known, and otherwise defaults to `:allbert`.
- `mix allbert.channels telegram simulate ...` and
  `mix allbert.channels email simulate ...` still work and record
  `active_app: :allbert` when no app context is present.
- A deliberately invalid app fixture fails `mix allbert.validate_app` with a
  clear diagnostic.
- `AllbertAssist.Surface.Encoder.to_a2ui/1` exists but reports that concrete
  protocol emission is not implemented.

## Non-Goals

- No app/plugin generator; that is v0.29.
- No memory namespace registration in `AllbertAssist.App.Registry`; that is
  v0.27.
- No canvas implementation; that is v0.24.
- No workspace shell replacement for `/agent`; v0.18 declares the current
  route, v0.24 upgrades it.
- No permission grants from app registration, surface registration, plugin
  registration, action bindings, or active app context.
- No dynamic route loading from app metadata.
- No runtime compilation or loading of arbitrary app/plugin modules.
- No AG-UI/A2UI package dependency for the local web path.
- No AG-UI protocol emission or streaming endpoint.
- No app-scoped memory promotion or semantic retrieval.
- No hosted accounts, teams, roles, or per-user app administration.

## Definition Of Done

v0.18 is done when the app/surface contract is implemented through all layers
except memory namespace registration; `CoreApp` implements
`SurfaceProvider`; `/agent` is declared as the built-in `:allbert` chat
surface; runtime requests default to `active_app: :allbert` when no known app
context exists; surface validation rejects unknown or unsafe nodes/action
bindings; `mix allbert.validate_app` validates conforming and invalid apps;
`docs/how-to-create-an-allbert-app.md` exists; ADR 0015 and roadmap docs are
current; user acceptance checks pass; and v0.20 StockSage can implement the
same contract from day one without a v0.15-to-v0.18 migration.
