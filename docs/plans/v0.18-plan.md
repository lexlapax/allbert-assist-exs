# Allbert v0.18 Memory Review And Retrieval

## Purpose

v0.18 improves memory after security, execution, imports, jobs, and channels
exist, so review, correction, promotion, and retrieval are grounded in how
Allbert is actually used across real surfaces.

Markdown memory remains the inspectable source of truth. Summaries, compiled
views, and retrieval indexes are derived artifacts that must be rebuildable.

This milestone was formerly tracked as v0.14 before the post-v0.10 roadmap
was unified into one v0.xx sequence.

## Expected Inputs

- v0.02 Settings Central exists for memory review cadence, sensitivity policy,
  and retention preferences.
- v0.05 Security Central can label risk, redaction, trust boundary, and audit
  metadata.
- v0.08 through v0.10 execution and import traces record concrete side effects.
- v0.13 jobs can schedule review or summary workflows.
- v0.16 channels can contribute channel-tagged memory candidates.
- v0.12 conversation history provides SQLite `Thread` and `Message` records,
  surfaced through `AllbertAssist.Conversations` and `mix allbert.threads`.
  These remain distinct from markdown long-term memory.
- Runtime traces can identify memory effects and selected skill/action context.

## Expected Direction

- Add memory review, correction, promotion, and pruning workflows.
- Add summaries and compiled runtime views over markdown memory.
- Introduce embeddings or retrieval only after the markdown source of truth and
  review path are stable across CLI, LiveView, execution, imports, scheduled
  jobs, and additional channels.
- Use settings for memory review cadence, sensitivity policy, and promotion
  preferences.
- Make derived memory artifacts rebuildable and clearly marked as derived.

## Two Memory Tiers

By v0.18, Allbert may have two distinct memory-adjacent stores:

- Markdown long-term memory under `<ALLBERT_HOME>/memory/`, the existing
  inspectable and operator-edited source of truth for durable memory.
- SQLite conversation history from v0.12, represented by `Thread` and
  `Message` tables for structured, ordered, paginated turn records.

These are not the same artifact. v0.18 review workflows apply to markdown
memory. Conversation history is not automatically promoted to markdown
entries. Selective promotion, such as extracting a lesson from a thread into a
memory entry, is an explicit confirmed action rather than an automatic
pipeline.

## v0.12 Handoff

v0.18 memory review may read v0.12 thread/message records as source material
for operator-selected promotion, but it must keep promotion explicit and
confirmed. The existence of persisted conversation turns is not consent to
create markdown memories.

## Non-Goals

- No embeddings or vector database as the first implementation step.
- No opaque memory mutation without markdown auditability.
- No deleting user-owned markdown memory without explicit operator action.
- No channel-specific memory stores.
- No automatic promotion of sensitive entries.
- No automatic promotion of conversation history turns to markdown memory
  entries.

## Definition Of Done

v0.18 is done when:

- Operators can review, correct, promote, and prune memory entries.
- Summaries and compiled runtime views are generated from markdown sources.
- Derived memory artifacts are rebuildable and clearly marked as derived.
- Retrieval improves recall without replacing the markdown source of truth.
- Sensitive memory promotion follows settings-backed policy and audit records.
