# Allbert v0.22 StockSage LiveViews

Formerly tracked as M-D3a.

## Purpose

v0.22 adds a standard Phoenix LiveView surface for StockSage inside the Allbert
shell. It is not canvas work; it proves the app surface as ordinary LiveViews
before v0.26 canvas consumes the full surface contract.

## Expected Inputs

- v0.15 app registration exists.
- v0.19 Python bridge provides real analyses.
- v0.14 session scratchpad provides `active_app`.

## Expected Direction

- Add `StockSageLive.WorkspaceLive`, `AnalysisLive`, `QueueLive`, and
  `TrendsLive`.
- Mount `/stocksage/...` routes statically in the Allbert web router while
  using registered app surfaces for navigation.
- Stream analysis progress through Phoenix.PubSub and LiveView `stream/3`.
- Set `active_app: :stocksage` in scratchpad when navigating under
  `/stocksage/`.
- Keep LiveViews thin: domain behavior stays behind StockSage contexts,
  registered actions, and runtime boundaries.

## Non-Goals

- No canvas component registration.
- No `canvas_ops`.
- No PostgreSQL or Oban dependency.

## Test Plan

- Full browser analysis cycle from enqueue to completed analysis.
- Live progress updates without polling/page refresh.
- Trends page loads from SQLite records.
- Route navigation sets `active_app: :stocksage`.

## Definition Of Done

v0.22 is done when StockSage has usable plain LiveViews for analysis, queue,
and trends, with real-time progress and app-context integration.
