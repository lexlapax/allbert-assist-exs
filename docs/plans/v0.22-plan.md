## Purpose

v0.22 brings real analysis results into Allbert through a supervised Python
bridge around the existing TradingAgents baseline. The bridge is transitional:
it proves the execution path that later native Jido agents (now v0.25 after
the project-direction rethink renumbering, formerly v0.23) will reuse
without changing the action, permission, confirmation, or result-persistence
contracts.

By the end of v0.22, a user can ask Allbert to analyze a stock, receive a
confirmation request, approve it, and see a real analysis result stored in
StockSage's local domain records and queryable through existing list/show
actions. This is the first end-to-end StockSage user loop.

## M2 Reopened — Audit Closeout (2026-05-16)

An operator audit against the M1–M5 implementation found that v0.22 did not
meet several plan-level acceptance promises. Five gaps were identified;
this section records the operator-confirmed disposition of each and the
implementation choices made.

### Audit gaps and dispositions

| # | Gap (audit finding) | Disposition | Implementation choice |
|---|---|---|---|
| 1 | `bridge.py` used `run_tradingagents_stub`; plan promised real TradingAgents integration in M2 | **Finish M2 before tag** | Real `TradingAgentsGraph.propagate` call. Stub path retained behind a `force_stub: true` flag in the bridge protocol so tests and credential-less dev environments don't need LLM/market-data credentials. Loud failure if `tradingagents` cannot be imported and `force_stub` is not set: returns `tradingagents_unavailable` error rather than silently stubbing. |
| 2 | Natural-language `analyze AAPL` with `active_app: :stocksage` ranks `run_analysis` as a candidate but does not select it; selected action is `direct_answer` | **Fix the ranker** | Extend the existing scorer in `AllbertAssist.Intent.Ranker` with an active-app boost. Deterministic scoring per ADR 0019; no new app-scoped predicate before the engine. Focused selection test added (not just candidate-presence). |
| 3 | Invalid `queue_entry_id` still writes an analysis row because `update_queue/5` silently no-ops when `Queue.get_entry/2` fails | **Bug fix** | Validate the queue entry exists and is consumable (`status in ["queued", "running"]`) before the bridge call. Mismatched `user_id` is treated as not-found to preserve cross-user isolation. Bridge is not invoked and no analysis row is written when validation fails. |
| 4 | `allbert.stocksage.analysis_requested`/`analysis_completed`/`analysis_failed` are named in the plan but not emitted | **Emit in v0.22** | Named signals fire at three lifecycle points in `RunAnalysis`: requested (after queue validation, before bridge call), completed (after success persistence), failed (after failure persistence or queue-validation rejection). Built via `Jido.Signal.new/3`, redacted via `AllbertAssist.Signals.redact/1`, logged via `AllbertAssist.Signals.log/1` so they reach the same log path as `allbert.action.*` lifecycle signals. |
| 5 | `stocksage.bridge_max_output_bytes` (default 1 MiB) is named as the bound for persisted detail, but `run_analysis.ex` hard-codes a 16_000 byte cap | **Unify on the setting** | Persisted detail row body is bounded by `stocksage.bridge_max_output_bytes` end-to-end. The 16_000 hard cap is removed. Operators who want a tighter trace/redaction posture lower the setting. |

### Moderate-gap dispositions

- **`approval_resume?/1` trusts a context flag.** Documented as
  internal-API-only in `RunAnalysis`. Hardening (signed resume marker or
  caller verification) deferred to v0.23 Jido State-Machine Convergence
  when `Confirmations.Store` becomes a `Jido.Agent` and gains lifecycle
  hooks suitable for verified-resume gating.
- **Fresh `ALLBERT_HOME` smoke fails with `no such table: scheduled_job_runs`.**
  Smoke instructions in `v0.22-request-flow.md` add the explicit
  `mix cmd --app allbert_assist mix ecto.migrate.allbert` (or equivalent)
  step before running the bridge or job flows.
- **CLI `--queue-id` negative-path test missing.** Added alongside the
  Gap 3 fix.
- **Bridge crash test doesn't prove in-flight caller gets `{:error,
  :bridge_crashed}`.** Test added that opens a bridge, issues a long
  call, simulates a port crash mid-flight, and asserts the pending caller
  receives `{:error, :bridge_crashed}` (not just port-recovery on next
  call).
- **Trace redaction coverage narrow.** Test added that runs a real-shape
  `RunAnalysis` result with secret-like and oversized fields through the
  trace path, asserting no leakage.

### Implementation choices recorded for posterity

The operator selected the recommended option for each of four design
questions during audit closeout:

1. **Stub fallback location:** `force_stub: true` flag in request payload.
   No env-var gates; no auto-detect of missing install.
2. **Bridge timeout default:** `stocksage.bridge_timeout_ms` raised from
   `300_000` to `600_000` (10 minutes) to tolerate real TradingAgents
   propagate runtimes. Operators can lower it for stub-mode tests via
   Settings Central.
3. **Intent ranker approach:** Extend existing scorer with an active-app
   boost rather than adding a new app-scoped predicate. Preserves the
   ranking-based ADR 0019 posture.
4. **Output cap unification:** Make `stocksage.bridge_max_output_bytes`
   govern persisted detail too (vs. the rejected option of documenting
   the lower 16_000 cap as a separate bound, or lowering the setting
   default).

### Operator Setup For TradingAgents

The real TradingAgents path requires three operator-managed pieces. The
stub path (tests, dev) requires none of them.

1. **Python venv with `tradingagents` installed from a pinned GitHub
   release.** v0.22 ships pinned to upstream **v0.2.5** (May 11, 2026 —
   "Grounded Sentiment Analyst, Qwen/GLM/MiniMax dual-region support, and
   ticker path-traversal fix") via
   `plugins/stocksage/priv/python/requirements.txt`. Setup:

   ```sh
   python3 -m venv ~/.venvs/allbert-stocksage
   ~/.venvs/allbert-stocksage/bin/python -m pip install --upgrade pip
   ~/.venvs/allbert-stocksage/bin/pip install -r \
     plugins/stocksage/priv/python/requirements.txt
   ```

   Then point Allbert at the venv's Python:

   ```sh
   mix allbert.settings set stocksage.python_path \
     $HOME/.venvs/allbert-stocksage/bin/python
   ```

   Bumping the upstream version is an operator-visible decision: edit the
   git tag in `requirements.txt`, re-install in the venv, and verify
   with a single-stock real-bridge run before tagging the new pin into
   git. v0.22 audit closeout deliberately rejected the "local checkout"
   pattern so production installs are reproducible from `requirements.txt`
   alone.

2. **LLM credentials in that venv's environment.** TradingAgents uses
   LangChain and calls OpenAI (or another provider per `llm_provider` in
   its config). The venv environment must have `OPENAI_API_KEY` (or the
   equivalent for the configured provider) available when the Port is
   spawned. The bridge does not inject keys; it inherits the operator's
   environment.
3. **Market-data API access.** TradingAgents defaults to `yfinance` for
   data vendors, which needs no API key for basic data. Operators can
   override via the bridge protocol's `config` field (allowed override
   keys: `deep_think_llm`, `quick_think_llm`, `max_debate_rounds`,
   `max_risk_discuss_rounds`, `max_recur_limit`, `online_tools`,
   `data_vendors`, `output_language`, `checkpoint_enabled`).

A real `propagate` call routinely takes 2–10 minutes (depending on
debate-round count and model). The raised `bridge_timeout_ms` default
(600_000 ms) tolerates this; operators running stub-mode smoke flows
should lower the timeout temporarily or accept the long default. The
bridge protocol returns `tradingagents_unavailable` (specifically
`tradingagents_import_failed: ...`) when `tradingagents` cannot be
imported and `force_stub` is not set, so missing-install gaps surface
loudly on the first non-stub call rather than silently degrading.

## Handoff Note For The Project-Direction Rethink

v0.22 is a single-action execution path. `RunAnalysis` runs one analysis
per confirmation; there is no private objective loop, no per-analysis
queue across confirmations, and no shared work state across turns. That
shape is correct for v0.22 and ships unchanged.

The project-direction rethink
(`docs/plans/project-direction-rethink-01.md`) inserts two new milestones
ahead of native trading agents:

- v0.23 Jido State-Machine Convergence converts
  `AllbertAssist.Confirmations.Store` and `AllbertAssist.Jobs.Scheduler`
  to `Jido.Agent`. v0.22's confirmation flow continues to behave
  identically; only the implementation substrate of `Confirmations.Store`
  changes.
- v0.24 Objective Runtime Foundation adds the durable multi-step work
  substrate. `StockSage.Actions.RunAnalysis` gains optional
  `objective_id`/`step_id` parameters, threaded through confirmations,
  audit, trace, and the `stocksage_analyses` row. v0.22 invocations
  without those parameters continue to work as today (both columns
  are NULL).

v0.22 itself does not change to accommodate v0.23 or v0.24. The handoff
is that future StockSage analyses (v0.24+) can be invoked as objective
steps; existing v0.22 direct invocations remain valid.

## Architecture Readiness Audit

The v0.22 plan is consistent with the current architecture:

- ADR 0006 keeps Security Central as the permission and risk boundary. A new
  `:stocksage_analyze` permission class covers bridge execution. The safety
  floor is `:needs_confirmation`; Settings Central cannot lower it. Risk tier
  is `high`.
- ADR 0007 requires runtime-facing, effectful operations to enter through
  registered Jido actions. `StockSage.Actions.RunAnalysis` is the registered
  action boundary. `StockSage.TraderBridge` is an internal plugin-owned
  implementation detail behind that boundary.
- ADR 0008 defines durable confirmation requests. `RunAnalysis` creates a
  pending confirmation record when Security Central returns `:needs_confirmation`
  and does not invoke the bridge until approved. The v0.07 `approve_confirmation`
  workflow resumes the action after re-checking policy.
- ADR 0017 defines plugins as contribution packages. All bridge code —
  `StockSage.TraderBridge`, `StockSage.Bridge.Protocol`, `bridge.py`, and
  `StockSage.Actions.RunAnalysis` — lives under `./plugins/stocksage/`. Allbert
  core does not import bridge internals directly.
- ADR 0018 places `StockSage.Actions.RunAnalysis` in the Python bridge
  milestone. `:stocksage_write` authorizes local domain writes only; it does
  not authorize bridge execution. This plan adds `:stocksage_analyze` as the
  distinct execution permission.
- ADR 0019 governs intent engine candidates. When `active_app: :stocksage`,
  RunAnalysis is a candidate scored through the standard engine; no new
  StockSage-specific core predicates are added.
- ADR 0020 (proposed in M1) defines the bridge technology, JSON-over-stdio
  protocol envelope, plugin-ownership boundary, `:stocksage_analyze` permission
  class, and the v0.28 (formerly v0.26) handoff for market-data API call
  hardening.

No existing ADR needs revision for v0.22 if implementation follows ADR 0018
and ADR 0020. ADR 0018 already anticipates `RunAnalysis` in this milestone.

native Jido trading agents become default only after parity tests prove they

## Bridge Technology And Protocol

The bridge uses a JSON-over-stdio Erlang Port. See ADR 0020 for the full
technology choice rationale.

### Protocol Envelope

Request (newline-delimited JSON object):

```
{"id": "<uuid>", "action": "run_analysis", "ticker": "AAPL", "analysis_date": "2026-05-01", "engine": "tradingagents", "max_output_bytes": 1048576, "force_stub": false, "config": {}}
```

`max_output_bytes`, `force_stub`, and `config` are optional. They are added
by the M2 audit closeout:

- `max_output_bytes` (int, default 1 MiB): hard cap on the bounded `summary`
  and serialized `raw` fields in the response. Set from
  `stocksage.bridge_max_output_bytes`.
- `force_stub` (bool, default false): when `true`, the bridge uses the
  deterministic stub path regardless of whether `tradingagents` is
  importable. Tests and credential-less smoke flows pass this; production
  callers do not.
- `config` (object, default `{}`): bounded TradingAgents config overrides.
  Allowed keys: `deep_think_llm`, `quick_think_llm`, `max_debate_rounds`,
  `max_risk_discuss_rounds`, `max_recur_limit`, `online_tools`,
  `data_vendors`, `output_language`, `checkpoint_enabled`. Other keys are
  silently dropped.

Response on success:

```
{"id": "<uuid>", "status": "ok", "result": { <analysis fields including stub: true|false> }}
```

The `stub` field in the result is `true` when the response came from the
stub path (either `force_stub: true` or the historical pre-M2-closeout
stub-only mode) and `false` for real TradingAgents responses. The Elixir
side persists this in `stocksage_analysis_details.payload.stub` so
operator inspection of an analysis row makes the source obvious.

Response on error:

```
{"id": "<uuid>", "status": "error", "reason": "<bounded string, max 500 chars>"}
```

Specific error reasons added in the M2 audit closeout:

- `tradingagents_import_failed: <ExceptionClass>: <message>` — the
  `tradingagents` Python package could not be imported in the bridge
  venv. Returned when `force_stub` is not set; the bridge does not
  silently degrade to stub mode.
- `tradingagents_error: <message>` — `TradingAgentsGraph.propagate`
  raised. The exception class and bounded message are surfaced;
  TradingAgents stack traces are not included in the response.

Ping/pong for health check:

```
{"id": "<uuid>", "action": "ping"}
{"id": "<uuid>", "status": "ok", "result": "pong"}
```

Rules enforced by `StockSage.Bridge.Protocol`:

- `id` is always required and echoed back unchanged.
- `action` controls dispatch; unknown actions return an error response.
- Missing required params return an error without dispatching to TradingAgents.
- The protocol rejects any request that lacks a recognized `action` field.
- Response bodies from TradingAgents are bounded to `stocksage.bridge_max_output_bytes`
  before serialization; values exceeding the limit are truncated and the
  response includes `"truncated": true`. The same bound is used for the
  persisted detail row body in `stocksage_analysis_details.content`
  (M2 audit closeout — gap 5).

### Python Bridge Script

Location: `./plugins/stocksage/priv/python/bridge.py`

Contract:

- Reads newline-delimited JSON from `stdin`.
- Writes newline-delimited JSON to `stdout` (flushed immediately).
- `action: "ping"` responds `{id, status: "ok", result: "pong"}`.
- `action: "run_analysis"` dispatches to TradingAgents, returns result or error.
- Unknown or malformed input returns a bounded error response; does not crash.
- All output to `stderr` is diagnostic only; it does not carry protocol data.
- Does not accept shell strings, file paths, module names, or arbitrary code.
- Does not write to any file path other than through TradingAgents' standard
  behavior; it does not write to Allbert Home or any Elixir-managed path.

## Module Shapes

All modules live under `./plugins/stocksage/` — none in `apps/allbert_assist/`


### `StockSage.Bridge.Protocol`

Location: `./plugins/stocksage/lib/stocksage/bridge/protocol.ex`

```elixir
@spec encode_request(map()) :: {:ok, binary()} | {:error, term()}
@spec decode_response(binary()) :: {:ok, map()} | {:error, term()}
@spec valid_action?(String.t()) :: boolean()
```

`encode_request/1` validates required fields and returns a newline-terminated
JSON binary. `decode_response/1` parses a JSON line and returns a map with
at least `id` and `status` fields. `valid_action?/1` returns true for `"ping"`
and `"run_analysis"`.

### `StockSage.TraderBridge`

Location: `./plugins/stocksage/lib/stocksage/trader_bridge.ex`

A GenServer that wraps a long-lived Erlang Port process.

```elixir
@spec start_link(keyword()) :: GenServer.on_start()
@spec analyze(map()) :: {:ok, map()} | {:error, term()}
@spec ping() :: :ok | {:error, term()}
@spec bridge_status() :: :running | :disabled | :crashed
```

Startup behavior:

- Reads `stocksage.bridge_enabled` from Settings Central at startup.
- When `bridge_enabled` is false, the GenServer starts but does not open a
  Port; `analyze/1` returns `{:error, :bridge_disabled}`.
- When `bridge_enabled` is true, opens a Port to `bridge.py` using
  `Port.open/2` with `{:spawn_executable, python_path}`, args pointing to
  the bridge script resolved from `:code.priv_dir(:stocksage)`.
- `python_path` is resolved from the `stocksage.python_path` setting;
  default `"python3"`.

Request handling:

- `analyze/1` accepts a validated map `{ticker, analysis_date, engine}`,
  generates a UUID request id, encodes the request via `Protocol.encode_request/1`,
  sends the binary to the Port, registers a pending entry keyed by request id,
  and awaits the response with a timeout of `stocksage.bridge_timeout_ms`
  (default 300_000 ms).
- Responses arrive as `{port, {:data, {:eol, line}}}` messages. The GenServer
  decodes the line, matches by id, and replies to the waiting caller.
- Timeout: the pending entry is cleared; the caller receives `{:error, :timeout}`.

Port crash handling:

- On `{:EXIT, port, reason}`: all pending callers receive `{:error, :bridge_crashed}`.
  The port reference is cleared. The next call to `analyze/1` attempts to reopen
  the Port.
- The crash does not propagate to Allbert core supervision.

### `StockSage.Actions.RunAnalysis`

Location: `./plugins/stocksage/lib/stocksage/actions/run_analysis.ex`

```elixir
use Jido.Action,
  name: "run_analysis",
  description: "Run a StockSage analysis for a ticker through the Python bridge.",
  schema: [
    ticker:         [type: :string, required: true],
    analysis_date:  [type: :string, required: true],  # ISO 8601 date string
    engine:         [type: :string, default: "tradingagents"],
    user_id:        [type: :string, required: true],
    queue_entry_id: [type: :string, required: false]
  ]
```

Action flow:

1. Validate schema. Invalid ticker format or unparseable date returns an error
   before Security Central evaluation.
2. Evaluate `:stocksage_analyze` through Security Central.
3. When `:needs_confirmation` (the default):
   - Build a confirmation record with ticker, analysis_date, engine, user_id,
     selected action, permission, risk tier `high`, and a disclosure note that
     TradingAgents will make external market-data API calls.
   - Create the durable pending confirmation record through the v0.07 workflow.
   - Return `:needs_confirmation` with confirmation id and bounded display summary.
4. When the confirmation is approved (`approve_confirmation` resume path):
   - Re-check Security Central with confirmation context.
   - Call `StockSage.TraderBridge.analyze/1` with `{ticker, analysis_date, engine}`.
   - On bridge success: persist the result (see Result Persistence).
   - On bridge error: persist a failed analysis record; return structured error.
   - If a `queue_entry_id` was provided, mark the queue entry completed or failed.
5. Return a structured result map with `analysis_id`, `status`, `ticker`,
   `analysis_date`, `engine`, and a bounded `summary` string.

Registration: `StockSage.Plugin.actions/0` returns the `RunAnalysis` capability
metadata struct. The action is listed in `allbert_plugin.json` under
`contributions.actions`.

### Result Persistence

On a successful bridge response, the action writes:

- `stocksage_analyses` row: `{id, user_id, ticker, analysis_date, engine,
  status: "completed", summary: bounded(result.summary, 500), thread_id,
  session_id, queue_entry_id, trace_id, request_id, bridge_duration_ms,
  inserted_at}`.
- `stocksage_analysis_details` row: `{id, analysis_id, raw_result:
  bounded(result.raw, max_output_bytes), truncated: result.truncated,
  inserted_at}`.

On a failed bridge response, the action writes:

- `stocksage_analyses` row: `{status: "failed", summary: bounded(reason, 200),
  ...}`.
- No `stocksage_analysis_details` row.

On bridge disabled or crashed (before confirmation approval), no row is written.

Queue integration: when `queue_entry_id` is provided, the action updates the
matching `stocksage_analysis_queue` row's status to `"completed"` or `"failed"`
and writes a `stocksage_queue_runs` row recording `{queue_entry_id,

## Settings Central Schema

All new settings are added to the StockSage plugin's Settings Central schema
contribution, not to the Allbert core settings namespace.

New keys under `stocksage.*`:

- `stocksage.bridge_enabled` — boolean, default `true`, writable. When false,
  `StockSage.TraderBridge` starts but does not open a Port; `RunAnalysis`
  returns `{:error, :bridge_disabled}` before confirmation.
- `stocksage.python_path` — string, default `"python3"`, writable. Path or
  executable name for the Python 3 interpreter. Used by `TraderBridge` when
  opening the Port.
- `stocksage.bridge_timeout_ms` — integer, default `600_000` (10 min),
  writable. Per-request timeout; exceeding it returns `{:error, :timeout}`.
  The default was raised from `300_000` to `600_000` in the M2 audit
  closeout because a real TradingAgents `propagate` call routinely takes
  5–10 minutes. Operators running stub-mode tests or fast smoke flows can
  lower it via Settings Central.
- `stocksage.bridge_max_output_bytes` — integer, default `1_048_576` (1 MiB),
  writable. Bridge response bodies exceeding this are truncated and marked
  `truncated: true` before persistence. **The same bound applies to the
  persisted detail row body in `stocksage_analysis_details.content`**
  (M2 audit closeout — gap 5). Operators who want tighter trace and
  redaction posture for persisted detail should lower this setting; the
  bridge response and the persisted detail row are bounded together.
- `stocksage.analysis_engine` — enum `["tradingagents"]`, default
  `"tradingagents"`, writable. The default engine passed to `RunAnalysis` when
  no `engine` param is supplied. `"tradingagents"` is the only valid value in
  v0.22; v0.25 (formerly v0.23) adds `"native"`.

Existing `permissions.stocksage_analyze` key (added in M1):

- Settings Central key: `permissions.stocksage_analyze`
- Default: `needs_confirmation`
- Safety floor: `needs_confirmation` (Security Central enforces)
- Risk tier: `high`
- Writable: true (but only tightenable; floor prevents setting to `allowed`)

## Signals And Trace Metadata

Named StockSage lifecycle signals are emitted from `RunAnalysis` via
`Jido.Signal.new/3`, redacted through `AllbertAssist.Signals.redact/1`, and
logged through `AllbertAssist.Signals.log/1` so they reach the same log
path as the runtime's `allbert.action.*` lifecycle signals. They were
named in the original M5 plan but not emitted; the M2 audit closeout
(gap 4) wires them up.

`allbert.stocksage.analysis_requested` — emitted by `RunAnalysis` *after*
queue-entry validation and *before* the bridge call (i.e., once the
analysis is committed to running). Includes: ticker, analysis_date,
engine, user_id, queue_entry_id (when present), confirmation_id (from
context when resumed), input_signal_id, trace_id. Does not include raw
analysis content, secrets, or LLM credentials.

`allbert.stocksage.analysis_completed` — emitted on successful bridge
return and persistence. Includes: analysis_id, ticker, analysis_date,
engine, user_id, queue_entry_id (when present), bridge_duration_ms,
truncated flag, stub flag (true when the response came from the stub
path). Does not include raw result body.

`allbert.stocksage.analysis_failed` — emitted on bridge error, crash,
timeout, or queue-entry-validation rejection (gap 3 closure). Includes:
analysis_id (when a failure row was persisted), ticker, analysis_date,
engine, user_id, queue_entry_id (when present), bridge_duration_ms (or
zero when validation failed before the bridge call), bounded `error`
string. Does not include partial TradingAgents output, stack traces, or
secrets.

**Authority note (M2 audit closeout):** these signals are diagnostic; they
do not authorize anything. The action runner's existing
`allbert.action.*` lifecycle signals continue to fire alongside them, and
Security Central remains the only authority boundary at the action
runner.

Trace sections added in v0.22:

- `## StockSage Analysis` — appears in traces for `RunAnalysis` runs. Includes:
  analysis_id or confirmation id, ticker, analysis_date, engine, bridge status
  (`running` / `disabled` / `crashed`), confirmation status, result summary
  (bounded to 200 chars), and truncated flag. Raw result body is never included.

Existing trace sections that pick up RunAnalysis data:

- `## Action Metadata` — includes `StockSage.Actions.RunAnalysis` as the
  selected action with permission `:stocksage_analyze`.
- `## Confirmation Metadata` — includes the RunAnalysis confirmation record
  when `:needs_confirmation` is returned, per the v0.07 confirmation workflow.

- No native Jido trading agents as default (v0.25, formerly v0.23).
- No StockSage LiveViews or workspace UI (v0.27, formerly v0.25).
- No per-market-data-source Resource Access Security Posture confirmations
  (v0.28, formerly v0.26, hardening).

  JSON-structured and action-dispatched only.

  v0.24/v0.28).
- No StockSage memory namespace registration (v0.29, formerly v0.27).
