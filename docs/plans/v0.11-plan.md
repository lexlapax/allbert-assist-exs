# Allbert v0.11 Execution-Aware Intent, URI-Based Resource Access, And Approval Handoff

## Purpose

v0.11 introduces the first structured intent decision contract over real risky
capabilities, the shared Approval Handoff contract for channel-native
approvals, and the URI-based Resource Access Security Posture consumption
layer that lets higher level consumers use local, network, package, skill, and
future MCP/agent resources safely.

Allbert needs this before scheduled jobs or new channels consume shell, script,
package, network, and online import capabilities. The contract makes selection,
confirmation, denial, and refusal explainable without adding new execution
powers.

Resource access security posture is the key v0.11 generalization. A user-facing
request such as "check this URL and summarize it", "inspect this local file",
"import this skill directory", "install this remote skill URL", "call this MCP
resource", or "delegate to this agent" is not a special skill flow and not a
generic browser or file explorer. It is an execution-aware decision over a
canonical resource URI, operation class, access mode, scope, limits, and
downstream consumer. The user sees those details before approval; approved
work then resumes only through the registered action runner and the v0.08,
v0.09, or v0.10 execution substrate that owns the actual effect. MCP and
agent URI schemes remain representable but unsupported unless a later plan
adds the corresponding action, security, approval, and adapter story.

Skills remain explicit special cases inside that posture. Direct remote skill
URL import uses `https://... + import_skill`; local skill directory import uses
`file://... + import_local_skill`; trusted skill script execution uses
`skill://... + run_skill_script`; existing online skill source actions remain
source-profile consumers over `allbert://sources/...` style identities. None
of those grants automatically authorize trust, activation, dependency install,
or script execution.

Approval Handoff is the bridge between an action that needs confirmation and
the user's current channel. CLI/REPL, web chat, scheduled jobs, and future
Telegram-style adapters should all translate a risky action result into native
approve, deny, and details affordances without owning confirmation storage,
security policy, or execution.

## Roadmap And Vision Fit

v0.11 is the bridge from Allbert's early safe-action substrate to the later
workspace roadmap. It must preserve the Jido-centered vision: user input
becomes a signal, an agent produces an inspectable decision, registered actions
perform bounded work, Security Central checks permission, confirmations pause
risky effects, and traces explain what happened.

This milestone prepares later work without implementing it:

- v0.12 uses the reserved `user_id` and `thread_id` fields for SQLite
  conversation history.
- v0.13 jobs consume Approval Handoff and resource posture when background
  work needs confirmation.
- v0.14 and v0.15 use `session_id` and `active_app` for scratchpad and
  app-scoped routing, but v0.11 does not route by app.
- v0.16 channels render the same Approval Handoff data natively.
- v0.19 cross-surface intent enrichment uses the v0.11 Decision contract as
  the structured base for hybrid deterministic and model-assisted ranking over
  real skills, actions, session context, and app registry inputs.
- v0.28 (formerly v0.26) security evals build on the operation-class, URI,
  and handoff metadata introduced here.

The implementation should be conservative: no new execution powers, no new
browser or crawler substrate, no conversation storage, and no app contract
work. v0.11 makes risky intent explainable and approvable over the existing
v0.08-v0.10 primitives.

## Implementation Boundaries

v0.11 should add plain-data contracts and routing glue around existing
subsystems instead of replacing those subsystems.

Authoritative existing boundaries:

- Registered actions: `AllbertAssist.Actions.Registry` and
  `AllbertAssist.Actions.Runner.run/3`.
- Resource identity and operation vocabulary:
  `AllbertAssist.Resources.ResourceURI`, `AllbertAssist.Resources.Ref`, and
  `AllbertAssist.Resources.OperationClass`.
- Remembered resource grants: `AllbertAssist.Resources.Grants`.
- Confirmation storage and rendering metadata:
  `AllbertAssist.Confirmations.Store` and
  `AllbertAssist.Confirmations.ResourceMetadata`.
- Security decisions: `AllbertAssist.Security`, `AllbertAssist.Security.Policy`,
  and `AllbertAssist.Security.Decision`.
- Intent routing entrypoint: `AllbertAssist.Agents.IntentAgent`, called by
  `AllbertAssist.Runtime.submit_user_input/1`.

New v0.11 contracts should live under `AllbertAssist.Intent` as inert structs
or schema-like modules:

- `AllbertAssist.Intent.Decision`
- `AllbertAssist.Intent.ResourceAccess`
- `AllbertAssist.Intent.ApprovalHandoff`

Those modules validate and render decision metadata. They do not fetch,
import, run, summarize, approve, deny, remember grants, or write
confirmations by themselves.

## Intent Decision Contract

Introduce a structured intent decision, tentatively
`AllbertAssist.Intent.Decision`, with fields equivalent to:

- `:intent`
- `:confidence`
- `:reason`
- `:risk_summary`
- `:selected_skill`
- `:candidate_skills`
- `:selected_action`
- `:candidate_actions`
- `:permission`
- `:confirmation`
- `:execution_mode`
- `:resource_access`
- `:foreground_or_background`
- `:user_id`
- `:thread_id`
- `:session_id`
- `:active_app`
- `:approval_handoff`
- `:alternatives`
- `:diagnostics`
- `:trace_metadata`

Field semantics:

- `intent`: atom or string name for the selected intent family.
- `confidence`: number from `0.0` to `1.0`.
- `reason`: concise operator-facing explanation of the selected route.
- `risk_summary`: redacted summary suitable for CLI, LiveView, traces, and
  audits.
- `selected_skill`: skill name or nil; must resolve through the skill registry
  when present.
- `candidate_skills`: ranked list of skill summaries with rejection reasons.
- `selected_action`: registered action name or nil; must resolve through
  `AllbertAssist.Actions.Registry` when present.
- `candidate_actions`: ranked list of action capability summaries with
  rejection reasons.
- `permission`: permission atom from the selected action capability, nil only
  when no action is selected.
- `confirmation`: one of `:not_required`, `:required`, `:pending`,
  `:unsupported`, or `:refused`.
- `execution_mode`: execution mode from the selected action capability, nil
  only when no action is selected.
- `resource_access`: normalized list of `ResourceAccess` entries. Empty list
  means the decision does not consume a local or remote resource.
- `foreground_or_background`: `:foreground` for v0.11 interactive turns;
  `:background` is reserved for v0.13 jobs.
- `approval_handoff`: handoff data when a request is confirmation-required or
  pending; nil otherwise.
- `alternatives`: safe alternatives or next actions the operator can choose.
- `diagnostics`: developer-facing validation details, never secret-bearing.
- `trace_metadata`: compact map included in runtime traces.

The decision object is plain data. It does not execute anything. Runtime still
executes through registered actions, the action lifecycle runner, Security
Central, the permission gate compatibility layer, and confirmation state.

`user_id` is a reserved optional string identity field that defaults from the
existing `operator_id` value or `"local"`. `thread_id`, `session_id`, and
`active_app` are reserved optional context fields for v0.12 conversation
history and v0.15 app-scoped routing; v0.11 records the shape but
does not wire conversation storage or app routing.

Validation rules:

- A selected action must exist in `AllbertAssist.Actions.Registry`.
- Selected action `permission`, `execution_mode`, confirmation requirement,
  and `resumable?` are copied from registry capability metadata, not model
  output.
- Candidate actions that do not resolve are retained only as rejected
  diagnostics.
- Resource entries must validate through `AllbertAssist.Resources.Ref` and the
  closed `OperationClass` vocabulary.
- Unsupported `mcp://`, `agent://`, and `agent+https://` resource URIs may be
  represented only with `unsupported_resource_workflow` or a refusal; they
  cannot resolve to an executable target action in v0.11.
- If Security Central denies a permission, the decision is refused or
  confirmation-blocked before action execution.
- Invalid model output cannot invent skill names, action names, permissions,
  execution modes, operation classes, URI schemes, or approval scopes.

Each `ResourceAccess` entry is posture metadata, not an executor. It should
carry canonical resource URI, redacted/display URI, derived origin kind and
canonical id when needed, source/profile label, operation class, access mode,
scope, expected content kind when applicable, accepted content types when
applicable, byte/output cap, redirect/retry posture when applicable,
digest/cache expectation, downstream consumer, summarizer or parser
requirement, origin channel/surface, response target when available, and
allowed approval scopes. Initial operation classes should include:

- `read_local_path`
- `write_local_path`
- `run_shell_command`
- `run_skill_script`
- `import_local_skill`
- `external_service_request`
- `summarize_url`
- `inspect_document`
- `import_skill`
- `online_skill_search`
- `online_skill_detail`
- `online_skill_audit`
- `online_skill_import`
- `package_install`

Operation class is a security boundary. A remembered approval for
`summarize_url` must not authorize `import_skill`; an `inspect_document`
approval must not authorize package installation; a local skill directory
import approval must not authorize trusted script execution; an import approval
must not authorize skill activation or dependency installation.

Initial workflow mapping:

| Request family | Operation class | Target action | Permission |
| --- | --- | --- | --- |
| Plan a shell command | `run_shell_command` as prospective resource context | `plan_shell_command` | `:command_plan` |
| Run a shell command | `run_shell_command` | `run_shell_command` | `:command_execute` |
| Run a trusted skill script | `run_skill_script` | `run_skill_script` | `:skill_script_execute` |
| Call an approved external service | `external_service_request` | `external_network_request` | `:external_network` |
| Plan a package install | `package_install` | `plan_package_install` | `:read_only` |
| Run a package install | `package_install` | `run_package_install` | `:package_install` |
| Search or inspect online skills | `online_skill_search` / `online_skill_detail` / `online_skill_audit` | existing online skill actions | `:external_network` |
| Import from online skill source | `online_skill_import` | `import_online_skill` | `:online_skill_import` |
| Import direct HTTPS skill URL | `import_skill` | `import_remote_skill` | `:online_skill_import` |
| Import local skill directory | `import_local_skill` | `import_local_skill` | `:skill_write` |
| Summarize a URL | `summarize_url` | `external_network_request` plus summary handoff, or unsupported result | `:external_network` |
| Inspect a remote document | `inspect_document` | `external_network_request` plus extractor handoff, or unsupported result | `:external_network` |
| Inspect a generic local file | `read_local_path` / `inspect_document` | `unsupported_resource_workflow` in v0.11 unless a registered bounded read action is explicitly added with tests | `:read_only` plus confirmation if added |
| MCP or agent URI | planned operation only | `unsupported_resource_workflow` | `:read_only` |

v0.11's required implementation is the decision, posture, handoff, and safe
execution path over already registered primitives. A working URL summary may
only fetch through the confirmed v0.10 `Req` adapter. A working document
inspection may only fetch through the same adapter and a registered extractor.
If the summarizer, extractor, or generic local file reader is absent, the
approved target result reports that capability as unavailable; it must not
fall back to shell commands, ad hoc file reads, browser automation, or
model-generated scripts.

## Approval Handoff Contract

Introduce Approval Handoff as plain data attached to the intent decision or
confirmation-needed action result. It should carry:

- confirmation id
- redacted action summary
- resource access summary when the risky action reads, writes, runs, fetches,
  imports, installs, or inspects a resource: canonical URI/resource URI,
  redacted/display URI, derived origin kind, operation class, access
  mode, scope, source/profile, expected content kind when applicable,
  byte/output cap, destination consumer, and whether a summarizer, parser,
  installer, or importer is expected after acquisition
- origin channel and surface
- origin response target when the channel supports durable follow-up
- allowed approval surfaces
- remembered approval choices, scoped to exact resource, local directory,
  remote URL, or remote URL hierarchy/prefix and operation class
- same-channel and cross-channel resolution expectations
- channel render hints for approve, deny, details, pending, resolved, and
  result-return states

Channels may render the handoff in their native UX, but approval and denial
must still call `approve_confirmation` and `deny_confirmation`. Channels must
not mutate confirmation records directly, bypass Security Central, or invoke
execution adapters.

The handoff should normalize existing confirmation metadata rather than create
a second confirmation store. It may be built from a pending confirmation
record, `AllbertAssist.Confirmations.ResourceMetadata`, and the intent
decision. The expected shape is:

- `confirmation_id`
- `status`: `:pending`, `:approved`, `:denied`, `:expired`, or `:resolved`
- `target_action`: registered action name plus redacted params summary
- `resource_access`: list of normalized resource posture entries
- `risk_summary`
- `origin`: channel, surface, response target, and user/session context when
  present
- `allowed_actions`: approve, deny, details, and allowed remember choices
- `render_hints`: compact labels and redacted details for CLI and LiveView
- `result_return`: same-channel or cross-channel expectations

The handoff cannot be used as authorization. Approval still re-checks Security
Central and resumes only the registered target action stored in the
confirmation record.

## Milestones

### Milestone 1: Execution-Aware Intent Decision Contract

Status: implemented. The runtime now builds an inert
`AllbertAssist.Intent.Decision` before action dispatch, validates selected
actions, selected skills, resource posture, Security Central policy, and
reserved identity/context fields, and surfaces decision/resource metadata in
runtime responses and markdown traces. Shell execution denied by policy now
stops at the decision layer before invoking the shell action. Focused coverage
lives in `AllbertAssist.Intent.DecisionTest`,
`AllbertAssist.Agents.IntentAgentTest`, and
`AllbertAssist.RuntimeIntentAgentTest`.

Goal: make risky capability selection inspectable before execution or
confirmation.

Implementation expectations:

- Define the intent decision data shape and validation rules.
- Include selected and candidate skills/actions, permission, risk,
  confirmation, execution mode, alternatives, diagnostics, and trace metadata.
- Emit URI-backed `resource_access` data for risky resource operations. The
  decision should name canonical URI/resource URI, operation class, access
  mode, scope, limits, downstream consumer, and target action without executing
  anything.
- Validate decisions against known skills, registered actions, known
  permissions, Security Central, Settings Central, and confirmation state.
- Cover shell, local file/path, skill script, local skill directory import,
  package-manager, external service, online skill, remote skill URL import,
  and URL/document decisions so Approval Handoff can show the actual risky
  target, not just "network", "file", or "install".
- For package installs, include exact package specs, target root, dry-run argv,
  npm execution argv preview, timeout/output caps, and pip preview-only
  warnings.
- Update `AllbertAssist.Agents.IntentAgent` to produce a `Decision` struct as
  part of its `respond/1` result. The current routing produces a tuple-based
  route then calls `run_route/3` in one pass; M1 splits this into route →
  Decision production → validation → execution so the Decision is inspectable
  before any action runs. Update `AllbertAssist.Runtime` to extract and surface
  the decision, approval handoff state, resource access posture, and
  diagnostics in runtime responses and traces.
- Keep the decision object inert; it never executes work.

### Milestone 2: URI-Based Approval Handoff Contract

Status: implemented. Pending confirmation responses now attach shared
`ApprovalHandoff` data to the intent decision, top-level runtime response,
and first action record. The handoff includes confirmation id, target action,
origin, resource posture, approval/detail actions, remember-scope choices,
render hints, and same-channel result-return expectations. Focused coverage
lives in `AllbertAssist.Intent.ApprovalHandoffTest` and the runtime intent
confirmation tests.

Goal: define the plain-data handoff that lets every channel render pending
approval UX consistently for URI-backed risky resource operations.

Implementation expectations:

- Add `approval_handoff` to the decision/action result shape for work that
  resolves to `:needs_confirmation`.
- Include confirmation id, redacted summary, origin channel/surface, response
  target, allowed approval surfaces, and render hints.
- Include resource details for local and remote consumers: canonical
  URI/resource URI, redacted/display URI, operation class, access mode, scope,
  source/profile, content or output limits, expected consumer,
  summarizer/parser availability, import destination when relevant, and
  redaction/cache posture.
- Include remembered approval options as data, not policy: approve once,
  remember exact resource for this operation class, remember local directory
  for this operation class, remember URL hierarchy/prefix for this operation
  class, or remember the exact package/source URI when policy allows it.
  Channels pass the target action permission when asking whether a remembered
  grant applies; the grant store does not infer permissions from skill,
  summary, import, package, MCP, agent, or channel workflows.
- Define result-return expectations for same-channel and cross-channel
  approval so a resolver sees the immediate outcome and the origin channel can
  show durable follow-up when it supports callbacks. Channels must preserve the
  v0.10 distinction between an approved confirmation and a failed resumed
  target, such as an online source HTTP failure with `target_status=failed`.
- Keep approval handoff data descriptive only; it must not become authority to
  execute, approve, or deny.

### Milestone 3: Existing Channel Approval UX Baseline

Status: implemented. `mix allbert.ask` renders Approval Handoff summaries,
resource posture, remember-scope choices, and concrete confirmation commands.
`mix allbert.confirmations` exposes pending details, approve, deny, and
remember-scope commands while still resolving through registered confirmation
actions. `AllbertAssistWeb.AgentLive` renders the same handoff as native
Approve, Deny, and Details controls and calls `approve_confirmation` or
`deny_confirmation` through `AllbertAssist.Actions.Runner`. Focused coverage
lives in the CLI task tests and `AllbertAssistWeb.AgentLiveTest`.

Goal: make the first CLI/REPL-style and web surfaces render channel-native
approvals over the shared handoff.

Implementation expectations:

- CLI/REPL-style surfaces should show pending approval details and expose
  approve, deny, and details commands over the confirmation actions.
- For resource access, CLI/REPL output should show the canonical or redacted
  URI appropriate for the operator surface, operation class, access mode,
  scope, content/output limit, destination consumer, and remember-scope choices
  before approval.
- Web LiveView/chat surfaces should render the same handoff as native approve,
  deny, and details controls.
- For resource access, web surfaces should show the same URI/resource summary,
  operation class, access mode, scope, limits, destination consumer,
  same-channel/cross-channel resolution expectation, and remembered-scope
  options.
- Both surfaces must call `approve_confirmation` and `deny_confirmation`
  through the action runner.
- Neither surface may write confirmation files directly, make policy
  decisions, or invoke shell, script, package, network, import, or other
  execution adapters directly.

### Milestone 4: First User-Facing URI Resource Consumers

Status: implemented. URL summary and remote document-inspection prompts now
route to the existing confirmed `external_network_request` substrate with
operation-scoped `summarize_url` or `inspect_document` resource refs,
Approval Handoff metadata, and no fetch before approval. Approved fetches report
the missing summarizer or extractor as unavailable instead of inventing a
downstream consumer. Generic local file inspection now returns inert
`file://...` `read_local_path` posture through `unsupported_resource_workflow`
with an explicit no-shell-fallback message. Focused coverage lives in
`AllbertAssist.Actions.ExternalNetworkRequestTest`,
`AllbertAssist.Agents.IntentAgentTest`, `AllbertAssist.RuntimeIntentAgentTest`,
`Mix.Tasks.Allbert.AskTest`, and `AllbertAssistWeb.AgentLiveTest`.

Goal: add the first user-facing URI resource workflows over the v0.10
substrate without making any one consumer the architecture center or adding an
unbounded file/browser/crawler primitive.

Implementation expectations:

- Document and implement the baseline UX for "check this URL and summarize
  it" as operation class `summarize_url`: fetch approval happens first;
  summarization receives only approved, bounded, extracted content. Fetching
  must run through the confirmed v0.10 `external_network_request` substrate.
- Document and implement the baseline UX for "inspect/read this document URL"
  as operation class `inspect_document`, with bounded fetch/extraction and a
  clear unsupported-content message for formats not yet extractable. Fetching
  must run through the confirmed v0.10 `external_network_request` substrate.
- Document the baseline UX for "read/inspect this local file" as a
  `file://...` posture and Approval Handoff. Generic local file reads remain
  unavailable in v0.11 unless implemented as a separately registered bounded
  action with canonical path handling, traversal/symlink defense,
  confirmation, content/output caps, redaction, and focused tests. Do not use
  shell commands as a local read fallback.
- If no summarizer agent/action exists, an approved fetch or read result must
  report that summarization is unavailable instead of hiding the missing
  capability.
- Keep HTTPS GET as the initial posture default for URL/document consumers and
  canonical workspace/Allbert Home-relative path handling for local consumers.
  Block local/private-network targets, credentials in URLs, hidden redirects,
  unbounded downloads, path traversal, symlink escapes, and raw secret-bearing
  output.
- Ensure remembered `summarize_url`, `inspect_document`, or local read grants
  cannot be reused for `import_skill`, package install, skill activation, or
  script execution.

### Milestone 5: Specialized URI Consumers And Future Scheme Boundaries

Status: implemented. Direct skill URL import, local skill directory import,
package install planning, shell execution, trusted skill scripts, online skill
source consumers, and unsupported MCP/agent schemes are now covered as
operation-scoped URI resource consumers in the decision/handoff path. The M5
implementation preserves the v0.10 action substrates: remote skill imports are
`https://... + import_skill`, local skill directories are
`file://... + import_local_skill`, package planning emits `pkg:` plus target-root
resources without shell authority, and MCP/agent URIs remain inert
`unsupported_resource_workflow` decisions. Focused coverage lives in
`AllbertAssist.Agents.IntentAgentTest`,
`AllbertAssist.Actions.SkillImportActionsTest`,
`AllbertAssist.Actions.PackageInstallActionsTest`,
`AllbertAssist.Actions.ResourceRefsTest`, and
`AllbertAssist.Resources.GrantsTest`.

Goal: make skills, packages, MCP, and future agents explicit consumers of the
URI posture instead of separate permission models.

Implementation expectations:

- Document direct skill URL import as `https://... + import_skill`; it should
  fetch through the same posture, validate with the existing Agent Skills
  parser/registry, audit, and import disabled/untrusted.
- Document local skill directory import as `file://... + import_local_skill`;
  it should inspect canonical local resources, validate with the existing Agent
  Skills parser/registry, audit, and import disabled/untrusted without granting
  script execution.
- Document MCP resource/tool calls and `agent://`/`agent+https://` delegation
  as explicitly unsupported URI resource consumers unless a later release adds
  the adapter and permission story.
- Treat package install as PURL-compatible `pkg:` resources plus target-root
  local resources, not shell authority.
- Treat `skills.sh` as one source profile. Direct `SKILL.md` or skill archive
  URLs should use `https://... + import_skill` and the same posture object.
- Preserve v0.10 online skill search/detail/audit/import as source-profile
  consumers of the same posture, not a separate marketplace-specific path.
- Preserve v0.08 shell command and v0.09 trusted skill script UX as local
  resource-access consumers that expose cwd/path/script resource details in
  approval handoff without changing their execution substrate.
- Ensure remembered `summarize_url`, `inspect_document`, or
  `import_local_skill` approvals cannot be reused for `import_skill`, package
  install, skill activation, or script execution.

## Test Plan

Focused implementation tests should cover:

- intent decision validation for known actions, permissions, execution modes,
  URI-backed resource access operation classes, and invalid model output
- Approval Handoff rendering data for CLI/REPL and web surfaces
- pending URL summary request creates confirmation before fetch
- pending remote document inspection creates confirmation before fetch or
  reports unsupported extraction without fetching
- generic local file inspection renders `file://` posture and unsupported or
  unavailable status unless a registered bounded read action is implemented
- pending local skill directory import creates confirmation before local import
- pending direct skill URL import creates confirmation before fetch or import
- unsupported `mcp://`, `agent://`, and `agent+https://` decisions render
  clear unavailable states without execution
- denial performs no local read/import, fetch, extraction, summarization,
  import, or grant write
- approved URL summary fetches bounded content through the v0.10 adapter and
  returns a summary when a summarizer agent/action is available
- approved URL summary reports summarizer unavailable when no summarizer exists
- document inspection reports unsupported content types or extractor failures
  without hiding the approved fetch outcome
- remembered exact-resource, local-directory, exact-URL, and URL-prefix grants
  are scoped by operation class
- `summarize_url` and `inspect_document` grants do not authorize
  `import_skill`
- `import_local_skill` grants do not authorize `run_skill_script`
- direct skill URL import uses resource access posture, parser and registry
  validation, import audit, disabled/untrusted result flags, and no activation,
  dependency installation, or script execution
- local skill directory import uses resource access posture, parser and
  registry validation, import audit, disabled/untrusted result flags, and no
  activation, dependency installation, or script execution
- jobs and future channel adapters can consume the same handoff metadata
  without owning confirmation storage, policy, fetch, summarization, or import

## Non-Goals

- No new local or network primitive beyond the v0.08, v0.09, and v0.10
  execution substrates and the URI-first resource identity substrate completed
  in v0.10 closeout.
- No model-selected side effects without validation.
- No new external channels; v0.16 consumes the handoff contract for
  Telegram/email/SMS-style adapters.
- No broad cross-channel routing; later channel-routing work owns that.
- No memory retrieval or embeddings work.
- No conversation history wiring. Thread context is v0.12 work that plugs into
  the reserved `user_id`, `thread_id`, and `session_id` fields after v0.11
  ships.
- No channel-specific intent fork.
- No browser, crawler, arbitrary recursive fetcher, unrestricted local file
  explorer, or unbounded document ingestion.
- No automatic trust, enablement, activation, package install, or script
  execution from imported local or fetched network content.

## Definition Of Done

v0.11 is done when:

- Runtime turns can include an intent decision object.
- Risky capability prompts produce inspectable selected/candidate decisions
  before execution or confirmation.
- Resource access prompts produce operation-scoped posture data before any
  local read/import, command/script run, fetch, install, or inspection,
  including canonical URI/resource URI, access mode, scope, limits, consumer,
  and allowed approval scopes.
- Confirmation-needed prompts produce an Approval Handoff that CLI/REPL-style
  and web surfaces can render as channel-native approvals.
- URL summarization, document inspection, direct skill URL import, local skill
  directory import, shell command execution, and trusted skill script execution
  are represented as consumers of the same posture, with clear unavailable or
  unsupported states when summarization, extraction, validation, import, or
  execution cannot proceed.
- Remembered approvals are operation-scoped and cannot cross from summaries or
  document inspection into skill import, package install, activation, or script
  execution, or from local skill import into script execution.
- Invalid model or route output cannot invent skill names, action names,
  permissions, or execution modes.
- Intent traces explain selected and rejected candidates plus approval handoff
  state when a confirmation is pending or resolved.
- Approval still resumes work only through `approve_confirmation`; no channel
  gets direct execution authority.
