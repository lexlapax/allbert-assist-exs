# Allbert v0.11 Additional Channels

## Purpose

v0.11 adds channel adapters after CLI, LiveView, settings, skills,
confirmations, execution adapters, jobs, and intent decisions share the same
runtime core.

Channels translate external messages into Allbert signals and render responses
back to their medium. They must not own agent logic, permission policy,
settings semantics, memory semantics, skill execution, or risky execution
policy.

## Expected Inputs

- v0.02 Settings Central exists for channel enablement, preferences, and
  credentials.
- v0.05 confirmation workflow can handle interrupted actions across operator
  surfaces.
- v0.06 through v0.08 risky execution adapters are action-backed and
  traceable.
- v0.09 intent decisions expose selected skill/action, risk, permission, and
  confirmation metadata.
- v0.10 jobs can emit runtime signals without becoming a separate runtime.
- CLI and LiveView remain the reference operator surfaces.

## Expected Direction

- Add channel adapters after CLI and LiveView share the same runtime core.
- Candidate channels include email, SMS, Discord/Telegram-style chat, browser
  capture, and native UI surfaces.
- Channels translate external messages to signals and render responses; they do
  not own agent logic.
- Channels read and update shared settings through the settings engine.
- Channel traces include channel metadata and confirmation handoff behavior.

## Non-Goals

- No channel-specific intent engine.
- No channel-specific settings store.
- No direct action execution from adapters.
- No remote channel credentials outside Settings Central secrets.
- No broad provider matrix until one adapter is boring and tested.

## Definition Of Ready

v0.11 is ready to implement when:

- Runtime input and response shapes are stable across CLI, LiveView, jobs, and
  execution-aware intent decisions.
- Settings Central can store channel enablement, response style, and
  credentials.
- Confirmation decisions can be routed back to an operator surface.

## Definition Of Done

v0.11 is done when:

- At least one additional channel adapter can submit messages through the
  runtime.
- Channel messages include channel metadata in traces.
- Channel credentials are stored as Settings Central secrets.
- Channel enablement and response style are settings-backed.
- The adapter has no embedded agent, permission, execution, or memory logic.
