# Allbert v0.05 Request Flow: Security Central Foundation

This document is the living request-flow map for v0.05 Security Central
Foundation. It should be updated during implementation as planned module names
become actual module names.

## Status

Status: released as v0.05 on 2026-05-02.

v0.05 is a foundation milestone. It adds no new execution powers. It widens the
security vocabulary before v0.06 action-backed skills and later confirmations,
execution adapters, jobs, channels, and security hardening.

## Boundary

Security Central evaluates runtime context against policy. It does not store
operator policy itself and does not execute actions.

Settings Central remains responsible for:

- operator settings
- permission defaults
- skill trust and enablement settings
- provider and model profiles
- encrypted secret storage
- settings audit records

Security Central is responsible for:

- normalizing actor, channel, session, request, skill, action, resource,
  secret-status, and external-content context
- resolving effective policy from Settings Central and built-in safety floors
- returning structured security decisions
- naming risk tier, confirmation requirement, redaction obligations, audit
  metadata, trace metadata, policy source, and trust boundary
- providing compatibility for `PermissionGate.authorize/2`
- producing read-only security status data for operator inspection

The v0.04 action runner remains the runtime execution boundary. v0.05 should
not replace the runner or move effectful behavior into security modules.

## Current Runtime Shape

Current action invocation already flows through:

```text
channel or runtime caller
  -> AllbertAssist.Runtime or operator surface
  -> AllbertAssist.Actions.Runner.run/3
  -> AllbertAssist.Actions.Registry.resolve/1
  -> registered Jido action run/2
  -> AllbertAssist.Security.PermissionGate.authorize/2
  -> action response with permission_decision
  -> runner_metadata
  -> action lifecycle signals
  -> trace recording through record_trace when enabled
```

v0.05 keeps that shape at first. The compatibility bridge is:

```text
PermissionGate.authorize(permission, context)
  -> AllbertAssist.Security.authorize(permission, context)
  -> Security.Context.normalize/2
  -> Security.Policy.resolve/2
  -> Security.Risk.classify/2
  -> Security.Decision.build/1
  -> compatibility map with widened metadata
```

This lets every current action receive Security Central decisions without a
broad action rewrite. Later milestones can call `AllbertAssist.Security`
directly from new actions once the compatibility bridge is stable.

## Implemented Module Map

- `AllbertAssist.Security`: facade for authorization and read-only security
  status.
- `AllbertAssist.Security.Context`: normalized actor, channel, session,
  request, skill, action, resource, secret-status, and external-content
  context.
- `AllbertAssist.Security.Decision`: canonical decision data and compatibility
  conversion.
- `AllbertAssist.Security.Policy`: Settings Central lookup plus built-in
  safety floors.
- `AllbertAssist.Security.Risk`: risk-tier vocabulary and classification.
- `AllbertAssist.Security.Redactor`: central redaction policy for values,
  metadata, signals, traces, audits, errors, and status surfaces.
- `AllbertAssist.Security.Audit`: audit event shape and metadata helpers.
- `AllbertAssist.Security.Status`: read-only operator status summary.
- `AllbertAssist.Security.PermissionGate`: compatibility layer.

Implemented read-only operator action:

- `AllbertAssist.Actions.Security.Status` with action name `security_status`.

Pure storage, parsing, schema, and formatting helpers remain plain Elixir
behind these boundaries.

## Context Normalization

Security context should accept sparse runtime context and normalize it into a
predictable shape:

```elixir
%{
  actor: %{
    id: "local",
    kind: :operator
  },
  channel: %{
    name: :cli | :live_view | :runtime | :unknown,
    trust: :local | :future_remote | :unknown
  },
  session: %{
    id: nil,
    source_signal_id: nil,
    request_id: nil
  },
  action: %{
    name: "direct_answer",
    module: AllbertAssist.Actions.Intent.DirectAnswer,
    registered?: true,
    internal?: false
  },
  skill: %{
    name: nil,
    source_scope: nil,
    trust_status: nil,
    capability_contract: nil,
    resources: []
  },
  resource: %{
    kind: nil,
    path: nil,
    external_uri: nil
  },
  secret_status: %{
    references: [],
    raw_secret_present?: false
  },
  external_content: %{
    present?: false,
    source: nil
  }
}
```

The implemented map layout preserves the categories above because v0.06 and
later milestones depend on them.

## Decision Shape

Security decisions should preserve current compatibility fields:

```elixir
%{
  permission: :read_only,
  decision: :allowed,
  reason: "Read-only inspection is allowed locally.",
  requires_confirmation: false,
  source: AllbertAssist.Security.PermissionGate
}
```

The widened decision should add:

```elixir
%{
  risk: %{tier: :minimal, reasons: []},
  redaction: %{
    obligations: [:redact_secrets, :redact_credentials],
    surfaces: [:signals, :traces, :audits, :cli, :live_view, :logs]
  },
  audit: %{
    event: "security.decision",
    actor_id: "local",
    channel: :cli,
    permission: :read_only,
    action: "direct_answer",
    skill: nil,
    decision: :allowed,
    policy_source: :built_in_default
  },
  trace: %{
    risk_tier: :minimal,
    policy_source: :built_in_default,
    trust_boundary: :local_operator_to_registered_action
  },
  context: %{},
  trust_boundary: %{},
  policy: %{
    setting_key: "permissions.read_only",
    configured: nil,
    effective: :allowed,
    source: :built_in_default,
    safety_floor: :allowed
  }
}
```

Action responses, runner metadata, traces, and status surfaces receive a
redacted, inspectable map form.

## Policy Resolution

Policy resolution order:

1. Normalize permission and context.
2. Read Settings Central permission policy when a setting exists.
3. Apply skill trust, selected action registration, secret status, and future
   resource context as inputs.
4. Apply built-in safety floors.
5. Return the effective decision and explain whether settings tightened the
   decision or a safety floor capped it.

Initial Settings Central keys already include:

- `permissions.memory_write`
- `permissions.command_plan`
- `permissions.command_execute`
- `permissions.external_network`
- `permissions.settings_write`

v0.05 added these explicit writable Settings Central keys so permission
defaults can be edited from `/settings` while effective decisions remain capped
by Security Central safety floors.

Built-in safety floors:

- unknown permission: deny
- unknown action or unregistered module: deny or inert context
- raw secret read: deny
- command execution: deny until v0.08
- external network: confirmation-needed at most until v0.10 adds confirmed
  external adapters
- skill metadata: never grants permission by itself
- scripts, package installs, online imports: deny/inert until later milestones

## Risk Vocabulary

Initial tiers:

- `:minimal`: read-only local inspection
- `:low`: local memory writes and non-executing command planning
- `:medium`: safe Settings Central writes and operator-visible configuration
- `:high`: secret writes, external network requests, future shell/process
  execution, future script execution, future package installs, and future
  online imports
- `:critical`: raw secret reads, unknown permissions, unknown actions,
  cross-session access, or bypass attempts

Risk does not decide by itself. It explains why policy allows, denies, or
requires confirmation.

## Redaction Flow

Current redaction exists in multiple places:

- `AllbertAssist.Settings.Secrets.redact/2`
- `AllbertAssist.Signals.redact/1`
- settings audit formatting
- action responses that avoid echoing raw secrets

v0.05 centralizes security-facing redaction:

```text
action params / action response / security decision / error / audit metadata
  -> Security.Redactor
  -> lifecycle signals
  -> runner metadata
  -> trace markdown
  -> CLI / LiveView / logs / tests
```

Central redaction must cover:

- keys containing `api_key`, `apikey`, `secret`, `token`, `password`, or
  `credential`
- `secret://...` references when a surface should show only status
- provider credential values
- nested maps, lists, structs, errors, action params, action responses, and
  decisions

## Audit And Trace Flow

Security decisions should produce audit metadata even if v0.05 does not add a
durable security audit store:

```text
Security.Decision
  -> audit metadata
  -> action response metadata
  -> runner metadata
  -> trace metadata
  -> future confirmation/execution audit stores
```

Trace output should include a compact security section:

```markdown
## Security Metadata

- Permission: read_only
- Decision: allowed
- Risk: minimal
- Confirmation: false
- Policy source: built_in_default
- Trust boundary: local_operator_to_registered_action
- Redaction: secrets and credentials redacted
- Audit event: security.decision
```

The exact markdown may change, but traces must remain readable, portable, and
safe for raw operator inspection.

## Trust Boundaries

Security Central should model at least these boundaries:

- local operator input
- CLI and LiveView channels
- future remote channels
- request/session identity
- selected skill and skill source
- trusted, untrusted, disabled, imported, project, user, and built-in skills
- skill resources such as scripts, references, and assets
- registered action execution
- internal actions such as `record_trace`
- Settings Central policy and secret references
- memory and trace persistence
- future shell/process execution
- future external content and external services
- future online skill import and package installation

v0.05 names these boundaries and includes them in decisions. It does not yet
implement all future checks.

## Operator-Visible Behavior

v0.05 adds security and permission inspection in both CLI and the existing
Settings LiveView.

The web surface is a Security & Permissions section inside `/settings`, not a
new top-level Security Center yet. In v0.05, the operator can edit Settings
Central permission settings and see Security Central's effective policy, safety
floors, trust, secrets posture, and redaction posture next to those settings. A
dedicated Security Center makes more sense after v0.07 confirmations, v0.08
execution audits, and v0.16 hardening workflows create review queues and
incident-style workflows.

Useful fields:

- editable Settings Central permission defaults
- built-in safety floors
- effective Security Central decisions after safety floors
- current safety boundaries
- skill trust summary
- secret status counts, never raw secrets
- trace/redaction posture
- planned sandbox/import policy posture

These surfaces must not enable execution. They may modify permission policy
only through Settings Central's existing registered settings actions,
validation, and audit path. Security Central remains an evaluator, not the
settings store.

## Flow: Settings LiveView Security And Permissions Section

`/settings` displays Security Central status and edits Settings Central
permission settings without owning security semantics.

Read flow:

```text
SettingsLive.mount/3 or handle_event/3
  -> AllbertAssist.Actions.Runner.run("security_status", %{}, context)
  -> AllbertAssist.Actions.Registry.resolve/1
  -> AllbertAssist.Actions.Security.Status.run/2
  -> AllbertAssist.Security.status/1
  -> Security.Status.summary/1
  -> Security.Redactor
  -> SettingsLive assigns
  -> Security or Permissions section/tab
```

Write flow for permission settings:

```text
SettingsLive.handle_event("save_permission_setting", params, socket)
  -> AllbertAssist.Actions.Runner.run("update_setting", params, context)
  -> AllbertAssist.Actions.Settings.UpdateSetting.run/2
  -> AllbertAssist.Settings.put/3
  -> Settings Central validation and audit
  -> refresh AllbertAssist.Actions.Runner.run("security_status", %{}, context)
  -> show configured policy plus effective decision after safety floors
```

The LiveView may own:

- section/tab selection
- loading and error display state
- stable DOM IDs
- formatting for compact operator display

The LiveView must not own:

- permission policy resolution
- safety-floor logic
- skill trust semantics
- secret-store reads
- redaction policy
- audit semantics
- action execution or confirmation behavior

The LiveView may edit Settings Central permission settings only by invoking
registered settings actions through the runner. It must not call
`AllbertAssist.Settings`, `AllbertAssist.Settings.Secrets`,
`AllbertAssist.Security.Policy`, or skill registry modules directly.

The Security section should include stable DOM IDs for:

- `security-status`
- `security-permission-defaults`
- `security-safety-floors`
- `security-skill-trust-summary`
- `security-secret-status`
- `security-redaction-posture`
- `security-future-boundaries`
- permission forms named `permission-<permission-name>-form`, such as
  `permission-command-execute-form`

The effective security status rows are read-only in v0.05. Permission setting
controls are editable because they are Settings Central settings. Settings
edits can tighten policy immediately, but they cannot relax Security Central's
built-in safety floors. Future confirmation and audit workflows should define
their own write paths.

## Compatibility Requirements

Existing callers must keep working:

- actions that pattern match `permission_decision.decision`
- actions that call `PermissionGate.allowed?/1`
- actions that call `PermissionGate.response_status/1`
- runner metadata that copies `permission_decision`
- trace rendering that inspects a decision map
- CLI and LiveView output that use current action response shapes

Compatibility tests should fail if v0.05 removes these fields:

- `permission`
- `decision`
- `reason`
- `requires_confirmation`
- `source`

## Verification Sketch

Implementation should verify:

- old PermissionGate decisions remain behavior-compatible
- widened decisions include risk, confirmation, redaction, audit, trace,
  context, policy, and trust-boundary metadata
- raw secrets are never rendered in security status, traces, audits, action
  metadata, lifecycle signals, CLI, LiveView, logs, or tests
- untrusted skills and unknown actions do not gain permission from metadata
- unknown permissions and execution modes deny by default
- Settings Central can tighten policy, but built-in safety floors prevent
  premature execution
- `record_trace` receives only redacted security metadata
- `/settings` renders a Security or Permissions section backed by registered
  settings actions for writes and the registered security status action for
  read-only effective status
- no v0.05 code path adds shell, script, package-install, external network, or
  online import execution

Closeout `rg` checks should look for:

```sh
rg -n "PermissionGate\\.authorize\\(" apps/allbert_assist/lib
rg -n "Security\\.Policy|permissions\\." apps/allbert_assist/lib apps/allbert_assist_web/lib
rg -n "api_key|secret|token|password|credential" apps/allbert_assist/lib apps/allbert_assist_web/lib
rg -n "command_execute|external_network|run_shell|script|package|import" apps/allbert_assist/lib
```

## User Verification Notes

Before release tagging, operators should verify with a temporary Allbert home:

- `mix allbert.security status` shows effective Security Central decisions,
  safety floors, skill trust summary, secret status, redaction posture, and
  future inert boundaries without raw secrets.
- `mix allbert.settings set permissions.command_execute allowed` writes the
  Settings Central default, and a following `mix allbert.security status` still
  shows command execution effectively denied by the v0.05 safety floor.
- `/settings` renders the Security & Permissions section with editable
  permission defaults and read-only effective Security Central status.
- A command-execution request remains denied, and any trace contains
  `## Security Metadata` with redacted decision metadata.
