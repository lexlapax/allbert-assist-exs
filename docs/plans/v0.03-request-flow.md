## Boundary

v0.03 is compatibility and importability only.

Agent Skills are portable manifests, instructions, and resources. A `SKILL.md`
file or bundled script never grants execution authority. v0.03 may parse,
validate, discover, trust, list, read, activate, and trace skills, but it must
not execute scripts, shell commands, package installs, network calls, or Jido
actions because of skill metadata.

v0.04 owns action-backed execution. The expected v0.04 path is:

```text
activated/trusted skill
  -> Allbert metadata contract
  -> action runner
  -> registered Jido action
  -> permission gate
  -> result plus action lifecycle signals
  -> trace
```

## Discovery Flow

Implemented in Milestone 3.

Implemented scan scopes:

1. built-in skills packaged with the app, for example
   `Application.app_dir(:allbert_assist, "priv/skills")`
2. project native skills at `./.allbert/skills/`
3. project interoperable skills at `./.agents/skills/`
4. user native skills at `<ALLBERT_HOME>/skills/`
5. user interoperable skills at `~/.agents/skills/`
6. imported cache skills at `<ALLBERT_HOME>/cache/skills/`
7. operator-configured extra roots from `skills.scan_paths`

Implemented behavior:

- Use `AllbertAssist.Paths` for Allbert Home-derived user and cache paths.
- Use Settings Central for extra scan paths, disabled/enabled skills, trust
  decisions, and imported cache policy.
- Search only bounded skill directories for immediate child folders containing
  an exact `SKILL.md`.
- Project skills are pending or hidden until the project root is trusted.
- User-owned skills can enter the catalog when valid and enabled.
- Duplicate names resolve deterministically; untrusted project duplicates do
  not override trusted user or built-in skills.
- Built-in names are reserved in v0.03.
- Imported cache skills are disabled unless
  `skills.imported_cache_policy: enabled_manual_trust` and the individual skill
  is enabled.
- Until M4 lands the built-in `SKILL.md` pack, a temporary
  `:built_in_legacy` bridge exposes the v0.01 static capabilities through the
  registry with inert legacy contracts. When any built-in `priv/skills`
  declarations are present, the bridge is not used.

Settings Central changes implemented in M3:

- `skills.scan_paths` is writable as a string list
- `skills.trusted_project_roots` is writable as a string list
- `skills.enabled` is writable as a string list
- `skills.disabled` is writable as a string list
- `skills.imported_cache_policy` is writable as `disabled` or
  `enabled_manual_trust`

## Parser And Diagnostics Flow

Implemented in Milestone 2.

Parser input:

```text
skill-root/
  SKILL.md
  scripts/
  references/
  assets/
```

Implemented parser modules:

- `AllbertAssist.Skills.Parser`
- `AllbertAssist.Skills.AgentSkillSpec`
- `AllbertAssist.Skills.Resource`

Public parser functions:

- `AllbertAssist.Skills.Parser.parse_dir/1` returns
  `{:ok, %AllbertAssist.Skills.AgentSkillSpec{}}` for loadable skill
  directories or `{:error, diagnostics}` for fatal parser errors.
- `AllbertAssist.Skills.Parser.parse_many/1` returns
  `%{specs: specs, diagnostics: diagnostics}` and reports duplicate names
  without crashing.

Implemented parser steps:

1. read `SKILL.md`
2. split YAML frontmatter from markdown body
3. parse frontmatter using the existing YAML dependency
4. require `name` and `description`
5. preserve optional `license`, `compatibility`, `metadata`, and
   `allowed-tools`
6. inventory `scripts/`, `references/`, and `assets/`
7. return an `AgentSkillSpec` plus diagnostics

Resource inventory entry:

```elixir
%AllbertAssist.Skills.Resource{
  path: "references/example.md",
  kind: :reference,
  byte_size: 1234,
  sha256: "..."
}
```

Diagnostic shape:

```elixir
%{
  severity: :error | :warning | :info,
  code: :invalid_yaml,
  message: "...",
  path: ".../SKILL.md",
  field: "description",
  value: "optional inspected value"
}
```

Implemented diagnostics:

- missing or empty `description`: skip the skill
- unparseable YAML: skip the skill
- missing `SKILL.md`: skip the skill
- missing or empty `name`: skip the skill
- invalid `name`: compatibility warning when the skill can still be safely
  identified
- parent-directory mismatch: compatibility warning
- oversized description or compatibility field: compatibility warning
- unknown top-level frontmatter fields: warning and preserve in
  `external_fields`
- unknown Allbert metadata: warning, not execution authority
- bundled scripts: inventory plus an informational diagnostic, never execute
- hidden resource files: skip by default
- duplicate skill names through `parse_many/1`: warning diagnostics

Test fixtures live under
`apps/allbert_assist/test/support/fixtures/skills/` and cover valid standard
skills, valid Allbert metadata, invalid YAML, missing required fields,
duplicate names, resources, and scripts.

Verification:

```sh
mix test apps/allbert_assist/test/allbert_assist/skills/parser_test.exs
```

Result on 2026-05-02: 7 tests, 0 failures.

M3 routes these parser diagnostics into `AllbertAssist.Skills.diagnostics/1`
without crashing the assistant loop.

## Registry Flow

Implemented in Milestone 3.

Implemented module split:

- `AllbertAssist.Skills`: small facade that keeps existing action call sites
  stable. Implemented in M3.
- `AllbertAssist.Skills.Registry`: discovery, precedence, trust, enablement,
  and lookup. Implemented in M3.
- `AllbertAssist.Skills.AgentSkillSpec`: parsed standard skill manifest.
  Implemented in M2.
- `AllbertAssist.Skills.Resource`: parsed resource inventory entry.
  Implemented in M2.
- `AllbertAssist.Skills.Parser`: parser and validator for Agent Skill
  directories. Implemented in M2.
- `AllbertAssist.Skills.Skill`: normalized registry record.
  Implemented in M3.
- `AllbertAssist.Skills.CapabilityContract`: inert v0.03 Allbert metadata
  draft. Implemented in M3.

Implemented facade API:

- `AllbertAssist.Skills.list(context \\ %{})`
- `AllbertAssist.Skills.get(name, context \\ %{})`
- `AllbertAssist.Skills.read(name, context \\ %{})`
- `AllbertAssist.Skills.diagnostics(context \\ %{})`
- `AllbertAssist.Skills.activate(name, context \\ %{})` returns
  `{:error, :activation_not_available}` until M5.

Implemented skill kinds:

- `:instruction`: context only.
- `:workflow`: context plus resources, no execution.
- `:capability_candidate`: metadata maps to actions but remains inert in
  v0.03.
- `:native_action`: built-in skill wrapper around a registered Allbert action;
  descriptive only in v0.03.
- `:external_candidate`: discovered/importable but not trusted or executable.

Implemented model-facing catalog:

- include only trusted and enabled skills
- include name, description, kind, source scope, and activation handle
- exclude disabled, invalid, untrusted, or denied skills entirely so the model
  does not waste turns trying to activate them

Built-in names:

- `direct-answer`
- `append-memory`
- `read-recent-memory`
- `list-skills`
- `read-skill`
- `plan-shell-command`
- `external-network-request`

Snake-case aliases such as `append_memory` may be accepted at public read and
activation boundaries, but registry records store canonical kebab names.

## List, Read, And Activate Flow

Planned runtime pipeline:

```text
skill directory
  -> SKILL.md parser
  -> AgentSkillSpec
  -> AllbertSkill registry record
  -> trusted model-facing catalog
  -> activate_skill action
  -> wrapped instructions plus resource inventory
  -> trace metadata
```

Implemented actions:

- `list_skills`: implemented in M3; lists trusted registry skills.
- `read_skill`: implemented in M3; returns one trusted skill declaration and
  diagnostics by name.
- `activate_skill`: implemented in M5; returns wrapped instructions, resource
  inventory, inert capability contract metadata, diagnostics, and a read-only
  permission decision.

`activate_skill` is the only v0.03 activation path exposed to the runtime and
model-facing catalog. It:

- verify the skill is trusted and enabled
- load the full `SKILL.md` body on demand
- wrap the instructions with source, trust, diagnostics, and resource inventory
- include Allbert metadata as inert context
- refuse to execute scripts or external tools
- write trace metadata when tracing is enabled

Activation output:

```elixir
%{
  name: "append-memory",
  title: "Append Memory",
  kind: :native_action,
  source_scope: :built_in,
  trust_status: :trusted,
  instructions: "...markdown body...",
  resource_inventory: [],
  capability_contract: %{status: :draft, actions: ["append_memory"]},
  diagnostics: [],
  permission_decision: %{permission: :read_only, decision: :allowed}
}
```

## Trace Flow

When tracing is enabled, skill list/read/activate turns should include:

- selected skill name
- source scope and source path or redacted path when needed
- trust status
- skill kind
- activation mode
- diagnostics
- resource inventory counts and paths
- inert capability contract summary when present
- permission decision, normally `:read_only` for v0.03 activation

`AllbertAssist.Trace` should render skill metadata explicitly, not only inside
the inspected `actions` block.

Implemented in M5: traces include a `Skill metadata` summary line in the
runtime turn header and a dedicated `## Skill Metadata` section rendered from
the selected action's `skill_metadata` map.

Traces must not include raw secrets and must not claim that skill activation
executed scripts, shell commands, package installs, network calls, or Jido
actions.

## skills.sh And Import Flow

v0.03 local operation must not depend on skills.sh.

Implemented v0.03 local operation:

- scan local directories and skill folders already installed by other tools
- document skills.sh search/detail/audit/import as future adapters
- do not call skills.sh, npm, git, external package installers, or network
  APIs from v0.03 skill activation

Future confirmation-gated import flow:

1. search local skills first
2. ask before external search
3. use `Req` for skills.sh APIs only after confirmation
4. consider audit status, install count, source, duplicate state, and
   relevance
5. write imported files only after confirmation
6. keep imported skills disabled until parsed, validated, and trusted

## Third-Party Skill Installation

v0.03 reads standard Agent Skill folders that already exist locally. A skill is
available for discovery when it is an immediate child directory containing an
exact `SKILL.md` file under one of the scanned parent directories.

Supported local parent directories:

- built-in: `Application.app_dir(:allbert_assist, "priv/skills")`
- project native: `./.allbert/skills/`
- project interoperable: `./.agents/skills/`
- user native: `<ALLBERT_HOME>/skills/`
- user interoperable: `~/.agents/skills/`
- imported cache: `<ALLBERT_HOME>/cache/skills/`
- operator-configured extras from `skills.scan_paths`

Operator notes:

- Project skills are pending until the current project root is listed in
  `skills.trusted_project_roots`.
- Imported cache skills remain disabled unless
  `skills.imported_cache_policy` is `enabled_manual_trust` and the individual
  skill name appears in `skills.enabled`.
- `skills.disabled` hides a skill even when it is otherwise trusted.
- `skills.scan_paths`, `skills.trusted_project_roots`, `skills.enabled`, and
  `skills.disabled` are string lists in Settings Central.
- `skills.imported_cache_policy` accepts `disabled` or `enabled_manual_trust`.
- skills.sh search, detail, audit, and import remain future confirmation-gated
  adapters. v0.03 does not add credentials, API settings, external search, or
  installation commands for skills.sh.
