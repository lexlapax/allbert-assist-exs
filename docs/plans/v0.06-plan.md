# Allbert v0.06 Action-Backed Allbert Skills

## Purpose

v0.06 takes the Agent Skills substrate from v0.03 and wires a trusted subset of
skills into Allbert's registered Jido action model through Security Central.

v0.03 answers: "Can Allbert discover, validate, trust, read, and activate
standard skills?"

v0.04 answers: "Do internal runtime requests converge on signals, agents, and
Jido action boundaries without wrapping pure helper modules?"

v0.05 answers: "Can Allbert make security decisions with one shared policy,
risk, trust, redaction, audit, and trace vocabulary?"

v0.06 answers: "Can a trusted skill safely map to a known Allbert action, carry
the right security contract, and leave a trace?"

## Expected Inputs

- v0.03 skill registry, resource inventory, trust policy, and activation traces
  exist.
- v0.04 action runner, Boundary Actions rule, internal `record_trace` action,
  and action lifecycle metadata exist.
- v0.05 Security Central exists for policy resolution, decision shape, risk,
  redaction, audit metadata, and compatibility with `PermissionGate`.

## Expected Direction

- Validate Allbert metadata overlays into executable capability contracts only
  when they reference registered Jido actions and known permission classes.
- Reject unknown actions, unknown permissions, unknown confirmation policies,
  module loading, shell/script execution, package installs, and external
  network access.
- Use the existing action lifecycle runner that emits
  `allbert.action.requested` and `allbert.action.completed`.
- Have the intent agent call actions through the runner and attach selected
  skill, contract, and Security Central decision metadata.
- Wire action-backed built-in skills through conservative deterministic routing
  while preserving v0.01 safety behavior.
- Add a local skill creation and validation helper for standard `SKILL.md`
  directories that can reference only existing registered action names.

## Non-Goals

- No arbitrary script execution.
- No auto-converting skill folders or bundled scripts into Elixir modules.
- No runtime loading of Elixir modules from arbitrary skill folders.
- No external-network skill search or installation.
- No shell command execution.
- No autonomous skill creation.
- No broad intent-engine enrichment.

## Definition Of Done

v0.06 is done when:

- Trusted Allbert capability skills map only to known Jido actions.
- Skill contracts map only to known permission classes and Security Central
  policy.
- Action execution goes through a common lifecycle runner.
- Traces include selected skill, contract, action, security decision, and
  outcome.
- The first skill creation and validation helper exists.
- v0.01 safety boundaries still hold.
