# Allbert v0.06 Action-Backed Allbert Skills

## Status

Status: in implementation after the v0.05 release and v0.06 architecture
readiness sweep on 2026-05-02. Milestones 1 through 5 are complete and
tested.

v0.06 is not release-complete until Milestone 6 closeout gates pass. Update
`docs/plans/v0.06-request-flow.md` during each implementation milestone as
planned module names become actual module names.

## Purpose

v0.06 takes the Agent Skills substrate from v0.03 and wires a trusted subset of
skills into Allbert's registered Jido action model through Security Central.

v0.03 answers: "Can Allbert discover, validate, trust, read, and activate
standard skills?"

v0.04 answers: "Do internal runtime requests converge on signals, agents, and
Jido action boundaries without wrapping pure helper modules?"

v0.05 answers: "Can Allbert make security decisions with one shared policy,
risk, trust, redaction, audit, and trace vocabulary?"

v0.06 answers: "Can a trusted skill safely map to a known Allbert action, carry
the right security contract, run through the action runner, and leave a trace?"

## Readiness Decision

v0.06 is ready to implement once this planning pass lands. The implementer
should not reopen the big product decisions about format, execution boundary,
trust, permission authority, or confirmation timing.

Implementation may make small naming adjustments if the code strongly prefers
them, but the behavioral contract is fixed:

- Agent Skills remain standard `SKILL.md` directories.
- Allbert metadata validates against registered actions and known permission
  classes.
- Valid metadata never grants permission by itself.
- Action invocation goes through `AllbertAssist.Actions.Runner.run/3`.
- Security decisions come from Security Central.
- v0.06 adds no script, shell, package-install, external-network, online-import,
  or autonomous skill-creation power.

No additional external research is required for this milestone. v0.06 depends
on the local v0.03 Agent Skills contract, the v0.04 action runner contract, and
the v0.05 Security Central decision contract.

## Architecture Readiness Audit

Decision: v0.06 is implementation-ready to take to completion if the
implementation follows the milestones in this plan and updates
`docs/plans/v0.06-request-flow.md` as each milestone lands.

No open architecture blocker remains in the plan. The work is Jido-centric and
consistent with `AGENTS.md` and `DEVELOPMENT.md` because it keeps the runtime
shape explicit:

- user-facing input still enters through `AllbertAssist.Runtime.submit_user_input/1`
  or a lower registered action/context boundary
- user input remains observable through Allbert signals
- `AllbertAssist.Agents.IntentAgent` decides and routes; it does not own
  side-effect logic
- runtime-facing capabilities resolve through `AllbertAssist.Actions.Registry`
- effectful, security-relevant, or observable work runs through registered
  Jido actions with schemas and structured results
- action invocation uses `AllbertAssist.Actions.Runner.run/3` so lifecycle
  signals, runner metadata, permission decisions, redaction, and traces stay
  consistent
- Security Central remains the policy/risk/redaction/audit evaluator at the
  action boundary
- skill metadata, model output, YAML, `allowed-tools`, scripts, and markdown
  never grant permission or execute by themselves
- pure parsing, validation, schema normalization, formatting, and storage
  helpers may stay plain Elixir behind the action boundary
- CLI, LiveView, jobs, and future channels must stay thin and call runtime,
  signal, agent, or action boundaries rather than owning skill, settings,
  security, memory, or trace semantics
- any new operator-supplied settings belong in Settings Central, not
  `config.exs`
- any durable local runtime data written by v0.06 must derive from
  Allbert Home, with tests using temporary homes or temp-specific roots
- request-flow docs, traces, and operator output must remain redacted and
  markdown-inspectable

Implementation should pause and update this plan, and add or update an ADR if
needed, before doing any of the following:

- adding a new execution mode
- adding a new permission class without Settings Central and Security Central
  coverage
- bypassing `Actions.Registry` or `Actions.Runner.run/3`
- moving security policy into a skill, LiveView, Mix task, or agent route
- letting a skill folder name an arbitrary Elixir module to execute
- adding script, shell, package-install, external-network, or online-import
  execution before the later roadmap milestones that own those powers
- storing durable skill/helper data outside Allbert Home

## Vision And Roadmap Alignment

The vision says skills are user-readable bundles around executable
capabilities, while actions are the validated side-effect boundary. v0.06 is
the first milestone where those two layers meet:

- skills explain capability intent, workflow, provenance, memory effects, trace
  effects, and expected permissions
- actions perform bounded work through schemas, registered modules, lifecycle
  signals, runner metadata, and observable results
- Security Central evaluates actor, channel, session, selected skill, selected
  action, permission, trust, risk, redaction, audit, and policy

The roadmap remains consistent:

- v0.06 binds trusted skills to already registered actions.
- v0.07 adds pending confirmation workflow.
- v0.08 adds confirmed shell execution through a sandbox.
- v0.09 adds confirmed skill script execution through that sandbox.
- v0.10 adds confirmed external adapters, package-install actions, and online
  skill import.
- v0.11 broadens intent selection over risky execution-aware capabilities.

This means v0.06 may make existing safe capabilities skill-backed, but it must
not sneak later execution milestones forward.

## Boundary Contract

v0.06 promotes skill metadata from inert contract data to validated action
binding data. It does not make `SKILL.md`, scripts, YAML, or model output
executable authority.

Validation and execution are separate:

```text
SKILL.md metadata
  -> parsed CapabilityContract
  -> action and permission validation
  -> executable contract summary or diagnostics
```

```text
user input
  -> Runtime.submit_user_input/1
  -> IntentAgent deterministic route
  -> selected trusted skill and validated contract
  -> Security Central context and decision
  -> Actions.Runner.run/3
  -> registered Jido action
  -> action lifecycle signals
  -> trace with skill, contract, security, action, and outcome metadata
```

The selected skill should travel in runner context as structured metadata, not
as raw markdown authority. The selected action should still be resolved by
`AllbertAssist.Actions.Registry`, and the action itself should still call
Security Central through `PermissionGate.authorize/2` or the Security facade.

## Scope Rules

- Built-in `:native_action` skills are the first action-backed execution
  surface.
- Trusted user, project, or configured skills may be validated as capability
  candidates, but validation alone must not create broad model-driven routing.
- A trusted non-built-in capability may run in v0.06 only when an explicit,
  deterministic router binding exists for its action and parameter shape.
- Multi-action workflow contracts may validate for diagnostics, but v0.06
  should not execute multi-step workflows. They remain future work for
  execution-aware intent and workflow planning.
- Skill activation remains progressive disclosure. `activate_skill` still
  loads instructions and resource inventory; it does not run the activated
  skill's declared action.
- Existing v0.01 safety behavior remains visible: shell command requests plan
  only, and external network requests report confirmation requirements without
  making calls.

## Expected Direction

- Add canonical action capability metadata for registered actions. The metadata
  should include action name, module, required permission, exposure
  (`:agent` or `:internal`), execution mode, and whether the action is
  available for skill-backed routing.
- Validate Allbert metadata overlays into executable capability contracts only
  when they reference registered Jido actions and known permission classes.
- Consume Security Central's selected skill trust/provenance and safety-floor
  behavior instead of letting capability metadata grant permission directly.
- Reject unknown actions, unknown permissions, unknown confirmation policies,
  module loading, shell/script execution, package installs, and external
  network adapter execution.
- Preserve the v0.04 action lifecycle runner that emits
  `allbert.action.requested` and `allbert.action.completed`.
- Have the intent agent route existing deterministic capability requests through
  selected skill contracts before calling the runner.
- Attach selected skill, validated contract, action capability metadata, and
  Security Central decision metadata to traces.
- Add local skill validation and creation helpers for standard `SKILL.md`
  directories that can reference only existing registered action names and
  known permission classes.

## Contract Validation Rules

A v0.06 executable contract is valid only when:

- the skill is trusted and enabled
- the contract status is `:draft` or `:legacy`
- every declared action resolves through `AllbertAssist.Actions.Registry`
- every declared action is explicitly allowed for skill-backed routing
- every declared permission is in
  `AllbertAssist.Security.Policy.permission_classes/0`
- the action metadata and declared permissions agree
- confirmation metadata is known, for example `not_required`,
  `future_confirmation_required`, or `required`
- the contract does not declare script execution, shell execution, package
  installation, external network execution, module loading, or online import as
  granted authority
- resource references are inventory metadata only

The standalone validation helper may report a local skill contract as
structurally valid before it is trusted or enabled, but it must also report
`execution_eligible?: false` or equivalent until registry policy allows the
skill to enter the trusted catalog.

An invalid contract should not hide the skill if it is otherwise trusted and
enabled. It should keep the skill inspectable, add diagnostics, and leave it
activation-only or instruction-only until fixed.

## Initial Skill-Backed Action Surface

The first v0.06 surface is the current built-in skill pack:

| Skill | Action | Contract permission | v0.06 behavior |
| --- | --- | --- | --- |
| `direct-answer` | `direct_answer` | `read_only` | answer without side effects |
| `append-memory` | `append_memory` | `memory_write` | write markdown memory through the action |
| `read-recent-memory` | `read_recent_memory` | `read_only` | read recent markdown memory |
| `list-skills` | `list_skills` | `read_only` | list trusted registry skills |
| `read-skill` | `read_skill` | `read_only` | read one trusted skill declaration |
| `plan-shell-command` | `plan_shell_command` | `command_plan` | plan only; command execution remains denied |
| `external-network-request` | `external_network_request` | `external_network` | report confirmation need; no adapter call |

This table intentionally does not imply every registered action needs a
catalog skill in v0.06. Settings, security-status, trace-recording, and future
internal actions may stay action-only until there is a user-facing reason to
wrap them as skills.

## Non-Goals

- No arbitrary script execution.
- No shell command execution.
- No skill script execution.
- No package installation.
- No external network adapter calls.
- No online skill search, audit, or import.
- No auto-converting skill folders or bundled scripts into Elixir modules.
- No runtime loading of Elixir modules from arbitrary skill folders.
- No autonomous skill creation.
- No broad intent-engine enrichment.
- No multi-action workflow execution.
- No confirmation queue or approval UI; v0.07 owns that.

## Definition Of Done

v0.06 is done when:

- trusted built-in Allbert capability skills map only to known registered Jido
  actions
- skill contracts map only to known Security Central permission classes
- invalid contracts are visible through diagnostics but cannot execute
- existing deterministic runtime capabilities execute through selected skill
  contracts and the common action lifecycle runner
- Security Central receives selected skill, contract, action, actor, channel,
  session, and request context
- traces include selected skill, contract, action, security decision, risk,
  policy, runner metadata, and outcome
- the first local skill validation and creation helper exists without adding
  executable scripts, generated modules, package installs, or network adapters
- v0.01 safety boundaries still hold
- docs and request-flow maps describe the implemented behavior
- focused tests and the milestone warning gate pass
