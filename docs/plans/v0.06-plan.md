# Allbert v0.06 Local Execution Sandbox And Shell Adapter

## Purpose

v0.06 adds Allbert's first real local execution path: confirmed shell command
execution through registered Jido actions, bounded sandbox policy, permission
checks, and inspectable traces.

This comes after action-backed skills and confirmations. Shell execution must
not enter as model authority, `SKILL.md` authority, or an unchecked helper. It
is an ordinary Allbert capability: selected by intent, represented as a known
action, authorized by the permission gate, confirmed by the operator, executed
inside policy, and traced.

## Expected Inputs

- v0.04 action-backed skill contracts and action lifecycle runner exist.
- v0.05 pending confirmation requests can approve, deny, and resume actions.
- Settings Central can provide permission defaults and execution policy.
- Runtime traces can record selected skill, selected action, permission,
  confirmation, and outcome.

## Expected Direction

- Add a local execution policy in Settings Central for allowed working roots,
  default timeout, max output bytes, environment allowlist, and destructive
  command defaults.
- Add `run_shell_command` as a registered Jido action with schema validation,
  permission `:command_execute`, confirmation requirement, stdout/stderr
  capture, exit status, timeout handling, and redaction.
- Keep `plan_shell_command` as the low-risk planning action and make
  `run_shell_command` opt-in through confirmation.
- Execute only after the command is normalized, policy-checked, confirmed, and
  bound to an explicit working directory.
- Record execution metadata in traces and audit records, including redacted
  command summary, working directory, timeout, exit status, output sizes, and
  denial reason when blocked.

## Non-Goals

- No autonomous shell execution.
- No skill script execution; v0.07 owns that.
- No package installs; v0.08 owns that.
- No external network calls unless an existing command performs them inside
  policy and the command was explicitly confirmed.
- No broad terminal session, PTY, or long-running daemon management.
- No bypass around `AllbertAssist.Security.PermissionGate`.

## Definition Of Ready

v0.06 is ready to implement when:

- v0.05 can persist and resolve pending confirmations.
- v0.04 action runner can resume a confirmed action with original context.
- The permission gate can express command execution policy and denial reasons.
- Trace output can represent action lifecycle and confirmation metadata.

## Definition Of Done

v0.06 is done when:

- `run_shell_command` exists only as a registered action.
- Confirmed allowed commands execute with bounded working root, timeout, output
  capture, redaction, and trace metadata.
- Destructive, ambiguous, out-of-root, over-timeout, or denied commands do not
  run and produce clear diagnostics.
- CLI and LiveView can show the confirmation, result, and trace reference.
- Existing `plan_shell_command` safety behavior remains intact.
