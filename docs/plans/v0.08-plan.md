# Allbert v0.08 Local Execution Sandbox And Shell Adapter

## Purpose

v0.08 adds Allbert's first real local execution path: confirmed shell command
execution through registered Jido actions, bounded sandbox policy, Security
Central decisions, and inspectable traces.

Shell execution must not enter as model authority, `SKILL.md` authority, or an
unchecked helper. It is an ordinary Allbert capability: selected by intent,
represented as a known action, evaluated by Security Central, confirmed by the
operator, executed inside policy, and traced.

## Expected Inputs

- v0.05 Security Central policy, risk, redaction, audit, and sandbox policy
  shapes exist.
- v0.06 action-backed skill contracts and action lifecycle runner exist.
- v0.07 pending confirmation requests can approve, deny, expire, audit, and
  resume eligible registered actions through `Actions.Runner.run/3`.
- v0.07 established that approval re-checks Security Central and does not
  bypass safety floors; v0.08 must explicitly replace the `:command_execute`
  safety floor only after the sandbox action is implemented and tested.
- v0.07 approvals that cannot resume a target adapter resolve as
  `adapter_unavailable` with `target_resumed?: false`; v0.08 is the first
  milestone that should add a real confirmed-resume target for shell execution.

## Expected Direction

- Add local execution policy in Settings Central for allowed working roots,
  default timeout, max output bytes, environment allowlist, and destructive
  command defaults.
- Add `run_shell_command` as a registered Jido action with permission
  `:command_execute`, confirmation requirement, stdout/stderr capture, exit
  status, timeout handling, and redaction.
- Keep `plan_shell_command` as the low-risk planning action.
- Execute only after the command is normalized, policy-checked, confirmed, and
  bound to an explicit working directory.
- Record redacted command summary, working directory, timeout, exit status,
  output sizes, and denial reason in traces and audit metadata.
- Consume the v0.07 confirmation id and approval metadata instead of adding a
  shell-specific approval path.
- Treat v0.07 `adapter_unavailable` as the baseline behavior to replace only
  for the new registered shell adapter, after command policy, sandbox limits,
  stored params, confirmation id, target action, and current Security Central
  decision are verified.

## Non-Goals

- No autonomous shell execution.
- No skill script execution; v0.09 owns that.
- No package installs; v0.10 owns that.
- No broad terminal session, PTY, or long-running daemon management.

## Definition Of Done

v0.08 is done when:

- `run_shell_command` exists only as a registered action.
- Confirmed allowed commands execute with bounded working root, timeout, output
  capture, redaction, and trace metadata.
- Destructive, ambiguous, out-of-root, over-timeout, or denied commands do not
  run and produce clear diagnostics.
- Existing `plan_shell_command` safety behavior remains intact.
