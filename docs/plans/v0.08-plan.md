# Allbert v0.08 Local Execution Sandbox And Shell Adapter

## Status

Implementation-ready after the v0.07 confirmation workflow release and tag on
2026-05-02.

v0.08 is the first release that may turn an operator-approved request into a
local process. It must stay deliberately narrow: confirmed shell execution
through a registered Jido action, bounded by Settings Central policy, Security
Central decisions, the v0.07 durable confirmation queue, output redaction, and
trace/audit records.

## Purpose

v0.08 adds Allbert's first real local execution path: confirmed shell command
execution through a policy sandbox. In this release, "sandbox" means an
application-level local process boundary, not OS-level containment.

Shell execution must not enter as model authority, `SKILL.md` authority, raw
prompt text, or an unchecked helper. It is an ordinary Allbert capability:
selected by intent or an operator CLI, represented as a known action, evaluated
by Security Central, confirmed by the operator, executed inside policy, and
traced.

## Research Summary

Elixir and Erlang provide good process lifecycle tools for local command
execution, but BEAM processes and OTP supervision are not security isolation.
`System.cmd/3` and Erlang ports can run external programs with explicit args,
working directory, environment, exit status, and stdout/stderr handling. The
safe Allbert shape is therefore to avoid shell interpolation, run a concrete
executable with a concrete argv list, constrain cwd/env/output/time, and keep
the effect behind a supervised adapter and registered Jido action.

External agent systems point to the same split:

- Hermes treats local terminal execution as host execution protected by
  dangerous-command approval, redacted env passthrough, and optional container
  backends for production or untrusted workloads.
- OpenClaw models exec as policy plus allowlists plus optional approval, with
  strict fallback behavior when no UI can approve and narrower safe-bin rules.
- OpenHands defaults to a Docker sandbox for a broad coding agent, but still
  warns that mounted read-write workspaces can be modified by the agent.

The implication for Allbert: v0.08 should implement a boring, inspectable
Level 1 local sandbox first. Container, Podman, Mac/Linux container, and remote
sandbox backends should be designed for later adapter replacement, not made a
v0.08 dependency.

## Sandbox Levels

Allbert will use explicit sandbox levels so future releases can increase
isolation without changing the action contract.

- Level 0, inert planning: no process is spawned. This is the v0.01 through
  v0.07 `plan_shell_command` posture.
- Level 1, local policy sandbox: a host process may run only through a
  registered action with explicit executable/args, allowed cwd roots, env
  allowlist, timeout, output limits, confirmation, redaction, and trace/audit.
  This is v0.08.
- Level 2, trusted project/process sandbox: still host execution, but with
  stricter per-project roots, per-skill or per-action allowlists, package
  manager constraints, and reusable execution profiles. v0.09 and v0.10 may
  consume this shape.
- Level 3, container sandbox: Docker, Podman, Linux containers, Mac containers,
  or another container backend with explicit bind mounts, user/capability
  constraints, resource limits, and network policy. This is future work, not
  required for v0.08.
- Level 4, remote or microVM isolation: remote builders, cloud sandboxes, or
  microVM-backed execution for hostile code, multi-user deployments, or
  untrusted online imports. This is future work.

## v0.08 Decision

Implement Level 1 only.

The v0.08 runner must launch no shell by default. It must execute an absolute
or resolved executable path plus argv list. `sh -c`, `bash -c`, `zsh -c`,
inline interpreter eval, command substitutions, shell chaining, redirection,
glob expansion, backgrounding, and PTY/session management are out of scope
unless a future action and policy explicitly add them.

Operator-visible distinctions must be crisp. In user language, v0.08 adds
"local shell execution," but internally that means policy-bounded local command
execution:

- Local read-only commands in an allowed working directory, such as `pwd`,
  `ls -la`, `rg TODO .`, `cat README.md`, `head README.md`, `tail README.md`,
  or `wc -l README.md`, should run after policy checks and confirmation when
  their executable, cwd, path operands, env, timeout, and output cap all pass.
- Local project/developer commands, such as `mix test`, `mix compile`, or a
  project-local script, are in scope for the v0.08 framework only when the
  operator explicitly allows the executable/profile in Settings Central. They
  still require confirmation and the same cwd, path, env, timeout, output, and
  blocked-argument checks.
- Local filesystem commands with path-like operands outside
  `execution.local.allowed_roots`, such as `ls -la /etc`, `cat ~/.ssh/config`,
  or `rg token /`, must be denied before execution.
- Network-shaped commands or interpreter-eval requests, such as
  `curl https://example.com`, `wget`, `python -c`, `node -e`, or asking Allbert
  to read a URL from the internet, must not run through the v0.08 shell
  adapter. v0.10 owns the first confirmed `Req`-based network adapters.

Containers do not belong in v0.08 because they add installation assumptions,
cross-platform path and filesystem semantics, daemon availability, image
management, mount policy, and false confidence when the project directory is
mounted read-write. The implementation should, however, introduce an adapter
boundary so a future container backend can replace the local process backend
without changing confirmation, action, trace, or Security Central semantics.

## Expected Inputs

- v0.05 Security Central policy, risk, redaction, audit, and sandbox policy
  shapes exist.
- v0.06 action-backed skill contracts and action lifecycle runner exist.
- v0.07 pending confirmation requests can approve, deny, expire, audit, and
  resume eligible registered actions through `Actions.Runner.run/3`.
- v0.07 established that approval re-checks Security Central and does not
  bypass safety floors.
- v0.07 approvals that cannot resume a target adapter resolve as
  `adapter_unavailable` with `target_resumed?: false`; v0.08 replaces that
  baseline only for the new registered shell adapter after policy and sandbox
  checks pass.

## Expected Architecture

### Command Spec

Add a structured command spec behind the action boundary. The durable request
and runner params should store redacted, normalized fields, not a shell string:

- executable name or path
- resolved executable path, when available
- argv list
- cwd
- stdin mode, default `none`
- timeout ms
- output limit bytes
- env allowlist and resolved env summary
- path operand summary, including whether path-like args resolve inside
  `execution.local.allowed_roots`
- command class, such as `read_only`, `mutating`, `interpreter`, `network`, or
  `destructive`
- command profile, when a non-default local executable is allowed by operator
  policy
- policy decision and denial reason

The first implementation may derive command class through conservative rules
and allowlists; ambiguous commands must be denied or require future policy, not
guessed into execution.

### Settings Central

Add an `execution` settings namespace for Level 1 policy:

- `execution.local.enabled`, default `false` until the milestone flips it after
  tests are in place.
- `execution.local.allowed_roots`, default empty. Operators must explicitly add
  project/workspace roots for host execution.
- `execution.local.allowed_commands`, default narrow read-only command names
  such as `pwd`, `ls`, `find`, `rg`, `cat`, `sed`, `head`, `tail`, and `wc`.
- `execution.local.command_profiles`, default empty. Operators may add
  explicitly named local developer commands, such as `mix test`, only with
  command class, cwd/path rules, env policy, timeout/output caps, and
  confirmation behavior recorded in Settings Central.
- `execution.local.blocked_arg_patterns`, including in-place editing,
  destructive find flags, inline interpreter eval, shell `-c`, command
  substitution, backgrounding, and path writes to sensitive roots.
- `execution.local.require_path_operands_in_allowed_roots`, default `true` and
  non-bypassable in v0.08 for commands that accept filesystem operands.
- `execution.local.default_timeout_ms`, default `5000`.
- `execution.local.max_timeout_ms`, default `30000`.
- `execution.local.max_output_bytes`, default `65536`.
- `execution.local.env_allowlist`, default `PATH`, `LANG`, `LC_ALL`, and
  `MIX_ENV`; no provider keys or token-like vars by default.
- `execution.local.require_confirmation`, default `true` and non-bypassable in
  v0.08.

The implementation may choose exact module names, but all operator-supplied
policy belongs in Settings Central and must be explainable through CLI and
LiveView.

### Security Central

`permissions.command_execute` remains denied until `run_shell_command`,
settings validation, local policy checks, confirmation resume, output
redaction, and tests exist.

After the execution action is complete, Security Central may return
`:needs_confirmation` for `:command_execute` when all of these are true:

- local execution is enabled
- the command spec is normalized
- the executable is allowed by default read-only policy or an explicit operator
  command profile
- cwd is under an allowed root
- path-like operands resolve under allowed roots or are omitted
- env passthrough is allowlisted
- timeout and output caps are within policy
- blocked args and destructive floors do not match

Denied safety floors still win over operator approval. Approval is a decision
for one pending request, not a global grant to execute commands.

### Local Runner

Add a local execution adapter behind the registered action. It may use
`System.cmd/3` for the first bounded implementation or Erlang ports if
streaming, richer timeout, or kill semantics require it. In either case:

- do not use `System.shell/2`
- do not build a shell string
- pass argv as a list
- set cwd per command instead of changing global process cwd
- construct env from an allowlist
- capture stdout/stderr without leaking directly to the terminal
- enforce timeout and output caps
- kill or terminate the child on timeout where supported
- return structured status, exit code, signal/timeout status, output sizes,
  redacted preview, and diagnostics

### Registered Action

Add `run_shell_command` as a registered Jido action with permission
`:command_execute`.

The action owns command schema validation, Security Central authorization,
confirmation creation, approval resume validation, local runner invocation,
redaction, trace metadata, and structured result formatting. CLI tasks,
LiveViews, intent agents, and future channels must use the action/runner path
instead of spawning commands themselves.

`plan_shell_command` remains the low-risk planning action and should still be
used when the user asks what could be run rather than asking Allbert to run it.

## Request Flow

Detailed request flow: `docs/plans/v0.08-request-flow.md`.

At a high level:

1. User asks Allbert to run a local command, or an operator CLI builds a command
   spec directly.
2. Runtime routes to `run_shell_command`; planning-only prompts still route to
   `plan_shell_command`.
3. The action normalizes executable, args, cwd, timeout, output limit, and env.
4. Security Central evaluates Level 1 policy.
5. If denied, no confirmation is created and nothing runs.
6. If confirmation is required, v0.07 durable confirmation stores the redacted
   command spec and origin channel.
7. Approval re-checks Security Central with confirmation context.
8. The local runner executes only after the re-check succeeds.
9. Result, output preview, output sizes, exit status, timeout, policy metadata,
   origin/resolver channel, and trace/audit links are recorded.

## Pre-Release Operator Test Matrix

Before v0.08 is marked ready for user testing, run these smokes against a
disposable `ALLBERT_HOME` and a temporary allowed workspace root:

- Allowed local working-directory read: ask Allbert to run `pwd` and `ls -la`
  in the allowed workspace, approve each pending confirmation, and verify the
  resolved result includes capped/redacted output, exit status, sandbox level,
  cwd, and `target_resumed?: true`.
- Allowed local file search/read: create fixture files under the allowed
  workspace, run commands such as `rg fixture .`, `cat README.md`,
  `head README.md`, `tail README.md`, or `wc -l README.md`, and verify path
  operand metadata resolves under the allowed root.
- Allowed operator-profiled local command: add one non-default local developer
  command profile in Settings Central, such as `mix test` against a tiny
  fixture or another harmless project-local executable, then verify it requires
  confirmation, uses the configured timeout/output/env policy, and records the
  command profile in traces/audits.
- Denied unprofiled local command: ask for a local executable that is not in
  default `execution.local.allowed_commands` and not in
  `execution.local.command_profiles`; verify it does not run and the denial
  names the missing command policy.
- Denied out-of-root read: ask Allbert to run `ls -la /etc`,
  `cat ~/.ssh/config`, or `rg token /` and verify no child process runs, no
  confirmation is needed unless policy explicitly chooses confirmation-first
  UX, and the denial says the path operand is outside
  `execution.local.allowed_roots`.
- Denied mutating/destructive local command: ask for `rm`, `mv`, `cp` to an
  out-of-root target, `sed -i`, `find -delete`, or a similar mutating command
  without an explicit v0.08 operator profile; verify it is denied before the
  runner can invoke it.
- Denied shell escape: ask for `sh -c "ls -la"`, `bash -c "ls -la"`, or
  `ls -la && pwd` and verify blocked-arg/shell-string diagnostics.
- Denied network escape: ask for `curl https://example.com`, `wget
  https://example.com`, or `python -c`/`node -e` network code and verify v0.08
  does not use the shell adapter as an internet backdoor.
- External-network placeholder unchanged: ask Allbert to fetch/read a URL
  through the normal prompt path, approve the resulting `external_network_request`,
  and verify it still resolves as `adapter_unavailable` until v0.10.
- Cross-surface consistency: create from CLI, resolve from `/settings`, and
  create from `/settings`/prompt then resolve from CLI where supported; verify
  origin and resolver channel metadata are visible.
- Trace/audit consistency: inspect the trace and markdown audit records for
  command summary, cwd, path operand policy, timeout, output cap, redaction,
  Security Central decisions, confirmation id, and result/denial reason.

## Non-Goals

- No autonomous shell execution.
- No unconfirmed command execution.
- No shell strings, PTY sessions, terminal emulation, command chaining,
  redirection, background processes, long-running daemons, or interactive
  prompts.
- No skill script execution; v0.09 owns that.
- No package installs; v0.10 owns that.
- No external network execution; v0.10 owns that.
- No Docker, Podman, Linux container, Mac container, remote sandbox, or microVM
  backend in v0.08.
- No promise that Level 1 prevents all host filesystem or network effects. It
  is a policy sandbox, not an OS isolation boundary.

## Definition Of Done

v0.08 is done when:

- `run_shell_command` exists only as a registered action.
- Confirmed allowed local commands execute with bounded working root, path
  operand policy, command profile/default allowlist, timeout, output capture,
  redaction, and trace/audit metadata.
- Destructive, ambiguous, out-of-root, over-timeout, blocked-arg, disabled, or
  policy-denied commands do not run and produce clear diagnostics.
- Pre-release testing covers default read-only local commands, explicitly
  profiled local developer commands, unprofiled local commands, out-of-root
  path operands, mutating/destructive commands, shell escapes, and network
  escape attempts.
- Existing `plan_shell_command` safety behavior remains intact.
- Existing external-network confirmations still resolve as
  `adapter_unavailable` until v0.10.
- CLI and `/settings` show the same durable pending/resolved command metadata.
- The release docs explain that v0.08 provides Level 1 local policy sandboxing,
  not container isolation.
