# Allbert v0.01 Implementation Plan

## Purpose

v0.01 turns Allbert from a demo Phoenix/Jido scaffold into the first usable
local assistant loop. The goal is a small, trustworthy runtime that can receive
input, route it through Jido signals, use a primary intent agent, execute a tiny
set of validated actions, record traces, and store user-owned memory in
markdown.

This is not the full personal assistant. It is the smallest slice that proves
the architecture in `docs/plans/allbert-jido-vision.md`.

The current request path is documented in
`docs/plans/v0.01-request-flow.md`.

## Current Baseline

- Phoenix umbrella app with runtime code in `allbert_assist` and web code in
  `allbert_assist_web`.
- `AllbertAssist.Application` already starts `AllbertAssist.Jido`,
  `Jido.Signal.Bus`, Phoenix PubSub, and the repo under OTP supervision.
- `AllbertAssist.Agents.SampleAgent` demonstrates `Jido.AI.Agent`.
- `AllbertAssist.Actions.Multiply` demonstrates `Jido.Action`.
- Model aliases are configured under `:jido_ai` so agent modules can refer to
  aliases like `:local`, `:fast`, `:capable`, and `:thinking`.
- `mix format --check-formatted` and `mix precommit` are clean as of
  Milestone 1.
- `AllbertAssist.Runtime.submit_user_input/1` exists as the first
  signal-first runtime boundary as of Milestone 2, and the `/agent` LiveView
  routes through it instead of calling the sample agent directly.
- `AllbertAssist.Agents.IntentAgent` is the default runtime agent as of
  Milestone 3. It exposes the first v0.01 intent action surface while keeping
  memory persistence and command execution out of scope.
- `AllbertAssist.Security.PermissionGate` is the central permission decision
  point as of Milestone 4. Intent actions include permission decisions in their
  structured results.
- `AllbertAssist.Memory` provides markdown-backed durable memory as of
  Milestone 5. The memory root defaults to `var/allbert/memory` and can be
  overridden with `ALLBERT_MEMORY_ROOT` or application config.

## Non-Goals

- No autonomous shell execution without confirmation.
- No broad skill creation or self-modifying code.
- No embeddings, vector store, or nightly model distillation.
- No browser/search capture yet.
- No messaging channels beyond CLI/REPL and the existing Phoenix surface.
- No complex distributed runtime or multi-node deployment work.

## Milestone Handoff Rule

Every milestone must leave the system in a testable state for the operator. A
milestone is not complete until its automated checks pass and its operator test
can be run from a clean shell with clear pass/fail output or visible UI
behavior.

Each milestone handoff should include:

- Commands the operator can run.
- Any environment variables required.
- The exact prompt or UI action to try.
- The expected response, file, trace, or permission result.
- A note if the milestone intentionally preserves a limited or mocked behavior.

## Target Runtime Contract

Use this contract unless implementation discovers a concrete reason to revise
it and updates this plan first.

- `AllbertAssist.Runtime.submit_user_input/1` is the channel-facing entrypoint.
- It accepts a map with at least:
  - `:text` user input
  - `:channel` such as `:cli`, `:live_view`, or `:test`
  - `:user_id` or `:operator_id`, defaulting to a local operator identity in
    development
  - optional `:metadata`
- It returns `{:ok, response}` or `{:error, reason}`.
- A successful response includes:
  - `:message`
  - `:status` such as `:completed`, `:needs_confirmation`, or `:denied`
  - `:trace_id` when tracing is enabled
  - `:actions` as the actions selected or considered
- CLI/REPL and LiveView must use this same function by the end of v0.01.

## Implementation Order

### Milestone 1: Clean Baseline

Status: complete and tested.

Create a safe starting point for implementation.

Implementation tasks:

- Fix existing formatter issues reported by `mix format --check-formatted`.
- Run `mix precommit` and fix any issues that are not unrelated external
  failures.
- Keep the sample agent/action available until the replacement path is working.
- Add or update docs only if the formatter/precommit work reveals a needed
  clarification.

Acceptance criteria:

- `mix format --check-formatted` passes.
- `mix precommit` passes.
- No behavior changes are introduced beyond formatting and clearly documented
  cleanup.

Operator test:

- Run `mix format --check-formatted`.
- Run `mix precommit`.
- Start the app with `mix phx.server`.
- Visit `http://localhost:4000/agent` and confirm the existing agent page still
  renders; if model credentials are not configured, the page may show a
  startup or provider error after submitting, but the page itself must load.

### Milestone 2: Signal-First Runtime Boundary

Status: complete and tested.

Introduce one public runtime entrypoint that receives user input and emits Jido
signals instead of letting channels call agents directly.

Implemented shape:

- `AllbertAssist.Runtime.submit_user_input/1` accepts atom-keyed or
  string-keyed request maps.
- The runtime validates non-empty text, normalizes channel/operator metadata,
  emits `allbert.input.received`, invokes the configured agent runner, emits
  `allbert.agent.responded`, and returns `{:ok, response}` or
  `{:error, reason}`.
- The default runner now uses `AllbertAssist.Agents.IntentAgent`. Tests can
  still inject a deterministic runner through application config so runtime and
  LiveView behavior stay stable without depending on a live model.
- The `/agent` LiveView submits through the runtime boundary and displays the
  response message, status, and response signal id.
- The remaining signal names are declared through `Runtime.signal_types/0` for
  follow-on milestones even when they are not yet persisted or routed through a
  durable trace store.

Implementation tasks:

- Add a runtime boundary module, tentatively `AllbertAssist.Runtime`, with a
  function that accepts a user input request and returns a structured response.
- Define the first signal names under an `allbert.*` namespace:
  - `allbert.input.received`
  - `allbert.agent.responded`
  - `allbert.action.requested`
  - `allbert.action.completed`
  - `allbert.memory.appended`
  - `allbert.trace.recorded`
- Route the existing LiveView flow through the runtime boundary.
- Keep the runtime boundary synchronous at first so tests and UI integration
  are straightforward.

Acceptance criteria:

- A caller can submit text through the runtime boundary and receive a response.
- The LiveView no longer needs to know how to start or call the agent directly.
- Signal names are documented in module docs or a small docs section.

Operator test:

- Run `iex -S mix`.
- Call:

      AllbertAssist.Runtime.submit_user_input(%{
        text: "Say hello from the runtime boundary.",
        channel: :test,
        operator_id: "local"
      })

- Confirm the return shape is `{:ok, response}` with `response.message`.
- Start `mix phx.server`, visit `http://localhost:4000/agent`, submit the same
  prompt, and confirm the page displays the runtime response.
- Confirm logs or trace output show an `allbert.input.received` signal. If trace
  persistence is not implemented yet, logging the signal is acceptable for this
  milestone.

### Milestone 3: Primary Intent Agent

Status: complete, tested, and operator-verified.

Replace the sample demo posture with a real first Allbert agent.

Implemented shape:

- `AllbertAssist.Agents.IntentAgent` uses `Jido.AI.Agent` and declares the
  first six intent tools:
  - `direct_answer`
  - `append_memory`
  - `read_recent_memory`
  - `list_skills`
  - `read_skill`
  - `plan_shell_command`
- M3 routing is deterministic over those named `Jido.Action` modules so the
  operator loop is fast, local, and testable. The supervised Jido agent
  definition and tool surface remain ready for later LLM-driven selection once
  permissions, memory, and traces are stronger.
- `append_memory` and `read_recent_memory` select the memory capability but do
  not persist durable markdown yet; that remains Milestone 5.
- `plan_shell_command` never executes shell commands. Command-like prompts
  return `:denied` with an inert plan/safety note.
- `AllbertAssist.Skills` holds the temporary static skill declarations until
  the v0.02 skill registry replaces them with file-backed declarations.
- The module/function request path is documented in
  `docs/plans/v0.01-request-flow.md`.

Implementation tasks:

- Add `AllbertAssist.Agents.IntentAgent` using `Jido.AI.Agent`.
- Give the agent a narrow system prompt:
  - understand user intent
  - select from available actions
  - ask for confirmation when an action is sensitive
  - answer plainly when no action is required
- Start with a tiny action set:
  - respond/echo or direct answer support
  - append memory
  - read recent memory
  - list available skills
  - read a skill declaration
  - plan a shell command without executing it
- Keep `SampleAgent` only as a temporary compatibility/demo module or remove it
  once the new loop is covered by tests.

Acceptance criteria:

- The primary agent can answer a plain text prompt.
- The primary agent can choose a memory action for an explicit memory request.
- The primary agent can describe which action it would use for a capability
  request without executing unsafe work.

Operator test:

- Ensure the required model provider environment variables are configured, or
  switch the configured model alias to an available local provider.
- Run `iex -S mix`.
- Submit:

      AllbertAssist.Runtime.submit_user_input(%{
        text: "Hello Allbert. What can you do right now?",
        channel: :test,
        operator_id: "local"
      })

- Confirm the answer names only v0.01-safe capabilities.
- Submit:

      AllbertAssist.Runtime.submit_user_input(%{
        text: "Remember that I prefer short implementation updates.",
        channel: :test,
        operator_id: "local"
      })

- Confirm the response says the memory action was selected or will be used.
- Submit:

      AllbertAssist.Runtime.submit_user_input(%{
        text: "Run rm -rf /tmp/example",
        channel: :test,
        operator_id: "local"
      })

- Confirm Allbert refuses to execute and offers only a plan or explanation.

### Milestone 4: Actions And Permission Gate

Status: complete, tested, and operator-verified.

Make executable capabilities explicit, typed, and inspectable.

Implemented shape:

- `AllbertAssist.Security.PermissionGate` defines the v0.01 permission classes:
  - `:read_only`
  - `:memory_write`
  - `:command_plan`
  - `:command_execute`
  - `:external_network`
- The gate returns a structured decision with `:permission`, `:decision`,
  `:reason`, `:requires_confirmation`, and `:source`.
- Read-only actions, memory-write intent, and command planning return
  `:allowed`.
- Command execution returns `:denied`.
- External network access returns `:needs_confirmation`.
- Every M4 intent action includes `permission_decision` in the top-level action
  result and inside the selected action metadata.
- `plan_shell_command` records both the allowed `:command_plan` decision and
  the denied requested `:command_execute` decision.
- `external_network_request` was added as an explicit inert action so
  network-shaped prompts do not silently pass through as plain answers.

Implementation tasks:

- Implement each capability as a named `Jido.Action` with schema validation,
  description, structured result, and structured error behavior.
- Add a permission gate module, tentatively
  `AllbertAssist.Security.PermissionGate`.
- Use these initial permission classes:
  - `:read_only`
  - `:memory_write`
  - `:command_plan`
  - `:command_execute`
  - `:external_network`
- Treat `:command_execute` and `:external_network` as denied or
  confirmation-required in v0.01.
- Include the permission decision in the action result or trace data.

Acceptance criteria:

- Read-only actions can run without confirmation.
- Memory-write actions are recorded and traceable.
- Command execution is not available silently.
- Permission decisions are covered by tests.

Operator test:

- Run the permission tests for the security boundary.
- In `iex -S mix`, submit a read-only prompt such as:

      AllbertAssist.Runtime.submit_user_input(%{
        text: "List the skills you can inspect.",
        channel: :test,
        operator_id: "local"
      })

- Confirm the response status is `:completed`.
- Submit a memory-write prompt and confirm the response status is `:completed`
  or `:needs_confirmation`, depending on the implemented default.
- Submit an external-network or command-execution prompt and confirm the status
  is `:needs_confirmation` or `:denied`, never silent execution.

### Milestone 5: Markdown Memory v0

Status: complete and tested.

Create the first durable memory system with markdown as the source of truth.

Implemented shape:

- `AllbertAssist.Memory` owns the markdown source of truth.
- Memory root precedence is:
  - `config :allbert_assist, AllbertAssist.Memory, root: "..."`
  - `ALLBERT_MEMORY_ROOT`
  - `var/allbert/memory` under the project root
- `Memory.ensure_root!/0` creates the initial category directories:
  - `notes`
  - `preferences`
  - `traces`
  - `skills`
- `Memory.append/1` writes one markdown file with timestamp, category, source
  signal, actor, agent, channel, summary, and body.
- `Memory.recent/1` reads markdown entries and ranks them with simple query
  token matching.
- `append_memory` now writes durable markdown and returns the file path in
  action metadata.
- `read_recent_memory` now returns markdown-backed entries instead of a planned
  placeholder.

Implementation tasks:

- Add a configurable memory root. Use a development default that is inside the
  app's local writable data area or a clearly documented local path.
- Create initial memory categories:
  - `notes`
  - `preferences`
  - `traces`
  - `skills`
- Implement append and recent-read actions for markdown memory.
- Store entries with timestamp, source signal, actor/agent, summary, and body.
- Keep retrieval simple: recent entries and explicitly selected files only.

Acceptance criteria:

- Asking Allbert to remember a fact creates a markdown entry.
- Asking Allbert what it recently remembers reads from markdown-backed memory.
- Memory files remain human-readable and portable.

Operator test:

- Set the memory root to a temporary directory documented by the implementation,
  for example `ALLBERT_MEMORY_ROOT=/tmp/allbert-memory-v001`.
- Submit:

      AllbertAssist.Runtime.submit_user_input(%{
        text: "Remember that my planning docs should be implementation-ready.",
        channel: :test,
        operator_id: "local"
      })

- Inspect the memory root and confirm a markdown file was created under the
  expected category.
- Open the markdown file and confirm it includes timestamp, source signal,
  actor/agent, summary, and body.
- Submit:

      AllbertAssist.Runtime.submit_user_input(%{
        text: "What do you remember about my planning docs?",
        channel: :test,
        operator_id: "local"
      })

- Confirm the response uses the markdown-backed memory entry.

### Milestone 5.1: Personal Preference Heuristics

Status: planned.

Make the first personal facts and preferences usable without requiring every
memory-worthy statement to start with "remember that".

Current baseline:

- `AllbertAssist.Memory` can append and read markdown entries from the memory
  root.
- The current categories are `notes`, `preferences`, `traces`, and `skills`.
- `append_memory` writes durable markdown and returns the created file path.
- `read_recent_memory` reads markdown-backed entries with simple query-token
  ranking.
- `AllbertAssist.Agents.IntentAgent` selects actions deterministically for the
  v0.01 operator loop.

Non-goals:

- No embeddings or vector store.
- No profile database or compiled persona document.
- No automatic overwriting or deletion of older facts.
- No extraction of highly sensitive personal data unless the prompt explicitly
  asks Allbert to remember it.
- No LLM-only memory extraction. The first implementation should be predictable
  and covered by tests.
- No broad natural-language preference ontology. Start with high-value personal
  facts and communication preferences.

Heuristic scope:

- Identity statements:
  - "my name is Sandeep"
  - "I am Sandeep"
  - "I'm Sandeep"
  - "call me Sandeep"
- Communication preferences:
  - "I prefer short updates"
  - "I like concise handoffs"
  - "Please keep responses brief"
  - "I want concrete next steps"
- Local working preferences:
  - "my timezone is America/Los_Angeles"
  - "I usually test in the browser first"
  - "I prefer docs before implementation"
- Identity recall:
  - "what is my name?"
  - "who am I?"
  - "what should you call me?"
- Preference recall:
  - "what do you know about my preferences?"
  - "how should you update me?"
  - "how should you communicate with me?"
- Local working context recall:
  - "what timezone am I in?"
  - "how do I like to test?"
  - "what do you remember about my planning preference?"

Implementation tasks:

- Add deterministic heuristics for basic identity statements, communication
  preferences, and local working preferences.
- Route recognized personal statements through the existing markdown-backed
  `append_memory` action.
- Route direct recall questions through the existing markdown-backed
  `read_recent_memory` action with expanded query terms.
- Keep all stored memory user-readable under the current `preferences`
  category.
- Avoid silent storage of sensitive personal data unless explicit memory intent
  is present.
- Add small deterministic predicate/extraction helpers in
  `AllbertAssist.Agents.IntentAgent` or a nearby private helper module:
  - `personal_fact_statement?/1`
  - `personal_preference_statement?/1`
  - `personal_recall_request?/1`
  - an extraction helper that returns `{category, summary, body, tags}` or an
    equivalent plain map.
- Include enough text in the stored markdown body to keep the memory
  user-readable without a compiled profile:
  - the original user statement
  - the inferred fact or preference
  - the heuristic family, such as `identity.name` or
    `communication.preference`
  - the source signal id when available.
- Route direct identity recall with expanded query terms. For example, "what is
  my name?" should query terms like `name call me identity`.
- Prefer a concise answer that cites the remembered markdown entry in plain
  language. It does not need formal source citation yet, but action metadata
  should still carry the memory path.

Sensitive data boundaries:

- Allowed without extra confirmation:
  - preferred name or display name
  - timezone
  - communication style
  - testing or planning preference
  - non-sensitive local working preferences
- Require explicit "remember" wording or defer to a future confirmation flow:
  - email addresses
  - phone numbers
  - home address
  - government identifiers
  - financial, health, legal, or credential-like data
  - secrets, API keys, tokens, passwords, or private keys
- If a prompt looks sensitive and lacks explicit memory intent, Allbert should
  answer normally or say it will not store that detail automatically.

Acceptance criteria:

- "my name is Sandeep" creates a markdown memory entry.
- "what is my name?" recalls the name from markdown memory.
- "I prefer short implementation updates" creates a preference memory.
- "how should you update me?" recalls that preference from markdown memory.
- Sensitive-looking personal data is not silently stored.
- Automated tests cover the identity statement, preference statement, recall
  query, and sensitive-data boundary.
- The operator can test every behavior through `AllbertAssist.Runtime` and the
  existing `/agent` LiveView.

Operator test:

- Use a temporary memory root so the test is inspectable and disposable:

      export ALLBERT_MEMORY_ROOT=/tmp/allbert-m5-1-memory
      rm -rf "$ALLBERT_MEMORY_ROOT"
      iex -S mix

- In IEx, submit:

      AllbertAssist.Runtime.submit_user_input(%{
        text: "my name is Sandeep",
        channel: :test,
        operator_id: "local"
      })

- Confirm the response status is `:completed`, the response says the personal
  fact or preference was saved, and a markdown file exists under
  `/tmp/allbert-m5-1-memory/preferences`.
- Submit:

      AllbertAssist.Runtime.submit_user_input(%{
        text: "what is my name?",
        channel: :test,
        operator_id: "local"
      })

- Confirm the response identifies the remembered name as Sandeep and action
  metadata includes at least one markdown memory result.
- Submit:

      AllbertAssist.Runtime.submit_user_input(%{
        text: "I prefer short implementation updates.",
        channel: :test,
        operator_id: "local"
      })

- Submit:

      AllbertAssist.Runtime.submit_user_input(%{
        text: "how should you update me?",
        channel: :test,
        operator_id: "local"
      })

- Confirm the response recalls the short implementation update preference from
  markdown memory.
- Start `mix phx.server`, visit `http://localhost:4000/agent`, repeat the same
  four prompts, and confirm the visible responses match the IEx behavior.

Test strategy:

- Add focused `IntentAgent` tests for:
  - identity statement routing to `append_memory`
  - preference statement routing to `append_memory`
  - direct identity recall routing to `read_recent_memory`
  - preference recall routing to `read_recent_memory`
  - sensitive-looking statement without explicit memory intent is not stored
- Add action-level assertions only where M5.1 changes action parameters or
  memory category selection.
- Use temporary memory roots in tests through application config.
- Keep LiveView tests focused on runtime integration and visible status, not
  exact prose.

### Milestone 6: Trace, Cost, And Diagnostics

Make the loop inspectable before making it more autonomous.

Implementation tasks:

- Record each assistant turn as a trace entry:
  - input signal
  - selected agent
  - selected action or no-action decision
  - permission decision
  - model alias
  - approximate token/cost metadata when available
  - final response
  - errors, retries, and denied actions
- Add a tracing toggle via config or runtime setting.
- Store traces in markdown or a markdown-friendly JSONL block format.
- Prefer plain data structures that can later be indexed.

Acceptance criteria:

- A successful turn creates an inspectable trace when tracing is enabled.
- A denied or confirmation-required action is visible in the trace.
- Trace writing failures do not crash the user-facing loop.

Operator test:

- Enable tracing using the documented config or runtime toggle.
- Submit one successful memory prompt and one denied command-execution prompt.
- Inspect the trace location and confirm each turn has a trace entry.
- Confirm the successful trace includes input signal, selected agent, selected
  action, model alias, final response, and trace id.
- Confirm the denied trace includes the permission decision and no command
  execution result.
- Temporarily point the trace path at an unwritable location and confirm the
  user-facing response still returns with a diagnostic error recorded in logs.

### Milestone 7: First Usable CLI And LiveView Loop

Expose the same runtime through terminal and web surfaces.

Implementation tasks:

- Add a simple CLI or Mix task for sending one prompt to Allbert.
- Update the existing LiveView to call the same runtime boundary as the CLI.
- Present response, permission status, and trace reference in the UI.
- Keep UI changes functional and modest; richer product design can come later.

Acceptance criteria:

- CLI/REPL demo:
  - ask Allbert to remember something
  - ask what it remembers
  - ask what action it would use for a command-like request
  - inspect the generated trace
- LiveView demo performs the same loop through the web UI.
- Both paths use the same runtime boundary and core actions.

Operator test:

- Run the documented CLI or Mix task with:

      remember that I like concise milestone handoffs

- Run the CLI or Mix task again with:

      what do you remember about milestone handoffs?

- Confirm the response cites or summarizes the markdown memory entry.
- Run the CLI or Mix task with:

      run a destructive shell command

- Confirm the response is `:needs_confirmation` or `:denied`.
- Start `mix phx.server`, visit `http://localhost:4000/agent`, and repeat the
  same three prompts.
- Confirm CLI and LiveView both produce trace references and use the same memory
  root.

## Test Strategy

- Unit-test every `Jido.Action` for valid input, invalid input, success, and
  structured error behavior.
- Unit-test permission decisions for each permission class.
- Test the runtime boundary without relying on `Process.sleep/1`; use monitored
  processes or synchronous calls where possible.
- Test markdown memory writes in a temporary directory.
- Test trace writes separately from agent reasoning so diagnostics remain
  deterministic.
- Add one integration test that exercises input -> runtime -> agent/action ->
  memory/trace -> response.
- Add LiveView tests only after the runtime boundary stabilizes; assert against
  key DOM IDs, not raw HTML strings.

## Definition Of Done

v0.01 is done when:

- `mix precommit` passes.
- The v0.01 CLI/REPL demo works.
- The LiveView demo uses the same runtime core.
- Markdown memory can append and read recent entries.
- Tracing records successful and denied/confirmation-required turns.
- Command execution is not silently available.
- The roadmap is updated with v0.01 status and the next milestone candidate.
