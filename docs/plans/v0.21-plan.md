## Purpose

v0.21 upgrades Allbert's memory layer from append-only markdown files into an
operator-reviewable, correctable, pruneable, promotable, and indexed store.
Markdown remains the inspectable source of truth; derived artifacts — compiled
indexes and category summaries — must be clearly marked as derived and
rebuildable on demand.

The upgrade matters because by v0.21 Allbert has been writing memory entries
for months across categories, channels, jobs, execution traces, and StockSage
domain context. Entries accumulate without review, stale entries survive, and
keyword ranking over raw files is weak. v0.21 adds the operator controls and
compiled runtime views that the vision calls for before memory retrieval can
be trusted as an intent signal.

v0.21 also wires the compiled index into the v0.19 intent engine as a new
`:memory` candidate source, fulfilling the plug-in point described in ADR 0019:
*"Future milestones (v0.21 memory retrieval, v0.24 workspace shell) can plug
into the engine as additional candidate sources or scoring signals without
changing the authority model."*

## Readiness Decision

v0.21 is ready for implementation when the plan, request-flow doc, roadmap, and
ADRs agree on these points:

- `AllbertAssist.Memory` is the existing markdown-backed append/read store.
  v0.21 adds review, correction, pruning, promotion, indexing, and retrieval
  improvement on top of the same markdown source of truth.
- Markdown files under `<ALLBERT_HOME>/memory/` remain the source of truth for
  durable memory. Derived indexes and summaries are rebuildable artifacts.
- Memory deletion requires an explicit confirmed action guarded by the v0.07
  confirmation workflow. Pruning without confirmation is not allowed.
- Promotion of conversation threads to memory entries is an explicit confirmed
  action. Automatic promotion is not allowed.
- Memory candidates fed to the v0.19 intent engine are proposal data under ADR
  0019 invariants. They can influence ranking but cannot grant permissions,
  bypass Security Central, or invent actions.
- Settings Central remains authoritative for memory policy. v0.21 makes
  existing `memory.*` settings writable and adds new ones.
- v0.21 does not add embeddings, vector databases, or semantic similarity
  search. Index-based retrieval uses scored keyword + recency + category
  weighting over the compiled JSON/ETS index.

## Architecture Readiness Audit

The v0.21 plan is consistent with the current architecture:

- ADR 0006 keeps Security Central as the permission, risk, redaction, and audit
  boundary. Memory write (`append`, `update`, `delete`) still requires the
  `:memory_write` permission class. Memory read (`list`, `read`, `search`)
  uses `:read_only`. Promotion from threads uses `:memory_write` and
  `:confirmation_decide`.
- ADR 0007 requires runtime-facing, effectful, observable operations to enter
  through registered actions. All v0.21 memory workflows — review, correct,
  delete, prune, promote, compile-index, summarize — are registered Jido
  actions invoked through `AllbertAssist.Actions.Runner.run/3`.
- ADR 0008 defines durable confirmation requests. `delete_memory_entry` and
  `promote_conversation_turn` both use the existing v0.07 confirmation
  workflow. v0.21 does not extend or modify confirmation semantics.
- ADR 0014 defines `user_id` as the identity boundary. All memory operations
  include `user_id`; entries written under one user are not returned to
  another. The default is `"local"`.
- ADR 0019 defines memory candidates as an allowed plug-in source for the
  intent engine. Memory candidates must follow all seven ADR 0019 invariants.
  In particular, memory source in candidates is `:memory`, not `:authority`.
  Memory signals can boost action scores but cannot grant permissions or bypass
  Security Central.

No new ADR is required for v0.21. ADR 0019 already provides the invariant
boundary for memory as intent signal. If a future implementation needs to
change those invariants, that requires a revision to ADR 0019 or a superseding
ADR, not a v0.21-specific one.

## Vision And Roadmap Alignment

The vision says memory should be markdown-first and runtime-compiled: the
source of truth is inspectable files; runtime speed comes from compiled and
indexed forms. Memory pruning should be deliberate. Selective promotion from
threads should be an explicit confirmed action.

v0.21 directly implements those postures:

- Markdown entries remain inspectable and portable.
- The compiled index is a derived artifact, clearly labeled, rebuildable.
- Pruning and deletion are explicit confirmed actions.
- Promotion from threads is confirmed and scoped to explicitly selected turns.

Roadmap dependency order is intact:

- v0.01 supplied the initial markdown memory append/read path.
- v0.02 supplied Settings Central and `ALLBERT_HOME`.
- v0.07 supplied the confirmation workflow that memory delete and promote use.
- v0.12 supplied `Thread` and `Message` conversation history that promotion
  reads as source material.
- v0.13 supplied scheduled jobs that can invoke memory review or compile-index
  on a cadence.
- v0.19 supplied the intent engine plug-in point and ADR 0019 invariants for
  memory candidates.
- v0.20 StockSage domain `memory_entries` records are available as structured
  context distinct from markdown Allbert memory.

v0.21 must not pull later work forward. It does not add embeddings, vector
stores, semantic retrieval, automatic memory promotion pipelines, AI-generated
rewriting of memory content, native workspace shell surfaces, canvas components,
or StockSage analysis execution.

## Guiding Rules

1. Markdown is the source of truth. The compiled index and category summaries
   are derived artifacts. If they diverge from the markdown source, the
   markdown wins. Derived files must include a machine-readable header marking
   them as derived with a rebuild timestamp.

2. Deletion, pruning, and promotion are confirmed. `delete_memory_entry`,
   `prune_memory_entries`, and `promote_conversation_turn` go through the
   v0.07 confirmation workflow when their respective
   `memory.delete_requires_confirmation`,
   `memory.prune_requires_confirmation`, and
   `memory.promotion_requires_confirmation` settings are true (all default
   true). No memory entry is deleted silently.

3. Memory write operations use `:memory_write` permission. Read operations use
   `:read_only`. The index compile action is `:read_only` (reads markdown,
   writes a derived artifact). Index write is not a memory-write operation;
   it is a derived-artifact rebuild.

4. Memory candidates respect ADR 0019. Memory entries that appear in intent
   candidates are proposal data. They do not grant permissions, bypass Security
   Central, or authorize actions. Their `source` field is `:memory`. Their
   scores influence ranking only.

5. User isolation is strict. All memory operations that accept `user_id` filter
   by it. The CLI defaults to `"local"` only at the operator boundary. Actions
   never default `user_id` internally.

6. Promotion from threads is explicit and bounded. The action accepts a
   specific `thread_id` and `message_id`. It does not crawl or summarize entire
   thread histories automatically. The operator selects which turn to promote.

7. Pruning is non-destructive first. `prune_memory_entries` dry-run mode
   identifies candidates by staleness, duplication, or flag status without
   writing. Write mode requires confirmation per the settings default.

8. Derived artifacts are rebuildable. `compile_memory_index` and
   `summarize_memory_category` always produce output that matches the current
   markdown state. If the derived file is stale or absent, the fallback is
   direct markdown reads.

9. No automatic memory mutations. Scheduled jobs may invoke review or
   compile-index through the action boundary, but they may not auto-delete,
   auto-promote, or auto-prune without explicit operator confirmation.

## Memory Entry Format

The existing format from v0.01:

```markdown
# Memory: {summary}

- Timestamp: {iso8601}
- Category: {notes|preferences|traces|skills}
- Source signal: {signal_id}
- Actor: {user_id}
- Agent: {agent_module}
- Channel: {channel}

## Body

{body text}
```

v0.21 adds an optional `## Review` section and `## Derived` marker:

```markdown
# Memory: {summary}

- Timestamp: {iso8601}
- Category: {notes|preferences|traces|skills}
- Source signal: {signal_id}
- Actor: {user_id}
- Agent: {agent_module}
- Channel: {channel}

## Body

{body text}

## Review

- Reviewed: {iso8601 | ""}
- Reviewed by: {user_id | ""}
- Status: {kept|flagged|prune_nominated}
- Correction note: {optional operator note}
```

The `## Review` section is added by `review_memory_entry` and
`update_memory_entry`. Entries without a `## Review` section are treated as
unreviewed. The parser must not fail on entries missing the section.

Derived files (index, summaries) include a header:

```
# DERIVED - DO NOT EDIT
# Source: <ALLBERT_HOME>/memory/{category}/*.md
# Rebuilt: {iso8601}
```

## Settings Central Schema

v0.21 updates and extends the `memory.*` namespace:

Existing keys made writable:
- `memory.review_cadence` — `enum ["manual","daily","weekly"]`, default
  `"manual"`, `writable?: true`. When `"daily"` or `"weekly"`, a scheduled
  job invokes `compile_memory_index` on that cadence through the v0.13
  scheduler. Setting this value to `"manual"` pauses the managed cadence job
  when one exists.
- `memory.auto_promote_sensitive_entries` — boolean, default `false`,
  `writable?: true`. Kept false by default; true allows future scheduled
  promotion (still requires confirmation per `promotion_requires_confirmation`).
- `memory.retention_policy` — extend allowed values to
  `["preserve_markdown", "prune_traces_after_30d", "prune_traces_after_90d"]`,
  `writable?: true`.

New keys:
- `memory.delete_requires_confirmation` — boolean, default `true`, writable.
  When false, `delete_memory_entry` deletes without a confirmation record.
- `memory.prune_requires_confirmation` — boolean, default `true`, writable.
  When false, `prune_memory_entries` write mode archives prune candidates
  immediately without a confirmation record. It is intentionally separate from
  `memory.delete_requires_confirmation` so single-entry delete and bulk prune
  can have different operator friction later.
- `memory.promotion_requires_confirmation` — boolean, default `true`, writable.
  When false, `promote_conversation_turn` writes the memory entry directly.
- `memory.max_entries_per_category` — integer, default `500`, writable.
  `prune_memory_entries` dry-run warns when a category exceeds this limit.
- `memory.index_enabled` — boolean, default `true`, writable.
  When false, `search_memory` falls back to direct markdown reads.
- `memory.max_index_entries` — integer, default `1000`, writable.
  The compiled index keeps at most this many entries per index rebuild.

### `AllbertAssist.Memory.Entry`

A normalized struct representing a parsed markdown entry:

```elixir
%AllbertAssist.Memory.Entry{
  path: String.t(),
  category: atom(),
  timestamp: String.t(),
  source_signal_id: String.t(),
  actor: String.t(),
  agent: String.t(),
  channel: String.t(),
  summary: String.t(),
  body: String.t(),
  review_status: :unreviewed | :kept | :flagged | :prune_nominated,
  reviewed_at: String.t() | nil,
  reviewed_by: String.t() | nil,
  correction_note: String.t() | nil
}
```

### `AllbertAssist.Memory.Index`

Compiled query interface over a JSON file derived from markdown entries:

```elixir
@spec build(String.t()) :: {:ok, map()} | {:error, term()}
@spec load(String.t()) :: {:ok, map()} | {:error, term()}
@spec query(map(), String.t(), keyword()) :: {:ok, [Memory.Entry.t()]}
@spec stale?(String.t()) :: boolean()
```

The index JSON is stored at `<ALLBERT_HOME>/memory/.index.json` and includes
bounded entry metadata (path, category, summary, timestamp, review_status,
keyword tokens) with a `derived_at` field. The full body is not stored in the
index; `query/3` returns paths; callers load entries from markdown when needed.

### `AllbertAssist.Memory.Compiler`

Builds the derived index and category summaries:

```elixir
@spec compile_index(String.t(), keyword()) :: {:ok, map()} | {:error, term()}
@spec summarize_category(String.t(), atom(), keyword()) :: {:ok, String.t()} | {:error, term()}
```

The index file is written to `<ALLBERT_HOME>/memory/.index.json`.
Category summaries are written to
`<ALLBERT_HOME>/memory/{category}/.summary.md`, clearly marked as derived.

### `AllbertAssist.Memory.Review`

Parse and write `## Review` sections in entry markdown:

```elixir
@spec parse_review(String.t()) :: %{review_status: atom(), reviewed_at: String.t() | nil, ...}
@spec write_review(String.t(), map()) :: {:ok, String.t()} | {:error, term()}
@spec prune_candidates(String.t(), keyword()) :: {:ok, [Memory.Entry.t()]}
```

`prune_candidates` identifies entries with `review_status: :prune_nominated`,
entries older than the retention policy window (for traces category), and
entries over `memory.max_entries_per_category`. Returns candidates; does not
delete.

### `AllbertAssist.Memory.Promotion`

Validates and constructs a markdown memory entry from a conversation turn:

```elixir
@spec from_thread_message(String.t(), String.t(), String.t(), map()) ::
        {:ok, map()} | {:error, term()}
```

Accepts `user_id`, `thread_id`, `message_id`, and optional metadata. Reads
the message content through `AllbertAssist.Conversations`. Validates that the
message belongs to `user_id`. Returns attrs ready for `Memory.append/1`. Does
not write directly; the calling action writes after confirmation.

## Actions

All new actions go under `AllbertAssist.Actions.Memory.*`.

### Read Actions (`:read_only`)

- `list_memory_entries` — list entries with optional filters: `category`,
  `review_status`, `limit`, `since`. Returns bounded `Entry` maps. Paths are
  included for downstream `read_memory_entry` calls.
- `read_memory_entry` — read one entry by `path`. Returns full `Entry` map
  including body and review section.
- `search_memory` — search entries by text query using the compiled index when
  available, falling back to direct markdown reads. Accepts `query`, `limit`,
  `categories`. Returns ranked `Entry` maps with match reasons.
- `list_memory_category_summary` — read the derived `.summary.md` for a
  category if it exists. Returns the summary text and its `derived_at`
  timestamp.

### Review And Correction Actions (`:memory_write`)

- `review_memory_entry` — set `review_status` on an entry: `kept`, `flagged`,
  or `prune_nominated`. Writes the `## Review` section to the markdown file.
  Requires `:memory_write`. Does not delete.
- `update_memory_entry` — correct an entry's body or summary. Writes the
  updated content to the markdown file in place. Requires `:memory_write`.
  Records correction note in `## Review`. Does not change the entry path or
  timestamp.

### Deletion And Pruning Actions (`:memory_write` + confirmation)

- `delete_memory_entry` — delete one entry by path. When
  `memory.delete_requires_confirmation` is true (default), creates a durable
  confirmation record through the v0.07 workflow and returns
  `:needs_confirmation`. On approval, the markdown file is moved to
  `<ALLBERT_HOME>/memory/deleted/<YYYY-MM>/` rather than hard-deleted.
  Entries are never permanently removed without operator review of the
  `deleted/` folder.
- `prune_memory_entries` — in dry-run mode (default), returns a list of
  candidate entries for pruning with reasons (prune-nominated, trace retention
  window, category over limit). In write mode, creates a bulk confirmation
  record listing all candidates when `memory.prune_requires_confirmation` is
  true (default). On approval, or immediately when the setting is false, moves
  each candidate to the `deleted/` archive. Requires `:memory_write`.

### Promotion Action (`:memory_write` + `:confirmation_decide` + confirmation)

- `promote_conversation_turn` — promote one conversation message to a markdown
  memory entry. Accepts `user_id`, `thread_id`, `message_id`, `category`
  (default `:notes`), and optional `summary` override. Validates ownership
  through `AllbertAssist.Conversations`. Constructs the entry body from the
  message content. When `memory.promotion_requires_confirmation` is true
  (default), creates a confirmation record and returns `:needs_confirmation`.
  On approval, calls `Memory.append/1` to write the markdown entry. Never
  promotes automatically.

### Index And Summary Actions (`:read_only`)

- `compile_memory_index` — rebuild the derived memory index from all markdown
  entries. Bounded by `memory.max_index_entries`. Writes the index JSON file
  and returns rebuild statistics: entry count, elapsed time, categories
  covered. The action is `:read_only` because it reads markdown and writes
  only a derived artifact.
- `summarize_memory_category` — generate a derived summary markdown file for
  one category. Writes `<ALLBERT_HOME>/memory/{category}/.summary.md` with
  the derived-file header. Returns summary path and entry count. The summary
  contains an entry table (timestamp, summary, review_status) followed by
  the most recently reviewed kept entries' bodies. Does not use a model; it
  is a compiled view.

## CLI Tasks

All new tasks under `mix allbert.memory`:

```sh
# List entries
mix allbert.memory list [--category notes|preferences|traces|skills] [--status unreviewed|kept|flagged|prune_nominated] [--limit N] [--since YYYY-MM-DD]

# Show one entry
mix allbert.memory show <path>

# Search entries
mix allbert.memory search <query> [--category ...] [--limit N]

# Review
mix allbert.memory review <path> --status kept|flagged|prune_nominated [--note "..."]

# Correct an entry body/summary
mix allbert.memory update <path> --body "..." | --summary "..."

# Delete (creates confirmation when setting is true)
mix allbert.memory delete <path>

# Prune (dry-run by default; --write creates confirmation)
mix allbert.memory prune [--dry-run] [--category ...]

# Promote a conversation turn (creates confirmation when setting is true)
mix allbert.memory promote-turn --thread-id THREAD_ID --message-id MSG_ID [--category notes] [--summary "..."] [--user USER]

# Rebuild index
mix allbert.memory compile-index

# View derived category summary
mix allbert.memory summarize [--category ...]
```

CLI rules:
- All commands call registered actions through `Actions.Runner.run/3`.
- No CLI task reads the markdown store or file system directly.
- Output is redacted, bounded, and stable enough for tests.
- `--user` defaults to `"local"` at the CLI boundary only.
- `mix allbert.memory delete` and `mix allbert.memory prune --write` display the
  confirmation id when a pending record is created.

## Intent Engine Integration

v0.21 adds `:memory` kind candidates to `Engine.collect_candidates/1` as the
plug-in point promised in ADR 0019.

Implementation:

1. When `memory.index_enabled` is true and the compiled index exists and is
   not stale, `Engine.collect_candidates/1` calls `Memory.Index.query/3` with
   the request text and a cap of 10 memory candidates.
2. Each matching memory entry becomes a `Candidate` with:
   - `kind: :memory`
   - `id`: the entry path (bounded)
   - `source: :memory`
   - `status: :candidate`
   - `score`: recency + keyword match score (between 0.1 and 0.5)
   - `reason`: bounded entry summary
   - `trace_metadata`: `%{category, timestamp, review_status, path}` (no body)
3. Memory candidates with `review_status: :flagged` or `:prune_nominated` are
   excluded from collection.
4. ADR 0019 invariants apply without exception:
   - Memory candidates do not grant permissions.
   - Memory candidates do not authorize actions.
   - Memory candidate scores cannot exceed permission floors.
   - Plugin provenance / memory provenance is explainability metadata only.
5. When the index is absent or stale, memory candidate collection is skipped
   silently and the engine proceeds without `:memory` candidates.

## Signals And Trace Metadata

Trace sections added or updated in v0.21:

- `## Memory Review` — appears when a review, update, delete, prune, or
  promote action runs. Includes entry path, category, review_status transition,
  and redacted confirmation id when relevant.
- `## Intent Candidates` — existing section now may include `:memory` candidates
  with category, recency, and match reason; bounded to 5 memory candidates in
  trace output.

Optional lifecycle signals for observability only:
- `allbert.memory.index_compiled` — emitted by `compile_memory_index` with
  entry count, elapsed time.
- `allbert.memory.entry_deleted` — emitted by `delete_memory_entry` approval
  with bounded entry summary and `user_id`. No body content.
- `allbert.memory.turn_promoted` — emitted by `promote_conversation_turn`
  approval with `thread_id`, `message_id`, and new entry summary.

Signals must not include entry body content, raw conversation turns, raw model
output, or `user_id`-scoped content beyond the confirmation context.

## Non-Goals

- No embeddings or vector database.
- No LLM-generated rewriting of memory entry bodies.
- No automatic promotion of conversation history turns to markdown entries
  without explicit operator selection and confirmation.
- No automatic promotion of StockSage domain `memory_entries` records to
  markdown Allbert memory.
- No channel-specific memory stores.
- No deletion or pruning of user-owned markdown entries without confirmation
  (by default).
- No memory mutation from model output or ambient agent activity without an
  explicit registered action call.
- No workspace shell, canvas components, or LiveView memory surfaces (v0.24
  and v0.25).
- No security hardening evals for memory workflows (v0.26).
- No StockSage analysis execution or Python bridge.
