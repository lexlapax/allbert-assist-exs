# Allbert v0.02 Settings Central, Secrets, And Operator Profile

## Purpose

v0.02 introduces Settings Central: one typed, inspectable, auditable control
plane for Allbert-owned user/operator configuration.

v0.02 also introduces Allbert Home: one canonical local directory for durable
runtime data. By default this is `~/.allbert`; operators can override it with
`ALLBERT_HOME`, and `ALLBERT_HOME_DIR` is accepted as a compatibility alias.
Settings, secrets, memory, the local database, user skills, imported caches,
and temporary runtime files should derive from this home unless a specific
test/dev override is provided.

`config.exs` is application and deployment configuration: endpoint setup, repo
setup, dependency configuration, release/runtime bootstrap, and other choices
that should not be edited by the assistant at runtime. It is not the place
where Allbert-owned user settings should live.

Everything the user/operator supplies to Allbert should go through Settings
Central, including API keys. Sensitive values should be handled by Settings
Central's encrypted secret store, not stored as plaintext in `settings.yml` and
not scattered through ad hoc subsystem config.

Settings Central becomes the shared source for:

- operator profile and local preferences
- provider profiles and user-supplied provider credentials
- model profiles and model aliases
- future agent definitions and agent selection policy
- trace defaults and diagnostic verbosity
- model/provider aliases that are safe to expose to the user
- skill scan paths, trust decisions, and enablement
- permission policy defaults
- confirmation behavior
- job defaults and timezone
- channel preferences and enablement
- memory review policy and retention preferences

This needs to exist before declarative agents, skills, confirmations, jobs, and
channels grow larger. Otherwise each subsystem will invent its own settings
surface and future channels will have no common way to inspect or change
behavior.

It also needs to exist before more persistence is added. Without a canonical
home, memory, settings, databases, imported skills, caches, and generated
runtime files will drift into separate path conventions and become hard to
backup, migrate, or inspect.

The v0.02 request flow must be kept current in
`docs/plans/v0.02-request-flow.md`. Every milestone below has a request-flow
handoff requirement.

## Research Grounding

The v0.02 design is grounded in:

- Elixir `Config` and application environment documentation: `config/config.exs`
  is build/application configuration, while `config/runtime.exs` is evaluated
  just before applications start. Both load application environment rather than
  acting as a user-editable domain settings store.
- Phoenix deployment guidance: endpoint host, adapters, release config, and
  deployment-time values belong in runtime/application config.
- Jido AI documentation: agents can be configured with model aliases, tools,
  prompts, iteration limits, timeouts, retries, and LLM options. Allbert should
  use that flexibility through its own provider/model profile layer instead of
  requiring every simple agent to be a compiled Elixir module.
- YAML Elixir documentation: `YamlElixir.read_from_file/2` and
  `YamlElixir.read_from_string/2` parse YAML into maps with string keys and
  structured parse errors.
- ymlr documentation: `Ymlr.document!/2` encodes maps as YAML documents and
  supports sorted map output. v0.02 should add `:yaml_elixir` and `:ymlr` as
  direct dependencies rather than relying on transitive packages or handwritten
  YAML output.

Implementation consequence: use `config.exs` and environment variables for
bootstrap and infrastructure, but store mutable user/operator settings,
provider credentials, provider profiles, and model profiles in
`AllbertAssist.Settings`.

## Design Boundary

Keep a hard distinction:

- `config.exs`, environment variables, and Phoenix runtime config are
  deployment/bootstrap settings. They are read mostly at boot and cover
  infrastructure details such as endpoint, repo, release, dependency, and
  initial path configuration. `ALLBERT_HOME`, `ALLBERT_HOME_DIR`, and the
  secret-store master key are bootstrap inputs.
- `AllbertAssist.Settings` is user/operator domain state. It is readable,
  auditable, channel-addressable, and safe to expose through Allbert actions
  according to permission policy.
- `AllbertAssist.Settings.Secrets` is part of Settings Central. It stores
  user-supplied sensitive values, such as API keys, encrypted at rest and
  exposed only as redacted status or explicit secret references.

Examples that belong in `config.exs`:

- database repo configuration
- Phoenix endpoint configuration
- mailer adapter
- release/runtime bootstrap
- dependency library bootstrap
- Allbert home or specific path overrides
- secret-store master-key bootstrap
- hard default paths for local development

Examples that belong in `AllbertAssist.Settings`:

- `operator.communication_style: concise`
- `operator.timezone: America/Los_Angeles`
- `providers.openai.enabled: true`
- `providers.openai.api_key_ref: secret://providers/openai/api_key`
- `providers.local_ollama.base_url: http://localhost:11434/v1`
- `model_profiles.fast.provider: openai`
- `model_profiles.fast.model: gpt-4o-mini`
- `agents.primary_intent.model_profile: local`
- `runtime.trace_default: denied_only`
- `skills.trusted_project_roots: [...]`
- `permissions.external_network: needs_confirmation`
- `channels.live_view.response_style: concise`
- `jobs.timezone: America/Los_Angeles`
- `memory.auto_promote_sensitive_entries: false`

## Allbert Home Decision

Add `AllbertAssist.Paths` as the single place that resolves Allbert-owned local
paths.

Path precedence:

1. Specific override when one exists, for example `ALLBERT_SETTINGS_ROOT`,
   `ALLBERT_MEMORY_ROOT`, `DATABASE_PATH`, or application config used by tests.
2. `ALLBERT_HOME`.
3. `ALLBERT_HOME_DIR`.
4. `~/.allbert`.

Canonical derived paths:

```text
<ALLBERT_HOME>/
  settings/
    settings.yml
    secrets.yml.enc
    .settings_key
    audit/YYYY-MM.md
  memory/
    notes/
    preferences/
    traces/
    skills/
  db/
    allbert.sqlite3
  skills/
  cache/
    skills/
  tmp/
```

Public API:

- `AllbertAssist.Paths.home/0`
- `AllbertAssist.Paths.ensure_home!/0`
- `AllbertAssist.Paths.settings_root/0`
- `AllbertAssist.Paths.memory_root/0`
- `AllbertAssist.Paths.db_path/0`
- `AllbertAssist.Paths.skills_root/0`
- `AllbertAssist.Paths.cache_root/0`
- `AllbertAssist.Paths.tmp_root/0`

Specific root overrides remain valid for tests, migrations, compatibility, and
operator escape hatches. Normal runtime code should derive paths from
`AllbertAssist.Paths` so new subsystems do not invent their own root
precedence.

Tests and CI must set a temporary `ALLBERT_HOME` or a specific temporary root.
They must never write to a real user's `~/.allbert`.

## Storage Decision

Use one canonical settings file:

```text
<ALLBERT_HOME>/settings/settings.yml
```

Use one encrypted local secret file:

```text
<ALLBERT_HOME>/settings/secrets.yml.enc
```

Use append-only audit records:

```text
<ALLBERT_HOME>/settings/audit/YYYY-MM.md
```

Settings root precedence:

1. `config :allbert_assist, AllbertAssist.Settings, root: "..."`
2. `ALLBERT_SETTINGS_ROOT`
3. `AllbertAssist.Paths.settings_root/0`

Secret-store key precedence:

1. `ALLBERT_SETTINGS_MASTER_KEY`, a base64-encoded 32-byte key.
2. `config :allbert_assist, AllbertAssist.Settings, master_key: "..."`
3. local development key at `<ALLBERT_HOME>/settings/.settings_key`, created
   with mode `0600` only when the app is not running in production.

Production should require an explicit master key source. Local development can
create the `.settings_key` bootstrap file to avoid making the first operator
flow miserable.

YAML parsing rules:

- Use `YamlElixir.read_from_file/2`.
- Do not use `atoms: true`; keep string keys to avoid atom creation from
  user-editable files.
- Treat a missing settings file as an empty user layer over built-in defaults.
- Treat malformed YAML as a structured diagnostic and refuse writes until the
  file is corrected or moved aside by the operator.

Writing rules:

- Validate before writing.
- Use `Ymlr.document!/2` with `sort_maps: true` for deterministic YAML output.
- Write to a temporary file in the same directory.
- Rename the temporary file over `settings.yml`.
- Append an audit record after a successful write.
- Return structured errors without partially updating in-memory state.

Secret writing rules:

- Store sensitive values only in `secrets.yml.enc`.
- Store only `secret://...` references in `settings.yml`.
- Use Erlang/OTP `:crypto` for authenticated encryption, preferably
  AES-256-GCM with a fresh nonce per write.
- Write encrypted secrets atomically, using the same temp-file-and-rename
  pattern as `settings.yml`.
- Never print raw secret values in CLI output, LiveView, traces, audits, or
  action metadata.
- Return redacted status such as `configured`, `missing`, or `invalid_ref`.

Markdown memory remains the source of truth for memories, traces, summaries,
and user-owned narrative context under `<ALLBERT_HOME>/memory` by default.
`settings.yml` is the source of truth for current operational preferences and
policy.

```yaml
  memory_write: "allowed"

  schedule_policy: "operator_approved"
memory:
  review_cadence: "manual"
  auto_promote_sensitive_entries: false
  retention_policy: "preserve_markdown"
```

Allowed v0.02 write keys:

- `operator.display_name`
- `operator.timezone`
- `operator.communication_style`
- `operator.handoff_detail`
- `runtime.trace_default`
- `runtime.diagnostics_verbosity`
- `providers.*.enabled`
- `providers.*.base_url`
- `providers.*.api_key_ref`, only when the referenced secret exists or is
  being written in the same operation
- `model_profiles.*.provider`
- `model_profiles.*.model`
- `model_profiles.*.temperature`
- `model_profiles.*.max_tokens`
- `model_profiles.*.timeout_ms`
- `channels.cli.response_style`
- `channels.live_view.response_style`

Sensitive values are written through secret APIs or provider credential
commands, not through generic `put/3`.

Agent definitions are read-only in v0.02. v0.03 owns the first write path for
declarative agents.

All other namespaces can be read and explained in v0.02, but writes should
return `{:error, {:read_only_setting, key}}` until the relevant milestone
implements the behavior.

## Layering

Settings resolve from explicit layers:

1. Built-in defaults from code.
2. Deployment overrides from application config and environment variables.
3. Operator/global settings from `settings.yml`.
4. Project/workspace settings, reserved for v0.04 skill trust and later.
5. Channel-specific settings.
6. Request/session overrides.

Every resolved setting returns explainability metadata:

```elixir
%{
  key: "operator.timezone",
  value: "America/Los_Angeles",
  source: :operator,
  layers: [
    %{source: :default, value: "UTC"},
    %{source: :operator, value: "America/Los_Angeles"}
  ]
}
```

## Core API

Add `AllbertAssist.Settings` with this public API:

- `root/0`
- `ensure_root!/0`
- `defaults/0`
- `schema/0`
- `safe_write_keys/0`
- `read_user_settings/0`
- `write_user_settings(map(), keyword())`
- `list(namespace_or_opts \\ [])`
- `get(key, context \\ %{})`
- `put(key, value, context \\ %{})`
- `resolve(key, context \\ %{})`
- `explain(key, context \\ %{})`
- `validate(settings_or_key_value, opts \\ [])`
- `list_provider_profiles/0`
- `list_model_profiles/0`
- `resolve_model_profile(name, context \\ %{})`

Add `AllbertAssist.Settings.Secrets` with this public API:

- `put_secret(secret_ref, value, context \\ %{})`
- `get_secret(secret_ref, context \\ %{})`
- `list_secret_status(namespace_or_opts \\ [])`
- `delete_secret(secret_ref, context \\ %{})`
- `redact(key_or_ref, value)`

Use dotted string keys at public boundaries, for example
`"operator.timezone"`. Internally, normalize atom keys only from trusted code.

Return shapes:

- `{:ok, value}`
- `{:ok, resolved_setting}`
- `{:ok, settings_map}`
- `{:error, :not_found}`
- `{:error, {:invalid_setting, key, reason}}`
- `{:error, {:read_only_setting, key}}`
- `{:error, {:settings_parse_failed, reason}}`
- `{:error, {:settings_write_failed, reason}}`
- `{:error, {:secret_not_found, secret_ref}}`
- `{:error, {:secret_write_failed, reason}}`
- `{:error, {:secret_decrypt_failed, reason}}`

## Permission Class

Add these permission classes to `AllbertAssist.Security.PermissionGate`:

- `:settings_write`
- `:settings_secret_write`
- `:settings_secret_read`

v0.02 default:

- safe settings writes are allowed.
- setting user-supplied provider credentials is allowed through explicit
  credential flows.
- raw secret reads are denied except inside trusted runtime code that is
  resolving a provider call.
- deployment config writes are denied.
- unknown settings writes are denied.

All settings actions must include the permission decision in action metadata.

## Auditing

Every successful settings write appends an audit record to
`audit/YYYY-MM.md`.

Audit entry fields:

- timestamp
- key
- old value, redacted when sensitive
- new value, redacted when sensitive
- actor/operator id
- channel
- source signal id when available
- permission decision
- validation result

Sensitive values are expected in v0.02, but only through the Settings Central
secret store. Audits must record secret writes as redacted status changes, for
example `missing -> configured`, never as raw values. Use the redaction helper
for keys containing `secret`, `token`, `password`, `api_key`, `private_key`, or
`credential`, and for every `secret://...` reference.

## Non-Goals

- No remote secrets manager.
- No remote sync.
- No multi-user authorization model.
- No full settings UI polish.
- No arbitrary editing of deployment config.
- No intent-engine rewrite.
- No channel adapters beyond CLI and LiveView.
- No project settings writes until v0.04 or later.
- No declarative agent creation until v0.03.
- No arbitrary agent module loading from settings.
- No raw secret display in any operator surface.

## Definition Of Done

v0.02 is done when:

- `AllbertAssist.Settings` exists as a typed, YAML-backed domain settings
  center.
- `AllbertAssist.Settings.Secrets` exists as an encrypted local secret store
  for user-supplied credentials.
- `AllbertAssist.Paths` exists and makes `ALLBERT_HOME` the canonical root for
  settings, secrets, memory, database, skills, cache, and temporary runtime
  files.
- `config.exs` remains deployment/bootstrap config and is not treated as user
  settings.
- Settings have layered resolution and explainability.
- Settings writes are validated, atomic, and audited.
- Secret writes are encrypted, atomic, audited with redaction, and never exposed
  in CLI, LiveView, traces, or audits.
- Provider and model profiles can be listed and resolved through Settings
  Central.
- Settings actions are exposed through the runtime boundary.
- CLI and LiveView can inspect and update safe settings through the same store.
- `docs/plans/v0.02-request-flow.md` has been updated after each milestone and
  matches the final implementation.
- Declarative agents, skills, confirmations, jobs, channels, memory review, and
  future intent work have a common settings foundation to depend on.
