# Allbert v0.10 External Capability Adapters

## Status

Implementation is complete through M11 and focused verified. v0.10
was reopened after M5 because post-M5 commits added online skill approval
clarity/search fixes and Resource Access Security Posture planning. M6
reconciled that history. M7 implemented the shared resource reference
contract. M8 implemented Settings-backed remembered resource grant storage and
matching. M9 refreshed release docs, future handoffs, and user-testing
instructions. M10 hardened canonical resource identity, local symlink scope
matching, source-profile grant drift, and registry-driven resumable action
metadata. M11 added remembered-grant operator UX and applied remembered grants
to existing v0.10 resource consumers.

After an M9 zoom-out release audit, v0.10 was reopened for M10-M14 closeout
before operator acceptance. These milestones do not move v0.11 channel-native
intent UX into v0.10; they finish the substrate so v0.11 can build on it
without inheriting ambiguous resource identity, unwired grants, or incomplete
skill import consumers.

Release/tag readiness is pending the remaining M12-M14 closeout sequence.
Expected release tag after operator acceptance remains `v0.10`. No v0.10 tag
has been created or pushed yet.

## Purpose

v0.10 adds Allbert's first confirmed external capability adapters:

- bounded HTTP/service calls through `Req`
- confirmed package-install workflows through package-manager profiles
- online Agent Skill search, audit, and import into the disabled imported cache

This release is deliberately after Security Central, confirmation workflow,
Level 1 local execution, and resource-gated skill scripts. By v0.10, Allbert
has enough action, confirmation, policy, redaction, and trace structure to let
external capabilities exist without becoming ambient model authority.

v0.10 does not make Allbert a browser, a crawler, a package manager shell, a
container runtime, or an untrusted-code execution environment. It adds a few
boring registered adapters and proves that external side effects still flow
through the Jido-centered runtime.

Post-M5 planning clarification: v0.10 is also where Allbert names a shared
resource access security posture. v0.08 and v0.09 already implemented local
special cases for host commands and trusted skill script resources. v0.10 adds
the matching network substrate and must now document the common posture without
rewriting those released milestones. Any new implementation work for shared
local/remote resource references, operation classes, approval scopes, or
remembered grants belongs in explicit v0.10 milestones. v0.11 owns the
execution-aware UX and Approval Handoff contract that consumes the posture.

Post-M9 release-audit clarification: M7/M8 created the right substrate, but
the substrate is not yet cohesive enough for release. M10 hardened canonical
resource identity and local scope matching. M11 exposed remembered-grant
operator controls and applied remembered grants to existing v0.10 actions
where policy allows. The remaining closeout now starts with a URI-first
resource identity refactor, then implements direct skill URL/local directory
import as concrete disabled/untrusted resource consumers. v0.11 still owns
channel-native Approval Handoff, execution-aware intent, URL/document summary
or extraction orchestration, and summarizer handoff.

## Research Notes

The implementation plan was checked against current public documentation:

- Req v0.5.17 documents built-in request, response, redirect, retry, Finch, and
  test-stub steps. Redirects and safe transient retries exist by default, so
  v0.10 must make redirect/retry policy explicit instead of inheriting hidden
  behavior. Req.Test provides concurrent-safe stubs and transport-error
  simulation for tests.
  Source: `https://hexdocs.pm/req/Req.Steps.html` and
  `https://hexdocs.pm/req/Req.Test.html`
- OWASP SSRF guidance recommends allowlists, input validation, defense in
  depth, and disabling redirect following when redirect chains could bypass
  validation.
  Source:
  `https://cheatsheetseries.owasp.org/cheatsheets/Server_Side_Request_Forgery_Prevention_Cheat_Sheet.html`
- OWASP path traversal and file upload guidance reinforces the local-resource
  half of the posture: canonicalize paths, accept known-good resource inputs,
  constrain storage/read locations, validate type/extension/signature where
  relevant, and apply size/parser/sandbox caution to untrusted files.
  Sources: `https://owasp.org/www-community/attacks/Path_Traversal` and
  `https://cheatsheetseries.owasp.org/cheatsheets/File_Upload_Cheat_Sheet.html`
- OWASP AI-agent and prompt-injection guidance reinforces that local files,
  fetched documents, API responses, and tool outputs are untrusted data unless
  a bounded action, permission, and approval posture says otherwise. Human
  approval must describe the real resource and operation, not a vague "tool
  call".
  Sources:
  `https://cheatsheetseries.owasp.org/cheatsheets/AI_Agent_Security_Cheat_Sheet.html`
  and
  `https://cheatsheetseries.owasp.org/cheatsheets/LLM_Prompt_Injection_Prevention_Cheat_Sheet.html`
- npm v11 install docs include `--dry-run`, `--package-lock-only`,
  `--ignore-scripts`, `--allow-git`, and related install controls. Lifecycle
  scripts and git dependencies remain risk boundaries that Allbert must make
  explicit.
  Source: `https://docs.npmjs.com/cli/v11/commands/npm-install/`
- pip docs include `--dry-run --report` for installation planning, while
  secure-install docs warn that default pip installs do not protect against
  remote tampering and can involve arbitrary code. Secure pip execution needs
  hash-checking and binary-only policy before it can be enabled broadly.
  Source: `https://pip.pypa.io/en/stable/cli/pip_install/` and
  `https://pip.pypa.io/en/stable/topics/secure-installs/`
- skills.sh currently documents a CLI-driven install flow using `npx skills
  add`, anonymous telemetry for ranking, and a warning that operators should
  review skills before installing. Allbert should not shell out to `npx skills
  add` in v0.10; it should use registered `Req` actions and keep imported
  skills disabled.
  Source: `https://skills.sh/docs`
- The Agent Skills specification keeps `SKILL.md` required and `scripts/`,
  `references/`, and `assets/` optional. v0.10 import should preserve that
  folder contract and feed the existing parser/registry instead of inventing a
  separate skill format.
  Source: `https://agentskills.io/specification`
- SLSA frames supply-chain integrity as a staged set of controls. v0.10 should
  record provenance, digests, source URLs, audit status, and operator decisions
  now, while deeper provenance verification remains future hardening.
  Source: `https://slsa.dev/`
- OpenClaw and Hermes both treat remote skill ecosystems as source and
  lifecycle-management concerns rather than a single marketplace authority.
  Anthropic's skills repository models skills as one plugin payload in a
  broader marketplace manifest. Allbert should therefore treat skills.sh as one
  source profile and keep the durable primitive as resource access provenance
  plus operation-scoped approval, not marketplace trust.
  Sources: `https://docs.openclaw.ai/tools/skills`,
  `https://docs.openclaw.ai/tools/skills-config`,
  `https://hermes-agent.nousresearch.com/docs/user-guide/features/skills`,
  and
  `https://github.com/anthropics/skills/blob/main/.claude-plugin/marketplace.json`.
- RFC 3986 defines URI as an extensible identifier for physical, abstract,
  local, remote, service, and collection resources. MCP resources are
  URI-identified, support `file://`, `https://`, `git://`, and custom schemes,
  and require URI validation and permission checks. Package URL / PURL provides
  a standard `pkg:` URI shape for package coordinates. Claude Code and Codex
  docs separate permissions, approvals, sandboxing, network controls, and
  telemetry instead of treating one surface as the resource model. Pi,
  OpenClaw, Hermes, and Anthropic skills docs reinforce that skills are
  lifecycle-managed resources over sources, trust, scans, locations, and
  precedence. The experimental `agent://` draft should be reserved for future
  compatibility but not treated as stable v0.10 execution scope.
  Sources: `https://www.rfc-editor.org/rfc/rfc3986`,
  `https://modelcontextprotocol.io/specification/2025-06-18/server/resources`,
  `https://github.com/package-url/purl-spec`,
  `https://code.claude.com/docs/en/permissions`,
  `https://developers.openai.com/codex/agent-approvals-security`,
  `https://pi.dev/docs/latest/skills`,
  and `https://www.ietf.org/archive/id/draft-narvaneni-agent-uri-03.html`.

## Architecture Readiness

v0.10 is the ninth item in the post-v0.01 dependency order:

1. Allbert Home and Settings Central exist.
2. Agent Skills parsing and registry exist.
3. Jido runtime convergence exists.
4. Security Central exists.
5. Action-backed skills exist.
6. Durable confirmations exist.
7. Level 1 local execution exists.
8. Trusted skill script execution exists.
9. External adapters, package installs, and online skill import can now be
   added without bypassing those earlier foundations.

The implementation must remain Jido-centric:

- Runtime-facing external work enters through registered Jido actions.
- `AllbertAssist.Actions.Runner.run/3` remains the action invocation path.
- Security Central owns permission decisions and safety floors.
- Settings Central owns operator policy, provider profiles, source profiles,
  package-manager profiles, and encrypted credentials.
- Durable confirmations are still the only approval workflow.
- CLI and LiveView are operator surfaces, not policy engines.
- Imports are skill registry inputs, not trusted execution authority.

## Fixed v0.10 Decisions

These decisions are part of the implementation contract:

- Keep `external_network_request` as the existing agent-facing action name and
  make it the first real confirmed `Req` adapter. Existing pre-v0.10 resolved
  `adapter_unavailable` records remain historical records.
- Add explicit package and import permissions:
  `:package_install` and `:online_skill_import`. Do not overload
  `:command_execute` or `:external_network` for those distinct risks.
- Keep `:external_network`, `:package_install`, and `:online_skill_import`
  capped by a `:needs_confirmation` safety floor.
- Use `Req` for all HTTP. Do not add `:httpoison`, `:tesla`, `:httpc`, or
  package-manager-specific HTTP clients.
- Disable automatic redirects and automatic retries by default for external
  side-effect actions. Allow redirects or retries only when the request spec and
  policy make them explicit and each redirected URL is revalidated.
- Do not allow user-provided arbitrary URLs as authority. Every request must
  resolve through an allowlisted service/source profile or a normalized
  allowlisted host/method/path policy.
- Treat fetched remote bytes as remote resource access with a declared
  operation class. Operation classes such as `external_service_request`,
  `summarize_url`, `inspect_document`, and `import_skill` carry different
  security meaning and must not share broad approval grants by accident.
- Treat URI identity as the remaining resource-reference target. Future
  `resource_refs` should carry canonical `uri` or `resource_uri` first, while
  `origin_kind`, `canonical_id`, and legacy scopes remain compatibility or
  derived fields. Matching authority is canonical URI plus operation class,
  access mode, downstream consumer, and the current Security Central
  permission decision.
- Reserve future URI schemes without granting execution authority:
  `allbert://` for Allbert-managed resources, `skill://` for skill inventory
  resources, `pkg:` for package coordinates, `mcp://` for future MCP resource
  envelopes, and experimental `agent://` or `agent+https://` for future agent
  endpoints. Unsupported schemes are inert and denied until a later plan adds
  action, policy, confirmation, adapter, redaction, trace, audit, and tests.
- Reject non-HTTP schemes, embedded URL credentials, localhost, loopback,
  link-local, private network, multicast, broadcast, Unix sockets, cloud
  metadata endpoints, and IP-literal targets unless a future policy explicitly
  adds a more privileged local-network mode.
- Keep credentials in Settings Central secrets. Traces, audits, CLI output,
  LiveView output, and tests must see redacted credential refs, never raw
  secrets.
- Treat package installs as package-manager actions, not shell execution. They
  may reuse v0.08 local runner primitives, but only with explicit manager
  profiles, target roots, preview, confirmation, output caps, and audit.
- Make npm the first executable package-manager profile in v0.10 because npm
  exposes documented dry-run and lifecycle-script controls. Keep pip
  preview/audit-only until hash-checking, binary-only, and target-root policy
  are implemented and tested.
- Do not call `npx skills add`, `git clone`, `pip install`, `npm install`, or
  any other external tool from skill activation, online search, or imported
  skill metadata.
- Online skill import writes only to `<ALLBERT_HOME>/cache/skills` and leaves
  imported skills disabled, untrusted, and pending review.
- `skills.sh` is one source profile and search convenience, not the platform
  model. Direct skill URL import should be planned as `remote_url +
  import_skill` resource acquisition plus skill validation/import under the
  `import_skill` operation class.
- Imported skill activation, trust, and script execution still require the
  existing registry validation, enablement, trust, capability-contract, and
  `run_skill_script` controls.
- v0.10 does not add Docker, Podman, Mac/Linux containers, remote sandboxes, or
  microVMs. If package or import work needs that, it must be denied or deferred
  to the container/remote sandbox future work.

## Settings Central Contract

Add the minimum policy surface needed for v0.10. Exact schema names may be
adjusted during implementation if tests and docs are updated together, but the
semantics must remain.

Permission settings:

- `permissions.external_network`: default `needs_confirmation`; writable
  values `needs_confirmation` or `denied`. `allowed` may be stored for
  compatibility but the safety floor still caps it to confirmation.
- `permissions.package_install`: default `denied`; writable values `allowed`,
  `needs_confirmation`, or `denied`; effective value is capped to
  `needs_confirmation` until a future security hardening release changes that.
- `permissions.online_skill_import`: default `denied`; writable values
  `allowed`, `needs_confirmation`, or `denied`; effective value is capped to
  `needs_confirmation`.

External service settings:

- `external_services.enabled`: default `false`.
- `external_services.allowed_hosts`: default `[]`.
- `external_services.blocked_hosts`: default common local/internal hostnames
  and metadata endpoints.
- `external_services.allowed_methods`: default `["GET"]`.
- `external_services.default_timeout_ms`: default `5000`.
- `external_services.max_timeout_ms`: default `30_000`.
- `external_services.max_response_bytes`: default `262_144`.
- `external_services.allow_redirects`: default `false`.
- `external_services.max_redirects`: default `0`.
- `external_services.retry_policy`: default `"none"`.

## Allbert Home Contract

Add or reuse the following durable roots:

- `<ALLBERT_HOME>/external/audit`: external service request audit markdown.
- `<ALLBERT_HOME>/cache/external-services`: bounded optional response/source
  metadata cache, not raw secret-bearing responses.
- `<ALLBERT_HOME>/execution/audit`: package install audit can share the
  execution audit root with an explicit `package_install` event type.
- `<ALLBERT_HOME>/cache/skills`: imported skill cache, already anticipated by
  v0.02 and v0.03.
- `<ALLBERT_HOME>/cache/skills/_sources`: source metadata, downloaded archive
  manifests, import manifests, and audit summaries.

Tests must always point these roots at temporary Allbert homes.

## Action Surface

v0.10 should register these action capabilities:

| Action | Exposure | Permission | Execution mode | Purpose |
| --- | --- | --- | --- | --- |
| `external_network_request` | agent | `:external_network` | `:req_http` | Confirm and run a bounded HTTP/service request after approval. |
| `plan_package_install` | agent or internal | `:read_only` | `:package_install_plan` | Build a preview from package manager, target root, package specs, and policy. |
| `run_package_install` | internal first | `:package_install` | `:package_manager_process` | Create a pending request and, after approval, run the allowed package-manager profile. |
| `search_online_skills` | agent or internal | `:external_network` | `:online_skill_search` | Search an allowed source without importing or enabling skills. |
| `show_online_skill` | agent or internal | `:external_network` | `:online_skill_detail` | Fetch source metadata and display redacted detail. |
| `audit_online_skill` | internal | `:read_only` | `:online_skill_audit` | Inspect cached candidate skill metadata/resources before import. |
| `import_online_skill` | internal | `:online_skill_import` | `:online_skill_import` | Write a reviewed candidate into `<ALLBERT_HOME>/cache/skills` disabled and untrusted. |

Implementation may keep search/detail agent-facing and import internal-only
until the v0.11 intent contract can represent the choice more cleanly. No
action may execute skill scripts, install dependencies, trust imported skills,
or bypass confirmation by virtue of a search/import result.

## Non-Goals

- No unconfirmed external network calls.
- No generic browser, crawler, or open internet research agent.
- No resource-reference authority by itself. M7 metadata does not approve,
  remember, fetch, import, install, summarize, or execute.
- No channel-owned remembered-grant writes or automatic action bypass. M8 adds
  Settings Central grant storage/matching; M11 adds registered grant actions
  and existing-flow grant application; v0.11 still owns
  channel-native Approval Handoff UX over those choices.
- No local directory skill import or direct remote URL skill import until M13
  implements them as explicit disabled/untrusted resource consumers.
- No URL/document summarization, document extraction, or summarizer handoff in
  v0.10. M14 may add clear unsupported/deferred UX; v0.11 owns the actual
  execution-aware orchestration.
- No MCP resource/tool calls or `agent://` delegation in v0.10. M12 may make
  those URI schemes representable as inert future metadata, but unsupported
  schemes remain denied.
- No hidden package installs from skill activation, script metadata, imported
  skill metadata, or model output.
- No auto-trusting or auto-enabling imported skills.
- No runtime loading of Elixir modules from imported skill folders.
- No arbitrary remote code execution.
- No broad provider matrix before one external adapter is boring and tested.
- No credentials outside Settings Central secrets.
- No package manager shell strings, command chaining, global installs, or
  lifecycle scripts by default.
- No pip execution without strict hash/binary/target policy.
- No Docker, Podman, Mac/Linux container, remote, or microVM backend.
- No claim that Level 1 or Level 2 host execution isolates the host from
  hostile code.
- No scheduled background use of these capabilities until v0.12.
- No new external chat channels until v0.13.

## Definition Of Done

v0.10 is done when:

- `external_network_request` can execute a confirmed, allowlisted, redacted
  `Req` HTTP/service request after approval and no longer resolves new v0.10
  requests as `adapter_unavailable`.
- Security Central includes `:package_install` and `:online_skill_import` with
  high-risk classification and confirmation safety floors.
- Settings Central exposes validated external service, package install, and
  online import policy.
- Package install actions can preview and execute the first npm profile only
  under explicit policy, target-root, confirmation, timeout, output, and audit
  controls.
- Pip remains preview/audit-only unless strict hash/binary/target policy is
  implemented and tested.
- Online skill search/detail/audit/import can fetch from an allowed source and
  write imported skills into `<ALLBERT_HOME>/cache/skills` without enabling,
  trusting, activating, or executing them.
- Confirmation approval resumes only eligible registered v0.10 target actions
  after Security Central re-check.
- CLI and `/settings` expose pending/resolved metadata without owning policy or
  execution.
- Traces and audits show external request, package install, and import metadata
  with secrets and large bodies redacted.
- Focused tests, final gates, and operator smoke steps pass.
- Release docs and future milestone handoffs are updated for v0.11, v0.12,
  v0.13, and v0.16.
- M10 hardens canonical resource identity, local symlink/realpath matching,
  source-profile drift, and resumable action metadata.
- M11 provides registered remembered-grant operator actions and applies grants
  to existing v0.10 resource consumers only after Security Central and
  target-specific validation.
- M12 refactors resource identity to URI-first matching while preserving
  existing grant and confirmation compatibility.
- M13 adds direct skill URL import and local skill directory import as
  disabled/untrusted import consumers.
- M14 refreshes unsupported URL/document workflow UX, final docs, final gates,
  and exact operator/user smoke steps.
