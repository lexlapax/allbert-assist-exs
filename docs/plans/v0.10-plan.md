# Allbert v0.10 External Services, Package Installs, And Online Skill Import

## Purpose

v0.10 adds the first confirmed external capability adapters: HTTP/service
calls, package-install workflows, and online skill search/audit/import.

This milestone is intentionally after Security Central, shell execution, and
skill scripts. By the time Allbert can reach out to the network or install
packages, it should already have action-backed contracts, confirmations,
sandbox policy, redaction, and trace/audit records.

## Expected Inputs

- v0.05 Security Central exists for policy, trust boundary, risk, redaction,
  and audit decisions.
- v0.06 action-backed skill execution and action runner exist.
- v0.07 confirmation workflow is stable across CLI and LiveView.
- v0.07 external-network confirmations currently resolve as
  `adapter_unavailable` with `target_resumed?: false`; v0.10 replaces that
  placeholder only through registered `Req`-based actions, Security Central,
  Settings Central credentials, and the shared confirmation context.
- v0.08 sandbox policy can constrain local install/execution side effects.
- v0.09 skill script runner can identify trusted skill resource context.
- Settings Central stores provider credentials, permission policy, skill import
  policy, and redaction behavior.

## Expected Direction

- Add confirmed `Req`-based external service actions for bounded HTTP/API calls
  with allow/block policy, credentials from Settings Central secrets, redacted
  request/response summaries, timeout, rate/cost visibility, and traces.
- Add package-install actions with stricter policy than ordinary shell
  execution: explicit package manager, package names, target directory, install
  command preview, confirmation, sandbox limits, output capture, and rollback
  notes when available.
- Add online skill search/detail/audit/import support, starting with skills.sh
  or another explicitly configured remote source.
- Write imported skills only to `<ALLBERT_HOME>/cache/skills`.
- Keep imported skills disabled and pending until parsed, validated, audited,
  individually enabled, and trusted through Settings Central.

## Non-Goals

- No unconfirmed external network calls.
- No hidden package installs from skill activation or script metadata.
- No auto-trusting imported skills.
- No arbitrary remote code execution.
- No broad provider matrix before one adapter is boring and tested.
- No credentials outside Settings Central secrets.

## Definition Of Done

v0.10 is done when:

- Confirmed external service actions use `Req`, Settings Central credentials,
  redaction, timeout, and trace metadata.
- Package install actions require explicit preview, confirmation, bounded target
  roots, and audit records.
- Online skill search/detail/audit/import can fetch metadata and write imported
  skills into the cache without enabling them.
- Imported skills remain non-executable until registry validation, enablement,
  and trust are explicit.
