# Allbert v0.07 Confirmation Workflow

## Status

Status: implementation-ready after the v0.06 release and v0.07 planning sweep
on 2026-05-02.

This is a docs-only readiness pass. No v0.07 implementation has been completed
yet. The implementation must proceed milestone by milestone and keep each
milestone warning-free before commit or handoff.

## Purpose

v0.07 adds the confirmation layer for capability requests that are known,
inspectable, permission-classed, and Security Central-decisioned, but should
not proceed without explicit operator approval.

This follows the v0.05 Security Central foundation and the v0.06 action-backed
skill release. Confirmations need stable settings-backed policy, known
permission classes, registered actions, risk metadata, redaction obligations,
selected skill/action context, and trace metadata before Allbert can safely
pause and resume runtime work.

The core product loop should become:

1. A user asks for work.
2. The runtime selects a registered action through the normal Jido/action path.
3. Security Central returns `:needs_confirmation`.
4. The action records a durable pending confirmation request and returns
   `:needs_confirmation` without performing the target side effect.
5. CLI or LiveView can approve or deny the same pending request through
   registered actions.
6. Approval may resume only the registered action and adapter that originally
   requested confirmation, and only when that adapter exists for the current
   version.

## Readiness Decision

v0.07 is ready for implementation when the plan, request-flow doc, roadmap,
development guide, and ADRs agree on these points:

- Confirmation requests are durable action state, not UI-only prompts.
- Approval and denial are registered Jido actions invoked through
  `AllbertAssist.Actions.Runner.run/3`.
- Confirmation storage is a plain implementation detail behind action
  boundaries.
- Security Central and Settings Central remain authoritative for permission
  decisions, risk, redaction, and operator policy.
- Approval never bypasses safety floors and never grants new execution powers
  by itself.
- CLI and LiveView share one queue and one approval/denial path.
- Confirmation records distinguish the originating channel from the approving
  or denying channel, so future channels can participate without adding
  channel-private queues.
- v0.07 keeps shell commands, skill scripts, package installs, online imports,
  and real external network calls inert.

## Architecture Readiness Audit

The v0.07 plan is consistent with the current architecture:

- ADR 0001 requires signal-first runtime coordination. v0.07 adds lifecycle
  signals for confirmation request creation and resolution instead of private
  UI state.
- ADR 0003 treats skill manifests as capability contracts, not authority. v0.07
  may include selected skill and contract metadata in confirmation records, but
  skill YAML still cannot grant permission or execute code.
- ADR 0006 makes Security Central the policy evaluation boundary. v0.07 stores
  Security Central decisions with each pending request and re-checks policy
  before approval resumes any work.
- ADR 0007 requires runtime-facing, effectful, security-relevant, and
  observable behavior to enter through registered actions. v0.07 approval,
  denial, expiration, listing, and target resumption follow that rule.
- ADR 0008 defines durable confirmation requests as action state and prevents
  approval from becoming a new execution bypass.

No new dependency or external research is required for this planning pass. If
implementation later depends on Phoenix, LiveView, Jido, YAML, Req, or another
library API detail, use Context7 before coding.

## Vision And Roadmap Alignment

The vision says Allbert should explain sensitive actions, ask for permission,
run validated actions, and record the result, cost, trace, and memory impact.
v0.07 is the missing "ask and remember the answer" layer between Security
Central and real execution adapters.

Roadmap dependency order remains intact:

- v0.05 supplied Security Central.
- v0.06 supplied action-backed skills and capability metadata.
- v0.07 supplies confirmation requests and approval/denial workflow.
- v0.08 can then add confirmed shell execution through the sandbox.
- v0.09 can run skill scripts only after confirmation and sandboxing exist.
- v0.10 can add confirmed external services, package installs, and online
  skill import.

v0.07 must not pull v0.08 through v0.10 forward. Its job is to prove the pause,
inspect, approve/deny, resume-or-decline, trace, and audit workflow before any
new risky adapter is introduced.

## Expected Inputs

- v0.05 Security Central exists for decision shape, risk, redaction, audit, and
  trace metadata.
- v0.05 exposes editable Settings Central permission defaults and read-only
  effective Security Central status through CLI and `/settings`.
- v0.06 has been released with action-backed skill contracts that map only to
  known Jido actions and known permission classes.
- v0.06 runner, signal, trace, and Security Central metadata include selected
  skill, contract validation, selected action capability, permission decision,
  risk, and outcome context.
- v0.06 local skill helper actions use the explicit `:skill_write` permission
  and are operator-invoked through `Actions.Runner.run/3`, not the intent-agent
  tool surface.
- CLI and LiveView can inspect the same runtime and settings state.
- Runtime/action context already carries actor, channel, and session metadata
  from Security Central traces; v0.07 must make that metadata first-class in
  confirmation records and approval decisions.

## Boundary Contract

Confirmation-required target actions must follow this contract:

- Evaluate permission through Security Central or the
  `PermissionGate.authorize/2` compatibility path at the action boundary.
- When the decision is `:needs_confirmation`, avoid the target side effect.
- Create a durable pending confirmation request with redacted params and
  security metadata.
- Store the originating actor, channel, surface, session, and response target
  separately from any later approval/denial channel.
- Return a structured `:needs_confirmation` response that includes the pending
  request id and safe display summary.
- Preserve action runner lifecycle metadata and trace metadata.

Confirmation-management actions must follow this contract:

- Resolve through `AllbertAssist.Actions.Registry`.
- Execute through `AllbertAssist.Actions.Runner.run/3`.
- Re-read pending records from Allbert Home instead of trusting client input.
- Re-check Security Central with the approving actor, channel, and session
  before approval resumes target work.
- Record the resolving channel and session without overwriting the origin
  channel.
- Deny, expire, or no-op resolved records idempotently.
- Emit traceable outcome metadata.

CLI tasks and LiveViews may render, approve, and deny confirmations only by
calling runtime/action boundaries. They must not own queue semantics, storage,
permission policy, redaction, or target action resumption.

## Channel-Aware Interaction Model

v0.07 must be channel-aware even though the implementation currently has only
CLI and LiveView operator surfaces. A pending confirmation is not owned by the
channel that created it. It is shared durable runtime state with explicit
origin and resolution metadata.

Each confirmation request should record:

- `origin_actor`
- `origin_channel`: for example `cli`, `liveview`, or a future channel adapter
- `origin_surface`: task name, route, channel adapter, or job name
- `origin_session_id`
- `origin_conversation_id`, `origin_thread_id`, or `origin_message_id` when the
  channel has that concept
- `origin_response_target`: where the result should be rendered or announced if
  the channel supports callbacks

Each approval, denial, expiration, or cancellation should record:

- `resolver_actor`
- `resolver_channel`
- `resolver_surface`
- `resolver_session_id`
- `resolution_reason`
- whether the resolution happened on the same channel as the request

Channel context is security and trace context, not standalone authority. In the
current local single-operator model, CLI may approve a request created in
LiveView and LiveView may approve a request created from CLI, unless Settings
Central disables that surface. The record and audit must make that handoff
visible. A future multi-user or hosted authorization model can tighten this by
requiring same actor, same channel, explicit role, or channel-specific policy.

When approval resumes target work, the result should be returned to the
approval channel immediately. If the origin channel supports a durable response
target, Allbert should also record or emit enough metadata for that channel to
show the final outcome. CLI has no persistent callback by default, so `list`,
`show`, traces, and audit records are the fallback for cross-channel follow-up.

## Durable Store

All durable confirmation data derives from Allbert Home. The planned default
layout is:

```text
<ALLBERT_HOME>/
  confirmations/
    pending/
      <confirmation-id>.yml
    resolved/
      <YYYY-MM>/
        <confirmation-id>.yml
    audit/
      YYYY-MM.md
```

The store may use YAML records plus markdown audit notes because the queue
needs deterministic read/write behavior and the audit should remain
human-inspectable. Implementation should use structured parsers/writers already
available in the repo and should not hand-roll YAML with ad hoc string
concatenation.

The implementation should add path helpers under `AllbertAssist.Paths` rather
than hard-coding paths in actions, CLI tasks, or LiveViews.

## Confirmation Record Shape

Each pending record should include at least:

- `id`
- `status`: `pending`, `approved`, `denied`, `expired`, `cancelled`, or
  `adapter_unavailable`
- `requested_at`
- `expires_at`
- `resolved_at`
- `origin`: actor, channel, surface, session, conversation/thread/message ids,
  and response target when available
- `target_action`: registered action name and module, if known
- `target_permission`
- `target_execution_mode`
- `selected_skill`: name, source, trust, and contract validation summary
- `capability_contract`: action, permission, confirmation mode, diagnostics
- `security_decision`: permission, decision, reason, risk, policy source,
  confirmation requirement, and redaction posture
- `source_signal_id`
- `source_trace_id`
- `runner_metadata`: request id, requested/completed signal ids, and status
- `params_summary`: redacted, bounded, display-safe action params
- `resume_params_ref`: an internal, redacted-enough representation needed to
  retry the target action, never raw secrets
- `operator_resolution`: approver/denier, reason, channel, surface, session,
  same-channel flag, and decision source
- `audit_path`

Records must not contain raw API keys, secret values, unredacted credential
references, raw command output, hidden environment values, or arbitrary model
scratchpad text.

## Status Vocabulary

Use one shared status vocabulary across store, actions, CLI, LiveView, traces,
and audit:

- `pending`: created and awaiting operator decision.
- `approved`: operator approved and the target action either resumed or is
  ready for a later adapter.
- `denied`: operator denied; the target action did not run.
- `expired`: request aged out according to Settings Central policy.
- `cancelled`: request was invalidated by a newer runtime state or explicit
  cancellation.
- `adapter_unavailable`: operator approved, but the target action has no real
  adapter in this version, so no target side effect occurred.

The existing action response status `:needs_confirmation` remains the runtime
status for target actions that create pending requests.

## Settings Central

v0.07 should add a `confirmations` settings namespace. Initial keys:

- `confirmations.default_ttl_minutes`: default 1440.
- `confirmations.auto_expire_on_startup`: default true.
- `confirmations.require_reason_for_denial`: default false.
- `confirmations.show_redacted_params`: default true.
- `confirmations.allow_cli_approval`: default true.
- `confirmations.allow_liveview_approval`: default true.
- `confirmations.allow_cross_channel_approval`: default true for the local
  single-operator runtime.

Existing permission settings remain under `permissions.*`. v0.07 should not
change the built-in safety floors:

- `:confirmation_decide` is the explicit permission for approval, denial, and
  expiration actions. Its Settings Central key is
  `permissions.confirmation_decide`, default `allowed`, safety floor `allowed`,
  and risk tier `medium`.
- `:command_execute` remains `:denied`.
- `:external_network` remains at most `:needs_confirmation` until v0.10 adds a
  real `Req` adapter.
- skill scripts, package installs, online imports, and arbitrary module loading
  remain denied or unavailable.

Settings may tighten policy, shorten TTLs, or disable an approval surface. They
must not allow approval to bypass Security Central safety floors.

## Initial Request Sources

v0.07 should support pending confirmation creation for the existing
confirmation-needed action path first:

- `external_network_request`: creates a pending request for
  `:external_network`, returns `:needs_confirmation`, and still makes no
  network call.

v0.07 may also support confirmation creation for existing actions whose
operator-configured permission policy resolves to `:needs_confirmation`, such
as settings writes or skill writes. If included, those actions must create a
pending request and avoid the write until approval resumes through the same
registered target action.

v0.07 must not create runnable confirmation requests for:

- shell command execution
- skill scripts
- package installs
- online skill import
- arbitrary network calls
- arbitrary Elixir module loading
- arbitrary file writes outside actions that already exist and own their
  permission story

## Approval Semantics

Approving a request means:

- The operator accepted this specific pending request.
- Security Central must be re-evaluated with confirmation context.
- The original registered target action may be invoked through
  `Actions.Runner.run/3` with a confirmation token or confirmation context.
- The target action still owns validation, permission handling, redaction, and
  side effects.
- If the target adapter is not implemented in v0.07, approval records
  `adapter_unavailable` and performs no target side effect.

Approval does not mean:

- future requests are automatically approved
- skill metadata becomes authority
- denied safety floors are bypassed
- shell, script, package, network, or import adapters become executable
- client-supplied params can replace the stored pending request

Denial means:

- the target action does not run
- the request moves to resolved state with a traceable reason
- any future identical request must create a new pending record

## Signals And Trace Metadata

v0.07 should add or document signal metadata for:

- `allbert.confirmation.requested`
- `allbert.confirmation.approved`
- `allbert.confirmation.denied`
- `allbert.confirmation.expired`
- `allbert.confirmation.adapter_unavailable`

Signals should include confirmation id, target action, permission, risk tier,
selected skill summary, origin actor/channel/session, resolver
actor/channel/session when present, source trace id, and redaction posture.
They should not include raw secrets or unbounded params.

Traces should show:

- pending confirmation id
- origin channel and resolver channel
- action status `:needs_confirmation`
- Security Central decision summary
- selected skill/action/capability metadata
- approval or denial outcome when available
- whether approval resumed target work or stopped at `adapter_unavailable`

### Milestone 4: Approval Resume Semantics

Goal: approval can safely resume eligible registered target actions, while
still leaving unavailable adapters inert.

Implementation details:

- Approval re-reads the pending record by id.
- Approval re-checks Security Central with confirmation context.
- Approval invokes the target action through `Actions.Runner.run/3` only when
  the target action supports confirmed resume.
- Target actions must verify the confirmation id, permission, target action,
  stored params, and current Security Central decision before performing any
  mutation.
- `external_network_request` approval resolves as `adapter_unavailable` in
  v0.07 and makes no network call.
- Denial moves records to resolved state and never invokes the target action.

Tests:

- Approval of external network request records approval but does not call the
  network.
- Approval of any implemented resumable write path performs exactly one
  mutation after approval and is idempotent on repeat approval.
- Denial does not run the target action.
- Policy changes between request and approval are respected.

Docs:

- Request-flow doc shows approved-with-adapter and approved-without-adapter
  paths separately.

Closeout:

- Complete. M4 updates `approve_confirmation` to re-read pending records,
  enforce approval-surface and cross-channel settings, re-check the target
  permission through Security Central with confirmation context, and resolve
  unavailable adapters as `adapter_unavailable` without invoking target work.
  If target policy changes to denied between request and approval, approval
  resolves the request as `denied` and records policy-block metadata. v0.07 has
  no implemented resumable target adapter, so no approval path performs a
  mutation or network call. Denial now records accurate same-channel metadata
  and still never invokes the target action.

## Operator Smoke For Implementation

The final v0.07 implementation should include an operator smoke flow using a
disposable home:

```sh
tmpdir=$(mktemp -d /tmp/allbert-v07-smoke.XXXXXX)
ALLBERT_HOME="$tmpdir" mix allbert.ask --trace "fetch https://example.com"
ALLBERT_HOME="$tmpdir" mix allbert.confirmations list
ALLBERT_HOME="$tmpdir" mix allbert.confirmations show <confirmation-id>
ALLBERT_HOME="$tmpdir" mix allbert.confirmations deny <confirmation-id> --reason "not needed"
ALLBERT_HOME="$tmpdir" mix allbert.confirmations list --resolved
```

If the implementation includes a resumable write path, add a second smoke that
configures that permission to `needs_confirmation`, proves no mutation occurs
before approval, approves the request, and proves exactly one mutation occurs.

The external network smoke must prove that approval does not make a network
call in v0.07.

## Non-Goals

- No arbitrary command execution.
- No shell adapter.
- No skill script execution.
- No package installs.
- No online skill import.

## Definition Of Done

v0.07 is done when:

- Confirmation-required actions create durable pending requests instead of
  running the target side effect.
- CLI and LiveView can approve or deny the same pending request through
  registered actions and `Actions.Runner.run/3`.
- Approval re-checks Security Central and resumes only registered, eligible
  target actions.
- Approval of an unavailable adapter records `adapter_unavailable` and performs
  no target side effect.
- Denial is traceable and does not run the target action.
- Confirmation decisions are redacted, audited, and visible in traces.
- Settings Central owns confirmation preferences.
- Release documentation and app version metadata are updated for v0.07.
- Safety floors still prevent shell, script, package, online import, and real
  network execution.
- The roadmap, request-flow doc, ADR index, and future milestone handoffs match
  the implemented behavior.
