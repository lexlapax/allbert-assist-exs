# Allbert v0.25 Native Jido Trading Agents

Formerly tracked as M-D2c, previously planned as v0.19, then v0.23 before
the v0.23 Jido State-Machine Convergence and v0.24 Objective Runtime
Foundation milestones were inserted by the project-direction rethink.

## Purpose

v0.25 replaces the transitional Python bridge as the default StockSage analysis
engine only after native Jido agents prove parity against the Python baseline.
Native agents consume the v0.24 objective runtime from day one: each analysis
runs as an objective step, with `objective_id` and `step_id` threaded through
confirmations, traces, and `stocksage_analyses` rows.

## Expected Inputs

- v0.22 Python bridge is working and provides golden fixtures.
- v0.20 StockSage plugin/app, `stocksage_*` domain storage, and queue records
  exist.
- v0.23 Jido State-Machine Convergence has shipped: `Confirmations.Store` and
  `Jobs.Scheduler` are `Jido.Agent` modules; specialist trading agents
  follow that pattern.
- v0.24 Objective Runtime Foundation has shipped: `Objectives.Engine.Agent`
  (built on the v0.23 `AllbertAssist.JidoBacked` substrate),
  `objective_steps.kind = :delegate_agent` (minimal-impl contract per
  v0.24 plan M4 + ADR 0021 Amendment A3), `AllbertAssist.Objectives.AgentRegistry`,
  the `AllbertAssist.Objectives.Actions.DelegateAgent` registered action,
  and `objective_id`/`step_id` threading exist. Native trading agents
  register themselves at boot in `AllbertAssist.Objectives.AgentRegistry`
  and are dispatched as `:delegate_agent` steps.
- Jido.AI.Agent and Jido actions remain the approved agent/action substrate.

## Expected Direction

- Add native agents: Orchestrator, Market Analyst, Sentiment Analyst, News
  Analyst, Fundamentals Analyst, Bull Researcher, Bear Researcher, Trader, and
  Portfolio Manager.
- Model the analysis workflow as a Jido pod/topology with registered actions
  for stock data, news, sentiment, and fundamentals.
- Add local OTP/Jido workers that reuse the v0.20 SQLite queue records for
  background analysis.
- Register all native analysis worker supervisors as children of
  `StockSage.Supervisor`. Confirm `StockSage.Plugin.child_spec/1` still returns
  the updated supervisor spec so native workers start, stop, and restart with
  the StockSage plugin lifecycle.
- Add `--engine native` and preserve `--engine python` fallback.
- Native becomes default only after golden-fixture and 20-stock smoke parity
  pass within a documented variance band.

## Non-Goals

- No default switch to native before parity.
- No hosted/distributed worker substrate.
- No direct market-data calls outside Resource Access Security Posture.

## Test Plan

- Golden fixture comparisons against Python baseline.
- 20-stock smoke batch.
- Engine-selection tests for `native`, `python`, and default behavior.
- Queue/resumption tests for local worker records.
- OTP supervision tests prove native worker supervisors are descendants of
  `StockSage.Supervisor`, not independently started from Allbert core.

## Definition Of Done

v0.25 is done when native Jido analysis is selectable, parity is documented,
all native workers live under the StockSage plugin supervisor, native
analysis runs as a v0.24 objective step with `objective_id`/`step_id`
threaded through confirmations and audit, and the project can safely choose
native as the default engine.
