# Allbert v0.24 Security Hardening And Evals

## Purpose

v0.24 returns to Security Central after real execution, imports, jobs, channels,
memory review, and richer intent exist. The goal is to harden the system with
evidence from actual runtime surfaces rather than guessing too early.

This milestone was formerly tracked as v0.16 before the post-v0.10 roadmap
was unified into one v0.xx sequence.

## Expected Inputs

- v0.05 Security Central foundation exists.
- v0.08 through v0.10 execution and import adapters are in use, including shell,
  skill script, confirmed `Req` external service calls, package-manager
  profiles, and online skill import manifests.
- v0.11 Resource Access Security Posture is in use for local path access, URL
  summarization, document inspection, local skill directory import, direct
  skill URL import, trusted skill script execution, and source-profile
  consumers.
- v0.10 M9 closed the first release-readiness baseline for external services,
  package installs, online skill import, resource references, and remembered
  resource grants; eval fixtures should preserve those cases.
- v0.13 jobs and v0.16 channels generate non-interactive and remote-context
  requests.
- v0.17 plugin registry exists and now owns shipped source-tree Telegram/email
  channel registration plus plugin-contributed skill/app/action fixtures.
- v0.18 StockSage local domain, queue, import, and safe local action surfaces
  exist.
- v0.19 memory review and v0.21 intent enrichment provide richer trace data.
- v0.12 conversation history is present, providing `user_id` and `thread_id`
  for cross-user and thread leakage eval coverage. v0.14 ETS session scratchpad
  may add session entries later; if v0.14 has not landed, session-specific
  fixtures should be marked pending with an explicit condition.
- v0.15 app registration may be present, providing app-scoped action routing
  via `AllbertAssist.App.Registry` for app-routing eval coverage. If v0.15 has
  not landed, those fixtures should be marked pending.
- v0.25 Full App Contract and Surface DSL follows v0.24 in the roadmap. v0.24
  should define pending app-registration eval fixtures and activation criteria
  for that future surface, but v0.25 is not an input to starting v0.24.
- v0.20 Python bridge may be present as a security surface. Eval coverage is
  conditional on v0.20 having landed.

## Expected Direction

- Add security eval fixtures for prompt injection, tool argument injection,
  SSRF attempts, unsafe redirects/retries, untrusted skill activation,
  malicious imports, package-manager lifecycle script abuse, unpinned package
  specs, URL/git/tarball/path/global package specs, command approval bypass,
  resource access operation-scope bypass, local path traversal, symlink escape,
  unsafe summarizer handoff, unsupported document extraction, credential
  leakage, cross-session data access, channel spoofing, and unsafe background
  execution.
- Add cross-user and thread leakage evals for attempts to read another user's
  thread history, inject a different `user_id` into a request, or
  cross-contaminate ETS scratchpad entries between sessions.
- Add plugin-boundary evals for malicious or malformed manifests, duplicate
  plugin ids, action-name collisions, disabled-plugin bypass attempts,
  path traversal in plugin skill roots, attempts to claim code-bearing
  contributions from `<ALLBERT_HOME>/plugins`, attempts to smuggle arbitrary
  `./plugins/*/lib` directories into compilation, and attempts to gain trust,
  permissions, resource grants, or confirmation bypass from plugin metadata.
- Add app-scoped action routing evals. StockSage actions registered through
  `AllbertAssist.App` must not be reachable when `active_app` is nil or set to
  a different app; `StockSage.Actions.RunAnalysis` requires explicit StockSage
  session context.
- If v0.20 has landed, treat the Python bridge as a security surface. Cover
  arbitrary command injection through the bridge JSON protocol, path traversal
  in bridge arguments, and bridge crash handling that does not leak partial
  analysis data.
- Add financial workflow authorization evals. StockSage analysis actions that
  trigger external market-data API calls must flow through Resource Access
  Security Posture and confirmation workflow; a remembered
  `external_service_request` grant for a general API must not automatically
  authorize financial analysis API calls.
- Define app-registration security eval fixtures for v0.25 activation:
  attempts to register a malicious app that bypasses `validate/1`, claims
  another app's `app_id`, injects unauthorized actions into the registry, or
  registers skill paths that resolve outside the app's `priv/skills` directory
  or `ALLBERT_HOME`. These fixtures may remain pending in v0.24 and should be
  enabled once v0.25 lands.
- Add operator-visible security review workflows for recent denials,
  confirmations, imports, external calls, and redaction incidents.
- Reassess sandbox, allowlist, safe-bin, Resource Access Security Posture,
  document extraction, summarizer handoff, and supply-chain policies against
  real traces.
- Add hardening docs for deployment posture, channel pairing, exposed services,
  local file permissions, and emergency disable switches.

## Non-Goals

- No broad hosted multi-user authorization model.
- No replacement for Settings Central or Security Central.
- No new execution capabilities.

## Definition Of Done

v0.24 is done when:

- Security evals cover the high-risk agent failure modes that now exist.
- Cross-user and thread leakage fixtures are present and passing when v0.12
  and v0.14 have landed, or documented as pending with an explicit condition.
- App-scoped action routing fixtures are present and passing when
  v0.15 has landed, or documented as pending.
- Plugin-boundary fixtures cover v0.17 plugin discovery, disabled-plugin
  behavior, skill-root path safety, duplicate contributions, and no runtime
  code loading from home plugins or arbitrary source-tree plugin folders.
- App registration security fixtures are present and passing when v0.25 has
  landed, or documented as pending.
- Operators can inspect recent security decisions and risky capability history.
- Security Central policy has been reviewed against real execution, channel,
  job, import, resource access, summarizer handoff, and memory behavior.
- The roadmap has a fresh risk reassessment for v0.27+.
