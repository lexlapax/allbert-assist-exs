the v0.23 Jido State-Machine Convergence and v0.24 Objective Runtime

Status: planning-ready after holistic revisit (2026-05-17). Implementation
has not started.

native Elixir/OTP/Jido design: reusable, encapsulated financial specialist

## Purpose

v0.25 introduces native financial specialist agents that produce StockSage
analyses as the primary execution engine.
The native implementation is **not** a one-to-one port of the Python
TradingAgents graph. It reuses the Python baseline's role intent, final-state
fields, comparison fixtures, and prompt material where available, but models
the work the Elixir/OTP/Jido way:

- A shared objective owns durable cross-turn state.
- Reusable specialist agents do one bounded financial-reasoning job each.
- Registered actions remain the only effectful capability boundary.
- Security Central, Resource Access Security Posture, confirmations, traces,
  and `stocksage_analyses` remain the operator-visible contract.
- The Python bridge remains available only as an explicitly requested
  comparison/reference path. Allbert never automatically falls back from native
  to Python.

The end-user loop should be:

1. The user asks Allbert or StockSage to analyze a ticker.
2. `RunAnalysis` creates or joins a v0.24 objective and asks for confirmation.
3. After approval, the native engine creates delegate-agent steps for the
   specialist agents.
4. The agents gather bounded evidence, produce role-specific reports, and
   synthesize a decision.
5. The result persists to the same StockSage analysis tables as v0.22, with
   `engine: "native"` and `objective_id` / `step_id` retained.
6. The user can inspect the objective, steps, trace, confirmation, and analysis
   rows through existing operator surfaces.

## Readiness Decision

v0.25 is ready for implementation when these documents agree:

- Native financial agents are reusable Allbert delegate agents, not private
  StockSage-only helpers.
- The Python bridge is a baseline and explicit comparison harness, not a
  fallback path and not the architecture template.
- Prompt material is adapted from the TradingAgents baseline only after a
  prompt inventory. `plugins/stocksage/priv/python/bridge.py` itself contains
  the JSON bridge and final-state field list, not the role prompts.
- All external market-data access is behind registered actions with Resource
  Access Security Posture metadata.
- No native agent writes `stocksage_*` rows directly. The final persistence
  still happens in `StockSage.Actions.RunAnalysis`.
- Native agents are supervised under `StockSage.Supervisor` through
  `StockSage.Plugin.child_spec/1`, and registered in
  `AllbertAssist.Objectives.AgentRegistry` on boot.

## Architecture Readiness Audit

- ADR 0006 keeps Security Central as the permission and risk boundary.
  Native agents cannot authorize themselves; `:stocksage_analyze` remains
  the execution permission for the analysis as a whole.
- ADR 0007 requires runtime-facing effectful behavior to enter through
  registered Jido actions. Specialist agents may reason and return advisory
  packets. Market-data fetches, persistence, confirmations, and settings writes
  still go through registered actions.
- ADR 0008 requires durable confirmations for high-risk actions. v0.25 reuses
  the `RunAnalysis` confirmation path from v0.22/v0.24.
- ADR 0017 says plugin code contributes capabilities through declared
  contracts. Native agents live under `./plugins/stocksage`, are supervised by
  `StockSage.Supervisor`, and are not added to Allbert core.
- ADR 0020 defines the Python bridge baseline. v0.25 keeps explicit
  `--engine python` comparison runs and the bridge protocol intact while
  making the native engine the normal analysis path. Python is never invoked as
  automatic recovery from native failure.
- ADR 0021 defines the objective and delegate-agent boundary. v0.25 is the
  first real consumer of `objective_steps.kind = "delegate_agent"` and
  `AllbertAssist.Objectives.AgentRegistry`.
- ADR 0022 binds the native financial specialist-agent contract introduced by
  this plan.

## Vision And Roadmap Alignment

The allbert vision says agents should be understandable, signal-driven, and
bounded by inspectable authority. v0.25 should make StockSage smarter without
creating a private autonomous trading system.

Roadmap dependency order remains intact:

- v0.20 supplied StockSage plugin/domain storage.
- v0.22 supplied the Python bridge baseline and real persisted analyses.
- v0.23 supplied the JidoBacked internal runtime substrate.
- v0.24 supplied durable objective/step state and delegate-agent dispatch.
- v0.25 now supplies reusable native financial specialist agents.
- v0.26 workspace shell can render objective progress and native agent steps.
- v0.27+ StockSage UI/polish can render richer native agent outputs.

v0.25 must not pull v0.26 LiveViews, v0.27 memory namespaces, v0.29 polish,
or v0.30 canvas work forward.

## Boundary Contract

### What Native Financial Agents Are

Native financial specialist agents are supervised Jido/Jido.AI agents that
accept a bounded request packet and return a bounded advisory report packet.
They are intentionally reusable:

- StockSage calls them through objective delegate steps.
- Future apps may call the same agents through `DelegateAgent` if they have
  the right permissioned action path.
- CLI/LiveView/channel surfaces do not call agent processes directly.
- Plugins do not maintain hidden registries; registration goes through
  `AllbertAssist.Objectives.AgentRegistry`.

### What Native Financial Agents Are Not

- Not registered Allbert capability actions.
- Not direct writers to `stocksage_*` tables.
- Not permission grants.
- Not market-data clients unless the data access is itself a registered action
  invoked through the action runner.
- Not a private durable task graph. Durable work belongs to
  `objectives`, `objective_steps`, `objective_events`, queue rows, and analysis
  rows.
- Not a textual clone of every Python TradingAgents role.

### Call Contract

Every specialist agent accepts this normalized request shape:

```elixir
%{
  "request_id" => binary(),
  "user_id" => binary(),
  "ticker" => binary(),
  "analysis_date" => "YYYY-MM-DD",
  "objective_id" => binary() | nil,
  "step_id" => binary() | nil,
  "trace_id" => binary() | nil,
  "role" => binary(),
  "task" => binary(),
  "evidence" => map(),
  "constraints" => map(),
  "prompt_version" => binary()
}
```

Every specialist agent returns this normalized report shape:

```elixir
%{
  "agent_id" => binary(),
  "role" => binary(),
  "status" => "completed" | "needs_more_evidence" | "failed",
  "summary" => binary(),
  "report" => binary(),
  "evidence_used" => [map()],
  "confidence" => float() | nil,
  "warnings" => [binary()],
  "data_requests" => [map()],
  "generated_at" => binary()
}
```

Reports are advisory. Only `RunAnalysis` persists the final result.

## Native Agent Topology

v0.25 implements a smaller, reusable agent set instead of mirroring the Python
graph one-for-one.

### Required Agent IDs

These IDs are registered in `AllbertAssist.Objectives.AgentRegistry` at
StockSage plugin boot:

| Agent id | Module | Purpose |
| --- | --- | --- |
| `stocksage.market_context` | `StockSage.Agents.MarketContext` | Summarize market/price/technical context from normalized evidence. |
| `stocksage.news_sentiment` | `StockSage.Agents.NewsSentiment` | Summarize news and sentiment evidence. |
| `stocksage.fundamentals` | `StockSage.Agents.Fundamentals` | Interpret fundamentals and company context. |
| `stocksage.bull_thesis` | `StockSage.Agents.BullThesis` | Produce the strongest bullish thesis from collected evidence. |
| `stocksage.bear_thesis` | `StockSage.Agents.BearThesis` | Produce the strongest bearish thesis from collected evidence. |
| `stocksage.decision_synthesizer` | `StockSage.Agents.DecisionSynthesizer` | Produce final rating, rationale, and investment plan. |
| `stocksage.quality_gate` | `StockSage.Agents.QualityGate` | Validate final report shape, missing evidence, and redaction/safety posture. |

`StockSage.Agents.NativeCoordinator` may exist as an internal JidoBacked
coordinator under `StockSage.Supervisor`, but it must not own durable objective
state. Its job is to submit delegate steps and gather report packets for the
current `RunAnalysis` call.

### Why This Is Not A Python Clone

TradingAgents has role names such as market analyst, sentiment analyst, news
analyst, fundamentals analyst, bull researcher, bear researcher, trader, and
portfolio manager. v0.25 keeps the useful role separation, but the native
shape is organized around reusable outputs:

- analyst agents produce bounded evidence reports;
- thesis agents produce opposing arguments;
- synthesizer produces a decision and investment plan;
- quality gate validates the final artifact.

The Python `portfolio_manager`/`trader` split can be represented by the native
`decision_synthesizer` until there is a concrete Allbert consumer that needs
them separate. The native design should split further only when tests or
operator UX show a real need.

## Prompt And Baseline Contract

Implementation must create a prompt inventory before coding agent prompts:

- Inspect `plugins/stocksage/priv/python/bridge.py` and confirm it contains
  bridge protocol, config, and final-state field names, not role prompts.
- Inspect the pinned TradingAgents package or source used by v0.22 for role
  prompts and debate instructions.
- Copy/adapt only prompt text that is license-compatible and appropriate for
  local source. If prompt text cannot be stored, record the prompt source and
  use an Allbert-authored prompt with the same role intent.
- Store reviewed prompt templates under
  `plugins/stocksage/priv/prompts/native_agents/`.
- Version prompts with `prompt_version`, include that version in agent reports,
  and persist it in `stocksage_analysis_details.raw`.

Prompt guidance:

- Use role prompts as operator-readable instructions, not hidden authority.
- Do not ask agents to execute trades, contact brokers, or give personalized
  financial advice.
- Require citations/evidence references to the local data packets used.
- Require uncertainty and warnings when evidence is incomplete.
- Keep "not financial advice" disclosure in the final report.

## Data And Tool Contract

Native agents need evidence packets. They may not fetch data by themselves.

Required registered actions or equivalent existing actions:

- `StockSage.Actions.FetchMarketData`
- `StockSage.Actions.FetchNews`
- `StockSage.Actions.FetchFundamentals`

Each action must:

- execute through `AllbertAssist.Actions.Runner.run/3`;
- include Resource Access Security Posture metadata for external URLs/APIs;
- inherit the approved `RunAnalysis` context (`user_id`, `objective_id`,
  `step_id`, `trace_id`) without treating those fields as permission;
- emit named StockSage/native-agent signals;
- return bounded, redacted evidence packets;
- support fixture mode for deterministic tests and operator smoke without
  external credentials.

If real data-provider integrations are too variable for v0.25 acceptance,
fixture mode still must exercise the full action/agent/objective/persistence
path. Do not bypass the action runner in fixture mode.

## Settings

Add settings through Settings Central:

| Key | Type | Default | Notes |
| --- | --- | --- | --- |
| `stocksage.native_enabled` | boolean | `true` | Master switch for native agents. |
| `stocksage.python_comparison_enabled` | boolean | `true` | Allows explicitly requested Python baseline/comparison runs. Does not enable fallback or default routing. |
| `stocksage.native_model_profile` | string | `"fast"` | Jido.AI model profile for native specialist agents. |
| `stocksage.native_timeout_ms` | integer | `600_000` | Total native analysis timeout. |
| `stocksage.native_agent_timeout_ms` | integer | `120_000` | Per-specialist timeout. |
| `stocksage.native_fixture_mode` | boolean | `false` | When true, data actions use fixture packets; no external market calls. |
| `stocksage.native_max_report_bytes` | integer | `65_536` | Bound persisted native report payloads. |
| `stocksage.native_parity_variance` | float | `0.25` | Acceptance band for fixture comparison metrics. |

Settings are not authority. They select implementation paths and bounds only.

## CLI And Operator Surfaces

Existing commands stay:

```sh
mix stocksage.analyze AAPL 2026-05-01 --engine python --user local
mix stocksage.analyze AAPL 2026-05-01 --engine native --user local
mix stocksage.analyze AAPL 2026-05-01 --compare-python --user local
mix stocksage.analyses list --user local
mix stocksage.analyses show <analysis_id> --user local
mix allbert.objectives show <objective_id> --user local
```

New operator inspection commands:

```sh
mix stocksage.agents list
mix stocksage.agents show stocksage.market_context
mix stocksage.agents smoke AAPL 2026-05-01 --fixture-data --user local
```

The new commands call registered actions through `Actions.Runner.run/3`.
They do not inspect processes or ETS tables directly.

## Signals

Add named signals:

- `allbert.stocksage.native.requested`
- `allbert.stocksage.native.agent.started`
- `allbert.stocksage.native.agent.completed`
- `allbert.stocksage.native.agent.failed`
- `allbert.stocksage.native.synthesized`
- `allbert.stocksage.native.quality_checked`
- `allbert.stocksage.native.completed`
- `allbert.stocksage.native.failed`

Payloads include `user_id`, `ticker`, `analysis_date`, `objective_id`,
`step_id`, `trace_id`, `engine`, `agent_id` when applicable, status, bounded
summary, and redacted diagnostics. Raw reports and provider secrets are never
signal payloads.

## Persistence

Native results use the existing tables:

- `stocksage_analyses.engine = "native"`
- `stocksage_analyses.summary` is the final bounded summary.
- `stocksage_analysis_details.raw` stores a bounded JSON object with:
  - `native_reports` keyed by agent id;
  - `final_trade_decision`;
  - `investment_plan`;
  - `market_report`;
  - `sentiment_report`;
  - `news_report`;
  - `fundamentals_report`;
  - `quality_gate`;
  - `prompt_versions`;
  - `fixture_mode`;
  - `parity` diagnostics when run against a fixture.

The shape intentionally preserves the v0.22 Python final-state field names so
existing list/show flows and v0.27 LiveViews can render both engines.

Explicit Python comparison runs keep the existing Python engine value, but
their confirmation, trace metadata, CLI output, and detail JSON must include a
clear comparison marker, for example:

```json
{
  "comparison_reference": true,
  "requested_engine": "python",
  "comparison_target": "native"
}
```

Native runs never silently create this Python comparison artifact.

### M2: Supervision, Agent Registry, And Callable Agent Contract

Goal: native agents boot under the StockSage plugin and are callable through
Allbert's delegate-agent substrate.

Tasks:

- Add `StockSage.Agents.Supervisor` under `StockSage.Supervisor`.
- Add required specialist agent modules and child specs.

  boot; unregister on shutdown when applicable.

  `StockSage.Agents.Contract`.

  `show_financial_agent` for operator inspection.
- Add `mix stocksage.agents list|show`.

Tests:

- All required ids register under `AgentRegistry`.
- Dead agent process eviction works with StockSage agents.
- Agents are descendants of `StockSage.Supervisor`.
- `Actions.Registry` does not contain private specialist agent command modules.
- `mix stocksage.agents list|show` output is bounded and secret-free.

### M3: Evidence Actions And Fixture Mode

Goal: data access is action-backed and safe before agents consume it.

Tasks:

- Add action-backed evidence providers for market, news/sentiment, and
  fundamentals.
- Add Resource Access Security Posture metadata for each external provider.
- Add fixture mode that uses local fixture packets through the same actions.
- Emit data-provider signals and trace metadata.

Tests:

- Fixture mode does not perform network calls.
- Non-fixture mode is rejected or confirmed according to Security Central and
  Resource Access posture.
- Evidence packets are bounded, redacted, and keyed by `ticker` and
  `analysis_date`.
- Cross-user context cannot read another user's queued analysis state.

### M4: Specialist Agent Reports

Goal: each specialist produces one bounded advisory report.

Tasks:

- Implement `MarketContext`, `NewsSentiment`, `Fundamentals`,
  `BullThesis`, `BearThesis`, `DecisionSynthesizer`, and `QualityGate`.
- Use Jido.AI agents for LLM-backed roles where useful; use deterministic
  Jido agents/actions for validation and quality gates.
- Feed agents with evidence packets and prompt templates, not direct DB or
  network access.
- Return normalized report packets with prompt version and evidence references.

Tests:

- Each agent has a fixture-mode happy path.
- Each agent rejects malformed input and oversize evidence.
- Agent reports redact secrets and bound output size.
- Missing evidence produces `needs_more_evidence` or warnings, not crashes.
- Prompts are loaded by version and recorded in report metadata.

### M5: Native RunAnalysis Integration

Goal: `RunAnalysis --engine native` uses objective delegate steps and persists
the same operator-visible result shape as the Python baseline.

Tasks:

- Extend `StockSage.Actions.RunAnalysis` engine selection:
  `"native"` as the normal path and `"python"` only when explicitly requested
  by the operator for comparison/reference.
- Preserve `:stocksage_analyze` permission, confirmation, and approval-resume
  behavior.
- On approval, create/continue a v0.24 objective step and delegate to the
  native agents through `DelegateAgent` / `AgentRegistry`.
- Persist final result to existing analysis tables with `engine: "native"`.
- Keep explicit `--engine python` and `--compare-python` comparison paths
  intact. Do not automatically retry or recover with Python when native fails.
- Add `mix stocksage.agents smoke TICKER DATE --fixture-data --user USER`.

Tests:

- Explicit `--engine native` creates confirmation, approval runs native path,
  and analysis row persists with `engine: "native"`.
- Explicit `--engine python` still uses the Python bridge and is clearly
  marked as a comparison/reference run.
- Default uses native; no persistent setting may make Python the default.
- Native failure does not invoke Python; the error remains visible unless the
  operator starts a separate explicit comparison/reference run.
- Objective show displays delegate-agent steps and native reports.
- Queue entries transition correctly for native success and failure.
- Denied confirmation does not start any native agent.

## User Acceptance

Operator-verifiable acceptance uses a disposable Allbert Home:

1. `mix ecto.migrate.allbert` applies all core and plugin migrations.
2. `mix stocksage.agents list` shows all required native agents.
3. `mix stocksage.agents smoke AAPL 2026-05-01 --fixture-data --user local`
   runs without external calls and produces a bounded report packet.
4. `mix stocksage.analyze AAPL 2026-05-01 --engine native --user local`
   creates a confirmation and does not run agents before approval.
5. `mix allbert.confirmations approve <id> --reason "native smoke"` runs the
   native agent path and persists an analysis row.
6. `mix stocksage.analyses show <analysis_id> --user local` shows
   `engine: native`, final decision, and bounded specialist report summaries.
7. `mix allbert.objectives show <objective_id> --user local` shows delegate
   steps with `objective_id` / `step_id` carried through.
8. `mix stocksage.analyze AAPL 2026-05-01 --engine python --user local` runs
   an explicitly requested Python comparison/reference analysis and is labeled
   as such in output, trace, confirmation, and persisted details.
9. Cross-user show/list attempts return not-found or empty results.

## Non-Goals

- No hosted/distributed worker substrate.
- No broker/trade execution.
- No personalized financial advice.
- No dynamic code generation from prompts.
- No direct market-data/network calls from agent modules.
- No automatic Python fallback.
- No StockSage LiveView buildout (v0.27).
- No canvas or workspace shell work (v0.26/v0.30).

## Definition Of Done

v0.25 is done when:

- ADR 0022 is accepted with the implemented native-agent contract.
- Native financial agents are supervised under `StockSage.Supervisor` and
  registered in `AllbertAssist.Objectives.AgentRegistry`.
- Agents are callable through the v0.24 delegate-agent path by StockSage and
  future Allbert runtime consumers.
- Agent prompts are inventoried, versioned, and operator-readable.
- Evidence access flows through registered actions and Resource Access posture.
- `RunAnalysis --engine native` persists the same durable analysis shape as
  the Python baseline.
- `--engine python` remains working only as an explicitly requested
  comparison/reference path and is documented/labeled accordingly.
- Objective, trace, confirmation, queue, and analysis rows carry
  `objective_id`, `step_id`, `user_id`, and `trace_id`.
- Fixture parity and smoke results are documented.
- Native is the only default operational engine. Python comparison never
  becomes an automatic fallback or persistent default.
