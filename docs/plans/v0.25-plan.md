This pass bakes in the decisions taken during the third-pass

the plugin-owned native coordinator graph with durable objective-step

explicit Python comparison/reference only, license-conservative
Allbert-authored prompt provenance, and a
fully runtime-only request-flow doc that owns user testing
per-milestone.

Post-implementation audit closeout: v0.25 ships real supervised
specialist modules, evidence actions, objective-step observability,
native persistence, parity comparison, and cross-app delegate reuse.
Non-quality native specialists call `Jido.AI.generate_object/3` when
`stocksage.native_llm_enabled` is true. Tests and explicit operator
smoke can disable that setting to exercise deterministic advisory

fallback from provider failure; when LLM mode is enabled and provider
generation fails, the graph records a bounded failed advisory packet.

## Purpose

v0.25 introduces native financial specialist agents that produce
StockSage analyses as the primary execution engine. The native
implementation is **not** a one-to-one port of the Python TradingAgents
graph. It reuses the Python baseline's role intent, final-state fields,
comparison fixtures, and role intent, but ships Allbert-authored prompt
control files in v0.25 and models the work the
Elixir/OTP/Jido way:

- A shared v0.24 objective owns durable cross-turn state.
- Reusable specialist agents do one bounded financial-reasoning job each.
- Registered actions remain the only effectful capability boundary.
- Security Central, Resource Access Security Posture, confirmations,
  traces, and `stocksage_analyses` remain the operator-visible contract.
- The Python bridge remains available only as an explicitly requested
  comparison/reference path. Allbert never automatically falls back from
  native to Python.

The end-user loop should be:

1. The user asks Allbert or StockSage to analyze a ticker.
2. `RunAnalysis` creates or joins a v0.24 objective and asks for
   confirmation.
3. After approval, `StockSage.Agents.NativeCoordinator` (JidoBacked)
   creates delegate-agent steps for the specialist agents, manages
   multi-round bull/bear/risk debate, and synthesizes the result.
4. The agents gather bounded evidence via 5 tiered Allbert evidence
   actions, produce role-specific reports, and synthesize a decision.
5. The result persists to the same StockSage analysis tables as v0.22,
   with `engine: "native"` and `objective_id` / `step_id` retained.
6. The user can inspect the objective, steps, trace, confirmation, and
   analysis rows through existing operator surfaces — and, for any
   registered specialist agent, invoke it through `mix allbert.delegate
   <agent_id> <args>` for cross-app reuse smoke.

## Readiness Decision

v0.25 is ready for implementation when these documents agree:

- Native financial agents are reusable Allbert delegate agents, not
  private StockSage-only helpers. Cross-app reuse is proven by an
  operator-visible `mix allbert.delegate` Mix task that lives in core
  Allbert, not StockSage.
- The Python bridge is a baseline and explicit comparison/reference
  harness, not a fallback path, not the architecture template, and not
  a Settings Central default. `--engine python` runs Python-only;
  `--engine both` runs both engines in parallel and records a parity
  comparison. Without an explicit engine option, `RunAnalysis` uses
  the native engine.
- Prompt material is license-conservative in v0.25. Prompt files are
  Allbert-authored and operator-readable; TradingAgents remains the
  behavioral baseline and comparison target, not a copied prompt
  source. Verbatim TradingAgents prompt adaptation requires a future
  explicit license audit. `plugins/stocksage/priv/python/bridge.py`
  itself contains the JSON bridge and final-state field list, not the
  role prompts.
- All external market-data access is behind registered tiered evidence
  actions with Resource Access Security Posture metadata.
- No native agent writes `stocksage_*` rows directly. The final
  persistence still happens in `StockSage.Actions.RunAnalysis`.
- Native agents are supervised under `StockSage.Supervisor` through
  `StockSage.Plugin.child_spec/1`, and registered in
  `AllbertAssist.Objectives.AgentRegistry` on boot.
- `StockSage.Agents.NativeCoordinator` is a JidoBacked agent owning
  multi-round debate state, agent dispatch order, and parity-run
  composition. It is distinct from `Objectives.Engine.Agent` (v0.24)
  — the coordinator is a per-analysis orchestrator; the engine is the
  durable cross-turn objective state machine.
- ADR 0022 status updated to Accepted at M6 with v0.25-specific
  amendments (12-agent topology, multi-round, parity metric,
  NativeCoordinator, tiered evidence actions, no persistent Python
  default).

## Architecture Readiness Audit

- ADR 0006 keeps Security Central as the permission and risk boundary.
  Native agents cannot authorize themselves; `:stocksage_analyze`
  remains the execution permission for the analysis as a whole. The
  5 evidence actions reuse the approved parent `RunAnalysis` context;
  outside an analysis (e.g., invoked through `mix allbert.delegate`), they
  use their own Resource Access Security Posture grants from v0.09+.
- ADR 0007 requires runtime-facing effectful behavior to enter through
  registered Jido actions. Specialist agents reason and return
  advisory packets (per ADR 0021 Section 4 authority rule). Market-data
  fetches, persistence, confirmations, and settings writes still go
  through registered actions.
- ADR 0008 requires durable confirmations for high-risk actions. v0.25
  reuses the `RunAnalysis` confirmation path from v0.22/v0.24
  unchanged.
- ADR 0017 says plugin code contributes capabilities through declared
  contracts. Native agents live under `./plugins/stocksage`, are
  supervised by `StockSage.Supervisor`, and are NOT added to Allbert
  core. The `mix allbert.delegate` Mix task IS in Allbert core (it
  exercises the v0.24 `DelegateAgent` registered action that any app
  can target).
- ADR 0020 defines the Python bridge baseline. v0.25 keeps explicit
  `--engine python` and `--engine both` comparison/reference runs and
  the bridge protocol intact while making the native engine the normal
  analysis path. Python is never invoked as automatic recovery from
  native failure, and no Settings Central key may make Python or
  parity mode the default operational path.
- ADR 0021 defines the objective and delegate-agent boundary. v0.25 is
  the first real consumer of `objective_steps.kind = :delegate_agent`,
  `AllbertAssist.Objectives.AgentRegistry`, and
  `AllbertAssist.Actions.Objectives.DelegateAgent`. v0.25 also
  validates the durable objective-step inspection path for a
  plugin-owned multi-agent graph.
- ADR 0022 binds the native financial specialist-agent contract;
  amended at M6 to enumerate the v0.25 topology + multi-round + parity
  metric + NativeCoordinator + tiered evidence actions.

## Vision And Roadmap Alignment

The Allbert vision says agents should be understandable, signal-driven,
and bounded by inspectable authority (`allbert-jido-vision.md`
L161-L205). v0.25 makes StockSage smarter without creating a private
autonomous trading system. Specialist agents may **propose**, rank,
predict, score, or summarize — they may not **authorize**, execute,
mark simulated state as observed truth, or short-circuit operator
confirmation (vision L198–L205; ADR 0021 §4).

Roadmap dependency order remains intact:

- v0.20 supplied StockSage plugin/domain storage.
- v0.22 supplied the Python bridge baseline and real persisted
  analyses.
- v0.23 supplied the JidoBacked internal runtime substrate.
- v0.24 supplied durable objective/step state, AgentRegistry,
  DelegateAgent action, and the generic app proposer substrate.
- **v0.25 now supplies reusable native financial specialist agents**,
  cross-app callable via `mix allbert.delegate`, with Jido.AI-backed
  report generation in the normal native path and deterministic
  advisory packets available only for fixture/test/operator-disabled
  smoke paths.
- v0.26 workspace shell will render objective progress and native agent
  steps.
- v0.27+ StockSage UI/polish will render richer native agent outputs.

v0.25 must NOT pull v0.26 LiveViews, v0.27 memory namespaces, v0.29
polish, or v0.30 canvas work forward.

The Jido agent surface grows materially in v0.25 — the project adds
11 LLM-capable specialist agents implemented as `Jido.Agent` signal
routers, 1 deterministic `Jido.Agent` quality gate, and 1 JidoBacked coordinator
alongside the existing `IntentAgent`. The pragmatic substrate rule
(`allbert-jido-vision.md` L120-L145; ADR 0007 v0.23 amendment)
supports this: each specialist is a named state machine with lifecycle
hooks (prompt/control file → evidence tools → bounded Jido.AI report
inside the delegate execute command)
that earns its keep.

## Guiding Rules

1. **Agents are advisory, actions are authoritative.** Specialist
   agents return bounded advisory packets. They never write
   `stocksage_*` rows, emit confirmations, call market-data
   providers directly, or bypass `Actions.Runner.run/3`. Per ADR 0021
   §4.

2. **Cross-app reusable by construction.** Every specialist agent is
   registered in `AllbertAssist.Objectives.AgentRegistry` at boot.
   The `mix allbert.delegate` Mix task (Allbert core, not StockSage)
   proves cross-app callability. Plugin code that calls a specialist
   does so through `Actions.Runner.run("delegate_agent", ...)` —
   never directly via PID or module call.

3. **Prompt provenance is per-file and auditable.** Every prompt at
   `plugins/stocksage/priv/prompts/native_agents/<role>.md` carries
   an `## Attribution` header naming source repo + commit + license,
   OR `## Attribution: Allbert-authored (Sandeep Puri, 2026-05-17)` for
   original content. M1 audits each one.

4. **Multi-round debate is a plugin-owned agent graph concern with
   objective-step observability.** The coordinator owns the
   StockSage-specific graph and round caps internally, while recording
   one durable `objective_step` per specialist turn. The generic
   v0.24 Objective Engine remains the lifecycle/inspection substrate;
   it is not asked to become a StockSage-specific graph scheduler.

5. **Parallel parity (`--engine both`) is a fan-out, not a fork.**
   Both engines run concurrently under a single `RunAnalysis`
   confirmation. Result persists as ONE `stocksage_analyses` row with
   both engines' fields populated and a parity diff computed at
   persistence time.

6. **Bounded LLM cost.** Each non-quality specialist has configurable
   model-profile routing (default `:fast`, with synthesizer + risk
   debaters defaulting to `:slow`/deep-think). v0.25 invokes Jido.AI
   only through the bounded specialist report boundary and only after
   the parent `RunAnalysis` confirmation. `quality_gate` stays
   deterministic.

7. **Fixture mode is a first-class operator surface, not a test-only
   shortcut.** `stocksage.native_evidence_mode = "fixture"` is
   supported in production (e.g., for smoke without market-data
   credentials). Fixtures
   are versioned, license-clear, and replayable.

8. **No native autonomous fallback to Python.** A native engine
   failure surfaces a clear error; the operator may choose to run an
   explicit Python reference/comparison command afterward. Silent
   fallback is forbidden by Readiness Decision.

9. **Coordinator is a JidoBacked agent.** `StockSage.Agents.NativeCoordinator`
   uses `use AllbertAssist.JidoBacked` per v0.23 substrate. Schema
   includes per-analysis projection (`current_analyses` map keyed by
   `objective_id`); state is a cache of the durable objective rows.

## Boundary Contract

### What Native Financial Agents Are

Native financial specialist agents are supervised Jido.AI agents (plain
Jido.Agent for deterministic gates like `quality_gate`) that accept a
bounded request packet and return a bounded advisory report packet.
Non-quality specialists use Jido.AI provider-backed structured
generation when `stocksage.native_llm_enabled` is true. They are
intentionally reusable:

- StockSage calls them through objective delegate steps via
  `StockSage.Agents.NativeCoordinator` → `Actions.Runner.run(
  "delegate_agent", ...)`.
- Any Allbert app may call the same agents through the same
  `delegate_agent` action. The `mix allbert.delegate` Mix task is the
  v0.25 operator-visible proof of this.
- CLI/LiveView/channels never call agent processes directly — always
  via the registered action.

### What Native Financial Agents Are Not

- They do not authorize execution. `:stocksage_analyze` (the parent
  RunAnalysis permission) and Resource Access Security Posture (the
  evidence-action permission set) are the only authorization gates.
- They do not write `stocksage_*` rows. Only
  `StockSage.Actions.RunAnalysis` persists. Coordinator passes the
  synthesized result back to `RunAnalysis` for persistence.
- They do not call market data providers directly. They call the 5
  registered evidence actions via `Actions.Runner.run/3`. The
  plugin-owned `StockSage.Agents` catalog enumerates exactly which
  evidence actions each specialist may invoke, and
  `StockSage.Agents.Commands.Execute` is the only delegate command
  that performs those calls (restricted tool scope per
  `project-direction-rethink-01.md` L390-L392).
- They do not emit confirmations. The parent `RunAnalysis`
  confirmation is the single approval gate.
- They do not schedule themselves. Multi-round debate is orchestrated
  by the coordinator; each specialist turn is recorded as a durable
  delegate-agent objective step.

### Request Packet

Every specialist agent accepts a normalized request packet (validated
via `Jido.Action` `schema:` per ADR 0017):

```elixir
%{
  request_id: String.t(),                  # UUID v7
  user_id: String.t(),                     # threaded from RunAnalysis
  ticker: String.t(),                      # bounded; uppercase
  analysis_date: String.t(),               # ISO 8601 date
  objective_id: String.t(),                # v0.24
  step_id: String.t(),                     # v0.24
  trace_id: String.t() | nil,              # v0.01 trace correlation
  role: atom(),                            # e.g., :market_context, :bull_thesis
  task: atom(),                            # e.g., :produce_report, :debate_round
  round_index: integer() | nil,            # debate cycle; nil for single-pass
  evidence: %{atom() => map()},            # prior evidence reports already gathered
  prior_reports: %{atom() => map()},       # prior round reports for multi-round agents
  constraints: %{
    max_tokens: pos_integer(),             # default per-agent profile
    timeout_ms: pos_integer(),             # default 180_000
    deep_think: boolean()                  # whether to use slow/deep model
  },
  prompt_version: String.t()               # `"v0.25.0"` initially; bumped per prompt revision
}
```

### Report Packet

Every specialist agent returns a normalized advisory report packet:

```elixir
%{
  agent_id: String.t(),                    # e.g., "stocksage.market_context"
  role: atom(),                            # same as request
  request_id: String.t(),                  # echoed back
  status: :ok | :error | :insufficient_evidence,
  summary: String.t() | nil,               # bounded 500 chars
  report: String.t() | nil,                # bounded 4000 chars; markdown
  evidence_used: [evidence_ref()],         # which evidence actions were called
  confidence: float() | nil,               # 0.0..1.0
  warnings: [String.t()],                  # bounded list of bounded strings
  data_requests: [%{action: String.t(), params: map()}],  # follow-up evidence needs
  generated_at: DateTime.t(),
  duration_ms: non_neg_integer(),
  model_profile: atom(),                   # which Jido.AI profile was used
  prompt_version: String.t()
}
```

Where `evidence_ref()` is:

```elixir
%{
  action: String.t(),                      # e.g., "stocksage_fetch_market_data"
  params: map(),                           # bounded; redacted
  result_summary: String.t() | nil,        # bounded 200 chars
  succeeded: boolean()
}
```

Reports are **advisory data only** (ADR 0021 §4). Only `RunAnalysis`
turns the coordinator's synthesized result into a `stocksage_analyses`
row.

## Native Agent Topology

v0.25 ships 12 specialist agents + 1 JidoBacked coordinator. All live
under `plugins/stocksage/lib/stocksage/agents/`. All are registered in
`AllbertAssist.Objectives.AgentRegistry` at app boot.

### Specialist Agents (12)

| Agent id | Module | Type | Role intent (Python source) | Default model profile |
|---|---|---|---|---|
| `stocksage.market_context` | `StockSage.Agents.MarketContext` | `Jido.Agent` signal router + Jido.AI execute command | Market/price/technical context, ≤8 indicators heuristic (Python `market_analyst.py`) | `:fast` |
| `stocksage.news_sentiment` | `StockSage.Agents.NewsSentiment` | `Jido.Agent` signal router + Jido.AI execute command | News + social sentiment (Python `news_analyst.py` + `sentiment_analyst.py` collapsed) | `:fast` |
| `stocksage.fundamentals` | `StockSage.Agents.Fundamentals` | `Jido.Agent` signal router + Jido.AI execute command | Fundamentals + company context (Python `fundamentals_analyst.py`) | `:fast` |
| `stocksage.bull_thesis` | `StockSage.Agents.BullThesis` | `Jido.Agent` signal router + Jido.AI execute command | Strongest bullish thesis; multi-round capable (Python `bull_researcher.py`) | `:fast` |
| `stocksage.bear_thesis` | `StockSage.Agents.BearThesis` | `Jido.Agent` signal router + Jido.AI execute command | Strongest bearish thesis; multi-round capable (Python `bear_researcher.py`) | `:fast` |
| `stocksage.risk_aggressive` | `StockSage.Agents.RiskAggressive` | `Jido.Agent` signal router + Jido.AI execute command | Aggressive risk debate (Python `risk_mgmt/aggressive_debator.py`) | `:slow` |
| `stocksage.risk_conservative` | `StockSage.Agents.RiskConservative` | `Jido.Agent` signal router + Jido.AI execute command | Conservative risk debate (Python `risk_mgmt/conservative_debator.py`) | `:slow` |
| `stocksage.risk_neutral` | `StockSage.Agents.RiskNeutral` | `Jido.Agent` signal router + Jido.AI execute command | Neutral risk debate (Python `risk_mgmt/neutral_debator.py`) | `:slow` |
| `stocksage.research_manager` | `StockSage.Agents.ResearchManager` | `Jido.Agent` signal router + Jido.AI execute command | Preliminary research decision from analysts + bull/bear debate (Python `research_manager.py`) | `:slow` |
| `stocksage.trader_plan` | `StockSage.Agents.TraderPlan` | `Jido.Agent` signal router + Jido.AI execute command | Bounded advisory investment plan from the preliminary research decision (Python `trader.py`) | `:slow` |
| `stocksage.decision_synthesizer` | `StockSage.Agents.DecisionSynthesizer` | `Jido.Agent` signal router + Jido.AI execute command | Final 5-point rating + investment_plan + trader_investment_plan + final_trade_decision after risk review (Python `portfolio_manager.py` posture, Allbert-bounded) | `:slow` |
| `stocksage.quality_gate` | `StockSage.Agents.QualityGate` | `Jido.Agent` (deterministic, no LLM) | Validate report shape, redaction, evidence-reference presence; reject malformed reports | n/a (no LLM call) |

The `decision_synthesizer` is not a bullish tie-breaker and does not
mechanically average agent ratings. It treats the bull, bear, and risk reports
as a committee record: conservative/neutral risk objections and the bear thesis
can dominate a constructive fundamentals packet when they identify material
valuation, balance-sheet, trend, catalyst, or evidence-completeness risk. This
keeps native parity work focused on LLM-backed committee reasoning instead of
post-hoc rating floors or ticker-specific correction logic.

The five-point scale follows the Python comparison engine's portfolio posture:
`Underweight` means reduced or below-benchmark exposure because forward
risk/reward is unfavorable. It is not the same as `Sell` and does not require a
broken business. Bear and conservative-risk specialists must preserve that
distinction so the final synthesizer can choose a cautious but non-exit stance
when valuation, per-share earnings, technical trend, catalyst, or data-quality
risks dominate.

The Jido.AI prompt envelope includes an advisory committee-context ledger built
from prior reports: ordered specialist stances, rating counts, risk committee
stances, and cautious-report summaries. This ledger is not a rule engine and
does not choose the final rating; it gives the LLM the same stable committee
shape every run so risk-manager and portfolio-manager semantics do not get lost
inside an unordered prior-report map.

### Coordinator (1)

| Module | Type | Purpose |
|---|---|---|
| `StockSage.Agents.NativeCoordinator` | `JidoBacked` (per v0.23 substrate) | Orchestrate multi-round bull/bear/risk debate; dispatch agents in correct order through the plugin-owned native graph while recording objective steps; compose parity (`--engine both`) by running coordinator + Python bridge concurrently; pass synthesized result back to `RunAnalysis` for persistence |

Coordinator state schema:

```elixir
schema: [
  current_analyses: [type: :map, default: %{}],     # objective_id => projection
  pending_dispatches: [type: :map, default: %{}],   # objective_id => [{agent_id, round_index}]
  parity_runs: [type: :map, default: %{}],          # objective_id => %{native: ref, python: ref}
  last_rebuilt_at: [type: :utc_datetime, default: nil]
]
```

`rebuild_state/1` cold-start callback rehydrates from in-flight
objectives where `status in [:open, :running, :blocked]` and a step
of `kind: :delegate_agent` exists.

### Why This Is Not A Python Clone

Python `tradingagents` (v0.2.5) ships 11+ agents organized as a
LangChain graph: 4 analysts, 2 researchers (bull/bear), 1 research
manager, 1 trader, 3 risk debaters, 1 portfolio manager.

v0.25's 12-agent topology preserves Python's research-manager,
trader, and final portfolio-manager handoff as separate native
specialists because live parity showed that collapsing them made the
native graph too neutral on risk-managed Underweight decisions. v0.25
still collapses news_analyst + sentiment_analyst into a single
`news_sentiment` (the v0.25 evidence actions handle source-tagged
fetches; the agent reasons over the combined stream). All three risk
debaters are preserved as independent agents because they materially
shape Python's final-decision quality and represent distinct
reasoning perspectives.

The 12-agent shape is documented in this plan and ADR 0022. v0.27+
may revise topology based on real-bridge parity data from v0.25 M6.

### Parity Diff Shape

```elixir
%{
  "rating_agreement" => 0.0..1.0,           # 1.0=exact, 0.5=adjacent, 0.0=distant on the 5-point scale
  "native_rating" => "Buy" | "Overweight" | "Hold" | "Underweight" | "Sell",
  "python_rating" => "Buy" | "Overweight" | "Hold" | "Underweight" | "Sell",
  "native_confidence" => 0.0..1.0,
  "python_confidence" => 0.0..1.0,
  "confidence_delta" => 0.0..1.0,           # |native - python|
  "within_variance" => boolean,             # confidence_delta < setting variance
  "parity_pass" => boolean,                 # rating_agreement >= 0.5 AND within_variance
  "computed_at" => DateTime.t()
}
```

"both"` rows. Surfaced via `mix stocksage.analyses show <id>` and
the future v0.26/v0.27 LiveView.

### Parity Tuning Closeout Posture

v0.25 parity hardening keeps the Python bridge as the trusted comparison
reference while moving the native graph closer to the TradingAgents role
sequence. The shipped native graph now preserves explicit
`research_manager`, `trader_plan`, risk-debate, and final
`decision_synthesizer` handoffs, and the final synthesizer receives a stable
committee-context ledger with ordered stances, rating counts, risk committee
summaries, and cautious-report excerpts.

This is not an exact-parity claim. `--engine both` is an inspectable comparison
mode for finding gaps, not an assertion that native and Python will always
return the same 5-point rating. Follow-up parity work belongs in a future
milestone and should focus on evidence-source coverage, prompt/role
calibration, and larger evaluation matrices. It must not introduce
ticker-specific post-processing, deterministic rating floors, or automatic
native-to-Python fallback.

present in existing settings schema, but v0.25 does not extend it and
`RunAnalysis` must not use it to choose the runtime engine. Engine
selection is request-scoped:

- no `engine` param → native;
- `engine: "native"` → native;
- `engine: "python"` → explicit Python reference/comparison if
  `stocksage.python_comparison_enabled = true`;
- `engine: "both"` / `--compare-python` → explicit parity run if
  `stocksage.python_comparison_enabled = true`;
- `"tradingagents"` remains a legacy alias for `"python"` only for
  old rows/callers and must never become the default.

### `:stocksage_evidence_fetch` permission class

- **Class name:** `:stocksage_evidence_fetch`

- **Safety floor:** `:resource_access_required` (Resource Access Security Posture from v0.09+)
- **Risk tier:** `:medium` (external API call; bounded by Resource Access grants)

## Non-Goals

- **No one-to-one port of TradingAgents.** v0.25 ships 12 specialist
  agents that preserve the research-manager/trader/final-decision handoff
  where parity requires it, while still consolidating Python roles where
  collapsing is natural (news_sentiment = news_analyst+sentiment_analyst).
- **No autonomous trading or order placement.** v0.25 produces
  advisory analyses only. Trades require operator action through
  channels Allbert does not provide in v0.25.
- **No silent Python fallback.** Native engine failures surface
  errors; operators choose whether to run an explicit Python
  reference/comparison command afterward.
- **No LLM-assisted acceptance evaluator for objectives.** v0.24's
  deterministic Evaluator continues to govern objective acceptance.
  Native specialist agents' outputs are evidence into the
  synthesizer; they do not authorize acceptance.
- **No v0.26 workspace shell or v0.27+ StockSage LiveViews.**
  Existing v0.20/v0.22 LiveViews continue to render analyses; richer
  native-agent visualization is v0.26/v0.27.
- **No advisory provider behaviour extraction** (ADR 0021 §3 reserved
  vocabulary). v0.25 specialist agents ARE advisory providers in
  spirit, but extracting a generic behaviour is premature until v0.27+
  when at least two consumer types exist (StockSage + e.g., future
  research-assistant app).
- **No unbounded native parallel step execution.** v0.25 allows
  stage-local parallelism only where reports are independent: 3 analyst
  agents, 3 risk perspectives per risk round, and explicit native/Python
  parity. Bull/bear rounds remain sequential so each stance can respond
  to the prior report. Each specialist dispatch is bounded by
  `stocksage.native_agent_timeout_ms`.
- **No public plugin hook contribution API.** Future Allbert apps can
  call existing StockSage specialist agents via DelegateAgent, but
  cannot contribute hooks into the coordinator pipeline.
