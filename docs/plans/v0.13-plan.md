# Allbert v0.13 Scheduled Jobs

## Purpose

v0.13 adds scheduled jobs that emit signals into the same runtime used by CLI,
LiveView, skills, actions, confirmations, execution adapters, Security Central,
and intent decisions.

Jobs should not become a separate automation system. They are another channel
into Allbert's runtime, with settings-backed schedules, security-aware actions,
confirmation behavior, and traceable outcomes.

This milestone was formerly tracked as v0.12 before the post-v0.10 roadmap
was unified into one v0.xx sequence.

## Expected Inputs

- v0.02 Settings Central exists for timezone, active/paused state, and schedule
  policy.
- v0.05 Security Central exists for policy, risk, redaction, and audit
  decisions.
- v0.06 action-backed skill contracts exist for known low-risk capabilities.
- v0.07 confirmation workflow exists for actions that require operator
  approval.
- v0.08 through v0.10 risky execution adapters are registered, confirmed, and
  traceable, including shell, skill script, external service, package install,
  and online skill import actions.
- v0.11 intent decisions can explain selected actions, confirmation needs,
  Approval Handoff, and resource access operation classes.
- v0.11 request-flow docs define the operator-visible manual verification
  baseline that jobs must preserve when they create confirmation-required
  resource access requests.
- v0.12 local workspace identity and conversation history provide originating
  `user_id` and `thread_id` context for jobs that are created from an
  interactive request. `app_id` remains nil/general until app-contract
  milestones add app-specific routing.

## Expected Direction

- Add cron-like jobs that emit signals into the same runtime.
- Start with registry health checks, trace summaries, and low-risk daily
  briefs.
- Use settings for timezone, active/paused state, and schedule policy.
- Keep scheduled jobs observable through traces and registered skills/actions.
- Pause confirmation-required job actions until the operator approves or denies
  the pending request.
- Jobs that create external service, package install, or online import requests
  must preserve job origin metadata and wait for the shared confirmation
  workflow; no scheduled job may silently approve its own v0.10 action.
- Jobs that request local path access, URL summaries, document inspection,
  local skill directory import, direct skill URL import, or other resource
  access consumers must emit the same posture and Approval Handoff metadata as
  CLI/web requests. They may create pending requests, but they must not read,
  fetch, summarize, import, execute, or remember approvals invisibly.
- Jobs preserve the originating `user_id`, `thread_id`, and `app_id` from the
  triggering request when those fields are present. This keeps job traces and
  audit records ready for the implemented v0.12 ownership context and later
  v0.15 app context without requiring a hosted accounts schema or later
  migration.
- Package-install jobs may create pending `run_package_install` requests only
  from exact package specs and allowed target roots. Job traces must preserve
  the package dry-run/execution argv preview so approval does not look like a
  generic background command.

## v0.11 Handoff

Jobs consume the v0.11 decision and Approval Handoff contracts as another
runtime channel. They should preserve `decision`, `resource_access`,
`approval_handoff`, confirmation id, and operation-scoped grant metadata in job
run records and traces. A scheduled URL summary, document inspection, skill
import, package install, shell command, or skill script request may create a
pending confirmation, but it must not self-approve or hide the target result
after approval.

## v0.12 Handoff

Jobs that originate from runtime turns should preserve the implemented v0.12
`user_id` and `thread_id` in job definitions, run records, signals, traces, and
pending confirmation origins. A job may create new conversation messages only
through the runtime path; job storage must not invent a separate conversation
history or bypass user-scoped thread checks.

## Non-Goals

- No background shell execution without confirmation.
- No external network polling without confirmation-aware adapters.
- No background package installs or online skill imports without durable
  confirmation.
- No background local path access, URL/document fetching, summarization,
  extraction, local skill directory import, or direct skill URL import without
  resource access posture and durable confirmation.
- No invisible autonomous side effects.
- No distributed scheduler.
- No hosted accounts or role-based ownership. Local jobs carry a string
  `user_id`, defaulting to `"local"`, and optional `thread_id` and `app_id`
  context from the originating request. These are strings, not foreign keys to
  an accounts table; real accounts and roles are deferred to a hosted
  production milestone.

## Definition Of Done

v0.13 is done when:

- Jobs are stored, listed, paused, resumed, and manually run.
- Scheduled runs emit normal runtime signals and traces.
- Low-risk built-in jobs can run without unsafe side effects.
- Confirmation-required job actions pause for operator approval.
- CLI and LiveView can inspect job state and recent run outcomes.
