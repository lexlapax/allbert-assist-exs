# Allbert v0.13 Scheduled Jobs

Formerly tracked as v0.12 before the post-v0.10 roadmap was unified into one
v0.xx sequence.

## Purpose

v0.13 adds local scheduled jobs that emit signals into the same Allbert runtime
used by CLI, LiveView, skills, actions, confirmations, Security Central, and
intent decisions.

Jobs are not a second automation system and not a shortcut around the runtime.
They are another local channel into the supervised Allbert kernel: schedules
are stored in SQLite, a supervised scheduler finds due work, each run uses
`AllbertAssist.Runtime.submit_user_input/1` or
`AllbertAssist.Actions.Runner.run/3`, and risky work still pauses through the
durable confirmation workflow.

v0.13 should make boring recurring work possible without adding hosted
accounts, distributed scheduling, app routing, new execution powers, or
autonomous side effects. The acceptance surface is CLI/runtime/signals/traces,
with thin LiveView inspection only.

## Expected Inputs

- v0.02 Settings Central exists. The current settings schema already contains
  `jobs.timezone` with default `"America/Los_Angeles"`,
  `jobs.default_state` with default `"paused"`, and
  `jobs.schedule_policy` with default `"operator_approved"` as read-only
  placeholders; v0.13 makes those specific keys writable and wires them to job
  behavior.
- v0.05 Security Central exists for policy, risk, redaction, and audit
  decisions.
- v0.06 action-backed skill contracts exist for known low-risk capabilities.
- v0.07 durable confirmation workflow exists, and confirmation records preserve
  origin and resolver context.
- v0.08 through v0.10 risky execution adapters are registered, confirmed, and
  traceable, including shell, trusted skill script, external service, package
  install, and online skill source actions.
- v0.11 intent decisions explain selected actions, confirmation needs,
  Approval Handoff, and resource access operation classes.
- v0.11 request-flow docs define the operator-visible manual verification
  baseline that jobs must preserve when they create confirmation-required
  resource access requests.
- v0.12 local workspace identity and conversation history are implemented.
  Runtime requests and responses carry `user_id`, `operator_id`, `thread_id`,
  and optional `session_id`; jobs must preserve originating `user_id`,
  `thread_id`, and nil/general `app_id` when available.

## Roadmap And Vision Fit

The Allbert/Jido vision says scheduled work should emit signals into the
supervised runtime instead of becoming a private automation stack. v0.13 is the
first background-work milestone and should prove that pattern with a small,
local, inspectable scheduler.

This milestone prepares later work without implementing it:

- v0.14 can add session scratchpad without changing job ownership or schedule
  storage.
- v0.15/v0.24 app contracts can add app-owned jobs later through `app_id`;
  v0.13 stores nil/general `app_id` only.
- v0.16 additional channels can reuse the same confirmation queue and job run
  inspection semantics.
- v0.20 intent enrichment can consider job context after jobs exist as normal
  runtime signals.
- v0.23 security evals can test background cross-user/thread leakage and
  confirmation bypass attempts against real job records.

The implementation should keep Allbert local-first and transparent: a due job
becomes a signal, the run is recorded, work enters the same runtime/action
boundaries as interactive work, and the operator can inspect what happened.
Job plans should remain Settings Central-aware, Security Central-aware, and
Allbert Home-local; they should not drift into hosted automation, invisible
memory promotion, external worker orchestration, or app-specific routing.

## v0.11 Handoff

Jobs consume the v0.11 decision and Approval Handoff contracts as another
runtime channel. They must preserve `decision`, `resource_access`,
`approval_handoff`, confirmation id, operation class, resource URI, approval
scope, and grant metadata in job run records and traces when those values are
available.

A scheduled URL summary, document inspection, skill import, package install,
shell command, or skill script request may create a pending confirmation, but
it must not self-approve, hide the pending state, or perform target work before
approval. Approval and denial remain registered confirmation actions.

## v0.12 Handoff

Jobs that originate from runtime turns should preserve the implemented v0.12
`user_id` and `thread_id` in job definitions, run records, signals, traces, and
pending confirmation origins. A job may create new conversation messages only
by submitting through the runtime path. Job storage must not invent a separate
conversation history, write directly to conversation tables, or bypass
user-scoped thread checks.

When a job has no stored `thread_id`, runtime prompt jobs use the v0.12 default
thread selection for their `user_id` unless the job explicitly uses
`thread_mode: "new_thread_per_run"`.

## Implementation Boundaries

Authoritative existing boundaries:

- Runtime entrypoint: `AllbertAssist.Runtime.submit_user_input/1`.
- Action execution: `AllbertAssist.Actions.Runner.run/3`.
- Action registry: `AllbertAssist.Actions.Registry`.
- Confirmation actions: `approve_confirmation`, `deny_confirmation`,
  `show_confirmation`, `list_confirmations`, and `expire_confirmations`.
- v0.11 contracts: `AllbertAssist.Intent.Decision`,
  `AllbertAssist.Intent.ResourceAccess`, and
  `AllbertAssist.Intent.ApprovalHandoff`.
- Conversation history: `AllbertAssist.Conversations`; jobs must not write
  conversation rows directly.
- Database: `AllbertAssist.Repo` with SQLite migrations under
  `apps/allbert_assist/priv/repo/migrations`.
- Settings: `AllbertAssist.Settings`; job settings belong there, while job
  definitions and run history belong in SQLite.

Preferred new implementation shape:

- `AllbertAssist.Jobs` context for creating, updating, listing, pausing,
  resuming, manually running, and inspecting jobs and runs.
- `AllbertAssist.Jobs.Job` Ecto schema for scheduled job definitions.
- `AllbertAssist.Jobs.Run` Ecto schema for durable run records.
- `AllbertAssist.Jobs.Schedule` for schedule validation and next-due
  calculation.
- `AllbertAssist.Jobs.Scheduler` supervised process for local due-job polling.
- `AllbertAssist.Jobs.Runner` for one-run execution and run-state updates.
- `Mix.Tasks.Allbert.Jobs` for CLI inspection and operator control.

The scheduler should be a local supervised process, not Oban, Quantum, a
distributed worker system, or an OS cron dependency. It may use existing
project dependencies for schedule parsing if useful, but the v0.13 contract is
the persisted schedule semantics below, not a dependency-specific API.

## Data Model

Create SQLite tables with string public ids.

`scheduled_jobs`:

- `id`: string primary key, generated as opaque `job_...`.
- `name`: operator-visible name, required, unique per `user_id`, max 120
  characters.
- `description`: string, nullable, redacted in traces if needed.
- `target_type`: string, required; allowed values are `"runtime_prompt"` and
  `"registered_action"`.
- `target`: map, required. For runtime prompt jobs this contains `text` and
  optional runtime metadata. For registered action jobs this contains
  `action_name` and `params`. The map is stored in full for execution. It must
  not contain raw secrets; store Settings Central secret refs or source/profile
  ids instead. Target contents are redacted when rendered in traces, logs, CLI
  output, and run summaries.
- `schedule`: map, required; see schedule contract below.
- `timezone`: string, required; defaults from `jobs.timezone`.
- `status`: string, required; allowed values are `"paused"`, `"active"`, and
  `"blocked"`.
- `user_id`: string, required, defaults to `"local"`.
- `operator_id`: string, required compatibility alias, defaults to `user_id`.
- `thread_id`: string, nullable; preserved from v0.12 when available.
- `thread_mode`: string, required; allowed values are `"origin_thread"`,
  `"recent_general"`, and `"new_thread_per_run"`.
- `session_id`: string, nullable; accepted as inert context until v0.14.
- `app_id`: string, nullable; nil means general Allbert context in v0.13.
- `channel`: string, required, default `"job"`.
- `next_due_at`: UTC datetime, nullable for manual or blocked jobs.
- `last_run_at`: UTC datetime, nullable.
- `blocked_confirmation_id`: string, nullable. When set, automatic scheduling
  skips the job until the referenced confirmation is no longer pending.
- `metadata`: map, default `%{}` for non-secret operator context.
- timestamps.

`scheduled_job_runs`:

- `id`: string primary key, generated as opaque `run_...`.
- `job_id`: string foreign key to `scheduled_jobs.id`.
- `status`: string, required; allowed values are `"queued"`, `"running"`,
  `"completed"`, `"needs_confirmation"`, `"failed"`, and `"skipped"`.
- `trigger`: string, required; allowed values are `"scheduler"` and
  `"manual"`.
- `due_at`: UTC datetime, nullable for manual runs.
- `started_at`: UTC datetime, nullable.
- `finished_at`: UTC datetime, nullable.
- `duration_ms`: integer, nullable.
- `user_id`: string, required, duplicated for user-scoped diagnostics.
- `operator_id`: string, required compatibility alias.
- `thread_id`: string, nullable.
- `session_id`: string, nullable.
- `app_id`: string, nullable.
- `input_signal_id`: string, nullable.
- `response_signal_id`: string, nullable.
- `trace_id`: string, nullable.
- `confirmation_id`: string, nullable.
- `decision`: map, default `%{}`; redacted v0.11 decision summary.
- `resource_access`: list/map, default empty; redacted v0.11 posture summary.
- `approval_handoff`: map, default `%{}`; redacted handoff summary.
- `action_log`: map, default `%{}`; redacted actions/result summary.
- `error`: map, default `%{}`; redacted failure details.
- `metadata`: map, default `%{}`.
- timestamps.

Indexes:

- jobs by `{status, next_due_at}` for scheduler polling.
- jobs by `{user_id, status}` for operator inspection.
- runs by `{job_id, inserted_at}` for recent run history.
- runs by `{user_id, inserted_at}` for user-scoped diagnostics/tests.

Do not add accounts tables, role tables, distributed lock tables, or a second
confirmation store.

## Schedule Contract

Persist schedules as maps so CLI, tests, and future LiveViews share one shape:

- Manual:
  `%{"kind" => "manual"}`
- Daily:
  `%{"kind" => "daily", "at" => "08:00"}`
- Weekly:
  `%{"kind" => "weekly", "weekday" => "monday", "at" => "08:00"}`
- Cron-like:
  `%{"kind" => "cron", "expression" => "0 8 * * 1-5"}`

Rules:

- Times are interpreted in the job's stored `timezone` and converted to UTC
  `next_due_at`.
- `at` uses 24-hour `HH:MM`.
- Weekly weekdays are lowercase English names or `mon` through `sun`; store
  the normalized full name.
- Cron expressions are five-field minute/hour/day/month/weekday expressions
  with minute granularity. No seconds and no sub-minute schedules.
- Invalid timezones, invalid schedule maps, impossible next run calculations,
  and unsupported cron syntax fail validation before inserting or updating a
  job.
- Manual jobs have `next_due_at: nil` and run only through manual commands.
- Paused and blocked jobs are not automatically due.
- `jobs.schedule_policy == "operator_approved"` allows active jobs to run
  automatically when due.
- `jobs.schedule_policy == "paused"` disables automatic due-job execution even
  for active jobs; manual runs are still allowed.

Settings defaults:

- `jobs.timezone`: `"America/Los_Angeles"` when not configured.
- `jobs.default_state`: `"paused"` when not configured; new jobs do not
  automatically run until the operator activates them.
- `jobs.schedule_policy`: `"operator_approved"` when not configured; the
  scheduler runs active due jobs automatically.

## Job Target Contract

`target_type: "runtime_prompt"`:

- Calls `AllbertAssist.Runtime.submit_user_input/1`.
- Required target field: `text`.
- Runtime request carries `channel: :job`, `user_id`, `operator_id`,
  `thread_id` according to `thread_mode`, optional `session_id`, nil/general
  `app_id`, and job metadata.
- Conversation messages are persisted only by the runtime.
- Risky prompts produce normal v0.11 decisions and Approval Handoff.

`target_type: "registered_action"`:

- Calls `AllbertAssist.Actions.Runner.run/3`.
- Required target fields: `action_name` and `params`.
- `action_name` must resolve through `AllbertAssist.Actions.Registry`.
- The runner context carries `channel: :job`, `job_id`, `run_id`, `user_id`,
  `operator_id`, `thread_id`, optional `session_id`, nil/general `app_id`, and
  origin metadata.
- Effectful actions still perform their own Security Central checks and may
  create durable confirmations.

No target may execute anonymous functions, shell strings, package manager
commands, HTTP requests, local path reads, or generated modules directly.

## Thread Mode Semantics

`thread_mode` controls only runtime prompt jobs:

- `"origin_thread"`: use the stored `thread_id`. If the thread no longer
  belongs to `user_id`, the run fails before runtime submission.
- `"recent_general"`: omit `thread_id` so v0.12 selects the user's recent
  general thread or creates one.
- `"new_thread_per_run"`: pass `new_thread: true` so each run creates a fresh
  general thread.

For registered action jobs, `thread_mode` is not used during execution. Store
`"recent_general"` as a neutral placeholder so the field is never nil. If a
registered action job stores a `thread_id`, that thread id is origin metadata
for traces, confirmations, and audits; it does not cause conversation history
writes.

Default:

- Jobs created from an interactive runtime turn use `"origin_thread"` when
  `thread_id` is present, or `"recent_general"` when it is absent.
- CLI-created jobs without `--thread` use `"recent_general"`.

## Scheduler Semantics

The scheduler is local and conservative:

1. On startup, it waits for Repo, Settings Central, PubSub, Jido Signal Bus,
   and Allbert runtime dependencies to be available.
2. On each tick, it reads `jobs.schedule_policy`.
3. If policy is `"paused"`, it emits/logs no due runs and sleeps.
4. It selects active jobs with non-nil `next_due_at <= now`.
5. It claims one run per due job in a transaction. A job with a currently
   running or pending-confirmation run is skipped.
6. It emits `allbert.job.due` and creates a `scheduled_job_runs` row.
7. It executes each claimed run through `AllbertAssist.Jobs.Runner`.
8. It updates run status, redacted result metadata, `last_run_at`, and the next
   `next_due_at`.

v0.13 does not need distributed locks. SQLite transactions and one supervised
local scheduler process are enough.

### Scheduler Restart Semantics

The scheduler must be restartable under OTP supervision and must keep no
authoritative in-memory queue. On startup or scheduler process restart:

1. Re-read active jobs from SQLite using `next_due_at`.
2. Preserve existing completed, failed, skipped, and needs-confirmation run
   history.
3. Detect any stale `"running"` run older than a test-configurable stale-run
   window, defaulting to five minutes.
4. Mark stale running runs as `"failed"` with redacted error kind
   `"scheduler_restarted"` so a crash cannot leave a job permanently locked.
5. Continue due-job polling from durable job rows without creating duplicate
   runs for a job that already has a non-stale running run.

## Signals And Observability

Use explicit job lifecycle signals:

- `allbert.job.due`
- `allbert.job.started`
- `allbert.job.completed`
- `allbert.job.needs_confirmation`
- `allbert.job.failed`
- `allbert.job.skipped`

Signals should include `job_id`, `run_id`, `trigger`, `user_id`,
`operator_id`, `thread_id`, `session_id` when present, `app_id`, `due_at`,
`started_at`/`finished_at` where relevant, and redacted status metadata.

Job run records are durable observability even when markdown trace recording is
disabled. When trace recording is enabled by existing runtime settings or a
job-specific trace option, job traces should include job/run ids and
user/thread context. Jobs must not write secrets or unredacted external
responses into run records, traces, or logs.

## Confirmation Semantics

If a job target returns `:needs_confirmation`:

- The run status becomes `"needs_confirmation"`.
- The run stores `confirmation_id`, redacted `decision`, `resource_access`,
  and `approval_handoff`.
- The job stores `blocked_confirmation_id` and `status: "blocked"` for
  automatic scheduling purposes.
- No target side effect occurs before approval.
- The confirmation origin includes `channel: "job"`, `job_id`, `run_id`,
  `user_id`, `operator_id`, `thread_id`, and `app_id` when available.
- Approval or denial still goes through registered confirmation actions.

v0.13 does not add a second job-specific approval queue and does not require
jobs to automatically continue after approval. Approval resumes the registered
target action exactly as v0.07-v0.11 define. The job run remains inspectable
with its confirmation id; later implementation may refresh resolved
confirmation status for display, but it must not invent new authority.

After the referenced confirmation is no longer pending, the operator may
resume the job or manually run it again. Automatic unblocking may be added if
it reads the existing confirmation state and does not execute target work as a
side effect of unblocking.

## CLI Contract

Add `mix allbert.jobs` with subcommands:

- `mix allbert.jobs list [--user USER] [--status active|paused|blocked]`
- `mix allbert.jobs show JOB_ID`
- `mix allbert.jobs runs JOB_ID [--limit N]`
- `mix allbert.jobs pause JOB_ID`
- `mix allbert.jobs resume JOB_ID`
- `mix allbert.jobs run JOB_ID`
- `mix allbert.jobs create runtime-prompt NAME --user USER --prompt TEXT
  [--thread THREAD_ID | --recent-thread | --new-thread-per-run]
  [--manual | --daily HH:MM | --weekly WEEKDAY@HH:MM | --cron EXPR]
  [--timezone TZ] [--active]`

Create commands default to `user_id: "local"`, `operator_id: user_id`,
`channel: "job"`, `status` from `jobs.default_state`, and timezone from
`jobs.timezone`. `--operator OPERATOR` is a compatibility alias for `--user`,
available from day one. If `--user` and `--operator` are both present and
differ, fail before writing. `--user alice --operator alice` is accepted.
Archive/delete controls are out of v0.13, so the list filter exposes only
`active`, `paused`, and `blocked`.

The initial v0.13 CLI may restrict arbitrary `registered_action` job creation
to built-in safe job templates. If a generic registered-action create command
is added, it must validate action registry metadata, permission class, params,
and redaction before storing the job.

## LiveView Contract

Add thin inspection in the existing operator web surface or settings surface:

- list jobs and statuses
- show recent runs and confirmation ids
- expose pause/resume/manual-run affordances only if they call the same
  `AllbertAssist.Jobs` context or registered actions used by CLI
- render Approval Handoff details for pending job confirmations by reusing
  existing handoff components

LiveView must not own schedule parsing, run execution, confirmation mutation,
security policy, or job storage semantics.

## Built-In Job Templates

v0.13 should provide a small set of documented, low-risk example job patterns.
These are CLI template presets the operator instantiates through
`mix allbert.jobs create template TEMPLATE_NAME ...`, not automatically seeded
database rows. Template creation expands into a normal `scheduled_jobs` row;
after creation, the scheduler does not treat template jobs specially. The job
metadata may record `template_name` for inspection.

Required initial templates:

- `daily-brief`: expands to a `runtime_prompt` job. It should use a
  side-effect-free default prompt unless the operator supplies `--prompt`.
- `registry-health`: expands to a `registered_action` job using a new
  read-only registered action for action/skill/settings registry health.
- `trace-summary`: expands to a `registered_action` job using a new read-only
  registered action for recent trace and job-run counts/failures.

The two registered-action templates require normal read-only Jido action
modules, registered in `AllbertAssist.Actions.Registry`, with permission
`:read_only`, execution mode `:read_only`, and confirmation `:not_required`.
They should return redacted summaries only.

Any template that can trigger shell, package, external network, local path,
skill import, or skill script behavior must go through the same runtime or
registered-action path and stop at confirmation when required.

## Acceptance

- `mix allbert.jobs create runtime-prompt daily-brief --user alice --prompt
  "give me a safe daily brief" --manual` creates a paused/manual job with
  `user_id: "alice"`.
- `mix allbert.jobs run JOB_ID` creates a run, emits job lifecycle signals,
  submits through the runtime, and records run metadata.
- `mix allbert.jobs create runtime-prompt alice-daily --user alice --daily
  08:00 --active --prompt "safe daily brief"` computes a UTC `next_due_at`
  from the stored timezone.
- An active due job runs from the supervised scheduler and updates
  `last_run_at` and the next due time.
- A job created with `--user alice --thread THREAD_ID` continues only an
  Alice-owned thread; Bob cannot run or inspect Alice's thread context through
  job commands.
- A risky URL/document/import/package/script prompt from a job returns
  `needs_confirmation`, creates a normal pending confirmation, records
  Approval Handoff in the run, and performs no target effect before approval.
- `mix allbert.confirmations show CONFIRMATION_ID` shows job origin metadata.
- After the scheduler supervisor process restarts, it resumes due-job polling
  from the durable `next_due_at` values in SQLite without losing run history.
- A stale `"running"` run left by a simulated scheduler crash is marked failed
  with `scheduler_restarted`, and the job can run again without duplicate
  in-flight runs.
- `mix allbert.jobs templates` lists the required templates, and
  `mix allbert.jobs create template daily-brief ...` creates a normal job row
  rather than relying on seeded rows.
- `mix allbert.jobs list`, `show`, and `runs` expose enough state for manual
  verification without reading raw database rows.
- LiveView can inspect jobs/runs and pending handoffs without owning execution
  or confirmation semantics.

## Non-Goals

- No background shell execution without confirmation.
- No external network polling without confirmation-aware adapters.
- No background package installs or online skill imports without durable
  confirmation.
- No background local path access, URL/document fetching, summarization,
  extraction, local skill directory import, direct skill URL import, or trusted
  skill script execution without v0.11 resource access posture and durable
  confirmation.
- No invisible autonomous side effects.
- No distributed scheduler, external cron dependency, Oban, Quantum, cluster
  leader election, or remote worker execution.
- No hosted accounts, authentication, roles, teams, or account foreign keys.
  Local jobs carry string `user_id`, defaulting to `"local"`, plus optional
  thread/app context.
- No app-specific job routing; nil/general `app_id` only until v0.15/v0.24.
- No session scratchpad semantics; `session_id` is propagated as inert context
  until v0.14.
- No archive/delete workflow. v0.13 uses pause/resume for operator control and
  preserves job/run history.
- No job-created markdown memory promotion. Runtime prompts may call explicit
  memory actions only through normal intent/action/confirmation boundaries.
- No separate confirmation queue, approval policy, grant store, trace store, or
  conversation-history writer for jobs.

## Test Plan

- Schema/migration tests prove job and run round-trips, required fields,
  allowed statuses, opaque ids, schedule maps, and indexes.
- Schedule tests cover daily, weekly, cron-like, manual, timezone conversion,
  invalid timezone, invalid time, invalid weekday, invalid cron syntax, and
  DST-adjacent next-due behavior where feasible.
- Jobs context tests cover create/list/show/pause/resume, user-scoped listing,
  status transitions, blocked confirmation ids, and next-due updates.
- Manual runner tests cover runtime prompt jobs, registered action jobs,
  success, failure, redacted error storage, trace id storage, and signal ids.
- Thread context tests prove `origin_thread`, `recent_general`, and
  `new_thread_per_run` semantics and cross-user thread rejection.
- Scheduler tests use short/test-configured intervals or direct tick helpers
  to prove due selection, one-run-per-job claiming, schedule policy pause,
  paused/blocked skips, next-due calculation, process restart recovery, and
  stale running-run cleanup.
- Confirmation tests prove risky job prompts/actions create pending
  confirmations, record Approval Handoff/resource posture, include job origin
  metadata, block duplicate pending runs, and do not execute before approval.
- CLI tests cover `mix allbert.jobs` list/show/runs/create/pause/resume/run,
  templates, template creation, invalid schedules, unknown jobs, invalid
  target/action, and user/thread isolation.
- LiveView tests cover read-only inspection and pause/resume/manual-run events
  delegating to the job context; no LiveView-owned execution logic.
- Existing v0.11/v0.12 focused tests continue to pass so jobs do not regress
  Approval Handoff or conversation history semantics.
- Final code gates follow `DEVELOPMENT.md`: `mix compile --warnings-as-errors`,
  `mix format --check-formatted`, `mix credo --strict`, `mix dialyzer`, and
  `mix precommit`.

## Definition Of Done

v0.13 is done when:

- Jobs are stored, listed, paused, resumed, manually run, and automatically run
  when due.
- Scheduled runs emit normal job lifecycle signals and use existing runtime or
  action-runner boundaries.
- Low-risk built-in jobs can run without unsafe side effects.
- Confirmation-required job work pauses for operator approval and records
  v0.11 Approval Handoff/resource posture.
- CLI and thin LiveView inspection can show job state and recent run outcomes.
- Job settings, run records, traces, and confirmations preserve v0.12
  `user_id`/`thread_id` context without adding hosted accounts or app routing.
