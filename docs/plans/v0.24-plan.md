# Allbert v0.24 Agentic Workspace Surface And Ephemeral UI Substrate

## Purpose

v0.24 upgrades the built-in `AllbertAssist.App.CoreApp` chat surface —
declared as a `SurfaceProvider` in v0.18 — from the rudimentary `/agent`
prompt into a signal-driven LiveView workspace. It is `CoreApp`'s surface
implementation, not a new shell floating outside the app contract.

Text remains valid, but Allbert should be able to open a persistent canvas or
task-scoped ephemeral UI when structured inspection, approval, editing, review,
or steering is clearer than another chat turn. The conversation timeline is the
primary surface; canvas and task surfaces are additive.

This milestone was formerly tracked as the old v0.17 before the post-v0.10
roadmap was unified into one v0.xx sequence, then shifted to v0.27 when the
plugin substrate was inserted ahead of StockSage, then to v0.24 when moved
before StockSage LiveViews.

## Expected Inputs

- v0.16 additional channels emit channel metadata and consume Approval Handoff.
- v0.18 app/surface contract and Surface DSL have landed:
  `AllbertAssist.App.Registry.registered_apps/0` is available for workspace
  shell navigation, `AllbertAssist.Surface` provides the component DSL for
  canvas artifacts, and `AllbertAssist.App.SurfaceProvider` is the behaviour
  for apps with interactive surface components. The canvas shell does not
  invent its own app discovery or surface node format.
- v0.19 intent enrichment explains selected skills, actions, jobs, channels,
  alternatives, confidence, and refusals.
- v0.21 memory review and retrieval expose operator review workflows.
- v0.22 Python bridge and v0.23 Native Jido agents both exist so the workspace
  can surface analysis state from either engine.
- Existing LiveView surfaces still call runtime, action, settings,
  confirmation, and resource-grant boundaries instead of owning domain logic.

## Expected Direction

- Implement the workspace shell lifecycle using v0.18's
  `AllbertAssist.Surface` component DSL and `AllbertAssist.App.SurfaceProvider`
  behaviour. v0.24 owns the runtime semantics: canvas persistence (tiles survive
  navigation), ephemeral surface scoping and discard (task-scoped, cleaned up
  on close), surface provenance, fallback text rendering when a component
  catalog entry is missing, redaction posture, and allowed event bindings.
  v0.24 owns the runtime semantics and does not redefine the component type
  catalog or app discovery format that v0.18 provides.
- Design the LiveView workspace shell around runtime state: conversation and
  signal timeline, canvas, task surfaces, approval inspector, trace inspector,
  memory review, job state, channel context, and security posture.
- Use Phoenix LiveView and OTP primitives as the local substrate: PubSub topics
  for runtime events, LiveView streams for timelines, `start_async/3` for
  long-running requests, and JS hooks only where canvas interaction needs
  client-side pointer, pan, zoom, or layout state.
- Route every surface event back through runtime or registered actions.
  LiveView renders and collects operator input; it does not approve, execute,
  fetch, import, summarize, mutate memory, or grant resources directly.
- Treat A2UI, AG-UI, MCP Apps, ChatGPT Canvas, Claude Artifacts, Gemini
  generative UI, BISCUIT, and Athena as research references. v0.24 should
  produce Allbert-native contracts first; external protocol compatibility is
  later work.

## Non-Goals

- No arbitrary model-generated HTML, JavaScript, Elixir modules, or LiveView
  components.
- No third-party MCP Apps or sandboxed remote UI execution.
- No A2UI renderer compatibility or AG-UI bridge as part of the first local
  substrate.
- No new execution, import, network, package, file, or memory powers.
- No UI-owned security policy, confirmation storage, resource-grant matching,
  intent routing, or action execution.

## Test Plan

- Workspace shell lifecycle tests prove `/agent` routes to the implemented
  signal-driven workspace surface declared by `CoreApp`.
- Canvas persistence tests prove persistent tiles survive navigation/reload,
  while ephemeral task surfaces are discarded on close or task completion.
- Surface validation tests reject unknown component ids, invalid action
  bindings, cross-app component theft, unredacted secret-bearing payloads, and
  model-invented node shapes.
- Fallback rendering tests prove missing catalog entries render bounded text
  with provenance instead of arbitrary HTML or crashing the LiveView.
- LiveView tests prove surface events call runtime or registered actions and
  never mutate confirmations, memory, resource grants, or security policy
  directly.
- Navigation tests prove registered apps from `AllbertAssist.App.Registry`
  populate workspace navigation.

## Definition Of Done

v0.24 is done when:

- `/agent` is replaced by the implemented signal-driven operator workspace
  surface declared by `CoreApp`.
- The workspace shell enforces v0.18's declarative surface
  contract at render time: validation against the component catalog, provenance
  metadata on all generated nodes, fallback text for missing catalog entries,
  redaction, accessibility, canvas and ephemeral lifecycle rules, and action
  binding constraints are all applied by the shell, not re-defined by it.
- Canvas and ephemeral UI semantics are distinct: canvas persists user-facing
  work, while ephemeral surfaces are task-scoped and cleanly discarded.
- Runtime, action, confirmation, memory, job, channel, and trace events can be
  rendered without duplicating domain logic in LiveView.
- Registered apps from `AllbertAssist.App.Registry` populate workspace shell
  navigation.
- Canvas component catalogs are validated against the `AllbertAssist.Surface`
  DSL, not arbitrary model-generated node shapes.
- Security tests or eval plans prove generated surfaces cannot invent actions,
  permissions, resource identities, scripts, URLs, or secret-bearing output.
- The roadmap records which protocol interop work belongs to later post-v0.29
  work.
