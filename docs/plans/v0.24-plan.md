## Purpose

v0.24 introduces an objective runtime layer between intent selection
and action execution. The intent layer continues to interpret the
user's immediate input and rank candidate routes. The action layer
continues to execute one bounded, authorized capability at a time.
Objectives become responsible for durable outcome state across one or
more steps, confirmations, jobs, channels, and turns.

The milestone adds three durable record types (`objectives`,
`objective_steps`, `objective_events`), one supervised
`JidoBacked`-pattern agent (`AllbertAssist.Objectives.Engine.Agent`), a
seven-stage state machine, four registered CLI/LiveView actions for
inspection and lifecycle, eleven new signals, and four new settings
keys. It threads `objective_id` and `step_id` through Confirmations,
Jobs, traces, and StockSage `RunAnalysis`. It does not ship advisory
provider behaviours, world-model abstractions, capability inventory
aggregation, route proposal, acquisition options, or any planner beyond
the deterministic step proposer needed for the first one-step and
two-step objectives.

Scope discipline matters here. The earlier rethink draft proposed a
sixteen-stage pipeline with 60+ named hook points, two provider
umbrellas, nine reserved provider roles, and 15+ settings keys. v0.24
ships the minimum implementation slice the rethink doc now describes:
seven engine stages, six step kinds (including the **minimal
`:delegate_agent`** that unblocks v0.25), five named hook points, four
settings keys, three tables, eleven signals. Reserved vocabulary
(advisory providers, world model, capability/route/acquisition,
parallelism settings) is documented in ADR 0021 and the research note
but not implemented.

## Architecture Readiness Audit

- ADR 0006 keeps Security Central as the permission, risk, redaction,
  and audit boundary. Objective steps that mutate, fetch, send,
  execute, or contact external systems still flow through
  `Actions.Runner.run/3` and Security Central. `objective_id` carries
  no permission.
- ADR 0007 requires runtime-facing effectful behavior to enter through
  registered Jido actions. All objective lifecycle actions
  (`list_objectives`, `show_objective`, `cancel_objective`,
  `continue_objective`) are registered Jido actions. The engine agent
  itself dispatches commands internally; external callers go through
  the action runner.
- ADR 0008 keeps durable confirmation requests in SQLite. Confirmation
  records gain optional `objective_id` and `step_id` columns. Existing
  confirmations from prior versions remain valid (NULL in both
  columns).
- ADR 0014 keeps `user_id` as the identity boundary. Every objective
  has `user_id`. Cross-user objective access returns `:not_found`.
- ADR 0015 keeps `active_app` as session context, not authorization.
  Objectives capture `active_app` at creation; ranking and rendering
  can use it; permission decisions cannot.
- ADR 0019 invariants for intent candidates hold. v0.24 adds
  `:objective` as a candidate kind via the new
  `Intent.Engine.collect_candidates/2` arity. Objective candidates are
  proposal data; they do not grant permissions or change execution mode.
- ADR 0021 (accepted at M6) defines the intent / objective / step /
  observation / capability / route / acquisition / advisory provider /
  world model / action authority boundary. This plan derives from ADR
  0021 and amends it for v0.24-specific additions (see Readiness
  Decision above).

## Vision And Roadmap Alignment

The vision describes Allbert as a signal-driven runtime where intent
is interpreted, capabilities are selected through registered actions,
permissions are checked, results and traces are recorded, and the
operator can inspect and steer work. v0.24 makes the "across multiple
turns" part of that loop a first-class concept.

Roadmap dependency order:

- v0.07 supplies the confirmation workflow that objective steps
  depend on for risky-action gating.
- v0.12 supplies the `Thread` + `Message` records that objectives
  reference via `source_thread_id`.
- v0.13 supplies the scheduled-job runtime that may create or advance
  objectives.
- v0.18 supplies the app/surface contract that objective views are
  rendered through.
- v0.19 supplies the intent engine that v0.24 extends with
  `:objective` candidate kind.
- v0.20 supplies StockSage as the first proving app.
- v0.21 supplies memory promotion that objective reflection steps may
  propose candidates for.
- v0.22 supplies the `RunAnalysis` confirmation flow that v0.24
  rewraps inside an objective.
- v0.23 supplies the `JidoBacked` substrate so the engine can compose
  with `Confirmations.Store.Agent` and `Jobs.Scheduler.Agent` via the
  same pattern.

v0.24 must not pull later work forward. It does not implement:

- v0.25 native trading agents (Objectives.Engine has only a
  deterministic step proposer; LLM-assisted planners and StockSage
  specialist agents are v0.25+). v0.24 ships the minimal
  `:delegate_agent` step kind contract so v0.25 has a binding target;
  v0.24 does not implement the specialist agents themselves.
- v0.26 workspace shell or canvas (objective views render through
  existing LiveView surfaces; canvas integration is v0.30).
- v0.28 security hardening evals over objective surfaces (the
  engine's security tests cover correctness; adversarial evals are
  v0.28).
- Advisory provider behaviours, world-model abstractions, capability
  inventory aggregation, route proposal, acquisition options,
  diffusion proposers, market allocators, or probabilistic inference
  providers. These are reserved vocabulary in ADR 0021 only.

## Guiding Rules

1. **Objective state is not authority.** `objective_id` never grants
   permission. `active_app` may scope ranking and objective context,
   but not permission. Every step that mutates, fetches, sends,
   executes, or contacts external systems grounds to a registered
   action and Security Central. The `:objective_write` permission
   class (kept for symmetry with other `_write` classes and for future
   per-objective ACL scoping in v0.31+) defaults to `:allow` with
   safety floor `:allow`; ADR 0021 is amended to enumerate it.

2. **SQLite is authoritative.** `objectives`, `objective_steps`, and
   `objective_events` rows are durable truth. The engine agent's
   in-memory state is a bounded projection of the current active
   objective(s); it rebuilds from SQLite on restart or cold start.

3. **Bounded loops with diagnostic verdict.** Every objective has
   explicit caps: `objectives.max_steps_per_turn` (default 3),
   `objectives.max_loop_count` (default 5). Exceeding a cap records an
   `impasse` event. When the acceptance evaluator returns
   `:needs_more_steps` AND `loop_count >= max_loop_count`, the cap
   wins (transition to `:blocked`/`:impasse`); the would-have-continued
   verdict is recorded in `objective_events.payload` for diagnostics so
   the impasse is never a black box.

4. **Cooperative cancellation only.** `cancel_objective` transitions
   status to `:cancelled`, blocks new step creation, and lets in-
   flight registered actions complete normally. Mid-action
   interruption is deferred to v0.25+.

5. **Coexisting signals with trace correlation.** `allbert.objective.*`
   signals emit alongside existing `allbert.action.*` and
   `allbert.runtime.turn.*` events. No existing signal is removed or
   renamed. For single-step objectives, both
   `allbert.runtime.turn.completed` and `allbert.objective.completed`
   fire; both carry the same `trace_id` so consumers can correlate.
   Trace volume is bounded by `objectives.trace_detail`.

6. **Cross-user isolation.** All objective operations filter by
   `user_id`. The CLI defaults to `"local"` only at the operator
   boundary. Actions never default `user_id` internally.

7. **Deterministic acceptance with hardcoded per-app proposer.**
   Acceptance criteria are evaluated deterministically against
   observation summaries and step results. The Proposer is a
   hardcoded Elixir module per app (`StockSage.Proposer`, future
   `Memory.Proposer`, etc.) exposing
   `propose(intent_decision, context)`; settings only toggle which
   apps are enabled, not the proposal rules themselves. An LLM-assisted
   evaluator is reserved as vocabulary in ADR 0021 but not implemented;
   v0.24 completion is a function of structured data, not free-form
   model output.

8. **Reserved vocabulary stays reserved.** Capability inventory,
   capability gap, route, acquisition option, advisory provider,
   world model, diffusion proposer, market allocator, and
   probabilistic inference provider are documented in ADR 0021 and
   the research note. They are not modules, settings, signals, or
   step kinds in v0.24.

9. **Pragmatic substrate rule.** `Objectives.Engine.Agent` is a
   JidoBacked agent (using the v0.23 substrate) because it is a stage
   state machine with lifecycle hooks that carry their weight. Per
   v0.23's pragmatic rule, this is a per-component decision, not a
   uniform policy.

10. **Eager rehydration with bounded prune.** On
    `AllbertAssist.JidoBacked.Supervisor` boot,
    `Objectives.Engine.Agent.rebuild_state/1` (JidoBacked callback)
    scans `objectives` where `status` in `[:open, :running, :blocked]`
    and loads them into the in-memory projection. Rows with
    `updated_at` older than 1 hour are marked `:abandoned` (new
    terminal status) and pruned from memory at boot; their SQLite rows
    are preserved for forensic inspection. This contract is documented
    in the engine's `@moduledoc` and verified by the M1 cold-start
    test.

v0.23 readiness requirement: `AllbertAssist.JidoBacked.__using__/1`
must pass this richer schema through to `use Jido.Agent` so `%{}` defaults
initialize as runtime maps. If v0.24 sees quoted syntax-tree defaults here,
fix the substrate before implementing objective commands.

In-memory state is bounded by `objectives.max_active_per_user` (reserved;
default 16). Engine state is a cache; SQLite is authoritative. Step rows
persist on **every status transition** (see Schema below) so a crash mid-
stage replays from the last persisted state.

**Naming convention** (renamed in v0.24 third validation pass per the
v0.23 substrate boundary):

- Private engine command modules live under
  `AllbertAssist.Objectives.Commands.*` (files:
  `apps/allbert_assist/lib/allbert_assist/objectives/commands/<name>.ex`).
  These use `Jido.Action` for schema validation and are dispatched only
  by `Objectives.Engine.Agent`. They are **not** registered in
  `AllbertAssist.Actions.Registry`, **not** intent candidates, and
  **not** operator-callable. The v0.23 boundary rule (ADR 0007
  amendment) applies: private Jido command modules are not Allbert
  capability actions.
- Registered operator-facing capability actions live under
  `AllbertAssist.Actions.Objectives.*` (files:
  `apps/allbert_assist/lib/allbert_assist/actions/objectives/<name>.ex`).
  These are registered in `Actions.Registry`, selectable by intent,
  callable through `Actions.Runner.run/3`, and gated by Security
  Central.

The two namespaces are intentionally distinct so a quick visual scan
distinguishes "engine internal" from "operator-facing capability".
`CancelObjective` and `ContinueObjective` appear in both — the
registered action accepts the operator call, runs Security Central,
and then dispatches into the private engine command via
`Objectives.Engine.Agent.cmd/2`. They are separate modules.

## `:objective_write` Permission Class

New permission class added to Security Central:

- **Class name:** `:objective_write`
- **Default policy:** `:allow` for the objective's owning `user_id`.
- **Safety floor:** `:allow` (this is not a high-risk class; the work
  that an objective performs goes through other action-level
  permission classes — `:stocksage_analyze`, `:memory_write`, etc.).
- **Settings Central key:** `permissions.objective_write` (writable).
- **Risk tier:** `:low` (operator can change policy without unlocking
  a higher tier).
- **Confirmation default:** `:allow` (no confirmation prompt for cancel/
  continue).

The class exists for symmetry with other write classes
(`:memory_write`, `:stocksage_write`, `:settings_write`) and for
future ability to scope objective creation/cancellation when hosted
multi-user lands (v0.31+). ADR 0021 is amended in M6 to explicitly
enumerate this class.

### Two acceptance flows for v0.24

**Single-step objective:**

```
User: "analyze AAPL" (with active_app: :stocksage)
  → Intent.Engine ranks RunAnalysis candidate
  → Engine frames objective: title "Analyze AAPL"
  → StockSage.Proposer.propose/2 returns one step
  → Engine authorizes; Security Central returns :needs_confirmation
  → Confirmation created with objective_id + step_id
  → Operator approves
  → Step transitions selected → running → completed
  → Engine evaluates acceptance: one completed analysis for AAPL on today's date → :met
  → Objective transitions to :completed
  → allbert.runtime.turn.completed + allbert.objective.completed emit with shared trace_id
```

**Two-step "analyze and compare" objective:**

```
User: "analyze AAPL and compare to MSFT"
  → Engine frames objective: title "Analyze AAPL and compare to MSFT"
  → StockSage.Proposer.propose/2 returns two steps; step 2.parent_step_id = step 1.id
  → Step 1 follows single-step flow → completed
  → Acceptance returns :needs_more_steps (one of two analyses present)
  → Engine increments loop_count, advances to step 2
  → Step 2 follows the same flow → completed
  → Acceptance returns :met
  → Objective transitions to :completed
```

Both flows are exercised in M5 acceptance smokes.

## Intent Engine Integration

ADR 0019 is amended at M2 to register `:objective` as a candidate kind.
`AllbertAssist.Intent.Engine` gains a new arity:

```elixir
@spec collect_candidates(decision_input :: map()) :: [Candidate.t()]
@spec collect_candidates(decision_input :: map(), opts :: keyword()) :: [Candidate.t()]
def collect_candidates(decision_input), do: collect_candidates(decision_input, [])
def collect_candidates(decision_input, opts) do
  objective = Keyword.get(opts, :objective)
  # ... existing candidates ...
  candidates ++ objective_candidates(objective, decision_input)
end
```

Behavior:

- When the operator has open objectives matching the current turn's
  topic (deterministic keyword + recency match), the engine returns an
  objective-resume candidate.
- The candidate's `source` is `:objective`; `score` is 0.2–0.5 based
  on recency and topic match; `reason` is the objective title.
- ADR 0019 invariants hold: objective candidates do not grant
  permissions; do not authorize actions; do not skip Security
  Central.

`Candidate.bound/2` adds `objective: 5` to `@default_kind_limits`.

Memory candidates (v0.21) and objective candidates can coexist in
the same ranking pass.

The new arity is backward-compatible: existing callers that use
`collect_candidates/1` continue to work; the engine sniffs `:objective`
from the new arg only.

## Non-Goals

- No advisory provider behaviour (reserved in ADR 0021).
- No world-model abstraction (reserved).
- No capability inventory aggregation as a module (reserved).
- No route, acquisition option, or planner beyond the deterministic
  per-app proposers.
- No diffusion proposer, market allocator, probabilistic inference
  provider.
- No parallel step execution (`max_parallel_steps` reserved at 1).
- No mid-action interruption (cooperative cancellation only).
- No LLM-assisted acceptance evaluator (deterministic only).
- No automatic memory promotion from objective observations; the
  v0.21 review surface remains the only path.
- No native StockSage trading agents (v0.25 — but v0.24 ships the
  minimal `:delegate_agent` step kind contract they will consume).
- No workspace shell upgrade (v0.26).
- No canvas integration (v0.30).
- No public plugin hook contribution API.
- No automatic capability acquisition (install plugin, write code,
  request credential) from an objective step.
- No settings-driven proposer rules (proposers are hardcoded Elixir
  per app).
