# Allbert v0.24 Objective Runtime Foundation

## Status

Status: planned (final implementation-readiness pass 2026-05-17).

v0.23 implementation is closed out and release docs are ready; the release
tag remains pending operator acceptance. This pass bakes in concrete
contracts exposed by the v0.23 substrate: JidoBacked signal-route command
dispatch, hybrid proposer contract for multi-step
propose-execute-reflect cycles, structured-map acceptance criteria,
Jido/OTP-registry-backed `AgentRegistry`, explicit web-side `SignalBridge`
for LiveView reactivity, default-config registration of Engine.Agent in
JidoBacked debug-trace, error policy, loop-count cadence, terminal
`:abandoned` semantics, and confirmation render staleness handling.
Directive-only JidoBacked dispatch support landed in v0.23 closeout and is an
expected input for v0.24, not new v0.24 work.

v0.24 introduces an objective runtime layer between intent selection and
action execution. It depends on v0.23 Jido State-Machine Convergence and
the `AllbertAssist.JidoBacked` substrate shipped there.

## Purpose

v0.24 introduces an objective runtime layer between intent selection
and action execution. The intent layer continues to interpret the
user's immediate input and rank candidate routes. The action layer
continues to execute one bounded, authorized capability at a time.
Objectives become responsible for durable outcome state across one or
more steps, confirmations, jobs, channels, and turns.

The milestone adds three durable record types (`objectives`,
`objective_steps`, `objective_events`), one supervised
`JidoBacked`-pattern agent (`AllbertAssist.Objectives.Engine.Agent`), a
seven-stage state machine, four registered CLI/LiveView actions for
inspection and lifecycle, eleven new signals, and four new settings
keys. It threads `objective_id` and `step_id` through Confirmations,
Jobs, traces, and StockSage `RunAnalysis`. It does not ship advisory
provider behaviours, world-model abstractions, capability inventory
aggregation, route proposal, acquisition options, or any planner beyond
the deterministic step proposer needed for the first one-step and
two-step objectives.

Scope discipline matters here. The earlier rethink draft proposed a
sixteen-stage pipeline with 60+ named hook points, two provider
umbrellas, nine reserved provider roles, and 15+ settings keys. v0.24
ships the minimum implementation slice the rethink doc now describes:
seven engine stages, six step kinds (including the **minimal
`:delegate_agent`** that unblocks v0.25), five named hook points, four
settings keys, three tables, eleven signals. Reserved vocabulary
(advisory providers, world model, capability/route/acquisition,
parallelism settings) is documented in ADR 0021 and the research note
but not implemented.

## Architecture Readiness Audit

- ADR 0006 keeps Security Central as the permission, risk, redaction,
  and audit boundary. Objective steps that mutate, fetch, send,
  execute, or contact external systems still flow through
  `Actions.Runner.run/3` and Security Central. `objective_id` carries
  no permission.
- ADR 0007 requires runtime-facing effectful behavior to enter through
  registered Jido actions. All objective lifecycle actions
  (`list_objectives`, `show_objective`, `cancel_objective`,
  `continue_objective`) are registered Jido actions. The engine agent
  itself dispatches commands internally; external callers go through
  the action runner.
- ADR 0008 keeps durable confirmation requests in SQLite. Confirmation
  records gain optional `objective_id` and `step_id` columns. Existing
  confirmations from prior versions remain valid (NULL in both
  columns).
- ADR 0014 keeps `user_id` as the identity boundary. Every objective
  has `user_id`. Cross-user objective access returns `:not_found`.
- ADR 0015 keeps `active_app` as session context, not authorization.
  Objectives capture `active_app` at creation; ranking and rendering
  can use it; permission decisions cannot.
- ADR 0019 invariants for intent candidates hold. v0.24 adds
  `:objective` as a candidate kind via the new
  `Intent.Engine.collect_candidates/2` arity. Objective candidates are
  proposal data; they do not grant permissions or change execution mode.
- ADR 0021 (accepted at M6) defines the intent / objective / step /
  observation / capability / route / acquisition / advisory provider /
  world model / action authority boundary. This plan derives from ADR
  0021 and amends it for v0.24-specific additions (see Readiness
  Decision above).

## Vision And Roadmap Alignment

The vision describes Allbert as a signal-driven runtime where intent
is interpreted, capabilities are selected through registered actions,
permissions are checked, results and traces are recorded, and the
operator can inspect and steer work. v0.24 makes the "across multiple
turns" part of that loop a first-class concept.

Roadmap dependency order:

- v0.07 supplies the confirmation workflow that objective steps
  depend on for risky-action gating.
- v0.12 supplies the `Thread` + `Message` records that objectives
  reference via `source_thread_id`.
- v0.13 supplies the scheduled-job runtime that may create or advance
  objectives.
- v0.18 supplies the app/surface contract that objective views are
  rendered through.
- v0.19 supplies the intent engine that v0.24 extends with
  `:objective` candidate kind.
- v0.20 supplies StockSage as the first proving app.
- v0.21 supplies memory promotion that objective reflection steps may
  propose candidates for.
- v0.22 supplies the `RunAnalysis` confirmation flow that v0.24
  rewraps inside an objective.
- v0.23 supplies the `JidoBacked` substrate so the engine can compose
  with `Confirmations.Store.Agent` and `Jobs.Scheduler.Agent` via the
  same pattern.

v0.24 must not pull later work forward. It does not implement:

- v0.25 native trading agents (Objectives.Engine has only a
  deterministic step proposer; LLM-assisted planners and StockSage
  specialist agents are v0.25+). v0.24 ships the minimal
  `:delegate_agent` step kind contract so v0.25 has a binding target;
  v0.24 does not implement the specialist agents themselves.
- v0.26 workspace shell or canvas (objective views render through
  existing LiveView surfaces; canvas integration is v0.30).
- v0.28 security hardening evals over objective surfaces (the
  engine's security tests cover correctness; adversarial evals are
  v0.28).
- Advisory provider behaviours, world-model abstractions, capability
  inventory aggregation, route proposal, acquisition options,
  diffusion proposers, market allocators, or probabilistic inference
  providers. These are reserved vocabulary in ADR 0021 only.

## Guiding Rules

1. **Objective state is not authority.** `objective_id` never grants
   permission. `active_app` may scope ranking and objective context,
   but not permission. Every step that mutates, fetches, sends,
   executes, or contacts external systems grounds to a registered
   action and Security Central. The `:objective_write` permission
   class (kept for symmetry with other `_write` classes and for future
   per-objective ACL scoping in v0.31+) defaults to `:allow` with
   safety floor `:allow`; ADR 0021 is amended to enumerate it.

2. **SQLite is authoritative.** `objectives`, `objective_steps`, and
   `objective_events` rows are durable truth. The engine agent's
   in-memory state is a bounded projection of the current active
   objective(s); it rebuilds from SQLite on restart or cold start.

3. **Bounded loops with diagnostic verdict.** Every objective has
   explicit caps: `objectives.max_steps_per_turn` (default 3),
   `objectives.max_loop_count` (default 5). Exceeding a cap records an
   `impasse` event. When the acceptance evaluator returns
   `:needs_more_steps` AND `loop_count >= max_loop_count`, the cap
   wins (transition to `:blocked`/`:impasse`); the would-have-continued
   verdict is recorded in `objective_events.payload` for diagnostics so
   the impasse is never a black box.

4. **Cooperative cancellation only.** `cancel_objective` transitions
   status to `:cancelled`, blocks new step creation, and lets in-
   flight registered actions complete normally. Mid-action
   interruption is deferred to v0.25+.

5. **Coexisting signals with trace correlation.** Objective namespace
   signals emit alongside existing `allbert.input.received`,
   `allbert.agent.responded`, and `allbert.action.*` signals. v0.24
   also formalizes the CoreApp-declared `allbert.runtime.turn.started`
   and `allbert.runtime.turn.completed` signals; no existing signal is
   removed or renamed. For single-step objectives, both
   `allbert.runtime.turn.completed` and `allbert.objective.completed` fire;
   both carry the same `trace_id` so consumers can correlate. Trace volume is
   bounded by `objectives.trace_detail`.

6. **Cross-user isolation.** All objective operations filter by
   `user_id`. The CLI defaults to `"local"` only at the operator
   boundary. Actions never default `user_id` internally.

7. **Deterministic acceptance with hardcoded per-app proposer.**
   Acceptance criteria are evaluated deterministically against
   observation summaries and step results. The Proposer is a
   hardcoded Elixir module per app (`StockSage.Proposer`, future
   `Memory.Proposer`, etc.) exposing
   `propose(intent_decision, context)`; settings only toggle which
   apps are enabled, not the proposal rules themselves. An LLM-assisted
   evaluator is reserved as vocabulary in ADR 0021 but not implemented;
   v0.24 completion is a function of structured data, not free-form
   model output.

8. **Reserved vocabulary stays reserved.** Capability inventory,
   capability gap, route, acquisition option, advisory provider,
   world model, diffusion proposer, market allocator, and
   probabilistic inference provider are documented in ADR 0021 and
   the research note. They are not modules, settings, signals, or
   step kinds in v0.24.

9. **Pragmatic substrate rule.** `Objectives.Engine.Agent` is a
   JidoBacked agent (using the v0.23 substrate) because it is a stage
   state machine with lifecycle hooks that carry their weight. Per
   v0.23's pragmatic rule, this is a per-component decision, not a
   uniform policy.

10. **Eager rehydration with bounded prune.** On
    `AllbertAssist.JidoBacked.Supervisor` boot,
    `Objectives.Engine.Agent.rebuild_state/1` (JidoBacked callback)
    scans `objectives` where `status` in `[:open, :running, :blocked]`
    and loads them into the in-memory projection. Rows with
    `updated_at` older than 1 hour are marked `:abandoned` (new
    terminal status) and pruned from memory at boot; their SQLite rows
    are preserved for forensic inspection. This contract is documented
    in the engine's `@moduledoc` and verified by the M1 cold-start
    test.

### Engine command dispatch

Engine.Agent follows the v0.23 JidoBacked signal-route pattern. Public helper
functions dispatch named signals with `AllbertAssist.JidoBacked.dispatch/4`;
private command modules are `Jido.Action` modules in
`AllbertAssist.Objectives.Commands.*`; `Jido.AgentServer` routes the signal to
the command module and the Jido strategy executes it. Do **not** define a
custom `cmd/3` on the agent: `use Jido.Agent` already provides `cmd/2` and
`cmd/3`, and overriding that generated function would fight the substrate.

The substrate's `command_modules/0` callback enumerates eligible modules for
the authority-boundary assertion API; it is not an action registry.

```elixir
defmodule AllbertAssist.Objectives.Engine.Agent do
  alias AllbertAssist.JidoBacked
  alias AllbertAssist.Objectives.Commands

  @frame_objective "allbert.objectives.engine.frame_objective"

  use JidoBacked,
    name: "allbert_objectives_engine",
    description: "Coordinates objective lifecycle stages.",
    schema: [...],
    signal_routes: [
      {@frame_objective, Commands.FrameObjective}
      # ...one route per private engine command
    ]

  def frame_objective(server \\ __MODULE__, params) do
    JidoBacked.dispatch(server, @frame_objective, params,
      source: "/allbert/objectives/engine",
      timeout: :infinity
    )
  end
end
```

Notes:

- Engine never calls `Actions.Runner.run/3` for its own private commands.
  Private commands are not registered capability actions (v0.23 boundary,
  ADR 0007 amendment).
- Engine DOES call `Actions.Runner.run/3` for the underlying step's
  target action (e.g., `RunAnalysis` for `kind: :action` steps). The
  runner handles Security Central and confirmation creation as in v0.07
  — the engine does not explicitly call `Confirmations.create/2`; when
  the target action returns `:needs_confirmation`, the runner creates
  the record with the `objective_id`/`step_id` threaded through context.
- Directives are honored: stage commands return `Jido.Agent.Directive`
  values such as `Directive.schedule/2` for delayed self-signals. The
  `Jido.AgentServer` directive queue drives the loop without explicit
  recursion.

### Engine command error policy

Hybrid: command modules return explicit `{:error, reason}` tuples for expected
domain failures and let unexpected raises bubble to the supervisor.

- On `{:error, reason}`: engine transitions the affected step to
  `:failed` with bounded reason; transitions objective to `:blocked`
  with reason `:step_failed`; emits `allbert.objective.step.failed` +
  `allbert.objective.blocked`; records `objective_events` row; **does
  not crash other active objectives** in the engine's
  `active_objectives` map.
- On uncaught raise (Ecto exception, KeyError, etc.): JidoBacked
  supervisor restarts Engine.Agent; `rebuild_state/1` rehydrates from
  SQLite. Affected objective's last successful state replays. Other
  active objectives lose transient in-memory state but reload from
  SQLite.
- `handle_command_error/3` is the single private helper that does the
  catch-side bookkeeping. Tested with synthetic `{:error, :db_busy}`
  and `{:error, %Ecto.Changeset{}}` shapes.

In-memory state is bounded by `objectives.max_active_per_user` (reserved;
default 16). Engine state is a cache; SQLite is authoritative. Step rows
persist on **every status transition** (see Schema below) so a crash mid-
stage replays from the last persisted state.

**Naming convention** (renamed in v0.24 third validation pass per the
v0.23 substrate boundary):

- Private engine command modules live under
  `AllbertAssist.Objectives.Commands.*` (files:
  `apps/allbert_assist/lib/allbert_assist/objectives/commands/<name>.ex`).
  These use `Jido.Action` for schema validation and are dispatched only
  by `Objectives.Engine.Agent`. They are **not** registered in
  `AllbertAssist.Actions.Registry`, **not** intent candidates, and
  **not** operator-callable. The v0.23 boundary rule (ADR 0007
  amendment) applies: private Jido command modules are not Allbert
  capability actions.
- Registered operator-facing capability actions live under
  `AllbertAssist.Actions.Objectives.*` (files:
  `apps/allbert_assist/lib/allbert_assist/actions/objectives/<name>.ex`).
  These are registered in `Actions.Registry`, selectable by intent,
  callable through `Actions.Runner.run/3`, and gated by Security
  Central.

The two namespaces are intentionally distinct so a quick visual scan
distinguishes "engine internal" from "operator-facing capability".
`CancelObjective` and `ContinueObjective` appear in both — the
registered action accepts the operator call, runs Security Central,
and then dispatches into the private engine command through the
Engine.Agent public helper (`cancel_objective/2` or `continue_objective/2`),
which uses `JidoBacked.dispatch/4`. They are separate modules.

### `AllbertAssist.Objectives.Proposer` (deterministic proposer dispatcher)

File: `apps/allbert_assist/lib/allbert_assist/objectives/proposer.ex`

```elixir
defmodule AllbertAssist.Objectives.Proposer do
  @moduledoc """
  Dispatch deterministic step proposals to per-app modules.
  Each app registers a module exposing `propose/2`.
  In v0.24, only `StockSage.Proposer` is registered.

  Supports hybrid propose-execute-reflect cycles via the {:more, hint}
  return shape: a proposer may stream steps batch-by-batch, signaling
  whether more are pending. The engine re-invokes propose after each
  Stage 7 observation, passing the prior hint back, until the proposer
  returns `:done` or `{:no_steps, _}`.
  """

  @type hint :: {app_id :: atom(), state :: map()}
  @type more_or_done :: {:more, hint()} | :done

  @spec propose(intent_decision :: map(), context :: map()) ::
          {:ok, [step_attrs :: map()], more_or_done()}
          | {:no_steps, reason :: atom()}
  def propose(intent_decision, context), do: # ...

  @spec register_app_proposer(app_id :: atom(), proposer_module :: module()) :: :ok
  def register_app_proposer(app_id, proposer_module), do: # ...
end
```

**Hybrid contract**: each call returns one of:

- `{:ok, [step_attrs, ...], :done}` — final batch; no more steps
  pending. Engine proceeds through Stages 5–7 for each step, then
  evaluates acceptance.
- `{:ok, [step_attrs, ...], {:more, hint}}` — first/intermediate
  batch; engine should call `propose/2` again after Stage 7 with
  `Keyword.put(context, :proposer_hint, hint)`. Engine persists the
  hint in durable SQLite (`objectives.proposer_hint`) and caches the
  decoded value in `Objectives.Engine.Agent` state under
  `proposer_hints[objective_id]` so a crash + rehydrate replays
  correctly.
- `{:no_steps, reason}` — no steps available. Engine records the
  reason in the framing event and does NOT create the objective (if
  this is the framing call) or transitions to `:blocked` with the
  reason as a `:no_more_steps` impasse (if mid-objective).

The hint is a **tagged tuple keyed by app_id** so the engine can
pattern-match without inspecting the inner state:

```elixir
# StockSage's hint shape:
{:stocksage, %{
  step_index: 1,                # next step's index (0-based)
  completed_steps: ["stp_aapl"], # ids of completed children
  total_planned: 2,
  remaining_tickers: ["MSFT"]
}}
```

The inner map shape is per-app and opaque to the engine, but it must be
JSON-encodable, redacted, and bounded. Engine stores the durable hint as
inspectable JSON in `objectives.proposer_hint`:

```json
{"app_id":"stocksage","state":{"step_index":1,"remaining_tickers":["MSFT"]}}
```

On rehydrate, Engine decodes the JSON, normalizes `app_id` through
`AllbertAssist.App.Registry.normalize_app_id/1`, and repopulates the
in-memory `proposer_hints` cache.

```elixir
defmodule StockSage.Proposer do
  @moduledoc """
  StockSage's deterministic step proposer. v0.24 handles single-ticker
  ("analyze AAPL") and two-ticker compare ("analyze AAPL and compare to
  MSFT") prompts via the hybrid contract.
  """

  @behaviour AllbertAssist.Objectives.ProposerBehaviour
  # (the behaviour is defined alongside Proposer dispatcher; callbacks:
  #  propose/2 with the same signature)

  @spec propose(intent_decision :: map(), context :: map()) ::
          {:ok, [map()], :done | {:more, term()}}
          | {:no_steps, atom()}
  def propose(intent_decision, context) do
    case parse_tickers(intent_decision.text) do
      [single] ->
        # Single-ticker: one step, done in one batch
        {:ok, [run_analysis_step(single, context)], :done}

      [first, second] ->
        case Keyword.get(context, :proposer_hint) do
          nil ->
            # First call: propose step 1, signal more pending
            {:ok, [run_analysis_step(first, context)],
             {:more, {:stocksage, %{step_index: 1, completed_steps: [],
                                    total_planned: 2,
                                    remaining_tickers: [second]}}}}

          {:stocksage, %{step_index: 1, remaining_tickers: [next | _]} = hint} ->
            # Second call after Stage 7: propose step 2 with parent_step_id
            parent_step_id = List.last(hint.completed_steps)
            {:ok, [run_analysis_step(next, context, parent_step_id: parent_step_id)],
             :done}
        end

      [] ->
        {:no_steps, :no_tickers_recognized}

      _too_many ->
        {:no_steps, :too_many_tickers_in_one_prompt}
    end
  end

  defp parse_tickers(_text), do: # deterministic ticker extraction
  defp run_analysis_step(ticker, ctx, opts \\ []), do: # action step attrs map
end
```

```elixir
defmodule AllbertAssist.Objectives.Evaluator do
  @moduledoc """

  map. Returns :met | :not_met | :needs_more_steps.

  :needs_more_steps, the engine treats the cap as winning (impasse);
  the would-have-continued verdict is recorded in the impasse event
  payload for operator diagnostics (per Guiding Rule 3).
  """

  @type verdict :: :met | :not_met | :needs_more_steps

  @spec evaluate(objective :: map(), steps :: [map()]) :: verdict()
  def evaluate(objective, steps), do: # ...
end
```

**Acceptance criteria structured shape** — `objectives.acceptance_criteria`
is a JSON-encoded TEXT column. The deserialized map shape:

```elixir
%{
  # Required: at least this many steps must reach :completed status
  "min_completed_steps" => 1,

  # Required: each clause is a "must hold" condition. All clauses must
  # be satisfied for :met. Clauses use simple typed matchers; no DSL.
  "required" => [
    %{
      "kind" => "step_completed_with_action",
      "action" => "StockSage.Actions.RunAnalysis",
      "params_match" => %{"ticker" => "AAPL"},
      "min_count" => 1
    },
    %{
      "kind" => "step_completed_with_action",
      "action" => "StockSage.Actions.RunAnalysis",
      "params_match" => %{"ticker" => "MSFT"},
      "min_count" => 1
    }
  ],

  # Optional: clauses that signal "in progress, expect more steps"
  # rather than "failed". Used to distinguish :not_met from
  # :needs_more_steps. Engine checks `pending_clauses` after the
  # `required` set partially matches.
  "needs_more_when" => [
    %{"kind" => "completed_step_count_below", "value" => 2}
  ],

  # Optional human-readable summary rendered in trace + LiveView.
  "summary" => "Two completed RunAnalysis steps for AAPL and MSFT."
}
```

**Clause kinds shipped in v0.24** (extensible; reserved names below):

| Kind | Semantics |
|---|---|
| `step_completed_with_action` | A `:completed` step exists whose `candidate_action` matches the string and whose `action_params` map super-set-matches `params_match`. Cardinality via `min_count` (default 1). |
| `completed_step_count_below` | Used in `needs_more_when` only. True if fewer than `value` steps are `:completed`. |
| `observation_contains` | A `:completed` step's `observation_summary` contains the literal `substring` (bounded plain text; not regex). Used for the StockSage two-ticker "comparison summary" check. |

Reserved clause kinds (named, not implemented in v0.24): `step_failed`,
`step_observation_matches_regex`, `total_duration_under_ms`,
`acceptance_via_llm` (reserved per ADR 0021 Section 3 "Planner /
evaluator"; LLM-assisted evaluation is not v0.24 scope).

**Evaluator algorithm** (deterministic):

1. Count `:completed` steps. If `min_completed_steps` not met → return
   `:not_met` (or `:needs_more_steps` if any `needs_more_when` clause
   holds).
2. Evaluate each `required` clause in order. If all hold → return
   `:met`.
3. If at least one `required` clause is unmet, check `needs_more_when`
   clauses. If any holds → return `:needs_more_steps`. Otherwise →
   `:not_met`.
4. Engine maps `:not_met` to objective `:blocked` (no more steps
   anticipated). Maps `:needs_more_steps` to "try Stage 4 again with
   the persisted proposer hint."

Test fixtures live under
`apps/allbert_assist/test/fixtures/v0.24/acceptance_criteria/` with
JSON exemplars for the two v0.24 acceptance flows (single-step
RunAnalysis; two-step AAPL + MSFT compare).

## Engine Stage Pipeline

Seven stages drive every objective lifecycle. Each stage is one private
Jido.Action command on `Objectives.Engine.Agent` routed through JidoBacked
signal dispatch; see "Engine command dispatch" in Module Shapes. The flow doc
(`docs/plans/v0.24-request-flow.md`) shows the per-stage diagrams; this
section documents the substrate-level loop semantics, persistence
cadence, and the hybrid proposer cycle.

| # | Stage | Command | Persistence on success | Signals emitted |
|---|---|---|---|---|
| 1 | Receive | `Receive` | none | `allbert.input.received` (existing) |
| 2 | Interpret intent | `InterpretIntent` (passes through to `Intent.Engine.collect_candidates/2`) | none | `allbert.runtime.turn.started` (new canonical alias) |
| 3 | Frame/resume objective | `FrameObjective` | INSERT `objectives` row (new) OR no-op (resume) | `allbert.objective.created` OR `.updated` |
| 4 | Propose and evaluate | `ProposeSteps` → `EvaluateSteps` | INSERT `objective_steps` row per proposed step (status `:proposed`); UPDATE status → `:selected` after evaluator | `.step.proposed`, `.step.selected` |
| 5 | Authorize | `AuthorizeStep` | UPDATE step status (`:selected` → `:blocked` if `:needs_confirmation`, else stay) | `.blocked` if confirmation needed; otherwise none |
| 6 | Execute | `ExecuteStep` | UPDATE step status (`:running` → `:completed`/`:failed`); result_summary populated | `.step.completed` OR `.step.failed` |
| 7 | Observe and advance | `ObserveStep` → `AdvanceObjective` | UPDATE objective `last_observation_summary`, `current_step_id`, `loop_count`; INSERT `objective_events` observation row | `.observed`; then `.completed` / `.blocked` / `.impasse` per evaluator verdict |

### Persistence cadence — every status transition

**Step rows** are UPDATEd on EVERY status transition. Happy path for a
single-step objective writes the same row id four times:

1. INSERT at Stage 4 with status `:proposed`
2. UPDATE at Stage 4 (evaluator approves) → status `:selected`
3. UPDATE at Stage 5 (confirmation needed) → status `:blocked`,
   OR at Stage 6 (execution starts) → status `:running`
4. UPDATE at Stage 6 (execution completes) → status `:completed` /
   `:failed`

Engine never holds step state in memory as authoritative; the SQLite
row is the source of truth. Crash mid-stage replays from the last
persisted state on rehydration.

**Objective rows** are UPDATEd on each Stage 7 observation
(`last_observation_summary`, `loop_count`, `current_step_id`) and on
each lifecycle transition (Stage 3 frame, Stage 7 acceptance).

**`objective_events` rows** are INSERTed (never UPDATEd) and
correspond 1:1 with the signal emissions in the table above.

### Stage 4 hybrid proposer loop

Stage 4 runs the **propose-evaluate-execute-observe** cycle until the
registered per-app proposer signals `:done`. Pseudocode:

```
loop:
  case Proposer.propose(intent_decision, ctx ++ proposer_hint) do
    {:ok, [step_attrs, ...], :done} ->
      persist_steps(step_attrs_list)
      foreach step: Stage 5 → Stage 6 → Stage 7
      acceptance_verdict = Evaluator.evaluate(objective, all_steps)
      handle_verdict(acceptance_verdict)
      break loop

    {:ok, [step_attrs, ...], {:more, hint}} ->
      persist_steps(step_attrs_list)
      store_hint(objective_id, hint)        # UPDATE objectives.proposer_hint + cache in Engine.Agent state
      foreach step: Stage 5 → Stage 6 → Stage 7
      acceptance_verdict = Evaluator.evaluate(objective, all_steps)
      case acceptance_verdict do
        :met -> handle_verdict(:met); break loop
        :needs_more_steps -> continue loop (re-propose with hint)
        :not_met -> handle_verdict(:not_met); break loop
      end

    {:no_steps, reason} ->
      record_impasse(reason); transition_blocked; break loop
  end
```

### Per-turn caps and impasse precedence

Two caps bound the loop per Guiding Rule 3:

- `objectives.max_steps_per_turn` (default 3) — total steps proposed
  in a single user turn. Tracked in turn-local context, not objective
  state. Exceeding this triggers
  `allbert.objective.impasse` with `cap_hit: :max_steps_per_turn`.
- `objectives.max_loop_count` (default 5) — total advance cycles per
  objective lifetime. Tracked in `objectives.loop_count` column.
  Incremented on each Stage 7 observation (per completed step). When
  `loop_count >= max_loop_count` AND the next evaluator verdict would
  be `:needs_more_steps`, the cap wins: engine records
  `would_have_continued_verdict: :needs_more_steps` in the impasse
  event payload and transitions to `:blocked`.

Cap precedence: **`max_loop_count` always wins over the evaluator's
`:needs_more_steps`**. The diagnostic verdict is preserved in the
impasse event so operators can see what the engine wanted to do.

### Error handling reference

Per "Engine command error policy" in Module Shapes:

- `{:error, _}` returns from any stage command → caught by
  `handle_command_error/3` → step (if any) transitions to `:failed`,
  objective transitions to `:blocked` reason `:step_failed`, events +
  signals emitted, other active objectives keep running.
- Unexpected raises → JidoBacked supervisor restarts Engine.Agent;
  eager rehydration replays from SQLite.

### Resume entry points

Three ways the engine re-enters an objective:

1. **Confirmation approval** (most common) — `approve_confirmation`
   action runs; engine's `ContinueObjective` command picks up at
   Stage 5 (re-authorize selected step) with the approved
   confirmation in context.
2. **Operator `mix allbert.objectives continue`** — registered
   `ContinueObjective` action calls Engine. Engine evaluates whether
   anything has changed:
   - If a confirmation linked to the objective is now resolved →
     advance via Stage 5 → 6 → 7.
   - If loop cap was the blocker AND it was raised via Settings →
     advance via Stage 4 (re-propose with stored hint).
   - Otherwise → return advisory `{:ok, status: :still_blocked,
     reason: "..."}` with no `loop_count` increment, no impasse
     event, no state change.
3. **Scheduled job advance** — a v0.13 scheduled job carries
   `objective_id`; when it runs, the job target's action result feeds
   Stage 6 → 7 of the linked objective.

`:abandoned` objectives are **terminal**: `continue` returns
`{:error, :objective_abandoned}`. Operators wanting to retry must
create a new objective (e.g., by re-asking).

## `:objective_write` Permission Class

New permission class added to Security Central:

- **Class name:** `:objective_write`
- **Default policy:** `:allow` for the objective's owning `user_id`.
- **Safety floor:** `:allow` (this is not a high-risk class; the work
  that an objective performs goes through other action-level
  permission classes — `:stocksage_analyze`, `:memory_write`, etc.).
- **Settings Central key:** `permissions.objective_write` (writable).
- **Risk tier:** `:low` (operator can change policy without unlocking
  a higher tier).
- **Confirmation default:** `:allow` (no confirmation prompt for cancel/
  continue).

The class exists for symmetry with other write classes
(`:memory_write`, `:stocksage_write`, `:settings_write`) and for
future ability to scope objective creation/cancellation when hosted
multi-user lands (v0.31+). ADR 0021 is amended in M6 to explicitly
enumerate this class.

## CLI

New under `mix allbert.objectives`:

```sh
# List objectives for the operator
mix allbert.objectives list [--user USER] [--status open|running|blocked|completed|cancelled|failed|abandoned] [--active-app APP] [--limit N]

# Show one objective by id
mix allbert.objectives show <id> [--user USER]

# Cancel an objective (--reason REQUIRED)
mix allbert.objectives cancel <id> --reason "..." [--user USER]

# Continue a blocked objective (advisory if nothing changed)
mix allbert.objectives continue <id> [--user USER]
```

CLI exit codes:

| Condition | Exit code |
|---|---|
| Success (incl. advisory `:still_blocked` from continue with no change) | `0` |
| Usage error (missing required arg, unknown subcommand, missing --reason on cancel) | `64` |
| Not found (cross-user or missing id) | `65` |
| `--user`/`--operator` mismatch | `66` |
| Action failure (Security Central denial, terminal objective like `:abandoned`, etc.) | `1` |

`continue` advisory contract:

- If the objective advances (confirmation now resolved, loop cap was
  raised, etc.), CLI prints the new status + brief diff and exits 0.
- If nothing has changed since the objective hit `:blocked`, the
  underlying `ContinueObjective` action returns
  `{:ok, %{status: :still_blocked, reason: "..."}}`. CLI prints
  `Objective <id> is still blocked: <reason>`. Exits 0. No
  `loop_count` increment. No impasse event. No state change. The
  operator can re-issue without consequence.
- If the objective is `:abandoned`, `ContinueObjective` returns
  `{:error, :objective_abandoned}`. CLI prints
  `Objective <id> is abandoned; create a new objective to retry.`
  Exits 1.

All commands call registered actions through `Actions.Runner.run/3`.
None reads SQLite directly. Output is redacted, bounded, and stable
for tests.

CLI module: `apps/allbert_assist/lib/mix/tasks/allbert.objectives.ex`
following the v0.22 `mix.tasks.stocksage.analyze.ex` style (one
module, multi-subcommand dispatch via `dispatch/1`).

### Redaction posture

Every new signal payload passes through
`AllbertAssist.Security.Redactor.redact/1` before emission. The
`sensitive_key_fragments` allowlist from v0.22 third-validation
(includes `raw_bridge`, `raw_final`, `raw_response`, `authorization`,
`bearer`) covers the new signal payloads without amendment. The four
free-text fields (`title`, `objective`, `acceptance_criteria`,
`observation_summary`) are bounded by character count, not by
key-name filtering, so M1 tests assert that no value derived from raw
action output exceeds the bound or leaks into the wrong signal.

### Trace section additions

Trace markdown gains per-section inline rendering. **Placement
decision (v0.24 fourth pass):** the `## Objective` block is rendered
inline as a `### Objective` subsection inside each top-level trace
section that touched the objective during the current turn. This
matches the spec originally drafted for `## Jido Debug` in v0.23
(which v0.23 walked back to single-tail for substrate diagnostics);
v0.24 commits to per-section inline for objective context because
operators reading the trace top-down see "what this section did to
the objective" immediately rather than scanning to a tail block.

Sections that may contain an inline `### Objective` subsection:

- `## Intent Candidates` — when an `:objective` candidate appears or
  the active objective influenced ranking.
- `## Selected Skill` (or `## Selected Action`) — when the selected
  action is the active objective's current step.
- `## Confirmation` — when the confirmation carries
  `objective_id`/`step_id`. Existing v0.07 section also gains
  `Objective: <id>` and `Step: <id>` lines for backward compatibility.
- `## Action Result` — when the action ran as an objective step.
- `## Memory Review` — when a reflection step proposed memory
  candidates.

Inline subsection contents (bounded):

- Objective id (full UUID)
- Title (bounded 60 chars)
- Status pill (`:open` / `:running` / `:blocked` / etc.)
- Current step id (full UUID) + kind
- Stage that touched this section
- Loop count

The dedicated `## Objective Steps` top-level section is also emitted
per turn (capped at 5 steps in operator detail; capped at 20 in debug
detail). This gives operators a single per-turn step-by-step view
alongside the per-section inline context.

When `objectives.trace_detail = :debug`, each per-section inline
subsection also includes the stage's `before_*` / `after_*` lifecycle
events for that section. At `:operator` (default), only status +
identifying fields render.

**JidoBacked substrate debug-trace** (`allbert.jido.debug_trace = true`)
continues to render at the trace tail as `## Jido Debug` per the v0.23
convention. Objective-specific trace details live inside the inline
`### Objective` subsections and the top-level `## Objective Steps`
section. The two channels do not overlap:

- `### Objective` (inline) = operator-facing per-section objective
  context. Always rendered when an objective touched the section.
- `## Jido Debug` (tail, gated) = substrate-internal Engine.Agent
  diagnostics (active objective count, last rebuild time, current
  stage map). Only when the operator opts in via the v0.23 setting.

Implementation note: the trace renderer needs a per-section
"touched-objective" affordance. Suggested mechanism — each trace
section emitter writes an optional `objective_context: %{id, title,
status, ...}` map alongside its existing fields; the renderer's
section template includes an `### Objective` subsection when the map
is present. M3 ships this affordance; the renderer is purely
additive (no v0.22 trace fields removed or renamed).

### Two acceptance flows for v0.24

**Single-step objective:**

```
User: "analyze AAPL" (with active_app: :stocksage)
  → Intent.Engine ranks RunAnalysis candidate
  → Engine frames objective: title "Analyze AAPL"
  → StockSage.Proposer.propose/2 returns one step
  → Engine authorizes; Security Central returns :needs_confirmation
  → Confirmation created with objective_id + step_id
  → Operator approves
  → Step transitions selected → running → completed
  → Engine evaluates acceptance: one completed analysis for AAPL on today's date → :met
  → Objective transitions to :completed
  → allbert.runtime.turn.completed + allbert.objective.completed emit with shared trace_id
```

**Two-step "analyze and compare" objective:**

```
User: "analyze AAPL and compare to MSFT"
  → Engine frames objective: title "Analyze AAPL and compare to MSFT"
  → StockSage.Proposer.propose/2 returns two steps; step 2.parent_step_id = step 1.id
  → Step 1 follows single-step flow → completed
  → Acceptance returns :needs_more_steps (one of two analyses present)
  → Engine increments loop_count, advances to step 2
  → Step 2 follows the same flow → completed
  → Acceptance returns :met
  → Objective transitions to :completed
```

Both flows are exercised in M5 acceptance smokes.

## Intent Engine Integration

ADR 0019 is amended at M2 to register `:objective` as a candidate kind.
`AllbertAssist.Intent.Engine` gains a new arity:

```elixir
@spec collect_candidates(decision_input :: map()) :: [Candidate.t()]
@spec collect_candidates(decision_input :: map(), opts :: keyword()) :: [Candidate.t()]
def collect_candidates(decision_input), do: collect_candidates(decision_input, [])
def collect_candidates(decision_input, opts) do
  objective = Keyword.get(opts, :objective)
  # ... existing candidates ...
  candidates ++ objective_candidates(objective, decision_input)
end
```

Behavior:

- When the operator has open objectives matching the current turn's
  topic (deterministic keyword + recency match), the engine returns an
  objective-resume candidate.
- The candidate's `source` is `:objective`; `score` is 0.2–0.5 based
  on recency and topic match; `reason` is the objective title.
- ADR 0019 invariants hold: objective candidates do not grant
  permissions; do not authorize actions; do not skip Security
  Central.

`Candidate.bound/2` adds `objective: 5` to `@default_kind_limits`.

Memory candidates (v0.21) and objective candidates can coexist in
the same ranking pass.

The new arity is backward-compatible: existing callers that use
`collect_candidates/1` continue to work; the engine sniffs `:objective`
from the new arg only.

### `AllbertAssistWeb.SignalBridge` (new GenServer)

File: `apps/allbert_assist_web/lib/allbert_assist_web/signal_bridge.ex`

Subscribes to the named `Jido.Signal.Bus` (`AllbertAssist.SignalBus`) on boot.
For each `allbert.objective.**` event received, broadcasts to a per-user Phoenix.PubSub
topic so AgentLive + ObjectiveLive can subscribe per session without
touching core Allbert modules.

```elixir
defmodule AllbertAssistWeb.SignalBridge do
  use GenServer

  @objective_topic_prefix "objectives:"

  @spec start_link(keyword()) :: GenServer.on_start()
  def start_link(opts), do: GenServer.start_link(__MODULE__, opts, name: __MODULE__)

  @impl true
  def init(_opts) do
    {:ok, _subscription_id} =
      Jido.Signal.Bus.subscribe(AllbertAssist.SignalBus, "allbert.objective.**")

    {:ok, %{}}
  end

  @impl true
  def handle_info({:signal, %Jido.Signal{data: data} = signal}, state)
      when is_map(data) do
    # Jido.Signal carries its CloudEvents-style payload on the `:data`
    # field (NOT `:payload`); `%Jido.Signal{}` does not implement Access,
    # so `get_in/2` on the struct does not work. Destructure or use
    # `signal.data` directly.
    case data do
      %{user_id: user_id} when is_binary(user_id) ->
        Phoenix.PubSub.broadcast(
          AllbertAssistWeb.PubSub,
          @objective_topic_prefix <> user_id,
          {:objective_event, signal}
        )

      %{"user_id" => user_id} when is_binary(user_id) ->
        # Same path for string-keyed payloads (e.g., deserialized from
        # the SignalBus event store).
        Phoenix.PubSub.broadcast(
          AllbertAssistWeb.PubSub,
          @objective_topic_prefix <> user_id,
          {:objective_event, signal}
        )

      _ ->
        # Objective signals without `user_id` (shouldn't happen per the
        # plan's "Signals And Trace Metadata" payload contract) are
        # dropped silently with a bounded warn.
        :ok
    end

    {:noreply, state}
  end

  def handle_info({:signal, _other}, state), do: {:noreply, state}

  @spec topic_for(user_id :: String.t()) :: String.t()
  def topic_for(user_id), do: @objective_topic_prefix <> user_id
end
```

Wired into `AllbertAssistWeb.Application` children alongside
`AllbertAssistWeb.Endpoint`. Web-side only — no core Allbert module
needs to know about Phoenix.PubSub. If the web app is not running
(headless CLI deployment), Engine signals fire but the bridge is
absent; this is by design.

Coverage requirements:

- Cross-user isolation strict: cross-user `show`/`cancel`/`continue`
  return `:not_found`.
- Loop caps enforced; would-have-continued verdict recorded in
  impasse event payload.
- Cancellation is cooperative: in-flight action completes; no new
  steps; cancel-then-approve preserves audit trail.

  loop_count increment; idempotent.

- Step row persistence cadence: every status transition writes the
  row.

  no secrets.
- Signal correlation: shared `trace_id` between turn.completed and

  topics; ignores non-matching signals; absence does not break
  LiveView (poll fallback).
- Confirmation rendering shows snapshot title + stale-warning when
  live objective status diverges, across CLI / Telegram / email /
  LiveView consistently.
- Hybrid proposer contract: `{:more, hint}` returns persist the

  rehydrate preserves hint.
- Acceptance evaluator: structured-map clauses evaluated

## Non-Goals

- No advisory provider behaviour (reserved in ADR 0021).
- No world-model abstraction (reserved).
- No capability inventory aggregation as a module (reserved).
- No route, acquisition option, or planner beyond the deterministic
  per-app proposers.
- No diffusion proposer, market allocator, probabilistic inference
  provider.
- No parallel step execution (`max_parallel_steps` reserved at 1).
- No mid-action interruption (cooperative cancellation only).
- No LLM-assisted acceptance evaluator (deterministic only).
- No automatic memory promotion from objective observations; the
  v0.21 review surface remains the only path.
- No native StockSage trading agents (v0.25 — but v0.24 ships the
  minimal `:delegate_agent` step kind contract they will consume).
- No workspace shell upgrade (v0.26).
- No canvas integration (v0.30).
- No public plugin hook contribution API.
- No automatic capability acquisition (install plugin, write code,
  request credential) from an objective step.
- No settings-driven proposer rules (proposers are hardcoded Elixir
  per app).
