# Allbert v0.09 Intent Engine Enrichment

## Purpose

v0.09 should turn the earlier routing heuristics into a real intent engine.

The origin note calls out intent understanding as the first point in the agent
loop: Allbert should understand what the user wants, choose skills and actions,
know when to ask for confirmation, and decide when work belongs in the
foreground, background, memory, or a future channel. v0.01 deliberately kept
this deterministic and small. v0.03 adds standard skill discovery and
activation. v0.04 binds trusted built-in skills to Jido actions.

v0.09 is the place to enrich intent without making execution more dangerous.
The output should be a structured decision that the runtime, permission gate,
traces, CLI, LiveView, future jobs, and future channels can all inspect.

## Why This Comes After Confirmation, Jobs, Channels, And Memory

Confirmation, jobs, channels, and memory review all generate concrete decision
pressure. They reveal what users actually ask for, which choices need
explanation, which settings matter, where confirmations interrupt the flow, and
which channel-specific context changes interpretation.

So v0.09 should introduce the shared intent decision contract after the first
skills, settings, confirmations, jobs, channels, and memory review surfaces
exist. This keeps the enriched engine grounded in real runtime behavior instead
of overfitting a classifier before the product has enough surfaces.

## Current Baseline

- `AllbertAssist.Agents.IntentAgent` currently selects routes with
  deterministic predicates.
- v0.03 is expected to provide skill discovery, trust, activation, and
  activation traces.
- v0.04 is expected to provide action-backed built-in skills, capability
  contracts, and an action lifecycle runner.
- v0.05 through v0.08 are expected to provide settings-backed confirmation,
  jobs, channels, and memory review signals that can inform richer intent
  evaluation.
- Permission decisions remain centralized in
  `AllbertAssist.Security.PermissionGate`.

## Non-Goals

- No new shell execution.
- No new external network execution.
- No autonomous skill creation.
- No unsupervised model-selected side effects.
- No embeddings or vector retrieval.
- No channel-specific intent forks.

## Intent Decision Contract

Introduce a structured intent decision, tentatively
`AllbertAssist.Intent.Decision`, with fields equivalent to:

- `:intent` such as `:direct_answer`, `:activate_skill`, `:run_skill_action`,
  `:append_memory`, `:read_memory`, `:needs_confirmation`, `:schedule_job`,
  `:unsupported`, or `:deny`
- `:confidence`
- `:reason`
- `:selected_skill`
- `:candidate_skills`
- `:selected_action`
- `:candidate_actions`
- `:permission`
- `:confirmation`
- `:channel`
- `:foreground_or_background`
- `:diagnostics`
- `:trace_metadata`

The decision object should be plain data. It should not execute anything. The
runtime still executes through registered actions, the action lifecycle runner,
and the permission gate.

## Engine Shape

Use a layered engine:

1. Normalize the request and channel metadata.
2. Run deterministic safety and explicit-command classifiers first.
3. Match explicit skill references and obvious skill/action triggers.
4. Rank candidate skills/actions using the registry and capability contracts.
5. Optionally ask a model for structured classification when deterministic
   confidence is low.
6. Validate the model's output against known skills, actions, permissions, and
   allowed statuses.
7. Return one decision plus alternatives and diagnostics.

The model may help classify intent, but it must not invent skill names, action
names, permission classes, or execution modes.

## Implementation Order

### Milestone 1: Decision Struct And Trace Shape

Status: planned.

Implementation tasks:

- Add an intent decision struct or plain map schema.
- Add trace fields for intent, confidence, selected/candidate skill, selected
  action, permission, confirmation, and decision reason.
- Keep current routing behavior while emitting the new decision shape.

Acceptance criteria:

- Existing v0.08 behaviors still work.
- Every turn has inspectable intent metadata when tracing is enabled.

### Milestone 2: Deterministic Classifier Extraction

Status: planned.

Implementation tasks:

- Move existing route predicates out of `IntentAgent` into an intent engine
  module.
- Preserve tested behavior for direct answers, memory append/read, skill
  list/read/activation, command planning, and external-network requests.
- Add confidence and candidate alternatives.

Acceptance criteria:

- Existing intent tests pass against the new engine.
- The decision explains why a route was selected.

### Milestone 3: Registry-Aware Skill And Action Ranking

Status: planned.

Implementation tasks:

- Use skill name, description, kind, trust state, and capability contract data
  as routing inputs.
- Rank trusted skills before untrusted or disabled candidates.
- Keep untrusted skills out of executable decisions.
- Include non-selected candidates in diagnostics.

Acceptance criteria:

- Explicit skill requests activate the named skill.
- Capability prompts select the expected built-in skill/action.
- Untrusted skills can be diagnosed but not selected for execution.

### Milestone 4: Structured Model Assist

Status: planned.

Implementation tasks:

- Add an optional model-assisted classifier behind config.
- Require structured output that validates against known registry entries.
- Fall back to deterministic decisions when model output is invalid, low
  confidence, or unavailable.
- Keep tests deterministic by injecting classifier fixtures.

Acceptance criteria:

- Model assistance can improve ambiguous routing without making unregistered
  actions executable.
- Invalid model output is recorded as a diagnostic and ignored.

### Milestone 5: Intent Evals

Status: planned.

Implementation tasks:

- Add eval fixtures for:
  - should activate skill
  - should not activate skill
  - should run safe action-backed skill
  - should require confirmation
  - should deny or stay inert
  - should route to future job candidate without scheduling
- Track false positives and false negatives.

Acceptance criteria:

- Intent behavior can be regression-tested against the skills, confirmation,
  jobs, channels, and memory review behavior that already exists.

## Definition Of Done

v0.09 is done when:

- Runtime turns pass through a structured intent decision.
- Existing v0.08 behavior is preserved.
- Skill/action selection is registry-aware and permission-aware.
- Intent traces explain selected and rejected candidates.
- Optional model-assisted classification is validated, bounded, and testable.
- Confirmation, jobs, and channels have one shared intent contract to consume.
