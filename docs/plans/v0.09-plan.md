# Allbert v0.09 Skill Script Runner

## Status

Implementation-ready after v0.08 release/tag on 2026-05-02.

v0.09 is the first release where trusted Agent Skill `scripts/` resources may
run. It is deliberately narrower than a scripting engine, package installer, or
container sandbox release.

## Purpose

v0.09 lets trusted Agent Skills run selected bundled scripts, but only through
Allbert's action, confirmation, sandbox, Security Central, and trace
boundaries.

v0.03 inventories `scripts/` as inert resources. v0.06 validates trusted skill
contracts against registered actions. v0.07 adds durable confirmations. v0.08
adds Level 1 local policy sandboxing. v0.09 connects those pieces with one
stricter rule: a script is executable only when it is part of the selected
skill's parsed resource inventory and a registered action chooses to run it.

## Architecture Readiness

v0.09 is implementation-ready when this planning pass lands. The implementer
should not reopen the core product decisions about arbitrary scripts, imported
skills, module loading, package installs, external network calls, or container
isolation.

The work is Jido-centric and consistent with `AGENTS.md` and `DEVELOPMENT.md`:

- runtime-facing script execution enters through a registered Jido action
- channels, CLI tasks, LiveViews, and future adapters stay thin
- selected skill metadata travels as context, not executable authority
- Security Central owns `:skill_script_execute` policy and risk
- confirmations pause execution and re-check policy before target resumption
- v0.08 runner primitives provide Level 1 host-process controls, not OS
  isolation
- traces and audits record skill, script, digest, confirmation, and result
  metadata

## Boundary Contract

Add `run_skill_script` as the only script execution entrypoint.

The action accepts a structured request equivalent to:

- `skill_name`
- `script_path`, relative to the selected skill root and matching an
  inventoried `:script` resource
- `args`, explicit argv only
- `cwd`, optional and constrained by `execution.local.allowed_roots`; when
  omitted, use a per-run temporary cwd under Allbert Home
- `timeout_ms` and `max_output_bytes`, capped by v0.08 execution limits
- `env`, filtered through the execution env allowlist plus non-secret
  Allbert-provided script metadata
- `source_text`, optional original operator prompt

Execution is allowed only when all of these are true:

- `execution.skill_scripts.enabled` is true
- `permissions.skill_script_execute` is configured so the effective decision is
  at most `:needs_confirmation`
- the selected skill is trusted, enabled, discoverable, and selected in the
  current action context or explicitly named by an operator CLI
- the skill's validated capability contract allows `run_skill_script`
- `script_path` exactly matches a parsed resource with `kind: :script`
- the script resource digest matches the inventory before pending creation and
  again immediately before execution
- the script path is relative, contains no traversal, and has no hidden path
  segments
- the launch vector is explicit executable plus argv; shell strings, command
  chaining, redirection, command substitution, backgrounding, and PTY sessions
  are refused

v0.09 should support direct executable script resources first. Interpreter
profiles for non-executable scripts may be added through Settings Central, but
they must be explicit, confirmed, and audited. No default interpreter profile
may make all `.sh`, `.py`, `.js`, or similar files executable without operator
configuration.

## Implementation Decisions

These decisions are fixed for v0.09 implementation:

- `run_skill_script` is a separate registered action. It must not call
  `run_shell_command`, must not add arbitrary script paths to
  `execution.local.allowed_commands`, and must not use shell strings.
- The first supported launch mode is a direct executable resource. The resolved
  script file must be regular, inventoried, digest-matched, and executable by
  the host user. A script with a shebang and executable bit is valid. A
  non-executable script returns `:script_not_executable` unless an explicit
  interpreter profile is implemented in the same milestone.
- Interpreter profiles are optional in v0.09. If implemented, they must be
  Settings Central entries that name the interpreter executable and fixed argv
  prefix. File extension alone never selects an interpreter.
- The script resource path is authorized by skill trust plus inventory, not by
  `execution.local.allowed_roots`. The cwd and any path-like args remain
  constrained by `execution.local.allowed_roots`.
- Path-like args use the v0.08 path operand rules: `.`, `..`, `~`, absolute
  paths, and strings containing `/` are expanded relative to cwd and denied if
  outside allowed roots. Non-path args are treated as opaque argv strings.
- Default cwd is `<ALLBERT_HOME>/execution/skill-scripts/runs/<run-id>/cwd`.
  This internal runner cwd is created per run and does not require the operator
  to add Allbert Home to `execution.local.allowed_roots`. Any operator-supplied
  cwd must be in `execution.local.allowed_roots`.
- Env is deny-by-default. Requested env passes only keys in
  `execution.local.env_allowlist`. The runner may add non-secret Allbert
  metadata env keys: `ALLBERT_SKILL_NAME`, `ALLBERT_SKILL_SCRIPT_PATH`,
  `ALLBERT_SKILL_SCRIPT_SHA256`, and `ALLBERT_RUN_ID`.
- The runner may reuse v0.08 lower-level timeout, output-buffer, redaction, and
  audit helpers, but script validation and launch policy belong to v0.09
  modules behind `run_skill_script`.
- Result status vocabulary is `:needs_confirmation`, `:denied`, `:completed`,
  `:failed`, `:timed_out`, `:stale`, and `:digest_mismatch`. Durable
  confirmation status remains the existing confirmation store vocabulary; script
  result details live under `operator_resolution.target_result`.

## Confirmation Metadata

Pending and resolved script confirmations should include trace-safe metadata
with these keys or equivalent names:

- `target_action.name`: `run_skill_script`
- `target_permission`: `skill_script_execute`
- `target_execution_mode`: `skill_script_process`
- `selected_skill`: skill name, source scope, trust status, enabled status, and
  capability contract summary
- `params_summary`: script path, sha256, byte size, args summary, cwd summary,
  timeout, max output bytes, env key summary, sandbox level, and launch mode
- `resume_params_ref`: skill name, script path, expected sha256, args, cwd,
  timeout, max output bytes, requested env keys, and source text
- `operator_resolution.target_result`: status, exit status, timed-out flag,
  truncated flag, output bytes, redacted output preview, digest re-check
  status, runner backend, and failure reason

## Settings And Permissions

Add the minimum policy surface needed for v0.09:

- `permissions.skill_script_execute`: default `denied`, allowed values
  `allowed`, `needs_confirmation`, `denied`, safety floor
  `needs_confirmation`.
- `execution.skill_scripts.enabled`: default `false`.
- `execution.skill_scripts.require_confirmation`: default `true`; v0.09 must
  still enforce confirmation through the safety floor.
- `execution.skill_scripts.interpreter_profiles`: default empty map. Each
  profile names an executable, allowed extensions, args prefix, timeout/output
  caps, and command class.

Reuse v0.08 `execution.local.allowed_roots`, env allowlist, timeout caps, and
output caps for cwd and process limits. Skill roots are authorized through the
trusted skill registry and resource inventory, not by broadening local allowed
workspace roots.

## Milestones

### Milestone 1: ADR, Security, Settings, And Capability Metadata

Goal: add the policy vocabulary without running scripts.

Implementation tasks:

- Add ADR 0010 and link it from plan/onboarding docs.
- Add `:skill_script_execute` to Security Central, Settings Central, risk
  status, and permission surfaces.
- Add `execution.skill_scripts.*` settings validation.
- Register planned `run_skill_script` capability metadata as confirmation
  required and skill-backed only.
- Update docs so activation remains inert and v0.09 does not add package,
  network, or container powers.

Tests:

- permission policy defaults deny script execution and cap allowed to
  `:needs_confirmation`
- settings schema accepts valid script policy and rejects malformed profiles
- security status shows script execution as planned/disabled until enabled
- action registry exposes `run_skill_script` as skill-backed, confirmation
  required, and `skill_script_process` execution mode

Implementation status: complete in M1. The permission, safety floor, risk tier,
Settings Central keys, interpreter-profile validation, Security status boundary,
registered `run_skill_script` placeholder action, and capability metadata are
implemented without adding script execution.

### Milestone 2: Resource-Gated Script Spec

Goal: normalize script requests into an inert, auditable spec.

Implementation tasks:

- Add a pure script-spec resolver behind the action boundary.
- Resolve `skill_name` through the registry and require trusted/enabled status.
- Match `script_path` against `AllbertAssist.Skills.Resource` entries with
  `kind: :script`.
- Recompute and compare `sha256` before creating a pending confirmation.
- Reject absolute paths, traversal, hidden segments, missing resources,
  non-script resources, invalid args, invalid env, and cwd outside allowed
  roots.
- Produce a redacted summary for confirmations, traces, CLI, and LiveView.
- Enforce direct executable launch mode and path-like arg validation.

Tests:

- trusted inventoried script resolves with digest and metadata
- untrusted, disabled, missing, non-script, hidden, and path-escaping resources
  are denied before confirmation
- digest drift is detected before pending creation
- non-executable scripts are denied unless an explicit interpreter profile is
  implemented and selected by policy
- path-like args outside allowed roots are denied

Implementation status: complete in M2. `AllbertAssist.Execution.SkillScriptSpec`
now resolves trusted/enabled skills through the registry, requires a validated
`run_skill_script` capability contract, matches exact inventoried
`AllbertAssist.Skills.Resource` script paths, recomputes SHA-256, denies stale
expected digests, rejects absolute/traversal/hidden/non-script/missing
resources, enforces direct executable launch mode, constrains operator cwd and
path-like args through `execution.local.allowed_roots`, validates env keys and
limits, and emits trace-safe summaries. The registered action now returns this
inert spec and clearly states that confirmation creation and running arrive in
M3/M4; nothing executes in M2.

### Milestone 3: Registered Action And Confirmation Resume

Goal: create and resume pending script execution through existing confirmation
semantics.

Implementation tasks:

- Extend the existing `AllbertAssist.Actions.Skills.RunSkillScript` registered
  action to create pending confirmations for allowed M2 `SkillScriptSpec`
  records when Security Central returns `:needs_confirmation`.
- Store resume params with skill name, script path, expected digest, cwd,
  args, env summary, timeout, output cap, and selected contract metadata.
- On approval, re-read the pending record, re-check Security Central, re-resolve
  the skill resource by calling the M2 spec resolver with the stored expected
  digest, re-check digest, then run exactly once.
- Resolve stale, denied, or drifted requests without executing.
- Persist script confirmation metadata with the shared keys from this plan.

Tests:

- pending record contains redacted script metadata and no side effect
- approval resumes only `run_skill_script` target records
- policy changes and digest drift between request and approval prevent
  execution
- repeated approval is idempotent
- resolved confirmation records include target result metadata and no raw
  unbounded output

### Milestone 4: Runner, Audit, And Operator Surfaces

Goal: execute trusted scripts with bounded output and make the result visible.

Implementation tasks:

- Add a script runner that uses explicit executable plus argv and the v0.08
  output buffer, timeout, env filtering, redaction, and audit conventions.
- Add execution audit events for requested, approved, denied, stale, succeeded,
  failed, timed-out, and digest-mismatch outcomes.
- Add CLI support under `mix allbert.skills run SKILL SCRIPT [--cwd PATH]
  [--timeout MS] [--max-output-bytes BYTES] -- [ARGS...]`, plus confirmation
  display for script metadata.
- Update `/settings` confirmation cards and resolved rows to show skill,
  script, digest, cwd, timeout, output cap, result, and output preview.
- Keep `activate_skill` read-only and explicit that activation does not execute
  scripts.

Tests:

- successful executable fixture script returns bounded output after approval
- timeout, non-zero exit, output truncation, and redaction are visible
- CLI and LiveView render pending/resolved script metadata through actions
- activation still loads instructions/resources only
- CLI smoke covers request, show, approve, resolved list, and denial output
- LiveView tests cover pending and resolved script confirmation cards

## Non-Goals

- No arbitrary script execution outside trusted skill resource inventory.
- No runtime loading of Elixir modules from skill folders.
- No generic scripting engine interface.
- No package installs or dependency bootstrap; v0.10 owns install policy.
- No external service adapter from script metadata.
- No network-isolation claim under Level 1 host execution.
- No persistent background script processes.
- No auto-enabling imported or untrusted skills.
- No Docker, Podman, Mac/Linux container, remote, or microVM backend.

## Pre-Release Operator Test Matrix

Before v0.09 is marked ready for release/tag, run these smokes against a
disposable `ALLBERT_HOME`, temporary trusted skill root, and temporary allowed
cwd root:

- Allowed trusted script: create a trusted skill with an inventoried executable
  script, request `run_skill_script`, approve it, and verify bounded output.
- Activation remains inert: activate the same skill and prove no script runs.
- Untrusted skill denial: disable or untrust the skill and verify no
  confirmation is created.
- Path escape denial: request `../`, absolute paths, hidden paths, and a
  `references/` resource as the script path.
- Digest drift denial: create a pending request, mutate the script, approve,
  and verify the target does not run.
- Policy denial: set `permissions.skill_script_execute` or
  `execution.skill_scripts.enabled` to deny/false and verify no run.
- Limit behavior: run fixtures for non-zero exit, timeout, output truncation,
  and secret-like output redaction.
- External-boundary smoke: scripts or metadata that imply package install or
  external network work are not treated as v0.10 capabilities.

Expected user-testing flow after implementation:

```sh
export ALLBERT_HOME="$(mktemp -d /tmp/allbert-v09-user.XXXXXX)"
export WORKSPACE="$(mktemp -d /tmp/allbert-v09-work.XXXXXX)"
export SKILL_ROOT="$ALLBERT_HOME/skills/demo-script"

mkdir -p "$SKILL_ROOT/scripts" "$WORKSPACE"
cat > "$SKILL_ROOT/SKILL.md" <<'SKILL'
---
name: demo-script
description: Demo trusted script skill.
metadata:
  allbert.kind: capability
  allbert.actions: run_skill_script
  allbert.permissions: skill_script_execute
  allbert.confirmation: required
---

Run the bundled demo script only through Allbert confirmation.
SKILL

cat > "$SKILL_ROOT/scripts/hello" <<'SCRIPT'
#!/usr/bin/env sh
printf 'hello from skill script\n'
SCRIPT
chmod +x "$SKILL_ROOT/scripts/hello"

mix allbert.settings set skills.enabled demo-script
mix allbert.settings set permissions.skill_script_execute allowed
mix allbert.settings set execution.skill_scripts.enabled true
mix allbert.settings set execution.local.allowed_roots "$WORKSPACE"

mix allbert.skills validate "$SKILL_ROOT"
mix allbert.skills run demo-script scripts/hello --cwd "$WORKSPACE" --
mix allbert.confirmations list
mix allbert.confirmations approve <confirmation-id> --reason "v0.09 smoke"
mix allbert.confirmations list --resolved
```

## Definition Of Done

v0.09 is done when:

- `run_skill_script` exists only as a registered action.
- Trusted inventoried scripts can run after confirmation through Level 1
  host-process controls.
- Untrusted, disabled, non-selected, non-inventoried, hidden, path-escaping,
  digest-drifted, or policy-denied scripts are refused before execution.
- Traces and audits include skill, script, resource digest, security decision,
  confirmation, sandbox level, runner result, and output summary.
- CLI and `/settings` expose the same pending/resolved script metadata.
- Skill activation alone still never executes scripts.
- Release docs state that v0.09 does not add package installs, external
  services, generic scripting engines, or container isolation.
