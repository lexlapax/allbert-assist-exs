# Allbert v0.17 Plugin Contract And Shipped Channel Plugins

## Purpose

v0.17 introduces Allbert's plugin substrate before StockSage becomes the first
real workspace app. The goal is to make plugins the general packaging and
discovery layer for developer extensions while keeping apps, channels, skills,
actions, settings, and supervision behind their existing safety boundaries.

This milestone also converts the v0.16 Telegram and email channels into
source-tree plugins shipped with Allbert under `./plugins/allbert.telegram/`
and `./plugins/allbert.email/`. After v0.17, the channel supervisor and
channel summaries should discover Telegram and email through plugin/channel
registration rather than hardcoded provider lists.

## Expected Inputs

- v0.02 Allbert Home and Settings Central exist.
- v0.03 Agent Skills substrate exists.
- v0.05 Security Central exists.
- v0.07 durable confirmations exist.
- v0.12 local `user_id`, `thread_id`, and SQLite conversation history exist.
- v0.14 session scratchpad and `active_app` exist.
- v0.15 minimal app registration contract exists.
- v0.16 channel substrate, `channel_events`, Telegram adapter, email adapter,
  and `mix allbert.channels` exist.
- The v0.16 channel-specific implementation currently lives under
  `apps/allbert_assist/lib/allbert_assist/channels/telegram/` and
  `apps/allbert_assist/lib/allbert_assist/channels/email/`; v0.17 moves those
  provider-specific modules into source-tree plugin folders while keeping the
  shared channel substrate in `allbert_assist`.
- ADR 0017 is accepted before implementation begins.

## Guiding Rules

- Plugin registration is contract data, not authority.
- Plugin ids are strings. Do not create atoms from plugin manifests, settings,
  channel input, model output, or operator input.
- A plugin can contribute apps, channels, actions, skill roots, settings
  schema entries, child specs, metadata, and diagnostics.
- Apps remain `AllbertAssist.App` modules. App registration is one plugin
  contribution type, not the plugin system.
- Channels remain delivery adapters around
  `AllbertAssist.Runtime.submit_user_input/1` and registered actions.
- Skills remain `SKILL.md` directories discovered by the skill registry and
  governed by existing trust/enablement policy.
- Actions remain registered Jido actions executed through
  `AllbertAssist.Actions.Runner.run/3`, Security Central, confirmations,
  traces, and audits.
- Settings Central owns operator-facing plugin configuration.
- No arbitrary runtime code loading from `<ALLBERT_HOME>/plugins`.
- Source-tree plugin code is compiled only by explicit project/release build
  configuration. Runtime plugin discovery never compiles `.ex`, loads `.beam`,
  runs package managers, or creates atoms from manifests.

## Definitions

### Plugin

A plugin is a package/discovery unit. It has a string `plugin_id`, display
name, version, source, trust status, and zero or more contributions.

Example contribution categories:

- `apps`: compiled `AllbertAssist.App` modules.
- `channels`: channel descriptors and supervised adapter children.
- `actions`: compiled `Jido.Action` modules.
- `skill_paths`: directories containing `SKILL.md` files.
- `settings_schema`: Settings Central schema entries.
- `children`: additional supervised children owned by the plugin.

### Shipped Source-Tree Plugin

A shipped source-tree plugin lives under `./plugins/<plugin-id>/`, is reviewed
with the project, and is compiled into the release by explicit build
configuration. v0.17 should ship at least:

- `allbert.telegram` for the Telegram channel.
- `allbert.email` for the email channel.

`allbert.core` can remain implicit in v0.17 if converting all core actions and
`CoreApp` would add unnecessary churn. The key requirement is that Telegram and
email stop being hardcoded channel supervisor children.

### Project Plugin

A project plugin lives under `./plugins/<plugin-id>/`. It is developer-owned
source-tree material. v0.17 may support manifests and skill roots immediately.
Code-bearing project plugins are allowed only when their modules are compiled
as part of the project/release by explicit developer action; discovery must not
compile code at runtime. v0.17's shipped Telegram and email plugins are the
first examples of this source-tree shape.

### Home Plugin

A home plugin lives under `<ALLBERT_HOME>/plugins/<plugin-id>/`. In v0.17 it
may contribute metadata and skill roots. It must not contribute new compiled
modules, supervised code, channel adapters, actions, or apps unless those
modules already exist in the running release and are explicitly allowed by a
future plan.

### Plugin Manifest

Each folder plugin may include `allbert_plugin.json`:

```json
{
  "schema_version": 1,
  "plugin_id": "example.research_skills",
  "name": "Research Skills",
  "version": "0.1.0",
  "kind": "skills",
  "description": "Example skill-only plugin.",
  "skill_paths": ["skills"],
  "contributions": {
    "apps": [],
    "channels": [],
    "actions": []
  }
}
```

v0.17 validates this manifest for diagnostics and skill discovery. It does not
load code from the manifest.

Shipped source-tree plugins may also declare a compiled plugin module:

```json
{
  "schema_version": 1,
  "plugin_id": "allbert.telegram",
  "name": "Allbert Telegram Channel",
  "version": "0.17.0",
  "kind": "channel",
  "module": "AllbertAssist.Plugins.Telegram"
}
```

The `module` field is metadata, not a dynamic loading instruction. Bootstrap
may use it only when the module is already compiled and the plugin id/module
pair is present in Allbert's shipped or explicitly configured allowlist.
Home-plugin manifests that name modules are rejected or ignored with
diagnostics in v0.17.

## Expected Direction

### Plugin Modules

Add:

- `AllbertAssist.Plugin` behaviour.
- `AllbertAssist.Plugin.Entry` struct for normalized registry entries.
- `AllbertAssist.Plugin.Manifest` parser/validator for folder manifests.
- `AllbertAssist.Plugin.Discovery` for shipped, project, and home plugin
  discovery.
- `AllbertAssist.Plugin.Registry` as the volatile runtime registry.
- `AllbertAssist.Plugin.Bootstrap` to discover and register plugins on boot.
- `AllbertAssist.Plugin.Supervisor` owning registry and bootstrap.

Suggested `AllbertAssist.Plugin` callbacks:

```elixir
@callback plugin_id() :: String.t()
@callback display_name() :: String.t()
@callback version() :: String.t()
@callback validate(keyword() | map()) :: :ok | {:error, [map()]}
@callback apps() :: [module()]
@callback channels() :: [map()]
@callback actions() :: [module()]
@callback skill_paths() :: [Path.t()]
@callback settings_schema() :: [map()]
@callback child_spec(keyword() | map()) :: Supervisor.child_spec() | :ignore
```

Provide defaults for optional callbacks through `use AllbertAssist.Plugin`.
`plugin_id/0`, `display_name/0`, `version/0`, and `validate/1` stay required.

### Source-Tree Plugin Layout And Compile Boundary

Create the repository plugin root:

```text
plugins/
  allbert.telegram/
    allbert_plugin.json
    README.md
    lib/
      allbert_assist/
        plugins/
          telegram.ex
        channels/
          telegram/
            adapter.ex
            client.ex
            parser.ex
            renderer.ex
  allbert.email/
    allbert_plugin.json
    README.md
    lib/
      allbert_assist/
        plugins/
          email.ex
        channels/
          email/
            adapter.ex
            imap_client.ex
            parser.ex
            renderer.ex
            smtp_client.ex
```

Move the v0.16 provider-specific modules from `apps/allbert_assist/lib/` into
these plugin folders. Keep shared channel code in the core app:

- `AllbertAssist.Channels`
- `AllbertAssist.Channels.Supervisor`
- `AllbertAssist.Channels.Event`
- `AllbertAssist.Channels.Identity`
- `AllbertAssist.Channels.Message`

Update `apps/allbert_assist/mix.exs` so the shipped plugin `lib/` paths are
compiled with `allbert_assist` in all environments. Keep this explicit for
v0.17; do not glob arbitrary `./plugins/*/lib` folders into compilation unless
a future plan defines the review, trust, and dependency story.

The runtime registry still discovers plugins by scanning `plugins.scan_paths`.
For code-bearing shipped plugins, discovery validates the manifest and then
registers the already-compiled allowlisted module. For skill-only project/home
plugins, discovery registers manifest metadata and skill roots only.

### Plugin Registry API

The registry should expose:

- `register_module(module, opts \\ [])`
- `register_manifest(manifest, opts \\ [])`
- `registered_plugins(opts \\ [])`
- `lookup(plugin_id, opts \\ [])`
- `diagnostics(opts \\ [])`
- `registered_apps(opts \\ [])`
- `registered_channels(opts \\ [])`
- `registered_actions(opts \\ [])`
- `registered_skill_paths(opts \\ [])`
- `registered_settings_schema(opts \\ [])`
- `plugin_id_for_action(module, opts \\ [])`
- `clear(opts \\ [])` for tests

Registry entries are volatile and rebuilt at boot. Durable operator choices
such as enabled/disabled plugins live in Settings Central, not ETS.

### Startup Order

Update `AllbertAssist.Application` startup order so plugins are available
before apps and channels consume them:

1. Repo and migrations.
2. PubSub, SignalBus, Jido.
3. `AllbertAssist.Plugin.Supervisor`.
4. `AllbertAssist.App.Supervisor`.
5. `AllbertAssist.Session.Scratchpad`.
6. `AllbertAssist.Jobs.Scheduler`.
7. `AllbertAssist.Channels.Supervisor`.

If plugin bootstrap fails for one plugin, Allbert should record diagnostics and
continue booting unless the plugin is a required shipped plugin and the
failure makes a previously released core feature unavailable.

### Plugin Paths And Settings

Add Settings Central keys:

- `plugins.enabled`: string list, default `[]`.
- `plugins.disabled`: string list, default `[]`.
- `plugins.scan_paths`: string list, default `["./plugins", "<ALLBERT_HOME>/plugins"]`.
- `plugins.trusted_project_roots`: string list, default `[]`.
- `plugins.load_policy`: enum, default `"shipped_and_skill_only"`.

`plugins.scan_paths` is the canonical Settings Central key for the operator's
plugin paths.

`plugins.enabled` is an explicit allowlist for optional folder plugins. An
empty list does not disable shipped source-tree plugins. Shipped plugins are
enabled by default unless listed in `plugins.disabled`. Folder plugins are
discoverable by default but unallowlisted code-bearing contributions are
rejected in v0.17; their skill roots follow the trust/enablement rules in
`AllbertAssist.Skills.Registry`.

`plugins.load_policy` allowed values for v0.17:

- `"shipped_and_skill_only"`: shipped compiled plugins plus folder
  manifests/skills only. This is the default.
- `"shipped_only"`: ignore project/home folder plugins.

Do not add a policy that compiles or loads arbitrary code from plugin paths in
v0.17.

### Folder Discovery

Discovery scans only direct child folders of each plugin path. Hidden folders
are skipped. Each plugin folder may include:

```text
allbert_plugin.json
skills/
README.md
```

For v0.17, folder plugins without `allbert_plugin.json` are ignored with a
debug diagnostic rather than treated as errors.

Skill paths from folder plugins are resolved relative to the plugin folder and
must stay inside that folder. Path traversal is rejected.

The default `./plugins` path is not optional implementation decoration: v0.17
must use it for the shipped Telegram and email plugins. That proves the same
folder/manifest/registry path later used by StockSage and developer plugins.

### Shipped Telegram And Email Channel Plugins

Add shipped plugin modules:

- `AllbertAssist.Plugins.Telegram`
- `AllbertAssist.Plugins.Email`

Each plugin contributes:

- a channel descriptor
- channel settings schema entries
- secret reference metadata
- adapter child spec
- any channel-specific CLI metadata needed by `mix allbert.channels`

The v0.16 adapter/client/parser/renderer modules keep their existing module
names, but their files move into `./plugins/allbert.telegram/lib/` and
`./plugins/allbert.email/lib/`. Registration ownership moves to
`AllbertAssist.Plugins.Telegram` and `AllbertAssist.Plugins.Email`.
`AllbertAssist.Channels.Supervisor` must start adapters from registered channel
descriptors, not a hardcoded child list.

Remove the v0.16 hardcoded provider map from `AllbertAssist.Channels`.
Channel summaries, credential status, settings lookups, session derivation
metadata, and adapter child specs should come from registered channel
descriptors. The v0.16 settings keys remain stable:

- `channels.telegram.*`
- `channels.email.*`

### Channel Registry Shape

Add either `AllbertAssist.Channel` plus `AllbertAssist.Channels.Registry`, or
make channel descriptors first-class entries in `AllbertAssist.Plugin.Registry`.
The implementation may choose the smaller shape, but the public channel
surface must support:

- `channel_id`: string, e.g. `"telegram"`, `"email"`.
- `provider`: string, e.g. `"telegram_bot_api"`, `"email_imap"`.
- `adapter`: compiled adapter module.
- `child_spec`: adapter child spec or `{module, opts}`.
- `secret_refs`: settings keys that point at Settings Secrets refs.
- `summary_fields`: redacted settings fields safe for CLI/action output.
- `identity_map_key`: settings key used by `AllbertAssist.Channels.Identity`.
- `session_strategy`: provider-specific session derivation metadata.
- `plugin_id`: contributing plugin id.
- `source`: `:shipped`, `:project`, or `:home`.

`AllbertAssist.Channels.list_channels/0`, `channel_settings/1`,
`credential_status/1`, and `Channels.Supervisor` should consume this registry.

### App And Action Contributions

Update `AllbertAssist.App.Bootstrap` so it registers apps from both:

- existing configured built-in apps
- plugin-contributed app modules

`StockSageStub` remains until v0.20 replaces it, but v0.17 should make the
replacement path clear: `StockSage.Plugin` contributes `StockSage.App`; app
bootstrap registers it through `AllbertAssist.App.Registry`.

Update `AllbertAssist.Actions.Registry` so it can merge:

- static core actions
- plugin-contributed action modules
- app-contributed action metadata

Duplicate action names must be diagnostics, not silent replacement. A plugin
action still needs normal permission metadata and still runs through
`AllbertAssist.Actions.Runner.run/3`.

### Skill Contributions

Update `AllbertAssist.Skills.Registry` so plugin skill roots are discovered
after app skill paths and before user roots, unless implementation analysis
shows app roots need to stay later for precedence. A recommended order:

1. built-in
2. project native/interoperable
3. app roots
4. plugin roots
5. user native/interoperable
6. configured scan paths
7. imported cache

Plugin skill roots are trusted only when the plugin source is trusted under
the v0.17 policy. Home plugin skill roots should be visible but not
auto-enabled if policy marks them pending.

### Settings Schema Contributions

`AllbertAssist.Settings.Schema` should keep core schema entries static and
merge plugin-contributed schema entries when the plugin registry is available.

Bootstrapping must avoid a circular dependency:

- Core plugin settings (`plugins.*`) are static.
- Plugin discovery reads only core settings.
- Plugin-contributed settings become valid after plugin registry startup.

Unknown plugin settings before registry startup should not crash boot; they can
be reported as validation diagnostics when read or written.

### CLI And Actions

Add read-only operator surfaces:

- `mix allbert.plugins list`
- `mix allbert.plugins show PLUGIN_ID`
- `mix allbert.plugins diagnostics`
- registered actions `list_plugins` and `show_plugin`

The plugin CLI must not print secret values or raw unbounded manifests.

`mix allbert.channels list/show`, Telegram simulate/poll-once, and email
simulate/poll-once must keep working after Telegram and email move to shipped
source-tree plugins.

## User Acceptance

- Allbert boots with no external plugin folders present.
- `mix allbert.plugins list` shows shipped Telegram and email plugins sourced
  from `./plugins/allbert.telegram` and `./plugins/allbert.email`.
- `mix allbert.channels list` still shows Telegram and email, now sourced from
  registered plugin channel descriptors.
- Telegram and email simulate flows still produce runtime responses, channel
  event rows, thread ids, and trace metadata.
- A temp `ALLBERT_HOME/plugins/example-skills/allbert_plugin.json` plus a
  `skills/example/SKILL.md` contributes a discoverable skill without loading
  code.
- Invalid plugin manifests are visible through diagnostics and do not crash
  boot.
- Disabled plugins do not contribute apps, channels, actions, or skill paths.
- A plugin-contributed action fixture cannot bypass Security Central or
  `Actions.Runner`.
- A home plugin manifest that claims an app, channel, action, or child module
  is rejected or ignored with diagnostics in v0.17.

## Test Plan

- Plugin behaviour validation unit tests.
- Manifest parser tests for valid manifest, missing required fields, invalid
  ids, oversized strings, invalid paths, path traversal, unknown schema
  versions, and code-bearing home plugin rejection.
- Registry tests for register, lookup, ordering, duplicate ids, diagnostics,
  clear, restart, and disabled registry behavior.
- Startup tests for default plugin paths with no folders, project plugin
  fixture, home plugin fixture, and invalid manifests.
- Compile-path tests or assertions proving only the two shipped plugin `lib/`
  roots are compiled explicitly in v0.17.
- Skill registry tests for plugin skill roots, precedence, trust/pending
  policy, duplicate skill diagnostics, and no runtime code execution.
- Channel tests proving Telegram and email are discovered through plugin
  channel descriptors and that all v0.16 channel tests still pass.
- Action registry tests for plugin-contributed actions and duplicate action
  diagnostics.
- App bootstrap tests for plugin-contributed app fixtures.
- CLI tests for `mix allbert.plugins list/show/diagnostics` and existing
  `mix allbert.channels` commands.
- Full gates from `DEVELOPMENT.md`: compile warnings-as-errors, format check,
  Credo strict, Dialyzer, precommit.

## Non-Goals

- No runtime compilation or loading of arbitrary code from plugin folders.
- No automatic compilation of arbitrary `./plugins/*/lib` directories.
- No remote plugin marketplace or installer.
- No dependency install or package-manager execution during plugin discovery.
- No code hot reload for plugins.
- No sandboxed untrusted plugin execution.
- No plugin-contributed LiveView routes.
- No `AllbertAssist.Surface` DSL; that remains v0.18.
- No hosted accounts, teams, roles, or per-user plugin administration.
- No automatic skill trust, action permission grants, resource grants, or
  confirmation bypass from plugin manifests.
- No StockSage domain implementation; that moves to v0.20.

## Handoffs

- v0.20 StockSage should enter through `./plugins/stocksage` as
  `StockSage.Plugin`, contributing `StockSage.App`, StockSage skill roots, and
  later StockSage actions.
- v0.19 intent enrichment should treat plugin provenance as ranking and trace
  metadata, not authorization.
- v0.26 security evals must include plugin spoofing, disabled-plugin bypass,
  action-name collision, path traversal, and malicious manifest fixtures.
- v0.18 app/surface contract should extend app/plugin contribution metadata
  rather than turning `AllbertAssist.App.Registry` into a general plugin
  registry.
- v0.29 generator should scaffold plugin packages and app packages together.

## Definition Of Done

v0.17 is done when Allbert has a supervised plugin registry, default plugin
path discovery, skill-only folder plugin support, plugin-contributed app/action
fixtures, Telegram and email moved into `./plugins` as shipped channel
plugins, existing channel behavior preserved, operator plugin inspection
commands, and docs/ADRs updated so StockSage can land as a plugin-contributed
app in v0.20.
