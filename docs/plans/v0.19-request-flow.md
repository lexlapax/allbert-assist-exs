## Flow 1: Neutral Runtime Turn

The user submits a normal request through CLI, LiveView, Telegram, email, or a
job-origin runtime call without an app-specific context.

1. The surface calls `AllbertAssist.Runtime.submit_user_input/1`.
2. Runtime normalizes `user_id`, `thread_id`, `session_id`, and `active_app`.
3. With no explicit app context, runtime uses the v0.18 default:
   `active_app: :allbert`.
4. The intent agent asks the enriched intent engine for candidates.
5. The engine collects core Allbert actions, trusted skills, relevant resource
   workflows, channels, jobs, and registered Allbert surfaces.
6. The engine ranks core Allbert candidates above app-specific candidates
   because the session is neutral.
7. The selected candidate is converted into the existing
   `AllbertAssist.Intent.Decision`.
8. If execution is safe, the selected action still runs through the normal
   action runner and Security Central.
9. The response, trace, and conversation metadata include the selected
   decision and bounded candidate explanations.

The user observes: a normal response. If they inspect the trace, they can see
why Allbert selected the route and why other candidates were rejected or ranked
lower.

## Flow 2: App-Scoped Runtime Turn

The user is working inside a registered app context, such as StockSage, and
submits a request from that session.

1. The request includes a `session_id`, explicit `active_app`, or scratchpad
   state that resolves to a known app.
2. Runtime passes `active_app: :stocksage` to the intent agent.
3. The engine collects the same global candidate set as a neutral turn.
4. The engine boosts candidates whose action capability, skill path, or
   surface metadata belongs to `:stocksage`.
5. The engine still validates the selected action, skill, or surface against
   the relevant registry.
6. Security Central applies normal permissions. `active_app` does not
   authorize execution.
7. The response and trace explain that app context influenced ranking.

The user observes: app-relevant results feel natural inside the app. The trace
shows app affinity as a ranking reason, not as a permission grant.

## Flow 3: Neutral Text That Mentions An App

The user mentions a registered app from neutral Allbert context, for example
"open StockSage" or "what can StockSage do?"

1. Runtime defaults the turn to `active_app: :allbert`.
2. The engine sees an explicit app name in the text and collects matching app
   and surface candidates.
3. If the request is navigation-like, the engine may select an inert surface
   navigation decision.
4. If the request would run an app-specific action, the engine selects it only
   when the prompt supplies enough explicit app/action evidence and the action
   validates through the registry.
5. Otherwise the engine returns a direct answer or clarification that points
   to the app surface.

The user observes: Allbert can point them to a registered app without silently
running app-specific actions from neutral context.

## Flow 4: Surface Navigation Candidate

The user asks to open a registered surface.

1. The engine collects surface metadata from the app registry.
2. The requested label/path/app matches a registered surface.
3. The engine selects an inert navigation decision.
4. No action is executed unless a separate registered navigation action exists
   and is selected intentionally.
5. The response includes a safe route suggestion, such as `/agent` or a future
   StockSage route.
6. The trace includes app id, surface id, path, and ranking reason.

The user observes: a route or navigation suggestion. v0.19 does not create
routes, generate UI, mount LiveViews, or render canvas components.

## Flow 5: Risky Resource Request

The user asks Allbert to summarize a URL, inspect a document, run a shell
command, install a package, import a skill, or run a skill script.

1. The engine recognizes the existing v0.11 resource workflow.
2. The matching resource candidate is ranked ahead of generic direct answers.
3. The selected decision contains the same resource access posture as v0.11:
   operation class, resource URI, access mode, scope, limits, and downstream
   consumer.
4. Security Central and confirmation rules are unchanged.
5. If confirmation is required, the response carries the existing Approval
   Handoff data.
6. Candidate metadata is added only as explanatory trace context.

The user observes: the same approval/refusal behavior as v0.11, with better
explanation of why the route was selected.

## Flow 6: Unsupported Resource Or Delegation Request

The user asks for MCP resource access, agent delegation, broad browsing, or a
resource workflow that this version does not implement.

1. The engine collects an unsupported-resource candidate.
2. The candidate validates only as an unsupported/refusal route.
3. The decision uses `unsupported_resource_workflow` or a refusal.
4. No browser, crawler, MCP client, agent call, file reader, or network
   adapter is created by the intent layer.
5. The trace records the unsupported resource and rejected executable
   candidates.

The user observes: a clear unsupported-state response, not a hidden execution
attempt.

## Flow 7: Model-Assisted Low-Confidence Classification

The operator has enabled model-assisted intent classification and Allbert sees
a low-confidence request.

1. The engine collects deterministic candidates first.
2. The engine sends only a bounded, redacted candidate summary to the
   classifier.
3. The classifier returns a structured selection proposal.
4. The engine validates the proposal against the already collected candidates.
5. If the proposal is valid and confident enough, it influences ranking.
6. If the proposal is invalid, unknown, timed out, or low confidence,
   deterministic ranking proceeds.
7. The selected decision still validates through the normal `Decision`,
   registry, Security Central, and confirmation path.

The user observes: better routing in ambiguous cases. The trace may say model
assistance was used, but it does not expose raw model prompts or completions.

## Flow 8: Channel-Origin Turn

A Telegram or email message enters through the v0.16/v0.17 channel plugins.

1. The channel adapter resolves external identity to local `user_id`.
2. The channel adapter derives `session_id` and submits the message to runtime.
3. Runtime resolves `active_app` from request data or scratchpad, defaulting
   to `:allbert`.
4. The enriched engine runs the same candidate collection and ranking as CLI
   and LiveView.
5. Approval-required work returns Approval Handoff data that the channel may
   render natively.
6. The channel does not own a separate intent fork.

The user observes: channel responses match CLI/web behavior for the same
identity and context.

## Flow 9: Job-Origin Turn

A scheduled job submits background work through the runtime.

1. The job runner passes originating `user_id`, `thread_id`, `session_id`,
   `app_id`/`active_app`, channel, and job metadata when available.
2. Runtime marks the request as background context.
3. The engine collects and ranks candidates using the same registries.
4. The decision records background/job context in trace metadata.
5. If confirmation is required, the job becomes blocked through the existing
   job/confirmation path.

The operator observes: job traces explain the selected candidate and preserve
the originating app/thread/user context.

## Flow 10: Unknown Or Malicious Candidate Input

A request, channel payload, job payload, or classifier output names an unknown
action, app, skill, surface, permission, execution mode, or operation class.

1. The engine attempts to match the value through the relevant registry.
2. Unknown values are retained only as rejected diagnostics.
3. Unknown app ids are not atomized.
4. Permissions and execution modes are copied only from registered action
   capabilities.
5. The engine selects a safe fallback: a known candidate, direct answer,
   clarification, unsupported workflow, or refusal.

The user observes: a safe response. The trace explains the rejected unknown
candidate without leaking raw payloads or creating atoms.

## Failure And Safety Cases

| Situation | Expected behavior |
| --- | --- |
| `active_app` is unknown | Fall back to `:allbert`, record diagnostic, do not atomize. |
| Model classifier times out | Deterministic ranking proceeds; trace records bounded diagnostic. |
| Model selects unknown action | Reject model proposal; no side effect. |
| Candidate references denied permission | Return refusal or confirmation-blocked decision; do not execute. |
| Candidate references untrusted skill | Reject executable selection; activation/read-only alternatives may remain. |
| Registered surface path missing at runtime | Return safe navigation/fallback text; do not generate routes. |
| Channel supplies forged app id | Validate through registry; unknown or mismatched context does not authorize. |
| Job-origin action needs confirmation | Block through existing job confirmation path. |
| Trace rendering encounters oversized candidates | Bound or drop candidate details and keep response safe. |

## CLI And Trace Inspection Flows

These flows describe what commands the user runs and what they observe. Full
acceptance criteria and disposable-home smoke instructions live in
`docs/plans/v0.19-plan.md` under "Operator Smoke For Implementation".

### Inspecting Intent Candidates Via Trace

```sh
mix allbert.ask --trace "what can you do?"
```

The user observes: a normal assistant response. The trace output includes an
`## Intent Candidates` section showing the selected candidate kind, id, and
reason, plus top rejected candidates with rejection reasons. No secrets or raw
model output appear.

### Inspecting App Context In Trace

```sh
mix allbert.sessions set-active-app --user local --session v019-smoke stocksage
mix allbert.ask --trace --user local --session v019-smoke "show recent analyses"
mix allbert.ask --trace "show recent analyses"
```

The user observes: the first ask (with `active_app: stocksage`) shows
StockSage-boosted candidates in the trace. The second ask (neutral context)
shows no StockSage action execution and a different candidate selection.

### Checking Model Assist Setting

```sh
mix allbert.settings get intent.model_assist_enabled
mix allbert.settings get intent.max_candidates
```

The user observes: `intent.model_assist_enabled` is `false` by default.
`intent.max_candidates` is `80` by default.

## Non-Goals Visible To Users

- Allbert will not browse, crawl, call MCP resources, delegate to external
  agents, or inspect arbitrary files just because the enriched engine can
  recognize those intents.
- Allbert will not dynamically create routes, LiveViews, canvas nodes, or
  generated UI in v0.19.
- Allbert will not auto-promote conversation turns into markdown memory or use
  v0.21 reviewed-memory retrieval before that milestone exists.
- Allbert will not run app-specific actions from neutral context unless the
  request explicitly names a registered app/action/skill and all normal safety
  checks pass.

## Milestone Update Log

Use this section to record what was implemented, what was deferred, and any
handoff changes during implementation. Each entry should record modules added

- M1: implemented. Added `AllbertAssist.Intent.Candidate`,
  `AllbertAssist.Intent.Ranker`, `AllbertAssist.Intent.Engine`, and
  `AllbertAssist.Intent.EvalFixtures`; added `intent.*` settings defaults;
  accepted ADR 0019; current routing behavior is unchanged while decisions now
  carry bounded selected-candidate metadata.
- M2: implemented. `AllbertAssist.Intent.Engine.collect_candidates/1` now
  collects bounded action candidates from `Actions.Registry`, trusted skill
  candidates from `Skills.Registry`, and registered app surface candidates
  from `App.Registry`; selected decisions include rejected registry candidates
  in bounded trace metadata.
- M3: implemented. `AllbertAssist.Intent.Ranker` now applies context-only
  `active_app` affinity and registered-surface text matching. `Engine.decide/1`
  can return an inert `:open_surface` decision with `surface_target` trace
  metadata for registered app surfaces, and `IntentAgent` only uses that path
  from the direct-answer fallback so existing explicit/risky routes continue
  to win. Unknown active-app strings fall back to `:allbert` without atom
  creation.
- M4: implemented. Existing v0.11-v0.13 risky/resource routes still execute
  through the pre-v0.19 deterministic route order before any surface fallback:
  URL summary/document/import/package/shell/MCP decisions continue to carry
  their `resource_access`, confirmation, and Approval Handoff metadata. Focused
  runtime/agent tests now assert selected intent candidates preserve
  `run_shell_command`, `external_network_request`, and
  `unsupported_resource_workflow` posture rather than being displaced by
  surface navigation.
- M5: implemented. Added `AllbertAssist.Intent.Classifier` with bounded,
  redacted candidate summaries, Settings Central gating, proposal validation,
  and trace diagnostics. Added inert default and process-local fake classifier
  modules; model assistance remains disabled by default and invalid/low
  confidence/timeout proposals fall back to deterministic ranking without
  inventing candidates or recording raw prompts/completions.
- M6: implemented. `AllbertAssist.Trace` renders a dedicated
  `## Intent Candidates` section; `explain_intent` and
  `list_intent_candidates` are registered read-only internal actions; version
  metadata is bumped to `0.19.0`; release docs describe v0.19 behavior and
  manual verification remains the final operator acceptance step before
  tagging.
