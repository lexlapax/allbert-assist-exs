# Allbert v0.12 Request Flow: Local Workspace Identity And Conversation History

This request-flow document is the implementation companion to
`docs/plans/v0.12-plan.md`. It describes the exact runtime and CLI behavior for
local string `user_id`, SQLite threads/messages, and bounded recent context.

## Request Normalization

Every runtime turn begins by normalizing identity and thread context:

1. Read `user_id` from `:user_id`/`"user_id"` when present.
2. Read `operator_id` from `:operator_id`/`"operator_id"` when present.
3. If both are present and differ after string conversion/trimming, return an
   error before signal emission, thread creation, or message persistence.
4. If only `operator_id` is present, use it as `user_id`.
5. If only `user_id` is present, set `operator_id` to the same value for
   compatibility.
6. If neither is present, use `"local"` for both fields.
7. Accept optional `session_id` and propagate it as context only.
8. Accept optional `thread_id` unless `new_thread` is true.
9. If `thread_id` and `new_thread` are both present, return an error before
   signal emission or persistence.

`user_id` is local identity context, not hosted authentication.

## Thread Selection

### Omitted Thread

For `mix allbert.ask --user alice "hello"` or a runtime request without
`thread_id`:

1. Resolve `user_id` to `"alice"`.
2. Find Alice's most recently updated general thread where `app_id` is nil.
3. If one exists, select it.
4. If none exists, create a new general thread for Alice.
5. Continue the runtime turn with the selected `thread_id`.

### New Thread

For `mix allbert.ask --user alice --new-thread "new topic"`:

1. Resolve `user_id` to `"alice"`.
2. Create a fresh general thread with id `thr_...`.
3. Derive the initial title from the first user message.
4. Continue the runtime turn with the new `thread_id`.

`--new-thread` always wins over recent-thread selection and is incompatible
with explicit `--thread`.

### Explicit Thread

For `mix allbert.ask --user alice --thread THREAD_ID "continue"`:

1. Resolve `user_id` to `"alice"`.
2. Fetch `THREAD_ID` scoped by `user_id: "alice"`.
3. If no row is found, return not found/rejected without revealing whether the
   thread belongs to another user.
4. Continue only with an Alice-owned thread.

## Runtime Turn

The happy-path runtime flow is:

1. CLI, LiveView, tests, jobs later, or other channels call
   `AllbertAssist.Runtime.submit_user_input/1`.
2. Runtime normalizes `user_id`, `operator_id`, `thread_id`, `session_id`, and
   `new_thread` using the rules above.
3. Runtime selects or creates the thread.
4. Runtime emits `allbert.input.received` with `user_id`, `operator_id`,
   `thread_id`, `session_id` when present, channel, text, and metadata.
5. Runtime persists the user message with `role: "user"` before invoking the
   agent.
6. Runtime loads up to 12 prior messages from that thread, excluding the
   current user message, in chronological order.
7. Runtime calls `AllbertAssist.Agents.IntentAgent.respond/1` with the current
   text and structured `thread_context`.
8. Runtime emits `allbert.agent.responded` with existing v0.11 response fields
   plus `user_id`, `operator_id`, `thread_id`, and `session_id` when present.
9. Runtime records the markdown trace as today, now with user/thread context.
10. Runtime persists the assistant message after response and trace id are
    known.
11. Runtime returns a response map including `user_id`, `operator_id`,
    `thread_id`, `session_id` when present, status, message, signal ids,
    trace id, actions, decision, resource access, approval handoff, and
    diagnostics.

If the agent fails before the runtime has a response, the user message remains
as submitted history and no synthetic assistant message is written.

## Intent-Agent Context

The runtime passes conversation history as structured context:

```elixir
%{
  text: "what did I just say?",
  channel: :cli,
  user_id: "alice",
  operator_id: "alice",
  thread_id: "thr_abc123",
  session_id: nil,
  thread_context: %{
    thread_id: "thr_abc123",
    user_id: "alice",
    limit: 12,
    messages: [
      %{
        role: "user",
        content: "hello",
        inserted_at: "2026-05-13T18:00:00Z",
        trace_id: nil
      },
      %{
        role: "assistant",
        content: "Hello.",
        inserted_at: "2026-05-13T18:00:01Z",
        trace_id: "/tmp/allbert/traces/example.md"
      }
    ]
  },
  metadata: %{},
  timeout_ms: 120_000,
  input_signal_id: "..."
}
```

The agent may use this for direct answers and intent selection. v0.12 does not
add semantic search, vector retrieval, summarization, or markdown memory
promotion.

## Persistence Details

### Thread

Stored fields:

- `id`
- `user_id`
- `title`
- `kind`
- `app_id`
- `last_message_at`
- `inserted_at`
- `updated_at`

In v0.12, `kind` is `"general"` and `app_id` is nil for runtime-created
threads. App-specific threads wait for the app contract milestones.

### Message

Stored fields:

- `id`
- `thread_id`
- `user_id`
- `role`
- `content`
- `action_log`
- `trace_id`
- `input_signal_id`
- `response_signal_id`
- `metadata`
- `inserted_at`

`action_log` is redacted operator/debug metadata, not an execution authority.
For assistant messages, it should include:

- status
- action summaries
- decision
- resource access
- approval handoff
- diagnostics
- input and response signal ids

Conversation history is SQLite-only. It is not written to markdown memory and
is not auto-promoted to notes, preferences, skills, or durable memory entries.

## CLI Flow

### Ask

- `mix allbert.ask "hello"` uses `user_id: "local"` and the recent local
  general thread, creating one if needed.
- `mix allbert.ask --user alice "hello"` uses Alice's recent general thread or
  creates one.
- `mix allbert.ask --operator alice "hello"` remains accepted and maps to
  `user_id: "alice"`.
- `mix allbert.ask --user alice --operator alice "hello"` is accepted.
- `mix allbert.ask --user alice --operator bob "hello"` fails before runtime
  submission.
- `mix allbert.ask --user alice --thread THREAD_ID "continue"` continues that
  thread only if it belongs to Alice.
- `mix allbert.ask --user alice --new-thread "new topic"` always creates a
  fresh thread.
- `mix allbert.ask --user alice --thread THREAD_ID --new-thread "bad"` fails
  before runtime submission.

CLI output should keep current v0.11 fields and add:

```text
User: alice
Thread: thr_...
```

Status, message, signal, trace, Approval Handoff, diagnostics, and action
rendering should remain compatible with v0.11.

### Threads

- `mix allbert.threads` lists `"local"` threads.
- `mix allbert.threads --user alice` lists Alice's threads.
- `mix allbert.threads --operator alice` lists Alice's threads through the
  compatibility alias.
- `mix allbert.threads --user alice --thread THREAD_ID` shows ordered messages
  for that Alice-owned thread.
- `mix allbert.threads --user alice --thread BOB_THREAD_ID` returns not found
  or rejected without naming Bob.
- `mix allbert.threads --user alice --limit 5` limits thread or message output.

Suggested list output:

```text
thr_abc123 user=alice kind=general app=general messages=4 updated=2026-05-13T18:00:01Z title=hello
```

Suggested show output:

```text
Thread: thr_abc123
User: alice
Kind: general
App: general

[2026-05-13T18:00:00Z] user: hello
[2026-05-13T18:00:01Z] assistant: Hello.
```

## Confirmation-Required Turns

For a risky v0.11 prompt such as:

```sh
mix allbert.ask --user alice --new-thread "check https://example.com/report and summarize it"
```

Expected flow:

1. User message is persisted before intent runs.
2. v0.11 decision/resource-access/Approval Handoff behavior runs unchanged.
3. Runtime response is `:needs_confirmation`.
4. Assistant message is persisted with pending status, confirmation id,
   resource-access entries, and Approval Handoff in `action_log`.
5. Approval or denial still uses `mix allbert.confirmations ...`; thread
   history does not approve, deny, fetch, import, or execute.

The confirmation origin should carry user/thread context when available, but
v0.12 does not require approval resolution to append a new conversation message.

## Acceptance Smoke

Use a disposable `ALLBERT_HOME` and test database.

```sh
export ALLBERT_HOME="$(mktemp -d /tmp/allbert-v012-manual.XXXXXX)"
export DATABASE_PATH="$ALLBERT_HOME/db/allbert.sqlite3"
mix ecto.create
mix ecto.migrate

mix allbert.ask --user alice --new-thread "hello"
mix allbert.ask --user alice "what did I just say?"
mix allbert.threads --user alice
mix allbert.threads --user bob
mix allbert.ask "hello from local"
```

Expected behavior:

- Alice's first command prints `User: alice` and `Thread: thr_...`.
- Alice's follow-up sees Alice's recent thread context.
- Bob does not see Alice's thread.
- Local/default usage prints `User: local`.
- Output includes signal id and trace id as before, plus thread context.

Isolation and conflict checks:

```sh
mix allbert.ask --user bob --thread ALICE_THREAD_ID "try to continue"
mix allbert.ask --user alice --operator bob "conflict"
mix allbert.ask --user alice --thread ALICE_THREAD_ID --new-thread "conflict"
```

Expected behavior:

- Bob cannot continue Alice's thread.
- Conflicting `--user`/`--operator` fails before runtime submission.
- Conflicting `--thread`/`--new-thread` fails before runtime submission.

Confirmation persistence smoke:

```sh
mix allbert.settings set external_services.enabled true
mix allbert.settings set external_services.allowed_hosts example.com
mix allbert.settings set external_services.allowed_paths /
mix allbert.ask --user alice --new-thread "check https://example.com/report and summarize it"
mix allbert.threads --user alice --thread THREAD_ID
```

Expected behavior:

- Ask output still renders v0.11 Approval Handoff.
- Thread show output includes an assistant message for the pending
  confirmation turn.
- The assistant message metadata includes pending status/confirmation context
  without approving or executing the target action.

## Guardrails

- No LiveView thread sidebar.
- No canvas/session UI.
- No semantic/vector retrieval.
- No account/auth schema.
- No scratchpad/session state beyond accepting and propagating `session_id`.
- No app-scoped routing or active-app behavior.
- No markdown memory promotion.
- No changes to v0.11 operation-scoped approvals or remembered grant matching.
