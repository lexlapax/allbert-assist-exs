# Allbert v0.12 Request Flow: Local Workspace Identity And Conversation History

## Runtime Turn

1. CLI, LiveView, or tests call `AllbertAssist.Runtime.submit_user_input/1`.
2. Runtime normalizes:
   - `user_id`: explicit `:user_id`, else `:operator_id`, else `"local"`
   - `operator_id`: retained compatibility alias
   - `thread_id`: explicit thread, selected recent thread, or newly created
   - `session_id`: accepted and propagated, but not interpreted until v0.14
3. Runtime persists the user message before invoking the agent.
4. Runtime passes the last 12 messages from the selected thread to the intent
   agent as bounded context.
5. Runtime emits input and response signals containing `user_id` and
   `thread_id`.
6. Runtime records trace metadata as today, now with user/thread context.
7. Runtime persists the assistant message after the response and trace id are
   known.
8. Runtime returns a response map including `user_id` and `thread_id`.

## CLI Flow

- `mix allbert.ask "hello"` uses `user_id: "local"` and the recent local
  general thread.
- `mix allbert.ask --user alice "hello"` uses Alice's recent general thread or
  creates one.
- `mix allbert.ask --user alice --thread THREAD_ID "continue"` continues that
  thread only if it belongs to Alice.
- `mix allbert.ask --user alice --new-thread "new topic"` always creates a
  fresh thread.
- `--operator` remains accepted. If `--operator` and `--user` are both present
  and differ, the task errors before runtime submission.
- `mix allbert.threads --user alice` lists Alice's threads.
- `mix allbert.threads --user alice --thread THREAD_ID` shows ordered messages
  for that thread.

## Storage Semantics

- `Thread`: `id`, `user_id`, `title`, `app_id`, timestamps.
- `Message`: `id`, `thread_id`, `user_id`, `role`, `content`, `action_log`,
  `trace_id`, inserted timestamp.
- `app_id` is nil/general in v0.12; app-specific threads arrive after the app
  contract milestones.
- Conversation history is SQLite-only and never auto-promoted to markdown
  memory.

## Acceptance Smoke

Use a disposable `ALLBERT_HOME` and test database.

```sh
mix allbert.ask --user alice --new-thread "hello"
mix allbert.ask --user alice "what did I just say?"
mix allbert.threads --user alice
mix allbert.threads --user bob
mix allbert.ask "hello from local"
```

Expected behavior:

- Alice's follow-up sees Alice's recent thread context.
- Bob does not see Alice's thread.
- Local/default usage still works.
- Output includes signal id and trace id as before, plus thread context where
  the CLI renders it.

## Guardrails

- No LiveView thread sidebar.
- No semantic/vector retrieval.
- No account/auth schema.
- No scratchpad/session state beyond accepting `session_id`.
- No markdown memory promotion.
