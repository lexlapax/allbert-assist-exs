## Authority Posture (unchanged)

Per ADR 0021 §4 and ADR 0023 §9:

- Every effectful action triggered from the workspace flows through
  `Actions.Runner.run/3`, Security Central, confirmations (when
  needed), audit. Same as v0.07/v0.22/v0.24.
- Fragments are **advisory**. They shape what the operator sees;
  clicking a button on a fragment-emitted tile invokes an action
  that re-evaluates Security Central as if the operator typed the
  invocation. Fragments cannot unlock permissions, bypass
  confirmations, or invent unregistered actions.
- The workspace renders state and dispatches action invocations
  through registered actions only. It owns zero authority decisions.

## Foundation Posture

The flows below use StockSage analysis where it gives the richest
end-to-end operator proof, but v0.26 is not a StockSage UI milestone.
The workspace must also work with StockSage disabled: empty shell,
core cards, confirmation ephemerals, trace viewer, canvas
persistence, multi-tab sync, and offline text/markdown editing all
remain core Allbert behavior.

## Vocabulary (Operator-Facing)

| Term | Meaning (operator-facing) |
|---|---|
| Workspace | The `/agent` page. The single operator workspace surface. |
| Region | A named area of the workspace (header, chat, canvas, ephemeral). |
| Canvas | The persistent work area inside the workspace. Holds Tiles. |
| Tile | A persistent unit of content inside the Canvas. Survives refresh + restart. Bound to the current thread. |
| Ephemeral Surface | A task-scoped overlay that appears when Allbert needs to focus you (approval card, trace inspector, single-shot prompt). Goes away when dismissed or when the thread closes. |
| Fragment | Any structured UI the runtime renders inside the workspace — a Tile or an Ephemeral Surface. Comes from registered actions, the objective engine, delegate agents, or any other in-BEAM component publishing to the workspace signal topic. |
| Thread | A v0.12 conversation. Each thread has its own Canvas + Ephemeral Surface set. Closing a thread closes its workspace state. |

## Flow 1: First-Time `/agent` Visit (Empty Workspace)

Operator opens `/agent` in a browser for the first time.

1. Phoenix LiveView mounts; workspace shell loads.
2. The shell determines the current thread (creates a new one if
   none exists for this user).
3. Workspace renders: a header (with theme toggle + active-objective
   badge if any), a chat region (composer + timeline), and an empty
   canvas region ("No tiles yet — start a conversation to see
   Allbert's work appear here").
4. No ephemeral surfaces visible (none pending for this thread).
5. Service worker registers for workspace shell assets (offline
   detection active; runtime data still rehydrates from Allbert on
   reconnect).

Expected operator experience: workspace loads under 1s on a local
machine. Layout responds to viewport width: two-pane above 768px
(chat-left, canvas-right), single-pane below (tab toggle between
chat, canvas, and ephemerals).

## Flow 2: Conversational Turn With Workspace-Emitted Tiles

Operator opens `/agent?app_id=stocksage` or resumes an existing
StockSage-active session, then types into the chat composer:
"analyze AAPL". From generic `/agent`, the same text remains a normal
Allbert conversation turn unless the operator explicitly switches the
active app.

1. Composer submits via existing v0.01 `Runtime.submit_user_input/1`
   path. Workspace LiveView assigns show "asking" state.
2. With StockSage as the active app, the intent agent selects
   `StockSage.Actions.RunAnalysis` (per v0.19 ranking). v0.24
   objective framing creates an objective.
3. Security Central returns `:needs_confirmation`. A confirmation
   record persists.
4. The confirmation publishes a Fragment to the workspace signal
   topic: an `:approval_card` ephemeral surface with the analysis
   details + approve/deny buttons.
5. Workspace receives the Fragment via PubSub; the ephemeral
   surface appears overlaid on the canvas region with focus
   trapped to its action buttons (a11y).
6. Operator clicks "Approve" with reason "smoke test".
7. The approval action runs via `Actions.Runner.run/3`. The
   ephemeral surface emits a "dismissed" event and disappears.
8. The objective engine runs the analysis (v0.25 native path).
   Workspace fragment emitters publish operator-visible progress:
   evidence summaries, objective steps, and the final analysis
   summary appear as Tiles in the canvas region. In v0.26, reserved
   StockSage card bodies are stubs that link to the existing
   StockSage routes; v0.27 replaces those stubs with full
   in-workspace cards.
9. When the objective completes, the chat timeline gets a response
   message; the Tiles persist in the canvas.

Expected operator experience:

- The ephemeral approval surface appears within ~500ms of the
  submit.
- Tiles stream into the canvas region as agents complete (live
  feedback during the multi-round debate, not a single
  end-of-analysis dump).
- The canvas region scrolls if needed; the chat region stays in
  view.
- After the analysis completes, refreshing the browser preserves
  the canvas state.

## Flow 3: Tile Inspection And Interaction

Operator clicks a Tile in the canvas (e.g., the `:analysis_card`).

1. The Tile renders its server-side content inline in `/agent`.
2. Text and markdown Tiles expose the Yjs-backed local editor when
   offline editing is enabled.
3. Reconciled stale-base edits render an inline conflict banner with
   a `revert_tile_revision` action.
4. StockSage analysis-card catalog entries render as explicit v0.26
   stubs; v0.27 owns full in-workspace StockSage card bodies.
5. Trace, objective, settings, and jobs components render as
   catalog-backed summaries or links rather than private tile-local
   workflows.

Expected operator experience: rendered Tiles are readable and
keyboard-reachable. Editable text/markdown Tiles preserve local draft
state. Modal ephemerals trap focus. Escape closes dismissible
ephemerals.

## Flow 4: Multi-Tab Sync

Operator has `/agent` open in two browser tabs (same thread).

1. In tab 1: operator triggers a new analysis (Flow 2). The
   approval ephemeral appears in tab 1.
2. **Tab 2 also displays the approval ephemeral.** Both tabs are
   subscribed to the per-(user, thread) workspace topic; the
   FragmentEnvelope broadcast reaches both.
3. Operator approves in tab 1. The ephemeral disappears in BOTH
   tabs simultaneously.
4. Tiles emitted during the analysis appear in both tabs as the
   agents complete.

Expected operator experience: tabs stay in sync within ~100ms of
each other via Phoenix PubSub. Closing one tab does not affect
the other's state. Closing the thread (or thread `completed_at`)
GCs all ephemerals on both tabs simultaneously.

## Flow 5: Navigating To Sibling Routes (And Back)

Operator clicks "Open objective view" from a Tile.

1. Browser navigates to `/objectives/<id>` (existing v0.24
   sibling route).
2. The standalone `ObjectiveLive` page renders (unchanged from
   v0.24).
3. Operator clicks "Back to workspace" (a workspace-aware link in
   the objective page header).
4. Browser returns to `/agent`. The thread is the same; canvas +
   ephemeral state restores from PubSub-cached assigns or
   re-fetches from SQLite/YAML.

Expected operator experience: sibling routes remain deep-linkable
for sharing/bookmarking. The workspace state survives the
navigation round-trip.

Variant: operator can open the same objective AS A TILE inside
the workspace by clicking "open in workspace" instead. The objective
shell renders as an `:objective_card` Tile in the canvas region.
Both paths work; the operator chooses.

## Flow 6: Theme Switching

Operator opens the theme toggle in the workspace header.

1. Three options: Light / Dark / System.
2. Selecting Dark: the workspace re-renders with `data-theme="dark"`
   on the root element; Tailwind dark variants take effect; all
   tiles re-render in dark mode within ~100ms (no page reload).
3. The setting persists via Settings Central
   (`workspace.theme = "dark"`).
4. Other open tabs pick up the setting on reload or their own next
   theme interaction; v0.26 does not broadcast theme changes across
   already-mounted tabs.

Expected operator experience: theme switch is instant + applies
to every region + every Tile + every ephemeral surface. High
contrast variant available via
`workspace.accessibility.high_contrast` setting.

## Flow 7: Offline Text/Markdown Editing And Reconciliation

Operator is connected; opens a text-kind Tile (e.g., a memory
review draft); edits content.

1. Each keystroke updates the browser-owned editor state and stores
   the unsynced edit log in IndexedDB. When online, the workspace
   also pushes bounded update payloads to Allbert and broadcasts
   accepted snapshots to other tabs.
2. Operator loses network connection (browser DevTools "Offline",
   real network loss, or laptop sleep).
3. The workspace shows an offline banner ("Working offline — your
   changes are saved locally and will sync when you reconnect").
4. Operator continues editing. Local updates accumulate in
   IndexedDB.
5. Operator regains network.
6. The service worker detects reconnect; the LiveView reconnects;
   accumulated updates and the latest text/markdown snapshot push
   to the server. Allbert validates the tile/user/thread/revision,
   records a revision row, updates the readable snapshot, and
   broadcasts accepted state to all tabs.
7. If the reconcile was non-trivial (e.g., another tab also edited
   the same tile offline), a per-tile "Conflict reconciled"
   banner appears with a "show merge details" expander. Operator
   can revert any reconciled change via a `revert_tile_revision`
   action button.

Expected operator experience:

- No data loss during offline period unless the browser clears local
  data or the unsynced payload exceeds
  `workspace.offline.indexeddb_quota_mb` (default 32 MB).
- Reconnect is automatic + transparent.
- Conflict banners are bounded — they appear only when the merge
  surfaced changes that need attention; trivial merges (e.g.,
  one tab idle, one editing) reconcile silently.

## Flow 8: Mobile Layout

Operator opens `/agent` on a phone (viewport width < 768px).

1. Workspace renders single-pane with a tab toggle at the top:
   [Chat] / [Canvas] / [Ephemeral].
2. Active tab fills the viewport.
3. Ephemeral surfaces (e.g., approval cards) stack as full-screen
   overlays with a back button to return to the previous tab.
4. The chat composer is anchored to the bottom of the viewport
   (mobile-keyboard-aware).

Expected operator experience: workspace usable on a phone for
common operations. Tile body editing (offline + local sync) works on
mobile keyboards. Pinch-zoom and rotation are CSS-handled.

## Flow 9: Tile Eviction (Cap Exceeded)

Operator is in a long-running thread with many tile emissions.

1. The 65th tile would push the canvas past
   `workspace.canvas.max_tiles_per_thread` (default 64).
2. The Canvas store evicts the oldest non-pinned tile (FIFO by
   `inserted_at`).
3. The evicted tile's body YAML is preserved (soft-delete:
   `deleted_at` set in SQLite; YAML file moves to
   `<tile-id>.deleted.<ts>.yml` under the same canvas directory).
4. The new tile appears at the end of the position order.
5. A `:badge_strip` Fragment is emitted into the canvas header
   ("1 older tile archived — view via `mix allbert.workspace canvas
   list --include-deleted`").
6. Operator can pin tiles to exempt them from eviction (Tile menu
   → "Pin", or `mix allbert.workspace canvas pin <tile-id>`).
7. If every tile in the thread is pinned and the cap is reached,
   the next emission fails with `:canvas_cap_exceeded`; the emitter
   surfaces an operator-readable error and no tile state changes.

Expected operator experience: caps are operator-discoverable
through the canvas-header badge. Pinning prevents surprise eviction
of in-use Tiles. Archived Tiles are inspectable via
`mix allbert.workspace canvas list --include-deleted` + `canvas
show <tile-id>`, and recoverable via `mix allbert.workspace canvas
restore <tile-id>` (clears `deleted_at`, moves the YAML back, and
places the tile at the end of position order).

## Flow 10: Cross-App Specialist Delegation (Operator-CLI, NOT a Workspace Surface)

Operator runs `mix allbert.delegate stocksage.market_context
--user local '{"ticker":"AAPL","analysis_date":"2026-05-17","evidence_mode":"fixture"}'`.

1. The Mix task creates a transient debug objective (v0.25
   contract).
2. The specialist agent runs end-to-end.
3. The report packet prints to stdout.
4. The debug objective marks `:completed`.

**Note**: `mix allbert.delegate` does NOT emit a workspace
Fragment in v0.26. It is an operator-CLI tool for cross-app
specialist reachability, separate from the workspace surface.
Workspace Fragments come from actions running inside an objective
or directly from the objective engine.

This is documented here so operators understand the boundary:
the workspace renders runtime state; the CLI delegate path is a
debug tool that lives alongside but doesn't populate the
workspace.

## Failure And Safety Cases

| Case | Operator-facing behavior |
|---|---|
| Fragment validation fails (invalid signature, unknown emitter, oversize, rate-limit, invalid catalog component) | Tile or ephemeral does NOT appear. A bounded warning logs to the trace. Operator can inspect dropped Fragments via the trace `## Workspace` section. No partial render. |
| LiveView disconnects (network blip, app restart) | Phoenix's automatic reconnect re-mounts the LiveView; state re-hydrates from SQLite + YAML; tiles + ephemerals restore. Brief "reconnecting" banner during the gap. |
| App restart with pending ephemeral | Ephemeral surface persists in SQLite; restores on reconnect. Operator sees the same approval card they were looking at. |
| App restart with offline-pending edits | Browser IndexedDB keeps the unsynced update log. On reconnect, the workspace pushes bounded updates and the latest snapshot; Allbert validates and records a revision before broadcasting. |
| Tile body YAML deleted from disk (e.g., operator manually rm'd) | Workspace shows a "tile body unavailable" placeholder; SQLite metadata row preserved for forensic recovery; operator can `mix allbert.workspace canvas purge <tile-id>` to clean up. |
| Fragment signing secret rotated mid-session | New Fragments emitted with the new secret validate; Fragments already in flight with the previous secret continue validating for the short rotation overlap window printed by `mix allbert.workspace rotate-signing-secret`. After that window, old-secret Fragments are dropped with bounded logs. |
| Theme change while LiveView disconnected | Theme setting persists; on reconnect, re-renders with the new theme. |
| Mobile rotation | CSS-only layout adapts; no LiveView reconnect. |
| `workspace.offline.enabled` set to false mid-session | On next page load, the service worker unregisters, editable tile hooks are not mounted, and the workspace shows an "Offline mode disabled" banner. Existing IndexedDB drafts are retained locally rather than automatically cleared; reconnect and sync drafts before disabling if automatic upload is required. |
| Operator attempts to add tile in a closed thread (thread.completed_at set) | Canvas becomes read-only; tile-add button hidden; CLI returns bounded error. |
| Cross-user `mix allbert.workspace canvas show <tile-id>` | Returns `:not_found`; CLI exits 65. Per ADR 0014 isolation rule. |
| Service worker registration fails (e.g., HTTPS required, not available in dev mode) | Workspace falls back to online-only mode; offline banner says "Offline mode unavailable in this environment". |
| Browser update payload is corrupt or server-rejected | Server state is unchanged. The browser keeps the local Yjs/IndexedDB draft and marks the fallback-shell manifest with recovery metadata so the operator can reopen the offline shell or re-enable sync without silent draft loss. |
| Tile body exceeds `workspace.canvas.tile_body_max_bytes` | Emission rejected with "tile body too large"; emitter gets a bounded error response; nothing renders. |
| Per-emitter rate limit exceeded (default 10/sec/user) | Excess emissions dropped with bounded log; emitter receives a back-pressure error. |
| Approval ephemeral pending when operator navigates to `/jobs` | Ephemeral remains in workspace state; visible when operator returns to `/agent`. |
| `mix allbert.workspace rotate-signing-secret` invoked | New emissions use the new secret; the previous secret is accepted only until the printed overlap expiry. The workspace continues rendering existing tiles + ephemerals because stored surfaces do not need re-validation. |
| Browser tab killed (force-quit) while editing offline | IndexedDB persists unsynced updates across browser sessions; reopening the same tab restores the edit state. |
| Operator clears browser data | All unsynced offline edits are lost; canvas state re-fetches from server. |
| Multi-tab race: two tabs each call `dismiss` on the same ephemeral simultaneously | First write wins; second receives idempotent "already dismissed" response. |

## Non-Goals Visible To Users

- No drag-drop tile reordering, tile resize handles, or durable
  region-layout settings. The `/agent` UI includes a client-side-only
  chat/canvas split-bar for operator ergonomics.
- No multiple canvases per thread.
- No canvas snapshot / undo / time-travel.
- No public AG-UI HTTP endpoint.
- No A2UI / MCP Apps interop.
- No multi-user collaborative cursors.
- No plugin-contributed workspace regions.
- No offline editing of non-text tile kinds (confirmation cards,
  approval inspectors, trace viewers stay read-only offline).
- No real StockSage analysis card rendering (v0.27 ships those;
  v0.26 stubs).
- No model-generated HTML/JS/Elixir at any layer.
