## Request Creation Flow

The default v0.07 pending request flow is:

1. A user request enters through CLI, LiveView, or another runtime surface with
   actor, channel, surface, session, and response-target context.
2. The surface calls the normal runtime/action boundary with that channel
   context.
3. The selected Jido action evaluates permission through Security Central.
4. Security Central returns `:needs_confirmation`.
5. The target action does not perform its target side effect.
6. The target action creates a durable pending confirmation request behind the
   action boundary.
7. The target action returns `:needs_confirmation` with a confirmation id,
   safe display summary, origin channel, permission, risk, selected skill
   summary, and trace metadata.
8. The runtime trace records the confirmation id, origin channel, and Security
   Central decision.

## Channel Context Flow

Confirmation requests are shared runtime state, not channel-private prompts.
The channel that creates a request and the channel that resolves it must be
recorded separately.

Request creation records:

- origin actor
- origin channel, such as CLI or LiveView
- origin surface, such as Mix task name, route, job, or future channel adapter
- origin session id
- origin conversation, thread, or message id when the channel has one
- origin response target when a channel can receive a later callback

Resolution records:

- resolver actor
- resolver channel
- resolver surface
- resolver session id
- resolution reason
- whether resolution happened on the same channel as request creation

The default local single-operator behavior may allow cross-channel approval:
CLI can approve a LiveView-created request and LiveView can approve a
CLI-created request. That handoff must be visible in the confirmation record,
trace, and audit. Future multi-user or hosted channels may tighten this through
Security Central and Settings Central policy.

Initial request source:

- `external_network_request` creates a pending `:external_network`
  confirmation and still makes no network call.

Optional v0.07 request sources:

- Existing settings-write or skill-write actions whose operator-configured
  permission policy resolves to `:needs_confirmation`, if the implementation
  adds resumable write support.

Explicitly out of scope:

- `plan_shell_command` does not create runnable approvals for
  `:command_execute`; the safety floor remains denied until v0.08.
- Skill scripts, package installs, online skill import, arbitrary network
  calls, and module loading do not create runnable approvals in v0.07.

## Durable Store Flow

Pending record creation:

1. Build a confirmation id.
2. Redact and bound target action params.
3. Capture origin actor, channel, surface, session, response target, source
   signal id, trace id, selected skill, capability contract, action capability,
   permission decision, risk, and runner metadata.
4. Write `<ALLBERT_HOME>/confirmations/pending/<id>.yml`.
5. Append a markdown audit entry under
   `<ALLBERT_HOME>/confirmations/audit/YYYY-MM.md`.
6. Emit or attach `allbert.confirmation.requested` metadata.

Resolution:

1. Re-read `<ALLBERT_HOME>/confirmations/pending/<id>.yml`.
2. Validate that status is still `pending`.
3. Apply approval, denial, expiration, cancellation, or adapter-unavailable
   resolution.
4. Move the YAML record to
   `<ALLBERT_HOME>/confirmations/resolved/<YYYY-MM>/<id>.yml`.
5. Append audit summary.
6. Preserve history rather than deleting user-owned confirmation data.

## CLI Flow

Planned commands:

```sh
mix allbert.confirmations list
mix allbert.confirmations list --resolved
mix allbert.confirmations show <confirmation-id>
mix allbert.confirmations approve <confirmation-id> [--reason "..."]
mix allbert.confirmations deny <confirmation-id> [--reason "..."]
mix allbert.confirmations expire
```

CLI rules:

- All commands call registered actions through `Actions.Runner.run/3`.
- The CLI never calls the confirmation store directly.
- Output is redacted, bounded, and stable enough for tests.
- Missing, malformed, resolved, expired, or denied ids return clear
  diagnostics.
- Approval respects `confirmations.allow_cli_approval` and
  `confirmations.allow_cross_channel_approval`.
- CLI approval records CLI as the resolver channel even when the request
  originated in LiveView or a future channel.

## LiveView Flow

The v0.07 UI lives in `/settings` as a Confirmation Requests section in
`AllbertAssistWeb.SettingsLive`.

Required behavior:

- Render pending requests from the shared queue.
- Show recently resolved requests if useful for operator confidence.
- Display status, target action, permission, risk, age, expiration, selected
  skill, and redacted params summary.
- Approve or deny through registered actions.
- Respect `confirmations.allow_liveview_approval`.
- Record LiveView as the resolver channel even when the request originated in
  CLI or a future channel.
- Never own permission decisions, queue state, or target action resumption in
  LiveView code.

Test expectations:

- A request created by CLI or runtime appears in LiveView.
- A request approved or denied in LiveView changes CLI output.
- Raw secret-like values do not appear in rendered HTML.

Implemented DOM and event details:

- `#confirmation-requests` contains `#pending-confirmations` and
  `#resolved-confirmations`.
- Pending rows use `#confirmation-pending-<id>`.
- Approval buttons use `#approve-confirmation-<id>` and submit the
  `approve_confirmation` event.
- Denial forms use `#deny-confirmation-<id>-form` and submit the
  `deny_confirmation` event with an optional reason.
- Recently resolved rows use `#confirmation-resolved-<id>`.
- When `confirmations.allow_liveview_approval` is false, the approve control is
  disabled; the action boundary remains the authoritative enforcement point.

## Approval Flow

Approval flow:

1. CLI or LiveView calls the approve action with confirmation id, optional
   reason, and resolver actor/channel/session context.
2. The approve action re-reads the pending record.
3. The approve action re-checks Security Central with confirmation context,
   including both origin and resolver channel metadata.
4. If policy is now denied, the request is resolved as denied or blocked and
   the target action is not invoked.
5. If the target action supports confirmed resume, approval invokes it through
   `Actions.Runner.run/3` with confirmation context.
6. If the target action has no implemented adapter in v0.07, approval resolves
   as `adapter_unavailable`, records `target_resumed?: false`, and performs no
   target side effect.
7. Trace and audit records include the approval result and any cross-channel
   handoff.

Implemented v0.07 behavior:

- `external_network_request` approval always resolves as `adapter_unavailable`
  after the approval and target policy re-checks pass. Operator-facing output
  explains this as approved, recorded, and not executed because the v0.07
  target has no adapter; external network execution is planned for v0.10.
- If the target permission is denied by current Settings Central policy during
  approval, `approve_confirmation` resolves the request as `denied` with
  policy-block metadata instead of invoking target work.
- `confirmations.allow_cli_approval`,
  `confirmations.allow_liveview_approval`, and
  `confirmations.allow_cross_channel_approval` are enforced at the approval
  action boundary. Disabled surface or cross-channel approval leaves the
  request pending.
- v0.07 has no implemented resumable target adapter, so no approval path
  performs a mutation, shell command, script, package install, import, or
  network call.

Approval must be idempotent. Re-approving a resolved request returns the stored
resolution and does not re-run target work.

When the target action returns after approval, the immediate response goes to
the resolver channel. If the origin channel has a durable response target, the
confirmation metadata should also let that channel show or fetch the final
outcome. CLI has no persistent callback by default, so `list`, `show`, traces,
and audit are the fallback.

## Denial Flow

Denial flow:

1. CLI or LiveView calls the deny action with confirmation id, optional reason,
   and resolver actor/channel/session context.
2. The deny action re-reads the pending record.
3. The deny action moves the record to resolved state as `denied`.
4. The target action is not invoked.
5. Trace and audit records include denial metadata and any cross-channel
   handoff.

If `confirmations.require_reason_for_denial` is true, denial without a reason
should fail validation before the record is resolved.

## Expiration Flow

Expiration flow:

1. `mix allbert.confirmations expire`, startup cleanup, or a future scheduled
   job asks the expiration action to scan pending requests.
2. Requests older than `confirmations.default_ttl_minutes` move to resolved
   state as `expired`.
3. Expiration emits traceable/auditable metadata.
4. Expiration never deletes user-owned confirmation records.

## Redaction Rules

Confirmation records, CLI output, LiveView HTML, traces, and audit notes must
not expose:

- raw API keys
- secret values
- unredacted credential references
- hidden environment values
- arbitrary model scratchpad text
- unbounded params or output
- raw command output

Use Security Central redaction metadata where available. When unsure, store a
summary and the registered action name rather than a raw value.
