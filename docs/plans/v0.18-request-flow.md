## Core Flow: Build And Boot

1. Mix compiles `apps/stocksage`.
2. `apps/stocksage/mix.exs` explicitly includes
   `../../plugins/stocksage/lib` in `elixirc_paths/1`.
3. `StockSage.Plugin` is compiled before runtime plugin bootstrap tries to
   register it.
4. `AllbertAssist.Plugin.Bootstrap` scans `./plugins/stocksage`.
5. Discovery reads `plugins/stocksage/allbert_plugin.json`.
6. Bootstrap validates that the manifest's module is already compiled and
   allowed as a source-tree project plugin.
7. `AllbertAssist.Plugin.Registry` stores the StockSage plugin entry,
   contributions, source, and diagnostics.
8. `AllbertAssist.App.Bootstrap` registers `StockSage.App` through
   `AllbertAssist.App.Registry`.
9. `AllbertAssist.App.StockSageStub` is no longer part of default app
   registration.
10. `AllbertAssist.Skills.Registry` sees `plugins/stocksage/skills`.
11. `AllbertAssist.Actions.Registry` sees the v0.18 StockSage actions.

Invariant: plugin discovery registers already-compiled StockSage modules. It
does not compile source files, load BEAM files, create atoms from manifest
input, or execute package managers.

## Plugin Manifest Flow

Expected manifest:

```json
{
  "schema_version": 1,
  "plugin_id": "stocksage",
  "name": "StockSage",
  "version": "0.18.0",
  "kind": "app",
  "module": "StockSage.Plugin",
  "skill_paths": ["skills"],
  "contributions": {
    "apps": ["StockSage.App"],
    "actions": [
      "StockSage.Actions.ListAnalyses",
      "StockSage.Actions.ShowAnalysis",
      "StockSage.Actions.GetTrends",
      "StockSage.Actions.QueueAnalysis"
    ]
  }
}
```

The manifest is inspected for diagnostics and operator visibility. The runtime
source of authority is the already-compiled `StockSage.Plugin` callbacks.

## App Registration Flow

1. `StockSage.Plugin.apps/0` returns `[StockSage.App]`.
2. App bootstrap passes `StockSage.App` to `AllbertAssist.App.Registry`.
3. `StockSage.App.app_id/0` returns `:stocksage`.
4. `StockSage.App.validate/1` succeeds under the v0.15 contract.
5. Registry rejects duplicate `:stocksage` entries.
6. `mix allbert.apps show stocksage` returns the real `StockSage.App`.
7. `mix allbert.sessions set-active-app --user alice --session sess-1 stocksage`
   succeeds because the app registry has the real app.

If `StockSage.App` fails validation, boot continues with diagnostics, but
StockSage does not contribute app/actions/skills. It must not silently fall
back to `StockSageStub`.

## Domain Storage Flow

StockSage uses `AllbertAssist.Repo` and the existing local SQLite database.

1. A central migration creates `stocksage_*` tables under
   `apps/allbert_assist/priv/repo/migrations`.
2. `apps/stocksage` defines Ecto schemas and contexts using
   `AllbertAssist.Repo`.
3. Context functions require or derive `user_id`, defaulting to `"local"` only
   at CLI/operator boundaries.
4. User-scoped list/get functions filter by `user_id` unless the function is
   explicitly an admin/debug helper for tests.
5. Contexts normalize ticker symbols to uppercase and reject blank symbols.

No `StockSage.Repo` is added in v0.18.

## Import Flow

Command:

```sh
mix stocksage.import_sqlite PATH --user alice
```

Flow:

1. CLI validates `PATH` exists and is a local file.
2. CLI opens the source SQLite database read-only.
3. Importer inspects expected legacy tables and columns.
4. Importer maps legacy rows into StockSage changesets.
5. Each imported row gets:
   - `user_id`
   - `source: "legacy_sqlite"`
   - `legacy_source`
   - `legacy_id`
   - `metadata.imported_at`
6. Dry-run mode runs steps 1-4 and reports counts without writes.
7. Write mode inserts or updates rows idempotently.
8. Output reports inserted, updated, skipped, invalid, and elapsed time.

`--user` defaults to `"local"`. If a compatibility `--operator` alias is
offered, it maps to `user_id`; if both flags are present and differ, import
fails before opening the source database.

Unsupported during import:

- Python execution
- market-data fetching
- remote document extraction
- package install
- skill import
- automatic markdown memory promotion

## Queue Flow

Command:

```sh
mix stocksage.queue create AAPL --user alice --thread-id THREAD_ID
```

Flow:

1. CLI normalizes symbol to `AAPL`.
2. CLI resolves `user_id`, failing if `--user` and any legacy `--operator`
   compatibility option differ.
3. Context creates a `stocksage_analysis_queue` row with status `queued`.
4. No bridge worker starts.
5. No market-data call happens.
6. The returned queue row includes id, user_id, symbol, status, and timestamps.

The queue is durable preparation for v0.20/v0.22. In v0.18 it is not an
execution engine.

## Local Action Flow

`StockSage.Actions.QueueAnalysis`:

1. Runs through `AllbertAssist.Actions.Runner.run/3`.
2. Validates symbol, user_id, optional thread/session context.
3. Creates a queue row through the StockSage context.
4. Emits normal action lifecycle signals/traces.
5. Returns a structured queued result.

`StockSage.Actions.ListAnalyses`, `ShowAnalysis`, and `GetTrends`:

1. Run through `AllbertAssist.Actions.Runner.run/3`.
2. Read only local SQLite data.
3. Filter by user context where applicable.
4. Return bounded summaries, not raw unbounded metadata payloads.

All v0.18 actions carry `app_id: :stocksage`. They do not authorize
cross-app routing by themselves.

## Skill Flow

StockSage skills are discovered from:

```text
plugins/stocksage/skills/
```

Expected skills:

- `queue-analysis`
- `list-analyses`
- `show-analysis`
- `get-trends`

Each skill names only v0.18-safe actions. Do not add `run-analysis` until
v0.20 adds `StockSage.Actions.RunAnalysis`.

## Manual Verification Matrix

1. Boot Allbert after v0.18 migrations.
2. Run `mix allbert.plugins show stocksage`; it shows plugin, app, action, and
   skill contributions.
3. Run `mix allbert.apps show stocksage`; it shows `StockSage.App`, not
   `AllbertAssist.App.StockSageStub`.
4. Run `mix allbert.sessions set-active-app --user alice --session sess-1
   stocksage`; it succeeds.
5. Run `mix allbert.skills list`; StockSage skills appear from
   `plugins/stocksage/skills`.
6. Run import with `--dry-run`; no rows are written.
7. Run import without `--dry-run`; rows are written.
8. Run the same import again; counts show updates/skips, not duplicates.
9. Run `mix stocksage.analyses list --user alice`; imported rows appear.
10. Run `mix stocksage.analyses list --user bob`; Alice's scoped rows do not
    appear.
11. Run `mix stocksage.queue create AAPL --user alice`; a queued row appears.
12. Confirm no Python process, market-data HTTP request, LiveView route, or
    canvas component is involved.

## Explicitly Unsupported In v0.18

- Python bridge execution.
- Native trading-agent execution.
- External market-data API calls.
- `StockSage.Actions.RunAnalysis`.
- StockSage LiveViews or route mounts.
- Canvas components or `canvas_ops`.
- PostgreSQL, Oban hard dependency, hosted auth, accounts, or roles.
- Automatic markdown memory promotion.
