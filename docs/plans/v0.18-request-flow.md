## Runtime Active App Flow

The default v0.18 runtime turn flow is:

1. A user request enters through CLI, LiveView, scheduled jobs, Telegram, email,
   or a future plugin channel.
2. The surface or adapter submits the request through the normal runtime
   boundary.
3. Runtime resolves app context:
   - explicit known `active_app` wins
   - known scratchpad `active_app` for the request `session_id` wins when no
     explicit app is present
   - otherwise the request defaults to `active_app: :allbert`
4. Runtime records the resolved app context in input signal metadata.
5. The intent agent receives the resolved app context as context, not
   authorization.
6. The selected action still runs through `Actions.Runner.run/3`, Security
   Central, confirmations, traces, and audits.
7. Runtime response metadata, assistant message metadata, and trace metadata
   include the resolved app context.

Unknown app ids are not atomized. They produce safe diagnostics and fall back
to `:allbert` unless a stricter caller explicitly treats them as invalid.

## Core App Surface Flow

The built-in chat surface flow is:

1. `AllbertAssist.App.CoreApp` is registered during app bootstrap.
2. CoreApp declares `/agent` through `AllbertAssist.App.SurfaceProvider`.
3. `AllbertAssist.App.Registry.registered_surfaces/0` exposes the validated
   `:allbert` `/agent` surface.
4. `mix allbert.apps show allbert` and `mix allbert.validate_app
   AllbertAssist.App.CoreApp` render the safe surface summary.
5. The existing Phoenix `/agent` route remains the reviewed implementation of
   that surface.
6. v0.24 can discover the same surface and upgrade it into the full workspace
   shell without inventing a second app discovery model.

Surface declaration does not mount routes, execute code, or grant access.

## App Validation Flow

App validation flow:

1. A compiled app module is passed to `mix allbert.validate_app MODULE`, app
   bootstrap, or plugin-contributed app registration.
2. Validation checks the `AllbertAssist.App` identity and lifecycle callbacks.
3. Validation checks declared actions against the registered action registry.
4. Validation checks skill paths, signal declarations, settings schema, and
   legacy navigation summaries.
5. If the module implements `AllbertAssist.App.SurfaceProvider`, validation
   checks surface declarations and component catalog entries through
   `AllbertAssist.Surface`.
6. Valid apps are registered with normalized summaries and diagnostics.
7. Invalid apps are rejected with bounded diagnostics. Other apps keep running.

Validation output must be safe for CLI, logs, and tests. It must not print raw
secrets, raw provider payloads, arbitrary app maps, raw model output, or
unbounded node data.

## Surface Action Binding Flow

Surface action binding flow:

1. A surface node declares an action binding by registered action name or
   compiled module.
2. Surface validation resolves the action through `Actions.Registry`.
3. Validation attaches capability metadata, including permission,
   confirmation requirement, app id, and plugin id when available.
4. Rendering may display the binding as an available UI affordance.
5. If the operator invokes the affordance, execution still enters
   `Actions.Runner.run/3`.
6. Security Central and confirmation workflows remain authoritative.

Invalid action bindings are rejected at validation time. A surface cannot grant
permission, skip confirmation, run shell commands directly, fetch remote URLs,
or introduce model-invented action names.

## Plugin-Contributed App Flow

The v0.18 plugin app flow is:

1. v0.17 plugin bootstrap registers a compiled plugin.
2. The plugin contributes an app module.
3. App bootstrap validates the app through the same v0.18 app/surface
   validator used for core apps.
4. The app registry records app identity, actions, skill paths, signals,
   settings schema, surfaces, and diagnostics.
5. Plugin inspection can show the app contribution.
6. App registration still does not grant permission or trust.

This is the path v0.20 StockSage uses.

## Channel Request Flow

Telegram and email channel flow after v0.18:

1. The channel adapter receives an inbound message or callback.
2. The adapter resolves external identity through Settings Central.
3. The adapter derives session id through its v0.17 plugin channel descriptor.
4. The adapter submits the request through `Runtime.submit_user_input/1`.
5. If no known app context exists for that session, runtime resolves
   `active_app: :allbert`.
6. The channel event row, runtime trace, and conversation message metadata
   preserve the resolved app context.
7. The channel renderer sends the normal response.

Channel adapters do not own app routing, surface rendering, permission policy,
or confirmation storage.

## Surface Fallback Flow

If a surface cannot be rendered:

1. The renderer asks the provider for fallback data or uses the validated
   surface fallback text.
2. The operator sees a bounded text fallback.
3. The fallback may point to the reviewed route path when one exists.
4. The fallback does not expose raw node data, secrets, model output, stack
   traces, or unvalidated component payloads.

## A2UI Adapter Stub Flow

The v0.18 adapter stub flow is:

1. Caller passes a validated `AllbertAssist.Surface` value to
   `AllbertAssist.Surface.Encoder.to_a2ui/1`.
2. The function returns a documented not-implemented result.
3. No AG-UI/A2UI dependency is loaded.
4. No protocol event is emitted.
5. Phoenix LiveView continues to use local Allbert surface data directly.

## User Testing Matrix

After implementation, a local operator should test:

1. Run `mix allbert.validate_app AllbertAssist.App.CoreApp`.
   Expected: command succeeds, prints `app_id: allbert`, and includes the
   `/agent` surface.
2. Run `mix allbert.apps show allbert`.
   Expected: output includes the `/agent` surface and omits raw node maps,
   secrets, provider payloads, and process internals.
3. Run `mix allbert.ask "hello from v0.18"`.
   Expected: normal answer, trace metadata includes `active_app: allbert`.
4. Run:

   ```sh
   mix allbert.sessions set-active-app --user local --session v018-smoke stocksage
   mix allbert.ask --user local --session v018-smoke "hello from stocksage context"
   ```

   Expected: known explicit session app is preserved.
5. Clear or use a fresh session, then run:

   ```sh
   mix allbert.sessions clear-active-app --user local --session v018-smoke
   mix allbert.ask --user local --session v018-smoke "hello after clear"
   ```

   Expected: fallback app is `allbert`.
6. Map a Telegram test identity and run:

   ```sh
   mix allbert.channels telegram map --external-user 123 --user alice
   mix allbert.channels telegram simulate --external-user 123 --chat 456 "hello from telegram"
   ```

   Expected: channel event, trace, and response metadata include
   `active_app: allbert` when no explicit app context exists.
7. Map an email test identity and run:

   ```sh
   mix allbert.channels email map --external-user alice@example.com --user alice
   mix allbert.channels email simulate --external-user alice@example.com "hello from email"
   ```

   Expected: same fallback behavior as Telegram.
8. Run the invalid app validation fixture through the CLI or focused test.
   Expected: clear validation diagnostics and non-zero CLI exit.
9. Confirm `AllbertAssist.Surface.Encoder.to_a2ui/1` exists and returns the
   documented not-implemented result.

## Failure And Safety Cases

- Unknown app id: no atom creation; fallback to `:allbert` with diagnostics.
- Invalid SurfaceProvider: app registration fails for that app only.
- Duplicate same-app surface id: app registration fails.
- Cross-app duplicate route path: diagnostics are visible; boot continues.
- Unknown component id: surface validation fails.
- Unknown action binding: surface validation fails.
- Secret-like props or raw HTML/script props: surface validation fails.
- Plugin-contributed invalid app: plugin/app diagnostics are visible; runtime
  keeps running.

## Non-Goals In Request Flow

- No generated UI.
- No workspace shell replacement.
- No canvas persistence.
- No AG-UI/A2UI emission.
- No dynamic route mounting.
- No permissions from app/surface/plugin registration.
- No memory namespace registration.
