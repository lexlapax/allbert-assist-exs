## Flow 1: Runtime Turn With No App Context

The user submits a request through any surface — CLI, LiveView, Telegram,
email — without any explicit app context.

1. The request enters through the normal surface boundary.
2. The runtime reads the session scratchpad. No `active_app` is set.
3. The runtime resolves `active_app` to `:allbert` as the default.
4. The intent agent receives the request with `active_app: :allbert` as
   session context. This is context, not authorization.
5. The agent selects a skill or action and routes through the normal
   action boundary.
6. Security Central applies the same permission check it would for any turn.
7. The response is returned to the surface.
8. The trace, conversation message, and response metadata include
   `active_app: allbert`.

The user observes: normal response with `active_app: allbert` visible in the
trace output.

## Flow 2: Runtime Turn With Known Session App Context

The user has previously set a session app context — for example by navigating
to a StockSage surface — and submits a new request.

1. The request enters through the surface with a `session_id`.
2. The runtime reads the scratchpad for that session and finds
   `active_app: :stocksage`.
3. `:stocksage` is a known registered app, so the runtime preserves it.
4. The intent agent receives `active_app: :stocksage` as session context and
   may prioritize StockSage skill paths and actions in its selection.
5. The selected action still runs through the normal action boundary. Security
   Central applies the same permission rules regardless of `active_app`.
6. The response, trace, and conversation metadata include `active_app: stocksage`.

The user observes: response with `active_app: stocksage` in trace. Switching to
a new session without a scratchpad app reverts to `active_app: allbert`.

## Flow 3: Runtime Turn With Unknown App Id

A turn arrives — from a channel, model output, or constructed request — with
an unrecognized app id string.

1. The runtime receives the request with `active_app: "phantom_app"`.
2. The runtime checks the app registry. `"phantom_app"` is not a known
   registered app.
3. The runtime falls back to `active_app: :allbert` and records an
   `:unknown_app_id` diagnostic in the response.
4. No atom is created from the unknown string. The fallback proceeds safely.
5. The turn completes normally with `active_app: allbert` in the trace.

The user observes: normal response with `active_app: allbert` and a visible
diagnostic noting the unknown app id was ignored.

## Flow 4: App Validation At Boot

Registered apps are validated when the app registry starts.

1. The app bootstrap loads the list of configured core apps and
   plugin-contributed apps.
2. Each app module is validated against the full v0.18 contract: identity,
   lifecycle, agents, actions, signals, skill paths, settings schema, and
   surface provider surfaces.
3. Valid apps are registered and their surfaces, agents, signals, and settings
   schema are stored in the registry.
4. Invalid apps are rejected with bounded diagnostics. Other registered apps
   continue running.
5. Cross-app duplicate route paths produce a warning diagnostic visible through
   app registry inspection but do not fail the registering app.

The user observes: the system starts normally. `mix allbert.apps list` shows
only valid registered apps. App inspection surfaces any registry warnings.

## Flow 5: CoreApp Surface Declaration

The built-in chat surface is declared through the app/surface contract at boot.

1. `AllbertAssist.App.CoreApp` is registered during app bootstrap.
2. `CoreApp` implements `AllbertAssist.App.SurfaceProvider` and declares the
   `/agent` chat surface.
3. The surface is validated: its path is local, its nodes use known catalog
   components, and any action bindings reference registered actions.
4. The validated surface is stored in the app registry.
5. `mix allbert.apps show allbert` and `mix allbert.validate_app
   AllbertAssist.App.CoreApp` expose the validated surface summary.
6. The existing Phoenix `/agent` route remains the reviewed implementation of
   the surface. v0.18 does not replace it with generated UI.

The user observes: `mix allbert.apps show allbert` output includes a surface
section with id `:agent` and path `/agent`. The app remains reachable at the
existing route.

## Flow 6: App Validation Via CLI

The operator validates a compiled app module.

1. The operator runs `mix allbert.validate_app AllbertAssist.App.CoreApp`.
2. The task resolves the module name safely. Only compiled, loaded modules
   are accepted.
3. Validation runs against the full v0.18 contract including SurfaceProvider.
4. On success: the task prints app id, display name, version, action count,
   skill path count, agent count, settings schema entry count, signal counts,
   and surface ids and paths. The task exits zero.
5. On failure: the task prints clear diagnostics. The task exits non-zero.
   No secrets, raw node data, raw provider payloads, or raw model output
   appear in the output.

The user observes: readable summary or readable diagnostics. Exit code reflects
validity.

## Flow 7: Surface Action Binding

A surface declares a button that invokes a registered action.

1. An app's `SurfaceProvider.surfaces/0` returns a surface node with an action
   binding referencing a registered action name.
2. At validation time, the surface validator looks up the action in the
   registered action registry. If the action is not found, the binding is
   rejected and the surface fails validation.
3. If the action is found, the binding is enriched with the action's permission
   class and confirmation requirement from the registry. The permission is
   stored as metadata; it is not modified.
4. The validated surface is registered and its binding metadata is visible in
   `mix allbert.apps show`.
5. When the user interacts with the surface affordance, the invocation still
   routes through the standard action boundary. Security Central and
   confirmation workflows are unchanged.

The user observes: the surface displays the affordance. Invocation behaves
exactly as if the user had called the action through any other surface. No
new permissions are granted by the surface declaration.

## Flow 8: Plugin-Contributed App Registration

A plugin contributes an app that implements the full v0.18 contract.

1. The plugin is loaded through the plugin registry at boot.
2. The plugin's `apps/0` callback lists the contributed app module.
3. The app bootstrap retrieves the app module from the plugin registry.
4. The app module is validated through the same full v0.18 app validator used
   for core apps.
5. If valid, the app is registered. Its surfaces, agents, signals, and settings
   schema contributions join the global registries.
6. Plugin inspection (`mix allbert.plugins show <id>`) shows the app
   contribution. App inspection (`mix allbert.apps show <id>`) shows the
   registered app.

This is the path v0.20 StockSage uses. No special migration or lite-to-full
path exists; plugin-contributed apps use the full contract from day one.

## Flow 9: Settings Schema From App And Plugin Contributions

An operator reads or validates a settings key contributed by an app or plugin.

1. The operator runs `mix allbert.settings get apps.myapp.some_setting` or
   the runtime reads that key during a turn.
2. Settings Central merges the static core schema with schema entries
   contributed by registered apps and plugins.
3. The contributed key is validated against its declared type and default.
4. The value is returned (or the default is used if not set).
5. If the app registry or plugin registry is unavailable (e.g., during early
   boot), Settings Central proceeds with the static schema and logs a warning.
   It does not crash.

The operator observes: app and plugin settings keys behave the same as core
settings keys. Schema-declared defaults are applied. The source of any setting
value is traceable.

## Flow 10: Channel Turn Active App Fallback

A Telegram or email channel turn arrives with no prior session app context.

1. The channel adapter receives an inbound message.
2. The adapter derives a session id from the channel descriptor.
3. The adapter submits the turn through the standard runtime boundary.
4. The runtime reads the scratchpad for the derived session. No `active_app`
   is set.
5. The runtime resolves `active_app: :allbert` as the default.
6. The turn completes normally. The channel event row, trace, and conversation
   metadata include `active_app: allbert`.

The user observes (from the channel): normal response. In trace output visible
via CLI, `active_app: allbert` is present.

## Flow 11: Surface Fallback When Rendering Cannot Use Nodes

A renderer cannot render a validated surface's nodes.

1. The renderer calls the surface provider's `fallback_surface/1` with the
   surface id, or uses the surface's stored `fallback_text`.
2. The operator sees a bounded text description of the surface.
3. The fallback may reference the route path (e.g., "Allbert chat is available
   at /agent").
4. No raw node data, secrets, model output, stack traces, or unvalidated
   component payloads appear in the fallback.

## Flow 12: AG-UI Adapter Stub

A caller asks for an AG-UI/A2UI encoding of a validated surface.

1. The caller passes a validated `AllbertAssist.Surface` value to the encoder.
2. The encoder returns a documented not-implemented result.
3. No AG-UI or A2UI dependency is loaded.
4. No protocol event is emitted.
5. The Phoenix LiveView path continues to use local Allbert surface data
   directly.

The v0.26 (formerly v0.24) workspace shell and post-v0.31 (formerly post-v0.29)
adapter work will replace the stub with real encoding when external protocol
compatibility is needed.

## Manual Verification Matrix

After full implementation, verify v0.18 end-to-end with the following steps.
Use a disposable Allbert Home to avoid touching `~/.allbert`.

```sh
export SMOKE_HOME=$(mktemp -d /tmp/allbert-v018-smoke.XXXXXX)
```

### Step 1: Validate CoreApp

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.validate_app AllbertAssist.App.CoreApp
```

Expected:
- Exit code 0.
- Output includes `app_id: allbert`.
- Output includes a surface entry with `id: :agent` and `path: /agent`.
- Output shows action, agent, settings schema, and signal counts.
- Output does not include raw node maps, secrets, or process internals.

### Step 2: Show Allbert App

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.apps show allbert
```

Expected:
- Exit code 0.
- Output includes a surface provider section with the `/agent` surface.
- Output includes counts for actions, agents, settings schema entries, and
  signals.
- Output does not include raw node data, raw provider payloads, process pids,
  or ETS internals.

### Step 3: Ask With No App Context

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.ask --trace "hello from v0.18"
```

Expected:
- Normal assistant response.
- Trace output includes `active_app: allbert`.

### Step 4: Set Session App And Ask

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.sessions set-active-app \
  --user local --session v018-smoke stocksage

ALLBERT_HOME="$SMOKE_HOME" mix allbert.ask --trace \
  --user local --session v018-smoke "hello from stocksage context"
```

Expected:
- `set-active-app` command succeeds.
- Ask response trace includes `active_app: stocksage`.
- Response content is normal regardless of `active_app` context.

### Step 5: Clear Session App And Ask

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.sessions clear-active-app \
  --user local --session v018-smoke

ALLBERT_HOME="$SMOKE_HOME" mix allbert.ask --trace \
  --user local --session v018-smoke "hello after clear"
```

Expected:
- `clear-active-app` command succeeds.
- Ask response trace includes `active_app: allbert` (fallback restored).

### Step 6: Telegram Channel Simulate

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.channels telegram simulate \
  --external-user 123 --chat 456 "hello from telegram"
```

Expected:
- Channel simulate completes without error.
- Trace or metadata includes `active_app: allbert`.

### Step 7: Email Channel Simulate

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.channels email simulate \
  --external-user alice@example.com "hello from email"
```

Expected:
- Channel simulate completes without error.
- Trace or metadata includes `active_app: allbert`.

### Step 8: Invalid App Validation

Run `mix allbert.validate_app` against a deliberately invalid module or a known
invalid fixture:

```sh
# Example: module missing required app_id/0 callback
ALLBERT_HOME="$SMOKE_HOME" mix allbert.validate_app Elixir.String
```

Expected:
- Exit code non-zero.
- Output prints a clear diagnostic explaining the validation failure.
- Output does not include stack traces, secrets, or unbounded data.

### Step 9: AG-UI Stub Existence

Confirm the stub exists in a focused test or IEx session:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix run -e \
  'IO.inspect AllbertAssist.Surface.Encoder.to_a2ui(%AllbertAssist.Surface{id: :agent, app_id: :allbert, label: "test", path: "/agent", kind: :chat, status: :available, fallback_text: "test"})'
```

Expected:
- Output is `{:error, :not_implemented}`.
- No exception raised.
- No AG-UI dependency loaded.

### Step 10: Settings Merge Verification

Verify that a settings key contributed by a registered app is readable:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.settings list
```

Expected:
- Output lists settings keys.
- If any app or plugin has contributed schema entries, those keys appear in
  the list (or are at least not missing from validation).
- Static core settings keys remain present.

## Failure And Safety Cases

| Scenario | Expected behavior |
|----------|-------------------|
| Unknown app id string in request | Falls back to `:allbert`; `:unknown_app_id` diagnostic in response; no atom creation |
| App with invalid SurfaceProvider | That app's registration fails; other registered apps continue running |
| Same-app duplicate surface id | App validation/registration fails; diagnostics are visible via validation output |
| Cross-app duplicate route path | Both apps register; a `:duplicate_route_path` warning diagnostic is recorded |
| Unknown component in surface node | Surface validation fails; app registration fails for that app |
| Action binding referencing unknown action | Surface validation fails; the binding is rejected at validation time |
| Secret-like prop key in surface node | Surface validation fails; app registration fails for that app |
| Raw HTML string in surface node prop | Surface validation fails; app registration fails for that app |
| App Registry unavailable during settings read | `runtime_schema/0` proceeds with static schema; warning is logged; no crash |
| Plugin Registry unavailable during settings read | `runtime_schema/0` proceeds with static + app schema; warning is logged; no crash |
| `mix allbert.validate_app` receives non-module string | Task prints clear error; exits non-zero; no atom creation from arbitrary input |
| `to_a2ui/1` called with valid Surface | Returns `{:error, :not_implemented}`; no protocol event emitted |
| Plugin-contributed app with invalid surface | Plugin-app diagnostics visible; runtime keeps running |
