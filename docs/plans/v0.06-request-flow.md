# Allbert v0.06 Request Flow: Action-Backed Allbert Skills

This document is the living request-flow map for v0.06 Action-Backed Allbert
Skills. It should be updated during implementation as planned module names
become actual module names and operator verification notes become real results.

## Status

Status: in implementation after the v0.05 release and v0.06 architecture
readiness sweep on 2026-05-02. Milestones 1 through 5 are complete and tested.

v0.06 makes trusted skills action-backed for existing registered Allbert
actions. It adds no shell execution, script execution, package installation,
external-network adapter call, online skill import, autonomous skill creation,
or confirmation queue.

## Boundary

Skills describe and bind capabilities. Actions execute capabilities.
Security Central evaluates whether a selected skill/action/permission context
may proceed. The action runner remains the invocation boundary.

v0.06 must not treat any of these as executable authority:

- `SKILL.md` markdown instructions
- YAML metadata
- `allowed-tools`
- bundled scripts
- resource inventory entries
- arbitrary Elixir module names
- package manifests
- model output

Only registered Jido actions can execute, and only through
`AllbertAssist.Actions.Runner.run/3`.

## Architecture Guardrails

The v0.06 implementation should preserve the architecture contract from
`AGENTS.md` and `DEVELOPMENT.md`:

- runtime-facing work starts at `AllbertAssist.Runtime.submit_user_input/1` or
  a lower registered action/context boundary
- Jido agents decide, route, or delegate
- registered Jido actions perform bounded work with schemas, permissions,
  structured results, and lifecycle metadata
- the action registry is the source of truth for executable action names and
  modules
- the action runner is the only runtime invocation path for selected actions
- Security Central evaluates policy, risk, trust, redaction, audit, and trace
  metadata at the action boundary
- Settings Central owns any new operator policy or helper configuration
- Allbert Home owns durable local skill/helper data
- CLI, LiveView, jobs, and future channels stay thin
- pure contract validation and markdown rendering helpers may stay plain
  Elixir behind the action boundary

Expected implementation pattern:

```text
surface or runtime caller
  -> signal or runtime/action boundary
  -> IntentAgent route decision
  -> skill contract validation helper
  -> Actions.Registry resolution
  -> Actions.Runner.run/3
  -> registered Jido action
  -> Security Central decision at action boundary
  -> lifecycle signals, runner metadata, trace
```

Do not continue implementation under this request flow if a milestone appears
to require direct action-module invocation, private security policy in an
operator surface, config-backed operator settings, durable writes outside
Allbert Home, or execution powers assigned to skill metadata. Update the plan
and add an ADR first if that architectural shape becomes necessary.

## Current Baseline

Implemented by earlier milestones:

- `AllbertAssist.Skills.Parser` parses standard Agent Skills directories.
- `AllbertAssist.Skills.Registry` discovers trusted and enabled skills.
- `AllbertAssist.Skills.CapabilityContract` preserves inert Allbert metadata.
- Built-in skill wrappers live under `apps/allbert_assist/priv/skills/`.
- `list_skills`, `read_skill`, and `activate_skill` are registered actions.
- `activate_skill` loads wrapped instructions and resource inventory only.
- `AllbertAssist.Actions.Registry` resolves registered action names/modules.
- `AllbertAssist.Actions.Registry` exposes canonical action capability
  metadata through `capabilities/0`, `agent_capabilities/0`,
  `internal_capabilities/0`, and `capability/1`.
- `AllbertAssist.Actions.Capability` stores non-executing action capability
  metadata for validation and traces.
- `AllbertAssist.Skills.CapabilityContract.validate/2` validates contract
  actions, permissions, confirmation policy, single-action execution shape,
  and execution eligibility.
- `AllbertAssist.Skills.Skill` records carry `contract_validation` summaries.
- `list_skills`, `read_skill`, and `activate_skill` surface compact contract
  validation status and execution eligibility without running declared
  actions.
- `AllbertAssist.Skills.ActionPlan` validates a selected skill/action pair
  before the intent agent invokes the action runner.
- Deterministic intent routes for the initial built-in surface now pass
  selected skill metadata through runner context.
- `AllbertAssist.Actions.Runner` carries selected skill metadata and action
  capability metadata into runner metadata and action entries.
- `AllbertAssist.Signals` includes selected skill, contract status, and
  redacted contract/action capability metadata in action lifecycle signals.
- `AllbertAssist.Security.Context` includes selected action capability and
  selected skill contract validation details.
- `AllbertAssist.Skills.LocalHelper` validates local skill directories and
  scaffolds standard local `SKILL.md` wrappers without creating executable
  code.
- `validate_skill` and `create_skill` are registered helper actions invoked
  through the action runner, not through the intent-agent tool surface.
- `mix allbert.skills validate PATH` and `mix allbert.skills create ...`
  provide thin operator CLI access to those helper actions.
- Security Central now knows the `:skill_write` permission class through
  Settings Central key `permissions.skill_write`.
- `AllbertAssist.Actions.Runner.run/3` emits action lifecycle signals and
  attaches runner metadata.
- `AllbertAssist.Security` evaluates policy, trust, risk, redaction, audit,
  trace, selected skill, selected action, and safety-floor metadata.

## Contract Validation Flow

Implemented in Milestone 1 for pure validation and surfaced through
registry/list/read/activation output in Milestone 2.

Implemented flow:

```text
skill directory
  -> SKILL.md parser
  -> AgentSkillSpec
  -> CapabilityContract
  -> AllbertAssist.Actions.Registry.capability/1
  -> Security.Policy.permission_classes/0
  -> executable contract summary or diagnostics
  -> Skill registry record / activation output / trace metadata
```

Validation inputs:

- selected skill name
- source scope and source path
- trust status
- parsed Allbert metadata
- declared action names
- declared permission classes
- declared confirmation mode
- resource inventory
- action registry metadata

Validation outputs should include:

```elixir
%{
  status: :valid | :invalid | :none,
  execution_eligible?: true,
  actions: [
    %{
      name: "append_memory",
      module: AllbertAssist.Actions.Intent.AppendMemory,
      permission: :memory_write,
      registered?: true,
      skill_backed?: true
    }
  ],
  permissions: [:memory_write],
  confirmation: :not_required,
  diagnostics: []
}
```

Implemented validation entrypoint:

```elixir
AllbertAssist.Skills.CapabilityContract.validate(contract, skill: skill)
```

Implemented action metadata entrypoints:

```elixir
AllbertAssist.Actions.Registry.capabilities()
AllbertAssist.Actions.Registry.agent_capabilities()
AllbertAssist.Actions.Registry.internal_capabilities()
AllbertAssist.Actions.Registry.capability("append_memory")
```

Diagnostic examples:

- unknown action
- unregistered module reference
- unknown permission
- permission/action mismatch
- unknown confirmation mode
- multi-action workflow not executable in v0.06
- script or package-install declaration is inventory-only
- external network execution is unavailable until v0.10
- selected skill is pending, disabled, imported-disabled, or untrusted

Invalid contracts remain inspectable. They do not become runnable.
Structurally valid contracts for local skill folders that are not yet trusted
or enabled should report `execution_eligible?: false` until registry policy
allows them into the trusted catalog.

## Skill-Backed Runtime Flow

Implemented in Milestone 3 for the initial deterministic built-in surface.

Current prompt flow:

```text
CLI / Agent LiveView
  -> AllbertAssist.Runtime.submit_user_input/1
  -> allbert.input.received
  -> IntentAgent deterministic route
  -> selected built-in skill
  -> AllbertAssist.Skills.ActionPlan.build/4
  -> action params from existing deterministic param builder
  -> Actions.Runner.run(action_name, params, ActionPlan.runner_context(plan))
  -> allbert.action.requested
  -> registered action
  -> Security Central decision through the action boundary
  -> allbert.action.completed
  -> allbert.agent.responded
  -> record_trace action when tracing is enabled
```

The selected skill should be explicit in context:

```elixir
%{
  selected_skill: "append-memory",
  skill_metadata: %{
    selected_skill: "append-memory",
    source_scope: :built_in,
    trust_status: :trusted,
    kind: :native_action,
    capability_contract: %{status: :valid, actions: ["append_memory"]}
  },
  selected_action: "append_memory"
}
```

The action is still resolved and executed by the runner. The intent agent does
not call action modules directly and does not execute an action just because
skill metadata says it exists.

## Deterministic Route Map

v0.06 preserves current route behavior while making the selected skill explicit:

| Prompt shape | Selected skill | Runner action | Params source |
| --- | --- | --- | --- |
| plain answer | `direct-answer` | `direct_answer` | original prompt text |
| remember/store note | `append-memory` | `append_memory` | existing memory text builder |
| identity/preference heuristic | `append-memory` | `append_memory` | existing personal memory builder |
| recall/recent memory | `read-recent-memory` | `read_recent_memory` | existing recall query builder |
| what skills are available | `list-skills` | `list_skills` | empty params |
| read/show skill | `read-skill` | `read_skill` | existing skill-name parser |
| run/execute/shell prompt | `plan-shell-command` | `plan_shell_command` | existing command parser |
| fetch/browse/download/call network | `external-network-request` | `external_network_request` | existing network request parser |

`activate_skill` remains the progressive-disclosure action. It loads the target
skill's instructions and resources, but it does not run that target skill's
declared action.

## Security Decision Flow

Security Central should see both the selected skill and selected action:

```text
Runner context
  -> action_metadata / selected_action_module
  -> selected_skill / skill_metadata / capability_contract
  -> Security.Context.normalize/2
  -> Security.Policy.resolve/2
  -> Security.Risk.classify/2
  -> Security.Decision.build/1
  -> action response permission_decision
  -> runner metadata
  -> lifecycle signals
  -> trace markdown
```

Rules:

- Unknown permissions deny.
- Unknown actions deny.
- Unregistered modules deny.
- Missing, disabled, pending, or undiscoverable selected skills deny
  non-read-only permissions.
- Skill metadata never grants permission by itself.
- Settings Central may tighten policy, but safety floors still cap premature
  shell, script, package-install, network, online-import, raw-secret-read,
  unknown-action, and unknown-permission authority.

## Trace Flow

When tracing is enabled, traces should show:

- selected skill name
- skill source scope and trust status
- contract validation status
- contract actions and permissions
- selected action name and module
- permission decision
- risk tier
- confirmation requirement
- policy source and safety floor
- trust boundary
- runner requested/completed signal ids
- action outcome
- memory path or category when a memory write happens

Trace output should keep the existing `## Skill Metadata` and
`## Security Metadata` sections if possible. If the implementation adds a new
section, use a compact name such as `## Skill Action Metadata`.

Traces must remain markdown, inspectable, portable, and redacted.

## Lifecycle Signal Flow

`allbert.action.requested` should include redacted summaries of:

- action name
- action module
- params summary
- selected skill
- contract status
- source signal id
- actor, channel, and session when available

`allbert.action.completed` should include redacted summaries of:

- action name
- action module
- status
- duration
- permission/security decision
- selected skill
- contract status
- response summary
- sanitized error when present

No lifecycle signal should include raw secrets, raw credential values, or raw
untrusted resource contents.

## Validation And Creation Helper Flow

Implemented validation flow:

```text
operator CLI
  -> mix allbert.skills validate PATH
  -> Actions.Runner.run("validate_skill", %{path: PATH}, context)
  -> AllbertAssist.Actions.Skills.ValidateSkill
  -> PermissionGate.authorize(:read_only, context)
  -> AllbertAssist.Skills.LocalHelper.validate_dir/1
  -> parse local skill directory
  -> validate Agent Skills fields
  -> validate Allbert action contract
  -> return diagnostics and execution eligibility
```

Implemented creation flow:

```text
operator CLI
  -> mix allbert.skills create NAME ACTION PERMISSION DESCRIPTION... [--root ROOT]
  -> Actions.Runner.run("create_skill", params, context)
  -> AllbertAssist.Actions.Skills.CreateSkill
  -> PermissionGate.authorize(:skill_write, context)
  -> AllbertAssist.Skills.LocalHelper.create_skill/1
  -> normalize skill name
  -> verify action exists, is registry-marked skill-backed, and permission matches
  -> render standard SKILL.md only
  -> write under requested local root or <ALLBERT_HOME>/skills
  -> validate the written skill
  -> return path and diagnostics
```

Creation helper constraints:

- no generated Elixir modules
- no script folders
- no package manifests
- no dependency install instructions
- no external network adapters
- no online imports
- no automatic execution
- no automatic trust escalation outside existing registry policy

Implemented security surface:

| Permission | Setting key | Default | Safety floor | Risk tier |
| --- | --- | --- | --- | --- |
| `:skill_write` | `permissions.skill_write` | `allowed` | `allowed` | `medium` |

The helper actions are registered as internal/operator helpers. They are
available to CLI and future operator surfaces through the action runner, but
they are not included in `IntentAgent.action_modules/0`; this keeps v0.06 from
adding autonomous skill creation. A skill that declares `create_skill` or
`validate_skill` as its action is invalid for skill-backed routing because the
registry marks both helper actions as `skill_backed?: false`.

## Operator Test Sketch

Use a disposable home:

```sh
export ALLBERT_HOME=/tmp/allbert-v006-demo
export ALLBERT_TRACE_ENABLED=true
rm -rf "$ALLBERT_HOME"
```

Expected commands after implementation:

```sh
mix allbert.ask --trace "what skills are available?"
mix allbert.ask --trace "remember that I prefer concise release handoffs"
mix allbert.ask --trace "what do you remember?"
mix allbert.ask --trace "read skill append-memory"
mix allbert.ask --trace "activate skill append-memory"
mix allbert.ask --trace "run ls"
mix allbert.ask --trace "fetch https://example.com"
mix allbert.skills validate apps/allbert_assist/priv/skills/append-memory
mix allbert.skills create demo-memory append_memory memory_write "Save a short memory helper" --root "$ALLBERT_HOME/skills"
mix allbert.security status
```

Expected behavior:

- built-in skills have valid action-backed contracts
- memory write and read behavior remains the same from the operator view
- shell command requests produce plans only and do not execute
- external network requests report confirmation need and do not make calls
- skill activation does not execute the activated skill
- local skill validation reports valid contracts with `execution_eligible?: false`
  until registry trust policy includes the local skill
- local skill creation writes only `<root>/<skill-name>/SKILL.md`
- traces show selected skill, validated contract, action, Security Central
  decision, runner metadata, and outcome
- output and traces are redacted

## Closeout Checks

Suggested closeout `rg` checks:

```sh
rg -n "Code\\.eval|Module\\.create|Code\\.compile|:code\\.load|load_file" apps/allbert_assist/lib apps/allbert_assist_web/lib
rg -n "System\\.cmd|Port\\.open|run_shell|script execution|package install|online import|Req\\." apps/allbert_assist/lib apps/allbert_assist_web/lib
rg -n "AllbertAssist\\.Actions\\.Intent\\.[A-Za-z]+\\.run\\(" apps/allbert_assist/lib apps/allbert_assist_web/lib
rg -n "AllbertAssist\\.Security\\.Policy|AllbertAssist\\.Settings|Settings\\.Secrets" apps/allbert_assist_web/lib/allbert_assist_web/live apps/allbert_assist/lib/mix/tasks
```

Interpretation:

- direct action module `run/2` calls from runtime surfaces should be replaced
  by `Actions.Runner.run/3`
- `Req` may appear only in already inert or future-planned text until v0.10
- security policy and secret internals should not be called directly from
  operator surfaces
- code compilation/loading should not appear in skill paths
