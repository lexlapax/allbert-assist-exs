# Allbert v0.04 Action-Backed Allbert Skills

## Purpose

v0.04 should take the Agent Skills substrate from v0.03 and wire a trusted
subset of skills into Allbert's existing Jido action and permission model.

v0.03 answers: "Can Allbert discover, validate, trust, read, and activate
standard skills?"

v0.04 answers: "Can a trusted built-in skill safely map to a known Allbert
action, carry the right permission contract, and leave a trace?"

It does not answer: "Can Allbert execute arbitrary skill scripts or turn
third-party skill folders into Elixir modules?" The answer to that remains no.
Agent Skills are portable manifests, instructions, and resources. Allbert
execution lives in registered Elixir/Jido actions.

## Dependency From v0.03

v0.04 assumes v0.03 has:

- a `SKILL.md` parser and validator
- discovery scopes and trust state
- a registry-backed skill list/read/activate path
- resource inventories for `scripts/`, `references/`, and `assets/`
- inert Allbert metadata parsing
- trace metadata for skill activation

As of v0.03 closeout, the parser, validator, discovery, trust policy,
registry-backed list/read/activate foundation, built-in `SKILL.md` pack,
progressive-disclosure activation, resource inventory output, trace skill
metadata, and CLI/LiveView operator coverage exist in
`AllbertAssist.Skills.Parser`, `AllbertAssist.Skills.AgentSkillSpec`,
`AllbertAssist.Skills.Resource`, `AllbertAssist.Skills.Registry`,
`AllbertAssist.Skills.Skill`, `AllbertAssist.Skills.CapabilityContract`,
`AllbertAssist.Actions.Intent.ActivateSkill`, `AllbertAssist.Trace`, and
`apps/allbert_assist/priv/skills/`.

If those are not boring and tested, v0.04 should not start.

## Non-Goals

- No arbitrary script execution.
- No auto-converting `SKILL.md` files or bundled scripts into Elixir modules.
- No runtime loading of Elixir modules from arbitrary skill folders.
- No external-network skill search or installation without confirmation.
- No shell command execution.
- No autonomous skill creation.
- No broad intent-engine enrichment beyond the existing conservative routing.

## Capability Contract

v0.04 promotes valid Allbert metadata from inert data to an executable
capability contract.

A contract may include:

- action names, mapped to registered Jido action modules
- permission classes, validated against `PermissionGate.permission_classes/0`
- confirmation policy
- memory effects
- trace effects
- resource/script policy

The permission gate remains the source of truth. `metadata.allbert.*` and
`allowed-tools` never grant permission by themselves.

Executable Allbert skills are standard `SKILL.md` wrappers plus registered
Elixir/Jido actions. Built-in wrappers should point at existing action names;
third-party skills with Allbert metadata remain non-executable unless they are
trusted and their metadata maps to registered actions and known permission
classes. Bundled scripts remain resources to inspect or plan around, not a v0.04
execution substrate.

The v0.04 runtime pipeline is:

```text
activated/trusted skill
  -> Allbert metadata contract
  -> action runner
  -> registered Jido action
  -> permission gate
  -> result plus action lifecycle signals
  -> trace
```

## Implementation Order

### Milestone 1: Contract Validator

Status: planned.

Implementation tasks:

- Validate Allbert metadata overlays into `CapabilityContract` records.
- Reject unknown action names, unknown permission classes, unknown confirmation
  policies, and executable script policies.
- Reject contracts that request module loading, shell/script execution, package
  installs, or external network access.
- Surface contract errors in skill diagnostics.

Acceptance criteria:

- Valid built-in contracts pass.
- Invalid third-party or fixture contracts are visible but non-executable.
- Unknown action or permission makes the skill non-executable.

### Milestone 2: Action Lifecycle Runner

Status: planned.

Implementation tasks:

- Add one internal runner for action invocation, tentatively
  `AllbertAssist.Actions.Runner`.
- Have the intent agent call actions through the runner instead of invoking
  action modules directly.
- Emit or log `allbert.action.requested` before action execution.
- Emit or log `allbert.action.completed` after action execution or structured
  failure.
- Attach selected skill and contract metadata to action context and result
  metadata.
- Refuse runner requests that do not resolve to a registered Jido action.

Acceptance criteria:

- Every selected action path includes action requested/completed signal
  metadata.
- Trace output includes selected skill, action name, permission decision, and
  outcome.
- Command, network, and script execution remain blocked unless a later
  milestone adds confirmation and sandboxing.

### Milestone 3: Minimal Intent Agent Integration

Status: planned.

Implementation tasks:

- Let the existing deterministic routing attach a registered built-in skill
  before choosing an action.
- Use skill name, description, kind, and contract metadata only for the current
  v0.01 route families.
- Use built-in `SKILL.md` wrappers around existing Allbert actions; do not
  generate executable code from skill markdown.
- Preserve v0.01 safety behavior for memory, recall, command planning, and
  external network prompts.

Acceptance criteria:

- Memory prompts select the built-in memory skill and then the `append_memory`
  action.
- Recall prompts select the memory-read skill and then the
  `read_recent_memory` action.
- Command execution remains denied.
- External network remains confirmation-required and not executed.
- General intent ranking, confidence, alternatives, and model-assisted routing
  remain v0.09 work.

### Milestone 4: First Skill Creation Workflow

Status: planned.

Implementation tasks:

- Add a local helper for drafting a standard skill, tentatively
  `mix allbert.skills.new`.
- Generate a spec-compatible `SKILL.md` template.
- For action-backed skills, require selecting from existing action names and
  permission classes.
- If a new side effect is needed, the helper may scaffold a separate Elixir
  action file, but generated code must be reviewed, tested, compiled, and
  registered like ordinary app code before any skill can invoke it.
- Add a validation command, tentatively `mix allbert.skills.validate`.
- Add a small eval file shape for activation and safety checks.

Acceptance criteria:

- A developer can create a standard Agent Skill skeleton from Allbert.
- A developer can validate a standard or Allbert-extended skill.
- The generated skill is usable by other Agent-Skills-compatible clients.

### Milestone 5: Operator Loop And Documentation

Status: planned.

Implementation tasks:

- Update request-flow documentation for skill-backed actions.
- Add CLI and LiveView operator tests.
- Update the roadmap with v0.04 status and the next milestone.

Acceptance criteria:

- `mix precommit` passes.
- CLI and LiveView can list, read, activate, and execute a trusted
  action-backed built-in skill.
- Traces show selected skill, selected action, permission decision, action
  lifecycle signal metadata, and final response.
- No unsafe execution capability has been added.

## Definition Of Done

v0.04 is done when:

- Trusted Allbert capability skills map only to known Jido actions.
- Skill contracts map only to known permission classes.
- Allbert does not auto-convert skill folders, markdown, or scripts into
  executable Elixir modules.
- Action execution goes through a common lifecycle runner.
- Traces include selected skill, contract, action, permission, and outcome.
- The first skill creation/validation helper exists.
- v0.01 safety boundaries still hold.
