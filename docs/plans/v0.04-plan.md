# Allbert v0.04 Jido Runtime Convergence Refactor

## Purpose

v0.04 makes Allbert's existing runtime shape more honest before Security
Central and executable skills add more policy pressure.

The rule is Boundary Actions: externally invoked, effectful, security-relevant,
or observable domain operations should enter through signals, internal agents,
and Jido actions. Pure parsing, validation, schemas, formatting, and data
transformations remain plain Elixir behind those boundaries.

This is a convergence refactor, not a new capability release. It plans and
implements the shared runtime boundary that Settings Central, Skills, Memory,
Trace, Security Central, jobs, and channels can use without inventing private
logic paths.

## Expected Inputs

- v0.01 signal-first runtime, primary intent agent, and Jido actions exist.
- v0.02 Settings Central exists with CLI and LiveView surfaces.
- v0.03 Skills substrate exists with list/read/activate actions and registry
  APIs.
- ADR 0001 already says channels should converge on a signal-driven core.

## Expected Direction

- Add a shared action runner boundary for invoking registered Jido actions.
- Have the intent agent delegate selected actions through the runner instead of
  manually calling `SomeAction.run/2`.
- Emit consistent `allbert.action.requested` and `allbert.action.completed`
  metadata for action success, denial, and failure.
- Move user-facing settings flows toward settings actions or settings signals
  rather than direct `Settings`/`Secrets` calls from Mix tasks and LiveView.
- Keep direct domain calls valid inside actions, tests, and pure internal
  modules.
- Plan internal domain agents where coordination is useful:
  `SettingsAgent`, `SkillsAgent`, `SecurityAgent`, and memory/trace actions or
  agents.
- Document that Security Central, action-backed skills, confirmations,
  execution, jobs, channels, and memory review must consume these boundaries.

## Non-Goals

- No Security Central decision implementation; v0.05 owns that.
- No action-backed skill execution; v0.06 owns that.
- No confirmation queue; v0.07 owns that.
- No shell, skill script, package install, external network, or online import
  capability.
- No wrapping pure helper modules as agents or actions.
- No broad rewrite of existing Settings, Skills, Memory, or Trace storage
  internals.

## Definition Of Done

v0.04 is done when:

- ADR 0007 records the Boundary Actions rule.
- `docs/plans/v0.04-request-flow.md` documents the converged signal, agent,
  action runner, domain action, trace, and audit flow.
- Runtime-facing docs identify current direct-call debt and the target action
  boundary for settings, skills, memory, trace, and future security work.
- Existing user-facing behavior remains unchanged.
- Security Central is shifted to v0.05 and all later milestones are renumbered.
