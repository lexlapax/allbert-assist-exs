# Allbert v0.04 Jido Runtime Convergence Refactor

## Purpose

Status: implementation-ready.

v0.04 makes Allbert's existing runtime shape consistent before Security
Central and executable skills add more policy pressure.

The rule is Boundary Actions: externally invoked, effectful, security-relevant,
or observable domain operations enter through signals, internal agents or
runtime routers, and registered Jido actions. Pure parsing, validation,
schemas, formatting, storage helpers, and data transformations remain plain
Elixir behind those boundaries.

This is a convergence refactor, not a new capability release. It must preserve
current CLI, LiveView, memory, settings, skills, trace, and permission behavior
while removing private runtime paths that would otherwise fork policy later.

## Expected Inputs

- v0.01 signal-first runtime, primary intent agent, Jido actions, permission
  gate, memory, and traces exist.
- v0.02 Settings Central exists with CLI and LiveView surfaces.
- v0.03 Skills substrate exists with list/read/activate actions and registry
  APIs.
- ADR 0001 says channels should converge on a signal-driven core.
- ADR 0007 defines Boundary Actions.

## Boundary Contract

Implement these boundaries without changing user-visible capability:

- `AllbertAssist.Actions.Registry`: the canonical list of registered action
  modules. `IntentAgent.action_modules/0` should delegate to it.
- `AllbertAssist.Actions.Runner.run/3`: the only runtime-facing action
  invocation path for agents, CLI tasks, LiveViews, jobs, and future channels.
  It accepts an action module or registered action name, params, and context.
- `AllbertAssist.Signals` or an equivalent helper: constructs Allbert signal
  structs and centralizes action lifecycle metadata. v0.04 may keep the current
  log-only signal behavior; it must not invent durable bus semantics.
- Domain agents only where they coordinate runtime-facing flows:
  `SettingsAgent`, `SkillsAgent`, and memory/trace action boundaries. v0.05
  owns `SecurityAgent` behavior.

Runner results must preserve existing action return shapes and add consistent
metadata:

- requested action name and module
- requested/completed signal ids
- status: `:completed`, `:needs_confirmation`, `:denied`, or `:error`
- permission/security decision when present
- duration in milliseconds
- selected skill when present
- sanitized error reason when execution fails

## Direct-Call Policy

Allowed direct calls:

- actions calling plain domain modules such as `Settings`, `Skills`, `Memory`,
  `Trace`, parsers, schemas, codecs, and storage helpers
- pure unit tests for those modules
- migrations, seeds, and bootstrap code where no runtime behavior is involved

Disallowed runtime-facing direct calls after v0.04:

- `IntentAgent` directly calling `SomeAction.run/2`
- Settings LiveView directly calling `Settings` or `Settings.Secrets`
- `mix allbert.settings` directly calling `Settings` or `Settings.Secrets`
- runtime trace recording directly performing durable trace effects without a
  trace action or trace signal boundary
- future jobs/channels bypassing the runner for settings, skills, memory,
  trace, or security behavior

### Milestone 2: Action Registry, Signal Helper, And Runner

Status: complete.

Implementation tasks:

- Add `AllbertAssist.Actions.Registry` with all current intent and settings
  actions.
- Add `AllbertAssist.Actions.Runner.run/3`.
- The runner resolves actions only through registered modules or registered
  action names.
- The runner creates `allbert.action.requested` before invocation and
  `allbert.action.completed` after success, denial, confirmation need, or
  error.
- Unknown actions return a structured `:denied` or `:error` result without
  invoking arbitrary modules.
- Keep action modules responsible for domain work and permission checks until
  v0.05 Security Central changes that preflight story.

Implemented modules:

- `AllbertAssist.Actions.Registry`
- `AllbertAssist.Actions.Runner`
- `AllbertAssist.Signals`

Implementation notes:

- The registry includes every current intent and settings runtime action in a
  stable order and intentionally excludes test-only actions such as
  `AllbertAssist.Actions.Multiply`.
- The runner resolves registered action names or modules only, invokes the
  current action `run/2` callback, and returns structured denial for unknown
  names or unregistered modules.
- The signal helper creates log-oriented `allbert.action.requested` and
  `allbert.action.completed` Jido signals with redacted params and response
  summaries. v0.04 still does not add a durable signal bus.
- Runner metadata is attached to the top-level response and to each action
  entry in `response.actions`.
- YAML error helper cleanup removed stale `normalize_yaml_error` and
  `normalize_error` helpers in favor of explicit exception-aware message
  normalization.

Acceptance criteria:

- Every registered action can be invoked through the runner.
- Unknown action names and unregistered modules do not run.
- Runner metadata is consistent for completed, denied, confirmation-needed,
  and error outcomes.
- Existing action modules do not need to know who called them.

### Milestone 3: Intent Agent Uses The Runner

Status: planned.

Implementation tasks:

- Change deterministic routing to return `{action_name, params}` or equivalent
  action selections.
- Replace all direct `SomeAction.run/2` calls in `IntentAgent` with
  `Actions.Runner.run/3`.
- Keep the current route predicates and prompt behavior unchanged.
- Attach request, selected route, selected action, selected skill, and input
  signal metadata to the runner context.

Acceptance criteria:

- `IntentAgent` no longer directly invokes action modules.
- Existing intent agent tests still pass with the same statuses and messages.
- Trace action lists include runner lifecycle metadata for selected actions.
- No new model-assisted routing is introduced.

### Milestone 4: Settings Surface Convergence

Status: planned.

Implementation tasks:

- Route `mix allbert.settings` list/get/explain/set/providers operations
  through settings actions via the runner.
- Route Settings LiveView list/update/provider credential operations through
  settings actions via the runner.
- Add or adjust a settings credential action so explicit CLI/LiveView provider
  key entry can call the encrypted secret store through an action boundary
  without rendering raw secrets.
- Keep `Settings`, `Settings.Store`, `Settings.Schema`, `Settings.Secrets`,
  and `Settings.YamlCodec` as plain modules behind actions.

Acceptance criteria:

- Settings CLI output remains behavior-compatible.
- `/settings` can list settings, update safe settings, and set provider keys.
- Raw secrets do not appear in CLI output, LiveView, action metadata, traces,
  audits, logs, or tests.
- `mix allbert.settings` and Settings LiveView no longer own settings
  semantics directly.

### Milestone 5: Skills, Memory, And Trace Boundary Convergence

Status: planned.

Implementation tasks:

- Keep skills list/read/activate behavior behind the existing registered skill
  actions.
- Add a skills coordination boundary only if it simplifies routing; do not move
  parser, registry, or struct logic into an agent.
- Add a trace-recording action or internal trace signal path so durable trace
  writes are observable as runtime work.
- Keep memory append/read behavior unchanged and still markdown-first.
- Do not rename public skill, memory, or trace outputs unless tests require the
  new runner metadata.

Acceptance criteria:

- Skill list/read/activate still use the v0.03 registry behavior.
- Trace writing is observable through the v0.04 boundary.
- Memory append/read behavior and markdown file layout are unchanged.
- Skill activation still does not execute scripts, shell commands, package
  installs, network calls, or action-backed contracts.

## Definition Of Done

v0.04 is done when:

- ADR 0007 records the implemented Boundary Actions rule.
- `docs/plans/v0.04-request-flow.md` reflects actual module names and request
  flow.
- All user-facing action execution can pass through `Actions.Runner`.
- `IntentAgent` no longer directly calls action modules.
- Settings CLI and Settings LiveView no longer own settings or secrets
  semantics directly.
- Trace recording has an action or signal boundary.

- Existing user-facing behavior is unchanged except for added lifecycle
  metadata.
- All focused tests and `mix precommit` pass.
- `git diff --check` passes for docs.
