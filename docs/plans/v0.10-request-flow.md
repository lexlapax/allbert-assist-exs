# Allbert v0.10 Request Flow

v0.10 turns external service calls, package install workflows, and online skill
import into confirmed registered actions. It does not add a generic browser,
generic package manager shell, untrusted-code runtime, or external channel.

## Common Runtime Shape

Every v0.10 side effect follows the same path:

1. User input enters through CLI, LiveView, or another existing local surface.
2. The channel submits to `AllbertAssist.Runtime.submit_user_input/1` or calls
   a registered action through `AllbertAssist.Actions.Runner.run/3`.
3. The selected action validates params and builds a redacted request spec.
4. Security Central evaluates permission, risk, selected action, selected
   skill, source/profile, secrets, and policy.
5. If confirmation is needed, the action writes a durable confirmation record
   and performs no target side effect.
6. CLI or `/settings` displays the pending request from the confirmation store.
7. `approve_confirmation` re-reads the record, re-checks Security Central, and
   resumes only the target action named in the record.
8. The target action re-validates params, policy, and source/target metadata.
9. The side effect runs through the relevant adapter.
10. The resolved confirmation, trace, and audit entry record the result.

No channel writes confirmation files directly, bypasses Security Central,
calls `Req` directly for runtime work, runs package managers directly, or
trusts imported skills directly.

## Resource Access Security Posture Handoff

v0.10 names the shared resource access security posture that connects the
historical local foundations from v0.08/v0.09 with the new network substrate in
v0.10. This section is documentation/handoff framing; it does not rewrite
v0.08 or v0.09 as if they implemented the generalized contract.

Local resource consumers include:

- local command cwd and path-like operands from v0.08
- trusted skill script resources from v0.09
- future local skill directory import
- future local document/file inspection

Remote resource consumers include:

- API JSON or service responses
- web pages
- PDFs, markdown, text, or other documents
- direct `SKILL.md` URLs or skill archives
- package metadata
- future summarization inputs

The common primitive is a URI-backed resource reference, not a marketplace
record or raw path/URL string. M7 implements the first inert
`AllbertAssist.Resources.Ref` data with `origin_kind`, `canonical_id`,
`operation_class`, `access_mode`, `scope`, source/profile, method, limits,
digest, redaction, downstream consumer, and metadata fields. The reference may
also be rendered through `AllbertAssist.Confirmations.ResourceMetadata` in
CLI, `/settings`, audits, and traces. It describes the resource posture only;
it does not approve, remember, fetch, import, install, summarize, or execute.

M10 hardens the identity rules behind that primitive. M12 is expected to make
URI identity explicit through a canonical `uri` or `resource_uri` field while
preserving `origin_kind`, `canonical_id`, and existing scope fields as
compatibility metadata. Canonical fields are the only authority for policy,
remembered grants, and matching. Display fields are for operators. For
example, an external request may store canonical URL
`https://example.com/a?token=secret` while rendering display URL
`https://example.com/a?[REDACTED]`; the redacted display value must not become
the remembered grant scope. Under the URI-first model, host local paths become
`file://...`, Allbert-managed resources may use `allbert://home/...`, source
profiles may use `allbert://sources/<kind>/<id>`, skill inventory resources
may use `skill://...`, package specs should prefer PURL-compatible `pkg:`
identities, and future MCP/agent resources may be represented as inert
`mcp://...`, `agent://...`, or `agent+https://...` metadata until a later plan
adds execution authority. Local path scopes are canonicalized with
intermediate symlink resolution before subtree comparisons. Source-profile
grants carry endpoint fingerprint metadata so a same-id source profile whose
base/API URL changes requires re-approval. Confirmation approval uses
registered resumable-action metadata before attempting target resume.

Operation class matters. A future approval or remembered grant for
`summarize_url` cannot authorize `import_skill`; `inspect_document` cannot
authorize package installation; `import_local_skill` cannot authorize
`run_skill_script`; and a remote skill source grant cannot authorize
activation, trust, dependency install, or script execution.

M8 stores remembered grants under `resource_grants.remembered` in Settings
Central. The record is generic local/remote resource approval memory:
resource origin, canonical scope, operation class, access mode, downstream
consumer, origin/resolver channel, timestamps, revocation, audit path, reason,
and metadata. `AllbertAssist.Resources.Grants.find_applicable/2` requires the
caller to pass the current action permission so Security Central is re-checked
without making the grant store a workflow router. v0.10 does not implement
arbitrary URL/document summarization, but the external adapter, local
execution metadata, confirmation resume, audit, trace, resource references,
and remembered grant metadata are the substrate that v0.11 builds on.

Example future consumer flow:

1. The user asks "check this URL and summarize it" from CLI, REPL, web chat,
   or another channel.
2. Intent emits an execution-aware decision with a remote resource reference
   and operation class `summarize_url`.
3. Security Central evaluates the fetch/read portion before any download.
4. Approval Handoff renders approve, deny, details, and remember-scope
   affordances in the requesting channel when possible.
5. Approved execution fetches bounded content through the v0.10 `Req` adapter.
6. A summarizer agent/action receives extracted content only after approval. If
   no summarizer exists, the result must say so clearly instead of pretending
   the summary ran.

Direct skill URL import is the same posture with operation class
`import_skill`: fetch approved remote `https://...` content, validate it as
Agent Skills content, audit it, import it disabled and untrusted, and never use
a summary or inspect grant as import authority.

Local skill directory import is the local counterpart with operation class
`import_local_skill`: canonicalize the directory to a `file://...` resource,
constrain it to an allowed
local resource scope, validate it as Agent Skills content, audit it, import or
register it according to Settings Central policy, and never use an import grant
as script execution authority.

## External Service Request Flow

### Request Creation

`external_network_request` remains the agent-facing action for network-shaped
requests.

Required input:

- `request` or normalized service request intent
- direct URLs embedded in free-form `request` text are extracted before policy
  validation
- optional `source_text`
- optional service profile
- method
- path or URL constrained by service/source profile
- optional query, headers, JSON/body/form params
- timeout and response limit, capped by Settings Central

The action builds `External.RequestSpec`:

- normalized method
- service/source profile
- operation class, defaulting to `external_service_request` unless a higher
  level consumer such as future `summarize_url`, `inspect_document`, or
  `import_skill` supplies a more specific class
- redacted URL summary
- host and path
- redacted headers
- body type and redacted body summary
- timeout
- max response bytes
- redirect policy
- retry policy
- credential ref status

If policy allows the shape but requires confirmation, the pending record stores:

- target action: `external_network_request`
- target permission: `external_network`
- target execution mode: `req_http`
- redacted params summary
- resume params ref
- service/source profile
- request spec digest
- origin channel/surface/response target
- Security Central decision
- trace/signal/runner metadata

No HTTP request is made during pending creation.

### Approval Resume

`approve_confirmation` resumes `external_network_request` only when:

- confirmation record is pending
- approval surface is allowed
- cross-channel policy allows the resolver channel
- Security Central still returns confirmation-eligible or allowed for
  `external_network`
- `external_services.enabled` is true
- service/source profile is enabled
- method, host, path, timeout, redirect, retry, and response cap still pass
- secret refs are configured when required
- request spec digest still matches the stored safe summary

The approved action calls `External.HttpClient` with `Req.request/1`.

Default request posture:

- `retry: false`
- `redirect: false`
- bounded `receive_timeout`
- bounded connect timeout
- no Unix socket
- no URL credentials
- no private/loopback/link-local/multicast/metadata targets
- redacted request/response headers
- capped body
- structured transport/error result

The CLI entrypoint is:

```sh
mix allbert.external request --url https://example.com/status --method GET
mix allbert.external request --profile test_echo --path /status
```

The task calls `external_network_request`; it does not call `Req` directly.
It creates a pending confirmation or returns a policy denial.

Resolved confirmation status:

- `approved` with `target_resumed?: true` when the request ran
- `denied` when policy drift or validation prevents resume
- `failed` or approved target metadata, depending on existing confirmation
  status vocabulary chosen during implementation, when the HTTP target ran but
  returned a transport or HTTP failure
- historical `adapter_unavailable` remains valid only for pre-v0.10 resolved
  records or unsupported old target records

### External Service Tests

Use `Req.Test` or the adapter option for all focused tests. Do not rely on live
internet access in the automated suite.

Required cases:

- successful GET to allowed test service profile
- redacted Authorization and Set-Cookie headers
- disabled external services deny
- unallowlisted host deny
- non-HTTP scheme deny
- URL credential deny
- localhost/private/link-local/metadata target deny
- redirect response not followed by default
- retry not attempted by default
- transport timeout becomes structured result
- response body cap and truncation flag
- approval re-check detects changed policy

## Package Install Flow

### Plan First

Package install is split into planning and execution.

`plan_package_install` builds a preview with no install side effect:

- manager
- target root
- exact package specs
- intended save mode: `prod`, `dev`, `optional`, `peer`, or `no-save`
- manager profile: executable, argv prefixes, allowed roots, timeout, output
  cap, and confirmation/lifecycle/git/global posture
- dry-run argv, including npm `--dry-run` and pip preview-only
  `--dry-run --ignore-installed --quiet --report -`
- execution argv preview for npm only
- policy warnings
- expected files or lockfile changes when manager output supports them later

Package specs must be normalized before confirmation. v0.10 default policy
rejects:

- global installs
- path installs
- tarball installs
- URL installs
- git dependencies
- shell metacharacters
- unpinned npm specs; use exact `name@x.y.z` or `@scope/name@x.y.z`
- unpinned pip specs; use exact `name==version` for preview
- lifecycle scripts
- target roots outside allowed roots
- unknown package managers

### npm Execution

`run_package_install` creates a durable confirmation for the allowed npm
profile. The approved path runs npm through v0.08 local runner primitives with
explicit executable/argv, cwd, env allowlist, timeout, and output cap.

Default npm posture:

- target root must be inside `package_installs.allowed_roots`
- no shell string
- no global install
- no git dependencies
- lifecycle scripts disabled
- dry-run preview recorded before execution
- exact package specs required by the default v0.10 policy
- stdout/stderr capped and summarized
- audit entry records argv preview, result, truncation status, and rollback
  note

The default npm argv shape is:

```text
npm install <package@x.y.z> --save-prod --json --save-exact --no-fund --no-audit --ignore-scripts --allow-git=none
```

The plan preview adds `--dry-run`; save mode swaps `--save-prod` for
`--save-dev`, `--save-optional`, `--save-peer`, or `--no-save`.

### pip Preview

pip is preview/audit-only in v0.10 unless implementation also adds and tests:

- `--dry-run --report`
- pinned requirements
- `--require-hashes`
- `--only-binary :all:`
- explicit `--target`
- no direct URL/VCS installs
- no source distributions
- target-root and output policy

If those controls are not present, pip execution returns a clear denial:
"pip execution requires strict hash and binary policy; preview only in v0.10."

### Package Confirmation Metadata

Pending and resolved records include:

- manager
- package specs
- target root
- dry-run summary
- execution argv preview
- execution availability
- timeout
- output cap
- lifecycle script policy
- warnings such as pip preview-only status
- git dependency policy
- package-manager warnings
- target result metadata

### Package Tests

Required cases:

- plan accepted npm package spec
- npm dry-run preview parsed or stored
- npm approved execution through local runner with test fixture executable
- disabled package installs deny
- out-of-root target deny
- global install deny
- URL/git/tarball/path package specs deny
- lifecycle scripts disabled by default
- pip execution denies unless strict policy implemented
- approval re-check detects policy drift
- audit and trace redact environment and output

## Online Skill Search, Audit, And Import Flow

### Search And Detail

`search_online_skills` and `show_online_skill` use allowed source profiles.
For v0.10, the first source is `skills_sh`, backed by Settings Central
`skills.online_import.sources.skills_sh.*` fields and Req test stubs for the
skills.sh-compatible JSON/page shapes.

`skills_sh` is a convenience source profile, not the durable model. Future
sources may include direct `SKILL.md` URLs, skill archives, source manifests,
or other registries. They should all flow through the same remote network
content posture with operation-scoped approval.

Because search and detail are external reads, v0.10 creates durable
confirmations before fetching. Approval resumes the same registered action and
returns the source result in confirmation metadata.

Search/detail output should include, when available:

- source id
- source profile
- skill name
- owner/repository/source URL
- description
- install/ranking metadata
- license
- audit badges or source warnings
- whether the source can be imported automatically
- clear diagnostics when a source profile is disabled or incompatible

Search/detail does not write files and does not enable skills.

### Audit

`audit_online_skill` is also a confirmed external read. After approval it
fetches bounded candidate metadata/files and reports:

- valid `SKILL.md` frontmatter
- name/description/license/compatibility
- resource inventory
- scripts present
- package manifests present
- external links or network instructions
- suspicious credential requests
- source URL
- fetched digest
- duplicate skill name or source conflict
- import eligibility

Audit does not trust, enable, activate, run scripts, or install packages.

### Import

`import_online_skill` requires confirmation before fetching or writing. After
approval it fetches the detail payload, runs the static audit, and writes only
to:

```text
<ALLBERT_HOME>/cache/skills/<source>/<safe-owner-repo>/<safe-skill-name>
```

It also writes source/import metadata under:

```text
<ALLBERT_HOME>/cache/skills/_sources
```

After writing, it runs the existing parser/registry validation. Imported skills
remain:

- disabled
- untrusted
- source-scoped as imported cache
- non-executable
- pending operator review

The import result records target root, manifest path, parser result, audit
result, `enabled?: false`, and `trusted?: false`.

Trust and enablement remain separate Settings Central decisions. Script
execution remains a separate `run_skill_script` flow after trust, enablement,
resource digest checks, and confirmation.

Future direct URL import should not require a source-specific search result.
It should create a `remote_url + import_skill` resource reference for the
operator-provided URL, ask for approval, fetch through the same v0.10 adapter,
then run the existing Agent Skills parser/registry and import audit. The result
remains disabled, untrusted, and non-executable.

### Online Skill Tests

Required cases:

- source disabled deny
- search with Req stub returns redacted results
- detail with Req stub returns candidate metadata
- audit flags scripts/package manifests/external links
- import writes only inside imported cache
- invalid `SKILL.md` denies import
- duplicate skill does not override built-in/project/user skill
- imported skill remains disabled and untrusted
- no `npx`, `git`, package manager, script, or activation side effect occurs
- source-profile search/import and direct URL import use the same remote
  network content confirmation/audit shape once direct URL import exists

Remembered grant operator smoke:

```sh
mix allbert.confirmations approve <confirmation-id> --reason "remember this resource" --remember exact
mix allbert.resources grants list

mix allbert.resources grants revoke <grant-id> --reason "done testing"

```

For actions with multiple resource references, use
`mix allbert.confirmations approve <confirmation-id> --remember exact --remember-all`
when the operator intends to remember every exact resource ref for that
operation. Package install grant reuse requires all current resource refs to
match; a remembered target-root grant alone must not authorize a package
registry/package-spec request.

`/settings` now provides a thin operator surface over the same registered
actions: pending confirmations can be approved once, approved with an exact
remembered resource grant, or approved while remembering all exact refs; the
Remembered Resource Grants section lists active/revoked grants and revokes
through `revoke_resource_grant`. The LiveView does not write Settings Central
grant records directly.

The default `skills_sh` source has `base_url=https://skills.sh` and
`api_url=https://skills.sh/api`; search appends `/search` and sends `q` plus
the configured listing limit. Detail fetches try the source API first and may
fall back to the public skill page.

Approval and target execution are distinct. If the operator approves an online
skill search/detail/audit/import and the configured source then returns an
HTTP, transport, size, or parse failure, the confirmation resolves as
`approved` with `operator_resolution.target_status=failed`. CLI, `/settings`,
audit, and trace renderers should show the `Failure:` reason instead of
collapsing the outcome into `denied`.
