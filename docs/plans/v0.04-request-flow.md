# Allbert v0.04 Request Flow: Jido Runtime Convergence Refactor

This document is the living request-flow map for v0.04 Jido Runtime
Convergence Refactor. It should be updated during implementation as planned
module names become actual module names.

## Status

Status: planned.

v0.04 adds no new execution powers. It makes the existing runtime use one
clear boundary rule before Security Central, confirmations, executable skills,
jobs, and channels expand the system.

## Boundary Rule

All externally invoked, effectful, security-relevant, or observable domain
operations should pass through:

```text
surface or internal request
  -> Jido signal
  -> agent or runtime router
  -> action runner
  -> registered Jido action
  -> plain domain module as needed
  -> action lifecycle signal
  -> trace/audit metadata
```

Plain modules remain valid behind the action boundary for:

- parsing and validation
- schema and YAML encoding
- skill manifest normalization
- settings merge and source explanation
- redaction helpers
- formatting
- storage implementation details

## Current Direct-Call Debt

- `IntentAgent` selects routes but calls action modules directly.
- Settings LiveView and `mix allbert.settings` call `Settings` and
  `Settings.Secrets` directly.
- Trace recording writes through memory internals instead of a trace action or
  internal trace signal.
- Permission checks are inside individual actions; v0.05 Security Central and
  the action runner should make preflight and result metadata consistent.

## Planned Converged Flow

```text
CLI / LiveView / future channel / job
  -> AllbertAssist.Runtime or domain-specific signal ingress
  -> allbert.*.requested signal
  -> internal agent or runtime router
  -> AllbertAssist.Actions.Runner
  -> registered Jido.Action
  -> Settings / Skills / Memory / Trace / Security plain modules
  -> allbert.*.completed signal
  -> trace and audit metadata
```

## Planned Internal Agents

- `SettingsAgent`: settings read/write, credential request, and settings status
  coordination.
- `SkillsAgent`: list/read/activate/diagnostic coordination over the skill
  registry.
- `SecurityAgent`: security decision, status, audit, and refusal coordination
  after v0.05.
- `MemoryAgent` or memory/trace actions: durable memory and trace side effects.

These agents are coordination boundaries. They do not replace pure modules such
as `Settings.Schema`, `Settings.YamlCodec`, `Skills.Parser`, or data structs.

## Verification Sketch

Implementation should verify:

- all user-facing action execution can pass through the shared runner
- action lifecycle metadata is consistent for success, denial, and failure
- direct domain calls remain allowed in pure internals and focused unit tests
- CLI and LiveView behavior is unchanged after convergence
- no shell, script, package, network, or import execution is added
