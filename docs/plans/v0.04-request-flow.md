# Allbert v0.04 Request Flow: Jido Runtime Convergence Refactor

This document is the living request-flow map for v0.04 Jido Runtime
Convergence Refactor. It must be updated during implementation as planned
module names become actual module names.

## Status

Status: implementation in progress. Milestones 1 through 3 are complete.

v0.04 adds no new execution powers. It makes the existing runtime use one
clear boundary rule before Security Central, confirmations, executable skills,
jobs, and channels expand the system.

## Boundary Rule

Runtime-facing requests should pass through:

```text
surface or internal request
  -> Jido signal
  -> agent or runtime router
  -> AllbertAssist.Actions.Runner
  -> registered Jido action
  -> plain domain module as needed
  -> action lifecycle signal
  -> trace/audit metadata
```

Plain modules remain valid behind the action boundary for parsing,
validation, schemas, YAML encoding, skill manifest normalization, settings
merge and source explanation, redaction helpers, formatting, and storage
implementation details.

## Action Registry

Implemented canonical registry:

```text
AllbertAssist.Actions.Registry
```

It exposes:

- all current intent actions
- all current settings actions
- lookup by action module
- lookup by registered action name
- duplicate-name diagnostics for tests

Initial registered action names:

- `direct_answer`
- `append_memory`
- `read_recent_memory`
- `list_skills`
- `read_skill`
- `activate_skill`
- `plan_shell_command`
- `external_network_request`
- `list_settings`
- `read_setting`
- `update_setting`
- `explain_setting`
- `list_provider_profiles`
- `set_provider_credential`

The registry intentionally does not include test-only or private modules such
as `AllbertAssist.Actions.Multiply`. If implementation adds a dedicated
provider credential write action, register it here and keep raw secret output
redacted everywhere.

## Runner Contract

Implemented public boundary:

```elixir
AllbertAssist.Actions.Runner.run(action_or_name, params, context)
```

Inputs:

- `action_or_name`: registered module atom or registered action name string
- `params`: map passed to the action
- `context`: runtime context with request, agent, channel, operator,
  input-signal, selected-skill, and trace metadata when available

Outputs:

- `{:ok, response}` for action success, denial, or confirmation-needed results
- structured `{:ok, response}` error results for action exceptions, invalid
  action returns, unknown action names, and unregistered modules

Runner metadata attached to the top-level response and each
`response.actions` entry:

- `runner_action_id`
- `requested_signal_id`
- `completed_signal_id`
- `action_name`
- `action_module`
- `status`
- `duration_ms`
- `permission_decision` when present
- `selected_skill` when present
- `error` when present

Unknown action behavior:

- unknown registered name: return a structured denial result
- unregistered module: return a structured denial result
- no dynamic module loading
- no action execution from arbitrary skill metadata

## Lifecycle Signals

v0.04 uses the existing Allbert signal vocabulary and makes it consistent:

- `allbert.action.requested`
- `allbert.action.completed`

Requested signal data:

- action name
- action module
- params summary with secret redaction
- source signal id when present
- channel and operator id when present
- selected skill when present

Completed signal data:

- action name
- action module
- status
- duration
- permission/security decision when present
- response summary with secret redaction
- error reason when present and sanitized

v0.04 may keep current log-only signal handling. It should not add a new
durable event store or broad bus dispatch semantics.

Implemented helper:

```text
AllbertAssist.Signals
```

The helper constructs Jido signals with `Jido.Signal.new/3`, recursively
redacts sensitive key names such as API keys, secrets, tokens, passwords, and
credentials, and logs the lifecycle signal summary. It does not persist or
dispatch durable events in v0.04.

## Flow: Normal User Prompt

```text
CLI / Agent LiveView
  -> AllbertAssist.Runtime.submit_user_input/1
  -> allbert.input.received
  -> IntentAgent route selection
  -> Actions.Runner.run(selected_action, params, context)
  -> allbert.action.requested
  -> registered action
  -> plain domain module if needed
  -> allbert.action.completed
  -> allbert.agent.responded
  -> trace recording boundary
```

Implementation note: `IntentAgent` may keep deterministic route predicates in
v0.04, but route execution must go through the runner.

Implemented note: `AllbertAssist.Agents.IntentAgent.action_modules/0`
delegates to `AllbertAssist.Actions.Registry.modules/0`, and every
deterministic route now calls `AllbertAssist.Actions.Runner.run/3` instead of
calling action modules directly. The runner context carries the original
request, agent, selected route, selected action, source text, input signal, and
selected skill when present.

## Flow: Settings CLI

```text
mix allbert.settings ...
  -> structured settings command parser
  -> Actions.Runner.run(settings_action, params, context)
  -> allbert.action.requested
  -> registered settings action
  -> Settings / Settings.Secrets plain modules
  -> allbert.action.completed
  -> CLI renderer
```

The Mix task may parse command-line arguments and render action results. It
must not own settings validation, settings writes, secret writes, redaction, or
provider profile semantics.

## Flow: Settings LiveView

```text
/settings event
  -> event params normalization
  -> Actions.Runner.run(settings_action, params, context)
  -> allbert.action.requested
  -> registered settings action
  -> Settings / Settings.Secrets plain modules
  -> allbert.action.completed
  -> assign updated action result
```

The LiveView may own form state and display. It must not write settings or
secrets directly.

## Flow: Skill List, Read, And Activate

```text
prompt or surface skill request
  -> action selection
  -> Actions.Runner.run(skill_action, params, context)
  -> registered skill action
  -> Skills registry plain modules
  -> action lifecycle metadata
  -> response / trace
```

The registry and parser stay plain Elixir. Skill activation remains
progressive-disclosure context only; it does not execute scripts or action
contracts.

## Flow: Trace Recording

```text
runtime response
  -> trace recording action or internal trace signal
  -> Trace plain module
  -> Memory plain module for markdown persistence
  -> allbert.trace.recorded
```

Trace persistence must become observable runtime work, but the markdown memory
implementation remains plain Elixir.

## Direct-Call Debt To Remove

Implementation should remove or isolate these runtime-facing direct calls:

- `AllbertAssist.Agents.IntentAgent` direct `SomeAction.run/2` calls in route
  execution.
- `AllbertAssistWeb.SettingsLive` direct `Settings.*` and
  `Settings.Secrets.*` calls.
- `Mix.Tasks.Allbert.Settings` direct `Settings.*` and
  `Settings.Secrets.*` calls.
- `AllbertAssist.Runtime` direct durable trace writes through `Trace.record_turn/1`
  without an action/signal boundary.

Allowed direct calls remain:

- actions to plain domain modules
- pure module tests
- parser, schema, codec, storage helper, and struct code
- setup, migration, seed, and bootstrap code

## Baseline Behavior Lock

Milestone 1 freezes the current behavior before runtime code movement. The
implementation must preserve:

- deterministic intent routing and current user-visible messages
- settings CLI list/get/explain/set/provider flows
- Settings LiveView list/update/provider key flows
- skill list/read/activate behavior from the v0.03 registry
- memory append/read behavior and markdown layout
- trace write behavior and trace settings
- command execution denial
- external network confirmation-needed behavior
- raw secret redaction in inspectable surfaces

## Verification Matrix

Automated verification should cover:

- runner invokes each registered action
- runner rejects unknown names and unregistered modules
- lifecycle metadata appears for completed, denied, confirmation-needed, and
  error outcomes
- `IntentAgent` has no direct action-module execution path
- settings CLI and Settings LiveView write through actions
- provider credential writes stay encrypted and redacted
- trace recording uses the new boundary
- skill activation remains non-executable
- command execution remains denied
- external network remains confirmation-needed and unexecuted

Operator verification should use a temporary `ALLBERT_HOME` and exercise:

- `mix allbert.ask --trace "hello"`
- `mix allbert.ask --trace "what skills are available?"`
- `mix allbert.ask --trace "activate skill append-memory"`
- `mix allbert.ask --trace "remember that I prefer concise milestone handoffs"`
- `mix allbert.ask --trace "run rm -rf /tmp/example"`
- `mix allbert.ask --trace "fetch https://example.com"`
- `mix allbert.settings list`
- `mix allbert.settings set operator.communication_style concise`
- `/agent`
- `/settings`

Release readiness checks:

```sh
rg -n "AllbertAssist\\.Actions\\..*\\.run\\(" apps/allbert_assist/lib/allbert_assist/agents
rg -n "Settings\\.|Settings\\.Secrets\\." apps/allbert_assist_web/lib/allbert_assist_web/live/settings_live.ex apps/allbert_assist/lib/mix/tasks/allbert.settings.ex
git diff --check
mix precommit
```

The `rg` checks should either return no prohibited direct runtime calls or only
documented exceptions.

v0.04 should not be considered release-ready until the automated gate,
temporary-home operator smoke tests, redaction checks, trace inspection, docs
alignment, and release metadata checks have all passed.
