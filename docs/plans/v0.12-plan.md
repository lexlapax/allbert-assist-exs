# Allbert v0.12 Local Workspace Identity And Conversation History

Formerly tracked as M-D1a.

## Purpose

v0.12 adds the local workspace identity and conversation-history foundation
needed for Allbert to become a multi-app personal workspace without adopting a
hosted accounts model. It introduces canonical string `user_id`,
SQLite-backed conversation threads/messages, and CLI/runtime thread selection
while preserving existing single-user behavior.

This milestone keeps the implementation centered on the runtime, CLI, traces,
signals, tests, and the v0.11 intent/approval contracts. It does not add a new
LiveView thread sidebar, session scratchpad, app routing, semantic retrieval,
or memory-review workflow. Richer web thread UX belongs to later workspace
surface work.

## Expected Inputs

- v0.11 is released and tagged. Runtime responses, traces, decisions,
  resource-access entries, and Approval Handoff metadata exist and reserve
  `user_id`, `thread_id`, `session_id`, and `active_app`.
- v0.10/v0.11 risky execution behavior remains in force: URL/document/import,
  package, shell, skill-script, MCP, and agent URI prompts still produce
  inspectable decisions and Approval Handoff before any effect.
- SQLite via `ecto_sqlite3` is the project database, and app/test aliases
  already run `ecto.create` and `ecto.migrate`. v0.12 implementation should
  verify that manual CLI smoke can isolate the local database with
  `DATABASE_PATH`/`ALLBERT_HOME` so it does not touch a real operator database.
- `AllbertAssist.Runtime.submit_user_input/1` accepts atom and string keys and
  already treats `:user_id` as a fallback alias for `:operator_id`.
- ADR 0014 local workspace identity is accepted before implementation begins.

## Roadmap And Vision Fit

v0.12 is the first durable workspace-context milestone after execution-aware
intent. It should make Allbert remember conversation turns as ordered,
queryable local history while keeping markdown memory distinct and
operator-owned.

This milestone prepares later work without implementing it:

- v0.13 scheduled jobs can preserve originating `user_id` and `thread_id`
  when a job is created from an interactive request.
- v0.14 session scratchpad can key volatile state by `{user_id, session_id}`
  without changing thread storage.
- v0.15/v0.24 app contracts can add app-specific routing and surfaces later;
  v0.12 stores only nil/general `app_id`.
- v0.18 memory review can selectively promote lessons from threads into
  markdown memory as explicit confirmed actions; v0.12 never promotes turns
  automatically.
- v0.23 security evals can test cross-user/thread leakage against real
  persisted conversation records.

The implementation should follow the Allbert/Jido vision: input becomes a
signal, the runtime normalizes context, the intent agent receives bounded
recent history, registered actions still own effects, and traces/messages make
the turn inspectable.

## v0.11 Handoff

v0.12 plugs identity and thread context into the v0.11 decision/runtime shape.
It must preserve the v0.11 behavior that risky URL/document/import/package/
script prompts produce inspectable decisions and Approval Handoff before
execution. New thread storage must not auto-promote conversation turns to
markdown memory and must not change operation-scoped grant matching.

When a runtime turn returns `:needs_confirmation`, the assistant message should
record the pending state, confirmation id, resource-access entries, and
Approval Handoff summary. Approval or denial still resolves through registered
confirmation actions; v0.12 does not make thread storage a new approval path.

## Implementation Boundaries

Authoritative existing boundaries:

- Runtime entrypoint: `AllbertAssist.Runtime.submit_user_input/1`.
- Intent routing: `AllbertAssist.Agents.IntentAgent.respond/1`.
- v0.11 contracts: `AllbertAssist.Intent.Decision`,
  `AllbertAssist.Intent.ResourceAccess`, and
  `AllbertAssist.Intent.ApprovalHandoff`.
- Tracing: `AllbertAssist.Trace` and the `record_trace` action.
- Markdown memory: `AllbertAssist.Memory`; v0.12 must not store conversation
  turns there.
- Database: `AllbertAssist.Repo` with SQLite migrations under
  `apps/allbert_assist/priv/repo/migrations`.

New conversation-history code should live outside the markdown-memory module so
the distinction stays obvious. Preferred shape:

- `AllbertAssist.Conversations` context for thread selection, creation,
  ownership checks, persistence, and bounded context reads.
- `AllbertAssist.Conversations.Thread` Ecto schema.
- `AllbertAssist.Conversations.Message` Ecto schema.
- `Mix.Tasks.Allbert.Threads` for operator CLI inspection.

Thread storage is local context, not authorization. User isolation checks
prevent accidental cross-user reads in local UX and tests, but `user_id` is not
hosted auth and does not replace Security Central.

All durable conversation data must use the configured `AllbertAssist.Repo`.
Manual/dev smoke should be able to point the repo at a disposable SQLite file
under `ALLBERT_HOME` or `DATABASE_PATH`; tests should continue to use the SQL
sandbox and temp-specific roots.

## Data Model

Create SQLite tables with string public ids:

- `conversation_threads`
  - `id`: string primary key, generated as an opaque `thr_...` id.
  - `user_id`: string, required, indexed.
  - `title`: string, generated from the first user message and editable later.
  - `kind`: string, required, default `"general"`.
  - `app_id`: string, nullable; nil means general Allbert context in v0.12.
  - `last_message_at`: UTC datetime, updated whenever a message is appended.
  - timestamps.
- `conversation_messages`
  - `id`: string primary key, generated as an opaque `msg_...` id.
  - `thread_id`: string foreign key to `conversation_threads.id`.
  - `user_id`: string, required, duplicated for isolation queries.
  - `role`: string, required; allowed values are `"user"` and `"assistant"`.
  - `content`: text, required.
  - `action_log`: map, default `%{}`; redacted summary of actions, decision,
    resource access, approval handoff, status, and diagnostics.
  - `trace_id`: string, nullable.
  - `input_signal_id`: string, nullable.
  - `response_signal_id`: string, nullable.
  - `metadata`: map, default `%{}` for bounded non-secret context.
  - inserted timestamp.

Indexes:

- threads by `{user_id, app_id, kind, last_message_at}` for recent-thread
  selection.
- messages by `{thread_id, inserted_at, id}` for ordered reads.
- messages by `{user_id, inserted_at}` for user-scoped diagnostics/tests.

Do not add an accounts table, account foreign key, role table, team table, or
hosted ownership model.

## Runtime Request Contract

Normalize runtime requests with these rules:

- `user_id`: explicit `:user_id`/`"user_id"`, else `:operator_id`/
  `"operator_id"`, else `"local"`.
- `operator_id`: compatibility alias. If only `user_id` is present, set
  `operator_id` to the same value for legacy code. If only `operator_id` is
  present, set `user_id` to the same value. If both are present and differ,
  return a validation error before emitting signals or writing messages.
- `thread_id`: explicit thread id, selected recent general thread for the
  user, or newly created general thread.
- `new_thread`: boolean; when true, always create a new general thread and
  reject simultaneous explicit `thread_id`.
- `session_id`: accepted and propagated as optional context, but not
  interpreted until v0.14.
- `app_id`: nil/general only in v0.12; app-specific routing waits for the app
  contract milestones.

Runtime responses must include `user_id`, `operator_id`, `thread_id`, and
`session_id` when present, alongside the existing v0.11 response fields.
Input/response signals, decision context, trace context, confirmation origin
metadata when available, and message action logs should carry `user_id` and
`thread_id`.

## Thread Selection Semantics

When no `thread_id` is supplied:

1. Resolve the canonical `user_id`.
2. Look up the most recently updated thread where `user_id` matches,
   `kind == "general"`, and `app_id` is nil.
3. Order by `last_message_at`, then `updated_at`, then `inserted_at`, newest
   first.
4. Create a general thread if none exists.

When `--new-thread` or `new_thread: true` is supplied:

1. Reject any simultaneous explicit `thread_id`.
2. Create a new general thread for the resolved user.
3. Use the first user message to derive the title, truncated to a concise
   single-line summary.

When an explicit `thread_id` is supplied:

1. Fetch the thread scoped by both `thread_id` and `user_id`.
2. If not found, return `{:error, {:thread_not_found, thread_id}}` or an
   equivalent CLI error that does not reveal whether another user owns it.
3. Continue only if the thread belongs to the resolved user.

## Message Persistence Semantics

- Persist the user message after request normalization and thread selection,
  before invoking the intent agent.
- Pass bounded recent prior context to the intent agent. The current prompt
  remains `request.text`; the context should contain up to the last 12 earlier
  messages in chronological order.
- Persist the assistant message after the runtime response and trace id are
  known. If tracing is disabled, store `trace_id: nil`.
- Assistant `action_log` should include redacted `status`, `actions`,
  `decision`, `resource_access`, `approval_handoff`, `diagnostics`,
  `input_signal_id`, and `response_signal_id`.
- If the agent returns an error before a response exists, keep the user message
  and do not invent an assistant message. If the runtime returns a structured
  error response later, that later behavior should be documented in the
  implementation PR before adding it.
- Update `conversation_threads.last_message_at` when each message is appended.

## Intent-Agent Context Contract

The runtime should call the intent agent with structured conversation context,
not by silently rewriting the user prompt. Preferred shape:

```elixir
%{
  text: request.text,
  channel: request.channel,
  user_id: request.user_id,
  operator_id: request.operator_id,
  thread_id: request.thread_id,
  session_id: request.session_id,
  thread_context: %{
    thread_id: request.thread_id,
    user_id: request.user_id,
    limit: 12,
    messages: [
      %{role: "user", content: "...", inserted_at: "...", trace_id: nil},
      %{role: "assistant", content: "...", inserted_at: "...", trace_id: "..."}
    ]
  },
  metadata: request.metadata
}
```

The intent agent may use this context for direct answers and decision
selection, but v0.12 must not add embeddings, vector search, summarization,
semantic recall, or memory distillation.

## CLI Contract

Update `mix allbert.ask`:

- Add `--user USER`.
- Preserve `--operator OPERATOR`.
- Add `--thread THREAD_ID`.
- Add `--new-thread`.
- If `--user` and `--operator` are both present and differ, fail fast before
  runtime submission.
- If `--thread` and `--new-thread` are both present, fail fast before runtime
  submission.
- Render `User: USER_ID` and `Thread: THREAD_ID` in the result, while
  preserving current status, message, signal, trace, Approval Handoff, and
  action rendering.

Add `mix allbert.threads`:

- `mix allbert.threads --user alice` lists Alice's threads.
- `mix allbert.threads --user alice --thread THREAD_ID` shows ordered messages
  for that thread.
- Omitted `--user` defaults to `"local"`.
- `--operator` is accepted as a compatibility alias with the same conflict
  rule as `mix allbert.ask`.
- `--limit N` limits list/show output with a conservative default.
- Showing a thread owned by another user returns not found or rejected without
  leaking the owner.

## Milestones

### Milestone 1: Conversation Data Model

Status: implemented.

Goal: add durable SQLite conversation history without touching runtime
behavior.

Implementation expectations:

- Add migrations, schemas, changesets, and the `AllbertAssist.Conversations`
  context.
- Ensure local CLI/manual verification can use a disposable SQLite database via
  `DATABASE_PATH`/`ALLBERT_HOME`; do not write v0.12 smoke data into a real
  operator database.
- Generate opaque `thr_...` and `msg_...` ids in application code.
- Add functions for create/select recent thread, fetch user-scoped thread,
  append user message, append assistant message, list user threads, show
  ordered messages, and load bounded recent prior context.
- Keep all tests on temporary test databases and temporary Allbert homes.

### Milestone 2: Runtime Identity Normalization And Thread Selection

Status: implemented.

Goal: normalize local identity and thread context before signals/agent work.

Implementation expectations:

- Extend runtime request normalization with canonical `user_id`,
  compatibility `operator_id`, optional `thread_id`, optional `session_id`,
  and `new_thread`.
- Enforce conflict errors for differing `user_id`/`operator_id` and
  simultaneous `thread_id`/`new_thread`.
- Select or create the correct general thread before input signal emission.
- Include `user_id`, `operator_id`, `thread_id`, and optional `session_id` in
  input signals and runtime responses.

### Milestone 3: Message Persistence And Bounded Context

Goal: persist runtime turns and feed bounded recent history into intent.

Implementation expectations:

- Persist user messages before intent-agent invocation.
- Load up to 12 prior messages in chronological order and pass them as
  structured `thread_context`.
- Persist assistant messages after response and trace metadata are known.
- Store redacted action/decision/resource/approval metadata in `action_log`.
- Preserve v0.11 Approval Handoff and confirmation behavior unchanged.

### Milestone 4: CLI Ask And Threads Surface

Goal: make v0.12 testable and usable from the CLI without adding web UI.

Implementation expectations:

- Add `--user`, `--thread`, and `--new-thread` to `mix allbert.ask`.
- Preserve `--operator` compatibility and conflict behavior.
- Render user/thread context in ask output.
- Add `mix allbert.threads` list/show behavior with user-scoped isolation.
- Do not add an `AgentLive` sidebar, canvas component, or session UI.

### Milestone 5: Trace, Signal, And Handoff Hardening

Goal: make conversation context visible in observability while preserving
security boundaries.

Implementation expectations:

- Add user/thread context to trace summaries/body and trace action context.
- Ensure v0.11 decisions receive `user_id` and `thread_id`.
- Ensure confirmation-required turns persist an assistant message with pending
  status and handoff metadata, without changing approval execution.
- Add cross-user/thread negative tests for runtime continuation and threads
  CLI.
- Confirm no markdown memory writes occur from ordinary conversation turns
  apart from the existing explicit trace recorder when tracing is enabled.

### Milestone 6: Release Readiness And User Testing Docs

Goal: close v0.12 with manual verification steps and future handoffs.

Implementation expectations:

- Update README, CHANGELOG, roadmap, v0.12 plan/request-flow status, and
  v0.13/v0.14/v0.18/v0.23 handoff notes as needed.
- Include manual tests for default `"local"` behavior, Alice/Bob isolation,
  recent-thread selection, explicit thread continuation, `--new-thread`,
  conflicting `--user`/`--operator`, conflicting `--thread`/`--new-thread`,
  trace metadata, and confirmation-required turns.
- State release/tag readiness and the operator manual verification path.

## Acceptance

- `mix allbert.ask --user alice --new-thread "hello"` creates a thread and
  records the user and assistant messages.
- A follow-up `mix allbert.ask --user alice "what did I just say?"` uses
  Alice's selected/recent thread context.
- `mix allbert.ask "hello"` defaults to user `"local"` and the local recent
  general thread.
- `mix allbert.ask --operator alice "hello"` remains accepted and maps to
  `user_id: "alice"`.
- `mix allbert.ask --user alice --operator bob "hello"` fails before runtime
  submission.
- `mix allbert.ask --user alice --thread THREAD_ID "continue"` continues only
  Alice-owned threads.
- `mix allbert.threads --user alice` lists Alice's threads; showing a Bob
  thread as Alice is rejected or returns not found.
- Runtime responses, input/response signals, traces, decisions, and persisted
  assistant action logs include `user_id` and `thread_id`.
- Risky URL/document/import/package/script prompts still produce v0.11
  decisions and Approval Handoff before execution.
- No `AgentLive` thread sidebar, canvas/session UI, scratchpad, app routing,
  semantic retrieval, or markdown-memory promotion is included.

## Non-Goals

- No hosted accounts, authentication, roles, teams, or account foreign keys.
- No ETS scratchpad; that is v0.14.
- No `active_app` routing or app-specific thread UX.
- No LiveView thread-management UI.
- No embeddings, vector search, semantic recall, summaries, or structured fact
  extraction.
- No automatic markdown memory promotion from conversation turns.
- No changes to resource grant matching, approval execution, shell/package/
  network policy, or Security Central authorization semantics.
- No thread title editing CLI. Thread titles are schema-editable for future
  milestones but no `mix allbert.threads rename` or equivalent ships in v0.12.

## Test Plan

- Schema/migration tests prove thread and message round-trips, indexes support
  recent selection, and invalid roles/blank users are rejected.
- Conversations context tests cover recent-thread selection, explicit
  user-scoped fetch, new-thread creation, ordered messages, bounded context,
  and cross-user not-found behavior.
- Runtime tests cover omitted `thread_id`, explicit `thread_id`, `new_thread`,
  conflict errors, user message before agent, assistant message after response,
  trace id persistence, and no assistant message on pre-response agent error.
- Intent-agent/runtime integration tests prove `thread_context` reaches the
  agent and v0.11 decisions include `user_id` and `thread_id`.
- CLI tests cover `--user`, `--operator`, conflict errors, `--thread`,
  `--new-thread`, rendered user/thread output, and `mix allbert.threads`
  list/show isolation.
- Confirmation-required prompt tests prove pending Approval Handoff metadata is
  persisted in assistant `action_log` without auto-approval.
- Markdown memory tests prove ordinary conversation turns do not create notes,
  preferences, skills, or trace-memory entries beyond the existing explicit
  trace recorder when tracing is enabled.
- Focused runtime, intent-agent, trace, CLI, and Mix task tests pass.
- Final code gates follow `DEVELOPMENT.md`: `mix compile --warnings-as-errors`,
  `mix format --check-formatted`, `mix credo --strict`, `mix dialyzer`, and
  `mix precommit`.
