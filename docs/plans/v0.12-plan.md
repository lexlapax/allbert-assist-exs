# Allbert v0.12 Local Workspace Identity And Conversation History

Formerly tracked as M-D1a.

## Purpose

v0.12 adds the local workspace identity and conversation-history foundation
needed for Allbert to become a multi-app personal workspace without adopting a
hosted accounts model. It introduces string `user_id`, SQLite-backed
conversation threads/messages, and CLI/runtime thread selection while
preserving existing single-user behavior.

This milestone keeps the implementation centered on the runtime, CLI, traces,
signals, and tests. It does not add a new LiveView thread sidebar; richer web
thread UX belongs to later workspace surface work.

## Expected Inputs

- v0.10 is released and SQLite via `ecto_sqlite3` is the project database.
- v0.11 reserves `user_id`, `thread_id`, `session_id`, and `active_app` in the
  intent decision shape.
- `AllbertAssist.Runtime.submit_user_input/1` already accepts `user_id` as a
  fallback alias for `operator_id`.
- ADR 0014 local workspace identity is accepted before implementation begins.

## Expected Direction

- Add `AllbertAssist.Memory.Thread` and `AllbertAssist.Memory.Message` Ecto
  schemas and migrations.
- Use string `user_id`; do not add `AllbertAssist.Accounts.User`, auth, roles,
  or account foreign keys.
- Add `thread_id`, `user_id`, and optional `session_id` to normalized runtime
  requests, input/response signal data, trace context, and runtime responses.
- If no `thread_id` is passed, select the user's most recently updated general
  thread, or create one if none exists. `--new-thread` always creates a new
  thread.
- Persist user messages before the agent runs. Persist assistant messages
  after response and trace metadata are known.
- Pass bounded recent thread context to the intent agent, initially the last 12
  messages. Do not add semantic/vector retrieval.
- Preserve `--operator`; add `--user`; if both are present and differ, fail
  fast instead of silently choosing one.
- Add `mix allbert.threads` for listing user threads and showing thread
  messages.
- Store conversation history only in SQLite. Do not promote conversation turns
  into markdown memory.

## Acceptance

- `mix allbert.ask --user alice --new-thread "hello"` creates a thread and
  records the user and assistant messages.
- A follow-up `mix allbert.ask --user alice "what did I just say?"` uses the
  selected/recent thread context.
- `mix allbert.ask "hello"` still defaults to user `"local"`.
- `mix allbert.threads --user alice` lists Alice's threads; showing a Bob
  thread as Alice is rejected or returns not found.
- Runtime responses include `user_id` and `thread_id`.
- No `AgentLive` thread sidebar or canvas/session UI work is included.

## Non-Goals

- No hosted accounts, authentication, roles, teams, or account foreign keys.
- No ETS scratchpad; that is v0.14.
- No LiveView thread-management UI.
- No embeddings, vector search, semantic recall, or structured fact
  extraction.
- No automatic markdown memory promotion from conversation turns.

## Test Plan

- Thread and message schemas round-trip through the repo.
- Runtime creates/selects threads correctly for omitted `thread_id`,
  explicit `thread_id`, and `--new-thread`.
- User and assistant messages persist in order with trace id on the assistant
  message when tracing is enabled.
- `--operator` compatibility remains; conflicting `--operator` and `--user`
  fails fast.
- Alice/Bob isolation is enforced in `mix allbert.threads` and runtime thread
  continuation.
- Focused runtime, intent-agent, trace, CLI, and Mix task tests pass.
- Final code gates follow `DEVELOPMENT.md`.
