# Allbert v0.23 Request Flow — Jido State-Machine Convergence

## Status

Status: implemented through M5 closeout on 2026-05-16. Operator manual
verification and release tagging remain pending.

v0.23 does not add a new operator feature. It changes the internal substrate
for confirmations and scheduled jobs while preserving the request flows that
already exist. **Operator-visible behavior is preserved from v0.22** for the
canonical scenarios covered by the focused M1/M2 regression suites and the
full closeout gate.

This document describes the runtime/user flows that must continue to work
after implementation. Implementation details (module shapes, settings
schema, byte-parity testing, milestones) live in `docs/plans/v0.23-plan.md`;
this file stays focused on what an operator or runtime caller observes and
on the substrate posture that makes the convergence safe.

## Pre-Flight: Allbert DB Migrations And Substrate Posture

Before running smoke flows, use a disposable Allbert Home and initialize the
local SQLite database (jobs/runs persist through migrations; confirmations
live as YAML files under Allbert Home):

```sh
export SMOKE_HOME=$(mktemp -d /tmp/allbert-v023-smoke.XXXXXX)
ALLBERT_HOME="$SMOKE_HOME" mix cmd --app allbert_assist mix ecto.migrate.allbert
```

The confirmation records in these flows live under
`$SMOKE_HOME/confirmations`. Scheduled jobs and runs live in the local SQLite
database for that Allbert Home.

### Substrate Reference (operator-facing summary)

v0.23 reshapes two internal coordinators into Jido.Agent-backed state
machines using a shared `AllbertAssist.JidoBacked` behaviour. Operators
do not see this directly under default settings; the substrate becomes
visible only through an explicit debug setting:

| Operator switch | Effect |
|---|---|
| `mix allbert.settings set allbert.jido.debug_trace true` | Trace markdown gains a bounded top-level `## Jido Debug` subsection for configured JidoBacked agents. Default is `false` (no operator-visible change). |

This switch changes only trace diagnostic volume. It does not change
confirmation, job, audit, signal, or permission behavior. Default-off keeps
the operator surface identical to v0.22.

### Authority Posture (unchanged)

All effectful work still flows through registered actions →
`Actions.Runner.run/3` → Security Central → confirmations (when needed) →
audit. The JidoBacked agents coordinate internal state but never grant
permission, change confirmation policy, or skip audit. ADR 0007, ADR 0008,
ADR 0021 Section 4 (authority rule) all hold.

## Flow 1: Confirmation Lifecycle

1. The operator asks for work that requires confirmation, such as a URL
   summary or another risky registered action.
2. The runtime selects a registered action and calls
   `AllbertAssist.Actions.Runner.run/3`.
3. Security Central returns `:needs_confirmation`.
4. The target action creates a durable confirmation record and does not
   perform the target side effect.
5. The confirmation store writes the pending YAML record and audit entry
   first, then updates its rebuildable in-memory projection.
6. `allbert.confirmation.requested` signal is emitted with the same payload
   contract as v0.07.
7. The operator runs `mix allbert.confirmations list` and sees the pending
   request.
8. The operator runs `mix allbert.confirmations show <id>` and sees the
   same redacted target summary, origin metadata, permission, risk, and
   approval instructions as before v0.23.
9. The operator approves or denies through
   `mix allbert.confirmations approve|deny`. The store re-reads the pending
   YAML, writes the resolved record, removes the pending file, appends audit,
   then updates its projection.
10. The corresponding signal (`allbert.confirmation.approved` or
    `allbert.confirmation.denied`) is emitted with the v0.07 payload
    contract preserved byte-for-byte.
11. The resolved record moves to the resolved confirmation area and audit
    output is appended.

Expected operator result: identical confirmation status, display fields,
audit shape, and trace behavior to v0.22. The M1 regression suite asserts
this, with retained v0.23 confirmation audit fixtures under
`apps/allbert_assist/test/fixtures/v0.23/confirmations/`.

Smoke:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.ask "summarize https://example.com"
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations list
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations show <id>
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations approve <id> --reason "v0.23 smoke"
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations list --resolved
```

## Flow 2: Confirmation Expiry

1. The operator or scheduler creates a confirmation-required request.
2. The confirmation reaches its configured TTL.
3. The operator runs `mix allbert.confirmations expire`.
4. The store agent's `Expire` command enumerates pending files and resolves
   expired records as expired; target work is not performed.
5. `allbert.confirmation.expired` signal fires per expired record.
6. `mix allbert.confirmations list --resolved` shows the expired status.

Expected operator result: same expiry semantics and audit shape as v0.22.

Smoke:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.settings set confirmations.default_ttl_minutes 1
ALLBERT_HOME="$SMOKE_HOME" mix allbert.ask "summarize https://example.com"
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations expire
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations list --resolved
```

## Flow 3: Scheduled Job Lifecycle

1. The operator creates a job through `mix allbert.jobs create ...`.
2. The job is persisted in SQLite (unchanged from v0.13).
3. The operator lists/shows the job through `mix allbert.jobs`.
4. The scheduler tick reads due jobs from SQLite, claims due jobs durably,
   and dispatches each run through the existing runner path.
5. A manual or scheduled run claims a durable `scheduled_job_runs` row.
6. The job target enters the same runtime/action path as an interactive
   request.
7. The run finishes as `completed`, `failed`, `skipped`, or
   `needs_confirmation`.
8. The operator inspects run history with `mix allbert.jobs runs
   <job-id>`.
9. Pause/resume/cancel behavior remains unchanged.

Expected operator result: same job statuses, run records, CLI output, and
trace metadata as v0.22/v0.13. The M2 regression suite asserts this, with
retained v0.23 scheduler summary fixtures under
`apps/allbert_assist/test/fixtures/v0.23/jobs/`.

Smoke:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs create runtime-prompt memory-list \
  --prompt "list my memory entries" --manual --user local
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs list --user local
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs run <job-id>
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs runs <job-id>
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs pause <job-id>
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs resume <job-id>
```

## Flow 4: Risky Scheduled Job

1. The operator creates a scheduled or manual job whose target requires
   confirmation.
2. The scheduler/manual run claims the run row.
3. The target action returns `:needs_confirmation`.
4. The job is marked blocked or needs-confirmation according to existing
   v0.13 behavior. `allbert.job.needs_confirmation` signal fires with
   the v0.13 payload.
5. The operator approves or denies through the confirmation CLI.
6. Approval resumes through registered confirmation actions; the job does
   not self-approve.

Expected operator result: same blocked-job and confirmation handoff
behavior as v0.13/v0.22.

Smoke:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs create runtime-prompt risky-url \
  --prompt "summarize https://example.com" --manual --user local
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs run <job-id>
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations list
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations approve <id> --reason "v0.23 job smoke"
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs show <job-id>
```

## Flow 5: Restart And Rehydration

1. The operator creates a pending confirmation and a job.
2. The supervised application is restarted (or just the
   `JidoBacked.Supervisor`).
3. The store agent's `Rebuild` command runs on startup and reconstructs
   the projection from `confirmations/pending/*.yml`.
4. The scheduler agent's startup `CleanupStaleRuns` runs, then a
   `Tick` is scheduled per `jobs.scheduler.poll_on_start`.
5. The operator lists confirmations and jobs again.
6. Pending confirmations reappear from Allbert Home files.
7. Jobs and run history reappear from SQLite.

Expected operator result: no pending confirmation or scheduled job
disappears because process memory was lost. In-memory state is never
authoritative.

Smoke:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations list --all
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs list --user local
# Restart the Mix/iex process or supervised app.
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations list --all
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs list --user local
```

## Flow 7: Debug Trace Gate (operator-visible substrate window)

1. Operator opts in: `mix allbert.settings set allbert.jido.debug_trace true`.
2. Operator runs any turn (`mix allbert.ask --trace "..."`).
3. Trace markdown now contains a bounded top-level `## Jido Debug`
   subsection for configured JidoBacked agents.
4. Operator opts out: `mix allbert.settings set allbert.jido.debug_trace false`.
5. Next trace contains no `## Jido Debug` section; rest of trace
   byte-identical to v0.22 for the same input.

Expected operator result: a low-friction window into JidoBacked internals
without changing any other behavior. No raw secrets, no full record bodies,
no operator-meaningful field changes.

Smoke:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.settings set allbert.jido.debug_trace true
ALLBERT_HOME="$SMOKE_HOME" mix allbert.ask --trace "list my memory entries"
# Inspect trace; ## Jido Debug subsection present.
ALLBERT_HOME="$SMOKE_HOME" mix allbert.settings set allbert.jido.debug_trace false
ALLBERT_HOME="$SMOKE_HOME" mix allbert.ask --trace "list my memory entries"
# Inspect trace; no ## Jido Debug subsection.
```

## Channel Context Preservation Guarantee

v0.07 defined the origin/resolver channel separation: confirmation records
carry both the origin (where the request entered) and the resolver (where
the approval was issued). v0.23 must preserve this contract:

- Confirmation creation populates `origin.{actor, channel, surface,
  session_id, thread_id}` from the same context fields as v0.22.
- Confirmation resolution populates
  `operator_resolution.{resolver_actor, resolver_channel, resolver_surface,
  resolver_session_id}` from resolver context.
- `same_channel?` flag (computed at resolve time) preserves v0.07
  semantics.
- Focused confirmation regression coverage asserts same-channel/cross-channel
  metadata shape and byte-stable audit fields for the canonical paths.

## Redaction Rules

v0.23 inherits all v0.07 + v0.22 redaction obligations:

- `AllbertAssist.Security.Redactor` runs against any payload before signal
  emission, before audit YAML write, and before CLI/trace rendering.
- The v0.22 third-validation closeout expanded
  `sensitive_key_fragments` to cover `raw_bridge`, `raw_final`,
  `raw_response`, `authorization`, `bearer`. v0.23 does not change this
  set.
- JidoBacked debug-trace output passes through the same Redactor before
  rendering the `## Jido Debug` subsection.
- Audit YAML for resolved confirmations preserves v0.07 redaction
  posture; M1 retained fixtures and focused regression assertions cover this.

## Failure And Safety Cases

| Case | Expected behavior |
| --- | --- |
| Confirmation agent restarts with pending YAML files | Pending records remain listable and approvable via projection rebuild from disk. |
| Confirmation pending file is malformed (invalid YAML, missing required field) | List/expire skips with bounded error log; app keeps running; per-file error is bounded in the log and never propagated to the operator's CLI list output. |
| Confirmation was already resolved | Approve/deny returns existing v0.07 resolved/not-pending behavior with the same error message. |
| Scheduler agent restarts | Due jobs are read from SQLite on the next tick/run-once; in-flight runs are recovered through the v0.13 stale-run cleanup at boot. |
| Scheduler disabled by config (`jobs.scheduler.enabled = false`) | `run_once` returns the same disabled summary as before; no signals emitted; no tick scheduled. |
| Scheduler tick directive cannot express required timing | Documented fallback in agent `@moduledoc` wraps `Process.send_after/3`; operator-visible behavior unchanged. |
| Job target needs confirmation | Job blocks/records needs-confirmation; no self-approval; confirmation record carries job context. |
| Blocked job is manually run | Existing blocked-job rejection behavior is preserved with the same error message. |
| `allbert.jido.debug_trace = true` with concurrent turns | Each turn's trace gets its own bounded top-level `## Jido Debug` subsection; no cross-turn data leakage. |
| Internal Jido command module accidentally appears in the Allbert action registry | Release gate fails; private state-machine commands must not become capability actions or intent candidates. |
| Transitional compatibility module reference appears post-M5 | Codebase grep gate fails; M5 is not done. |

## Non-Goals Visible To Users

- No new CLI commands.
- No new LiveView surfaces.
- No new channel behavior.
- No new permission classes, safety floors, risk tiers.
- No new signals.
- No schema migrations.
- No change to confirmation YAML record shape, audit markdown, or trace
  section content at default settings.
- No change to operator-facing CLI output at default settings.

### M2: Jobs.Scheduler Conversion

Status: implemented in code (M2 focused tests passing).

Record runtime parity notes here:

- `AllbertAssist.Jobs.Scheduler` is now a facade over
  `AllbertAssist.Jobs.Scheduler.Agent`; direct `start_link/1`,
  `run_once/2`, and `cleanup_stale_runs/2` compatibility remains. The
  facade does not expose `child_spec/1`; supervision happens through
  `AllbertAssist.JidoBacked.Supervisor` so the scheduler cannot be

  `Confirmations.Store.Agent` and `Jobs.Scheduler.Agent`. The previous
  application-level scheduler child was removed so there is exactly one
  scheduler process.

  `AllbertAssist.Jobs.Scheduler.Commands.*`; they are private Jido actions
  and are absent from `AllbertAssist.Actions.Registry`.
- Startup stale-run cleanup remains in the agent rebuild path. Tick
  scheduling uses `Jido.Agent.Directive.schedule/2`, which Jido currently
  implements with `Process.send_after/3`.
- Initial parity coverage:
