# Allbert v0.01 Request Flow

This document describes the current v0.01 M5 request path. It is meant to help
an implementer trace one user prompt through the Phoenix UI, runtime boundary,
Jido signals, primary intent agent, permission gate, markdown memory, and
action surface.

## Browser / LiveView Flow

1. `AllbertAssistWeb.Router`
   - Route: `live "/agent", AgentLive`
   - Module file:
     `apps/allbert_assist_web/lib/allbert_assist_web/router.ex`

2. `AllbertAssistWeb.AgentLive.mount/3`
   - Initializes the prompt, response, error, status, signal id, and loading
     assigns.
   - Default prompt: `Hello Allbert. What can you do right now?`

3. `AllbertAssistWeb.AgentLive.handle_event/3`
   - Handles the `"ask"` form submit from `#agent-form`.
   - Reads `%{"prompt" => prompt}`.
   - Calls `start_async(:ask, fn -> ... end)` so the LiveView stays responsive.
   - The async function calls:

       `AllbertAssist.Runtime.submit_user_input/1`

     with:

       `%{text: prompt, channel: :live_view, operator_id: "local"}`

4. `AllbertAssistWeb.AgentLive.handle_async/3`
   - Handles `{:ok, {:ok, response}}`.
   - Assigns:
     - `response.message` to `@response`
     - `response.status` to `@status`
     - `response.signal_id` to `@signal_id`
   - The template renders:
     - `#agent-response`
     - `#agent-status`
     - `#agent-signal`

## Runtime Boundary

All channel adapters should enter through:

`AllbertAssist.Runtime.submit_user_input/1`

Current function flow:

1. `submit_user_input/1`
   - Accepts atom-keyed or string-keyed maps.
   - Returns `{:ok, response}` or `{:error, reason}`.

2. `normalize_request/1`
   - Builds the normalized runtime request.
   - Requires non-empty `:text`.
   - Defaults:
     - `:channel` to `:unknown`
     - `:operator_id` to `:user_id` or `"local"`
     - `:metadata` to `%{}`
     - `:timeout_ms` to `120_000`

3. `new_input_signal/1`
   - Calls `Jido.Signal.new/3`.
   - Emits signal type:

       `allbert.input.received`

   - Source is built by `channel_source/1`, for example:

       `/allbert/channels/live_view`

4. `log_signal/1`
   - Logs the input signal id, type, and source.

5. `agent_runner/0`
   - Reads optional test/runtime override from:

       `Application.get_env(:allbert_assist, AllbertAssist.Runtime)`

   - Defaults to:

       `run_intent_agent/2`

6. `run_intent_agent/2`
   - Adds the input signal id/type to the normalized request as
     `:input_signal_id` and `:input_signal_type`.
   - Calls:

       `AllbertAssist.Agents.IntentAgent.respond/1`

   - v0.01 currently uses deterministic routing for the first usable local
     loop. The module still declares a `Jido.AI.Agent` and its tool surface so
     later milestones can move more selection into supervised agent execution.

7. `new_response_signal/3`
   - Calls `Jido.Signal.new/3`.
   - Emits signal type:

       `allbert.agent.responded`

   - Includes:
     - input signal id
     - response message
     - response status
     - selected/considered actions

8. `build_response/3`
   - Returns the channel-facing response map:
     - `:message`
     - `:status`
     - `:trace_id`
     - `:signal_id`
     - `:input_signal_id`
     - `:actions`

## Primary Intent Agent

The primary agent is:

`AllbertAssist.Agents.IntentAgent`

It has two roles in the current v0.01 loop:

- It is declared with `use Jido.AI.Agent`, model `:local`, and a tool list so
  it matches the Jido-centered architecture.
- It exposes `respond/1` for deterministic routing so operator tests do not
  depend on LLM variability.

Current function flow:

1. `respond/1`
   - Receives the normalized runtime request.
   - Trims `request.text`.
   - Builds an action context:

       `%{request: request, agent: AllbertAssist.Agents.IntentAgent}`

   - Calls `route/1`, then `run_route/3`.

2. `route/1`
   - Chooses one of the current intent routes:
     - `:plan_shell_command`
     - `:external_network_request`
     - `:append_memory`
     - `:read_recent_memory`
     - `{:read_skill, name}`
     - `:list_skills`
     - `:direct_answer`

3. Route predicate helpers
   - `command_request?/1`
   - `external_network_request?/1`
   - `memory_append_request?/1`
   - `memory_read_request?/1`
   - `read_skill_request?/1`
   - `capability_request?/1`

4. Input extraction helpers
   - `memory_text/1`
   - `requested_command/1`
   - `network_request/1`
   - `skill_name/1`

5. `run_route/3`
   - Calls one typed `Jido.Action` module:
     - `AllbertAssist.Actions.Intent.DirectAnswer.run/2`
     - `AllbertAssist.Actions.Intent.AppendMemory.run/2`
     - `AllbertAssist.Actions.Intent.ReadRecentMemory.run/2`
     - `AllbertAssist.Actions.Intent.ListSkills.run/2`
     - `AllbertAssist.Actions.Intent.ReadSkill.run/2`
     - `AllbertAssist.Actions.Intent.PlanShellCommand.run/2`
     - `AllbertAssist.Actions.Intent.ExternalNetworkRequest.run/2`

## Permission Gate

The central permission module is:

`AllbertAssist.Security.PermissionGate`

Functions:

- `permission_classes/0`
- `authorize/2`
- `response_status/1`
- `allowed?/1`

Current permission classes:

- `:read_only`
- `:memory_write`
- `:command_plan`
- `:command_execute`
- `:external_network`

Current decisions:

- `:read_only`, `:memory_write`, and `:command_plan` return
  `decision: :allowed`.
- `:command_execute` returns `decision: :denied`.
- `:external_network` returns `decision: :needs_confirmation`.

Every M5 intent action includes the gate result as `permission_decision` in the
top-level action result and inside the selected action metadata. The runtime
passes action metadata through in `response.actions`.

## Markdown Memory

The markdown memory module is:

`AllbertAssist.Memory`

Functions:

- `root/0`
- `ensure_root!/0`
- `categories/0`
- `append/1`
- `recent/1`

Memory root precedence:

1. `config :allbert_assist, AllbertAssist.Memory, root: "..."`
2. `ALLBERT_MEMORY_ROOT`
3. `var/allbert/memory` under the project root

Initial categories:

- `notes`
- `preferences`
- `traces`
- `skills`

Write flow:

1. `AllbertAssist.Actions.Intent.AppendMemory.run/2` receives a
   memory-shaped prompt.
2. It asks `AllbertAssist.Security.PermissionGate.authorize/2` for
   `:memory_write`.
3. It calls `AllbertAssist.Memory.append/1`.
4. `Memory.append/1` writes a markdown file under the selected category.
5. The file includes timestamp, category, source signal, actor, agent, channel,
   summary, and body.
6. The action response includes `memory_path`, `memory_category`, and
   `durable: true`.

Read flow:

1. `AllbertAssist.Actions.Intent.ReadRecentMemory.run/2` receives a recall
   prompt.
2. It asks the permission gate for `:read_only`.
3. It calls `AllbertAssist.Memory.recent/1`.
4. `Memory.recent/1` reads markdown files and ranks recent entries with simple
   query-token matching.
5. The action response includes `memories` and `memory_count`.

## Intent Actions

Every M5 intent capability is a named `Jido.Action`.

`AllbertAssist.Actions.Intent.DirectAnswer.run/2`
- Handles plain prompts without side effects.
- Returns status `:completed`.
- Returns action metadata with permission `:read_only`.
- Includes an allowed `:read_only` permission decision.

`AllbertAssist.Actions.Intent.AppendMemory.run/2`
- Handles explicit "remember..." requests.
- Writes durable markdown memory.
- Returns status `:completed`.
- Returns action metadata with permission `:memory_write`.
- Sets `durable: true` and returns `memory_path`.
- Includes an allowed `:memory_write` permission decision.

`AllbertAssist.Actions.Intent.ReadRecentMemory.run/2`
- Handles memory recall requests.
- Reads from markdown-backed memory.
- Returns status `:completed`.
- Returns matching entries and `memory_count`.
- Includes an allowed `:read_only` permission decision.

`AllbertAssist.Actions.Intent.ListSkills.run/2`
- Lists static v0.01 skill declarations.
- Calls `AllbertAssist.Skills.list/0`.
- Returns status `:completed`.
- Includes an allowed `:read_only` permission decision.

`AllbertAssist.Actions.Intent.ReadSkill.run/2`
- Reads one static v0.01 skill declaration.
- Calls `AllbertAssist.Skills.get/1`.
- Returns status `:completed`, including `status: :not_found` inside action
  metadata when the named skill is missing.
- Includes an allowed `:read_only` permission decision.

`AllbertAssist.Actions.Intent.PlanShellCommand.run/2`
- Handles command-shaped requests.
- Never executes the command.
- Returns status `:denied`.
- Marks execution as `:not_available`.
- Flags obvious destructive commands with `destructive: true`.
- Includes an allowed `:command_plan` decision and a denied requested
  `:command_execute` decision.

`AllbertAssist.Actions.Intent.ExternalNetworkRequest.run/2`
- Handles external-network-shaped requests.
- Never makes a network call.
- Returns status `:needs_confirmation`.
- Marks execution as `:not_available`.
- Includes a `:needs_confirmation` permission decision for `:external_network`.

## Skill Declarations

Temporary static skill declarations live in:

`AllbertAssist.Skills`

Functions:

- `list/0`
- `get/1`

This is a small bridge until v0.02 adds a real skill registry with readable
file-backed declarations.

## REPL / CLI-Style Flow

The REPL path skips Phoenix but uses the same runtime boundary:

```elixir
AllbertAssist.Runtime.submit_user_input(%{
  text: "Hello Allbert. What can you do right now?",
  channel: :test,
  operator_id: "local"
})
```

Flow:

1. Caller invokes `AllbertAssist.Runtime.submit_user_input/1`.
2. Runtime emits `allbert.input.received`.
3. Runtime calls `AllbertAssist.Agents.IntentAgent.respond/1`.
4. Intent agent routes to one action module.
5. Runtime emits `allbert.agent.responded`.
6. Caller receives `{:ok, response}`.

## Current Boundaries

- M5 persists explicit memory requests as markdown and reads from the same
  markdown source of truth.
- M5 does not execute shell commands. Command-shaped prompts return `:denied`.
- M5 does not make external network calls. Network-shaped prompts return
  `:needs_confirmation`.
- M5 does not write durable traces. Signal ids and permission decisions are
  returned and logged where signals are emitted.
- M5 keeps the default path deterministic; LLM-driven agent selection can grow
  after M6 traces are in place.
