# Allbert v0.05 Security Central Foundation

## Purpose

v0.05 introduces Security Central: the shared security evaluation surface that
future action-backed skills, confirmations, execution adapters, jobs, channels,
and memory workflows must use.

The current permission gate is intentionally small. That was right for v0.01
through v0.03, but the next roadmap arc needs one vocabulary for permission
decisions, risk, trust boundaries, redaction, provenance, audit, and trace
metadata before Allbert starts executing richer capability contracts.

Security Central is not a second settings store. Settings Central remains the
durable source for operator policy and secrets. Security Central evaluates
runtime context against that policy and returns structured decisions.

## Expected Inputs

- v0.04 Runtime Convergence has established Boundary Actions,
  `AllbertAssist.Actions.Registry`, `AllbertAssist.Actions.Runner`,
  action lifecycle signals, and signal/action ownership rules.
- v0.02 Settings Central exists for policy settings, secret references, and
  auditable operator configuration.
- v0.03 skill registry exists for skill trust, source scope, provenance, and
  resource inventory.
- Current actions already call `AllbertAssist.Security.PermissionGate`.
- Traces can render selected action, selected skill, and permission metadata.

## Expected Direction

- Add Security Central documentation and initial module boundaries for:
  security context, decisions, policy resolution, risk tiers, redaction, audit
  event shape, and operator-visible security status.
- Keep `AllbertAssist.Security.PermissionGate.authorize/2` as the compatibility
  entrypoint, but define the future decision shape behind it.
- Expand permission decisions from `permission + decision + reason` to include
  risk tier, confirmation need, redaction obligations, trace metadata, audit
  metadata, actor/channel/session, selected skill/action, and trust boundary.
- Centralize redaction policy for settings output, traces, audits, action
  metadata, and future adapter responses. Encrypted secret storage remains in
  `AllbertAssist.Settings.Secrets`.
- Define sandbox and supply-chain policy shapes without adding shell, script,
  package-install, network, or import execution.
- Add read-only operator status surfaces only if they stay informational:
  permission defaults, current safety boundaries, skill trust summary, secret
  status counts, and trace/redaction posture.

## Non-Goals

- No action-backed skill execution; v0.06 owns that.
- No confirmation queue; v0.07 owns that.
- No shell, skill script, package install, external network, or online skill
  import capability.
- No replacement for Settings Central.
- No multi-user authorization model.
- No prompt-injection scanner implementation; v0.05 only names the trust
  boundary and policy hook.

## Definition Of Ready

v0.05 is ready to implement when:

- v0.03 skill registry and trace metadata are released.
- v0.04 Runtime Convergence has established that security decisions are
  consumed at action/runtime boundaries, not private UI or channel code paths.
- Settings Central can resolve permission defaults, skill trust settings, and
  secret status without exposing raw secrets.
- The roadmap and ADRs agree that Security Central precedes action-backed
  skill execution.

## Definition Of Done

v0.05 is done when:

- ADR 0006 records the Security Central decision.
- `docs/plans/v0.05-request-flow.md` documents how runtime, actions, skills,
  settings, traces, and future confirmations call Security Central.
- PermissionGate compatibility behavior is preserved in the plan and tests for
  the later implementation.
- Security decisions have a documented shape covering permission, decision,
  reason, risk, confirmation, redaction, audit, trace, actor/channel/session,
  selected skill/action, and trust boundary.
- The roadmap points v0.06 to action-backed skills and all later milestones are
  shifted forward.
