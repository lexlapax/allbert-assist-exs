# Allbert v0.05 Security Central Foundation

## Purpose

Status: implementation-ready after the v0.04 release.

v0.05 introduces Security Central: the shared security evaluation surface that
future action-backed skills, confirmations, execution adapters, jobs, channels,
and memory workflows must use.

The current permission gate is intentionally small. That was right for the
first local assistant loop, but the next roadmap arc needs one vocabulary for
permission decisions, risk, trust boundaries, redaction, provenance, audit, and
trace metadata before Allbert starts executing richer capability contracts.

Security Central is not a second settings store. Settings Central remains the
durable source for operator policy and secrets. Security Central evaluates
runtime context against that policy and returns structured decisions.

v0.05 adds no new execution powers.

## Vision And Roadmap Alignment

The vision says the security gate belongs at the action boundary: before an
action mutates files, runs commands, spends money, contacts outside services, or
sends messages, Allbert should know which actor, channel, skill, action,
permission, and trace are involved.

The roadmap now has the needed predecessor in place. v0.04 released Boundary
Actions, the registered action registry, the shared action runner, lifecycle
signals, and trace recording through `record_trace`. v0.05 should consume that
boundary instead of reopening it.

Security Central sits between the completed foundations and the risky future
features:

- v0.02 Settings Central provides policy settings, secret references, and
  audited writes.
- v0.03 Agent Skills provides trust, source, provenance, resources, and inert
  capability metadata.
- v0.04 Runtime Convergence provides the action boundary and runner lifecycle
  metadata Security Central consumes.
- v0.05 Security Central provides policy evaluation, risk, redaction, audit,
  trace, context, and trust-boundary vocabulary.
- v0.06 action-backed skills may then bind skill contracts to known registered
  actions through Security Central.
- v0.07 confirmations, v0.08 shell execution, v0.09 skill scripts, v0.10
  external adapters/imports, v0.12 jobs, v0.13 channels, and v0.16 security
  hardening all depend on this shared decision vocabulary.

## Boundary Contract

Security Central owns policy evaluation and security explanations. It should:

- normalize actor, channel, session, request, selected skill, selected action,
  permission, resource, external-content, and secret-status context
- resolve effective policy from Settings Central plus built-in safety floors
- classify risk and trust boundaries
- return one canonical security decision shape
- name redaction obligations before data reaches signals, traces, audits, CLI,
  LiveView, logs, or tests
- produce audit and trace metadata
- preserve `PermissionGate.authorize/2`, `PermissionGate.allowed?/1`, and
  `PermissionGate.response_status/1` compatibility

Security Central should not:

- store operator policy or secrets
- execute actions
- approve shell, script, package-install, network, or online-import execution
- treat skill metadata as permission by itself
- replace the v0.04 action runner
- introduce multi-user authorization
- implement a prompt-injection scanner

## Canonical Decision Shape

Security decisions should preserve the current compatibility fields:

- `permission`
- `decision`
- `reason`
- `requires_confirmation`
- `source`

Security decisions should also include:

- `risk`: tier and reasons
- `redaction`: obligations for values, metadata, traces, audits, and operator
  surfaces
- `audit`: event name, actor, channel, session, permission, action, skill,
  decision, reason, and policy source
- `trace`: compact metadata safe to render in markdown traces
- `context`: normalized actor/channel/session/request/skill/action/resource
  context
- `trust_boundary`: summary of local operator input, channel, skill source,
  action registration, secret status, external content, and future sandbox or
  import boundaries
- `policy`: resolved setting value, built-in safety floor, source layer, and
  whether policy tightened or was capped by the safety floor

Initial risk tiers:

- `:minimal`: local read-only inspection
- `:low`: local memory writes and command planning that do not execute
- `:medium`: safe settings writes and operator-visible configuration changes
- `:high`: secret writes, external network requests, shell/process execution,
  script execution, package installs, and online imports
- `:critical`: raw secret reads, unknown permissions, unknown actions,
  cross-session access, or attempts to bypass registered actions

Initial decision outcomes remain:

- `:allowed`
- `:needs_confirmation`
- `:denied`

## Built-In Safety Floors

Settings Central may tighten decisions, but v0.05 built-in safety floors must
preserve current behavior and prevent premature authority:

- `:read_only` remains allowed.
- `:memory_write` remains allowed for current markdown memory behavior.
- `:command_plan` remains allowed because it plans without execution.
- `:command_execute` remains denied until the v0.08 sandbox milestone.
- `:external_network` remains confirmation-needed at most; v0.05 still does
  not add a network adapter that can execute.
- `:settings_write` remains allowed only for safe Settings Central keys.
- `:settings_secret_write` remains allowed only through explicit credential
  flows that never echo raw secrets.
- `:settings_secret_read` remains denied for user-facing surfaces.
- Unknown permissions and unknown execution modes deny by default.
- Skill manifests, capability overlays, scripts, package declarations, and
  model output never grant permission by themselves.

## Non-Goals

- No action-backed skill execution; v0.06 owns that.
- No confirmation queue; v0.07 owns that.
- No shell, skill script, package install, external network, or online skill
  import capability.
- No replacement for Settings Central.
- No multi-user authorization model.
- No durable security event database unless the implementation discovers a
  minimal audit append-only file is needed for the read-only status surface.
- No prompt-injection scanner implementation; v0.05 only names the trust
  boundary and policy hook.

## Definition Of Ready

v0.05 is ready to implement when:

- v0.04 is released and the action runner/registry boundary is the accepted
  runtime invocation path.
- v0.03 skill registry and trace metadata are released.
- Settings Central can resolve permission defaults, skill trust settings, and
  secret status without exposing raw secrets.
- ADR 0006 and ADR 0007 agree that Security Central consumes the action runner
  rather than replacing it.
- The roadmap points v0.06 to action-backed skills through Security Central and
  the existing runner.

Status: ready.

## Definition Of Done

v0.05 is done when:

- `AllbertAssist.Security` and supporting modules provide normalized context,
  policy resolution, risk, redaction, audit metadata, trust-boundary metadata,
  and canonical decisions.
- `AllbertAssist.Security.PermissionGate.authorize/2` delegates to Security
  Central while preserving current compatibility behavior.
- Security decisions cover permission, decision, reason, risk, confirmation,
  redaction, audit, trace, actor/channel/session, selected skill/action,
  policy, and trust boundary.
- Action runner metadata, action lifecycle signals, and trace output include
  redacted security decision metadata.
- Settings, skill trust/provenance, secret status, and built-in safety floors
  participate in decisions without granting new execution powers.
- A read-only operator security status action exists.
- `/settings` includes a Security or Permissions section that can edit
  Settings Central permission settings through registered settings actions and
  displays read-only effective status from the registered security status
  action.
- `docs/plans/v0.05-request-flow.md` documents how runtime, actions, skills,
  settings, traces, and future confirmations call Security Central.
- PermissionGate compatibility behavior is preserved in tests.
- Raw secrets are not rendered in security status, traces, audits, action
  metadata, CLI, LiveView, logs, or tests.
- The roadmap points v0.06 to action-backed skills through Security Central and
  the existing action runner.
