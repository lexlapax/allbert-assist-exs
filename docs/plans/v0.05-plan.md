# Allbert v0.05 Confirmation Workflow

## Purpose

v0.05 adds the confirmation layer for capability requests that are known,
inspectable, and permission-classed, but should not run without explicit
operator approval.

This comes after Settings Central and action-backed skills because
confirmations need stable settings-backed policy, known permission classes,
registered actions, and trace metadata before they can safely interrupt the
runtime.

## Expected Inputs

- v0.02 Settings Central exists with permission and confirmation-related
  settings.
- v0.03 skill registry exists for discovery, trust, and activation context.
- v0.04 action-backed skill contracts exist and map only to known Jido actions
  and known permission classes.
- CLI and LiveView can inspect the same runtime state.

## Expected Direction

- Add pending capability requests for actions that require confirmation.
- Let CLI and LiveView approve or deny a pending request through the same
  runtime boundary.
- Use settings-backed permission defaults and confirmation preferences.
- Persist confirmation decisions in traces and memory-friendly audit records.
- Keep command execution and external network adapters inert until this flow is
  tested end-to-end.

## Non-Goals

- No arbitrary command execution.
- No external network execution.
- No autonomous approval.
- No channel-specific confirmation forks.
- No long-term policy learning beyond explicit settings and audit records.

## Definition Of Ready

v0.05 is ready to implement when:

- v0.04 traces can identify selected skill, selected action, permission class,
  and outcome.
- Settings Central has stable permission defaults and confirmation preferences.
- CLI and LiveView can both render pending runtime state without duplicating
  agent logic.

## Definition Of Done

v0.05 is done when:

- Confirmation-required actions create pending requests instead of running.
- CLI and LiveView can approve or deny the same pending request.
- Approval resumes through the runtime/action boundary.
- Denial is traceable and does not run the action.
- Confirmation decisions are redacted, audited, and visible in traces.
