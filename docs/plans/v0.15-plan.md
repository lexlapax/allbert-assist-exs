# Allbert v0.15 Minimal App Registration Contract

Formerly tracked as M-AppContract-Lite.

## Purpose

v0.15 gives StockSage and future workspace apps a small public surface to
register identity, validation, child supervision, registered Jido actions, skill
paths, and navigation entries with Allbert without reaching through private
internals.

This milestone is deliberately small. It introduces the `AllbertAssist.App`
behaviour, `AllbertAssist.App.Registry`, and `AllbertAssist.App.Supervisor`,
adds an optional `:app_id` field to registered action capability metadata,
replaces the v0.14 `AllbertAssist.Session.AppId` allowlist with registry-backed
validation, and surfaces app inspection through registered Jido actions and a
`mix allbert.apps` CLI. It does not add the `AllbertAssist.App.SurfaceProvider`
behaviour, `AllbertAssist.Surface` DSL, canvas storage, AG-UI/A2UI encoders,
generator scaffolding, dynamic route injection, or any new execution authority.
Those arrive in v0.18 (full contract) and v0.26 (formerly v0.24) workspace
shell.

The acceptance surface is the runtime, registered actions, signals, traces, the
`mix allbert.apps` CLI, the `Mix.Tasks.Allbert.Sessions` CLI from v0.14, and
tests. Phoenix LiveView may render `AllbertAssist.App.Registry.registered_surfaces/0`
data later; v0.15 does not add a shell sidebar, an `/apps` LiveView, or any new
HTTP route.

## Expected Inputs

- v0.11 is released and tagged. `AllbertAssist.Intent.Decision` already
  carries `active_app` as a validated atom field.
- v0.12 local string `user_id` and SQLite conversation history are implemented.
- v0.13 scheduled jobs are implemented; jobs carry nil/general `app_id` and
  must continue to do so after v0.15 unless a later plan defines app-scoped
  jobs.
- v0.14 ETS session scratchpad is implemented. Scratchpad entries can carry
  `active_app`. v0.14 ships a compile-time `AllbertAssist.Session.AppId`
  allowlist (`:allbert`, `:stocksage`, plus nil aliases); v0.15 replaces that
  allowlist with `AllbertAssist.App.Registry` validation.
- `AllbertAssist.Actions.Registry` exists, is compile-time, and exposes
  `capabilities/0`, `agent_capabilities/0`, `internal_capabilities/0`,
  `resolve/1`, and `capability/1`. The `AllbertAssist.Actions.Capability` struct
  exposes `name`, `module`, `permission`, `exposure`, `execution_mode`,
  `skill_backed?`, `confirmation`, `resumable?`, and `notes`. v0.15 adds an
  optional `:app_id` field without changing the compile-time core list.
- `AllbertAssist.Skills.Registry` exists as a stateless, settings-driven loader
  over `skills.scan_paths` and built-in/user roots. v0.15 makes it consult
  `AllbertAssist.App.Registry.registered_skill_paths/0` at load time without
  caching that result.
- `AllbertAssist.Settings` exists with a static schema in
  `AllbertAssist.Settings.Schema`. v0.15 introduces no new writable settings
  keys.
- `AllbertAssist.Runtime.submit_user_input/1` remains the runtime entrypoint.
  Channels, jobs, CLI, and LiveView continue to call it instead of calling
  agents directly.
- ADR 0015 is accepted before implementation, with v0.15 marked as the lite
  contract slice.

## Roadmap And Vision Fit

The Allbert/Jido vision treats workspace apps as first-class participants in
the supervised runtime: an app declares identity, validation, supervision,
registered Jido actions, skills, and navigation surfaces and plugs into
Allbert through public contracts rather than private hooks. v0.15 is the
first slice of that contract. It is intentionally weaker than the full v0.18
contract:

- App registration carries identity, validation, child supervision, action
  modules, skill paths, and navigation surfaces only.
- App registration never grants permission. Registered actions still run
  through `AllbertAssist.Actions.Runner.run/3`, Security Central, confirmation
  workflow, traces, and audits.
- Surfaces are display data describing existing routes. v0.15 does not register
  new routes, mount LiveView modules from app folders, or render a workspace
  shell.
- Apps do not declare signals, settings schemas, memory namespaces, surface
  providers, canvas components, or surface DSL nodes in v0.15. Those are v0.18
  responsibilities.

This milestone prepares later work without implementing it:

- v0.16 additional channels can map external identities into local `user_id`
  values without needing per-channel app routing; channel adapters can read
  registered app metadata for display but must not invent permission grants
  from the registry.
- v0.17 plugin work can feed app modules into `AllbertAssist.App.Registry`
  from source-tree plugin packages without turning app registration into the
  general extension registry.
- v0.20 StockSage can register a real `StockSage.App` implementing the v0.18
  full contract through `StockSage.Plugin`. The v0.15
  `AllbertAssist.App.StockSageStub` is removed by v0.20 when StockSage is
  contributed by the plugin layer.
- v0.19 intent enrichment can prioritize app-scoped actions when `active_app`
  matches a registered app and the action carries that `app_id`.
- v0.27 (formerly v0.25) StockSage LiveViews can mount as standard Phoenix
  routes and set `active_app: :stocksage` when the operator is in StockSage
  context; they continue to use statically configured routes.
- v0.28 (formerly v0.26) security evals can test app-scoped action routing,
  cross-app session leakage, and registry-bypass attempts.
- v0.18 app/surface contract extends `AllbertAssist.App` to include declared
  signals, settings schemas, `AllbertAssist.App.SurfaceProvider`,
  `AllbertAssist.Surface` DSL, and `mix allbert.validate_app`. v0.15 must not
  ship anything that v0.18 would have to remove.
- v0.26 (formerly v0.24) workspace shell and ephemeral UI consume
  `App.Registry.registered_surfaces/0` and `App.Registry.registered_apps/0`
  for navigation discovery without inventing its own app catalog.
- v0.31 (formerly v0.29) app generator builds on the v0.18 app/surface
  contract; the generator must
  produce app plugins that register only through the public
  `AllbertAssist.Plugin` and `AllbertAssist.App` behaviours and not load
  runtime modules from arbitrary folders.

v0.15 must keep Allbert local-first and inspectable. The registry should be a
supervised process with a clear API, predictable validation diagnostics, and
boring restart behavior.

## Handoffs

### v0.12 Handoff

App registration does not touch `user_id`, `operator_id`, `thread_id`, or
`session_id` propagation. Runtime requests continue to carry these v0.12
fields exactly as v0.13 and v0.14 define. App context is `active_app`, not
identity, and is owned by the v0.14 scratchpad lookup, not the app registry.

### v0.13 Handoff

Jobs continue to store nil/general `app_id` in `scheduled_jobs` and
`scheduled_job_runs` rows. v0.15 does not introduce app-scoped jobs, app-owned
schedules, or per-app job templates. A job whose stored `session_id` matches a
v0.14 scratchpad entry with `active_app: :stocksage` may receive that
`active_app` in its runtime decision context, but the job target still runs
through unchanged Security Central, confirmation, and resource-access
boundaries.

### v0.14 Handoff

v0.14 reserves a compile-time `AllbertAssist.Session.AppId` allowlist
(`:allbert`, `:stocksage`, plus nil aliases). v0.15 replaces that allowlist
with `AllbertAssist.App.Registry` validation:

- `AllbertAssist.Session.AppId.normalize/1` and the registered actions
  `set_active_app`, `clear_active_app`, and `show_session_scratchpad` delegate
  app-id parsing to `AllbertAssist.App.Registry.normalize_app_id/1` after the
  existing nil/general alias normalization. The registry helper accepts known
  atoms, safely resolves known binary names through existing atoms, and returns
  `{:error, :unknown_app}` for everything else. v0.15 never creates new atoms
  from external app-id input.
- `AllbertAssist.Intent.Decision.new/1` rejects candidate `active_app` values
  that are not registered by adding a structured diagnostic and falling back
  to known context or nil. Runtime/session write paths still return
  `:unknown_app` errors for invalid operator input.
- v0.15 ships two built-in app modules so the registry is never empty after
  boot:
  - `AllbertAssist.App.CoreApp` with `app_id: :allbert`, no actions, no
    skills, no surfaces. This makes `:allbert` always valid and matches the
    "general Allbert context" semantics v0.14 documents.
  - `AllbertAssist.App.StockSageStub` with `app_id: :stocksage`, no actions,
    no skills, no surfaces. This is a transition artifact only. It exists so
    v0.14 acceptance for `set_active_app stocksage` keeps passing between
    v0.15 release and v0.20 release. It is removed in v0.20 when
    `StockSage.Plugin` contributes `StockSage.App`.

The `:allbert` and `:stocksage` atoms remain present because
`AllbertAssist.App.CoreApp` and `AllbertAssist.App.StockSageStub` reference
them at compile time. Registry membership, not atom existence, decides whether
input is accepted. In v0.15, `nil` is the neutral/general context; v0.18
later establishes `active_app: :allbert` as the runtime default for
channel-delivered requests, replacing nil as the baseline.

### v0.20 Handoff

v0.20 introduces `./plugins/stocksage` with `StockSage.Plugin` contributing
`StockSage.App` implementing the v0.18 full app/surface contract:

- Registers `StockSage.Plugin` through the v0.17 source-tree plugin substrate.
- Removes `AllbertAssist.App.StockSageStub` and its config-list entry.
- Registers StockSage's actions, skill paths, and navigation surfaces through
  the v0.18 app/surface contract public registry API.
- Tags StockSage actions with `app_id: :stocksage` via capability metadata.

v0.15 must keep `AllbertAssist.App.Registry`'s public API stable enough that
v0.18 can expand it and v0.20 can use the expanded API without touching
v0.14/v0.15-era code.

### v0.18 Handoff

v0.18 expands `AllbertAssist.App` into the app/surface contract: identity
and OTP lifecycle, agents/actions/signals, skills, UI surface
(`AllbertAssist.App.SurfaceProvider`, `AllbertAssist.Surface`), and
data/settings. Memory namespace registration remains deferred to v0.29
(formerly v0.27). v0.15
must not pre-define those callbacks in a way that constrains v0.18's design.
The v0.15 `surfaces/0` callback returns display data only; v0.18 may add
`surface_providers/0`, `signals/0`, `settings_schema/0`, and
`canvas_components/0` callbacks. v0.15 callers must not assume those callbacks
exist.

## Implementation Boundaries

Authoritative existing boundaries:

- Application supervision tree:
  `apps/allbert_assist/lib/allbert_assist/application.ex`. Current order is
  Repo → DNSCluster → PubSub → SignalBus → `AllbertAssist.Jido` → Scratchpad
  (v0.14) → Scheduler (v0.13).
- Action registry: `AllbertAssist.Actions.Registry` and
  `AllbertAssist.Actions.Capability` in
  `apps/allbert_assist/lib/allbert_assist/actions/`.
- Action runner: `AllbertAssist.Actions.Runner.run/3`.
- Skills registry: `AllbertAssist.Skills.Registry` in
  `apps/allbert_assist/lib/allbert_assist/skills/registry.ex`.
- Settings: `AllbertAssist.Settings` and `AllbertAssist.Settings.Schema`.
- Decision contract: `AllbertAssist.Intent.Decision`.
- Session scratchpad and AppId helper: `AllbertAssist.Session.Scratchpad`,
  `AllbertAssist.Session`, `AllbertAssist.Session.AppId` (v0.14).
- CLI namespace: `Mix.Tasks.Allbert.*` under
  `apps/allbert_assist/lib/mix/tasks/`.

Preferred new implementation shape:

- `AllbertAssist.App`: public behaviour defining the lite contract callbacks
  (`app_id/0`, `display_name/0`, `version/0`, `validate/1`, `child_spec/1`,
  `actions/0`, `skill_paths/0`, and optional `surfaces/0`).
- `AllbertAssist.App.Registry`: supervised `GenServer` that owns a protected
  ETS table holding registered app records and a diagnostics map. Public API:
  `register/2`, `unregister/1`, `lookup/1`, `registered_apps/0`,
  `registered_surfaces/0`, `registered_skill_paths/0`, `known_app_id?/1`,
  `normalize_app_id/1`, `actions_for/1`, `app_id_for_action/1`,
  `diagnostics/0`.
- `AllbertAssist.App.DynamicSupervisor`: `DynamicSupervisor` for registered
  apps' `child_spec/1` children. Apps with no children may return `:ignore` or
  a no-op `Supervisor` child; the registry tolerates both.
- `AllbertAssist.App.Bootstrap`: small `GenServer` that registers configured
  apps in `handle_continue/2` and then stays idle. It is not a completing
  permanent `Task`, so a normal bootstrap completion cannot create a supervisor
  restart loop.
- `AllbertAssist.App.Supervisor`: top-level supervisor that wraps `Registry`,
  `DynamicSupervisor`, and `Bootstrap` with `:one_for_all` semantics so a
  registry, app-child supervisor, or bootstrap crash restarts the whole
  volatile app-registry group and forces clean re-registration.
- `AllbertAssist.App.CoreApp`: built-in `:allbert` app, registered by default.
- `AllbertAssist.App.StockSageStub`: built-in `:stocksage` placeholder,
  registered by default, removed by v0.20.
- `AllbertAssist.App.Validator`: pure module performing the registration-time
  validation described under the App Contract section.
- Capability struct change:
  `AllbertAssist.Actions.Capability` gains optional `:app_id` field
  (`atom() | nil`), default `nil`.
- Registered actions:
  `AllbertAssist.Actions.ListApps`, `AllbertAssist.Actions.ShowApp`.
- CLI: `Mix.Tasks.Allbert.Apps`.

The ETS table is owned by the `AllbertAssist.App.Registry` GenServer. Runtime,
CLI, registered actions, tests, and future channel adapters reach the registry
through the GenServer API. No external caller touches ETS directly.

`AllbertAssist.App.Supervisor` belongs in the application supervision tree
between `AllbertAssist.Jido` and `AllbertAssist.Session.Scratchpad` so that
session writes that consult the registry can rely on a populated registry, and
so that scheduler-originated session lookups have a registry to validate
against. It must start after Repo/PubSub/Jido so child app supervisors can use
those primitives if needed.

## App Contract

`AllbertAssist.App` is a `@behaviour` module. Implementing modules use
`use AllbertAssist.App` or `@behaviour AllbertAssist.App` and define the
following callbacks:

Required:

- `@callback app_id() :: atom()` — compile-time atom identifier for the app.
  Must match `~r/^[a-z][a-z0-9_]*$/` after `Atom.to_string/1`. Reserved
  values: `:allbert` (used by `CoreApp`) and `nil`/`:none`/`:general` (used
  by `AllbertAssist.Session.AppId` as nil aliases) must not be reused by
  other apps. v0.15 reserves `:stocksage` for `StockSage.App` and the v0.15
  `StockSageStub` only.
- `@callback display_name() :: String.t()` — operator-visible name. Required,
  trimmed, 1..64 characters.
- `@callback version() :: String.t()` — semver-like string. Required, 1..32
  characters. v0.15 does not enforce SemVer comparison; the string is
  diagnostic.
- `@callback validate(opts :: keyword() | map()) :: :ok | {:error, [diagnostic()]}` —
  called at registration time. Receives the registration opts passed to
  `AllbertAssist.App.Registry.register/2` (typically `[]`). Returns `:ok` or
  a list of structured diagnostic maps `%{kind: atom(), message: String.t(), detail: map()}`.
  Validation must be pure and side-effect-free.
- `@callback child_spec(opts :: keyword() | map()) :: Supervisor.child_spec() | :ignore` —
  returns a child spec for the app's own private supervision tree, or
  `:ignore` if the app has no children. The child spec is started under
  `AllbertAssist.App.DynamicSupervisor`. v0.15 does not require apps to have
  any children; `CoreApp` and `StockSageStub` both return `:ignore`.
- `@callback actions() :: [module()]` — list of registered Jido action
  modules contributed by this app. Each module must already be present in
  `AllbertAssist.Actions.Registry`'s compile-time list. v0.15 does not load
  action modules from arbitrary folders or generate them at runtime. The
  returned list may be empty.
- `@callback skill_paths() :: [Path.t()]` — list of absolute filesystem paths
  to skill packs. Each path must be a binary, absolute, and exist (or be
  expected to exist) on disk. Paths get `trust: :trusted` posture in skills
  registry resolution because app modules are compiled into the umbrella. The
  returned list may be empty.

Optional:

- `@callback surfaces() :: [surface_entry()]` — list of navigation surface
  entries. `surface_entry()` is the v0.15 display-data shape:

  ```elixir
  %{
    id: atom(),
    label: String.t(),
    path: String.t(),
    app_id: atom(),
    icon: String.t() | nil,
    description: String.t() | nil
  }
  ```

  Rules:

  - `id` is a stable per-app atom, unique within the app's surfaces. v0.15
    does not enforce uniqueness across apps but `App.Registry` records
    diagnostics for cross-app duplicates.
  - `path` is a static URL path string. v0.15 does not register routes from
    surfaces; the path must already correspond to an existing Phoenix route
    or remain operator-visible documentation only. The validator checks that
    `path` starts with `/`, is 1..128 characters, and does not contain query
    strings, fragments, or whitespace.
  - `app_id` must equal the app's `app_id/0`.
  - `icon` and `description` are optional, non-secret diagnostic strings.

  The default implementation returns `[]`.

`use AllbertAssist.App` provides default implementations for `actions/0`,
`skill_paths/0`, and `surfaces/0` that return empty lists, plus a
default `child_spec/1` that returns `:ignore`. `app_id/0`, `display_name/0`,
`version/0`, and `validate/1` have no defaults and must be implemented.

## App Registry State Contract

The ETS table is the only data store v0.15 introduces. Apps' own data lives in
their own contexts and (for v0.20 StockSage) in their own SQLite tables.

ETS table:

- Name: `:allbert_app_registry` (atom, named table; overridable via
  application config for tests).
- Type: `:set`.
- Access: `:protected`. The supervised `AllbertAssist.App.Registry` process
  owns the table. Writes are serialized through the GenServer.
- Concurrency: `read_concurrency: true`, `write_concurrency: false`.
- Lifetime: created in the GenServer's `init/1`. When the owning process
  terminates for any reason, the table is destroyed. There is no disk
  persistence and no migration path.

Entries are keyed by `app_id :: atom()`.

Stored entry shape:

```elixir
%{
  app_id: :stocksage,
  module: StockSage.App,
  display_name: "StockSage",
  version: "0.1.0",
  actions: [StockSage.Actions.RunAnalysis, ...],
  skill_paths: ["/abs/path/to/stocksage_skills"],
  surfaces: [%{id: :workspace, label: "StockSage", path: "/stocksage", ...}, ...],
  child_id: :stocksage_supervisor | :ignore,
  child_pid: #PID<0.123.0> | :ignore | nil,
  registered_at_ms: 12_345,
  metadata: %{}
}
```

Rules:

- Duplicate `app_id` registrations fail with `{:error, {:app_id_taken, :stocksage}}`
  without disturbing the existing registration.
- A registration whose `validate/1` returns an error is recorded in the
  registry diagnostics map under that `app_id` and produces `{:error,
  {:validation_failed, diagnostics}}`. No ETS entry is created. No child
  spec is started.
- A registration whose `validate/1` succeeds but whose `child_spec/1` returns
  an invalid spec produces `{:error, {:child_spec_failed, detail}}` and is
  recorded in diagnostics. No ETS entry is created.
- An app that returns `child_spec/1 = :ignore` stores `child_id: :ignore`
  and `child_pid: :ignore`. The registry is still considered registered.
- An app that starts a child stores the stable child spec `:id` as
  `child_id` for diagnostics and stores the started pid as `child_pid` for
  `unregister/1` termination. `DynamicSupervisor.which_children/1` does not
  preserve dynamic child ids, so v0.15 treats the top-level app child as a
  stable app-owned supervisor process; that child should supervise its own
  internals rather than relying on the registry to re-associate restarted
  dynamic children.
- `unregister/1` removes the ETS entry and terminates the app's child under
  `AllbertAssist.App.DynamicSupervisor` if present. `unregister/1` is exposed
  for tests; production code does not call it during normal operation.
- The registry diagnostics map is observable via `diagnostics/0` and is
  always rendered in `mix allbert.apps list` for any app that failed to
  register.

CLI/log/trace output may show `app_id`, `display_name`, `version`, action
counts, skill-path counts, and surface counts, but must not dump arbitrary
`metadata` values or app child supervision internals.

## Supervision And Lifecycle

`AllbertAssist.App.Supervisor` runs as a supervised `Supervisor` started from
`AllbertAssist.Application`. Boot order becomes:

1. `AllbertAssist.Repo` and migrations.
2. `DNSCluster`.
3. `Phoenix.PubSub` (`AllbertAssist.PubSub`).
4. `Jido.Signal.Bus` (`AllbertAssist.SignalBus`).
5. `AllbertAssist.Jido`.
6. `AllbertAssist.App.Supervisor` (new in v0.15).
7. `AllbertAssist.Session.Scratchpad` (v0.14).
8. `AllbertAssist.Jobs.Scheduler` (v0.13, when enabled).

`AllbertAssist.App.Supervisor`'s children, in order, with `:one_for_all`
strategy:

1. `AllbertAssist.App.Registry` — owns ETS table.
2. `AllbertAssist.App.DynamicSupervisor` — supervises registered apps'
   child specs.
3. `AllbertAssist.App.Bootstrap` — registers configured apps once.

`:one_for_all` treats the registry ETS table, app child supervisor, and
bootstrap process as one volatile unit. If any of them crashes, the entire
group restarts: the registry starts empty, the app child supervisor starts
with no children, and Bootstrap re-reads the `:apps` config list and
re-registers each configured app. This avoids stale child PIDs or stale action
mappings after a supervisor restart and preserves v0.15's registry volatility
guarantee, exactly like v0.14's scratchpad.

`AllbertAssist.App.Registry` responsibilities:

- `init/1`: create the named ETS table, set up an empty diagnostics map,
  and log a startup line.
- `handle_call/3` serializes writes: `register`, `unregister`, `clear`
  (test-only).
- Reads (`lookup`, `registered_apps`, `registered_surfaces`,
  `registered_skill_paths`, `known_app_id?`, `actions_for`,
  `normalize_app_id`, `app_id_for_action`, `diagnostics`) may go through the
  GenServer for simplicity or read the protected table directly through the
  public module. Either choice is acceptable as long as every external caller
  goes through the public `AllbertAssist.App.Registry` module and never
  touches ETS directly.
- `terminate/2` does nothing special; ETS dies with the process.

`AllbertAssist.App.Bootstrap` responsibilities:

- Read `Application.get_env(:allbert_assist, :apps, [AllbertAssist.App.CoreApp, AllbertAssist.App.StockSageStub])`.
- For each module, call `AllbertAssist.App.Registry.register(module, [])` and
  log the outcome. Aggregated diagnostics are visible via `diagnostics/0`.
- Once all configured apps have been processed, transition to a permanent
  idle state. Bootstrap should not be implemented as a permanent child that
  exits normally after registration, because that would create a supervisor
  restart loop.

Test isolation:

- `AllbertAssist.App.Registry.register/2` is callable directly from tests;
  it does not require the bootstrap path. Tests may register fixture apps
  under a sandboxed registry name.
- Application config keys (under
  `Application.get_env(:allbert_assist, AllbertAssist.App.Registry)`) for
  test-only or deployment-only options:
  - `enabled?`: defaults to `true`. When `false`, the supervised process
    either is not started or starts in a disabled mode that returns
    `:not_found` from lookups and `:disabled` from writes.
  - `table_name`: defaults to `:allbert_app_registry`. Tests may pass an
    alternate name to start an isolated registry instance.
- Bootstrap may be disabled via
  `Application.put_env(:allbert_assist, :apps_bootstrap, false)` so tests can
  set up the registry without auto-registering `CoreApp` and `StockSageStub`.
  Otherwise the v0.15 default is to bootstrap the two built-ins.

## Configuration

v0.15 does not introduce any new writable Settings Central keys. Operator
configuration of which apps to register lives in `config.exs` (or releases
overlay):

```elixir
config :allbert_assist, :apps, [
  AllbertAssist.App.CoreApp,
  AllbertAssist.App.StockSageStub
]
```

v0.20 will replace `AllbertAssist.App.StockSageStub` with
`./plugins/stocksage` contributing `StockSage.Plugin` and `StockSage.App`
through the v0.17 plugin substrate.

Test config may set `:apps` to `[]` to start with a fully empty registry, or
to a list of test fixture apps. Test config may also set
`:apps_bootstrap, false` to skip Bootstrap entirely and require tests to call
`AllbertAssist.App.Registry.register/2` explicitly.

## Action Registry Integration

`AllbertAssist.Actions.Capability` gains an optional `:app_id` field of type
`atom() | nil`, default `nil`. All existing compile-time core actions retain
`app_id: nil`. App-contributed actions store the registering app's
`app_id` so that capability metadata can be filtered by app at lookup time.
`AllbertAssist.Actions.Capability.summary/1` includes `app_id` when present so
decision metadata, traces, and tests observe one consistent shape.

`AllbertAssist.Actions.Registry` gains:

- A read-time merge with `AllbertAssist.App.Registry.app_id_for_action/1`
  results so capability lookups stamp `app_id` consistently even if a future
  refactor changes how registration happens.
- `capabilities_for_app(app_id :: atom())` accessor returning capabilities
  tagged with the given `app_id`. Returns `[]` when no app actions are
  registered for that id.
- No change to existing public functions' semantics. Core actions continue to
  appear in `capabilities/0` exactly as today, with `app_id: nil`.

App registration does not extend the executable action set. It tags a subset
of actions that are already compiled and already present in
`AllbertAssist.Actions.Registry`. v0.18 may use those tags for active-app
ranking; v0.15 only records and exposes them.

App registration does not load action modules dynamically. The app's
`actions/0` callback must return modules already present in
`AllbertAssist.Actions.Registry`'s compile-time list. The validator rejects
any action module that does not resolve through `AllbertAssist.Actions.Registry.resolve/1`.
This keeps v0.15 inside the existing compile-time discipline: app authors add
modules to the action registry at compile time and to the app's `actions/0`
list at registration time. Both must agree.

Tagging path:

1. At registration, `AllbertAssist.App.Registry` records `{app_id, action_module}`
   pairs.
2. At capability read time, `AllbertAssist.Actions.Registry.capability/1`
   consults `AllbertAssist.App.Registry.app_id_for_action/1` and fills the
   capability's `app_id` field.
3. Intent decisions and trace metadata observe the tagged capability and may
   include `app_id` in their context. v0.15 does not introduce intent
   prioritization based on `app_id`; v0.19 does.

## Skills Registry Integration

`AllbertAssist.Skills.Registry` consults
`AllbertAssist.App.Registry.registered_skill_paths/0` on every `load/1` call.
`registered_skill_paths/0` returns entries shaped as
`%{app_id: atom(), path: Path.t()}` rather than bare strings so the skills
registry can attach `source_scope: :app` and app diagnostics without guessing
ownership.
The merged source list is, in precedence order:

1. Built-in root (`priv/skills`) — trusted, precedence 0.
2. Project native (`.allbert/skills`) — trusted/pending, precedence 1.
3. Project interoperable (`.agents/skills`) — trusted/pending, precedence 2.
4. **App-contributed paths** (new in v0.15) — trusted, precedence 3.
5. User native (`~/.allbert/skills`) — trusted, precedence 4.
6. User interoperable (`~/.agents/skills`) — trusted, precedence 5.
7. Configured `skills.scan_paths` — operator-controlled trust, precedence 6+.
8. Imported cache (`~/.allbert/cache/skills`) — pending/trusted per policy,
   precedence 100.

App-contributed paths default to `trust: :trusted` posture because app modules
are compiled into the umbrella; operators configured the app by adding it to
the `:apps` config list. v0.15 does not let an app id alone elevate trust for
arbitrary paths; only paths returned by a registered app's `skill_paths/0`
callback receive trusted posture.

The implementation adds `:app` to `AllbertAssist.Skills.Skill.source_scope/0`
and to the registry precedence table. App skills report `source_scope: :app`
and include `app_id` in diagnostics where practical, but they still pass
through the same `SKILL.md` parser, capability-contract validation, duplicate
resolution, and enablement filters as every other skill source.

`Skills.Registry` does not cache the app-contributed paths between loads; it
reads them on every `load/1` call exactly as it reads `skills.scan_paths`.
When an app is unregistered (test-only) or the registry restarts empty, the
next `load/1` reflects that change.

## Surface Data Contract

`AllbertAssist.App.Registry.registered_surfaces/0` returns a flat list of all
surface entries across all registered apps, in registration order (which
matches `:apps` config list order). Surface entries are the display-data
shape defined in the App Contract section.

v0.15 does not:

- Register Phoenix routes from surface entries.
- Mount LiveView modules from surface entries.
- Validate that `path` corresponds to an existing route; the operator/app
  author is responsible for keeping `path` aligned with statically defined
  routes in `apps/allbert_assist_web/lib/allbert_assist_web/router.ex`.
- Render a workspace shell, app sidebar, or `/apps` LiveView. v0.26
  (formerly v0.24) introduces the workspace shell that consumes this data.

CLI output and registered action results may include surface entries; raw
`metadata` and per-surface `description` strings are rendered when present
and trimmed.

## Decision Validation Handoff

`AllbertAssist.Intent.Decision.new/1` keeps its existing strict validation
shape. v0.15 changes the `active_app` validation step:

- Context-supplied `active_app` from the v0.14 scratchpad is preserved when
  the candidate decision does not explicitly set one.
- An `active_app` value in a candidate decision is normalized through
  `AllbertAssist.App.Registry.normalize_app_id/1`. Unknown values are rejected
  as structured diagnostics rather than silently accepted.
- When candidate output supplies an unknown `active_app`, the decision falls
  back to the context-supplied active app when that context value is known;
  otherwise the decision proceeds with `active_app: nil`.

`AllbertAssist.Session.AppId.normalize/1` is updated:

- nil, `""`, `"none"`, and `"general"` continue to normalize to nil.
- Atom and binary input delegate to
  `AllbertAssist.App.Registry.normalize_app_id/1`. Unknown binary input is
  rejected with `:unknown_app`. v0.15 retains the v0.14 invariant that no new
  atoms are created from external app-id input.

The v0.14 compile-time allowlist in `AllbertAssist.Session.AppId` is removed.
The atom reservations for `:allbert` and `:stocksage` move into
`AllbertAssist.App.CoreApp` and `AllbertAssist.App.StockSageStub`
respectively, which the bootstrap registers by default.

## Registered Actions

v0.15 exposes app-registry inspection through registered Jido actions so
intent agents, CLI, future channels, and tests share one contract. Actions
run through `AllbertAssist.Actions.Runner.run/3` and emit the standard
`allbert.action.requested`/`allbert.action.completed` lifecycle signals.

Registered actions:

- `list_apps`: no params. Permission `:read_only`, exposure `:agent`,
  execution mode `:settings_read`, confirmation `:not_required`. Returns a
  list of registered apps with `app_id`, `display_name`, `version`,
  `action_count`, `skill_path_count`, and `surface_count`. Diagnostics for
  failed registrations are included as a separate `diagnostics` list. Raw
  skill paths and surface metadata are not included.
- `show_app`: params `app_id` (string or atom). Permission `:read_only`,
  exposure `:agent`, execution mode `:settings_read`, confirmation
  `:not_required`. Returns
  `app_id`, `display_name`, `version`, the sorted list of registered action
  names (strings), the list of skill paths (absolute paths), and the list of
  surfaces (full display data). Missing apps return `:not_found`. Binary
  `app_id` goes through `AllbertAssist.App.Registry.normalize_app_id/1`;
  unknown binary returns `:unknown_app`.

The registered action capability metadata must follow the v0.06 contract
(permission, execution mode, confirmation requirement, resumability,
redaction posture). Both actions call `PermissionGate.authorize/2` with
`:read_only`. Neither declares itself as a resource-access consumer; neither
touches `AllbertAssist.Resources.*`; neither creates durable confirmations;
neither requires new Security Central permission classes or safety floors.

v0.15 does not register `register_app`, `enable_app`, `disable_app`, or
`unregister_app` actions. Registration happens through the configured
`:apps` list and `AllbertAssist.App.Registry.register/2` only. Mutating
registry membership at the action boundary is deferred until a later plan
defines what those mutations imply for security, traces, and audits.

## CLI Contract

Add `mix allbert.apps` with subcommands:

- `mix allbert.apps list`
  - Renders one row per registered app: `app_id`, `display_name`, `version`,
    `actions` count, `skills` count, `surfaces` count.
  - Renders one row per failed registration with the validation diagnostic
    kind and a redacted message.
- `mix allbert.apps show APP_ID`
  - Renders `app_id`, `display_name`, `version`, implementing module, action
    names (sorted), skill paths, surfaces (full display data including
    `label`, `path`, `description`, `icon`).
  - Missing app returns a clear not-found message.
- `mix allbert.apps validate MODULE`
  - Resolves the given module name (e.g., `StockSage.App`) against already
    compiled module atoms from loaded OTP application specs and currently
    loaded modules, confirms it implements `AllbertAssist.App`, runs
    `validate/1` with `[]`, and prints the resulting `:ok` or diagnostics
    list.
  - Does NOT register the app, does NOT start its child spec, does NOT
    touch the registry. Pure inspection for app-author iteration.
  - Unknown or unloaded modules return a clear error; v0.15 must not crash
    the task on bad module names.

CLI rules:

- All three subcommands are read-only. They never mutate ETS state.
- Binary `APP_ID` goes through
  `AllbertAssist.App.Registry.normalize_app_id/1`; unknown binary returns
  `:unknown_app` with a redacted message.
- `validate MODULE` must not call `String.to_atom/1` or `Module.concat/1` on
  arbitrary operator input. If the module name cannot be matched to an already
  compiled module atom, the command returns a clear error.
- CLI delegates `list` and `show` to the same registered actions (`list_apps`,
  `show_app`) via `AllbertAssist.Actions.Runner.run/3` so the CLI and channel
  callers share one code path. `validate` is CLI-only because it operates on
  a module name the operator types, not on registry state.

## Signals, Traces, And Message Metadata

App registry membership changes are observable through:

- `list_apps` / `show_app` registered action lifecycle signals
  (`allbert.action.requested`, `allbert.action.completed`) with redacted
  parameters.
- `mix allbert.apps` CLI output for operator inspection.

v0.15 introduces no new signal types. The registry does not emit
`allbert.app.*` lifecycle signals. Bootstrap registration produces log lines
(level `:info` for success, level `:warning` for diagnostic failure), but no
signal bus events.

Action runtime traces and decision metadata may include `app_id` when the
selected action's capability is tagged with one. v0.15 does not change the
existing trace shape beyond adding `app_id` alongside the existing action
name; trace consumers must treat missing `app_id` as the nil/general case.

The v0.14 `active_app` propagation through signals, decisions, traces, and
assistant `action_log` is unchanged. v0.15 only adds registry-backed
validation; signal/trace fields do not gain new keys.
