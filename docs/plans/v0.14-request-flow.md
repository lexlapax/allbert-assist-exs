# Allbert v0.14 Request Flow: Session Scratchpad And Active App Context

This request-flow document is the implementation companion to
`docs/plans/v0.14-plan.md`. It describes how volatile session state is created,
read by the runtime, expired, inspected, and kept separate from durable memory,
conversation history, app registration, and authorization.

Status: planned.

## Common Shape

Every v0.14 session lookup follows the same shape:

1. Resolve local string `user_id`.
2. Resolve optional `session_id`.
3. If `session_id` is absent, skip scratchpad lookup and use
   `active_app: nil`.
4. If `session_id` is present, look up `{user_id, session_id}` in
   `AllbertAssist.Session.Scratchpad`.
5. If the entry exists and has not expired, pass its `active_app` into runtime
   context.
6. If the entry is missing or expired, continue with `active_app: nil`.

Scratchpad lookup never authorizes work, never grants app permissions, and
never changes thread ownership. It is context, not authority.

## Scratchpad Entry Lifecycle

### Create Or Update

For:

```sh
mix allbert.sessions set-active-app \
  --user alice \
  --session sess-1 \
  stocksage
```

The flow is:

1. CLI validates identity. Omitted `--user` defaults to `"local"`.
2. `--operator` remains a compatibility alias. If `--user` and `--operator`
   are both present and differ, fail before mutating scratchpad.
3. CLI normalizes `session_id`; blank session ids are rejected.
4. CLI normalizes `APP`. `"stocksage"` maps to `:stocksage`;
   `"general"`, `"none"`, and `""` map to nil. Unknown binary app ids are
   rejected before v0.15 app-registry validation exists.
5. `AllbertAssist.Session.set_active_app/3` creates or updates the
   entry for `{user_id, session_id}`.
6. The entry stores `active_app`, preserves existing `working_memory`, updates
   monotonic `updated_at_ms`, and sets `expires_at_ms` using
   `sessions.scratchpad_ttl_minutes`.
7. CLI prints the normalized `user_id`, `session_id`, `active_app`, and
   remaining TTL.

The operation writes only ETS state. It does not write SQLite conversation
history, markdown memory, job rows, confirmations, traces, or settings.

### Show

For:

```sh
mix allbert.sessions show --user alice --session sess-1
```

The flow is:

1. CLI validates identity and session id.
2. Scratchpad returns the entry only if it exists and has not expired.
3. CLI prints `user_id`, `session_id`, `active_app`, remaining TTL,
   metadata keys, and `working_memory` key count.
4. CLI does not print raw working-memory values by default.

If the entry is missing or expired, the command returns a clear not-found
message and does not create a new entry.

### List

For:

```sh
mix allbert.sessions list --user alice
```

The flow is:

1. CLI validates identity.
2. Scratchpad sweeps or ignores expired entries.
3. CLI lists only entries whose `user_id` is Alice.
4. Each row includes session id, active app, remaining TTL, and key counts.

Alice and Bob can both use `sess-1`; list/show must keep those entries
separate.

### Clear Active App

For:

```sh
mix allbert.sessions clear-active-app --user alice --session sess-1
```

The flow is:

1. CLI validates identity and session id.
2. If the entry exists and is not expired, set `active_app: nil`.
3. Preserve `working_memory`.
4. Touch the entry and extend TTL.

If the entry is missing or expired, return not found. Do not create a new empty
entry for clear operations.

### Clear Session

For:

```sh
mix allbert.sessions clear --user alice --session sess-1
```

The flow is:

1. CLI validates identity and session id.
2. Delete the `{user_id, session_id}` entry from ETS if present.
3. Print whether an entry was removed.

This does not delete any durable thread, message, job, memory, confirmation, or
setting data.

## Runtime Ask Flow

For:

```sh
mix allbert.ask --user alice --session sess-1 "what should I look at?"
```

The flow is:

1. CLI parses `--session` and includes `session_id: "sess-1"` in the runtime
   request.
2. `AllbertAssist.Runtime.submit_user_input/1` normalizes `user_id`,
   `operator_id`, `thread_id`, `new_thread`, and `session_id` using v0.12
   rules.
3. Runtime selects or creates the SQLite conversation thread using v0.12
   semantics.
4. Runtime asks `AllbertAssist.Session.Scratchpad` for
   `{user_id: "alice", session_id: "sess-1"}`.
5. If the entry exists and active app is `:stocksage`, runtime sets
   `active_app: :stocksage`.
6. Runtime emits `allbert.input.received` with `active_app`.
7. Runtime persists the user message. Message metadata may include
   `session_id` and `active_app`; it must not include raw `working_memory`.
8. Runtime passes bounded thread context and `active_app` to
   `AllbertAssist.Agents.IntentAgent.respond/1`.
9. Runtime emits `allbert.agent.responded` with `active_app`.
10. Runtime records trace metadata and persists the assistant message with
    `active_app` context.
11. CLI prints `Session: sess-1` and `Active app: stocksage`.

If no scratchpad entry exists, the same request succeeds with
`active_app: nil`. Runtime must not create a scratchpad entry just because a
session id was supplied.

## Runtime Context Shape

The intent-agent request receives:

```elixir
%{
  text: "what should I look at?",
  channel: :cli,
  user_id: "alice",
  operator_id: "alice",
  thread_id: "thr_abc123",
  session_id: "sess-1",
  active_app: :stocksage,
  thread_context: %{
    thread_id: "thr_abc123",
    user_id: "alice",
    limit: 12,
    messages: []
  },
  metadata: %{},
  timeout_ms: 120_000,
  input_signal_id: "..."
}
```

`active_app` may be nil. The agent must treat nil as general Allbert context.
v0.14 does not require the agent to rank or route app actions differently.

## Decision And Approval Handoff Flow

When the intent agent builds a v0.11 decision:

1. Runtime passes `active_app` in the decision context.
2. `AllbertAssist.Intent.Decision` includes `active_app` in the decision map
   and trace metadata when present.
3. Resource Access Security Posture and Approval Handoff behavior remain
   unchanged.
4. A risky prompt in a StockSage session still stops at the same confirmation
   boundary as a general Allbert prompt.

`active_app` is not a grant. It must not make a remembered external-service
grant authorize app-specific financial workflow calls.

## Scheduled Job Flow

For a runtime prompt job with `session_id: "sess-1"`:

1. `AllbertAssist.Jobs.Runner` builds the runtime request with the stored
   `session_id`.
2. Runtime performs the same scratchpad lookup as CLI asks.
3. If an unexpired entry exists for `{job.user_id, "sess-1"}`, the runtime
   prompt receives `active_app`.
4. The job run stores normal v0.13 run metadata. It does not update or clear
   scratchpad.

Registered action jobs do not read scratchpad in v0.14. Their action context
continues to carry stored `session_id` as origin metadata only.

## Expiry And Sweep Flow

The scratchpad process uses monotonic milliseconds:

1. Entry writes store `inserted_at_ms`, `updated_at_ms`, and `expires_at_ms`.
2. Runtime reads and CLI show/list ignore expired entries.
3. Runtime reads of existing entries may touch them and extend TTL.
4. `mix allbert.sessions sweep` removes expired entries.
5. The supervised process also sweeps periodically using
   `sweep_interval_ms`.

Expired entries behave like missing entries:

- runtime `active_app` becomes nil
- CLI show returns not found/expired
- durable Allbert data is untouched

## Restart Flow

On scratchpad process crash or application restart:

1. OTP restarts `AllbertAssist.Session.Scratchpad`.
2. The process creates a fresh ETS table.
3. No prior entries are restored.
4. Runtime asks with the same `session_id` continue to work with
   `active_app: nil`.
5. SQLite threads/messages/jobs/settings and markdown memory remain intact.

This volatility is required by ADR 0014. Do not add persistence in v0.14.

## Working Memory Flow

`working_memory` is API-only in v0.14. It exists so later app/channel surfaces
can keep short-lived state without inventing another store.

Rules:

1. API callers pass a map.
2. The scratchpad performs a shallow merge into the existing entry.
3. Oversize payloads are rejected before storage.
4. Values are not logged, traced, or printed by CLI by default.
5. Expiry/restart drops the map.

Do not use `working_memory` for secrets, API keys, durable preferences,
financial records, thread summaries, or canvas documents.

## Manual Verification Matrix

### Active App Set And Ask

Commands:

```sh
mix allbert.sessions set-active-app --user alice --session sess-1 stocksage
mix allbert.ask --user alice --session sess-1 "hello from StockSage context"
```

Expected:

- sessions command prints `active_app=stocksage`
- ask output prints `Session: sess-1`
- ask output prints `Active app: stocksage`
- runtime response contains `active_app: :stocksage`
- no StockSage action is routed solely because active app is set

### General Context

Command:

```sh
mix allbert.ask --user alice "hello from general context"
```

Expected:

- ask output prints `Active app: none` or omits a non-nil app
- response contains `active_app: nil`
- v0.12 recent-thread behavior still works

### Alice/Bob Isolation

Commands:

```sh
mix allbert.sessions set-active-app --user alice --session shared stocksage
mix allbert.sessions show --user bob --session shared
```

Expected:

- Bob does not see Alice's session entry
- Bob's ask with `--session shared` uses `active_app: nil`

### Clear Active App

Commands:

```sh
mix allbert.sessions clear-active-app --user alice --session sess-1
mix allbert.ask --user alice --session sess-1 "back to general"
```

Expected:

- session entry remains inspectable
- `active_app` becomes nil
- working-memory key count is preserved

### Restart Volatility

Process:

1. Set active app for a session.
2. Restart the application or scratchpad process in a test/manual dev run.
3. Show the same session.

Expected:

- session entry is gone
- durable threads/messages/jobs/settings remain present
- asks with that session use `active_app: nil`

### Risky Prompt Regression

Command:

```sh
mix allbert.ask --user alice --session sess-1 "fetch https://example.com"
```

Expected:

- v0.11 Resource Access and Approval Handoff behavior still applies
- `active_app` appears only as context/metadata
- no hidden fetch, import, shell, package, or skill-script execution occurs

## Registered Action Flow

CLI subcommands and channel adapters reach the scratchpad through the same
registered Jido actions so the operator surface, future channels, and tests
share one durable contract. The actions run through
`AllbertAssist.Actions.Runner.run/3` and emit
`allbert.action.requested`/`allbert.action.completed` lifecycle signals.
Mutating session actions reuse the existing `:settings_write` permission class;
inspection uses `:read_only`.

For:

```sh
mix allbert.sessions set-active-app --user alice --session sess-1 stocksage
```

The runtime flow is:

1. CLI normalizes identity, session id, and app id.
2. CLI calls `AllbertAssist.Actions.Runner.run/3` with `set_active_app`,
   params `%{user_id: "alice", session_id: "sess-1", app_id: "stocksage"}`,
   and context `%{channel: :cli}`.
3. The action validates `app_id` through the v0.14 `AppId` allowlist using
   `String.to_existing_atom/1`. Unknown values return `:unknown_app`.
4. The action calls `AllbertAssist.Session.set_active_app/3`.
5. The runner emits standard action lifecycle signals with redacted
   parameters (no `working_memory` values, no monotonic-time internals).
6. CLI prints the normalized `user_id`, `session_id`, `active_app`, and
   remaining TTL from the action result.

`clear_active_app` mirrors the same shape and returns `:not_found` when no
entry exists. `show_session_scratchpad` returns `active_app`, sorted
`working_memory_keys`, remaining TTL, and metadata keys; raw working-memory
values never cross the action boundary.

Direct context-module calls (`AllbertAssist.Session.set_active_app/3`,
`clear_active_app/2`, `merge_working_memory/3`) remain available for
runtime internals, supervised tests, and v0.15+ app activation paths that
need lower-level access.

## CLI Flag Interactions

`mix allbert.ask` flag interactions in v0.14:

- `--session` is orthogonal to `--thread` and `--new-thread`. A session id
  does not imply a thread id; a thread id does not imply a session id. The
  runtime treats them as independent context fields and propagates both
  when present.
- `--user` and `--operator` continue to follow v0.12 rules. If they differ,
  the command fails before runtime submission and before any scratchpad
  read.
- `--session SESSION_ID` must be non-empty after trimming and at most 128
  characters; otherwise the command fails before runtime submission and
  before any scratchpad read.
- Omitted `--session` skips scratchpad lookup. The runtime sets
  `active_app: nil` and traces/signals match v0.13 shape exactly.

## Failure Modes

- Invalid `--user`/`--operator` mismatch: fail before any scratchpad
  read/write and before runtime submission.
- Blank or oversize `--session` (>128 chars): fail before any scratchpad
  read/write and before runtime submission.
- Unknown `app_id` (string not in the v0.14 `AppId` allowlist): the
  registered action returns `:unknown_app`; the CLI prints the redacted
  error; no entry is created.
- `:canvas_tiles` key in a `merge_working_memory` call: the context
  module returns `:reserved_key`; no merge occurs.
- Oversize `working_memory` payload (>64 KiB): the context module returns
  `:working_memory_too_large`; no merge occurs.
- Expired entry on read: `active_app` becomes nil; the entry is removed by
  the next sweep tick.
- Scratchpad process crash mid-write: OTP restarts the supervisor child
  with an empty ETS table; the in-flight write is lost; the failing turn
  surfaces a diagnostic; durable data (threads, jobs, settings, memory) is
  untouched.
- Scratchpad lookup error during a runtime turn: the runtime continues
  with `active_app: nil` and records a redacted diagnostic; it must not
  drop, retry, or abort the turn.
- `Application.get_env(:allbert_assist, AllbertAssist.Session.Scratchpad)`
  has `enabled?: false`: reads return `:not_found`; writes return
  `:disabled`; the runtime continues with `active_app: nil`.

## Out Of Scope

- Hosted accounts, sessions table, role table, or auth schema.
- Cross-machine or remote session sync.
- App registry, app-scoped action routing, app skill ranking, or
  StockSage domain behavior.
- AgentLive session sidebar, workspace shell, canvas state, or ephemeral
  UI surfaces.
- Automatic session creation from arbitrary `session_id` reads.
- Automatic markdown memory or conversation-history promotion from
  scratchpad contents.
- Generic dynamic atom creation from user input.
- Durable scratchpad audit table or `allbert.session.*` lifecycle
  signals.
- Working-memory value rendering in CLI, traces, signals, response
  payloads, or registered action results.
- New permission classes, new Security Central safety floors, confirmation
  requirements, resource grants, or operation classes. v0.14 uses existing
  `:settings_write` and `:read_only` permission decisions for its registered
  actions.

## Test Plan

- Scratchpad unit tests cover key normalization, create/update/get/list/
  clear, active-app normalization, TTL touch, expiry, sweep, restart
  volatility, oversize/`:canvas_tiles` rejection, blank/oversize session
  rejection, and alice/bob isolation.
- AppId allowlist tests cover known atoms, nil aliases, and unknown-string
  rejection.
- Settings tests cover `sessions.scratchpad_ttl_minutes` validation and
  defaults, plus the boot-time re-read rule.
- Registered action tests cover `set_active_app`/`clear_active_app`/
  `show_session_scratchpad` through `AllbertAssist.Actions.Runner.run/3`,
  including lifecycle signal emission and `working_memory_keys`-only
  output. Mutating actions prove `:settings_write`; show proves
  `:read_only`.
- Runtime tests cover no session, missing session, active session, expired
  session, scratchpad lookup failure, blank session input, oversize
  session input, and runtime touch behavior.
- CLI tests cover `mix allbert.ask --session` and all `mix allbert.sessions`
  commands, with explicit coverage for blank/oversize sessions, identity
  aliasing, invalid app ids, missing entries, and redacted output.
- Decision contract tests prove context-supplied `active_app` survives
  Decision validation and that model/agent output cannot overwrite it with
  an unknown atom.
- Observability tests prove `active_app` appears in signals/traces/
  decision/`action_log` only when non-nil and that raw `working_memory`
  never appears.
- Job integration tests prove runtime prompt jobs with `session_id` read
  scratchpad context without mutating it, and that registered-action jobs
  do not read scratchpad in v0.14.
- Regression tests prove existing v0.11-v0.13 confirmation/resource/job
  behavior remains unchanged.
