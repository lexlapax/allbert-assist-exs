# Allbert v0.02 Request Flow

This document is the living request-flow map for v0.02 Settings Central,
Secrets, Allbert Home, And Operator Profile.

It must be updated after every v0.02 milestone. When a section describes planned
behavior that is not implemented yet, mark it as planned. When the behavior is
implemented, replace the planned note with actual module/function/file names
and operator verification notes.

## Storage Flow

Implemented Allbert Home precedence:

1. Specific override when one exists, for example `ALLBERT_SETTINGS_ROOT`,
   `ALLBERT_MEMORY_ROOT`, `DATABASE_PATH`, or application config used by tests.
2. `ALLBERT_HOME`.
3. `ALLBERT_HOME_DIR`.
4. `~/.allbert`.

Implemented paths:

```text
<ALLBERT_HOME>/
  settings/
    settings.yml
    secrets.yml.enc
    .settings_key
    audit/YYYY-MM.md
  memory/
  db/
    allbert.sqlite3
  skills/
  cache/skills/
  tmp/
```

Implemented path API:

- `AllbertAssist.Paths.home/0`
- `AllbertAssist.Paths.ensure_home!/0`
- `AllbertAssist.Paths.settings_root/0`
- `AllbertAssist.Paths.memory_root/0`
- `AllbertAssist.Paths.db_path/0`
- `AllbertAssist.Paths.skills_root/0`
- `AllbertAssist.Paths.cache_root/0`
- `AllbertAssist.Paths.tmp_root/0`

Planned settings root precedence:

1. `config :allbert_assist, AllbertAssist.Settings, root: "..."`
2. `ALLBERT_SETTINGS_ROOT`
3. `AllbertAssist.Paths.settings_root/0`

Planned settings files:

```text
settings.yml
secrets.yml.enc
audit/YYYY-MM.md
```

Planned read flow:

1. `AllbertAssist.Settings.root/0`
2. `AllbertAssist.Settings.ensure_root!/0`
3. `AllbertAssist.Settings.read_user_settings/0`
4. `YamlElixir.read_from_file/2` without `atoms: true`
5. merge parsed user settings over built-in defaults
6. validate resulting shape
7. return value plus source-layer metadata

Planned module split:

- `AllbertAssist.Settings`: public facade.
- `AllbertAssist.Settings.Schema`: defaults, schema, validation, safe-write
  matching, dotted key helpers.
- `AllbertAssist.Settings.Store`: root resolution, read/merge/write, atomic
  settings writes, source-layer metadata.
- `AllbertAssist.Settings.YamlCodec`: YAML parsing and deterministic encoding.
- `AllbertAssist.Settings.Secrets`: encrypted local secret store.
- `AllbertAssist.Settings.Audit`: append-only audit writer, added in Milestone
  3.

Planned write flow:

1. `AllbertAssist.Settings.put/3`
2. validate key is known
3. validate key is in `safe_write_keys/0`
4. validate value
5. merge value into settings map
6. compute minimal user override map instead of dumping all defaults
7. encode deterministic YAML with `Ymlr.document!/2` and `sort_maps: true`
8. write temporary YAML file in settings root
9. rename temporary file over `settings.yml`
10. once Milestone 3 is complete, append audit record to `audit/YYYY-MM.md`
11. return updated resolved setting and audit diagnostics when relevant

Planned validation flow:

1. public calls accept dotted string keys, for example
   `operator.communication_style`
2. schema lookup rejects unknown keys
3. write authorization rejects read-only keys
4. value validation checks the exact type, enum, range, URI, timezone, or
   reference rule for the key
5. provider/model profile validation verifies cross-references after merging
6. validation failures return `{:error, {:invalid_setting, key, reason}}`
   without writing files

Planned atomic write rule:

1. validate before writing
2. encode to a temp file in the same directory
3. rename temp file over the target
4. preserve the previous target when validation, encode, temp write, or rename
   fails

Planned settings file policy:

- `settings.yml` stores only operator/user overrides.
- defaults stay in code.
- safe writes must not serialize raw secrets.

Planned secret write flow:

1. `AllbertAssist.Settings.Secrets.put_secret/3`
2. validate the `secret://...` reference
3. ask `PermissionGate` for `:settings_secret_write`
4. resolve the secret-store master key from
   `ALLBERT_SETTINGS_MASTER_KEY`, application config, or local development
   `<ALLBERT_HOME>/settings/.settings_key`
5. encrypt the secret payload into `secrets.yml.enc`
6. write the encrypted file atomically
7. store only the `secret://...` reference in `settings.yml`
8. once Milestone 3 is complete, append a redacted audit record
9. return redacted status such as `configured`

Planned secret store contract:

- `secrets.yml.enc` is YAML with `version`, `cipher`, `nonce`, `tag`, and
  `ciphertext`.
- `cipher` is `aes-256-gcm`.
- nonce is fresh 12-byte random data per write.
- associated data is `settings:v1`.
- ciphertext, nonce, and tag are base64 strings.
- decrypted plaintext is YAML with `version: 1` and a `secrets` tree.
- v0.02 writes only `secret://providers/<provider>/api_key` references.
- secret status values are `missing`, `configured`, `invalid_ref`, and
  `decrypt_failed`.
- raw secret reads are internal-only and denied to user-facing actions.

Tests and CI must set a temporary `ALLBERT_HOME` or specific temp roots. They
must never write to a real user's `~/.allbert`.

## Runtime Prompt Flow

Planned route examples:

- "show settings" -> `list_settings`
- "what is my timezone setting?" -> `read_setting`
- "explain operator.timezone" -> `explain_setting`
- "set my communication style to concise" -> `update_setting`
- "show provider profiles" -> `list_provider_profiles`
- "configure my OpenAI API key" -> start credential flow guidance
- "set my OpenAI API key to test-key" -> refuse raw prompt-secret write and
  redact diagnostics

Planned function flow:

1. channel calls `AllbertAssist.Runtime.submit_user_input/1`
2. runtime emits `allbert.input.received`
3. `AllbertAssist.Agents.IntentAgent.respond/1` recognizes explicit settings
   prompts
4. selected settings action asks `PermissionGate` for the relevant permission
5. action calls `AllbertAssist.Settings`
6. settings store reads/writes `settings.yml`
7. natural-language credential prompts never write raw secrets from prompt text
8. successful writes append redacted audit records
9. runtime emits `allbert.agent.responded`
10. tracing records setting key, action, permission decision, redacted secret
   status, and audit path
   when tracing is enabled

Planned settings actions:

- `AllbertAssist.Actions.Settings.ListSettings`
- `AllbertAssist.Actions.Settings.ReadSetting`
- `AllbertAssist.Actions.Settings.UpdateSetting`
- `AllbertAssist.Actions.Settings.ExplainSetting`
- `AllbertAssist.Actions.Settings.ListProviderProfiles`
- `AllbertAssist.Actions.Settings.SetProviderCredential`

Planned runtime safety rule:

The runtime action may say how to configure a provider key or point to CLI and
LiveView credential entry. It must not accept a raw API key inside normal
natural-language prompt text as a valid secret write.

## CLI Flow

Planned Mix task:

`Mix.Tasks.Allbert.Settings`

Planned commands:

```sh
mix allbert.settings list
mix allbert.settings get operator.timezone
mix allbert.settings explain operator.timezone
mix allbert.settings set operator.communication_style concise
mix allbert.settings providers list
mix allbert.settings providers set-key openai
```

Planned function flow:

1. task starts the OTP app
2. task parses subcommand and arguments
3. task calls `AllbertAssist.Settings` or settings actions
4. `providers set-key` reads the key from piped stdin or an interactive hidden
   prompt, not from a plain positional argument
5. task prints status, value, source layer, diagnostics, and audit path when
   relevant
6. task never prints raw secret values

Planned CLI output contract:

- success commands exit `0`
- invalid usage, validation failure, read-only write, parse failure, encrypt
  failure, or decrypt failure exits non-zero
- `list` prints key, value or redacted status, source, and writable state
- `get KEY` prints key, value or redacted status, and source
- `explain KEY` prints source layers and diagnostics
- `set KEY VALUE` prints written key, source `operator`, and audit path once
  M3 exists
- `providers set-key PROVIDER` prints provider and `configured`, never the key

## LiveView Flow

Planned route:

`live "/settings", SettingsLive`

Planned module:

`AllbertAssistWeb.SettingsLive`

Planned DOM IDs:

- `#settings-list`
- `#settings-key`
- `#settings-value`
- `#settings-save`
- `#settings-explanation`
- `#settings-diagnostics`
- `#provider-profiles`
- `#provider-key-form`

Planned function flow:

1. `SettingsLive.mount/3` loads settings list and selected setting
2. template renders list, selected value, source, diagnostics, provider
   profiles, model profiles, and redacted credential status
3. `"select_setting"` event reads one setting and explanation
4. `"save_setting"` event validates and writes safe keys
5. `"save_provider_key"` event writes a provider secret through
   `AllbertAssist.Settings.Secrets`
6. successful writes show updated value, redacted credential status, and audit
   reference
7. failed writes show diagnostics without changing `settings.yml` or
   `secrets.yml.enc`

Planned LiveView event contract:

- `select_setting`: `%{"key" => dotted_key}`
- `save_setting`: `%{"setting" => %{"key" => key, "value" => value}}`
- `save_provider_key`: `%{"provider" => provider, "api_key" => raw_key}`

After `save_provider_key`, the LiveView must clear the form value, refresh
provider status, and render no raw secret value in HTML or assigns.

## Audit And Secret Flow

Planned audit path:

```text
audit/YYYY-MM.md
```

Planned fields:

- timestamp
- key
- old value, redacted when sensitive
- new value, redacted when sensitive
- actor/operator id
- channel
- source signal id when available
- permission decision
- validation result
- audit version

Planned audit markdown entry:

```markdown
## 2026-05-01T12:00:00Z operator.communication_style

- actor: local
- channel: cli
- source_signal_id: sig_123
- permission: allowed
- validation: ok
- old: concise
- new: detailed
- audit_version: 1
```

Planned secret audit entries record status transitions only:

```markdown
## 2026-05-01T12:00:00Z secret://providers/openai/api_key

- actor: local
- channel: cli
- source_signal_id: none
- permission: allowed
- validation: ok
- old: missing
- new: configured
- audit_version: 1
```

Planned redaction:

Redact keys containing `secret`, `token`, `password`, `api_key`,
`private_key`, or `credential`, plus every `secret://...` reference.

Planned secret files:

- `settings.yml` stores only secret references.
- `secrets.yml.enc` stores encrypted secret payloads.
- audits, traces, CLI output, LiveView, and logs report only redacted status.

## Permission Flow

Planned v0.02 permission class:

- `:settings_write`
- `:settings_secret_write`
- `:settings_secret_read`

Planned decisions:

- safe settings writes: `:allowed`
- explicit provider credential writes: `:allowed`
- raw user-facing secret reads: `:denied`
- deployment config writes: `:denied`
- unknown settings writes: `:denied`
- read-only settings writes: `:denied`

Settings reads use `:read_only`.

## Trace Flow

When tracing is enabled, settings turns should include:

- input signal id
- response signal id
- selected settings action
- setting key
- permission decision
- redacted secret status, when relevant
- validation result
- audit path for successful writes
- diagnostics for parse/write/validation failures

Trace default resolution is planned to use:

1. per-request/session trace flag, for example `--trace`
2. `ALLBERT_TRACE_ENABLED`, when set
3. application config trace override, when set
4. `runtime.trace_default`
5. built-in default `disabled`

Environment/config trace overrides must continue to work.

Allowed `runtime.trace_default` values:

- `disabled`
- `enabled`
- `denied_only`
