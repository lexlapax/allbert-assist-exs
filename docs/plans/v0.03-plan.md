# Allbert v0.03 Agent Skills Substrate

## Purpose

v0.03 should make Allbert a skills-compatible agent client by adding the
substrate for standard `SKILL.md` discovery, parsing, validation, trust, and
progressive disclosure.

The origin note wants Allbert to find skills, explain which capability it would
use, ask for permission when needed, and eventually help create new skills. The
vision turns that into an Elixir/Jido shape: agents decide, Jido actions perform
bounded side effects, permissions are checked at the action boundary, and
skills remain readable bundles around those capabilities.

The broader skill ecosystem already has an emerging standard. Agent Skills are
directories containing a required `SKILL.md` file with YAML frontmatter and
markdown instructions. They can also include `scripts/`, `references/`, and
`assets/`, and clients are expected to use progressive disclosure: load only
name and description during discovery, load the full skill body on activation,
and load bundled resources only when needed.

Allbert should adopt that format rather than inventing an incompatible
`allbert-skills` format. v0.03 is the foundation layer: parse standard Agent
Skills, validate them, track trust and source, inventory resources, and activate
instructions. Optional Allbert metadata can be parsed and reported, but using it
to drive Jido action execution belongs in v0.04.

This is still not autonomous skill creation or arbitrary code loading. It is
the compatibility layer that makes those future features safer.

## External Grounding

The v0.03 design is informed by:

- Agent Skills overview and specification: `https://agentskills.io/home` and
  `https://agentskills.io/specification`
- Agent Skills client implementation guidance:
  `https://agentskills.io/client-implementation/adding-skills-support`
- Agent Skills creator guidance:
  `https://agentskills.io/skill-creation/best-practices`
- Agent Skills script guidance:
  `https://agentskills.io/skill-creation/using-scripts`
- skills.sh catalog, CLI, and API docs: `https://skills.sh/`,
  `https://skills.sh/docs`, and `https://skills.sh/docs/api`

## Why This Is Next

v0.01 proved the local assistant loop:

- signal-first runtime boundary
- primary intent agent
- named Jido actions
- central permission gate
- markdown memory
- markdown traces
- shared CLI and LiveView entrypoints

The next tempting features are richer memory, scheduled jobs, channels,
external network access, and shell execution. Each of those depends on a common
capability contract. Without it, jobs, channels, and agents would each need
their own way to discover skills, activate instructions, reason about
permissions, and explain what was run.

So v0.03 should build the first of two layers:

- Agent Skills compatibility, so Allbert can participate in the wider skill
  ecosystem.

v0.04 should then add the second layer:

- Allbert capability translation, so selected skills can bind safely to Jido
  actions, permission classes, traces, and memory effects.

## Current Baseline

- `AllbertAssist.Skills` is a static in-code list of v0.01 capabilities.
- v0.02 is expected to provide settings-backed scan paths, trust decisions, and
  enablement defaults for skills.
- `AllbertAssist.Agents.IntentAgent` hard-codes the available action modules in
  `action_modules/0` and deterministic route handling.
- `list_skills` and `read_skill` read from the static skill list.
- `AllbertAssist.Runtime.signal_types/0` already names
  `allbert.action.requested` and `allbert.action.completed`, but v0.01 does
  not yet make those signals the standard action lifecycle path.
- `AllbertAssist.Trace` records selected action metadata, but it does not yet
  record a selected skill manifest, skill version, or skill declaration path.
- `AllbertAssist.Memory` already has a `skills` category, but current durable
  memory entries are append-style records rather than stable skill
  declarations.

## Core Decision

Allbert should use the Agent Skills format as both its import/export format and
its native authoring format.

An installed skill is a directory with `SKILL.md` and optional supporting
folders. Allbert parses that into an internal `AllbertSkill` record. If the
skill includes Allbert metadata, v0.03 should preserve and validate it enough to
show diagnostics, but it should not use that metadata to execute actions yet.

Allbert-specific skills should therefore be standard Agent Skills plus a small
namespaced metadata overlay. Prefer fields that stay compatible with the
specification:

```markdown
---
name: append-memory
description: Save user-approved memory into Allbert's markdown memory. Use when the user asks Allbert to remember a fact, preference, or project note.
compatibility: Allbert v0.03+. Requires the built-in append_memory Jido action.
allowed-tools: allbert:action:append_memory
metadata:
  allbert.kind: capability
  allbert.version: "0.3.0"
  allbert.actions: append_memory
  allbert.permissions: memory_write
  allbert.confirmation: not_required
  allbert.memory-effects: writes_markdown_memory
  allbert.trace-effects: records_selected_skill,records_permission_decision
---

## Workflow

1. Confirm the memory is user-approved or low-risk under Allbert's memory
   policy.
2. Use the `append_memory` Allbert action.
3. Report the markdown path and category.
```

Notes:

- Agent Skill names use hyphens. Allbert action names may remain snake case.
- The frontmatter `metadata` map should use namespaced string keys because the
  Agent Skills spec reserves only a small set of core fields.
- `allowed-tools` is treated as advisory and experimental. It must never grant
  permission by itself.
- If the ecosystem later standardizes richer permissions, Allbert can translate
  to or from those fields without changing the directory format.

## Skill Kinds

The registry should classify skills into these kinds:

- `:instruction`: standard Agent Skill with no Allbert action metadata. It can
  be discovered, read, and activated as context. It cannot run new side effects.
- `:workflow`: instruction skill with checklists, references, templates, or
  scripts. In v0.03, scripts are inspectable resources only.
- `:capability_candidate`: skill whose Allbert metadata appears to map to Jido
  actions and permission classes, but is not executable through the skill layer
  until v0.04.
- `:native_action`: built-in Allbert/Jido capability represented as an Agent
  Skill wrapper for user-facing discovery and activation. In v0.03 this is
  descriptive; v0.04 wires it to action execution.
- `:external_candidate`: skill discovered through skills.sh or an untrusted
  source but not yet installed or enabled.

This distinction lets Allbert import ecosystem skills without pretending every
skill is safe executable code.

## Non-Goals

- No autonomous skill creation.
- No arbitrary code loading from skill files.
- No scripting engine.
- No shell command execution.
- No external network calls.
- No embeddings or vector search.
- No additional user channels.
- No skill-driven Jido action execution.
- No action lifecycle runner.
- No generic skill creation helper beyond documentation and examples.
- No automatic `npx skills add` or skills.sh API calls without explicit
  external-network confirmation.
- No scheduled job runtime beyond what is needed to keep the registry design
  compatible with future jobs.

## Discovery Scopes

Allbert should scan multiple scopes with deterministic precedence and trust
metadata:

- Built-in: packaged with the app, for example
  `apps/allbert_assist/priv/skills/`.
- Project native: `./.allbert/skills/`.
- Project interoperable: `./.agents/skills/`.
- User native: `<ALLBERT_HOME>/skills/`, defaulting to
  `~/.allbert/skills/`.
- User interoperable: `~/.agents/skills/`.
- Imported cache: `<ALLBERT_HOME>/cache/skills/` for skills imported from
  skills.sh, git, or a local path after user confirmation.

Project skills should be gated by a trust decision before they can affect the
agent catalog. A newly cloned repository should not be able to silently inject
instructions into Allbert.

User-owned and imported skills should resolve through `AllbertAssist.Paths` so
they follow the same Allbert Home rules as settings, secrets, memory, and the
local database. Project-local `./.allbert/skills/` remains a separate
trust-gated scope because it is owned by the current workspace, not by the
operator's global Allbert home.

The existing markdown memory `skills` category should not become the install
location for executable skill folders. Use memory for skill notes, activation
traces, review records, and user preferences about skills. Installed skills
should remain ordinary directories.

## Translation Model

The runtime should follow this pipeline:

```text
Skill directory
  -> SKILL.md parser
  -> AgentSkillSpec
  -> AllbertSkill record
  -> inert CapabilityContract draft, when Allbert metadata exists
  -> activation context
  -> trace metadata
```

Suggested internal structs:

- `AllbertAssist.Skills.AgentSkillSpec`
  - source path
  - name
  - description
  - license
  - compatibility
  - allowed tools
  - metadata
  - markdown body
  - resource inventory
- `AllbertAssist.Skills.Skill`
  - normalized name
  - title/display name
  - source scope
  - trust status
  - kind
  - activation mode
  - original spec
  - diagnostics
- `AllbertAssist.Skills.CapabilityContract`
  - action names
  - permission classes
  - confirmation policy
  - memory effects
  - trace effects
  - script/resource policy

In v0.03, capability contracts are drafts. The registry validates them enough to
surface diagnostics against:

- known Jido action names
- known permission classes from `PermissionGate.permission_classes/0`
- supported confirmation policies
- resource and script policy
- trust status
- duplicate name precedence

v0.04 decides how valid draft contracts become executable through the action
runner.

## Registry Rules

- Built-in skills should ship with the app as stable markdown declarations.
- User-owned skills should live in Agent-Skills-compatible directories, not in
  opaque database records.
- Every `SKILL.md` must validate the required Agent Skills frontmatter before
  it enters the catalog.
- The model-facing catalog should include only enabled and trusted skills.
- A capability candidate may reference only known Allbert action names.
- A skill may not name an arbitrary Elixir module and cause it to execute.
- A skill may not execute bundled scripts in v0.03.
- `allowed-tools` and `metadata.allbert.*` are hints to the registry; the
  permission gate remains the source of truth.
- Invalid or untrusted skill files should be visible in registry diagnostics
  without crashing the assistant loop.
- Duplicate skill names should resolve deterministically. Project-level skills
  may override user-level skills only when the project is trusted; otherwise
  they should be hidden or marked pending.

## Activation Model

Allbert should implement progressive disclosure explicitly:

- Catalog: at runtime startup or first use, load only skill name, description,
  scope, trust state, and kind.
- Activation: when the user or intent agent selects a skill, load the body of
  `SKILL.md` and return it wrapped as skill context.
- Resources: enumerate `scripts/`, `references/`, and `assets/` during
  activation, but load a file only when a selected skill's instructions call for
  it and the permission model allows reading it.

Model-driven activation can remain deterministic in v0.03 through an internal
`activate_skill` action selected by explicit prompts. More autonomous
model-driven selection belongs after the registry and trust model are stable.
User-explicit activation should be supported through plain prompts at first, and
later through CLI or UI syntax such as `/skill append-memory` once the
interface design catches up.

For v0.03, activation changes context and trace metadata. It should not grant
new execution powers.

## skills.sh And Import Policy

skills.sh matters in two roles:

- discovery catalog: search, leaderboard, install counts, official sources, and
  detail pages
- risk signal: audit endpoints can return pass/warn/fail status and risk
  levels from security partners

Allbert should not depend on skills.sh for local operation. v0.03 should work
with local folders and skills already installed by other tools. skills.sh
integration can be designed in v0.03 but should remain confirmation-gated
because it requires external network access and may write files.

Planned import flow:

1. User asks for a skill or says Allbert needs a capability.
2. Allbert searches local installed skills first.
3. If nothing fits, Allbert can propose an external search.
4. External search uses `Req` and the skills.sh API only after confirmation.
5. Results are ranked by relevance, source reputation, install count, official
   status, duplication status, and audit risk.
6. Installation/import writes to an Allbert-controlled skill cache or delegates
   to `npx skills add` only after explicit user confirmation.
7. Imported skills are disabled until parsed, validated, and trusted.

The Skills CLI is useful for interoperability, but Allbert should not make
Node/npm a hard runtime dependency. The registry should read standard skill
directories directly. A future import adapter can call `npx skills` when
available.

## Future Allbert Skill Creation Rule

Allbert-specific skills should eventually be created as Agent Skills first.
This is a design rule for v0.04 and later, not a v0.03 implementation
requirement. The creation workflow should be:

1. Start from a real repeated task, trace, correction, or user preference.
2. Decide whether the skill is instruction-only, workflow, or action-backed.
3. Draft a standard `SKILL.md` with a precise description that says when to use
   it.
4. Add Allbert metadata only if the skill needs known Jido actions or permission
   semantics.
5. If a new side effect is needed, create the Elixir `Jido.Action` and
   permission contract first; then create the skill wrapper.
6. Add small test prompts:
   - should activate
   - should not activate
   - should refuse or stay read-only
7. Run the skill against the tests and inspect traces.
8. Iterate the description and instructions based on actual false positives,
   false negatives, failures, and user corrections.
9. Promote the skill from draft to trusted only after validation passes.

This gives Allbert a path to self-improving skills later: traces and memory can
suggest a skill draft, but the runtime still validates and asks before enabling
it.

## Implementation Order

### Milestone 1: Compatibility Decision And ADR

Status: planned.

Implementation tasks:

- Record the decision to adopt Agent Skills as Allbert's import/export and
  native authoring format.
- Document the Allbert metadata overlay and translation rules.
- Document discovery scopes, trust precedence, and duplicate handling.
- Confirm the v0.02 direct YAML parser dependency supports `SKILL.md`
  frontmatter.

Acceptance criteria:

- The plan and ADR say clearly that Allbert is skills-compatible.
- No separate incompatible allbert-only skill format is introduced.

### Milestone 2: Agent Skill Parser And Validator

Status: planned.

Implementation tasks:

- Add a parser for `SKILL.md` YAML frontmatter and markdown body.
- Validate required Agent Skills fields:
  - `name`
  - `description`
- Validate Agent Skills naming constraints.
- Preserve optional fields:
  - `license`
  - `compatibility`
  - `metadata`
  - `allowed-tools`
- Inventory optional resource directories:
  - `scripts/`
  - `references/`
  - `assets/`
- Add registry diagnostics for malformed YAML, missing fields, oversized
  descriptions, invalid names, and parent-directory mismatches.

Acceptance criteria:

- A valid standard Agent Skill parses without Allbert-specific metadata.
- A valid Allbert capability skill parses with namespaced metadata.
- Invalid skills are reported as diagnostics rather than exceptions.

### Milestone 3: Discovery, Trust, And Registry

Status: planned.

Implementation tasks:

- Add registry modules, tentatively:
  - `AllbertAssist.Skills.Registry`
  - `AllbertAssist.Skills.AgentSkillSpec`
  - `AllbertAssist.Skills.Skill`
  - `AllbertAssist.Skills.CapabilityContract`
- Scan built-in, project, user, and interoperable skill paths.
- Track source scope, path, trust status, enabled/disabled status, and
  diagnostics.
- Implement deterministic duplicate handling.
- Keep `AllbertAssist.Skills` as a small facade so existing actions keep a
  stable call site.

Acceptance criteria:

- `AllbertAssist.Skills.list/0` returns registry-loaded skills.
- `AllbertAssist.Skills.get/1` reads by name from the registry.
- Project skills are hidden or pending unless trusted.
- Tests cover precedence, trust gating, disabled skills, malformed files, and
  duplicate names.

### Milestone 4: Built-In Allbert Skill Pack

Status: planned.

Implementation tasks:

- Seed built-in Agent Skill wrappers for the current v0.01 actions:
  - `direct_answer`
  - `append_memory`
  - `read_recent_memory`
  - `list_skills`
  - `read_skill`
  - `plan_shell_command`
  - `external_network_request`
- Use Agent Skill names with hyphens while mapping to snake-case action names.
- Include Allbert metadata as descriptive, non-executable contract data.
- Include instructions that preserve v0.01 safety boundaries.

Acceptance criteria:

- Each existing capability has one readable skill declaration.
- The built-in skill pack can be listed and read through the registry.
- No built-in skill grants command execution or external network access.

### Milestone 5: Activation And Progressive Disclosure

Status: planned.

Implementation tasks:

- Add an `activate_skill` action that returns wrapped skill instructions and a
  resource inventory.
- Update `list_skills` and `read_skill` to use registry data.
- Keep instruction activation separate from action execution.
- Record activated skill metadata in traces.

Acceptance criteria:

- "What skills are available?" lists trusted registry skills.
- "Read skill append-memory" displays the standard skill instructions.
- "Activate skill append-memory" returns wrapped skill content and resources.
- Activation does not execute scripts or external tools.

### Milestone 6: Operator Loop And Documentation

Status: planned.

Implementation tasks:

- Update the request-flow documentation for the registry and activation path.
- Add operator tests for CLI and LiveView.
- Document how to install third-party skills using external tools and where
  Allbert scans for them.
- Document why skills.sh search/install is confirmation-gated.
- Update the roadmap with v0.03 status and v0.04 as the action-backed skill
  milestone.

Acceptance criteria:

- `mix precommit` passes.
- The CLI can list skills, read a skill, activate a skill, and show
  registry/trace metadata.
- The LiveView can perform the same loop.
- Invalid skill declarations can be inspected without breaking the assistant.

## Operator Test Sketch

Use a temporary Allbert Home so user-owned skill declarations, imported caches,
memory, settings, and traces are easy to inspect:

```sh
export ALLBERT_HOME=/tmp/allbert-v004-demo
export ALLBERT_TRACE_ENABLED=true
rm -rf "$ALLBERT_HOME"
```

Then run:

```sh
mix allbert.ask --trace "what skills are available?"
mix allbert.ask --trace "read skill append-memory"
mix allbert.ask --trace "activate skill append-memory"
mix allbert.ask --trace "read skill from an invalid fixture"
```

Expected result:

- Skill list/read/activate output comes from Agent-Skills-compatible
  `SKILL.md` directories.
- Existing v0.01 memory and command behavior remains unchanged.
- Traces show activated skill, source scope, trust status, diagnostics, and
  resource inventory.
- No skill activation executes a script or Jido action.

## Definition Of Done

v0.03 is done when:

- Allbert can parse and validate standard Agent Skills directories.
- Built-in and user/project skills are loaded from predictable scopes with
  trust and precedence rules.
- Allbert-specific capability metadata is a compatibility-preserving overlay,
  not a forked skill format, and remains inert in v0.03.
- Instruction-only skills can be activated without gaining new side effects.
- Traces record selected skill, source, trust, diagnostics, and resources.
- CLI and LiveView expose the same registry-backed list/read/activate behavior.
- No new unsafe execution capability has been added.
- The roadmap is updated with v0.03 completion status and v0.04 action-backed
  skill work.
