# Allbert v0.08 Request Flow

v0.08 turns shell execution from an inert planning topic into a confirmed,
policy-bounded, local process capability. The request flow preserves the
existing Allbert architecture: channels translate user interaction into
runtime calls, Jido agents/actions own capability selection and side effects,
Security Central evaluates policy, Settings Central stores operator policy,
durable confirmations pause sensitive work, and traces/audits explain what
happened.

Post-v0.10 documentation framing: this request flow is a historical local
resource access special case for host process execution. It is not updated to
claim the generalized resource access contract now planned in v0.10; it only
provides the local command/cwd/path/env/output foundation that later docs
reference.

## Actors And Channels

- User/operator: the single local operator for v0.08.
- Origin channel: CLI, LiveView, or future channel that created the execution
  request.
- Resolver channel: CLI, LiveView, or future channel that approved, denied, or
  expired the request.
- Intent agent: routes prompts to `plan_shell_command` or `run_shell_command`.
- `run_shell_command`: registered Jido action and only command execution
  entrypoint.
- `mix allbert.exec`: deterministic CLI command-spec path for local operator
  testing; it never spawns directly.
- `mix allbert.confirmations`: CLI approval/denial/display path for durable
  confirmation records and shell result metadata.
- Security Central: evaluates `:command_execute` with Level 1 sandbox policy.
- Settings Central: stores `execution.local.*`, permission, confirmation, and
  redaction policy.
- `AllbertAssist.Execution.Policy`: resolves `execution.local.*` into Level 1
  policy for allowed roots, commands, profiles, env, timeout, and output caps.
- `AllbertAssist.Execution.CommandSpec`: normalizes executable/argv/cwd/env
  inputs and denies disabled, out-of-root, blocked, unprofiled, network-shaped,
  or inline-eval commands before any runner can spawn a process.
- `AllbertAssist.Execution.LocalRunner`: launches the approved process after
  policy and confirmation succeed. The first adapter uses `System.cmd/3` with
  explicit argv, command-local cwd, allowlisted env, `stderr_to_stdout: true`,
  timeout, and bounded output capture.

## Planning-Only Flow

1. The user asks what command could be run or asks for a shell plan.
2. The runtime submits input through `AllbertAssist.Runtime.submit_user_input/1`.
3. The intent agent selects `plan_shell_command`.
4. The action returns an inert plan and keeps command execution unavailable.
5. Trace metadata records command-planning permission and no side effect.

This flow must not create a confirmation or spawn a process.

## Direct Execution Request Flow

1. The user asks Allbert to run a shell command, or `mix allbert.exec` creates
   a command spec directly.
2. The channel calls runtime/action boundaries; it does not spawn commands.
3. The intent agent, `mix allbert.ask`, or `mix allbert.exec` selects
   registered action `run_shell_command`.
4. `run_shell_command` validates and normalizes:
   - executable name or path
   - argv list
   - cwd
   - path-like operands
   - stdin mode
   - timeout
   - output limit
   - env allowlist request
   - command class and command profile, when configured
5. The action builds a command spec and classifies the command conservatively.
6. Security Central evaluates `:command_execute` with action, command spec,
   origin channel, actor, selected skill when present, allowed roots, command
   allowlist or operator command profile, path operand policy, blocked args,
   env policy, timeout, output cap, and safety floors.
7. If policy returns `:denied`, the action returns a denial result and nothing
   runs.
8. If policy returns `:needs_confirmation`, the action writes a durable pending
   confirmation with redacted command spec and no side effect.
9. The user approves or denies from CLI or `/settings`.
10. Approval re-reads the pending record, checks expiration and resolver
    channel policy, and re-checks Security Central with confirmation context.
11. If the re-check denies, the confirmation resolves as denied and nothing
    runs.
12. If the re-check allows execution for target action `run_shell_command`,
    `approve_confirmation` calls the registered action through
    `AllbertAssist.Actions.Runner.run/3` with approval context and the durable
    `resume_params_ref`.
13. The runner executes only the normalized executable plus argv list, in the
    policy-approved cwd, with allowlisted env, timeout, and output caps.
14. The action resolves the confirmation with `target_resumed?: true` and
    stores structured result metadata under `operator_resolution.target_status`
    and `operator_resolution.target_result`.
15. Runtime output, traces, CLI, LiveView, and audit records show the result
    without raw secrets or unbounded stdout/stderr.

## Operator Surface Flow

- `mix allbert.exec --cwd "$WORKSPACE" -- ls -la` creates a pending
  `run_shell_command` confirmation when policy allows the command spec.
- `mix allbert.ask "run pwd"` routes through the runtime and intent agent to
  the same action boundary.
- `mix allbert.confirmations approve CONFIRMATION_ID --reason "smoke"` resumes
  the shell target once and prints command/result metadata.
- `mix allbert.confirmations list --resolved` and `/settings` show command,
  cwd, sandbox level, timeout, output cap, exit status, timeout/truncation
  flags, denial reason, and output preview from the durable record.
- Runtime traces include a bounded `Shell Command Metadata` section.
- Execution audit records are appended under `<ALLBERT_HOME>/execution/audit`
  for requested, approved, denied, succeeded, failed, and timed-out command
  events.

## Denial Flow

Denials happen before confirmation when possible.

Examples:

- local execution disabled
- missing or empty executable
- shell string instead of executable plus args
- cwd outside `execution.local.allowed_roots`
- path-like operand outside `execution.local.allowed_roots`
- command not in `execution.local.allowed_commands` and not covered by an
  explicit `execution.local.command_profiles` entry
- mutating/destructive command without an explicit allowed profile
- blocked arg pattern, such as `sed -i`, `find -delete`, `sh -c`, `python -c`,
  command substitution, or backgrounding
- network-shaped command, such as `curl`, `wget`, or interpreter eval that
  would fetch from the internet
- timeout above `execution.local.max_timeout_ms`
- output limit above `execution.local.max_output_bytes`
- requested env var outside `execution.local.env_allowlist`
- hard safety floor from Security Central

Denied results should explain which policy blocked execution and should include
the sandbox level, command summary, cwd summary, and no-output/no-side-effect
status.

## Timeout And Output Flow

The local runner must treat long-running or noisy commands as controlled
failures:

1. Start the child process with timeout and output cap metadata.
2. Capture stdout/stderr without writing raw child output directly to the
   operator terminal or logs.
3. If output exceeds the cap, stop recording additional bytes and mark the
   result `truncated?: true`.
4. If timeout expires, terminate the child where supported and mark
   `timed_out?: true`.
5. Return exit status, timeout status, output sizes, redacted previews, and
   diagnostics.
6. Trace and audit the limit that fired.

## Adapter-Unavailable Compatibility Flow

v0.08 must not remove v0.07 `adapter_unavailable` semantics. Approval can
resume only the target action named in the pending record. Shell execution
requests may resume because v0.08 adds the shell adapter. External network
requests still resolve as `adapter_unavailable` until v0.10 adds a registered
network adapter.

Operator-facing output should make the distinction clear:

- shell command approved and executed through Level 1 sandbox
- shell command approved but denied on re-check
- unrelated adapter approved but not executed because its adapter is still not
  implemented

Implementation note: as of M3, only the shell target resumes. External network
targets keep the v0.07 `adapter_unavailable` status and remain a v0.10 handoff.

## Trace And Audit Requirements

Every pending, denied, approved, timed-out, failed, or succeeded shell request
should include:

- confirmation id, when present
- origin channel and resolver channel
- selected action
- permission
- sandbox level
- executable summary
- argv summary with secret-like values redacted
- command profile, when present
- cwd summary
- allowed-root match
- path operand policy result
- env allowlist summary, not env values
- timeout and output cap
- Security Central decision before pending creation
- Security Central decision after approval
- runner backend, such as `local_process`
- exit status or timeout status
- stdout/stderr byte counts
- whether stderr was merged into captured output
- redacted output preview
- denial or failure reason

Implemented audit locations:

- confirmation lifecycle audit: `<ALLBERT_HOME>/confirmations/audit/YYYY-MM.md`
- shell execution audit: `<ALLBERT_HOME>/execution/audit/YYYY-MM.md`

## Implementation Checklist

- Keep LiveViews thin; they render confirmation/action results and call action
  boundaries.
- Keep CLI tasks thin; they parse operator args and call action boundaries.
- Do not mutate confirmation records directly from channels.
- Do not invoke `System.cmd/3`, Erlang ports, or any runner from a channel,
  intent agent, skill parser, skill registry, or skill metadata module.
- Use temporary Allbert homes and temporary cwd roots in tests.
- Do not add container dependencies in v0.08.
