# Allbert v0.24 Request Flow — Objective Runtime Foundation

## Status

Status: planned (fourth validation pass 2026-05-17 — post-v0.23 implementation).

This pass updates the flows to match the v0.24 plan's fourth-pass
specs: hybrid proposer cycle, `continue` advisory `:still_blocked`
return, `:abandoned` terminal lifecycle, confirmation rendering
snapshot + stale warning, web-side `SignalBridge` for LiveView
reactivity, and per-section inline `### Objective` trace rendering.

This document describes how a runtime turn flows through the objective
engine. It complements `docs/plans/v0.24-plan.md` (which owns modules,
schema, migrations, signal payload contracts, milestones, and tests).

Four flows are documented end-to-end:

1. Single-step objective (one action; one confirmation).
2. Multi-step objective (two sequential actions; `parent_step_id`
   populated).
3. Cooperative cancellation (including cancel-then-approve race).
4. Resume via `mix allbert.objectives continue`.

The Milestone Update Log at the bottom is updated as each milestone
lands.

## Pre-Flight: Allbert DB Migrations And Substrate Posture

Use a disposable Allbert Home and apply all four v0.24 migrations
(plus the v0.23 substrate is required):

```sh
export SMOKE_HOME=$(mktemp -d /tmp/allbert-v024-smoke.XXXXXX)
ALLBERT_HOME="$SMOKE_HOME" mix cmd --app allbert_assist mix ecto.migrate.allbert
```

`mix ecto.migrate.allbert` applies:

- v0.24 core migrations: `objectives`, `objective_steps`,
  `objective_events`; column additions on `confirmations` and
  `scheduled_jobs`.
- v0.24 StockSage plugin migration: `objective_id`/`step_id` columns on
  `stocksage_analysis_queue` and `stocksage_analyses`, plus btree index
  on `stocksage_analyses.objective_id`.

v0.23 `AllbertAssist.JidoBacked.Supervisor` and the
`AllbertAssist.JidoBacked` behaviour are prerequisites; v0.24 appends
`Objectives.Engine.Agent` under that supervisor at app boot through
`:extra_children` and adds it to the JidoBacked debug-agent list.

### Substrate Reference

| Operator switch | Effect |
|---|---|
| `mix allbert.settings set objectives.enabled true|false` | Master switch. `false` skips Stages 3–4 and 7; intent → action path runs as in v0.23/v0.22. |
| `mix allbert.settings set objectives.max_steps_per_turn N` | Cap per-turn step count; 1..16. Default 3. |
| `mix allbert.settings set objectives.max_loop_count N` | Cap advance cycles per objective; 1..32. Default 5. Exceeding triggers impasse. |
| `mix allbert.settings set objectives.trace_detail :operator|:debug` | Trace volume. `:operator` (default) renders `## Objective` + bounded `## Objective Steps`. `:debug` also renders stage and lifecycle events. |
| `mix allbert.settings set allbert.jido.debug_trace true|false` (v0.23) | When `true`, configured JidoBacked agents, including `Objectives.Engine.Agent`, render bounded `## Jido Debug` diagnostics in trace markdown. Default `false`. |

## Authority Posture (unchanged)

`objective_id` is **not** authorization. Every step that mutates,
fetches, sends, executes, or contacts external systems still flows
through registered actions → `Actions.Runner.run/3` → Security Central
→ confirmations (when needed) → audit. The new `:objective_write`
permission class (default `:allow`, floor `:allow`) governs only the
cancel/continue lifecycle actions; it exists for symmetry with other
`_write` classes and for future per-objective ACL scoping (v0.31+).

ADR 0021 Section 4 (authority rule) and Section 2 (authority boundary)
hold without exception.

## Stage Reference

The engine implements seven stages, each a declared command on the
`Objectives.Engine.Agent` JidoBacked agent:

| # | Stage | Engine command | What it does |
|---|---|---|---|
| 1 | Receive | `:receive_input` | Accept runtime turn; attach `objective_id` if resuming. |
| 2 | Interpret intent | `:interpret_intent` | Call `Intent.Engine.collect_candidates/2` with `:objective` opt; receive ranked candidates including objective-resume kind. |
| 3 | Frame/resume objective | `:frame_objective` | Decide: create new, resume existing, or no objective. Deterministic rules. |
| 4 | Propose and evaluate steps | `:propose_steps` + `:evaluate_steps` | `AllbertAssist.Objectives.Proposer` dispatches to the registered per-app proposer (e.g., `StockSage.Proposer`). Evaluator scores feasibility. |
| 5 | Authorize selected step | `:authorize_step` | Route through `Actions.Runner` + Security Central + Confirmations. |
| 6 | Execute | `:execute_step` | `Actions.Runner.run/3` |
| 7 | Observe and advance | `:observe_step` | Record observation; evaluate against acceptance; decide continue / block / complete. |

## Flow 1: Single-Step Objective

Example user input: `"analyze AAPL"` (in StockSage app context).

```

│   → emits allbert.runtime.turn.completed (trace_id: T1-trace,       │

└────────────────────────────────────────────────────────────────────┘
```

Trace markdown for this flow contains the standard v0.22 top-level
sections plus an inline `### Objective` subsection in each section
that touched the objective during this turn (per the v0.24 fourth-pass
placement decision):

- `## Intent Candidates` — standard v0.19 section with the
  RunAnalysis selection. Includes inline `### Objective` subsection
  (objective id, title bounded 60 chars, status pill, current step,
  stage, loop_count) when the `:objective` candidate kind appeared
  in the ranking or the active objective influenced scoring.
- `## Selected Skill` (or `## Selected Action`) — standard. Includes
  inline `### Objective` subsection when the selected action is the
  objective's current step.
- `## Confirmation` — standard v0.07 section with `Objective: <id>`
  and `Step: <step_id>` lines AND an inline `### Objective`
  subsection.
- `## Action Result` — standard. Includes inline `### Objective`
  subsection (the action ran as an objective step).
- `## Objective Steps` — dedicated top-level section listing all
  steps for the objective in this turn (cap 5 at `:operator`, cap
  20 at `:debug`). Includes per-step status, action, result_summary
  (bounded).
- `## Memory Review` — appears only if a reflection step proposed
  candidates (none in this flow).

At `objectives.trace_detail = :debug`, each inline subsection also
includes the stage's `before_*` / `after_*` lifecycle events for
that section. At `:operator` (default), only status + identifying
fields render.

If `allbert.jido.debug_trace = true`, a single top-level
`## Jido Debug` section renders at trace tail per the v0.23
substrate convention (covers Engine.Agent + Confirmations.Store.Agent
+ Jobs.Scheduler.Agent). Objective-specific details live inside the
inline `### Objective` subsections and the dedicated
`## Objective Steps` section — the two channels do not overlap.

## Flow 2: Multi-Step Objective (Hybrid Proposer Cycle)

Example user input: `"analyze AAPL and compare to MSFT"`.

The flow uses the **hybrid proposer contract** introduced in the v0.24
fourth pass: `StockSage.Proposer.propose/2` returns one step at a time
plus a `{:more, {:stocksage, hint}}` signal until the comparison is
ready, at which point it returns `:done`. The engine re-invokes the
proposer after each Stage 7 observation, passing the prior hint back.

Differences from Flow 1:

1. **Stage 3 — Frame objective.** `acceptance_criteria` is a structured
   JSON map (see v0.24 plan Evaluator section) with two
   `step_completed_with_action` clauses (one per ticker) and an
   `observation_contains` clause for the comparison summary. Stored
   in `objectives.acceptance_criteria` as JSON.

2. **Stage 4 — First propose call.**
   `Proposer.propose(intent_decision, ctx)` returns:
   ```
   {:ok, [step1_attrs], {:more, {:stocksage, %{
     step_index: 1,
     completed_steps: [],
     total_planned: 2,
     remaining_tickers: ["MSFT"]
   }}}}
   ```
   Engine persists the hint into `Engine.Agent.proposer_hints[obj_id]`.
   Inserts `objective_steps` row for step 1 (`status: :proposed`,
   `candidate_action: "StockSage.Actions.RunAnalysis"`,
   `action_params: %{ticker: "AAPL", ...}`, `parent_step_id: nil`).
   Evaluator scores feasibility → step 1 → `:selected`.

3. **Stages 5–7 run for step 1.** Same as Flow 1. Confirmation,
   approval, execution, observation. Step 1 completes.
   `loop_count` increments by 1 at Stage 7 (per completed step).

4. **Acceptance evaluator returns `:needs_more_steps`** since only one
   `RunAnalysis` step is completed (the comparison clause requires
   two).

5. **Engine re-invokes Stage 4 with persisted hint.** Calls
   `Proposer.propose(intent_decision,
   Keyword.put(ctx, :proposer_hint, {:stocksage, ...}))`. Proposer
   pattern-matches on the `:stocksage` tag and returns:
   ```
   {:ok, [step2_attrs], :done}
   ```
   where step2_attrs has `parent_step_id = step1.id`. Engine inserts
   `objective_steps` row for step 2; clears
   `Engine.Agent.proposer_hints[obj_id]`. Evaluator → step 2 →
   `:selected`.

6. **Stages 5–7 run for step 2.** Step 2 confirmation, approval,
   execution, observation. `loop_count` increments to 2 at Stage 7.

7. **Acceptance evaluator returns `:met`.** All required clauses
   satisfied (two completed RunAnalysis steps; comparison summary
   present in step 2's `observation_summary`). Objective transitions
   to `:completed`.

8. Both `allbert.runtime.turn.completed` (per turn that drove a step)
   and `allbert.objective.completed` (once, at the end) fire; all
   share `trace_id` lineage so consumers can correlate.

Two confirmations are created, one per step. The operator approves
each in turn. The engine does not auto-approve based on prior
approval ("smarter" auto-approve policies are a reserved-vocabulary
concern, not v0.24).

**Loop count cadence**: `loop_count` increments on each Stage 7
observation (per completed step), not on each propose-cycle. Default
`max_loop_count = 5` accommodates the two-step case (count = 2 at
completion) with headroom.

**Hint persistence under crash**: if the engine crashes between
step 1 completion and step 2 propose, the JidoBacked
`rebuild_state/1` reloads `proposer_hints` from the engine's
serialized state (alongside `active_objectives`). On rehydration,
Stage 4 resumes with the same hint and proposes step 2 correctly.
Tested in M4 with a synthetic crash-and-rehydrate test.

### Sub-flow 2a: Impasse via `max_loop_count`

If `loop_count >= max_loop_count` AND the evaluator returns
`:needs_more_steps`:

- Engine writes an `objective_events` row:
  ```
  kind: :impasse
  summary: "max_loop_count reached"
  payload: {
    "cap_hit": "max_loop_count",
    "would_have_continued_verdict": "needs_more_steps"
  }
  ```
- Objective row UPDATE: status `:blocked`.
- Engine emits `allbert.objective.impasse` and
  `allbert.objective.blocked`.

The `would_have_continued_verdict` is recorded so operators can see
why the engine wanted to keep going (per v0.24 plan Rule 3).

### Sub-flow 2b: Impasse via `max_steps_per_turn`

If the per-app proposer returns more steps than
`max_steps_per_turn` allows in one turn:

- Engine writes an `objective_events` row:
  ```
  kind: :impasse
  payload: { "cap_hit": "max_steps_per_turn" }
  ```
- Objective row UPDATE: status `:blocked`.

## Flow 3: Cancellation (Cooperative Only)

Example: operator runs `mix allbert.objectives cancel <id> --reason "..."`
while an objective is in `:blocked` (awaiting confirmation) or
`:running`.

```
┌────────────────────────────────────────────────────────────────────┐
│ Operator: mix allbert.objectives cancel <id> --reason "..."         │
│   --reason is REQUIRED (third-validation decision); missing it      │
│   exits with code 64.                                               │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│ AllbertAssist.Actions.Objectives.CancelObjective                    │
│   → Security Central: :objective_write (default :allow, floor      │
│     :allow)                                                         │
│   → Actions.Runner.run/3                                            │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│ Engine.Agent.cmd(:cancel_objective, %{                              │
│   objective_id: <id>,                                               │
│   reason: <bounded string>                                          │
│ })                                                                  │
│   → objective row UPDATE: status :cancelled                         │
│   → any :proposed steps → row UPDATE status :cancelled              │
│   → any :running step is NOT interrupted (cooperative only)         │
│   → in-flight registered action completes normally                  │
│   → its observation is still recorded                               │
│   → but no new steps proposed; engine stops advancing               │
│   → emits allbert.objective.cancelled                               │
│   → records objective_events row (kind=:cancelled, reason in        │
│       summary)                                                      │
└────────────────────────────────────────────────────────────────────┘
```

### Sub-flow 3a: Cancel-Then-Approve Race

If a confirmation is pending when cancellation runs:

- The confirmation record itself is NOT auto-expired. The operator
  can still approve or deny it.
- **If they approve**, the action runs (single-shot semantics); the
  action's own audit trail records the run; the result row persists
  with the original `objective_id`/`step_id`.
- The engine ignores the result for objective advancement — the
  objective is `:cancelled` and the engine's
  `:observe_step`/`:evaluate_steps` path is bypassed for cancelled
  objectives.
- This is a deliberate choice: cancellation does not destroy work
  already in flight. Operators who want to also reject the pending
  confirmation can do so via `mix allbert.confirmations deny`.

The M5 test "Cancel-Then-Approve" pins this race behavior.

### Sub-flow 3b: Mid-Action Interruption (deferred)

Killing the in-flight `RunAnalysis` Python process (or other
external work) is NOT v0.24 scope. It's deferred to v0.25+ when
native trading agents have a clearer interruption story. ADR 0021
Section 7 codifies this.

## Flow 4: Resume Via `mix allbert.objectives continue`

Three ways an objective gets resumed:

1. **Confirmation approval (most common).** After `approve_confirmation`
   runs, the engine's `:continue_objective` command picks up where the
   step was blocked. This is the path Flow 1 takes during the
   "Operator approves" pause.
2. **Operator `mix allbert.objectives continue <id>`.** Used when an
   objective is in `:blocked` status for a reason other than
   pending confirmation (e.g., loop-cap-hit impasse where the
   operator decides to bump `max_loop_count` and continue).
3. **Scheduled job advance.** A v0.13 scheduled job may carry an
   `objective_id`; when it runs, it advances the objective. This is
   the path used for long-running analyses or follow-ups across
   days.

For path 2:

```
┌────────────────────────────────────────────────────────────────────┐
│ Operator: mix allbert.objectives continue <id> [--user USER]        │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│ AllbertAssist.Actions.Objectives.ContinueObjective                  │
│   → Security Central: :objective_write                              │
│   → Actions.Runner.run/3                                            │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│ Engine.Agent.cmd(:continue_objective, %{objective_id: <id>})        │
│   → re-evaluates whether the objective can advance:                 │
│     A) confirmation linked to objective is now resolved →           │
│        advance via Stage 5 → 6 → 7; CLI prints new status           │
│     B) loop cap was the blocker AND it was raised via               │
│        `mix allbert.settings set objectives.max_loop_count N` →     │
│        advance via Stage 4 with stored proposer hint                │
│     C) status :abandoned → return                                   │
│        {:error, :objective_abandoned}; CLI prints                   │
│        "Objective <id> is abandoned; create a new objective to      │
│         retry."; exits 1                                            │
│     D) nothing changed (still pending confirmation, no cap raise,   │
│        no other state change) → return advisory                     │
│        {:ok, %{status: :still_blocked, reason: "..."}};             │
│        CLI prints "Objective <id> is still blocked: <reason>";      │
│        exits 0; NO loop_count increment; NO state change; NO        │
│        impasse event. Operator can re-issue without consequence.    │
│   → emits allbert.objective.updated (only on paths A or B)          │
└────────────────────────────────────────────────────────────────────┘
```

The advisory `:still_blocked` return is the v0.24 fourth-pass
decision: `continue` is idempotent for no-progress calls. Operators
can poll it without polluting `loop_count` or generating impasse
events. Resume-on-loop-cap-raise (path B) is the only way to push a
loop-cap impasse forward; resume-via-confirmation (path A) is the
common case driven by Flow 1 / Flow 2 approval.

`:abandoned` (path C) is **terminal**. Operators wanting to retry must
create a new objective (e.g., by re-asking the original prompt).
Engine never resurrects `:abandoned` rows.

## Confirmation Integration Detail

When a step requires confirmation, the confirmation record gets
two new fields (from the v0.24 M1 migration on `confirmations`):

- `objective_id`
- `step_id`

**Snapshot + stale warning** (v0.24 fourth-pass decision):

The confirmation's `params_summary` JSON gets two new keys at
creation time so renderers can detect divergence without an extra
DB read for the common (still-fresh) case:

```json
{
  "ticker": "AAPL",
  "analysis_date": "2026-05-17",
  "disclosure": "...",
  "objective_title": "Analyze AAPL",
  "objective_status": "running"
}
```

The renderer then fetches the live objective row by `objective_id`
and compares the live `status` to the snapshot `objective_status`.
If they match → render the snapshot. If they differ → prepend a
`Note: objective is now :<live_status>` line so the operator sees
that the objective changed since the confirmation was created.

`mix allbert.confirmations show <id>` renders these as:

Standard (objective status unchanged since confirmation creation):

```
Confirmation: <conf_id>
Action: StockSage.Actions.RunAnalysis
Status: pending
Created: 2026-05-17 14:23:01 UTC
TTL: expires 2026-05-17 15:23:01 UTC

Objective: <objective_id>
  Title: Analyze AAPL
  Status: :running
Step: <step_id>
  Kind: action
```

Stale (objective status changed; e.g., operator cancelled the
objective after the confirmation was created):

```
Confirmation: <conf_id>
Note: objective is now :cancelled (was :running at confirmation creation)
Action: StockSage.Actions.RunAnalysis
Status: pending
Created: 2026-05-17 14:23:01 UTC
TTL: expires 2026-05-17 15:23:01 UTC

Objective: <objective_id>
  Title: Analyze AAPL
  Status: :cancelled
Step: <step_id>
  Kind: action
```

Operators can still approve a stale confirmation; per Flow 3a
(cancel-then-approve), the action runs (single-shot v0.07 semantics)
and the audit trail records the run, but the engine ignores the
result since the objective is terminal.

Telegram and email confirmation rendering (M5) follows the same rule
— snapshot at creation, fetch + compare at render, prepend the
`Note:` line on divergence. Confirmation LiveView does the same on
mount and on each PubSub update.

The `Confirmations.Store.Agent` (v0.23 JidoBacked) writes the new
`params_summary` keys verbatim; existing v0.07 audit YAML schema is
preserved with the two new optional keys.

## Intent Engine Integration

The intent engine gains a new arity. ADR 0019 is amended at M2 to
register the new candidate kind.

```elixir
@spec collect_candidates(decision_input :: map()) :: [Candidate.t()]
@spec collect_candidates(decision_input :: map(), opts :: keyword()) :: [Candidate.t()]
```

The old `/1` arity continues to work and delegates to `/2` with
empty opts.

Behavior of the new `/2` arity:

- When the operator has open objectives matching the current turn's
  topic (deterministic keyword + recency match), an objective-
  resume candidate appears in the ranking.
- Selecting it routes to `:continue_objective` rather than creating
  a new objective.
- ADR 0019 invariants hold: the candidate's score cannot exceed
  permission floors; the candidate does not grant authority.

Candidate fields:

```elixir
%AllbertAssist.Intent.Candidate{
  kind: :objective,
  id: <objective_id>,
  source: :objective,
  score: 0.2..0.5,
  reason: <objective.title>,
  trace_metadata: %{
    objective_id: <id>,
    objective_status: :open | :running | :blocked,
    objective_active_app: <active_app>
  }
}
```

Memory candidates (`:memory`) and objective candidates (`:objective`)
can coexist in the same ranking pass without conflict.

## LiveView Surfaces

### Signal → PubSub bridge

`AllbertAssistWeb.SignalBridge` (new in v0.24; see v0.24-plan.md
Module Shapes) is a GenServer running under
`AllbertAssistWeb.Application`. It subscribes to
`AllbertAssist.SignalBus` at boot, filters `allbert.objective.*`
signals, and broadcasts each event to a per-user Phoenix.PubSub topic:

```
Phoenix.PubSub.broadcast(
  AllbertAssistWeb.PubSub,
  AllbertAssistWeb.SignalBridge.topic_for(user_id),  # "objectives:<user_id>"
  {:objective_event, signal}
)
```

LiveViews subscribe to this topic to receive real-time objective
updates without polling. If the web app is not running (headless CLI
deployment), the bridge is absent and Engine signals fire normally —
LiveViews fall back to their 5-second poll.

### `/agent` (AgentLive extension)

- Adds `#objective-badge-<id>` component when the current user has an
  active objective.
- On mount: `Phoenix.PubSub.subscribe(AllbertAssistWeb.PubSub,
  AllbertAssistWeb.SignalBridge.topic_for(current_user_id))`.
- `handle_info({:objective_event, signal}, socket)` re-fetches active
  objectives and re-renders the badge.
- 5-second `:timer.send_interval` poll fallback if SignalBridge is
  not running (graceful degradation).

### `/objectives/:id` (new ObjectiveLive)

DOM element ids:

| Id | Purpose |
|---|---|
| `#objective-header` | Title, status pill, created_at, current_step_id. |
| `#objective-acceptance` | Acceptance criteria block. |
| `#objective-steps` | Step list container. |
| `#objective-step-<step_id>` | Per-step row with status pill, kind label, action name, confirmation link. |
| `#objective-events` | Event timeline container. |
| `#objective-event-<event_id>` | Per-event row. |
| `#objective-cancel-button` | Cancel modal trigger. Modal collects required reason. |
| `#objective-cancel-modal` | Reason form. POSTs `cancel_objective` via `Actions.Runner`. |
| `#objective-continue-button` | Visible only when objective.status == :blocked. POSTs `continue_objective`. |

All UI actions go through registered actions; no direct SQLite reads
or writes from the LiveView.

## Settings Behavior

| setting | default | effect |
|---|---|---|
| `objectives.enabled` | `true` | When `false`, runtime turns skip Stages 3–4 and 7. Intent → action runs as in v0.23. |
| `objectives.max_steps_per_turn` | `3` | Engine refuses to propose more than N steps within a single turn (impasse if exceeded). |
| `objectives.max_loop_count` | `5` | After N advance cycles, engine records `:impasse` (`cap_hit=:max_loop_count`) and transitions to `:blocked`. |
| `objectives.trace_detail` | `:operator` | At `:operator`, trace contains `## Objective` and bounded `## Objective Steps`. At `:debug`, also stage and lifecycle events. |
| `allbert.jido.debug_trace` (from v0.23) | `false` | When `true`, configured JidoBacked agents render a bounded top-level `## Jido Debug` section. Objective-specific details remain in `## Objective`. |

A `mix allbert.settings set objectives.enabled false` instantly
disables objective creation for new turns. Existing open objectives
remain visible via `mix allbert.objectives list` until they are
cancelled or completed manually.

## Engine Rehydration Posture

Per v0.24 plan Rule 10, on `AllbertAssist.JidoBacked.Supervisor` boot:

1. `Objectives.Engine.Agent.rebuild_state/1` (JidoBacked callback)
   queries `objectives` where `status` in `[:open, :running, :blocked]`.
2. Rows with `updated_at` older than 1 hour:
   - SQLite row UPDATE: `status = :abandoned` (new terminal status
     added in v0.24 vs. ADR 0021 §3).
   - Not loaded into in-memory projection.
3. Rows within the 1-hour window are loaded into
   `active_objectives` projection map.
4. Engine resumes from the recorded `current_stage` for each active
   objective.

The 1-hour threshold is currently fixed. A future setting
(`objectives.rehydrate_window_minutes`) is reserved in ADR 0021 but
not implemented in v0.24.

## Redaction Rules

v0.24 inherits all v0.07 + v0.22 + v0.23 redaction obligations and
applies them to the new objective surface:

- `AllbertAssist.Security.Redactor` runs against every new signal
  payload (`allbert.objective.*`) before emission.
- `objective_events.payload` JSON is redacted before write.
- Trace `## Objective` and `## Objective Steps` sections pass through
  Redactor before render.
- `Intent.Engine.collect_candidates/2` objective candidates'
  `trace_metadata` is redacted before being added to the trace
  `## Intent Candidates` section.
- LiveView and channel rendering surfaces use the same Redactor
  output; no LiveView assigns or channel messages carry raw secret
  fields.

The v0.22 third-validation `sensitive_key_fragments` allowlist
(including `raw_bridge`, `raw_final`, `raw_response`, `authorization`,
`bearer`) covers v0.24's new fields without amendment.

## Failure And Safety Cases

| Case | Expected behavior |
| --- | --- |
| Objective creation fails (changeset error) | Engine logs bounded error; emits no `allbert.objective.created`; runtime falls back to v0.23 intent → action flow for the turn. |
| Engine crashes mid-objective | Supervisor restarts; `rebuild_state/1` rehydrates objectives with `updated_at` within 1h; older rows transition to `:abandoned`. |
| Step crashes mid-execution (action raises) | Step row UPDATE: status `:failed` with bounded error; objective transitions to `:blocked` with reason `:step_failed`; emits `allbert.objective.step.failed` and `allbert.objective.blocked`. |
| Confirmation pending when engine restarts | Confirmation record remains pending (v0.23 store rehydrates from Allbert Home YAML); on next operator action, engine looks up the linked confirmation via `confirmation_id` and continues. |
| Operator approves a confirmation for a cancelled objective | Action runs once (single-shot semantics); result persists; objective stays `:cancelled`; engine logs the discarded result. |
| `max_loop_count` reached with `:needs_more_steps` verdict | Impasse event recorded with `would_have_continued_verdict: :needs_more_steps`; objective transitions to `:blocked`. |
| `max_steps_per_turn` exceeded by proposer | Impasse event recorded with `cap_hit: :max_steps_per_turn`; objective transitions to `:blocked`. |
| `objectives.enabled = false` mid-objective | Existing objectives keep advancing through their natural lifecycle; no new objectives are created. |
| `mix allbert.objectives cancel <id>` without `--reason` | Exits with code 64; CLI prints usage line. |
| Cross-user `mix allbert.objectives show <id>` | Returns `:not_found`; CLI exits with code 65. |
| `:delegate_agent` step with unknown `delegate_agent_id` | Step row UPDATE: status `:failed`; error `agent_not_found`; objective transitions to `:blocked`. |
| `Intent.Engine.collect_candidates/1` (legacy arity) called | Delegates to `/2` with empty opts; no `:objective` candidates returned; back-compat preserved. |
| `Objectives.Engine.Agent` rehydrates row with FK pointing at deleted thread | `source_thread_id` NULLed; objective remains valid; warning logged. |
| Telegram/email plugin not loaded | Confirmation flow runs without objective context lines in those channels; CLI/LiveView still render full objective info. |
| Two operators acting on the same objective simultaneously | Last write wins on the objective row; per-step row UPDATEs are atomic; events are recorded for both actions. (Multi-operator coordination is v0.31+ scope.) |
| `mix allbert.objectives continue <id>` with no progress possible | Returns `{:ok, %{status: :still_blocked, reason: "..."}}`. CLI prints reason; exits 0; no `loop_count` increment; idempotent. |
| `mix allbert.objectives continue <id>` on `:abandoned` objective | Returns `{:error, :objective_abandoned}`. CLI prints "Objective <id> is abandoned; create a new objective to retry."; exits 1. Engine never resurrects `:abandoned` rows. |
| Confirmation rendered after objective status changed (e.g., objective cancelled mid-pending) | Renderer fetches live objective row; if status differs from snapshot in `params_summary.objective_status`, prepends `Note: objective is now :<live_status>` line. Operator can still approve; per cancel-then-approve race, action runs but engine ignores result. |
| Engine command returns `{:error, _reason}` (synthetic; not a raise) | `handle_command_error/3` catches; affected step → `:failed`; objective → `:blocked` reason `:step_failed`; events + signals emitted; other active objectives keep running. |
| Engine command raises unexpected exception | JidoBacked supervisor restarts Engine.Agent; eager `rebuild_state/1` rehydrates from SQLite within the 1-hour window. Active objectives lose transient in-memory state but reload from SQLite. |
| Proposer returns `{:ok, [steps], {:more, hint}}` but engine crashes before next propose | Hint persisted in `Engine.Agent.proposer_hints[objective_id]` (JidoBacked schema field); `rebuild_state/1` reloads. Next `continue` or approval re-invokes propose with the persisted hint. |
| `Jido.Registry` API insufficient for AgentRegistry (M1 verification) | Fall back to Elixir `Registry`; document the choice in `AgentRegistry.@moduledoc`; v0.25 specialist agents see the same `register/3` + `lookup/1` API either way. |
| SignalBridge not running (headless CLI deployment) | AgentLive / ObjectiveLive fall back to 5-second poll. Engine signals still fire; only the LiveView reactivity is degraded. |

## Non-Goals Visible To Users

- No advisory provider behaviour, world-model abstraction, capability
  inventory, route, or acquisition option.
- No parallel step execution.
- No mid-action interruption.
- No LLM-assisted acceptance evaluator.
- No automatic memory promotion from objective observations.
- No native StockSage trading agents in v0.24 (v0.25). v0.24 ships
  only the minimal `:delegate_agent` contract.
- No workspace shell upgrade (v0.26) or canvas integration (v0.30).
- No public plugin hook contribution API.
- No settings-driven proposer rules (proposers are hardcoded Elixir
  per app).
