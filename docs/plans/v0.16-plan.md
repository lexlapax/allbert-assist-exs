# Allbert v0.16 Additional Channels

## Purpose

v0.16 adds channel adapters after CLI, LiveView, settings, skills,
confirmations, execution adapters, jobs, Security Central, and intent decisions
share the same runtime core.

Channels translate external messages into Allbert signals and render responses
back to their medium. They must not own agent logic, security policy, settings
semantics, memory semantics, skill execution, or risky execution policy.

This milestone was formerly tracked as v0.13 before the post-v0.10 roadmap
was unified into one v0.xx sequence.

## Expected Inputs

- v0.02 Settings Central exists for channel enablement, preferences, and
  credentials.
- v0.05 Security Central exists for actor/channel/session context, trust
  boundary, risk, redaction, and audit metadata.
- v0.07 confirmation workflow can handle interrupted actions across operator
  surfaces.
- v0.08 through v0.10 risky execution adapters are action-backed and traceable,
  including external service, package install, and online import confirmation
  metadata.
- v0.11 intent decisions expose selected skill/action, risk, permission, and
  confirmation metadata, including Resource Access Security Posture and
  operation-scoped Approval Handoff data.
- v0.13 jobs can emit runtime signals without becoming a separate runtime.
- v0.12 local workspace identity provides the local string `user_id` that
  external channel identities resolve to through Settings Central.
- v0.10 M9 release-readiness docs define the current CLI/LiveView smoke
  baseline that additional channels must render natively rather than bypassing.

## Expected Direction

- Add channel adapters after CLI and LiveView share the same runtime core.
- Candidate channels include email, SMS, Discord/Telegram-style chat, browser
  capture, and native UI surfaces.
- Channels translate external messages to signals and render responses; they do
  not own agent logic.
- Channels read and update shared settings through the settings engine.
- Channel adapters map external identities, such as a Telegram user id, email
  address, or SMS number, to a configured local string `user_id` through
  Settings Central. The mapping is explicit and settings-backed; an external
  identity cannot implicitly claim an existing local `user_id` without
  operator configuration. Channel traces include both the external identity
  and the resolved local `user_id`.
- Channel traces include channel metadata and confirmation handoff behavior.
- Channels render v0.10 approval handoffs natively but still call
  `approve_confirmation` and `deny_confirmation`; they do not call `Req`,
  package managers, or import writers directly.
- Channels render resource access handoffs natively for requests such as URL
  summarization, document inspection, local skill directory import, direct
  skill URL import, and trusted skill script execution. They show the canonical
  path or URL, operation class, access mode, scope, limits, consumer, and
  remember-scope choices, but they do not read, fetch, extract, summarize,
  import, execute, or store grants directly.
- Package approval handoffs must render manager, exact package specs, target
  root, npm execution argv preview, and pip preview-only warnings as native
  channel details instead of collapsing them into a generic command prompt.

## Non-Goals

- No channel-specific intent engine.
- No channel-specific settings store.
- No direct action execution from adapters.
- No channel-owned package install, external service, or online import policy.
- No channel-owned local path access, URL fetch, document extraction,
  summarizer invocation, remembered resource grants, local skill directory
  import, or skill URL import.
- No remote channel credentials outside Settings Central secrets.
- No broad provider matrix until one adapter is boring and tested.

## Definition Of Done

v0.16 is done when:

- At least one additional channel adapter can submit messages through the
  runtime.
- Channel messages include channel metadata in traces and security context.
- Channel credentials are stored as Settings Central secrets.
- Channel enablement and response style are settings-backed.
- The adapter has no embedded agent, security, execution, or memory logic.
