# v0.16 Plan: Additional Channels

Status: implemented through M7. Ready for operator manual verification.

Formerly the "Additional Channels" future plan. v0.16 is the first concrete
non-CLI/non-LiveView channel milestone in the unified v0.xx sequence.

## Purpose

v0.16 proves the Allbert channel adapter boundary with two remote channels:
**Telegram** (instant-messaging, interactive, polling via Bot API) and
**email** (async inbox/SMTP, IMAP polling, typed-command approvals). Having two
real adapters in the same milestone demonstrates that the shared substrate
(context, message struct, identity resolution, event dedup, runtime submission,
Approval Handoff rendering) is genuinely provider-neutral, not Telegram-specific.

Each adapter translates external provider input into
`AllbertAssist.Runtime.submit_user_input/1`, renders runtime responses back to
the provider, and records enough durable event metadata to dedupe, trace, and
audit the exchange. The two providers differ in transport (HTTP polling vs IMAP),
dedup primitive (Telegram update_id vs email Message-ID), session semantics
(user+chat vs user-only), and approval affordance (inline buttons vs typed reply
commands). Those differences live inside each adapter's `Adapter`, `Client`,
`Parser`, and `Renderer` modules, not in the shared substrate.

v0.16 must not create a second runtime, a second confirmation system, or a
second security policy. Channels are delivery surfaces. All intent, action,
memory, resource access, confirmation, and job behavior continues to flow
through the existing runtime and registered Jido actions.

## Core Decisions

- Telegram and email are both implemented in v0.16. They share the channel
  substrate but differ in transport, dedup primitive, session semantics, and
  approval affordance. Neither owns intent, memory, confirmations, or security
  policy.
- External identities never implicitly become local users. Telegram user IDs
  and sender email addresses are each mapped to local string `user_id` values
  through explicit Settings Central `identity_map` configuration, following
  the same contract for both channels.
- Channel transports may talk only to their configured provider: Telegram Bot
  API for Telegram; IMAP server for inbox retrieval and SMTP server for email
  sending. Neither adapter is a general-purpose HTTP client, browser, crawler,
  document extractor, skill importer, or external service bridge.
- Telegram uses **inline callback buttons** for confirmation approval UX because
  the Bot API supports interactive keyboards. Email uses **typed reply commands**
  (`approve <id>`, `deny <id>`, `show <id>`) because email has no native
  interactive callback primitive. Both resolve durable confirmations through
  `AllbertAssist.Actions.Runner.run/3`; neither grants new powers.
- Telegram dedup uses the integer `update_id` as `external_event_id`. Email
  dedup uses the RFC 5322 `Message-ID` header as `external_event_id`. Both
  write to the same `channel_events` table with a partial unique index on
  `{channel, external_event_id}` for inbound and callback directions.
- Email IMAP polling fetches `UNSEEN` messages, processes each, and marks them
  `\Seen` after a successful `channel_events` insertion. The email adapter does
  not persist a separate numeric offset; `Message-ID` dedup handles restarts.
  Comments and a defined `ImapClient` callback boundary are left in place for
  future IMAP IDLE support. The SMTP client is structured to accept a provider
  API backend (Mailgun, Sendgrid) as a future adapter without changing the
  `Renderer` or `Adapter` layers.
- Channel event records store full execution-routing fields needed for dedupe
  and audit, but redacted/truncated summaries for operator display. Passwords,
  tokens, secrets, and raw provider payload dumps are never written to traces,
  logs, CLI output, LiveView, or channel event summaries.

## Roadmap And Vision Fit

The Allbert/Jido vision treats channels as delivery adapters around the
signal-driven runtime, not as separate assistant surfaces:

> All input and output surfaces should become channel adapters around the same
> signal-driven core. Each channel should translate external input into signals
> and render agent output back into the medium without owning the agent logic.

v0.16 is the first proof of that principle beyond CLI and LiveView. Having two
real adapters — one interactive/polling (Telegram) and one async/inbox (email)
— validates that the shared substrate genuinely holds across different transport
models, dedup strategies, and approval affordances.

Design lessons from JavaScript and Python agentic systems (openClaw, Hermes,
LangChain agents) converge on the same key patterns:

- Adapters implement a minimal receive/send interface; routing is the runtime's
  job.
- External identity is always mapped to an internal user id at the adapter
  boundary, never assumed.
- Approval rendering is channel-specific (buttons vs text) but confirmation
  resolution is shared.
- Dedup is channel-specific but uses the same storage substrate.

v0.16 embeds all four of those patterns in the Elixir/OTP idiom: supervised
GenServers for polling loops, pure modules for parsing/rendering, a shared Ecto
context for event persistence, and `AllbertAssist.Actions.Runner.run/3` as the
single confirmation resolution path.

This milestone prepares later work without implementing it:

- v0.17 StockSage can receive requests through Telegram or email immediately.
  If a user's session carries `active_app: :stocksage`, the runtime routes
  StockSage actions the same way for both channels. Adapters do not need to
  know which app is active.
- v0.18 memory review must not auto-promote channel conversation turns. The
  constraint that explicit confirmation is required for markdown memory
  promotion applies equally to Telegram and email turns.
- v0.20 cross-surface intent enrichment can use `channel`, `session_id`,
  `active_app`, and `thread_id` from incoming channel requests as additional
  routing signals. v0.16 already carries all four in runtime requests.
- v0.23 security evals have concrete cross-channel surfaces to test: identity
  spoofing across both channels, callback replay, provider payload injection,
  resource approval scope leakage, cross-user thread leakage, and secret
  redaction in provider error responses.
- v0.26 workspace surface work should treat the v0.11 Approval Handoff format,
  the Telegram compact callback format, and the email typed-command format as
  the three reference points for how canvas/ephemeral UI extensions should
  structure approval-resolution paths.

v0.16 must keep Allbert local-first and inspectable. Both channel supervisors
and adapters should have predictable startup, diagnostic idle behavior, and
boring restart semantics.

`AllbertAssist.Channels.Message` — normalized inbound message struct shared

  provider:            String.t(),         # "telegram_bot_api" | "email_imap"

  content_type:        :text | :callback_query | :command | :unsupported,
  external_event_id:   String.t(),         # Telegram update_id or email Message-ID
  external_user_id:    String.t(),         # Telegram user id or sender email address
  external_chat_id:    String.t() | nil,   # Telegram chat id; nil for email

  command_data:        String.t() | nil,   # present when content_type == :command

                                           #   (Telegram inline keyboard)
  subject:             String.t() | nil,   # email subject; nil for Telegram

### Ecto Schema: channel_events

Migration adds to the v0.12 SQLite database. Works for all channels.

```elixir
create table(:channel_events) do
  add :channel,             :string,  null: false  # "telegram" | "email"
  add :provider,            :string,  null: false  # "telegram_bot_api" | "email_imap"
  add :direction,           :string,  null: false  # "inbound" | "outbound" | "callback"
  add :external_event_id,   :string,  null: false  # Telegram update_id or email Message-ID
                                                    # or generated uuid for outbound
  add :external_user_id,    :string               # Telegram user id or sender address
  add :external_chat_id,    :string               # Telegram chat id; nil for email
  add :external_message_id, :string               # Telegram message_id or IMAP UID str
  add :user_id,             :string               # nil until identity resolution
  add :session_id,          :string               # nil until session derivation
  add :thread_id,           :binary_id            # nil until runtime submission
  add :input_signal_id,     :string               # nil until signal emission
  add :trace_id,            :string               # nil until action trace
  add :status,              :string,  null: false  # received|processed|rejected|failed
  add :reason,              :string               # bounded (<=500 chars)
  add :payload_summary,     :string               # redacted, bounded (<=500 chars)
  add :error,               :string               # redacted, bounded (<=500 chars)
  timestamps()
end

# Dedup index: only inbound and callback events across all channels
create unique_index(:channel_events, [:channel, :external_event_id],
  where: "direction IN ('inbound', 'callback')",
  name: :channel_events_inbound_callback_dedup)

create index(:channel_events, [:channel, :user_id])
create index(:channel_events, [:channel, :external_user_id])
create index(:channel_events, [:status])
```

`AllbertAssist.Channels.Event` is the Ecto schema module. Changesets enforce
field types, null constraints, bounded string lengths, and direction/status
enum membership. Business logic lives in `AllbertAssist.Channels` context
functions.

## Session Identity And Session_id Derivation

All channel adapters derive a bounded, deterministic `session_id` using SHA-256.
The derivation is public in `AllbertAssist.Channels.derive_session_id/3` and
shared across providers.

**Telegram**: `session_id` depends on channel, external user id, AND chat id
(because one Telegram user can interact in multiple chats, each warranting
a distinct session context):

```elixir
# channel = "telegram", external_user_id = Telegram user id str,
# external_chat_id = chat id str
raw = "#{channel}:#{external_user_id}:#{external_chat_id}"
hash = :crypto.hash(:sha256, raw) |> Base.encode16(case: :lower)
"ch_tg_#{String.slice(hash, 0, 32)}"
# e.g. "ch_tg_a3f2c1d9..." (max 39 chars)
```

**Email**: `session_id` depends on channel and sender address only. Email has
no chat concept; one session per sender:

```elixir
# channel = "email", external_user_id = sender email address,
# external_chat_id = nil
raw = "#{channel}:#{external_user_id}"
hash = :crypto.hash(:sha256, raw) |> Base.encode16(case: :lower)
"ch_em_#{String.slice(hash, 0, 32)}"
# e.g. "ch_em_b8c7f2e1..." (max 39 chars)
```

All derived session_ids are:

- Stable across adapter restarts (same inputs, same output).
- Bounded in length (≤39 chars).
- Not atoms (never created with `String.to_atom/1`).
- Not reversible to external identifiers (one-way hash).

The `session_id` is passed in `AllbertAssist.Runtime.submit_user_input/1`
requests so the v0.14 scratchpad can associate per-channel session state.
`channel_events` rows store the derived `session_id` for self-contained audit
records.

## Telegram Polling And Offset Management

Telegram's `getUpdates` API accepts an `offset` parameter. The adapter must
advance the offset to `max(update_id) + 1` after each processed batch.

### Offset Derivation Across Restarts

On `AllbertAssist.Channels.Telegram.Adapter.init/1`:

1. Call `AllbertAssist.Channels.max_inbound_integer_event_id("telegram")`.
2. Returns the highest `external_event_id` parsed as integer across
   `direction IN ('inbound', 'callback')` rows for `channel = "telegram"`.
   Returns `0` when no events exist.
3. First `getUpdates` offset = `max_event_id + 1`.

Events inserted with status `"rejected"` count as acknowledged (the adapter
will not re-process them). The unique-index is the second dedup guard.

### Telegram Polling State Machine

The adapter `GenServer` state:

```elixir
%{
  enabled:       boolean(),
  offset:        non_neg_integer(),
  backoff_ms:    non_neg_integer(),    # 0 when healthy
  poll_interval: pos_integer(),        # channels.telegram.poll_interval_ms
  poll_timeout:  pos_integer()         # channels.telegram.poll_timeout_seconds
}
```

Loop: `init/1` → disabled/misconfigured returns idle; valid config derives
offset and schedules `:poll`. `handle_info(:poll)` calls `Client.get_updates`,
processes each update, advances `offset` in state, resets or increases
`backoff_ms`, and schedules next `:poll`. Provider errors increase `backoff_ms`
exponentially (capped at 60 000 ms).

## Email IMAP Polling And Offset Management

Email does not use a sequential integer offset. Dedup is by `Message-ID`.

### IMAP Polling State Machine

The adapter `GenServer` state:

```elixir
%{
  enabled:       boolean(),
  backoff_ms:    non_neg_integer(),    # 0 when healthy
  poll_interval: pos_integer()         # channels.email.imap_poll_interval_ms
}
```

Loop: `init/1` → disabled/misconfigured returns idle; valid config opens an
IMAP connection (or marks connection pending), schedules `:poll`.
`handle_info(:poll)`:

1. Open or reuse IMAP connection: `ImapClient.connect/3` → `login/3` →
   `select_mailbox/3`. On connection failure: log redacted error, increase
   `backoff_ms`, schedule retry.
2. `ImapClient.search_unseen/1` → list of UIDs.
3. For each UID:
   a. `ImapClient.fetch_message/2` → raw RFC 5322 bytes.
   b. `Email.Parser.parse_email/1` → extract `from_address`, `message_id`,
      `subject`, `text_body`, etc.
   c. Attempt `Channels.create_event/1` with `external_event_id = message_id`.
      If the unique index fires: already processed, skip; still mark `\Seen`.
   d. On new event: route to text message or command flow.
   e. `ImapClient.mark_seen/2` after successful event insertion.
4. Close or keep-alive IMAP connection based on poll interval and settings.
5. Reset `backoff_ms`, schedule next `:poll` after `poll_interval`.

Mid-batch crash recovery: on restart, `search_unseen` re-fetches any message
not yet marked `\Seen`. If its `message_id` is already in `channel_events`, the
unique index prevents double-processing; the adapter still calls `mark_seen` and
moves on.

_Placeholder for IMAP IDLE_: A future `idle` mode would replace the periodic
`:poll` tick with a persistent IMAP connection that calls
`ImapClient.idle/2` and receives server-push `EXISTS` notifications, triggering
an immediate fetch rather than waiting for the poll interval.

## Supervision And Lifecycle

`AllbertAssist.Channels.Supervisor` is a `:one_for_one` supervisor. Full boot
order after v0.16:

1. `AllbertAssist.Repo` + migrations
2. `DNSCluster`
3. `Phoenix.PubSub`
4. `Jido.Signal.Bus`
5. `AllbertAssist.Jido`
6. `AllbertAssist.App.Supervisor` (v0.15)
7. `AllbertAssist.Session.Scratchpad` (v0.14)
8. `AllbertAssist.Jobs.Scheduler` (v0.13)
9. **`AllbertAssist.Channels.Supervisor`** (v0.16)

`Channels.Supervisor` children (`:one_for_one`, v0.16):

1. `AllbertAssist.Channels.Telegram.Adapter`
2. `AllbertAssist.Channels.Email.Adapter`

Each adapter reads its settings at `init/1` and either starts polling or stays
idle with diagnostics. A crashed adapter is restarted by `:one_for_one` without
affecting the other adapter. Exponential backoff handles provider errors within
the adapter's own state; the supervisor restart is reserved for process crashes.

Adapter startup rules (same for both):

1. Read `channels.<provider>.*` settings from Settings Central.
2. If `enabled` is `false`: log disabled diagnostics, return idle
   `{:ok, %{enabled: false}}`. No poll scheduled.
3. Resolve credentials from Settings Secrets. If missing or invalid: log
   diagnostics, remain idle. Must not crash-loop.
4. Validate remaining settings. Invalid settings → idle with diagnostics.
5. On valid configuration: initialize state, schedule first poll tick.

Adapter restart reads settings fresh in `init/1`. A settings change + adapter
kill picks up new settings without application restart.

## Configuration

v0.16 does not introduce new `Application.get_env/3` config keys for channels.
`AllbertAssist.Channels.Supervisor` is wired unconditionally; adapter
enabled/disabled state is controlled by `channels.<provider>.enabled` in
Settings Central, read at adapter `init/1`.

Tests may set Settings Central values in a temp Allbert home or override
specific settings before the supervisor starts. Both adapters must tolerate
disabled or missing settings without crashing.

### M5: Runtime Submission And Response Rendering

Status: implemented.

Implement the full inbound flow and rendering for both channels.

**Telegram text message flow:**

1. Parse update → `{:text_message, fields}`.
2. Validate text byte size against `max_text_bytes`.
3. Reject group chats unless `allow_group_chats` is true and chat id is listed.
4. Resolve identity → `{:ok, user_id}` or reject.
5. Derive `session_id` from channel/user_id/chat_id.
6. Detect `/new` command.
7. Build and submit runtime request.
8. Render response via `Telegram.Renderer.render_response/2`.
9. Send via `Telegram.Client.send_message/3`.
10. Update `channel_events` row with `thread_id`, `trace_id`, `status`.

**Email text message flow:**

1. Fetch and parse email → `from_address`, `message_id`, `text_body`, etc.
2. Insert `channel_events` row (unique index dedup guard).
3. Check `text_body` for command patterns via `Email.Parser.detect_command/1`.
   Command messages route to M6 approval flow, not to runtime submission.
4. Validate body byte size against `max_body_bytes`.
5. Resolve identity from `from_address`.
6. Derive `session_id` from channel/from_address (no chat_id).
7. Build runtime request:

```elixir
%{
  text:        cleaned_text_body,
  channel:     "email",
  user_id:     resolved_user_id,
  operator_id: resolved_user_id,
  session_id:  session_id,
  metadata: %{
    channel:             "email",
    provider:            "email_imap",
    external_event_id:   message_id,
    external_user_id:    from_address,
    external_chat_id:    nil,
    external_message_id: imap_uid_str,
    subject:             subject,
    in_reply_to:         in_reply_to
  }
}
```

8. Submit via `AllbertAssist.Runtime.submit_user_input/1`.
9. Render response via `Email.Renderer.render_response/2`.
10. Send reply via `Email.SmtpClient.send/5` with reply subject and headers.
11. Update `channel_events` row.

**Telegram rendering:**

- Normal responses: text chunks within Telegram limits.
- Approval Handoff: text + inline keyboard when callback data fits; typed
  commands otherwise.

**Email rendering:**

- Normal responses: plain-text email reply. No HTML in v0.16 unless
  `allow_html_replies` is true.
- Approval Handoff: plain-text email with:

```text
Allbert needs your approval:
  Action: <title>
  Risk: <risk>
  Resource: <resource_uri>
  Operation: <operation_class>

Reply with one of:
  approve <confirmation_id>
  deny <confirmation_id>
  show <confirmation_id>
```

Typed commands are case-insensitive. Confirmation id is included in the reply
body so the operator can copy-paste or use a reply client.

Thread behavior (both channels):

- Normal messages rely on v0.12 default thread selection.
- `/new <message>` (Telegram) or `Subject: New: <prompt>` convention (email,
  optional) creates a new thread. v0.16 may support only the Telegram `/new`
  command; email new-thread behavior is runtime default.
- Conversation history remains SQLite `Thread`/`Message`. No markdown memory
  promotion.

Implementation expectations:

- Full text message flow for Telegram: parse → validate → resolve → session_id
  → runtime → render → send → update event.
- Full text message flow for email: parse → dedup → detect command → validate
  → resolve → session_id → runtime → render → send reply → update event.
- `thread_id` and `trace_id` from runtime are written back to `channel_events`.
- Email renderer uses `in_reply_to` and sets `References` header so replies
  thread correctly in standard mail clients.
- Tests: Telegram mapped text → runtime submission → Thread/Message persisted;
  email mapped text → runtime submission; both: unmapped rejected, oversized
  rejected, runtime metadata fields correct.

### M6: Confirmation Callback Flow

Status: implemented.

Implement confirmation resolution for both channels.

**Telegram callback buttons:**

Callback data: `allbert:v1:<action>:<confirmation_id>` where action is
`approve`, `deny`, or `show`. No resource targets, commands, URLs, or
credentials embedded. Falls back to typed commands if callback data length
limit would be exceeded.

Resolution flow:

1. Parse callback data; reject malformed/overlong/unknown versions.
2. Insert callback `channel_events` row.
3. Resolve identity via `Identity.resolve/3`.
4. Call registered confirmation action via `AllbertAssist.Actions.Runner.run/3`
   with resolver metadata: channel, provider, external user/chat ids, callback
   query id, mapped user_id, session_id.
5. Render confirmation state back to Telegram.
6. `answer_callback_query/3` to dismiss the loading state.

**Email typed command callbacks:**

When `Email.Parser.detect_command/1` returns `{:command, action, confirmation_id}`:

1. Action is `"approve"`, `"deny"`, or `"show"`.
2. Insert callback `channel_events` row with direction `"callback"`.
3. Resolve identity from `from_address` via `Identity.resolve/3`.
4. Call registered confirmation action via `AllbertAssist.Actions.Runner.run/3`
   with resolver metadata: channel `"email"`, provider `"email_imap"`,
   `external_user_id` (sender address), `user_id`, `session_id`, original
   `message_id`, `in_reply_to`.
5. Render confirmation state via `Email.Renderer.render_approval_handoff/2`
   (shows current state) and send a reply email.

Approval scope rules (both channels):

- The action may approve, deny, or show.
- Remembered resource grants only through scopes exposed by v0.11/Security Central.
- Approval from one channel for one operation class does not authorize another.

Already-resolved: render current state without re-running the action.

Implementation expectations:

- Telegram callback flow fully implemented and tested.
- Email command flow fully implemented and tested.
- Tests cover: Telegram approve/deny/show, email approve/deny/show, unmapped
  sender rejected, already-resolved renders state, malformed callback/command
  rejected, resolver metadata propagated to confirmation action context.

## Handoffs

### v0.17 Handoff

v0.17 introduces `StockSage.App`. Both Telegram and email channels can route
StockSage-context requests immediately after v0.16. If a user's v0.14 scratchpad
carries `active_app: :stocksage` for their `session_id`, runtime routes
StockSage actions the same way for both channels. No channel-specific app
routing is needed.

### v0.18 Handoff

v0.18 memory review must not auto-promote channel conversation turns from
`channel_events` or `Thread`/`Message` rows. The v0.12 constraint (explicit
confirmation required for markdown memory promotion) applies equally to Telegram
and email turns. Email subjects and text bodies stored in `payload_summary` are
redacted diagnostics, not memory sources.

### v0.20 Handoff

v0.20 cross-surface intent enrichment can use `channel`, `session_id`,
`active_app`, and `thread_id` from incoming requests as additional routing
signals. v0.16 already carries all four for both channels. v0.20 may prefer
more concise response formats for `channel: "telegram"` and more detailed
formats for `channel: "email"`.

### v0.23 Handoff

v0.23 security evals must cover v0.16-introduced surfaces for both channels:

- Cross-user thread leakage through identity spoofing (Telegram user id swap;
  email `From` header spoofing).
- Callback replay: Telegram callback data sent twice; email `approve <id>`
  replied twice.
- Provider payload injection: malicious `text` (Telegram) or email body
  attempting privilege escalation, memory writes, or action injection.
- Email header injection in rendered replies (malicious `subject` or
  `from_address`).
- Resource approval scope leakage through callbacks/typed commands.
- Secret redaction in IMAP/SMTP connection errors and `channel_events` records.
- Cross-user channel event read paths.
- Session_id collision probability and boundary analysis.

### v0.26 Handoff

v0.26 workspace surface work should treat the v0.11 Approval Handoff format,
the Telegram compact callback format, and the email typed-command format as the
three reference points for how canvas/ephemeral UI extensions should structure
approval-resolution paths. All three converge on the same `Actions.Runner.run/3`
path; v0.26 need not reinvent confirmation resolution.
