# v0.16 Request Flow: Additional Channels

Status: planned.

This document names the implementation flow for v0.16's two additional
channels: Telegram and email. It is intentionally concrete so the
implementation can be tested without drifting into a generic provider
framework.

## Common Shape

Every v0.16 channel interaction follows the same boundary posture:

1. `AllbertAssist.Channels.Supervisor` starts after `Jobs.Scheduler` in the
   application supervision tree. Each adapter reads settings at `init/1` and
   either starts polling or stays idle with diagnostics.
2. Inbound provider updates enter the adapter through the polling loop. Each
   update is parsed by the channel parser, a `channel_events` row is inserted
   (dedup guard), and the update is dispatched to the text message or callback
   flow.
3. The text message flow resolves external identity through
   `AllbertAssist.Channels.Identity.resolve/3`, derives a bounded `session_id`,
   builds a runtime request, and calls
   `AllbertAssist.Runtime.submit_user_input/1`. The returned `thread_id` and
   `trace_id` are written back to the `channel_events` row.
4. The callback/command flow resolves identity, inserts a callback or command
   `channel_events` row, calls the appropriate registered confirmation action
   through `AllbertAssist.Actions.Runner.run/3`, and renders the confirmation
   state back through the channel.
5. All rendering goes through the channel's Renderer module. Telegram uses inline
   keyboard buttons and splits oversized text into bounded chunks. Email uses
   plain-text replies with typed command instructions for confirmations and
   chunks long responses into bounded paragraphs.

Channel adapters call the runtime. They do not call private action modules,
mutate confirmations directly, bypass Security Central, write memory, or
create new security policies.

## Telegram Channel

### Startup Flow

1. Application supervision starts `AllbertAssist.Channels.Supervisor`.
2. The supervisor reads Settings Central channel settings.
3. If `channels.telegram.enabled` is `false`, the supervisor records disabled
   diagnostics and starts no polling work.
4. If Telegram is enabled, the adapter validates:
   - `bot_token_ref` resolves to an available secret
   - `identity_map` is valid
   - `allowed_chat_ids` and group-chat settings are valid
   - polling intervals are within bounds
5. Invalid configuration leaves the adapter idle with diagnostics. It must not
   crash-loop.
6. Valid configuration starts one Telegram polling process under supervision.

### Settings And Identity Setup Flow

Operator command:

```text
mix allbert.channels telegram set-token TOKEN
```

Expected behavior:

- Writes the token through Settings Secrets at
  `secret://channels/telegram/bot_token`.
- Leaves `channels.telegram.bot_token_ref` pointing at that secret ref.
- Redacts the token in all output, traces, logs, and tests.

Operator command:

```text
mix allbert.channels telegram map --external-user 123 --user alice
```

Expected behavior:

- Adds or updates one enabled identity map entry.
- Requires both values to be non-empty strings.
- Does not create accounts, roles, auth users, or foreign keys.
- Does not allow duplicate `external_user_id` entries.

Unmapping removes or disables the entry. After unmapping, the external identity
cannot submit runtime input.

### Telegram Poll Flow

#### Startup Offset Derivation

Before the first poll, the adapter derives its starting offset:

1. Call `AllbertAssist.Channels.max_inbound_integer_event_id("telegram")`.
2. This queries `channel_events` for the maximum `external_event_id` (parsed as
   integer) across `direction IN ('inbound', 'callback')` rows.
3. Returns `0` when no events exist; otherwise returns the highest known
   `update_id`.
4. First `getUpdates` call uses `offset = max_event_id + 1`.

This ensures the adapter resumes after the last acknowledged update on restart,
even if the previous run crashed mid-batch. Events already in `channel_events`
are not re-processed; any events in a partial batch that were not yet inserted
are re-fetched and handled (the dedup index prevents double-processing).

#### Per-Poll Loop

1. The adapter calls `AllbertAssist.Channels.Telegram.Client.get_updates/3`
   with the current offset and the configured `poll_timeout_seconds`.
2. Each provider update is parsed by `AllbertAssist.Channels.Telegram.Parser.parse_update/1`
   into one of:
   - supported text message
   - supported callback query
   - unsupported media/document/voice/sticker/other update
   - malformed update
3. The adapter derives `external_event_id` from the Telegram update id (an
   integer serialized as a string, e.g. `"12345678"`).
4. It attempts to insert a `channel_events` row with status `"received"` and
   `direction: "inbound"` (or `"callback"` for callback queries).
5. If the partial unique index on `{channel, external_event_id}` rejects the
   insert, the update has already been processed or rejected and is skipped.
6. Supported text messages continue to the text message flow.
7. Supported callback queries continue to the callback flow.
8. Unsupported or malformed updates are marked `"rejected"` with a bounded,
   redacted reason and never call the runtime.
9. After processing all updates in the batch, the adapter advances its in-process
   offset to `max(update_id) + 1`. This offset is also the value that will be
   derived fresh from `channel_events` on next restart, so no separate state
   table is needed.
10. On provider error: the adapter logs a redacted error (no token, no raw body),
    applies exponential backoff (capped at 60 seconds), and schedules the next
    poll after the backoff delay.

### Telegram Text Message Flow

1. Validate text byte size against `channels.telegram.max_text_bytes`.
2. Reject group chats unless:
   - `channels.telegram.allow_group_chats` is true
   - the chat id is listed in `channels.telegram.allowed_chat_ids`
3. Resolve Telegram `external_user_id` through
   `channels.telegram.identity_map`.
4. If unmapped or disabled:
   - mark the event `"rejected"`
   - optionally send a short "not connected" response
   - do not call `AllbertAssist.Runtime.submit_user_input/1`
   - do not create a thread or message
5. Derive a deterministic bounded `session_id` from channel, external user id,
   and chat id:

```elixir
raw = "telegram:#{external_user_id}:#{external_chat_id}"
hash = :crypto.hash(:sha256, raw) |> Base.encode16(case: :lower)
"ch_tg_#{String.slice(hash, 0, 32)}"  # max 39 chars
```

6. Detect supported channel command prefixes:
   - `/new <prompt>` sets `new_thread: true` and uses `<prompt>` as input
   - plain text uses normal v0.12 default thread selection
7. Build a runtime request:

```elixir
%{
  text: prompt_text,
  channel: "telegram",
  user_id: resolved_user_id,
  operator_id: resolved_user_id,
  session_id: session_id,
  new_thread: new_thread?,
  metadata: %{
    channel: "telegram",
    provider: "telegram_bot_api",
    external_event_id: update_id,
    external_user_id: telegram_user_id,
    external_chat_id: chat_id,
    external_message_id: message_id
  }
}
```

8. Submit through `AllbertAssist.Runtime.submit_user_input/1`.
9. Store returned `thread_id`, `trace_id`, and signal id on the channel event.
10. Render the runtime response back to Telegram.
11. Mark the event `"processed"` or `"failed"`.

### Thread Semantics (Telegram)

- v0.16 does not create channel-specific thread tables.
- If no `thread_id` is passed, runtime uses v0.12 behavior: the user's most
  recently updated general thread, or a new one if none exists.
- `/new` always creates a new general thread for the resolved user.
- Channel event records reference the thread; they are not the conversation
  history source of truth.
- Conversation turns are not promoted to markdown memory.

### Telegram Response Rendering Flow

Normal runtime responses:

1. Take the runtime response text and metadata.
2. Redact secrets and sensitive values.
3. Split or truncate to Telegram-safe lengths.
4. Send via the Telegram client `sendMessage`.
5. Record outbound delivery status without raw token or raw provider response
   dumps.

Approval Handoff responses:

1. Detect the v0.11 Approval Handoff result.
2. Render the handoff lines:
   - action title
   - reason/risk
   - resource URI or target summary
   - operation class
   - access mode
   - scope choices
   - confirmation id
3. Render provider buttons only when callback data can safely fit the compact
   callback format.
4. When buttons cannot be safely rendered, include typed commands/instructions
   instead.
5. The rendered message must make it clear that no hidden execution happened.

### Telegram Confirmation Callback Flow

Callback data examples:

```text
allbert:v1:approve:<confirmation_id>
allbert:v1:deny:<confirmation_id>
allbert:v1:show:<confirmation_id>
```

Flow:

1. Parse callback data. Reject unknown versions, unknown actions, malformed
   confirmation ids, and overlong payloads.
2. Resolve the callback sender through the identity map.
3. If unmapped, acknowledge with a rejection message and do not mutate the
   confirmation.
4. Insert a `channel_events` row with direction `"callback"`.
5. Execute the matching registered confirmation action through
   `AllbertAssist.Actions.Runner.run/3`.
6. Pass origin/resolver metadata:
   - channel `"telegram"`
   - provider `"telegram_bot_api"`
   - external user id
   - external chat id
   - callback query id
   - mapped `user_id`
   - `session_id`
7. Render the resulting confirmation state back to Telegram.
8. Do not embed resource targets, commands, URLs, credentials, or remembered
   grant scopes directly in callback data.

### Telegram Failure Modes

Disabled channel:

- `show telegram` reports disabled.
- Polling does not start.
- `simulate` remains available as a local synthetic update path. It does not
  contact Telegram, does not require a bot token, and does not start polling,
  but it still requires an explicit identity mapping.

Missing token:

- Adapter stays idle with diagnostics.
- CLI output says token is missing without printing any secret ref contents.

Unmapped external user:

- Event is recorded as rejected.
- Runtime is not called.
- No thread/message is created.

Oversized text:

- Event is rejected before runtime.
- The response says the message is too large.

Unsupported media/document:

- Event is rejected.
- No file download or document extraction occurs.

Provider error:

- Adapter marks affected event failed when applicable.
- Poller backs off and remains supervised.
- Token and raw response bodies stay redacted.

Duplicate update:

- Unique insert detects the duplicate.
- Runtime is not called again.
- Existing event is left as the source of truth.

Already-resolved confirmation:

- Callback renders current confirmation state.
- It does not reopen or re-run the action.

### Telegram CLI Flows

```text
mix allbert.channels list
```

Shows configured channel ids, enabled/disabled status, diagnostics, and last
event summary.

```text
mix allbert.channels show telegram
```

Shows Telegram settings summary, identity-map count, allowed chat count,
polling status, and diagnostics. Secrets and raw identity details are redacted
or summarized unless the command explicitly names a safe operator-inspection
mode.

```text
mix allbert.channels telegram simulate --external-user 123 --chat 456 "hello"
```

Creates a synthetic Telegram text update and runs the normal channel flow. This
is the primary manual acceptance path and must use a temporary Allbert home in
tests.

```text
mix allbert.channels telegram poll-once
```

Runs one bounded provider poll for manual verification. It does not leave a
background loop running.

## Email Channel

### Email Startup Flow

1. Application supervision starts `AllbertAssist.Channels.Supervisor`.
2. The supervisor reads Settings Central channel settings.
3. If `channels.email.enabled` is `false`, the supervisor records disabled
   diagnostics and starts no polling work.
4. If email is enabled, the adapter validates:
   - `imap_password_ref` resolves to an available secret
   - `smtp_password_ref` resolves to an available secret
   - `imap_host`, `imap_port`, `imap_mailbox` are non-empty and valid
   - `smtp_host`, `smtp_port`, `from_address` are non-empty and valid
   - `identity_map` is valid
   - poll interval is within bounds
5. Invalid configuration leaves the adapter idle with diagnostics. It must not
   crash-loop.
6. Valid configuration starts one email polling process under supervision.

### Email Settings And Identity Setup Flow

Operator command:

```text
mix allbert.channels email set-password --imap PASSWORD
```

Expected behavior:

- Writes the IMAP password through Settings Secrets at
  `secret://channels/email/imap_password`.
- Leaves `channels.email.imap_password_ref` pointing at that secret ref.
- Redacts the password in all output, traces, logs, and tests.

```text
mix allbert.channels email set-password --smtp PASSWORD
```

Expected behavior:

- Writes the SMTP password through Settings Secrets at
  `secret://channels/email/smtp_password`.
- Leaves `channels.email.smtp_password_ref` pointing at that secret ref.
- Redacts the password in all output, traces, logs, and tests.

Operator command:

```text
mix allbert.channels email map --external-user alice@example.com --user alice
```

Expected behavior:

- Adds or updates one enabled identity map entry keyed by email address.
- Requires both values to be non-empty strings.
- Does not create accounts, roles, auth users, or foreign keys.
- Does not allow duplicate `external_user_id` entries.

Unmapping removes or disables the entry. After unmapping, messages from that
sender are rejected without calling the runtime.

### Email IMAP Poll Flow

#### Startup State

Email adapters do not derive an integer offset on startup. Dedup is provided by
the `channel_events` partial unique index on `{channel, external_event_id}`, where
`external_event_id` is the RFC Message-ID header value. Any UNSEEN message
already present in `channel_events` will be detected as a duplicate and skipped;
any message not yet in `channel_events` will be processed normally.

The adapter also relies on the IMAP `\Seen` flag: messages successfully
inserted (or detected as duplicates) are marked `\Seen` after processing. If the
adapter crashes before marking `\Seen`, it will re-fetch the same message on
restart, but the dedup index will prevent double-runtime submission.

#### Per-Poll Loop

1. The adapter calls `AllbertAssist.Channels.Email.ImapClient.connect/3`
   (if not already connected) using the configured host, port, SSL setting, and
   credentials.
2. It calls `ImapClient.select_mailbox/3` for the configured mailbox (default
   `"INBOX"`).
3. It calls `ImapClient.search_unseen/1` to retrieve a list of unseen message
   UIDs.
4. For each UID:
   a. Call `ImapClient.fetch_message/2` to retrieve headers and body.
   b. Parse the raw MIME email through
      `AllbertAssist.Channels.Email.Parser.parse_email/1`.
   c. Extract the `Message-ID` header value as `external_event_id`.
   d. Attempt to insert a `channel_events` row with status `"received"` and
      `direction: "inbound"`.
   e. If the partial unique index on `{channel, external_event_id}` rejects the
      insert, the message is a duplicate. Mark it `\Seen` and continue.
   f. If the body exceeds `channels.email.max_body_bytes`, reject with
      status `"rejected"` and reason `"oversized"`. Mark `\Seen` and continue.
   g. If only HTML is available and `channels.email.allow_html_replies` is
      false, reject with reason `"html_only"`. Mark `\Seen` and continue.
   h. Extract sender address as `external_user_id` from `From:` header.
   i. Detect command patterns in the reply body via
      `AllbertAssist.Channels.Email.Parser.detect_command/1`.
   j. If a command is detected, route to the email command flow.
   k. Otherwise, route to the email text message flow.
   l. Mark the message `\Seen` after processing (regardless of success/failure).
5. After processing all UNSEEN messages, the adapter closes or idles the IMAP
   connection according to the configured `imap_poll_interval_ms`.

   <!-- IMAP IDLE placeholder: In a future release, the poll loop above can be
   replaced by an IMAP IDLE session that receives server push notifications
   instead of polling. This would require a persistent TCP connection and proper
   IDLE timeout/re-issue handling. The dedup and processing logic (steps 4d–4l)
   remain identical. See future-features.md for prerequisites. -->

6. On IMAP connection error: the adapter logs a redacted error (no password, no
   raw server response), applies exponential backoff (capped at 60 seconds), and
   schedules the next poll after the backoff delay.

### Email Text Message Flow

1. Extract the plain-text body (prefer plain-text; fall back to HTML-stripped
   content only if `allow_html_replies` is true and HTML-only).
2. Validate body byte size against `channels.email.max_body_bytes`.
3. Extract `external_user_id` from the `From:` header (normalized lower-case
   email address).
4. Resolve `external_user_id` through `channels.email.identity_map`.
5. If unmapped or disabled:
   - mark the event `"rejected"`
   - optionally send a short "not connected" reply via SMTP
   - do not call `AllbertAssist.Runtime.submit_user_input/1`
   - do not create a thread or message
6. Derive a deterministic bounded `session_id` from channel and external user id
   (email has no chat concept; each sender address is its own session):

```elixir
raw = "email:#{external_user_id}"
hash = :crypto.hash(:sha256, raw) |> Base.encode16(case: :lower)
"ch_em_#{String.slice(hash, 0, 32)}"  # max 39 chars
```

7. Detect `/new` as the first line of the subject or body to request a new
   thread; otherwise use normal v0.12 default thread selection.
8. Build a runtime request:

```elixir
%{
  text: prompt_text,
  channel: "email",
  user_id: resolved_user_id,
  operator_id: resolved_user_id,
  session_id: session_id,
  new_thread: new_thread?,
  metadata: %{
    channel: "email",
    provider: "email_imap",
    external_event_id: message_id_header,
    external_user_id: from_address,
    external_chat_id: nil,
    external_message_id: message_id_header,
    subject: email_subject,
    in_reply_to: in_reply_to_header
  }
}
```

9. Submit through `AllbertAssist.Runtime.submit_user_input/1`.
10. Store returned `thread_id`, `trace_id`, and signal id on the channel event.
11. Render the runtime response back via SMTP reply.
12. Mark the event `"processed"` or `"failed"`.

### Thread Semantics (Email)

- Same as Telegram: no channel-specific thread tables.
- Each sender's `session_id` is stable across restarts; it maps to the same
  runtime session.
- `/new` in the subject or as the first line of the body creates a new general
  thread for the resolved user.
- The `in_reply_to` email header is surfaced in metadata for traceability but
  does not drive Allbert thread selection.
- Conversation turns are not promoted to markdown memory.

### Email Response Rendering Flow

Normal runtime responses:

1. Take the runtime response text and metadata.
2. Redact secrets and sensitive values.
3. Wrap text into bounded plain-text paragraphs (max `channels.email.max_text_bytes`
   per reply; if the response exceeds this, split into multiple sequential emails
   or truncate with a notice).
4. Send via `AllbertAssist.Channels.Email.SmtpClient.send/5`:
   - `To:` is the resolved sender address
   - `From:` is `channels.email.from_address` / `channels.email.from_name`
   - `Subject:` is `Re: <original subject>` (bounded, no secrets)
   - `In-Reply-To:` is the original `Message-ID` header
   - Body is plain text only (no HTML, no attachments)

   <!-- SMTP provider API placeholder: In a future release, SmtpClient can be
   backed by Mailgun, SendGrid, or another transactional email API through a
   bounded provider adapter. The send/5 interface remains unchanged; provider
   selection becomes a Settings Central key. -->

5. Record outbound delivery status without raw password or raw server response
   dumps.

Approval Handoff responses:

1. Detect the v0.11 Approval Handoff result.
2. Render the handoff as a plain-text email body including:
   - action title
   - reason/risk
   - resource URI or target summary
   - operation class
   - access mode
   - scope choices
   - confirmation id
3. Include typed command instructions in the email body:

```text
To approve, reply with this exact line (copy and paste):
  ALLBERT:APPROVE:<confirmation_id>

To deny:
  ALLBERT:DENY:<confirmation_id>

To see current status:
  ALLBERT:SHOW:<confirmation_id>
```

4. The rendered email must make it clear that no hidden execution happened.
5. Do not include resource targets, shell commands, URLs to fetch, or credential
   values in the typed command instructions.

### Email Confirmation Command Flow

Command patterns detected by `Email.Parser.detect_command/1` in the reply body:

```text
ALLBERT:APPROVE:<confirmation_id>
ALLBERT:DENY:<confirmation_id>
ALLBERT:SHOW:<confirmation_id>
```

Detection rules:

- Scan the plain-text body for lines matching the `ALLBERT:<action>:<id>` prefix.
- The pattern is case-insensitive and trimmed.
- `<confirmation_id>` must match a bounded alphanumeric format (no embedded
  commands, URLs, or credentials).
- If no matching line is found, the body is treated as a regular text message.
- If multiple commands are found, only the first is used and the rest are logged
  as rejected extras.

Flow:

1. Resolve the email sender through the identity map.
2. If unmapped, acknowledge with a rejection reply and do not mutate the
   confirmation.
3. Insert a `channel_events` row with direction `"callback"`.
4. Execute the matching registered confirmation action through
   `AllbertAssist.Actions.Runner.run/3`.
5. Pass origin/resolver metadata:
   - channel `"email"`
   - provider `"email_imap"`
   - external user id (from address)
   - external chat id (nil)
   - original message id
   - mapped `user_id`
   - `session_id`
6. Render the resulting confirmation state as a plain-text email reply.
7. Do not embed resource targets, commands, URLs, or credentials in the reply.

### Email Failure Modes

Disabled channel:

- `show email` reports disabled.
- Polling does not start.
- `simulate` remains available as a local synthetic path. It does not contact
  any IMAP or SMTP server, does not require credentials, but still requires an
  explicit identity mapping.

Missing credentials:

- Adapter stays idle with diagnostics.
- CLI output says credentials are missing without printing any secret ref
  contents or server addresses.

IMAP connection error:

- Adapter backs off and retries (exponential, capped at 60 seconds).
- Error logged redacted; no raw server response bodies in traces.

SMTP send failure:

- Event marked `"failed"` with a bounded, redacted error reason.
- No raw SMTP server response in logs or traces.
- Adapter continues polling; SMTP failures do not stop the IMAP poll loop.

Unmapped sender:

- Event recorded as rejected.
- Runtime is not called.
- No thread/message is created.

Oversized body:

- Event rejected before runtime.
- Reply (if sent) says the message is too large.

HTML-only email (when `allow_html_replies: false`):

- Event rejected.
- No HTML rendering, parsing, or script execution occurs.

Malformed Message-ID:

- If `Message-ID` is missing or malformed, the event is rejected with reason
  `"missing_message_id"`.
- The adapter does not invent a synthetic dedup key.

Unsupported attachment:

- Attachment presence does not block text processing.
- Attachments are never downloaded, extracted, or forwarded to the runtime.
- The `payload_summary` field notes the attachment count but not content.

Duplicate email (restart replay):

- Unique index detects duplicate Message-ID.
- Runtime is not called again.
- Message is marked `\Seen`.

Already-resolved confirmation command:

- Command reply renders current confirmation state.
- It does not reopen or re-run the action.

## Shared Signals And Trace Metadata

Emit channel lifecycle signals using the existing signal helper. Both Telegram
and email emit the same event types with provider-specific metadata values.

Signal events:

- `allbert.channel.update_received`
- `allbert.channel.message_rejected`
- `allbert.channel.runtime_submitted`
- `allbert.channel.response_sent`
- `allbert.channel.delivery_failed`
- `allbert.channel.callback_received`

Signal metadata includes:

- `channel` — `"telegram"` or `"email"`
- `provider` — `"telegram_bot_api"` or `"email_imap"`
- `external_event_id` — Telegram update_id string or email Message-ID header
- `external_user_id` — Telegram integer-as-string or email from-address
- `external_chat_id` — Telegram chat id; `nil` for email
- `user_id` — resolved local user id
- `session_id` — derived bounded session id
- `thread_id` — Allbert thread id when available
- `trace_id` — Allbert trace id when available
- `app_id` — when present in the runtime response

Signal metadata excludes:

- bot tokens or email passwords
- raw provider JSON dumps or IMAP/SMTP response bodies
- full oversized message bodies
- secrets, API keys, credentials, and bearer tokens

## Shared Security Boundaries

- Channel adapters call the runtime. They do not call private action modules,
  mutate confirmations directly, bypass Security Central, or write memory.
- Telegram transport can receive updates and send responses only through the
  bounded client.
- Email transport can fetch UNSEEN messages and send replies only through the
  bounded IMAP and SMTP clients.
- Message text is untrusted model/user input and must not become code,
  commands, URLs to fetch, skill sources, or SMTP relay instructions.
- External identity mapping is explicit Settings Central configuration.
- Approval callbacks (Telegram) and approval commands (email) are durable
  confirmation resolution requests, not capability grants.
- Resource Access Security Posture continues to decide every resource access
  operation class independently.
- Email attachment content is never downloaded, parsed, rendered, forwarded to
  the runtime, or used as memory.
