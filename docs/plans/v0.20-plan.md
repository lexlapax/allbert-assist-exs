## Purpose

v0.20 introduces StockSage as the first real shipped plugin workspace app.
It proves that a source-tree plugin can contribute an app, skill roots, local
domain storage, and safe read/write actions without private hooks into Allbert.

This milestone creates the StockSage local data foundation and import path. It
does not run the Python bridge, call market-data APIs, build LiveViews, or
start native trading agents.

## Readiness Decision

v0.20 is ready for implementation when:

- v0.17 plugin contract is implemented and released.
- v0.18 full app/surface contract is implemented and released.
- v0.19 cross-surface intent enrichment is implemented and released so
  `active_app: :stocksage` session context routes `mix allbert.ask` turns to
  registered StockSage actions through the engine without StockSage-specific
  core predicates.
- `AllbertAssist.App.StockSageStub` placeholder from v0.15 is still live and
  ready to be replaced by the real `StockSage.App`.
- ADR 0018 is accepted (accepted as part of M1).
- `AllbertAssist.Repo` and central migrations are the shared SQLite
  infrastructure; no `StockSage.Repo` will be introduced.

## Architecture Readiness Audit

The v0.20 plan is consistent with the current architecture:

- ADR 0014 defines local string `user_id` and optional `thread_id`,
  `session_id`, and `app_id`. v0.20 domain rows carry these as data provenance,
  not as foreign keys to an accounts table.
- ADR 0015 defines the v0.18 full app/surface contract. `StockSage.App`
  implements it from day one; there is no lite→full migration path.
- ADR 0017 defines plugins as contribution packages that do not compile
  arbitrary code at runtime. The host umbrella explicitly compiles reviewed
  shipped source-tree plugin code from `./plugins/stocksage/lib`; the plugin
  manifest does not trigger dynamic module loading.
- ADR 0018 defines the StockSage domain boundary: one SQLite database,
  `stocksage_*` tables, no `StockSage.Repo`, no Python execution, no
  LiveViews, no market-data calls in v0.20.
- ADR 0019 defines intent enrichment invariants. StockSage actions and surfaces
  plug into the engine through registries; `active_app: :stocksage` boosts
  matching candidates without granting additional permissions.

No new ADR is required for v0.20 if implementation follows ADR 0018 and adds
`:stocksage_write` as a normal Security Central permission class under ADR
0006. A new ADR or revision is needed only if v0.20 requires a second database,
a separate repo, a separate migration path, a financial API authorization
model, or a change to the app/plugin authority model.

## Vision And Roadmap Alignment

The vision says StockSage proves Allbert's app model with a concrete
financial-analysis workflow. v0.20 is the domain and data foundation: plugin
and app registration, SQLite domain records, legacy import, safe local actions,
discoverable skills, and focused CLI. It does not build the Python bridge,
native agents, LiveViews, or canvas.

Roadmap dependency order is intact:

- v0.17 supplied the plugin contract and source-tree plugin packages.
- v0.18 supplied the full app/surface contract that `StockSage.App` implements
  from day one.
- v0.19 supplied the intent engine so `active_app: :stocksage` routes assistant
  turns to registered StockSage actions without core-agent StockSage predicates.
- v0.20 is the StockSage data and domain foundation.
- v0.21 adds markdown memory review and retrieval.
- v0.22 adds the Python bridge and `StockSage.Actions.RunAnalysis`.
- v0.23 adds native Jido trading agents.
- v0.25 adds StockSage LiveViews on the v0.18 surface contract.
- v0.26 security evals cover financial authorization and the plugin/domain
  boundaries created here.
- v0.27 matures outcomes/trends and completes memory namespace registration.

v0.20 must not pull v0.21-v0.28 forward. It does not build the Python bridge,
native agents, market-data calls, LiveViews, canvas components, memory namespace
registration, or external resource posture for financial APIs.

## Guiding Rules

- StockSage enters through the plugin layer, not through a private bootstrap
  shortcut.
- StockSage uses the existing local SQLite database through
  `AllbertAssist.Repo`; v0.20 does not add `StockSage.Repo`.
- Every user-facing row carries string `user_id`. No row is created without
  an explicit `user_id`.
- Every read-by-id path also requires `user_id` and returns `:not_found` when
  the row belongs to another user. Durable ids are not authorization.
- Optional context fields use existing Allbert names: `thread_id`,
  `session_id`, `app_id`, `channel`, `input_signal_id`, and `trace_id`.
- Plugin/app registration is not authority. StockSage actions run through
  `AllbertAssist.Actions.Runner`, Security Central, traces, and audits.
- Imported legacy data is provenance-rich and idempotent. Import is read-only
  against the source database and never executes Python.
- Financial data fetches and analysis execution remain unavailable until v0.22
  adds Resource Access posture, confirmations, and bridge execution.
- Settings schema entries contributed through `StockSage.Plugin` validate
  through Settings Central's merged runtime schema; they are not compiled into
  `AllbertAssist.Settings.Schema` at boot.

### Settings Central Schema

`StockSage.Plugin.settings_schema/0` contributes entries under the
`stocksage.*` namespace. These are merged into Settings Central's runtime
schema at read/validation time and are not compiled into
`AllbertAssist.Settings.Schema` at boot.

Initial schema entries:

- `stocksage.import.default_user`: string, default `"local"`. The `user_id`
  applied when no `--user` flag is given to import tasks.
- `stocksage.import.batch_size`: integer, default `500`. Maximum rows per
  insert batch during import.
- `stocksage.import.unknown_tables_as_warnings`: boolean, default `true`.
  Unknown tables in the source database produce diagnostics, not errors.
- `stocksage.list.max_results`: integer, default `50`. Maximum rows returned
  by list CLI tasks and `ListAnalyses`/`GetTrends` actions.
- `stocksage.queue.default_priority`: enum `"low" | "normal" | "high"`,
  default `"normal"`. Default priority for new queue entries.

None of these settings grant execution authority or bypass Security Central.

### Security Central Permission Extension

v0.20 adds one local domain write permission class:

- `:stocksage_write`: local StockSage SQLite domain writes only. Default
  policy: `:allowed`. Default risk tier: `:low`. Settings key:
  `permissions.stocksage_write`.

This permission covers `QueueAnalysis` and StockSage CLI/context write paths
that create or update local `stocksage_*` records. It does not authorize
Python execution, market-data API calls, external services, package installs,
browser/search, local shell execution, or financial-analysis API requests.
Those remain later milestones and must use their own Resource Access posture
and confirmation story.

### Signals And Trace Metadata

StockSage actions use the standard action lifecycle signals emitted by
`AllbertAssist.Actions.Runner`:

- `allbert.action.requested`
- `allbert.action.completed`

Both signals carry `app_id: "stocksage"` through the action capability
metadata registered in `StockSage.Plugin.actions/0`.

Optionally, StockSage context functions may emit bounded informational signals:

- `stocksage.queue.created`: queue entry id, `user_id`, `symbol`, `status`.
- `stocksage.analysis.imported`: `user_id`, count, source, elapsed_ms.

Optional signals are observability only. No action, context function, or test
should require them to complete correctly.

Traces for StockSage actions follow the standard Allbert trace format and
include the selected StockSage action, `app_id: :stocksage`, plugin provenance
from the intent engine, and bounded result metadata. No raw analysis payload,
no unbounded metadata dump.

### Import Contract

```sh
mix stocksage.import_sqlite PATH --user local
mix stocksage.import_sqlite PATH --user alice --dry-run
mix stocksage.import_sqlite PATH --user alice --limit 50
```

Rules:

- Source database is opened read-only. The importer never writes to the source.
- Import accepts only a local filesystem path supplied by the operator.
- Import does not execute Python, run notebooks, fetch market data, call
  external services, install packages, or import skills.
- Unknown legacy tables and columns produce diagnostic warnings, not crashes.
- Rows are inserted or updated idempotently using `legacy_source` and
  `legacy_id` unique constraints.
- Every imported row gets `user_id` from `--user`, defaulting to `"local"`.
- Import returns counts by entity (inserted, updated, skipped, invalid) and
  elapsed time.
- `--dry-run` validates and reports without writing.
- If the original Python `stocksage.db` field shape is not stable, create a
  representative fixture under `plugins/stocksage/test/fixtures/` and document
  the field mapping in `StockSage.Import.SqliteImporter` module docs.

`--user` defaults to `"local"`. If a compatibility `--operator` alias is
offered, it must map to `user_id`; if both flags are present and differ, the
command fails fast before opening the source database.

### First StockSage Actions

Add safe local actions proving plugin-contributed actions without analysis
execution:

- `StockSage.Actions.ListAnalyses`
- `StockSage.Actions.ShowAnalysis`
- `StockSage.Actions.GetTrends`
- `StockSage.Actions.QueueAnalysis`

`QueueAnalysis` creates a local `stocksage_analysis_queue` row only. It does
not call Python, market data, external services, package managers,
browser/search, or native agents.

All actions include capability metadata with `app_id: :stocksage`.
`ListAnalyses`, `ShowAnalysis`, and `GetTrends` use `:read_only` permission.
`QueueAnalysis` uses `:stocksage_write`. All actions execute through
`AllbertAssist.Actions.Runner.run/3` and Security Central.

Action contracts:

- `ListAnalyses`: requires `user_id`; optional `symbol`, `limit`, `offset`;
  returns bounded analysis summaries with `id`, `symbol`, `status`,
  `analysis_date`, `recommendation`, `score`, `source`, and timestamps.
- `ShowAnalysis`: requires `user_id` and `analysis_id`; returns one analysis
  plus bounded details/outcomes. If the id belongs to another user, returns
  `:not_found`.
- `GetTrends`: requires `user_id`; optional `symbol`, `limit`; summarizes
  local imported/completed outcome records only. It does not calculate new
  market outcomes or fetch prices.
- `QueueAnalysis`: requires `user_id` and `symbol`; optional `thread_id`,
  `session_id`, `requested_for`, `priority`; creates one queued row and
  returns its id/status. It does not start a worker.

### CLI

```sh
mix stocksage.import_sqlite PATH [--user USER] [--dry-run] [--limit N]
mix stocksage.analyses list [--user USER] [--symbol SYMBOL]
mix stocksage.analyses show ANALYSIS_ID [--user USER]
mix stocksage.queue create SYMBOL [--user USER] [--thread-id THREAD_ID]

```

`--user` defaults to `"local"`. A compatibility `--operator` alias maps to
`user_id`; if both flags are present and differ, the command fails fast.

CLI output is bounded and never prints raw imported payloads unbounded. Output
contracts:

- `stocksage.import_sqlite`: prints source path summary, user_id, dry-run
  flag, per-entity inserted/updated/skipped/invalid counts, warning count, and
  elapsed_ms. It does not print raw source rows or full payload metadata.
- `stocksage.analyses list`: prints total/returned counts and one bounded row
  per analysis: id, symbol, status, source, recommendation, score,
  analysis_date, updated_at.
- `stocksage.analyses show`: prints one bounded detail view for the scoped
  `user_id`: summary fields, bounded details by section, bounded outcomes, and
  provenance. Cross-user ids print not-found.
- `stocksage.queue create`: prints queue id, user_id, symbol, priority,
  status, requested_for, inserted_at.
- `stocksage.queue list`: prints total/returned counts and bounded queue rows:
  id, symbol, status, priority, requested_for, analysis_id, updated_at.

`docs/plans/v0.20-request-flow.md` describes how those commands feel in the
operator/user flow.

### Intent Integration

After v0.19, the intent engine routes assistant turns to StockSage actions when
the request has `active_app: :stocksage` from a running session or one-turn CLI
context. No StockSage-specific predicates are needed in the core intent agent.

End-to-end user flow:

1. Operator sets StockSage as the active app for a running session, or passes
   equivalent one-turn CLI app context:
   ```sh
   mix allbert.sessions set-active-app --user local --session sess-1 stocksage
   mix allbert.ask --user local --active-app stocksage --trace "list my recent analyses"
   ```
2. In long-running surfaces, subsequent session turns carry the app context:
   ```sh
   mix allbert.ask --user local --session sess-1 --trace "list my recent analyses"
   ```
3. The v0.19 engine sees `active_app: :stocksage`, boosts
   `StockSage.Actions.ListAnalyses` as a candidate, validates it through
   `Actions.Registry`, and selects it.
4. Security Central authorizes the read-only action.
5. `ListAnalyses` runs through `Actions.Runner.run/3` and returns imported
   analyses for `user_id: "local"`.
6. Trace includes `app_id: :stocksage`, the selected StockSage candidate, and
   plugin provenance.
7. The same text in neutral context (`active_app: :allbert`) produces a direct
   answer or clarification, not StockSage data.

This end-to-end path must work by M4/M5 and is covered in the operator smoke.
