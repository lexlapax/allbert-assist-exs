# Allbert v0.20 StockSage Plugin, Umbrella App, And Domain

Formerly tracked as M-D2a.

## Purpose

v0.20 introduces StockSage as the first real plugin-contributed workspace app.
It proves that a source-tree plugin can contribute an app, skill roots, local
domain storage, and safe read/write actions without private hooks into Allbert.

This milestone creates the StockSage local data foundation and import path. It
does not run the Python bridge, call market-data APIs, build LiveViews, or
start native trading agents.

## Expected Inputs

- v0.12 local string `user_id`, optional `thread_id`, and conversation history
  exist.
- v0.14 session scratchpad and `active_app` exist.
- v0.15 minimal app registration contract exists, including
  `AllbertAssist.App.StockSageStub` as a temporary placeholder.
- v0.18 app/surface contract and Surface DSL exist. `StockSage.App`
  implements that contract from day one; there is no lite→full migration.
- v0.17 plugin contract exists, including source-tree plugin packages,
  plugin-contributed apps, skill roots, actions, settings schema entries,
  diagnostics, and explicit compile-path handling.
- ADR 0018 is accepted before implementation begins.
- `allbert-jido-vision.md` is the durable product direction for StockSage and
  the retired `aiworkspace-plan.md` is no longer needed for implementation.

## Guiding Rules

- StockSage enters through the plugin layer, not through a private bootstrap
  shortcut.
- StockSage uses the existing local SQLite database through
  `AllbertAssist.Repo`; v0.20 does not add `StockSage.Repo`.
- Every user-facing/request/queue/domain row carries string `user_id`.
- Optional context fields use existing Allbert names: `thread_id`,
  `session_id`, `app_id`, `channel`, `input_signal_id`, and `trace_id`.
- Plugin/app registration is not authority. StockSage actions still run
  through `AllbertAssist.Actions.Runner`, Security Central, traces, and audits.
- Imported legacy data is provenance-rich and idempotent. Import is read-only
  against the source database and never executes Python.
- Financial data fetches and analysis execution remain unavailable until later
  milestones add Resource Access posture, confirmations, and bridge/native
  execution.

### Application And Plugin Registration

Implement:

- `StockSage.Plugin` using `AllbertAssist.Plugin`.
- `StockSage.App` using the v0.18 full `AllbertAssist.App` contract.
- `StockSage.Supervisor`, returning no long-running analysis workers in v0.20
  unless implementation needs a lightweight domain supervisor.

`StockSage.Plugin` contributes:

- app module: `StockSage.App`
- action modules listed below
- skill root: `plugins/stocksage/skills`
- settings schema entries for local defaults and import behavior
- child spec: `StockSage.Supervisor` or `:ignore` if no children are needed

`StockSage.App` uses:

- `app_id: :stocksage`
- display name: `"StockSage"`
- version matching the app/milestone version
- actions from this milestone only
- skill paths from the plugin contribution
- implements `AllbertAssist.App.SurfaceProvider` with an empty or
  placeholder surface list; LiveView surfaces are added in v0.25

Remove `AllbertAssist.App.StockSageStub` from default app registration once
`StockSage.App` registers. `mix allbert.sessions set-active-app stocksage`
must still work because the real app owns `:stocksage`.

`apps/allbert_assist/priv/repo/migrations`. Table names are prefixed with
`stocksage_`.

Add schemas and context functions for:

- `StockSage.Domain.Analysis`
- `StockSage.Domain.AnalysisDetail`
- `StockSage.Domain.Outcome`
- `StockSage.Domain.AnalysisQueue`
- `StockSage.Domain.QueueRun`
- `StockSage.Domain.MemoryEntry`

Recommended tables:

```text

stocksage_outcomes
stocksage_analysis_queue
stocksage_queue_runs
```

All primary keys use string ids generated by StockSage/Allbert helpers. Use

### Import Contract

Add:

```sh
mix stocksage.import_sqlite PATH --user local
mix stocksage.import_sqlite PATH --user alice --dry-run
mix stocksage.import_sqlite PATH --user alice --limit 50
```

Rules:

- Source database is opened read-only.
- Import accepts only a local filesystem path supplied by the operator.
- Import does not execute Python, run notebooks, fetch market data, or call
  external services.
- Unknown legacy tables/columns are reported as diagnostics, not crashes.
- Rows are inserted or updated idempotently using `legacy_source` and
  `legacy_id` unique constraints where possible.
- Every imported row gets `user_id` from `--user`, default `"local"`.
- Import returns counts by entity, skipped rows, validation errors, and elapsed
  time.
- `--dry-run` validates and reports without writing.

CLI tasks use `--user`, default `"local"`. If a compatibility `--operator`
alias is offered, it maps to `user_id`; if both flags are present and differ,
the command fails fast.

If the original Python `stocksage.db` shape is not stable enough, create a
small representative fixture database under test support and document the
mapping assumptions in the import module docs. Do not defer the import path
entirely.

### First StockSage Actions

Add safe local actions that prove plugin-contributed actions without running
financial analysis:

- `StockSage.Actions.ListAnalyses`
- `StockSage.Actions.ShowAnalysis`
- `StockSage.Actions.GetTrends`
- `StockSage.Actions.QueueAnalysis`

`QueueAnalysis` creates a local queue record only. It does not call Python,
market data, external services, package managers, browser/search capture, or
native agents.

All actions include capability metadata with `app_id: :stocksage`. Read-only
actions use read-only permission posture. `QueueAnalysis` uses the existing
action runner and permission/security path for a local domain write.

### Skills

Add local skills under `plugins/stocksage/skills`:

- `queue-analysis`
- `list-analyses`
- `show-analysis`
- `get-trends`

The skills describe v0.20-safe behavior only. Do not include a `run-analysis`
skill that implies immediate market-data or Python execution before v0.22.
v0.22 can add or expand that skill when `StockSage.Actions.RunAnalysis` lands.

### CLI

Add focused operator CLI tasks:

```sh
mix stocksage.import_sqlite PATH [--user USER] [--dry-run] [--limit N]
mix stocksage.analyses list [--user USER] [--symbol SYMBOL]
mix stocksage.analyses show ANALYSIS_ID
mix stocksage.queue create SYMBOL [--user USER] [--thread-id THREAD_ID]
mix stocksage.queue list [--user USER] [--status queued]
```

CLI tasks use `--user`, default `"local"`. If a compatibility `--operator`
alias is offered, it maps to `user_id`; if both flags are present and differ,
the command fails fast.

CLI output redacts/limits metadata and never prints raw imported payloads
unbounded.

## User Acceptance

- `mix allbert.plugins show stocksage` shows `StockSage.Plugin`, app
  contribution, action contribution, and skill root without granting authority.
- `mix allbert.apps show stocksage` shows the real `StockSage.App`, not
  `AllbertAssist.App.StockSageStub`.
- `mix allbert.sessions set-active-app --user alice --session sess-1 stocksage`
  still succeeds.
- `mix allbert.skills list` includes StockSage skills from
  `plugins/stocksage/skills`.
- `mix stocksage.import_sqlite fixture.db --user alice --dry-run` reports
  counts without writing.
- Importing the same fixture twice for the same user is idempotent.
- `mix stocksage.analyses list --user alice` shows imported analyses; `bob`
  does not see Alice's user-scoped imported rows.
- `mix stocksage.queue create AAPL --user alice` creates a queued record and no
  analysis execution.
- No Python process starts, no market-data API is called, and no LiveView route
  is mounted in v0.20.

## Test Plan

- Plugin registration tests for `StockSage.Plugin`, `StockSage.App`, skill
  roots, action modules, settings schema entries, and diagnostics.
- App-registry tests proving `StockSageStub` is gone/replaced and
  `active_app: :stocksage` still validates.
- Migration/schema tests for every `stocksage_*` table, changeset validation,
  enum validation, required `user_id`, and timestamp types.
- Context tests for user isolation, symbol normalization, queue lifecycle, and
  idempotent updates.
- Import tests for dry-run, idempotency, unknown columns/tables diagnostics,
  invalid path errors, malformed row handling, and user assignment.
- Action tests proving list/show/get-trends/queue-analysis go through
  `Actions.Runner`, carry `app_id: :stocksage`, and do not perform external
  calls.
- Skill registry tests proving StockSage skills come from the plugin root and
  do not claim unavailable v0.22 execution.
- CLI tests for import, analyses list/show, queue create/list, and bounded
  output.
- Full gates from `DEVELOPMENT.md`: compile warnings-as-errors, format check,
  Credo strict, Dialyzer, precommit.

## Non-Goals

- No Python bridge execution; that is v0.22.
- No native Jido trading agents; that is v0.23.
- No StockSage LiveViews or route mounts; that is v0.25.
- No canvas components; that is v0.28.
- No PostgreSQL, Oban hard dependency, hosted auth, accounts, or roles.
- No external market-data calls or financial analysis API calls.
- No `StockSage.Actions.RunAnalysis` until v0.22.
- No automatic promotion of StockSage memory records to markdown Allbert
  memory.
- No plugin-system changes beyond consuming the v0.17 source-tree plugin
  contract.

## Handoffs

- v0.22 adds the Python bridge and `StockSage.Actions.RunAnalysis`, using the
  v0.20 queue, analysis, detail, and provenance records.
- v0.19 intent enrichment can rank StockSage actions only when
  `active_app: :stocksage` or equivalent session evidence exists.
- v0.23 native agents reuse the same analysis and queue records rather than
  introducing another persistence model.
- v0.25 LiveViews are thin over the v0.20 contexts/actions and do not own
  analysis logic.
- v0.26 security evals cover financial authorization and plugin/domain
  boundaries created here.
- v0.18 app/surface contract is already in place; `StockSage.App` implements
  `SurfaceProvider` from day one with placeholder surface metadata.
- v0.27 outcomes/trends mature the v0.20 outcome and memory tables rather than
  replacing them.

## Definition Of Done

v0.20 is done when StockSage exists as a source-tree plugin-contributed local
app with `StockSage.App` registered, `StockSageStub` removed, SQLite-backed
`stocksage_*` domain records, idempotent legacy SQLite import, safe local
StockSage actions, discoverable skills, focused CLI tasks, passing tests, and
clear handoffs to the bridge, native agents, LiveViews, security evals, and
canvas milestones.
