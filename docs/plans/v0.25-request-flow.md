## Pre-Flight

Use a disposable Allbert Home for smoke testing:

```sh
export SMOKE_HOME=$(mktemp -d /tmp/allbert-v025-smoke.XXXXXX)
ALLBERT_HOME="$SMOKE_HOME" mix cmd --app allbert_assist mix ecto.migrate.allbert
```

For fixture-mode native testing, no external credentials or market-data access
are required. For real native testing, operators must configure the model
profile and any data-provider credentials explicitly through documented
Settings Central keys and provider setup.

## Flow 1: Startup And Agent Registration

When Allbert starts with the StockSage plugin available:

1. Plugin discovery loads `./plugins/stocksage`.
2. `StockSage.Plugin.child_spec/1` contributes `StockSage.Supervisor`.
3. `StockSage.Supervisor` starts `StockSage.Agents.Supervisor`.
4. The specialist agents start as supervised Jido/Jido.AI agents.
5. Each agent registers its stable id in
   `AllbertAssist.Objectives.AgentRegistry`.
6. The registry monitors each process. If an agent exits, its registry entry is
   evicted and later delegation returns not-found instead of calling a dead pid.
7. `mix stocksage.agents list` shows registered agent ids and safe metadata.

The operator observes: Allbert starts, StockSage plugin is loaded, and
`mix stocksage.agents list` shows:

- `stocksage.market_context`
- `stocksage.news_sentiment`
- `stocksage.fundamentals`
- `stocksage.bull_thesis`
- `stocksage.bear_thesis`
- `stocksage.decision_synthesizer`
- `stocksage.quality_gate`

If `stocksage.native_enabled` is false, the agents do not register and
`--engine native` returns a clear disabled response before confirmation.

## Flow 2: Inspect A Native Agent

The operator checks whether a specialist agent is callable.

1. Run:

   ```sh
   mix stocksage.agents show stocksage.market_context
   ```

2. The CLI calls a registered read-only action through
   `AllbertAssist.Actions.Runner.run/3`.
3. The action reads safe registry metadata and returns role, module, prompt
   version, status, and last health summary.
4. No raw prompt secrets, model credentials, data-provider credentials, or
   process internals are printed.

The operator observes bounded metadata suitable for debugging startup and
v0.25 readiness.

## Flow 3: Fixture-Mode Agent Smoke

The operator runs one specialist path without external data or model calls.

1. Run:

   ```sh
   mix stocksage.agents smoke AAPL 2026-05-01 --fixture-data --user local
   ```

2. The CLI calls a registered smoke action through the runner.
3. The smoke action uses the same evidence actions as real native analysis,
   but with fixture mode enabled.
4. The request is sent to the relevant specialist agent through
   `AllbertAssist.Objectives.AgentRegistry`.
5. The agent returns a normalized report packet.
6. The CLI prints agent id, status, summary, warnings, prompt version, and
   whether fixture data was used.

The operator observes a complete report packet with no external calls and no
analysis row written. This verifies agent registration and report shape before
running a full analysis.

## Flow 4: Native Analysis Via CLI

The operator requests a native analysis.

1. Run:

   ```sh
   mix stocksage.analyze AAPL 2026-05-01 --engine native --user local
   ```

2. The CLI calls `StockSage.Actions.RunAnalysis` through
   `AllbertAssist.Actions.Runner.run/3`.
3. `RunAnalysis` validates ticker/date/user and confirms native analysis is
   enabled.
4. Security Central evaluates `:stocksage_analyze`. Default remains
   confirmation-required.
5. The action creates a durable confirmation request with engine `native`,
   ticker, date, user id, objective context if present, and disclosure that
   native agents may call configured market-data providers after approval.
6. No specialist agent runs before approval.
7. The operator approves:

   ```sh
   mix allbert.confirmations approve <confirmation_id> --reason "native smoke"
   ```

8. Approval re-enters `RunAnalysis` through the action runner.
9. `RunAnalysis` creates or joins the v0.24 objective context and schedules
   delegate-agent steps.
10. Evidence actions collect fixture or real data packets through registered
    actions and Resource Access posture.
11. Specialist agents run in bounded sequence:
    market context, news/sentiment, fundamentals, bull thesis, bear thesis,
    decision synthesizer, quality gate.
12. The final result persists into `stocksage_analyses` and
    `stocksage_analysis_details` with `engine: "native"`,
    `objective_id`, `step_id`, `trace_id`, report summaries, prompt versions,
    and bounded raw details.
13. The confirmation resolves and CLI prints the analysis id and bounded
    summary.

The operator observes: a normal StockSage analysis row exists, but the engine
is native and the objective contains delegate-agent steps.

## Flow 5: Native Analysis Via Natural Language

The user is in StockSage context and asks Allbert to analyze a ticker.

1. `active_app: :stocksage` is present in the session scratchpad.
2. User asks: `analyze AAPL and compare to MSFT`.
3. v0.19 intent ranking selects StockSage analysis candidates because active
   app evidence exists.
4. v0.24 objective runtime frames a multi-step objective.
5. `RunAnalysis` is invoked for the first ticker and asks for confirmation.
6. After approval, native agents run for AAPL.
7. The objective continues to the second ticker using durable proposer hints.
8. After the second approval, native agents run for MSFT.
9. The objective completes when acceptance criteria are met.

The user observes: confirmation requests, then analysis summaries. The
operator can inspect:

```sh
mix allbert.objectives show <objective_id> --user local
mix stocksage.analyses list --user local
```

The objective shows delegate-agent steps. The analysis list shows both native
analysis rows.

## Flow 6: Explicit Python Comparison

The operator can still invoke the Python bridge, but only as an explicit
comparison/reference run. Allbert never retries native failures through Python
automatically.

1. Run:

   ```sh
   mix stocksage.analyze AAPL 2026-05-01 --engine python --user local
   # or, for a side-by-side parity helper:
   mix stocksage.analyze AAPL 2026-05-01 --compare-python --user local
   ```

2. The CLI and confirmation output clearly label the run as Python comparison
   / reference behavior.
3. The flow follows v0.22 exactly after explicit approval: confirmation,
   approval, Python bridge, persistence.
4. The result row has `engine: "tradingagents"` or the existing Python-engine
   value.

The operator observes: Python is available for fixture parity, similarity
checks, and regression comparison. If native fails, Allbert reports native
failure; it does not silently or automatically run Python. A Python run happens
only when the operator explicitly requests it.

## Flow 7: Default Native Engine Selection

When no `--engine` is provided:

1. `RunAnalysis` uses native.
2. There is no persistent setting that makes Python the default.
3. If `stocksage.native_enabled` is false, the action returns a disabled error
   instead of falling back to Python.
4. Python comparison requires `--engine python` or another explicitly
   documented comparison flag.

The operator observes: default behavior is explicit and inspectable through
Settings Central. v0.25 does not silently switch engines.

## Flow 8: Failure And Recovery

| Case | Expected behavior |
| --- | --- |
| Native disabled | `--engine native` returns a disabled error before confirmation. |
| Confirmation denied | No native agent runs; confirmation resolves denied. |
| Evidence action requires confirmation | Analysis pauses at the existing confirmation boundary; no agent bypasses it. |
| Evidence provider fails | Agent report includes warning or failed status; final result is failed or blocked according to `RunAnalysis` policy. |
| Specialist agent process dies | `AgentRegistry` evicts it; delegate step fails with not-found/dead-agent diagnostic; objective blocks. |
| Model timeout | Agent returns failed/timeout report; objective blocks or final result fails with bounded reason. |
| Quality gate rejects final report | Analysis is failed or blocked; raw invalid report is not surfaced to CLI/traces. |
| Native report exceeds bound | Report is truncated with `truncated: true`; no unbounded trace or detail write. |
| Cross-user analysis show/list | Returns not-found or empty result. |
| Python bridge unavailable | Explicit Python comparison reports the bridge error; native analysis does not depend on the bridge. |

## User Testing Checklist

Use a disposable Allbert Home.

1. Migrate:

   ```sh
   ALLBERT_HOME="$SMOKE_HOME" mix cmd --app allbert_assist mix ecto.migrate.allbert
   ```

2. Verify agents:

   ```sh
   ALLBERT_HOME="$SMOKE_HOME" mix stocksage.agents list
   ALLBERT_HOME="$SMOKE_HOME" mix stocksage.agents show stocksage.market_context
   ```

3. Smoke fixture agent:

   ```sh
   ALLBERT_HOME="$SMOKE_HOME" mix stocksage.agents smoke AAPL 2026-05-01 --fixture-data --user local
   ```

4. Run native analysis:

   ```sh
   ALLBERT_HOME="$SMOKE_HOME" mix stocksage.analyze AAPL 2026-05-01 --engine native --user local
   ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations approve <id> --reason "native smoke"
   ```

5. Inspect:

   ```sh
   ALLBERT_HOME="$SMOKE_HOME" mix stocksage.analyses list --user local
   ALLBERT_HOME="$SMOKE_HOME" mix stocksage.analyses show <analysis_id> --user local
   ALLBERT_HOME="$SMOKE_HOME" mix allbert.objectives show <objective_id> --user local
   ```

6. Verify explicit Python comparison:

   ```sh
   ALLBERT_HOME="$SMOKE_HOME" mix stocksage.analyze AAPL 2026-05-01 --engine python --user local
   ```

7. Verify isolation:

   ```sh
   ALLBERT_HOME="$SMOKE_HOME" mix stocksage.analyses show <analysis_id> --user bob
   ```

Expected result: native path is end-to-end functional, inspectable, and
confirmation-gated; Python comparison runs only when explicitly requested;
cross-user access is blocked.
