### Operator-visible posture (Substrate Reference)

Settings the operator may tune (full list + ranges in
`docs/plans/v0.25-plan.md` Settings Central Schema section):

- `stocksage.native_evidence_mode` â€” default `"live"`; allowed `"live"`
  / `"fixture"` / `"compare"`.
- `stocksage.native_max_debate_rounds` â€” default `2`; cap on
  bull/bear cycles per analysis.
- `stocksage.native_max_risk_rounds` â€” default `1`; cap on risk-debate
  cycles per analysis.
- `stocksage.native_agent_timeout_ms` â€” default `180000`; per-specialist
  dispatch timeout for LLM-backed and deterministic native agents.
- `stocksage.native_parity_variance` â€” default `0.25`; confidence-delta
  threshold for parity acceptance.
- 9 per-agent + 1 global model-profile settings used for Jido.AI
  provider selection and recorded in report metadata.
- `stocksage.native_llm_enabled` â€” default `true`; when `false`,
  non-quality specialists use deterministic advisory packets with the
  same report shape for smoke/tests.
- `stocksage.python_comparison_enabled` â€” default `true`; when `false`,
  `--engine python` and `--engine both` are rejected.

Engine choice is per request. If the operator does not pass an engine
flag, StockSage uses native. Python (`--engine python`) and parity
(`--engine both` / `--compare-python`) are explicit comparison/reference
requests, never a persistent Settings Central default.

For v0.25 native analyses, the operator-managed prerequisites are:

1. `stocksage.native_engine_enabled = true` (default).
2. For live evidence: Resource Access Security Posture grants for the
   evidence-action providers the operator wants.
3. For fixture evidence: no market-data credentials required; fixtures
   ship with the v0.25 release. The M4 fixture smoke runs the
   deterministic advisory path through the same supervised Jido agents
   and evidence actions when `stocksage.native_llm_enabled = false`.
   With the default `true`, fixture evidence still uses Jido.AI for
   non-quality specialist reasoning.

## Authority Posture (unchanged)

Per ADR 0021 Â§4 and ADR 0022:

- Specialist agents are **advisory**. They propose, rank, predict,
  score, summarize. They do NOT authorize execution.
- `:stocksage_analyze` is the single authorization gate for an
  analysis. Confirmation is required per v0.22 baseline.
- The 5 evidence actions reuse the approved analysis context inside
  an approved analysis or use their
  own Resource Access Security Posture grants when called outside
  an analysis (e.g., via `mix allbert.delegate`).
- No agent writes `stocksage_*` rows directly. Only
  `StockSage.Actions.RunAnalysis` persists.
- No agent calls market-data providers directly. They call the 5
  registered evidence actions via `Actions.Runner.run/3` with
  restricted tool scope (each agent's `tools:` list).

## Flow 1: Native Analysis End-To-End (Single-Round)

Example user input: operator runs `mix stocksage.analyze AAPL
2026-05-17 --engine native --user local` against an Allbert Home with
LLM credentials and fixture-mode evidence.

1. CLI invokes `StockSage.Actions.RunAnalysis` via
   `Actions.Runner.run/3` with `engine: "native"`.
2. RunAnalysis validates ticker + analysis_date format.
3. Security Central evaluates `:stocksage_analyze` â†’ returns
   `:needs_confirmation`.
4. RunAnalysis creates a durable confirmation record carrying ticker,
   analysis_date, engine ("native"), user_id, risk tier `high`, and
   a disclosure that the analysis will run native specialist agents,
   Jido.AI generation when enabled, and market-data evidence actions
   when live evidence is requested.
5. CLI displays: confirmation id + approval command (per v0.07).
6. Operator runs `mix allbert.confirmations approve <id> --reason
   "..."`.
7. Approval re-enters RunAnalysis via the v0.07 resume path.
8. RunAnalysis creates a v0.24 objective (`status: :open`, title
   "Analyze AAPL native", `acceptance_criteria` JSON requiring one
   completed native analysis for AAPL on the date).
9. RunAnalysis dispatches `StockSage.Agents.NativeCoordinator.analyze/2`
   with the request packet (objective_id + step_id + ticker +
   analysis_date + user_id + trace_id).
10. The coordinator creates an `objective_steps` row of
    `kind: :delegate_agent` for the first stage: the 3 analyst agents.
11. The coordinator dispatches `market_context`, `news_sentiment`,
    `fundamentals` in parallel through the plugin-owned native agent
    graph. This is bounded stage-local parallelism.
    Each agent uses its declared tools to fetch
    evidence via the registered evidence actions; each returns a
    bounded advisory report packet.
12. Coordinator observes the 3 analyst reports and dispatches next
    stage: bull_thesis â†’ bear_thesis (sequential).
13. Coordinator observes the bull + bear reports and dispatches next
    stage: 3 risk debaters in parallel. Each specialist dispatch is
    bounded by `stocksage.native_agent_timeout_ms`; timeout marks that
    delegate step failed and lets the coordinator continue with a
    warning rather than hanging the whole analysis.
14. Coordinator observes the 3 risk reports and dispatches next
    stage: decision_synthesizer with all 8 prior reports in
    `prior_reports`.
15. Coordinator observes the synthesizer report and dispatches
    final stage: quality_gate (deterministic) to validate report
    shape + redaction + evidence presence.
16. Quality gate passes; coordinator returns synthesized result to
    RunAnalysis.
17. RunAnalysis persists one `stocksage_analyses` row with
    `engine: "native"`, populated final-state fields
    (`final_trade_decision`, `investment_plan`, etc.), and one
    `stocksage_analysis_details` row with the full agent reports +
    debate rounds in `payload` JSON.
18. RunAnalysis emits `allbert.stocksage.analysis_completed` (v0.22
    signal preserved) AND `allbert.objective.completed` AND
    `allbert.runtime.turn.completed` (all three share `trace_id`).
19. CLI displays: analysis id, ticker, date, engine ("native"),
    synthesized decision summary.

The operator observes: durable confirmation + approval + native
analysis completion. Fixture evidence removes market-data latency
variance; LLM latency depends on the configured model profile when
`stocksage.native_llm_enabled = true`.
The resulting analysis is inspectable via
`mix stocksage.analyses show <id>` with engine label and per-agent
breakdown in the details row.

## Flow 2: Native Analysis With Multi-Round Bull/Bear/Risk Debate

Same as Flow 1 but with `stocksage.native_max_debate_rounds = 3` and
`stocksage.native_max_risk_rounds = 2` set in Settings Central. Steps
1-10 identical. Differences begin at step 12:

12. Coordinator observes the 3 analyst reports and dispatches
    bull_thesis round 1.
13. Coordinator observes bull round 1 and dispatches bear_thesis
    round 1 with bull's report in `prior_reports`.
14. Coordinator observes bear round 1 and dispatches bull_thesis
    round 2 with bear's round-1 report in `prior_reports`.
15. (Continues: bear round 2, bull round 3, bear round 3 â€” 3 rounds
    each as configured.)
16. Coordinator proposes 3 risk debaters round 1 in parallel
    (aggressive, conservative, neutral) with all prior bull/bear
    rounds as evidence.
17. Coordinator observes risk round 1 and dispatches risk round 2 in
    parallel.
18. Coordinator proposes decision_synthesizer with ALL prior reports
    (3 analysts + 6 bull/bear + 6 risk = 15 reports in
    `prior_reports`).
19. Quality gate; persistence; signals; CLI display.

Each round = one `objective_steps` row per agent per round, visible
via `mix allbert.objectives show <id>` for full inspection.

The multi-round cap is local to the native coordinator and bounded by
Settings Central. This keeps intermediate specialist turns private to
the plugin-owned agent graph while still recording every turn as a
durable v0.24 objective step for inspection.

## Flow 3: Explicit Python Comparison (`--engine python`)

Operator runs `mix stocksage.analyze AAPL 2026-05-17 --engine python
--user local`. Steps 1-7 identical to Flow 1.

8. After approval, RunAnalysis dispatches to
   `StockSage.TraderBridge.analyze/2` (v0.22 path; unchanged).
9. TraderBridge runs the Python TradingAgents graph (5-10 minutes
   for live LLM debate; <1s for `--force-stub`).
10. TraderBridge returns the v0.22 final-state fields.
11. RunAnalysis persists one `stocksage_analyses` row with
    `engine: "python"` (or `"tradingagents"` for legacy callers).
12. CLI displays: analysis id, ticker, date, engine ("python"),
    decision summary.

No native specialist agents are involved. v0.22 path runs unchanged.

## Flow 4: Parity Run (`--engine both`)

Operator runs `mix stocksage.analyze AAPL 2026-05-17 --engine both
--user local` (or `--compare-python` as syntactic sugar).

Steps 1-7 identical to Flow 1 (confirmation covers both engines).

8. After approval, RunAnalysis dispatches to
   `NativeCoordinator.parity_run/2`.
9. Coordinator starts native analysis (Flow 1 stages 8-16) AND Python
   bridge call (Flow 3 stages 8-10) concurrently under supervised
   runtime work.
10. Coordinator awaits both with timeout = `stocksage.bridge_timeout_ms`
    (default 10 min).
11. Native completes (typically ~60-120s); Python completes (typically
    ~5-10 min for live).
12. Coordinator computes parity diff:
    - `rating_agreement`: 5-point scale (Buy/Overweight/Hold/
      Underweight/Sell). Exact match = 1.0; adjacent = 0.5; distant
      = 0.0.
    - `confidence_delta`: |native_confidence - python_confidence|.
    - `within_variance`: `confidence_delta < stocksage.native_parity_variance`
      (default 0.25).
    - `parity_pass`: `rating_agreement >= 0.5 AND within_variance`.
13. RunAnalysis persists ONE `stocksage_analyses` row with
    `engine: "both"`, top-level final-state fields populated from
    the native result, `parity_diff` JSON populated.
14. The `stocksage_analysis_details.payload` JSON carries both
    `native_report` and `python_report` keys plus the `parity_diff`.
15. RunAnalysis emits
    `allbert.stocksage.native.parity_run.completed` with the
    `parity_pass`, `rating_agreement`, `confidence_delta`.
16. CLI displays: analysis id, both engines' decisions, parity result.

If one engine fails (e.g., Python timeout): the row persists with the
successful engine's data populated; `parity_diff` records the failure
and `parity_pass: false`. Operator can inspect both engines'
contributions independently via `mix stocksage.analyses show`.

## Flow 5: Cross-App Specialist Agent Invocation Via `mix allbert.delegate`

Operator runs (from anywhere â€” does NOT need to be in a StockSage
context):

```sh
mix allbert.delegate stocksage.market_context \
  --user local \
  --command execute \
  '{"ticker":"AAPL","analysis_date":"2026-05-17","evidence_mode":"fixture"}'
```

1. Mix task validates args; resolves `<agent_id>` against
   `AllbertAssist.Objectives.AgentRegistry`. If not found â†’ exit 65.
2. Mix task creates a transient debug objective:
   `Objectives.frame/2` with `title: "debug.delegate.<agent_id>"`,
   `status: :open`, no acceptance_criteria. Stored in `objectives`
   table.
3. Mix task creates one `objective_steps` row of
   `kind: :delegate_agent` with the supplied params + the resolved
   delegate_agent_id.
4. Mix task invokes
   `Actions.Runner.run("delegate_agent", %{objective_id, step_id,
   delegate_agent_id, command, params}, context)`.
5. The v0.24 `DelegateAgent` registered action authorizes via
   `:objective_write` permission (default `:allow`).
6. DelegateAgent dispatches the `execute` command to the resolved
   agent through the monitored AgentRegistry.
7. The agent runs (with `tools:` access to evidence actions; evidence
   actions hit Resource Access if `evidence_mode = "live"` and the
   operator has not granted access â€” they will be prompted for
   confirmation per v0.09+).
8. The agent returns its normalized report packet.
9. Mix task prints the report packet (bounded; redacted).
10. Mix task marks the debug objective `:completed` on success or
    `:failed` on error.

This flow proves that **any registered specialist agent is callable
from outside StockSage**. The Mix task lives in Allbert core, not
StockSage. Future Allbert apps (e.g., a research-assistant or v0.26
workspace shell) can call `stocksage.fundamentals` through the same
v0.24 DelegateAgent path without coupling to StockSage modules.

## Flow 6: Fixture-Mode Operator Smoke (No Market-Data Credentials Needed)

Operator who lacks market-data credentials or wants a quick smoke runs:

```sh
mix allbert.settings set stocksage.native_evidence_mode fixture
mix stocksage.analyze AAPL 2026-05-17 --engine native --user local
mix allbert.confirmations approve <id> --reason "fixture smoke"
mix stocksage.analyses show <id>
```

The native engine uses fixture data for all 5 evidence actions. With
the default `stocksage.native_llm_enabled = true`, the 9 non-quality
specialists still make Jido.AI calls; set the flag to `false` for a
credential-free deterministic smoke. Before M4, the M2/M3 skeleton-agent
smokes prove registration and fixture evidence without provider-backed
execution.

Fixtures ship under
`plugins/stocksage/priv/fixtures/native_agents/` for AAPL, MSFT, NVDA
at one date (`2026-05-15`). Fixtures are versioned and license-clear
(synthetic data; no redistribution of Yahoo Finance / StockTwits /
Reddit content).

## Flow 7: Per-Agent Single-Shot Smoke Through Delegate Boundary

Operator runs:

```sh
mix stocksage.agents smoke stocksage.market_context \
  --ticker AAPL \
  --analysis-date 2026-05-17 \
  --fixture \
  --user local
```

1. Mix task validates args; resolves `<agent_id>` against
   AgentRegistry. If not found â†’ exit 65.
2. Mix task creates a synthetic request packet with the supplied
   ticker, date, fixture flag, role=`:market_context`,
   task=`:produce_report`, `round_index: nil` (single-shot).
3. Mix task creates a transient debug objective and invokes the
   v0.24 `delegate_agent` registered action through
   `Actions.Runner.run/3`, using the same authority boundary as
   `mix allbert.delegate`.
4. Agent runs end-to-end: prompt/control file context â†’ tools (fetch
   evidence from fixture) â†’ Jido.AI generation when enabled (or
   deterministic advisory synthesis when disabled) â†’ report packet.
5. Mix task prints the bounded, redacted report packet.

This flow is for **agent-level smoke testing**. It is not a substitute
for operator-visible analysis runs (`mix stocksage.analyze` is for
that), and it does not persist to `stocksage_analyses`. It still uses
the registered action runner and Security Central so the smoke proves
the same reusable delegate-agent path that other Allbert apps will
use.

## Flow 8: Native Engine Failure + Optional Explicit Python Comparison

Operator runs `mix stocksage.analyze AAPL 2026-05-17 --engine native
--user local`. Confirmation flow as Flow 1 steps 1-7.

8. After approval, native engine starts; one specialist agent times
   out (e.g., decision_synthesizer takes longer than
   `stocksage.native_agent_timeout_ms`).
9. Coordinator records the failure; `quality_gate` rejects an
   incomplete synthesis; analysis row marked `status: "failed"`
   with bounded error reason.
10. RunAnalysis emits `allbert.stocksage.analysis_failed` (v0.22)
    AND `allbert.objective.blocked` (v0.24).
11. CLI displays: failure message + the analysis/objective ids to
    inspect. It may mention that the operator can run a separate
    explicit Python reference/comparison command if desired, but it
    must not present Python as automatic recovery.
12. Operator (manually) may run:
    ```sh
    mix stocksage.analyze AAPL 2026-05-17 --engine python --user local
    ```
13. Fresh confirmation (no auto-resume from the failed native one;
    `:stocksage_analyze` re-evaluates per v0.07).
14. Python path runs (Flow 3); succeeds; persists a new
    `stocksage_analyses` row with `engine: "python"`.

**No automatic native â†’ Python fallback** (Readiness Decision; ADR
0022 Decision section). Operator always re-invokes explicitly.

## Failure And Safety Cases

| Case | Expected behavior |
|---|---|
| Native engine: one specialist agent times out | Coordinator records failure; quality_gate rejects incomplete synthesis; analysis row marked `failed`; operator sees inspection guidance and may separately run an explicit Python reference/comparison command. |
| Native engine: model-provider credentials missing while `stocksage.native_llm_enabled = true` | Each affected specialist returns a bounded failed advisory packet; coordinator records failed steps/warnings; quality_gate rejects incomplete synthesis when the synthesizer cannot produce a valid decision; operator can disable native LLM for deterministic smoke or configure credentials. |
| Native engine: all 5 evidence actions in live mode but no Resource Access grants | First evidence action call returns `:needs_confirmation`; operator must approve each evidence fetch OR switch to fixture mode. |
| Native engine inside an approved analysis: evidence actions reuse parent approval | No additional confirmations for the evidence actions covered by the already-approved `RunAnalysis`; outside that parent context, Resource Access applies normally. |
| `--engine python` with `stocksage.python_comparison_enabled = false` | RunAnalysis rejects with bounded error; CLI exits 64. |
| `--engine both` with native success + Python failure | `stocksage_analyses` row persists with native data; parity_diff records Python failure; `parity_pass: false`. |
| `--engine both` with native failure + Python success | `stocksage_analyses` row persists with Python data; parity_diff records native failure; `parity_pass: false`. |
| `--engine both` with both failures | Analysis row marked `failed`; no partial row persisted. |
| Multi-round debate reaches native round caps | Coordinator stops after the configured Settings Central caps; every completed specialist turn remains visible as an `objective_steps` row. |
| `mix allbert.delegate` on non-existent agent id | Exit code 65 ("not found"); CLI prints `Agent <id> not found in AgentRegistry`. |
| `mix allbert.delegate` cancelled mid-flight via Ctrl-C | Debug objective marked `:cancelled` (v0.24 cooperative cancellation); in-flight agent call completes; result discarded. |
| `mix stocksage.agents smoke` on non-existent agent id | Exit code 65; CLI prints registry-not-found message. |
| Fixture mode requested but fixture file missing for ticker | Evidence action returns bounded error; agent records `status: :insufficient_evidence`; coordinator records but continues with available evidence. |
| Coordinator crash mid-analysis | JidoBacked supervisor restarts coordinator; eager rehydration loads in-flight objectives; analysis continues from last persisted step row. |
| Confirmation pending when coordinator restarts | Confirmation record remains pending (v0.23 store rehydrates); on operator approval, engine looks up linked confirmation via `confirmation_id` and continues. |
| Quality gate rejects synthesized result | `quality_gate.rejected` signal emitted; analysis row marked `failed`; bounded rejection reason persisted; operator can retry. |
| Parity diff computation fails (e.g., one engine returned malformed result) | Row persists with available data; parity_diff records the computation failure; `parity_pass: false`. |
| Operator cancels native analysis mid-debate via `mix allbert.objectives cancel` | v0.24 cooperative cancellation: in-flight specialist agent call completes; result recorded; no new rounds proposed; objective marked `:cancelled`; analysis row marked `cancelled`. |
| Cross-user `mix allbert.delegate` against another user's debug objective | Returns `:not_found` (v0.24 isolation rule); CLI exits 65. |

## Non-Goals Visible To Users

- No autonomous trading or order placement.
- No automatic native â†’ Python fallback.
- No LLM-assisted objective acceptance evaluation (v0.24 deterministic
  Evaluator continues to govern).
- No v0.26 workspace shell rendering of native agent steps.
- No v0.27+ StockSage LiveView rendering of richer native outputs.
- No public plugin hook contribution API for the coordinator pipeline.
- No unbounded native parallel step execution. v0.25 allows only
  stage-local fan-out where inputs are independent: 3 analysts, 3 risk
  perspectives per risk round, and explicit native/Python parity.
