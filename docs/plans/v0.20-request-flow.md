## Flow 1: Startup Discovery

Allbert starts with StockSage available as a source-tree plugin.

1. Plugin bootstrap discovers `./plugins/stocksage`.
2. The plugin registry records StockSage's declared app, action, skill,
   settings, and child-spec contributions.
3. App bootstrap registers the real `StockSage.App`.
4. The old `AllbertAssist.App.StockSageStub` is absent.
5. The skill registry discovers StockSage skills from the plugin skill root.
6. The action registry exposes the v0.20-safe StockSage actions.
7. Startup diagnostics are available through plugin and app inspection
   commands.

The operator observes: `mix allbert.plugins show stocksage` lists plugin,
app, action, and skill contributions. No stub appears.

## Flow 2: Plugin Manifest Inspection

The operator asks to inspect the StockSage plugin manifest.

1. `mix allbert.plugins show stocksage` reads the registered plugin entry,
   not the raw JSON file.
2. The display includes `plugin_id`, `kind`, `module`, `skill_paths`,
   contributed apps, actions, settings keys, and any diagnostics from bootstrap.
3. The manifest JSON at `plugins/stocksage/allbert_plugin.json` is available
   for operator inspection but is not the authoritative source at runtime.

The operator observes: the manifest fields and the compiled module match. If
they diverge (module missing, contributions list wrong), diagnostics appear in
the plugin registry output.

## Flow 3: App Registration And Active-App Flow

The operator or a session sets `active_app: :stocksage`.

1. Startup registers the real StockSage app contribution.
2. The v0.18 app/surface contract accepts the app.
3. The app registry stores `:stocksage` with surface, action capability, and
   skill path metadata.
4. Duplicate `:stocksage` entries are rejected with a diagnostic.
5. `mix allbert.sessions set-active-app --user alice --session sess-1 stocksage`
   succeeds because the app registry has the real app.
6. Subsequent runtime turns in `sess-1` carry `active_app: :stocksage`.

If `StockSage.App` fails validation, boot continues with diagnostics but
StockSage does not contribute app/actions/skills. The system does not silently
fall back to any stub.

The operator observes: `mix allbert.apps show stocksage` shows `StockSage.App`
with its declared surfaces, action capabilities, and skill paths.

## Flow 4: Intent Routing Turn With App Context

The user is in a session where `active_app: :stocksage` is set and submits
a natural-language request.

1. Runtime passes `active_app: :stocksage` to `AllbertAssist.Intent.Engine`.
2. The engine collects the global candidate set from all registries.
3. StockSage action candidates (`list_analyses`, `show_analysis`, `get_trends`,
   `queue_analysis`) are boosted because their `app_id` matches the session
   `active_app`.
4. The engine selects the highest-ranked valid candidate.
5. `active_app` boosts scores but does not grant additional permissions. All
   candidates still pass Security Central authorization.
6. The selected decision is returned to `AllbertAssist.Agents.IntentAgent` and
   executed through `AllbertAssist.Actions.Runner.run/3`.
7. If `--trace` is passed, the response includes an Intent Candidates section
   naming the selected candidate, its `app_id`, and top rejected candidates
   with rejection reasons.

The user observes: StockSage-relevant responses feel natural inside a StockSage
session. The trace shows `active_app` affinity as a ranking reason, not a
permission grant.

## Flow 5: Intent Routing Turn From Neutral Context

The user is in a neutral Allbert session (`active_app: :allbert`) and asks
about StockSage analyses.

1. Runtime sets `active_app: :allbert`.
2. The engine collects the global candidate set.
3. StockSage candidates are not boosted.
4. If the request text contains explicit StockSage action or app evidence, the
   engine may select a StockSage candidate when it passes registry and Security
   Central validation.
5. If the request is ambiguous or lacks explicit evidence, the engine selects a
   core Allbert response, a clarification, or a surface navigation decision.
6. StockSage actions are never executed silently from neutral context.

The user observes: neutral requests do not trigger StockSage actions without
explicit intent. The trace, if requested, shows why StockSage candidates were
not promoted.

## Flow 6: Domain Storage

StockSage reads and writes through `AllbertAssist.Repo` using the shared
Allbert SQLite database.

1. Reviewed shipped plugin migrations in
   `plugins/stocksage/priv/repo/migrations` create `stocksage_*` tables when
   the Allbert host includes plugin migration paths.
2. StockSage Ecto schemas in `plugins/stocksage` reference
   `AllbertAssist.Repo`.
3. Context functions accept `user_id` as a required parameter. They do not
   default to any user unless at a CLI/operator boundary.
4. List and get functions filter by `user_id`. Alice's rows are never returned
   to Bob.
5. Read-by-id flows also require `user_id`; ids from another user return
   not-found, not data.
6. Contexts normalize ticker symbols to uppercase and reject blank symbols.

No `StockSage.Repo` is created. No PostgreSQL connection is added.

## Flow 7: Import Flow

The operator imports a legacy SQLite database.

```sh
mix stocksage.import_sqlite /path/to/stocksage.db --user alice --dry-run
mix stocksage.import_sqlite /path/to/stocksage.db --user alice
```

1. CLI validates the path is a local file. Remote URIs, shell expansions, and
   paths outside the filesystem are rejected.
2. CLI opens the source database read-only.
3. Importer inspects expected legacy tables; unknown tables produce warnings
   when `stocksage.import.unknown_tables_as_warnings` is true.
4. Importer maps legacy rows into StockSage changesets with `user_id`,
   `source: "legacy_sqlite"`, `legacy_source`, `legacy_id`, and
   `metadata.imported_at`.
5. Dry-run mode reports expected counts without writes.
6. Write mode inserts or updates rows idempotently using `legacy_id` as
   conflict key.
7. Output reports inserted, updated, skipped, invalid, and elapsed time per
   table.

The operator observes: running the same import twice shows updates/skips, not
duplicates. Alice's import rows are not visible under Bob's user scope.

No Python is executed. No market-data call is made. No remote document is
fetched. No skill is imported.

## Flow 8: Queue Flow

The operator or a skill creates an analysis queue entry.

```sh
mix stocksage.queue create AAPL --user alice --thread-id THREAD_ID
```

1. CLI normalizes the symbol to uppercase.
2. CLI resolves `user_id`; if `--user` and any legacy alias differ, the command
   fails before touching the database.
3. Context creates a `stocksage_analysis_queue` row with `status: :queued`.
4. No bridge worker starts. No market-data call happens.
5. Output shows id, user_id, symbol, status, and timestamps.

The operator observes: a durable queue row. The queue is preparation for
v0.22/v0.25 (formerly v0.22/v0.23) execution; it does nothing on its own in
v0.20.

## Flow 9: StockSage Action Execution

A StockSage action runs through the standard Allbert action runner.

`StockSage.Actions.QueueAnalysis`:

1. Runs through `AllbertAssist.Actions.Runner.run/3`.
2. Validates symbol, user_id, and optional thread/session context.
3. Authorizes `:stocksage_write` through Security Central.
4. Creates a queue row through the StockSage context.
5. Emits standard action lifecycle signals and trace entries.
6. Returns a structured queued result.

`StockSage.Actions.ListAnalyses`, `ShowAnalysis`, `GetTrends`, and
`ListQueue`:

1. Run through `AllbertAssist.Actions.Runner.run/3`.
2. Read only local SQLite data; no market-data requests.
3. Filter by user context; show-by-id returns not-found for another user's id.
4. Return bounded summaries, not unbounded raw metadata payloads.

All v0.20 actions carry `app_id: :stocksage`. App id does not authorize
cross-app routing or elevate permissions.

`StockSage.Actions.ImportSqlite` is an internal operator action used by
`mix stocksage.import_sqlite`; it authorizes `:stocksage_write` before opening
the local source database. v0.20 actions never default `user_id` internally.
The `"local"` default belongs only to CLI/operator entrypoints.

## Flow 10: Skill Invocation

StockSage skills are discovered from `plugins/stocksage/skills/`.

Expected skills in v0.20: `queue-analysis`, `list-analyses`, `show-analysis`,
`get-trends`. Each skill names only v0.20-safe actions.

1. `AllbertAssist.Skills.Registry` discovers skills from the registered skill
   path.
2. The intent engine may select a skill candidate for a matching user request.
3. The skill executes its named action through the normal action runner.
4. Security Central authorizes the action as it would any other.
5. `run-analysis` is absent; any prompt that resolves to it returns an
   unsupported-workflow response.

The user observes: `mix allbert.skills list` shows StockSage skills alongside
core skills. Skill invocation traces include the `plugin_id: "stocksage"` field.

a running node, while `--active-app stocksage` supplies equivalent one-turn app

silent StockSage action execution.

### Verifying Import And User Isolation

```sh
mix stocksage.import_sqlite /path/to/stocksage.db --user local --dry-run
mix stocksage.import_sqlite /path/to/stocksage.db --user local
mix stocksage.analyses list --user local
mix stocksage.analyses show ANALYSIS_ID --user local
mix stocksage.analyses list --user other
```

The user observes: dry-run reports counts but writes nothing. Write mode is
idempotent. The `other` user sees no rows from the `local` import. If `other`
tries to show a `local` analysis id, the command reports not-found.

### Verifying Queue Creation

```sh
mix stocksage.queue create AAPL --user local
```

The user observes: a queued row with id, user_id, symbol, and status. No
execution starts.

## Failure And Safety Cases

| Situation | Expected behavior |
| --- | --- |
| `plugins/stocksage` directory missing | Bootstrap skips, logs diagnostic, no StockSage contributions. |
| Manifest `module` field names an uncompiled module | Bootstrap rejects plugin, logs diagnostic, no StockSage contributions. |
| `StockSage.App` fails v0.18 validation | Boot continues; no StockSage app/actions/skills registered; no stub fallback. |
| `active_app: :stocksage` set but StockSage not registered | Engine falls back to `:allbert` behavior; logs bounded diagnostic; does not atomize unknown app id. |
| Import `--user` and legacy alias differ | Import fails before opening source database. |
| Import source path is a remote URI | CLI rejects before any database open. |
| Import row has unknown legacy table | Warning logged per `stocksage.import.unknown_tables_as_warnings` setting; row skipped. |
| Same import run twice | Second run shows updates/skips; no duplicate rows. |
| User asks to show another user's analysis id | Returns not-found; durable ids are not authorization. |
| StockSage action requested without sufficient permission | Security Central returns refusal or confirmation-blocked decision; no execution. |
| `permissions.stocksage_write` is denied | Queue creation returns a permission refusal; no queue row is created. |
| `run-analysis` skill or action named | Returns unsupported-workflow response; no bridge or market-data call starts. |
| Queue created but no v0.22 worker present | Queue row is durable and idle; nothing executes. |

## Non-Goals Visible To Users

- StockSage will not execute Python in v0.20.
- StockSage will not call external market-data APIs or fetch remote documents.
- `StockSage.Actions.RunAnalysis` does not exist in v0.20; requests for it
  return an unsupported-workflow response.
- StockSage will not mount LiveViews, create routes, or render canvas
  components.
- StockSage will not promote conversation turns into markdown memory or use
  v0.21 reviewed-memory retrieval before that milestone exists.
- `active_app: :stocksage` boosts StockSage candidates but does not grant
  StockSage actions elevated permissions or bypass Security Central.
